## Page 12

第 1 章
初识RocketMQ

随着分布式限构的盛行，服务被水平拆分和重直拆分，服务与服务之间的调用关系越来越复杂，
服务治理的技术挑战性也越来越大。例如，将拆分之前和拆分之后的调用链路关系复杂度进行对比，
后者通常都是前者的好几倍。架构师和技术专家在面对复杂的业务场最带来的技术挑战时，为了保
证核心业务的高吞吐量，通常要考虑将核心业务的接口退行异步设计。
消息中间件作为异步设计的一种技术和架构手段，不仅能解胰服务之间的强依赖关系，还能最
大限度地提升服务的QPS处理功能。开源的消息中间件有很多，RockeiMQ就是其中之一。它基
于分布式集群管理技术，并提供高并发、高性能和高可用的消息发布和订阅服务。
下面将会带着大家一起进入RocketMQ的世界。
1.1认识分布式架构
对于“分布式限构”，技术人员要用“技术小白”的视角认识它。
人的大脑在认知未知的知识领域时，如果用系统思维认知斯的知识，则通常要经历“从部分到
整体，再从整体到部分”的思考过程，因此，本节将分布式架购拆分为“分布式”和“架购”两部分。

名字服务器充当路由消息的提供者。生产者或消费者能通过名字服务器查找各主题相应的

推动式消费是Consumer消费的一种类型。在该模式下，Broker Server在收到数据后会主动

3.2NameServer的架构
Name Server是无状态的。我们可以先认识一下无状态架构，这样才能更加深刻地理解Name
Server的果构思部。
3.2.1认识无状态架构
1.什么是无状态服务和有状态服务
无状态服务（Stateless Service）是指在运行服务对应的实例时，不会在实例所在的机器节点
中持久化数据（缓存除外），并且多个实例对于同一个请求响应的结果是完全一致的。
无状态服务可以被任意创建和销级，服务中的数据都不会丢失。调用无状态服务的应用，可以
任意切映到无状态服务集群中的一个实例，而不会影响应用调用的可用性。
口提示：
典型的无状为设计有HTTP的无状为的速接和通信机制，Serveriess的无状为的品数调用机制等。
有状态的服务（Statefui Service）是指服务需要记录访问者的信息，并识别访问者的身份，根
据访问者的身份进行请求处理。
有状态的服务需要将数据持久化。所以，有状态服务只能同时有一个实例提供服务，因此它不
支持“自动化扩容”
提示：
一般来说，数据年服务、存储在本地文件系统中的配置文件成其他水久数据的应用程序可以采用
有状态服务模式。典型的有状态设计有业务状态机制、事务机制及Tomcat中的Sesion机制导。
2.为什么需要无状态化
随着服务从单体聚构向分布式限购的演进，服务无状态化对提升服务的稳定性和可用性显得尤
其的重要。在分布式架构中，服务无状态化的具体原因如下。
（1）业务被拆分为多个微服务，需要动态扩客，抗住流量洪峰。
当业务被拆分为多个微服务之后，为了抗住流量洪峰，每个微服务会采用多进程部署，每个进
程又会做主/从架构（如“一主多从”）。那么服务在不做重构的前提下，是否能够实施和落地“多主
多从”的分布式架构？答案是否定的。

4.2.6事务消息
在厘清什么是事务消息之前，我们先来看一看什么是消息。
从广文上来看，消息就是数据，发送端发送消息，接收端解析消息并处理消息。从狭义上来看，
消息可以分为很多种类型，如图片、文字、语音和视频等都是消息。
RockeMQ在普通消息定义的基础上，对事务消息扩展了两个相关的概念：半消息（预处理消
息）和消息状态回查。
( 1 ) Half ( Prepare ) Message; 半消息 (预处理消息)。
半消息是一种特殊的消息类型，该类型的消息暂时不能被Consumer消费，当一条事务消息被
成功投递到 Broker Server 上，但是 Broker Server 并没有接收到 Producer 发出的二次确认时，
该事务消息就处于“暂时不可被消费”状态，该状态的事务消息被称为“半消息”。
(2) Message Status Check;消息状态回重。
网格科动、 Producer 董启等原因可能金导数 Producer 网 Broker Server 发送的二次确认消
息没有成功送达。如果BrokerServer检测到某条事务消息长时间处于半消息状态，则主动向
Producer 端发起回查操作,查询该事务消息在 Producer 端的事务状态 （Commit 或 Rolback)。
可以看出，MessageStatus Check主要用来解决分布式事务中的超时问题。

2.事务消息的应用场题
在业务服务中，事务消息的应用场累非常多，最常见的就是微服务架构中的分布式事务的场景。
在完成业务服务拆分后，开发人员需要考虑购服务之间调用引起的数据库中数据一致性问题。
RockeMQ的事务消息是采用两阶段提交和回查本地事务状态（又被称为“第二阶段”提交的
补偿)实现的。
（1）“第一阶段”采用“同步”模式，确保“生产事务预处理消息”的可靠性。
（2）“第二阶段”采用“最多发送一次”模式，确保结束分布式事务的吞吐量和高性能。如果需
要回查本地事务的状态，则重新发起“第二阶段”。为了确保结束分布式事务的吞吐量和高性能，
RocketMQ是使用线程来异步处理的。
总体来说，RocketMQ事务消息的性能还是短高的。
