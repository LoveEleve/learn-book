# 内 容 提 要

本书从 JVM 和 Java API 两个方面，深入地介绍了 Java 8 和 Java 11 等新的 Java 长期支持版本中影响性能的因素。本书先从Java 应用程序的通用方法、基准测试的缺陷、性能监控工具等方面，分析了如何通过 JVM 的配置方式影响程序的性能；接着从即时编译、垃圾回收、堆内存与原生内存最佳实践、线程与同步的性能等方面，分析了常见的优化方向及其特性。第 2 版专注于更加成熟的 Java 8 和 Java 11，主要的更新内容涉及 G1 垃圾回收器和 Java 飞行记录器，以及Java在容器化环境下的性能变化。

本书面向关注性能的 Java 工程师，尤其是想了解 JVM 和 Java API 的各个方面如何影响性能的开发人员。

# 前言

当 O’Reilly 第一次找我，让我写一本关于 Java 性能优化的书时，我心存疑虑。我心想：Java 性能不是已经没什么好写的了吗？的确，我每天仍然致力于提升Java 应用程序（和其他应用程序）的性能，但是我更认为自己大部分时间在优化低效的算法和处理外部系统的瓶颈，而不是解决任何与Java性能优化直接相关的问题。

思考片刻后我确信，我像往常一样欺骗了自己。端到端的系统性能优化的确耗费了我大量的时间，我时不时也会碰到原本可以用 ${ \cal O } ( \log N )$ 1 算法却用了 $O ( n ^ { 2 } )$ 算法的代码。然而实际上，我每天仍会考虑垃圾回收（GC）性能与 JVM 编译器性能，也会思索如何从 Java API中获得最佳性能。

这并不是要抹去 20 多年来 Java 和 JVM 在性能方面的巨大进步。20 世纪 90 年代后期，当我在 Sun 公司担任 Java 布道师时，唯一真正可用的“基准测试”工具是Pendragon 软件的CaffeineMark 2.0。由于种种原因，该工具的设计很快就限制了其使用价值。然而在那时，我们很喜欢告诉大家，基于这个基准测试，Java 1.1.8 的性能比 Java 1.0 的性能高 8 倍。这当然是真的，因为Java 1.1.8有了真正的即时编译器，而Java 1.0几乎完全是解释型的。

之后，标准委员会开始制定更严谨的基准测试，Java 的性能围绕这些基准测试开始提升。结果，JVM 的各个方面都有了长足的进步，包括垃圾回收器、编译器和 API。这个过程如今当然还在持续，但一个重要的事实是，性能优化变得越来越难。通过引入即时编译器实现 8 倍的性能提升，这在工程上很简单。尽管编译器还在持续改进，但我们再也无法实现这样大的提升了。垃圾回收器的并行化也带来了巨大的性能提升，但是这些最近的改进更多是渐进式的。

这是典型的应用程序发展历程（JVM 本身也是应用程序）：在项目初期，很容易找到架构上的改进点（或代码缺陷），改进后会带来巨大的性能提升。在成熟的应用程序中，很少能找到这样的性能改进点。

在很大程度上，我最初担心的是工程界对于 Java 性能可能已经没什么可做的了。不过，有几件事让我确信自己错了。首先，我每天都会遇到关于在特定环境中运行 JVM 的这样或

那样的问题。新的工程师源源不断地进入 Java 领域，JVM 的运行机制在某些方面又比较复杂，因此，对其操作给出指导仍有裨益。其次，计算环境的变化似乎也改变了工程师如今面临的性能问题。

在过去的几年里，人们对性能的关注已经分化。一方面，大型机器已经很常见了，其上可运行使用大量堆内存的JVM。JVM 已经使用了新的G1 垃圾回收器来处理这些内存。作为一种新的技术，G1 垃圾回收器比传统垃圾回收器更需要手动优化。另一方面，云计算已经重新证明了小型单 CPU 机器的重要性：你可以去 Oracle、Amazon 或者其他公司租用一台便宜的单 CPU 机器，用来运行小型应用程序服务器。（你并不会得到一台真正的单 CPU机器，实际上你得到的是一台非常大的机器上的虚拟 OS 镜像，但是这个虚拟 OS 被限制只能使用一个 CPU。对于 Java 应用程序来说，这和运行在单 CPU 机器上是一样的。）在这些情况下，正确管理少量内存就变得非常重要。

Java 平台也在不断进化。每个新的 Java 版本都会提供新的语言特性和新的 API，虽然这并不一定能提升应用程序的性能，但能提高开发人员的生产力。善用这些语言特性有助于使缓慢迟钝的应用程序顺畅运行。平台的进化带来了一个值得注意的性能关注点：在两个程序之间用 JSON 交换信息，毫无疑问比使用高度优化的专有协议更加简洁和高效。为开发人员节省时间确实可以提升生产力，但是确保在生产力提升的同时性能也提升（至少打平）才是真正的目标。

如果你的网站要在周一早晨上线，而你直到周日深夜还在为一个性能问题寻找快速解决方案，那么本书不适合你。

Java 性能优化工作的一个有趣的方面是，开发人员的技术背景往往与性能组或QA 组的工程师截然不同。我认识一些开发人员，他们可以记住一些不常用的 API 方法签名，那些方法签名数量庞大又晦涩，但是他们不知道 -Xmn 标志的含义。我还认识一些测试工程师，他们通过给垃圾回收器设置各种标志来竭力地提升程序性能，却几乎无法写出一个像样的Hello World 程序。

Java 性能优化包括两个方面：一是编译器和垃圾回收器等的标志优化，二是 API 的最佳实践。本书假设你对如何编写 Java 应用程序有很好的理解，即使你的兴趣点不在 Java 的编程方面，我也会用相当大的篇幅讨论应用程序，包括含有大量数据的示例应用程序。

如果你的主要兴趣是 JVM 本身的性能优化，即不通过修改代码进行 JVM 优化，那么本书的大部分内容仍然对你有益。你可以随意跳过编码的部分，只专注于你感兴趣的内容。在这个过程中，也许你会深入了解 Java 应用程序如何影响 JVM 性能，并能给开发人员提出建议，从而使你的性能优化工作变得更轻松。

本书讨论 Java 性能优化的艺术与科学。

Java 性能优化是一门科学，这样说并不令人惊讶：关于性能的讨论包含大量数字、评测和分析。大部分性能优化工程师拥有科学类的学科背景，而且实现最佳性能的关键就是严谨地运用科学方法。

那么 Java 性能优化是艺术吗？性能优化既是一门科学，又是一门艺术，这种说法并不新鲜，但是我们在讨论性能时很少明确地承认这一点。部分原因是“艺术”的概念与我们的调试过程相悖。对于一些人来说，看起来是艺术的东西，基本上都基于丰富的知识和经验。而足够先进的技术和魔法无异，就像对于圆桌骑士来说，手机真的拥有魔法。同理，优秀的性能优化工程师的工作成果犹如艺术作品，只不过这些艺术作品源于渊博的知识、丰富的经验和敏锐的直觉。

本书不能在经验和直觉方面提供帮助，但是可以帮助你积累丰富的知识。积年累月地使用这些知识，能帮助你获得优秀 Java 性能优化工程师的必备技能。本书的目标是让你深入理解Java平台的性能。

这些知识分为两大部分。第一部分关于 Java 虚拟机（Java Virtual Machine，JVM）的性能优化，即如何通过 JVM 的配置方式影响程序的性能。使用其他语言的资深开发人员可能认为 JVM 优化有点令人厌烦，但实际上，JVM 优化类似于 $\mathrm { C } { + + }$ 程序员在编译阶段测试和挑选编译器参数，或者 PHP程序员在 php.ini 文件中设置合适的变量。

第二部分关于理解 Java 平台的特性如何影响性能。注意，这里使用了平台这个词：有些平台特性（比如线程和同步）是 Java 语言的一部分，而有些平台特性（比如字符串处理）是Java 标准 API 的一部分。尽管 Java 语言和 Java API 有着显著的区别，但在本书中，两者会被同等对待。这两方面在本书中都会讲到。

JVM 的性能主要取决于标志优化，而平台的性能更多地取决于是否在代码中应用了最佳实践。一直以来，开发人员只负责编写代码，性能优化组只负责测试并提出性能优化建议，这两方面被认为是独立的专业领域。这种区分从来就不太有用，因为任何使用 Java 工作的人都应该理解代码在 JVM 中的行为，也应该知道什么样的优化有助于提升性能。当项目转向 DevOps 模式后，这种区分开始变得模糊。只有全面掌握领域内的知识，才能让你的作品更具艺术气息。

# 1.2 平台和约定

本书讲的是 Java 性能，它受一些因素的影响：Java 本身的版本，以及对应的硬件和软件平台。

# 1.2.1 Java平台

本书介绍 Oracle HotSpot JVM 和 Java 开发工具包（JDK）的性能优化，对应的版本是 Java8 和 Java 11。这也是我们所熟知的 Java 标准版（Java SE，Java Standard Edition）。Java运行时环境（Java Runtime Environment，JRE）是 JDK 的子集，只包含 JVM，但是 JDK中的一些工具对性能分析很重要，所以 JDK 是本书的重点。实际上，本书还会介绍从OpenJDK 仓库衍生出来的其他平台，包括 AdoptOpenJDK 项目中的 JVM。严格意义上，在生产环境中使用 Oracle 公司的二进制文件需要得到许可，而 AdoptOpenJDK 的二进制文件使用的是开源协议，无须得到许可便能使用。对本书来说，两者是一样的，我们都称之为 JDK 或 Java 平台 1 。

这些大版本经历了许多 bug 修复版本。在我写这段话的时候，最新的 Java 8 版本是jdk8u222（222 版本），Java 11 版本是 11.0.5。如果不能使用更新的版本，至少要使用这两个版本，这很重要，特别是在用Java 8的时候。Java 8的早期版本（大约到jdk8u60）不涉及本书讨论的很多重要的性能增强措施和特性，尤其是关于垃圾回收和G1垃圾回收器的。

之所以选择上述版本的 JDK，是因为可以得到 Oracle 公司的长期支持（LTS）。Java 社区可以自由地发展自己的支持模式，但是目前他们都遵循了 Oracle 公司的模式。所以，上述版本在一段时间内都可以使用，并会得到免费的维护、更新等支持：Java 8 至少可以用到2023 年（前期通过 AdoptOpenJDK，后期通过延长的 Oracle 公司支持协议）；Java 11 至少可以用到 2026 年。

至于过渡版本，虽然 Oracle 公司和整个社区都已经不再支持了，但在讨论Java 11 的时候必然会涉及 Java 9 和 Java 10 引入的特性。有时候，我提及的版本信息会不准确，我可能会说 X 特性和 Y 特性最初是由 Java 11 引入的，而其实它们在 Java 9 和 Java 10 中就已经使用了。Java 11 是首个包含这些特性的 LTS 版本，而且 Java 9 和 Java 10 已经不再使用，那么某个特性最初的出现时间就不重要了。同样，在本书出版时 Java 13 会发布，但是本书并不会涵盖 Java 12 和 Java 13 的很多特性。你可以在生产环境中使用这些版本，但是 6个月之后就需要升级到更新的版本了。（当你读到这里的时候，Java 12 可能已经不被支持了，如果正在使用Java 13，那它马上也会被Java 14替代。）我们会粗略地了解这些过渡版本的一些特性，但是因为这些版本在大多数情况下不会用于生产环境，所以本书的重点还是 Java 8 和 Java 11。

Java 语言规范还有其他的实现，包括开源实现的分支。AdoptOpenJDK 提供了其中一个开源实现（Eclipse OpenJ9），剩下的由其他厂商提供。虽然所有平台必须通过兼容性测试才能使用 Java 名称，但是本书讨论的内容不会扩展到兼容性方面，调优标志尤其如此。所有的JVM实现都有一个或多个垃圾回收器，而在每个厂商的GC实现中，调优标志都是其产品所特有的。因此，虽然本书所述概念适用于任何 Java 实现，但具体的调优标志和建议只适用于 HotSpot JVM。

调优标志及其默认值会随着版本的变化而变化，早期版本的 HotSpot JVM 也是如此。本书讨论的调优标志对Java 8（特别是 222 版本）和 Java 11（特别是 11.0.5 版本）有效。这些信息在之后的版本中可能会有细微的变化。请随时查阅版本说明以知晓重要的变化。

在 API 层面，不同的 JVM 更具兼容性。尽管如此，在 Oracle HotSport Java 平台和其他平台上，某一特定类的实现可能仍然存在微小的差异。这些类在功能上等效，但是实际的实现方式可能有所不同。幸运的是，这种情况很少见，不太可能对性能产生重大影响。

在本书的剩余内容中，Java 和 JVM 特指 Oracle HotSpot 实现。严格意义上，“JVM 在第一次执行时不编译代码”这种说法是错误的。有些 Java 实现在第一次执行时的确会编译代码，但无论是写起来还是读起来，JVM 都要比Oracle HotSpot JVM简单得多。

# JVM 调优标志

除了少数例外，JVM 接收两种标志：布尔标志和附带参数的标志。

布尔标志使用的语法是：-XX:+FlagName 表示开启，-XX:-FlagName 表示关闭。

附带参数的标志使用的语法是：-XX:FlagName=something，表示设置 FlagName 的值为 something。其中，something 指表示任意值的符号。例如，-XX:NewRatio=N 表示 NewRatio 标志可以设成任意值 $N$ （ $N$ 的含义将是讨论的重点）。

本书在介绍每个标志时都会讨论其默认值。默认值的设定通常基于两个因素：JVM 所运

行的平台和传给 JVM 的其他命令行参数。如有疑问，可参见 3.2.1 节。该节展示了当给出特定命令时，如何使用 -XX:+PrintFlagsFinal 标志（默认为 false，即“关闭”状态）来确定特定环境中特定标志的默认值。根据运行环境自动调优标志的过程被称为自行优化（ergonomics）。

从 Oracle 网站和 AdoptOpenJDK 网站下载的 JVM，被称作产品版 JVM。当从源码构建JVM 时，可以构建多个其他版本：调试版、开发版等。这些版本通常有其他的附加功能。特别是开发版，它包含更多的调优标志，让开发人员可以尝试 JVM 各种算法最微小的操作。本书基本不涉及这些标志。

# 1.2.2 硬件平台

本书第 1 版出版时，硬件环境和今天不同。那时多核机器很受欢迎，但是 32 位处理器和单核 CPU 仍大量使用。现在使用更多的是虚拟机和软件容器。下面概述这些平台对本书主题的影响。

# 1. 多核硬件

如今，几乎所有的计算机都有多个核心。对于 JVM（或任何其他程序）来说，这表现为多个 CPU。每个核心往往都应用了超线程技术。超线程是 Intel 常用的术语，AMD（和其他厂商）则使用同时多线程，一些芯片制造商称之为核心内的硬件线程（hardware strandswithin a core）。它们指的是同一个东西，本书将这种技术称为超线程。

对于性能来说，一台机器的核心数量很重要。以一台 4 核机器为例：（多数情况下）每个核心都可以独立于其他核心运行，所以一台4 核机器可以实现单核机器4 倍的吞吐量（当然这还取决于软件情况）。

在大多数情况下，每个核心会包含两个硬件或超线程。这些线程并不是相互独立的，一个核心每次只能运行一个线程。线程有时候会暂停。例如，从主内存加载值时线程会暂停，这个过程需要几个 CPU 周期。单核心运行单线程时，线程暂停，这些 CPU周期就浪费了。单核心运行两个线程时，核心可以切换并执行另一个线程的指令。

所以，如果 4 核机器开启超线程，似乎就可以同时执行 8 个线程的指令（尽管从技术上讲，每个 CPU 周期只能执行4 条指令）。这对于操作系统，也就是Java 和其他应用程序来说，机器可以表现得像拥有 8 个 CPU 一样。但从性能的角度来讲，这些 CPU 并不等价。如果运行一个 CPU 密集型任务，它只会使用 1 个核心；运行两个 CPU 密集型任务才会用到第 2 个核心；以此类推，直到第 4 个核心。也就是说，独立运行 4 个 CPU 密集型任务可以得到4倍的吞吐量。

如果添加第 5 个任务，它只能在任何其他任务暂停时运行。该任务的运行时间平均占总时间的 $20 \%$ 到 $40 \%$ 。额外的任务都面临这一挑战。因此，添加第 5 个任务只能提高约 $30 \%$ 的性能。结论就是，使用超线程技术得到的8个CPU能使机器性能达到单核的五六倍。

你将在后文的几个章节中看到这个例子。垃圾回收很大程度上是 CPU 密集型任务，所以第 5 章介绍了超线程如何影响垃圾回收算法的并行化。第 9 章大致讨论了如何充分利用Java的多线程能力，你也会看到扩展超线程核心的例子。

# 2. 软件容器

这些年，Java 部署的最大变化是，应用程序经常部署在软件容器中。这一改变当然不只局限于Java。这是行业发展的趋势，迈向云计算的步伐加快了这一趋势。

有两个容器很重要。一个是虚拟机，它在运行的硬件子集上建立了完全隔离的操作系统副本。云计算的基础是其厂商必须有包含大型机器的数据中心。这些机器可能有 128 个核心，不过由于成本原因，有些机器可能核心数较少。从虚拟机的角度讲，这并不重要，因为虚拟机只可以访问硬件的子集。因此，给定的虚拟机可能有 2 个核心（4 个 CPU，因为它们通常支持超线程技术）和16 GB内存。

从 Java 和其他应用程序的角度讲，这个虚拟机和常规的 2 核 16 GB 内存的机器的区别很小。出于本书讲解优化性能的目的，你把它们当成一样的就行。

另一个比较重要的容器是Docker。Docker容器只是操作系统中的一个进程（可能受资源限制），因此，运行于其内的 Java 进程不一定知道它运行于其中（即便可以检测出来）。也正是如此，它对于其他进程的 CPU 和内存的隔离有些不同。你会在本书中看到 Java 8 的早期版本（192版本之前）和后期版本（包括Java 11）在处理方式上的异同。

# 虚拟机超额分配

云厂商可以在物理机上对虚拟机进行超额分配。假设物理机有32个核心，云厂商通常会部署8个4核虚拟机，这样每个虚拟机预期有4个专用核心。

为了省钱，厂商也可以部署 16 个 4 核虚拟机。原理是，16 个虚拟机不太可能同时处于工作状态；如果其中一半的虚拟机在工作，就有足够的物理核心满足它们；如果太多核心处于工作状态，它们就会争夺CPU周期，机器性能就会受到影响。

同样，云厂商还可以对虚拟机使用的 CPU 降频。这就允许虚拟机在突发情况下使用它所分配的 CPU，但是不会长时间维持这个状态。这在免费或者试用的产品中经常见到，此时你对性能会有不同的预期。

这些对性能的影响很大，并且不局限于影响Java。其影响对于 Java 和虚拟机中运行的其他程序来说并无区别。

默认情况下，Docker 容器可以自由使用机器上的所有资源，包括所有可用的 CPU 和内存。如果使用 Docker 仅仅是为了在机器上快速部署单一应用程序，并且机器只运行这一个 Docker 容器，那没问题。但其实经常需要在机器上运行多个 Docker 容器，还要限制每个容器的资源。鉴于我们的4 核机器有 16 GB 内存，我们可能希望运行两个Docker 容器，每个可获取2个核心和8 GB内存。

配置 Docker 来达到这个预期很简单，但在 Java 层面可能出现复杂的问题。这是因为大量的 Java 资源是根据运行 JVM 的机器的大小自动（或自行）分配的，包括默认的堆大小和垃圾回收器使用的线程数量。关于垃圾回收器的详细内容会在第 5 章讲到，一些线程池的设置会在第9章讲到。

如果你使用的是最新版本的 Java 8（192版本或更新的）或者 Java 11，那么 JVM会像你希望的那样处理上述问题：如果限制 Docker 容器只能使用 2 个核心，本来基于机器 CPU 数量设置的值，就会根据 Docker 容器的限制 2 进行调整。同样，堆和其他默认基于机器内存数量的设置，也会随着Docker容器的限制而调整。

在Java 8的早期版本中，JVM无法得知任何容器的限制：当检测机器剩余内存以确定默认的堆大小时，它会检测机器的所有内存，但我们更希望看到的是允许 Docker 容器使用的内存。同样，检查有多少 CPU 可以用来优化垃圾回收器时，它会检查机器上所有的 CPU，而不是只检查分配给 Docker 容器的 CPU。结果是，JVM 运行状态不理想：启用的线程过多，设置的堆过大。线程过多会导致性能下降，但真正的问题出在内存上：堆的最大大小比分配给 Docker 容器的内存还大，当堆增大到这个大小时，Docker 容器（以及 JVM）会停止运行。

在早期的 Java 8 版本里，可以手动设置合理的内存和 CPU 数量。在进行这些优化时，我会指出这种情况下需要怎么调整，但最好还是升级到Java 8的后期版本或者Java 11。

Docker 容器给 Java 提出了另一个难题：Java 有丰富的性能问题诊断工具集，而 Docker 容器里没有。我们将在第3章进一步讨论这个问题。

# 1.3 全面的性能

本书重点关注如何最好地利用 JVM 和 Java 平台 API，让程序能运行得更快。但是很多外部因素也会影响性能。这些外部影响因素会时不时地在书中出现，它们并不是 Java 独有的，所以不一定会详细讨论。要知道，JVM 和 Java 平台的性能只是性能优化主题的一小部分。

本节介绍了一些外部影响因素，这些因素和本书讨论的 Java 性能优化话题一样重要。本书以 Java 为基础的内容可以与这些因素互为补充，但其中许多已经超出了我们要讨论的范围。

# 1.3.1 写出更好的算法

本书会讨论Java影响应用程序性能的很多细节和大量调优标志，只是没有如-XX:+RunReallyFast的神奇设置。

归根结底，程序的性能取决于程序怎么写。如果使用循环遍历数组中的所有元素，那么JVM 可以优化数组的边界检查方式，让循环运行得更快，还可以展开循环操作以提供额外的加速。如果使用循环是为了查找某个特定的元素，那么世界上还没有可行的优化方式，能让基于数组的代码和使用哈希映射（hash map）一样快。

好的算法对于提升性能是至关重要的。

# 1.3.2 写更少的代码

我们中有些人写代码是为了赚钱，有些是为了好玩，有些是为了回馈社区，但我们所有人都在写代码，或在写代码的团队中工作。你很难觉得删减代码是在为项目做贡献，因为有些管理者还会用代码行数来评估开发人员。

我明白这种情况，但矛盾的是，写得好的小程序比写得好的大程序跑得快。对于所有的计算机程序都是如此，这自然也适用于 Java 程序。需要编译的代码越多，代码快速运行之前需要的时间就越长。需要分配后销毁的对象越多，垃圾回收器需要做的工作就越多。需要分配后持有的对象越多，GC 周期就越长。需要从磁盘加载到 JVM 的类越多，程序开始运行需要的时间就越长。执行过的代码越多，放入机器的硬件缓存的可能性就越小。需要执行的代码越多，执行的时间就越长。

我把这概括为“积少成多”原理。开发人员会说，他们只是加了一个非常小的功能，根本用不了多长时间（尤其是不使用该功能的时候），同一个项目中的其他开发人员也这么说。结果是，性能突然下降了几个百分点。下个版本中再次发生了这种事，现在程序的性能已经降低了 $10 \%$ 。这种事发生了几次之后，性能测试会触达某个资源的阈值——内存使用到达临界点、代码缓存溢出或其他类似的情况。在这些情况下，定期的性能测试可以找出性能下降的原因，性能优化团队也可以修复导致性能大幅下降的问题。不然随着时间的推移，小的性能损耗不断积累，修复就会变得越来越难。

# 我们最终会输掉这场战争

一个反常（令人沮丧）的地方是，每个应用程序的性能都会随着时间下降，准确地说是随着应用程序新版本的发布而下降。这种性能差异常常被忽略，因为硬件的改进可以保证新应用程序的运行速度。

想想看，如果在曾经运行 Windows 95 的计算机上运行 Windows 10 会发生什么。我之前最喜欢的一台计算机是 Mac Quadra 950，但是它无法运行 macOS Sierra（如果运行起来，相对于 Mac OS 7.5 它会非常非常慢）。从更低的层面看，Firefox 69.0 似乎比前几个版本运行得快，但这些只是小版本。随着标签式浏览、同步滚动和安全特性的增加，Firefox 要比 Mosaic 强大得多，但 Mosaic 加载我本地硬盘上的 HTML 文件的速度要比 Firefox 69.0 快 $50 \%$ 。

当然，Mosaic 已经无法加载几乎所有主流网站的真实 URL 了，我们不能再将 Mosaic作为主要的浏览器了。这也印证了普遍认同的一点，特别是在两个小版本之间，代码可以在优化后运行得更快。这是性能优化工程师应该关注的地方。我们擅长这个工作的话，就能赢得战斗。

这是有用且有价值的事。我的观点并不是不应该优化现有应用程序的性能。讽刺的是，为了赶超竞品，添加新特性和使用新标准只会导致应用程序越来越大、越来越慢。

我不是倡导永远不应在产品中添加新特性或新代码，增强程序显然会带来好处。但是注意做好权衡，尽可能提高效率。

# 1.3.3 过早优化

高德纳被认为是最早创造过早优化这个词的人。开发人员经常使用这个词来宣称，代码的性能重要不重要，得运行了才知道。也许你从不知道，完整的原话是这么说的：“大部分时间里，比如在 $9 7 \%$ 的时间里，我们不应该对细枝末节进行优化。过早优化是万恶之源。”3

这句话的意义在于，归根结底，你应该写出清晰明了、简单易懂的代码。这里的优化指的是，通过改变代码的算法和设计让结构复杂的程序也能有更佳的性能。这种类型的优化不应该先做，在分析了程序，发现这样做能带来很大益处后再去做。

这里所说的优化并不表示不能使用已知的性能糟糕的代码结构。如果对于每行代码都可以在两种简单、直接的代码中做出选择，那么请选择性能更好的那个。

在某种程度上，有经验的 Java 开发人员都能理解这一点（这是他们在日积月累中学到的一种优化艺术）。思考以下代码：

```javascript
log.log(Level.FINE, "I am here, and the value of X is " + calcX() + " and Y is " + calcY()); 
```

这段代码使用的字符串连接看起来没必要。除非日志级别很高，否则这条消息并不会记录到日志里。如果日志消息没有打印，就没必要调用 calcX() 方法和 calcY() 方法。有经验的 Java 开发人员会本能地抗拒这样写。有些 IDE 会标记这些代码并建议对其修改（然而工具是不完美的，NetBeans IDE 只会标记字符串连接，不会建议去掉不必要的方法调用）。

改成下面的日志代码会更好：

```javascript
if (log.isLoggable(LevelFINE)) {  
    log.log(Level.FINE, "I am here, and the value of X is {} and Y is {}, new Object[{calcX(), calcY()]);  
} 
```

这完全避免了字符串连接（消息格式不一定更高效，但更简洁），在不使用日志功能时也避免了方法调用和数组对象的分配。

用这种方式写的代码仍然简洁易读，但比之前的写法费不了多少力气。好吧，我们还是要敲几下键盘并多写一行逻辑的。但这并不是那种应当避免的过早优化，而是好的程序员都应该学会的技能。

不要让前辈们那些脱离了上下文的信条阻碍你对自己所写代码的思考。你会在本书中看到相关的其他例子，比如第 9 章讨论了一个看起来性能良好的循环结构，该循环用来处理Vector 中的元素。

# 1.3.4 其他：数据库永远是瓶颈

如果你在开发不依赖外部资源的独立 Java 应用程序，那么该应用程序本身的性能几乎是最重要的。一旦外部资源（例如数据库）添加进来，那两者的性能就都很重要了。在分布式环境中，如果有 Java REST 服务器、负载均衡器、数据库和后端企业信息系统，那么Java服务器的性能可能是最不重要的部分。

本书并不关注系统的整体性能。在这样的环境中，必须对系统的各个方面采取有组织的优化方法。系统的 CPU 使用率、I/O 延迟和各个部分的吞吐量都必须经过测量和分析。只有这样做，我们才能判断哪部分造成了系统瓶颈。有关该主题的资源非常丰富，相关的方法和工具也并不只适用于 Java。本书假设你已经完成了上述分析，并确定了你的运行环境中需要改进的是Java部分。

# bug 和性能问题不只局限于 JVM

本节只是以数据库性能为例进行说明，运行环境中的任何部分都可能是性能问题的根源。

我曾经有个客户在安装新版本的应用服务器时，遇到测试显示发送到服务器的请求越来越耗时，我根据奥卡姆剃刀原理考虑了可能导致这个问题的方方面面。

排除了一些因素后，性能问题依然存在。后端数据库没问题，所以最有可能出问题的是测试工具。经过测试后，我们发现负载生成器Apache JMeter是问题的源头。它将所有的响应都保留在一个列表中，当得到新的响应时，它会遍历整个列表去计算第90百分位响应时间（如果你不熟悉这个名词，可以查看第2章）。

整个系统的每一部分都可能造成性能问题。常见的案例分析表明，应该首先检测系统新增加的部分（经常在 JVM 应用程序里），但仍要准备好检查运行环境中的每一部分。

另外，不要忽视最初的分析。如果数据库是瓶颈（提示：它的确是），对访问数据库的Java 应用程序进行优化不会提升整体性能。实际上，这有可能适得其反。一个通用的准则是，给已经过载的系统增加负载，系统的性能会变差。如果在 Java 应用程序中做了一些更改让它更高效，这也会给过载的数据库增加负载，所以实际上整体性能是下降的。若你就此得出“不应使用某项JVM改进”的结论，那就危险了。

给系统中低效的部分增加负载会让整个系统变慢，这并不局限于数据库。例如，将负载添加到 CPU 密集型服务器上，或者让更多线程去获取已经有线程等待的锁，又或者在其他类似场景中，这个原理都适用。第9章将展示一个仅用了JVM的极端示例。

# 1.3.5 常见优化

我们很容易将性能的所有方面同等对待，特别是知道“积少成多”的典型表现之后。但我们应该关注常见的用例场景。这一原则体现在以下几个方面。

• 通过分析来优化代码，并专注于优化其中最耗时的操作。注意，这不代表只需考虑整体里的一部分（见第3章）。  
• 使用奥卡姆剃刀原理来诊断性能问题。对性能问题最简单的解释就是最可能的原因：新增代码的 bug 比机器配置更有可能带来性能问题，而后者比 JVM 和操作系统的 bug 更有可能带来性能问题。操作系统或 JVM 的确存在潜在的 bug，随着更多确定的因素被排除之后，的确有可能是测试用例在某种情况下触发了潜在的bug。但不要一开始就考虑这种可能性很小的情况。  
• 为应用程序的常见操作提供简单的算法。假设一个应用程序在求解一个数学公式，用户可以选择是在 $10 \%$ 的误差范围内还是 $1 \%$ 的误差范围内得到答案。如果多数用户对$1 0 \%$ 的误差感到满意，那就优化代码路径— 即使这意味着提供具有 $1 \%$ 误差的代码的速度会变慢。

# 1.4 小结

Java有很多工具和特性可以帮助 Java应用程序获得最佳性能。本书会帮你弄明白如何使用JVM所有的这些特性来让应用程序快速地运行。

不过在很多情况下，JVM 只是整体性能的一小部分。在 Java 环境中，数据库和其他后端系统的性能同 JVM 一样重要，因此需要对性能进行系统的分析。这并不是本书的重点，本书假设你已经做了详尽的分析，并且确定了运行环境中的Java组件是系统的重要瓶颈。

然而，JVM 和系统其他部分的交互对性能来说也很重要，不管是直接的交互（比如用最佳方式调用数据库），还是间接的交互（比如优化与大型系统中多个组件共享机器的应用程序的本地内存使用）。本书的内容也能帮你解决这方面的性能问题。

# 性能测试方法

本章讨论了获得性能测试结果的 4 项原则：测试真实的应用程序；理解吞吐量、批处理时间和响应时间；理解可变性；早测试、常测试。这些原则构成了后面几章的基础，也涵盖了性能工程的科学知识。能在应用程序上执行性能测试固然不错，但如果这些测试背后没有科学的分析过程，那么往往会导致分析结果不正确或者不完整。本章介绍如何确保测试得出有效的分析结果。

后面几章的许多例子使用了同一个模拟股票价格系统的应用程序，本章也使用了该应用程序。

# 2.1 测试真实的应用程序

第一个原则，即测试真实的应用程序，是指应该以实际产品的使用方式进行测试。大致而言，进行性能测试的代码可以分为 3 种：微基准测试（microbenchmark）、宏基准测试（macrobenchmark）和介基准测试（mesobenchmark）。每一种都各有优缺点，只有适合真实应用程序的代码才能提供最好的测试结果。

# 2.1.1 微基准测试

微基准测试是指通过测量一小部分代码的性能来确定多种实现中哪个最好。一些例子包括比较创建线程和使用线程池的开销，比较某个算术算法和其替代实现的执行时间，等等。

微基准测试似乎是测试性能的好办法，但是诸如即时编译和垃圾回收这些对开发人员很有吸引力的Java特性，给正确编写微基准测试增加了难度。

# 1. 必须读取测试的结果

微基准测试程序和常规程序有很多不同的地方。首先，Java 代码在执行的前几次都是解释

执行的，执行的时间越长，运行速度越快。出于这个原因，所有的基准测试（不仅是微基准测试）通常都包括一个预热期，在这期间，JVM可以将代码编译到最佳状态。

最佳状态包含了很多优化。举例来说，下面是一个看似简单的循环，用于计算第 50 个斐波那契数。

```java
public void doTest() { //主循环 double l; for (int i = 0; i < nWarmups; i++) { l = fibImpl1(50); } long then = System.currentTimeMillis(); for (int i = 0; i < nLoops; i++) { l = fibImpl1(50); } long now = System.currentTimeMillis(); System.out.println("Elapsed time: " + (now - then)); } 
```

这段代码旨在得出 fibImpl1() 方法的执行时间。它先预热了编译器，然后执行了编译后的方法。但这个执行时间可能是 0（或者更有可能的是，没有循环体的 for 循环的执行时间是 0）。因为 l 的值未被读取，所以编译器完全可以跳过它不进行计算。这取决于fibImpl1() 方法的内部结构，如果其内部仅仅是算术运算，那全都可以跳过。还有一种可能：方法只有一部分被执行，甚至产生了错误的 l 值。因为 l 值从未被读取，所以没人会注意到错误。（第4章将详细讲解如何消除循环。）

解决该问题的方法是：确保每个结果都被读取，而不只是写入。在实践中，将l 的定义从局部变量改为实例变量（用 volatile 关键字进行声明）即可测量这个方法的性能。（第 9章将解释l 实例变量必须用volatile 声明的原因。）

# 多线程微基准测试

在上述例子中，即使微基准测试是单线程的，也需要使用 volatile 变量。

在编写多线程微基准测试时要相当警惕。当多个线程同时执行一小段代码时，发生同步瓶颈（和其他线程问题）的可能性相当大。多线程微基准测试的结果往往会导致需要花费大量时间在优化同步瓶颈上，而不是在解决更紧迫的性能需求上，但同步瓶颈很少出现在实际代码中。

思考这样一种情况：在微基准测试中，有两个线程调用一个对象的同步方法。因为基准测试的代码量少，所以大部分时间在执行同步方法。即使同步方法的执行时间只占整个微基准测试的 $50 \%$ ，即使少到只有两个线程，同时执行同步方法的概率也会很大，所以基准测试会执行得很慢。而且随着线程的增加，资源竞争导致的性能问题会越来越糟糕。最终，测试变成了测量 JVM 处理多线程资源竞争的方式，这并不是微基准测试的初衷。

# 2. 必须测试一系列的输入值

即便读取了微基准测试结果，隐患依然存在。上述代码只有一个目的：计算第 50 个斐波那契数。足够智能的编译器可以分析出这一点，只执行一次循环体就能达成目的，或至少能减少迭代次数，毕竟这些操作是多余的。

此外，fibImpl1(1000) 和 fibImpl1(1) 的性能很可能相差很大。如果测试的目标是比较不同实现的性能，那就必须测试一系列的输入值。

这些值可以是随机的，举例如下：

```javascript
for (int i = 0; i < nLoops; i++) { l = fibImpl1(random.nextInt()); } 
```

这可能不是我们想要的结果，因为循环的执行时间包含了计算随机数的时间。现在测试检测的时间是计算 nLoops次斐波那契数列的时间加上生成nLoops 个随机数的时间。

所以，最好提前算好输入值。

```txt
int[] input = new int[nLoops];  
for (int i = 0; i < nLoops; i++) {  
    input[i] = randomnextInt();  
}  
long then = System.currentTimeMillis();  
for (int i = 0; i < nLoops; i++) {  
    try {  
        l = fibImpl1(input[i]);  
    } catch (IllegalArgumentExceptioniae) {  
        }  
}  
long now = System.currentTimeMillis(); 
```

# 3. 必须测量正确的输入值

你也许已经注意到，现在测试不得不在调用fibImpl1() 方法时捕获异常：输入值的范围可能包括负数（负数中没有斐波那契数），或者输入值大于1476（其结果超出了 double 的表示范围）。

当代码用于生产环境时，这些是可能出现的输入值吗？在这个例子里也许不是，但在你自己的基准测试中，情况可能就不一样了。考虑这样的情况：假设你测试的代码有两种实现，第一种实现能够快速计算出斐波那契数，但不会检查输入值的范围；第二种实现在输入值超出范围时立刻抛出异常，然后执行一个缓慢的递归操作来计算斐波那契数，如下所示。

public double fibImplSlow(int n) { if $(n <   0)$ throw new IllegalArgumentException("Must be $>0$ "）; if $(n > 1476)$ throw new ArithmeticException("Must be $<  1476$ "）; return recursiveFib(n);   
}

在输入值的范围很大的情况下，和原来的实现相比，上述这种新的实现更快，这仅仅是因为方法的开头进行了范围检查。

在现实中，如果用户总是给方法传入小于 100 的值，那么比较上述两种实现就会得出错误的结论。一般情况下，fibImpl1() 方法会更快。就像第 1 章解释的那样，我们应该针对常见的情况进行优化。（这显然是一个虚构的例子，在原来的实现上添加边界检查就能优化，生产环境中一般不太可能有这种情况。）

# 4. 代码在生产环境中可能有不同表现

到目前为止，我们关注的问题可以通过认真编写微基准测试来解决。代码被纳入到一个更大的系统中后，其他因素也会影响代码运行的最终结果。编译器利用代码的性能分析反馈（profile feedback）决定编译方法时的最佳优化方式。性能分析反馈基于方法被频繁调用、调用栈的深度、参数的实际类型（包括子类）等因素，而这些都是由代码的实际运行环境决定的。

因此，对于同样的代码，编译器在微基准测试中的优化方式常常与大型系统中的优化方式不同。

在垃圾回收方面，微基准测试也可能表现出截然不同的行为。考虑微基准测试的两种实现：第一种实现可以快速得出结果，但是会产生大量短期对象（short-lived object）；第二种实现慢一点，产生的短期对象也少一点。

当运行一个小程序来测试时，第一种实现可能更快。尽管它会触发更多的垃圾回收操作，但新生代垃圾回收器在回收时会快速丢弃短期对象，更短的整体时间有利于这种实现。当在服务器上使用多线程并发执行这段代码时，GC 性能分析看起来会不同：多个线程会很快填满新生代，许多在微基准测试中被快速丢弃的短期对象在多线程服务器环境下使用时，会被提升到老年代。这会导致频繁的（而且代价巨大的）Full GC。在这种情况下，Full GC花费的大量时间，让第一种实现不如第二种快，“更慢”的实现产生更少的垃圾。

最后，要确定微基准测试的实际含义。像我们现在讨论的基准测试，很多循环的整体时间差异是以秒为单位的，而单次迭代的时间差异是以纳秒为单位的。纳秒慢慢累加，“积少成多”就会频繁造成性能问题。但是要明确，特别是在回归测试中，纳秒级别的程序跟踪是否有意义。若一个集合被访问上百万次，那么每次访问节省的几纳秒就很重要（见第 12章的例子）。对于不那么频繁的操作（比如每次REST 调用时运行一次），修复微基准测试发现的纳秒级别的回归问题就会浪费时间，将这些时间用于优化其他操作必然更有益。

尽管有很多缺陷，但微基准测试还是很受欢迎，以至于 OpenJDK 有一个专门用来开发微基准测试的核心框架：Java 微基准测试工具（jmh，Java Microbenchmark Harness）。jmh 被JDK 开发人员用来构建针对 JDK 本身的回归测试，同时为开发人员提供了通用的基准测试框架。2.5.1节将讨论 jmh的更多细节。

# 什么是预热期？

Java 的一个性能特点是，代码执行得越多，性能就越好（这个话题会在第 4 章讲到）。因此，微基准测试必须包含一个预热期，让编译器有机会生成最佳代码。

本章稍后会深入讨论预热期的优点。微基准测试必须有预热期，否则它测量的就是编译性能，而不是代码性能。

# 2.1.2 宏基准测试

要测量一个应用程序的性能，最好的测量对象就是应用程序本身，外加它使用的任何外部资源。这就是宏基准测试的原则。如果应用程序通过调用目录服务（比如通过 LDAP1）来检查用户凭证，那就应该在这种模式下测试应用程序。去掉 LDAP 调用对于模块级别的测试有效果，但是应用程序必须在配置完整的情况下进行测试。

一个原因是，随着应用程序规模的增长，上述准则变得越来越重要，也越来越难实现。复杂系统并不是各个组成部分加在一起那么简单，当这些部分组合在一起时，它们的行为会截然不同。例如，模拟数据库调用可能意味着你再也不必担心数据库性能了——嘿，你是写 Java 的，为什么非要处理本该由 DBA 操心的性能问题呢？数据库连接会消耗大量的堆空间用于缓存；网络会因传送了太多数据而饱和；代码调用简单的方法（简单是相对于JDBC 驱动中复杂的代码而言的）时优化方式不同；CPU 传输和缓存短代码路径比长代码路径更高效……

需要测试完整应用程序的另一个原因是资源分配。在理想的情况下，有足够的时间优化应用程序中的每一行代码。在现实世界里，交付日期迫在眉睫，仅仅优化复杂运行环境的一部分可能不会有立竿见影的效果。

思考图 2-1 所示的数据流。用户登录后发起数据请求，然后系统进行数据业务处理，数据库基于此加载数据，之后经过专有计算，修改后的数据存入数据库，最后将结果返回给用户。每个方框中的数字表示该模块在单独测试时每秒能处理的请求数（RPS）。

但是，数据加载模块的处理速率仍然只有 100 RPS。即便有 200 RPS 输入数据库，也只有100 RPS 会输出到其他模块。虽然数据业务处理的效率翻倍了，但是系统的总吞吐量仍只有 100 RPS。在花时间改进运行环境的其他方面之前，进一步改进数据业务处理的算法只会是徒劳。

在这个例子中，优化数据业务处理所花费的时间并没有被完全浪费。一旦优化了其他的系统瓶颈，性能收益就会显现。准确地说，这是优先级的问题。在没测试整个应用程序的时候，不可能知道优化哪部分的性能会有效果。

# 多个 JVM上的系统整体测试

在测试完整的应用程序时，一个特别重要的情况是，多个应用程序同时运行在同一硬件上。JVM 的很多方面进行了调优，并假设所有硬件资源都是可用的。如果对这些JVM 进行单独测试，那么它们会表现得很好。如果在其他应用程序（包括但不限于其他JVM）正在运行时测试JVM，那么其表现则会大不相同。

后文将给出这样的例子。这里简单提一句：当执行一个 GC 周期时，单个 JVM（默认配置下）会使所有处理器的 CPU 使用率达到 $100 \%$ 。如果以程序执行时的平均水平来测量 CPU，那么 CPU 使用率可能会达到 $40 \%$ ——实际上，CPU 使用率有时是 $30 \%$ ，有时是 $100 \%$ 。这在 JVM 独立运行时还好，但是如果 JVM 和其他的应用程序一同运行，那么在 GC 时不可能 $100 \%$ 利用机器的 CPU。这时的性能会和单独运行时有明显差距。

这也是微基准测试和模块级别的基准测试不一定能让你了解应用程序性能全貌的另一个原因。

# 2.1.3 介基准测试

介基准测试介于微基准测试和宏基准测试之间。在与开发人员一起研究 Java SE 和大型Java 程序的性能时，我发现两者都有一套被称为“微基准测试”的测试。对于 Java SE 工程师，“微基准测试”意味着比 2.1.1 节中的例子更小的测试，用来测量很小的部分。Java程序开发人员则往往会把这个词用在其他基准测试上，即测量某个性能方面的基准测试，但是仍然要执行大量代码。

应用程序微基准测试的一个例子是，测量服务器返回简单REST 请求的响应有多快。与传统的微基准测试相比，测试这种请求的代码是相当多的：管理套接字的代码、读取请求的代码、编写响应的代码等。从传统的角度来看，这不算是微基准测试。

上述测试也不算是宏基准测试。它没有安全性（比如用户不用登录系统）、没有会话管理、没有大量使用其他应用程序特性。因为这种测试只是实际应用程序的子集，所以它属于中间地带——介基准测试。我用这个术语来指代做了一些实际工作但功能不全的基准测试。

介基准测试比微基准测试的缺陷更少，也比宏基准测试更容易操作。介基准测试不会有大

量可以被编译器优化的无效代码（除非无效代码存在于应用程序中，这时候优化是好事）。介基准测试更容易使用多线程。它们比完整的应用程序更有可能遇到同步瓶颈，不过，当实际的应用程序运行在大型硬件系统上且有大量负载时，这些瓶颈是不可避免的。

尽管如此，介基准测试仍不完美。如果使用这种基准测试来比较两台应用程序服务器的性能，那么开发人员很容易误入歧途。考虑两台 REST 服务器假定的响应时间，如表 2-1所示。

用介基准测试进行自动化测试也是很好的方法，特别是在模块级别的测试中。

# 2.2 理解吞吐量、批处理时间和响应时间

第二个原则，即理解吞吐量、批处理和响应时间，是指了解并选择合理的应用程序测试指标。性能可以通过吞吐量（RPS）、运行时间（批处理时间）或响应时间来测量。这3个指标是相互关联的。理解它们的关系，你就可以根据应用程序的目的，选择正确的测量指标。

# 2.2.1 测量批处理时间

要测量性能，最简单的方式就是测量一个任务的运行时间。例如，检索25 年内 1 万只股票的历史价格并计算这些股票价格的标准差，或者制作企业5 万名员工的工资福利报告，

抑或执行一个循环 100 万次。

对于静态编译语言，这种测试很简单：编写应用程序，并测量其执行时间。不过，Java 世界里的即时编译会带来额外的问题（这个过程在第 4 章详述）。实际上，这个过程需要几秒到几分钟（甚至更长）的时间将代码充分优化并以最佳性能运行。基于这个（和其他）原因，Java 性能研究很注重预热期，其性能的测量通常在相关代码运行足够长的时间、被编译和优化之后。

# 其他影响预热应用程序的因素

预热应用程序通常是指等待编译器对相关代码进行优化。根据代码运行时长，其他因素也会影响性能。

举例来说，Java（或 Jakarta）持久化 API（JPA）通常会缓存从数据库中读取的数据（见第 11 章），第 2 次使用该数据时会运行得更快，因为可以从缓存中获取数据，而无须再去访问数据库。同样，当应用程序读取一个文件时，操作系统常常会将文件存入内存。随后读取该文件时（例如在测量性能的循环中）速度会更快，因为数据已经在计算机的主内存里，不需要再从磁盘读取了。

总之，在很多情况下会缓存数据，并且预热期很重要，只不过并非所有这些情况都那么明显。

在很多情况下，应用程序从开始到结束的整体性能才是最重要的。一个处理一万个数据元素的报告生成器会在一定时间内完成执行。对于最终用户而言，前 5000 个元素的处理速度比后 5000 个慢 $50 \%$ 并不重要。以 REST 服务器为例，即使它的性能会随着时间的推移而提高，最初的性能仍然很重要。服务器达到性能峰值需要一定时间，对在这段时间内访问应用程序的用户来说，预热期的性能就很重要。

由于以上原因，本书中的很多例子是面向批处理的（虽然这不常见）。

# 2.2.2 测量吞吐量

对吞吐量的测量基于一定时间内可以完成的工作量。尽管最常见的吞吐量测量涉及服务器处理客户端发来的数据，但严格意义上这不是必要条件。一个独立的应用程序可以轻易地测量吞吐量，就像测量运行时间那样。

在客户端 / 服务器测试中，吞吐量测量并不考虑客户端的“思考时间”。假如有一个客户端向服务器发送了一个请求，当客户端收到响应后，立刻再发出一个新的请求。不断重复这个过程，在测试结束时，客户端报告它完成的操作总量。客户端常常有多个线程做这件事，吞吐量就是所有客户端完成的操作数量之和。通常这个数字是以每秒的操作数来报告的，而不是测量期间的总操作数。这种度量常常以每秒事务数（TPS）、每秒请求数（RPS）或者每秒操作数（OPS）为单位。

在客户端 / 服务器测试中，客户端的配置很重要，因为需要保证客户端向服务器发送数据的速度足够快。速度不够快的原因可能是客户端的机器没有足够的 CPU 周期来运行预期

数量的线程，或者客户端在发送新的请求之前需要大量时间来处理请求。在这些情况下，测试实际上是在测量客户端的性能，而不是服务器的性能，这通常不是最初的目标。

这种风险取决于每个客户端线程执行的工作量，以及客户端的线程数和配置。零思考时间（面向吞吐量）测试更有可能遇到这种情况，因为每个客户端线程需要执行更多的工作。因此，和测量响应时间的测试相比，吞吐量测试通常采用更少的客户端线程（更少负载）来执行。

吞吐量测试常常也会报告请求的平均响应时间。这是一条有趣的信息，只有在所报告的吞吐量相同时，平均响应时间的变化才表示性能有问题。举例来说，以0.5 秒的响应时间承载500 OPS的服务器，比以0.3秒的响应时间承载400 OPS的服务器性能好。

吞吐量测试几乎都是在适当的预热期之后进行的，因为所需测量的吞吐量并非一成不变。

# 2.2.3 测量响应时间

最后一种常见的测试指标是响应时间，即从客户端发送请求到收到响应之间的时间。

响应时间测试和吞吐量测试（假设吞吐量测试基于客户端/ 服务器模式）的区别在于，在响应时间测试中，客户端线程会在操作之间有一段休眠时间，这被称为思考时间。响应时间测试旨在更确切地模拟用户行为：用户在浏览器中输入 URL，花时间阅读返回的页面，再点击页面中的链接，花时间阅读新的页面，等等。

当将思考时间引入测试中时，吞吐量就变成固定的了：给定客户端的数量和思考时间，执行请求往往会产生相差无几的 TPS（略有差异，见框注）。在这时，请求的响应时间就变得重要了，因为服务器的效率取决于其对固定负载的响应速度。

# 思考时间和吞吐量

对于引入思考时间的客户端，吞吐量有两种测量方式。较为简单的方式是让客户端在请求之间休眠一段时间：

```txt
while (!done) { executeOperation(); Thread.currentThread().sleep(30*1000); } 
```

在这种情况下，吞吐量在一定程度上取决于响应时间。如果响应时间为 1 秒，那么客户端每 31 秒发送一次请求，产生的吞吐量为 0.032 OPS。如果响应时间是 2 秒，那么客户端每32秒发送一次请求，产生的吞吐量为 0.031 OPS。

另一种方式被称为周期时间（并不是思考时间）。周期时间会将请求之间的总时间设定为 30秒，这样一来，客户端的休眠时间就完全取决于响应时间了：

```javascript
while (!done) { time = executeOperation(); Thread.currentThread().sleep(30*1000 - time); } 
```

采用这种方式，每个客户端都会产生固定的吞吐量（0.033 OPS），而不会考虑响应时间（在这个例子里，假设响应时间总是小于30秒）。

测试工具中的思考时间常常会有所变化。它们平均下来是某个特定的值，但是会随机变化以更好地模拟用户行为。此外，线程调度从来都不是完全实时的，所以客户端请求之间的实际时间会略有不同。即使测量工具提供了周期时间而不是思考时间，报告的吞吐量也会略有差异。但是如果吞吐量与预期值相差甚远，就说明测试执行过程出现了问题。

我们可以用两种方式来测量响应时间。可以报告平均响应时间：将单个时间加在一起，再除以请求数。此外，也可以报告百分位响应时间。举例来说，如果 $90 \%$ 的响应小于 1.5秒， $1 0 \%$ 的响应大于1.5秒，那么1.5秒就是第90百分位响应时间。

平均响应时间和百分位响应时间的区别是，异常值会影响平均值计算。这是因为，在计算平均值时会用到异常值，异常值越大，它对平均响应时间的影响就越大。

# 2.3 理解可变性

第三个原则，即理解可变性，是指理解测试结果如何随时间变化。即使处理完全相同的数据集，应用程序每次运行也都会产生不同的结果。这是因为，后台进程会影响应用程序的运行，而且网络在运行时或多或少会出现拥堵。好的基准测试不会每次都使用完全相同的数据集，而是会在测试过程中构建随机行为以模拟真实世界。这就产生了一个问题：当比较两次运行的结果时，差异是由性能造成的，还是由随机变化的测试造成的？

这个问题可以通过多次运行测试后取平均值来解决。如果正在测试的代码有修改，那么可以再多运行几次测试，再次对结果取平均值，然后比较这两个平均值。这听起来很简单。

不幸的是，事情并没有那么简单，因为很难判断什么时候是真正的性能倒退，什么时候是随机变化。在性能优化这一重要领域，科学引领了方向，艺术也在发挥作用。

当比较基准测试结果的平均值时，我们不能绝对地肯定这些平均值的差异是由于真正的性能问题还是随机变化而导致的。最好的办法是假设“平均值是相同的”，然后判断这句话正确的概率。如果这句话错误的概率很高，我们就可以放心地认为平均值的差异是由于真正的性能问题（哪怕我们永远不能 $100 \%$ 确定）。

像这样因代码改进而进行的测试叫作回归测试（regression testing）。在回归测试中，原本的代码叫作基线（baseline），新的代码叫作样本（specimen）。以一个批处理程序为例，基线和样本各运行3次，表2-2给出了这个测试所用的时间：

表2-2：两个测试的假设执行时间  

<table><tr><td></td><td>基线</td><td>样本</td></tr><tr><td>第1次迭代</td><td>1.0秒</td><td>0.5秒</td></tr><tr><td>第2次迭代</td><td>0.8秒</td><td>1.25秒</td></tr><tr><td>第3次迭代</td><td>1.2秒</td><td>0.5秒</td></tr><tr><td>平均值</td><td>1秒</td><td>0.75秒</td></tr></table>

样本的平均值表明，代码改进后速度提升了 $2 5 \%$ 。我们有多大把握确信这个测试真实地反映了 $2 5 \%$ 的提升？结果看起来不错：在3个样本中有两个值比基线的平均值小，改进的幅度很大。但是将本节介绍的分析方式应用在这些结果上时，你会发现样本和基线性能相同的概率是 $43 \%$ 。观察到这样的数字，表明两个测试的性能在 $43 \%$ 的时间里是相同的，而在 $5 7 \%$ 的时间里性能不同。顺便一提，这和“ $5 7 \%$ 的时间里性能提升 $2 5 \% ^ { \ast }$ ”不完全一样，稍后详述原因。

上述概率似乎与预期不同，原因是测试结果的变化很大。一般来说，结果的变化越大，越难判断平均值的差异是由于真正的性能问题还是随机变化 3。

$43 \%$ 这个数字是基于T检验（T-test）的结果。T检验是基于数列和方差的统计分析方法4。它产生的数值叫作 $p$ 值，表示零假设为真的概率。

回归测试中的零假设（null hypothesis）就是假设两个测试有相同的性能。这个例子中的 $p$ 值大约是 $43 \%$ ，意思是我们对这两组测试的平均值相同有 $43 \%$ 的把握。反过来，我们对这两组测试的平均值不相同的把握是 $5 7 \%$ 。

这是什么意思呢？严格地说，这并不表示我们对性能提升 $2 5 \%$ 有 $5 7 \%$ 的把握，它只表示结果不同的情况占 $5 7 \%$ 。性能既可能提升 $2 5 \%$ ，也可能提升 $12 5 \%$ ，甚至可能是样本的性能比基线更差。最有可能的是，测试的差异和（被测量的）真实情况类似（特别是当 $p$ 值下降时），但我们永远无法确定这一点。

T 检验通常和 $\alpha$ 值一起用， $\alpha$ 值代表一个点（可任意选择），达到这个点的结果被认为具有统计显著性（statistical significance）。 $\alpha$ 值通常设为 0.1，这表示如果样本和基线只有 $1 0 \%$ （0.1）的时间结果相同（或者反过来， $90 \%$ 的时间不同），那么结果就具有统计显著性。其他常用的 $\alpha$ 值有 0.05（ $9 5 \%$ 的时间不同）和 0.01（ $9 9 \%$ 的时间不同）。如果测试的 $p$ 值大于 $1 - \alpha$ 值，那么就认为该测试的结果具有统计显著性。

# 统计数字及其语义表述

正确表述 T 检验结果的方法是这样的：样本和基线有差异的概率为 $5 7 \%$ ，差异的最佳估值是 $2 5 \%$ 。

一般这样表述结果：结果提升 $2 5 \%$ 的置信度是 $5 7 \%$ 。虽然这与前一种说法并不完全相同，也会让统计学家抓狂，但这种说法简洁且易于接受，也不是很离谱。概率分析总会涉及不确定性，准确的表述会让这种不确定性更容易理解。然而，特别是在一个根本问题已被透彻理解的领域，必然会出现语义上的简化。

因此，正确地查找代码性能倒退的方式是，确定一个统计显著性水平——比如 $90 \%$ （ $\alpha$ 值为 0.1），然后用 T 检验判断样本和基线的差异是否在这个统计显著性水平以内。必须清楚

的是，如果统计显著性检验失败意味着什么。在这个例子中， $p$ 值是 0.43，这表示平均值不同的置信度小于 $90 \%$ ，所以我们不能说结果具有统计显著性。测试结果不具有统计显著性，这并不代表这个结果无关紧要，而只表示测试尚无定论。

# 统计重要性

统计显著性并不同于统计重要性。测试变化很小的基线（如平均 1 秒）和变化很小的样本（如平均 1.01秒），其 $p$ 值可能是 0.01，即结果有差异的置信度是 $9 9 \%$ 。

差异本身只有 $1 \%$ 。现在假设另一个测试在基线和样本上有 $1 0 \%$ 的差异，但是 $p$ 值为 0.2，即不具有统计显著性。这两个测试之中，哪个更重要？这需要额外的时间去调查。

虽然 $10 \%$ 差异的测试置信度更低，但最好把时间花在这个测试上（如果可以的话，先用更多的数据验证其结果真的具有统计显著性）。这是因为， $1 \%$ 差异的可能性更大并不表示它更重要。

通常，一个检验没有统计结论的原因是验证的数据不够多。到目前为止，我们的例子都使用了 3 个基线结果和 3 个样本结果。如果再加上 3 个结果，会发生什么？表 2-3 列出了新加的数据。

通过运行额外的测试达到某个统计显著性水平，这样做并不总是可行的，严格来说也没有必要。决定统计显著性的 $\alpha$ 值是任意选择的，即使是大家普遍使用的 $\alpha$ 值，也只是常用而已。0.11 的 $p$ 值在 $90 \%$ 的置信度下不是统计显著的，但在 $89 \%$ 的置信度下就是统计显著的。

回归测试很重要，但它并不是非黑即白的二元选择。如果不做一些统计分析来理解这些数字的含义，就不能对这一系列数字（或它们的平均值）进行比较判断。但是，概率本身的含义是，即使做了这种分析，也不能得出一个完全确定的答案。性能优化工程师的工作就是查看数据、理解概率，并根据所有可用的数据决定应该先优化哪个问题。

# 快速小结

• 正确判断两个测试的结果是否有差异需要进行一定程度的统计分析，以确保感知到的差异不是随机波动造成的。  
• 要进行严谨的统计分析，可以使用 T检验比较测试结果。  
• T 检验的结果表示出现性能倒退的概率，但是它并不能显示出哪些倒退应该忽略，哪些应该追踪。在这两者间找到平衡是一种性能工程艺术。

# 2.4 早测试、常测试

第四个原则，也是最后一个原则，就是早测试、常测试。性能极客（包括我）建议将性能测试作为开发周期中的必要部分。在理想情况下，性能测试可以在代码检入中央仓库时，作为检入过程的一部分运行，导致性能倒退的代码会在检入时被阻。

这个原则和本章中的其他原则之间，以及和现实情况之间存在着一些内在矛盾。一个好的性能测试包含大量代码——至少一个中等规模的介基准测试就包含大量代码。它需要重复运行多次后，才能建立可信度，以确保新代码和旧代码之间的任何差异都是真实的差异，而非随机的差异。在一个大型项目中，这可能需要几天甚至一周的时间，这就使得在检入中央仓库之前运行性能测试不太现实。

典型的开发周期并没有让事情变得简单。项目计划通常会设置一个特性冻结日期：所有功能上的代码变更都必须在发布周期的早些时候检入中央仓库，而剩余时间用来检查新版本中的bug（包括性能问题）。这就给早期测试带来了两个问题。

受发布时间的限制，开发人员需要尽快检入代码。当初始代码都检入后，即使还有优化性能的时间，开发人员也不愿意花时间进行优化。这是因为在开发周期初期，如果检入的代码造成了 $1 \%$ 的性能倒退，那么负责检入的开发人员对于修复此问题会面临很大的压力。而如果直到特性冻结日当晚才检入代码，还造成了 $20 \%$ 的性能倒退，那么开发人员也可以在晚些时候再处理这个问题。  
代码的性能特征会随着代码的变化而变化。这和测试完整的应用程序（加之可能出现的任何模块级别的测试）的情况一样。比如，堆的使用情况会发生变化，代码的编译也会发生变化。

尽管有这些挑战，检测到的问题也不能立即解决，但在开发过程中频繁进行性能测试是很重要的。如果开发人员检入的代码造成了 $5 \%$ 的性能倒退，那么他可以在开发过程中解决这个问题。比如，他的代码依赖一个即将集成的特性，等这个特性可用时，稍微调整一下代码就不会再有问题。尽管这意味着性能测试不得不和这 $5 \%$ 的性能倒退并存好几周（不幸的是，这难免会掩盖其他的性能倒退问题），但这是合理的。

如果只需对架构做些改变即可消除因新代码而导致的性能倒退，那么最好是在其他代码基于新的实现之前，就捕获这个问题并解决它。这是一种平衡的艺术，需要仔细分析，常常还需要一些技巧。

遵循以下准则，早测试、常测试的原则会最有效。

自动化一切

所有的性能测试都应该脚本化（或者程序化，不过脚本化通常更容易）。必须通过脚本安装新代码并配置完整的运行环境（创建数据库连接、设置用户账户等），然后运行测试。不止于此，测试脚本还必须能够运行多次，以便进行 T 检验分析并生成报告，展示结果相同的置信度或在结果不同时展示差异。

在测试运行之前，必须通过自动化确保机器处于合适的状态。必须检查并确定没有额外的进程在运行，并且操作系统配置正确。只有当每次的运行环境都一样时，性能测试才是可重复的。自动化必须注意这一点。

测量一切

自动化必须收集可以想到的每一份数据，以便用于以后的分析。这包括运行过程中采集的系统信息，即 CPU 使用率、磁盘使用率、网络使用率和内存使用率等。这还包括应用程序的日志——应用程序生成的日志和垃圾回收器生成的日志。理想情况下，它还可以包括 Java 飞行记录器（JFR）的记录（见第 3 章）或者其他影响较小的性能分析信息、定期的线程栈，以及其他堆分析数据，比如直方图或全堆转储信息（不过全堆转储尤其需要大量空间，并且不能长期保存）。

如果可以，自动化监控的信息还必须包括系统其他部分的数据。如果程序使用了数据库，那么自动化就要收集数据库系统统计的数据，以及所有的数据库诊断输出（包括类似 Oracle 公司的 Automatic Workload Repository 的性能报告）。

这些数据可用于分析任何未发现的性能倒退。如果 CPU 使用率增加了，就需要查阅性能分析信息，看看是什么耗费了更多时间。如果花在GC 上的时间增加了，就需要查阅堆性能分析信息，看看是什么占用了更多内存。如果 CPU 使用率和 GC 时间都减少了，那么可能是某个地方的资源竞争降低了性能，比如，栈数据有助于找出特定同步瓶颈的原因（见第 9 章），JFR 记录有助于找出应用程序延迟的原因，数据库日志有助于找出导致数据库资源竞争加剧的原因。

在查找性能倒退的原因时，需要像侦探一样，可用的数据越多，能推断出的线索就越多。正如第 1 章所讨论的那样，性能倒退不一定是 JVM 导致的。测量一切，确保分析结果的正确性。

在目标系统上运行

在单核笔记本计算机上运行的测试和在 72 核计算机上运行的测试会有不同表现。就线程的运行效果来说，大型机器很明显会同时运行更多的线程，以减少应用程序线程访问CPU 时的资源竞争。同时，大型系统会出现同步瓶颈，这在小型笔记本计算机上基本会被忽略。

即使不那么明显，但其他性能差异也同样重要。很多重要的调优标志会基于 JVM 运行的底层硬件系统，计算出它们的默认值；不同平台的代码编译方式不同；软件缓存和更重要的硬件缓存，在不同的系统和不同的负载下的表现也不同。

因此，如果不在预期的负载下和预期的硬件上测试，就永远无法完全知道特定生产环境的性能。通过在较小的硬件上运行较小的测试，可以得出近似值以便做出推断。再加上，在现实世界中，复制用于测试的生产环境是相当困难或相当昂贵的。所以，推断只

是简单的预测，就算是在最好的情况下，预测也可能是错的。大规模系统不仅仅是各个部分的总和，并且没有什么能替代在目标平台上执行充分的负载测试。

• 频繁地进行性能测试很重要，但是不能脱离实际。对于正常的开发周期，需要权衡利弊。  
• 自动化测试系统会从所有机器和程序中尽可能多地收集数据，这将为查找性能倒退源头提供必要的线索。

# 2.5 基准测试示例

本书的一些例子使用 jmh 提供微基准测试。本节将深入介绍这样的微基准测试是如何开发出来的，以此作为编写 jmh 基准测试的例子。本书中的很多例子基于一个介基准测试的变体，这个介基准测试足够复杂，可以用于练习各种 JVM 特性，但是它没有实际的应用程序那么复杂。在了解 jmh 之后，我们将浏览在后面章节中会用到的介基准测试的代码示例，以便理解这些例子的上下文。

# 2.5.1 Java微基准测试工具

jmh 是一组类，这些类提供了一个编写基准测试的框架。jmh 中的 $m$ 曾经代表microbenchmark，即微基准测试，现在 jmh 宣称自己适用于纳（nano）/ 微（micro）/ 毫（milli）/ 宏（macro）等规模的基准测试。不过，它的典型用法仍然是微基准测试。虽然jmh 是随着 Java 9一起发布的，但它并没有与任何具体的 Java版本绑定，而且 JDK中没有支持jmh的工具。组成 jmh的类库兼容JDK 8及之后的版本。

jmh 能让你更方便地编写高质量的基准测试，但它不是“灵丹妙药”。你必须理解你所测试的代码，知道如何写出好的基准测试代码。jmh的功能只是让这一切变得简单。

本书在多个例子中用到了 jmh，包括第 12 章中 JVM 参数如何影响字符串保留（string interning）的测试。我们在这里用这个例子来解释如何用jmh编写基准测试。

可以从头开始写一个 jmh 基准测试，但是从 jmh 提供的主类开始，只编写基准测试相关的代码则更简单。可以通过各种工具（甚至是 IDE）获取必要的 jmh 类，但最基础的方法还是使用 Maven。下面的命令将创建一个 Maven 项目，我们可以在其中加入基准测试的相关代码：

```txt
$ mvn archetype:generate \
-DinteractiveMode=false \
-DarchetypeGroupId=org.openjdk.jmh \
-DarchetypeArtifactId=jmh-java-benchmark-archetype \
-DgroupId=net.sdo \
-DartifactId=string-intern-benchmark \
-Dversion=1.0 
```

这个命令在 string-intern-benchmark 目录下创建了 Maven 项目。在该目录下，又根据 groupId

创建了目录和一个名为 MyBenchmark 的基准测试框架类。这个类名没有什么特殊含义，你可以创建一个或多个其他的类名，因为 jmh 会通过名为 Benchmark 的注解来查找需要测试的类。

因为我们想测试 String.intern() 方法的性能，所以第一个基准测试方法应该这样写：

```java
import org.openjdk.jmh.annotations.Benchmark;   
import org.openjdk.jmh.infra.Blackhole;   
public class MyBenchmark { @Benchmark public void testIntern(Blackhole bh) { for (int i = 0; i < 10000; i++) { String s = new String("String to intern " + i); String t = s.intern(); bhconsume(t); } } 
```

上述关于testIntern() 方法的代码表示，我们要测量创建10 000个保留字符串的时间。这里用的 Blackhole 类是 jmh 的特性，它解决了微基准测试的一个重要问题：如果一个操作的值没有用到，那么编译器会直接优化这个操作。所以把值传递给 Blackhole 的 consume()方法可以确保值被使用。

在这个例子里，严格意义上并不需要 Blackhole。我们真正感兴趣的只是调用 intern() 方法后的情况，这个方法会将一个字符串插入到全局哈希表中。即使我们没有用到intern()方法本身的返回值，这个返回值也不会被编译掉。尽管如此，与其纠结是否有必要优化一个特定的值，不如养成习惯，确保代码会按照预期执行，并使用计算出来的值。

# 1. jmh 和参数

对于一个测试，你常常需要输入一系列的参数来进行验证。在这个例子中，假如想知道保留 1 个或者 10 000 个（甚至是 100 万个）字符串的效果，我们可以在 testIntern() 方法中引入一个参数，而不是硬编码：

```txt
@Param({"1","10 000"})   
private int nStrings;   
@Benchmark   
public void testIntern(Blackhole bh) { for (int i = 0; i < nStrings; i++) { String s = new String("String to intern " + i); String t = s.intern(); bhconsume(t); }   
} 
```

# 3. Setup 代码

之前的输出对格式进行了缩减，实际运行 jmh 测试时，在看到结果之前，你会先看到一段很长的警告。警告的要点是，“不要因为在 jmh 中写了几行代码，就认为你已经写出了很好的基准测试。测试你的代码，确保它测试的正是你所期望的”。

让我们再来看看上述基准测试。我们想测试保留 10 000个字符串需要多长时间，但是我们实际测试的是创建（通过字符串连接）10 000 个字符串的时间加上保留字符串的时间。这些字符串的范围在一定程度上是受限制的：开头是同样的 17 个字符，后面加上一个整数。对于之前写的那个斐波那契数的微基准测试，我们为其提前创建了输入。在这种情况下，我们也应该提前创建输入。

有人认为，字符串的范围对这个基准测试不重要，并且因为字符串连接的影响很小，所以最初的测试是完全准确的。这是有可能的，但证明这个观点需要一定的工作量。所以，最好在写基准测试时解决这些问题，而不是对将要发生的事情做假设。

我们还必须深入思考测试期间发生了什么。本质上说，持有保留字符串的表相当于一个缓存区：保留字符串既可能在那里（这种情况下测试，会返回值），也可能不在（这种情况下测试，会插入值）。在比较不同的实现时，我们会遇到一种情况：手动管理的ConcurrentHashMap 在测试中从来没有清空过。这意味着在第一个预热期，字符串被插入到了哈希映射中，在后续的测量期，字符串已经存在了，测试的缓存命中率是 $100 \%$ 。

保留字符串的表并不是这样工作的：此表中的键基本上是弱引用。因此，JVM 可能随时清除部分或者全部条目（因为插入到表中后，保留字符串就退出了其作用域）。这种情况下的缓存命中率是不确定的，但是不可能接近 $100 \%$ 。就目前情况来看，字符串保留测试会做更多的工作，因为它必须更频繁地更新内部字符串表（删除和重新添加条目）。

如果我们预先创建字符串并将其放到静态数组中，然后将其保留（或者插入到哈希映射中），那么上述两个问题都可以避免。这是因为，静态数组持有对字符串的引用，字符串表中的引用不会被清除。在测量期内，这两个测试都会有 $100 \%$ 的命中率，字符串的范围也会更加广泛。

# 设置代码与低级别测试

后面章节的一些 jmh 测试会报告特定的操作只相差几纳秒。这些测试并不是在测试单个操作后报告纳秒级时间，而是进行大量操作后报告统计方差内的平均值。jmh 会为我们管理这些。

请务必注意：如果每次调用都会执行 setup 代码，那么 jmh 很难执行平均调用分析。因为这个（和其他）原因，建议尽量少使用 Level.Invocation 注解，最好只在测试方法本身耗时较长的情况下使用。

如果需要的话，可以在其他情况下使用类似的 Teardown 注解来清除状态。

jmh 还有许多其他的功能，比如测量方法的单次调用、测量平均时间而不是吞吐量、将额外的 JVM 参数传递给派生的 JVM、控制线程同步等。我的目标不是详尽地介绍 jmh。相反，这个例子恰好说明，一个简单的微基准测试也有复杂的一面。

# 4. 控制执行和可重复性

一旦写了一个正确的微基准测试，你就需要以一种能让测量结果有统计意义的方式运行这个测试。

# 2.6 小结

性能测试需要权衡利弊。在相互制约的选项中做出正确的选择，这样做是成功优化系统性能的关键。

在构建性能测试的过程中，直觉和经验对选择哪方面进行测试将大有裨益。微基准测试主要能为一部分操作设定指导原则。这为其他测试留下了广阔的发展空间，无论是模块级别的小型测试，还是涉及多层环境的大型测试。这些测试都有各自的优点，而如何选择需要丰富的经验和敏锐的直觉。不管怎样，随着应用程序部署到生产环境中，测试完整的应用程序的过程是无可替代的。只有测试完整的应用程序才能了解所有与性能相关的问题及其影响。

同样，理解代码中哪些是性能倒退哪些不是，并非总是一个非黑即白的问题。程序总是展现出随机行为，一旦引入随机性，我们就永远无法完全确定数据意味着什么。对结果进行统计分析，可以让分析更客观，但即便如此，也还会存在一定的主观性。理解数据背后的概率及其含义，可以帮助减少这种主观性。

最后，有了本章介绍的关于性能测试的基础知识，就可以建立一个自动化测试系统，收集测试中生成的所有信息。了解了测试过程中所发生的事情及其背后的意义，性能分析工程师就可以将科学和艺术结合起来，让程序展现出最佳性能。

# Java性能工具箱

性能分析讨论的基础是所有过程和信息的可见性，即要了解应用程序内部以及承载应用程序运行的环境中正在发生什么。可见性的关键在于使用工具，所以性能优化是围绕工具展开的。

在第 2 章中，我们探讨了采用数据驱动的方法进行性能优化的重要性：你必须测量应用程序的性能并了解测量结果的意义。性能分析同样必须由数据驱动：为了优化应用程序，你必须掌握有关应用程序运行的数据。如何获取和理解这些数据是本章的主题。

数以百计的工具可以提供 Java 应用程序的运行信息，但是研究所有的工具是不切实际的。最重要的工具大都是 Java 开发工具包（JDK）自带的。尽管还有其他的开源工具和商用工具，但为方便起见，本章主要关注JDK 工具。

# 3.1 操作系统工具和分析

性能分析的起点和 Java 无关，而和操作系统自带的一组基本监控工具有关。在 Unix 系统中，有 sar（System Accounting Report）及其组成工具，例如 vmstat、iostat 和 prstat等。在Windows系统中，有图形化的资源监视器和类似typeperf 的命令行工具。

无论性能测试何时进行，都需要从操作系统收集数据。至少要收集CPU、内存和磁盘使用率的相关信息。如果应用程序使用了网络，也需要收集网络使用率信息。如果是自动化的性能测试，就需要使用命令行工具（甚至在 Windows 上也一样）。即使测试以交互式运行，最好也用命令行工具捕获输出，而不是盯着 GUI 图表猜测它的意思。在之后进行分析时，随时可以将捕获的输出以图表的方式展现出来。

# 3.1.1 CPU使用率

我们首先来看 CPU 监控，看看从中可以得到关于 Java 应用程序的哪些信息。CPU 使用率通常分为两类：用户时间和系统时间（Windows 上叫作特权时间）。用户时间指的是 CPU执行应用程序代码的时间所占的百分比，系统时间则是执行内核代码的时间所占的百分比。系统时间和应用程序相关。如果应用程序执行 I/O 操作，那么系统将执行内核代码，从磁盘读取文件，或者将缓冲的数据发送给网络。任何使用底层系统资源的操作都会让应用程序使用更多的系统时间。

性能优化的目标是，在尽可能短的时间内，让 CPU 使用率尽可能高。这听起来有点违反常理，因为你肯定有过这样的经历：CPU 使用率到了 $100 \%$ 之后，你只能坐在计算机屏幕前看着它挣扎。接下来让我们看看CPU使用率到底意味着什么。

首先需要牢记的是，CPU 使用率是一段时间内的平均值，也许是 5 秒、30 秒，甚至可能是 1 秒（不会比它还短了）。假设在执行一个应用程序的 10 分钟内，CPU 使用率平均是$50 \%$ ，这意味着 CPU 有一半时间是空闲的。如果优化这个应用程序以避免出现空闲时间（在没有其他瓶颈的情况下），那么这样做能让它的性能翻倍，即以 $100 \%$ 的 CPU 使用率在5分钟内结束运行。

如果我们优化这个应用程序的算法并让其性能再次翻倍，那么它会以 $100 \%$ 的 CPU 使用率在 2.5 分钟内结束运行。CPU 使用率反映了应用程序使用 CPU 的效率，所以这个数字越大，性能就越好。

• 应用程序在等待某些东西，比如数据库调用返回的响应。  
• 应用程序没有事情可做。

前两种情况往往表明问题可以解决。如果可以减少锁的竞争或者优化数据库以使响应返回得更快，应用程序就会运行得更快，平均 CPU 使用率也会上升（当然，此处假设没有其他类似问题会阻塞应用程序）。

第 3 种情况常常使人困惑。如果应用程序有事可做（而且没有等待锁或其他资源），那么CPU 必然会分配一些周期执行应用程序的代码。这是一个通用原则，并不局限于 Java。例

如你写了一个包含无限循环的简单脚本，脚本执行时会消耗 $100 \%$ 的 CPU 资源。下面的Windows批处理任务就是这么做的：

ECHO OFF

:BEGIN

ECHO LOOPING

GOTO BEGIN

REM We never get here...

ECHO DONE

试想一下，脚本没有消耗 $1 0 0 \%$ 的 CPU 资源意味着什么。这意味着操作系统还可以做其他的事——比如可以打印另一行 LOOPING——但是它选择空闲。空闲在这种情况下并没有什么帮助，如果正在进行有用却很耗时的计算，那么强制 CPU 周期性地空闲会让我们得到答案的时间变得更长。

如果你在单 CPU 机器上或者容器中运行这个脚本，那么多数时候你不会注意到它在运行。但是，如果尝试启动一个新的应用程序，或者测试另一个应用程序的性能，你就会注意到影响了。操作系统会为竞争 CPU 周期的应用程序分配时间片，但是新应用程序的可用CPU 周期较少，所以就会运行得很慢。有经验的人有时认为，留下一些空闲的 CPU 周期是件好事，以备不时之需。

但是操作系统不会猜测你下一步要做什么，它默认会执行任何可以执行的应用程序，而不是让 CPU 空闲。

# 限制程序使用CPU

尽可能地利用 CPU 周期运行程序，可以最大化程序的性能。不过，有时你可能并不想这么做。比如，运行 SETI@home 时，它会消耗机器上所有可用的 CPU 周期。在你不工作或者仅仅浏览网页和编写文档时还好，否则这会降低你的生产率。（我们还没考虑你在玩CPU密集型游戏时会发生什么！）

有许多与操作系统相关的机制可以人为地限制程序使用 CPU——实际上，就是强制CPU 留下空闲周期，以防有程序需要使用。也可以更改进程的优先级，这样一来，后台任务就不会和你想运行的程序争夺 CPU 周期，也不会留下空闲的 CPU 周期。这些技术不在我们现在的讨论范围内。（顺便说一句，SETI@home允许你对这些进行配置，它不会真的占用机器上所有的空闲周期，除非你告诉它这么做。）

# 1. Java 和单 CPU 的使用率

我们再回到 Java 应用程序的讨论，上述例子中的 CPU 周期性空闲意味着什么？这取决于应用程序的类型。如果应用程序属于批处理类型，那么其要完成的工作量是固定的。这种情况下，你不会看到 CPU 空闲，因为 CPU 空闲意味着无事可做。对批处理任务而言，榨干 CPU 的最后一点处理能力是其孜孜以求的目标。用得越多，任务完成的速度越快。即便 CPU 使用率已经达到 $100 \%$ ，你依然可以探索进一步优化的可能，从而让工作更快完成（同时尽量保持 $100 \%$ 的CPU使用率）。

如果应用程序是服务器处理类型，即应用程序需要从某处接收请求，那么 CPU 会因为没有工作可做而处于空闲状态。例如，Web 服务器已经处理了所有的 HTTP 请求，正在等待下一个请求。这时就要引入平均时间。前面 vmstat 样例的输出是从服务器获取的，服务器每秒接收一个请求。应用程序服务器处理请求的时间为 450 毫秒——在这 450 毫秒内，CPU1 $00 \%$ 忙碌，而剩下 550毫秒空闲。这会报告为 CPU是 $45 \%$ 忙碌的。

尽管 CPU 忙碌时间常常因为粒度太小而无法可视化，但运行有负载的应用程序时，CPU就是这样以短时突发的方式运行的。如果 CPU 每半秒接收一个请求且请求的平均处理时间是 225 毫秒，那么也可以在宏观层面看到同样的模式。CPU 会忙碌 225 毫秒，空闲 275毫秒，再次忙碌 225 毫秒，再次空闲 275 毫秒：平均来说， $45 \%$ 的时间忙碌， $5 5 \%$ 的时间空闲。

如果应用程序优化之后每个请求只需要 400 毫秒，那么总体的 CPU 使用率会降低至 $40 \%$ 。这是唯一降低CPU使用率有意义的情况——流入系统的负载量固定且应用程序不受限于外部资源。同时，这些优化可以让系统承担更多的负载，最终提高 CPU 使用率。在微观层面，这种优化仍然只是在短时间（执行请求的400 毫秒）内让 CPU 使用率达到 $100 \%$ 只是 CPU峰值的维持时间太短，不足以让大多数工具显示为 $100 \%$ 使用率。

# 2. Java 和多 CPU 的使用率

上述例子假设在单CPU机器上运行单个线程，一般情况下，这与多CPU多线程是相通的。多线程会以有趣的方式使平均 CPU 使用率偏斜——第 5 章就有这样的例子，它展示了多个 GC 线程对 CPU 使用率的影响。但总体来说，在多 CPU 机器上运行多线程的目标仍然是通过确保单个线程不被阻塞而提升 CPU 使用率，或者是在线程已经完成工作，正在等待更多的工作时，降低CPU使用率（在很长的时间间隔内）。

在多 CPU 多线程的情况下，还有一个关于 CPU 何时空闲的重要补充：CPU 在有工作可做时也可能空闲。如果没有处理那项工作的可用线程，这种情况就会发生。典型的情况是，应用程序以固定大小的线程池运行各种任务。线程任务以队列形式放置，当有线程空闲而且队列中有任务时，该线程会取出任务并执行。然而，每个线程一次只能执行一个任务，如果该任务被阻塞（例如，等待数据库的响应），那么该线程在这期间无法执行新的任务。此时，有任务要执行（有工作要完成）但是没有可用的线程来执行它们，结果就是 CPU空闲。

在这个具体的例子中，应该增加线程池的大小。但是，不要仅仅因为有空闲的 CPU 可用，就认为应该增加线程池的大小以完成更多的工作。程序无法获得 CPU 周期，还有我们之前提到过的两个原因——锁或外部资源的瓶颈。在确定行动方案之前，了解程序为什么没有获得CPU很重要。（有关这一主题的更多详细信息，请参见第9章。）

查看 CPU 使用率是了解程序性能的第一步，但这仅仅是看看代码是否使用了所有可用的CPU资源，或者看看是否出现了同步问题或资源问题。

# 3.1.2 CPU运行队列

Windows 系统和 Unix 系统都可以监控运行（意味着它们没有被 I/O 阻塞或者休眠）的线

程数。Unix 系统称之为运行队列，许多工具的输出包含运行队列的长度。3.1.1 节中的vmstat 的输出就包含这一数值：每行第一个数字就是运行队列的长度。Windows 系统称之为处理器队列，可以用 typeperf 命令查看（还有其他方式）：

```batch
C:\> typeperf -si 1 ""System\Processor Queue Length" "05/11/2019 19:09:42.678", "0.000000" "05/11/2019 19:09:43.678", "0.000000" 
```

这个输出和之前的输出的重要区别是：在 Unix 系统中，运行队列的长度（vmstat 示例输出中的 1 和 2）指的是，正在运行的线程的数量加上一旦 CPU 可用就可以运行的线程的数量。在示例中，总是有至少 1 个线程试图运行，即以单线程执行应用程序。因此，运行队列的长度至少是 1。记住，运行队列表示的是机器上所有的进程信息，所以有时其他线程（完全不同的进程）也试图运行，这就是为什么示例输出中的运行队列长度有时是2。

在 Windows 系统中，处理器队列长度不包含正在运行的线程的数量。因此在 typeperf 的样例输出中，处理器队列的长度是0，即便机器上运行的是同一个单线程应用程序，而且其中的线程始终在运行。

如果要运行的线程数量多于可用的 CPU，性能就会开始下降。总体来说，为确保性能，你需要让 Windows 系统的处理器队列长度等于 0，或者让 Unix 系统的运行队列长度小于等于 CPU 数。这不是一条硬性规定，系统进程和其他进程会定期地暂时提高这个值，这不会对性能产生重大影响。但是如果运行队列在相当长的时间内过长，那就说明机器已经过载，你需要想办法减少机器当前的工作量（通过将任务移至其他机器或者优化代码）。

# 3.1.3 磁盘使用率

监控磁盘使用率有两个重要目的。第一个目的与应用程序本身有关：如果应用程序正在执行大量磁盘I/O 操作，那么I/O很容易成为瓶颈。

很难知道磁盘 I/O 何时是瓶颈，因为它取决于应用程序的行为。如果应用程序不能有效地缓冲写入磁盘的数据（例子见第 12 章），磁盘 I/O 的统计数据就会很低。如果应用程序执行的 I/O 超过了磁盘的处理能力，那么磁盘 I/O 的统计数据会很高。注意，这两种情况的性能都需要优化。

监控磁盘使用率的第二个目的——即使预计应用程序不会进行大量 I/O 操作——是监控系统是否在进行内存交换。计算机有固定的物理内存，但是它可以用更大的虚拟内存来运行一系列应用程序。应用程序会保留超出其需要的内存量，且通常只在一部分内存上操作。在这两种情况下，操作系统会将没有用到的内存部分保留在磁盘上，只在需要的时候将之转入物理内存。

在大多数情况下，这种内存管理的效果很好，尤其是对于交互式应用程序和 GUI 应用程序（这是有好处的，否则你的笔记本计算机需要比原来多得多的内存）。这种内存管理对于服务器端应用程序效果较差，因为这些应用程序往往需要更多内存。由于 Java 堆的原因，这种内存管理对于任何 Java 应用程序（包括在你的桌面上运行的基于 Swing 的 GUI 应用程序）都很糟糕。更多详细信息参见第5章。

系统工具也可以报告系统是否在交换内存。比如，vmstat的输出中有两列数据（si代表换入，so代表换出），可以警示我们系统是否正在进行内存交换。磁盘活动也说明内存交换可能正在发生。请密切注意这些信息，因为系统在交换内存时——将数据分页从主内存移入磁盘或者进行相反的过程——往往性能相当差。必须配置系统，使其永远不会发生内存交换。

# 快速小结

• 监控磁盘使用率对于所有的应用程序都很重要。对于不直接写入磁盘的应用程序，系统内存交换会影响它们的性能。  
• 向磁盘写入数据的应用程序遇到瓶颈的原因有两个：一是写入数据的效率不高（吞吐量太低），二是写的数据太多（吞吐量太高）。

# 3.1.4 网络使用率

如果你运行的应用程序使用了网络——比如 REST 服务器——那么你必须监控网络流量。网络使用率低的原因类似于磁盘流量的情况：应用程序可能低效地使用了网络，以至于带宽太低，或者写入特定网络接口的数据总量超过了接口可以处理的量。

不幸的是，标准系统工具对于监控网络流量来说并不理想，因为它们通常只显示特定网络接口发送和接收的数据包数量和字节数。虽然有用，但这些信息并不能告诉我们网络是利用不足还是利用过度。

在 Unix 系统中，基本的网络监控工具是 netstat（大多数 Linux 发行版没有包含 netstat，必须单独获取）。在 Windows 系统中，可以在脚本中用 typeperf 监控网络使用率——某些情况下 GUI 有优势，比如标准的 Windows 资源监视器可以显示网络使用百分比的图表。不过，GUI在进行自动化性能测试的时候帮助不大。

幸好，许多开源工具和商用工具可以监控网络带宽。在 Unix 系统中，受欢迎的命令行工具是nicstat，它可以展示每个网络接口的流量概况，包括接口的使用程度：

% nicstat 5   

<table><tr><td>Time</td><td>Int</td><td>rKB/s</td><td>wKB/s</td><td>rPk/s</td><td>wPk/s</td><td>rAvs</td><td>wAvs</td><td>%Util</td><td>Sat</td></tr><tr><td>17:05:17</td><td>e1000g1</td><td>225.7</td><td>176.2</td><td>905.0</td><td>922.5</td><td>255.4</td><td>195.6</td><td>0.33</td><td>0.00</td></tr></table>

其中e1000g1是1000 MB的接口，它在这个例子中并没有被充分利用（ $( 0 . 3 3 \%$ ）。这个工具（和其他类似的工具）的用处在于它可以计算接口的使用率。在上面的输出中，数据正在以 225.7 KBps 的速率从接口读取，并以 176.2 KBps 的速率向接口写入。对于 $1 0 0 0 \mathrm { M B }$ 的网络，做除法之后可以得到约 $0 . 3 3 \%$ 的使用率，而nicstat工具可以自动地算出接口的带宽。

类似 typeperf 和 netstat 的工具会报告读写数据的总量，但是要计算网络使用率，你必须在获取带宽后用脚本进行计算。一定要记住，带宽的单位是位/ 秒（bps），而工具报告的一般是字节 / 秒（Bps）。1000 兆位网络每秒处理 125 兆字节（ $. 1 2 5 { \mathrm { ~ M B } } )$ ）。在上述例子中，读的速率为 225.7KBps，写的速率为 176.2KBps，总计 3.29Mbps。把它除以接口速率就可以得到约 $0 . 3 3 \%$ 的使用率。所以 nicstat 或类似的工具并没有什么神奇之处，它们只是使用起来更方便。

网络无法维持 $1 0 0 \%$ 的使用率。对于本地以太网，使用率持续超过 $40 \%$ 就说明接口饱和了。如果是分组交换网络或者利用了不同传输介质的网络，那么最大持续使用率可能会不一样。请咨询网络架构师以确定合适的数值。这个数值和 Java 无关，Java 只不过是利用了操作系统的网络参数和接口。

# 快速小结

• 对于基于网络的应用程序，要监控网络以确保它不是瓶颈。  
• 向网络写入数据的应用程序遇到瓶颈的原因有两个：一是写入数据的效率不高（吞吐量太低），二是写的数据太多（吞吐量太高）。

# 使用调优标志

JVM 可以设置大量调优标志，其中许多标志是本书的重点。追踪这些标志及其默认值有点令人生畏，上面最后两个 jcmd 的例子在这方面很有用。command_line 命令显示了命令行直接设定的标志，flags 命令显示了命令行设置的标志和 JVM 直接设置的一些标志（因为它们的值可以自动测出来），加上 -all 命令可以列出所有的标志。

JVM 调优标志有上百个，大多数令人费解。建议永远不要对其中大多数标志进行修改（见下面的框注“信息太多？”）。在诊断性能问题时，经常要搞清楚哪些标志是有效的，jcmd命令就可以在 JVM 运行时做到这一点。通常，你需要搞清楚特定 JVM 平台相关的默认值，这个时候在命令行添加 -XX:+PrintFlagsFinal 命令就很有用。最简单的方式就是执行以下命令：

% java other_options -XX:+PrintFlagsFinal -version

……上百行输出，包括……

uintx InitialHeapSize := 4169431040 {product}

intx InlineSmallCode = 2000 {pd product}

你应当在命令行中添加任何你想用的标志，因为一些标志（特别是与 GC 相关的标志）会影响另一些标志的最终值。这会打印出 JVM 标志及其值的完整列表（这和 jcmd 使用VM.flags -all 命令打印的内容是一样的）。

这些命令的标志数据会以上述例子中的两种方式显示。第一行输出中的冒号表示标志使用了非默认值。这可能是以下原因导致的：

• 标志的值直接通过命令行设定；  
• 其他标志间接地改变了这个标志的值；  
• JVM自动计算出了默认值。

第二行没有冒号，表示当前的值是这个版本 JVM 的默认值。某些标志的默认值在不同平台上可能不一样，最后一列会显示出来。product 表示该标志的默认设置在所有平台是统一的。pd product 表示该标志的默认值依赖于平台。

最后一列还可能出现的值有 manageable 和 C2 diagnostic。前者表示标志的值可以在运行期间动态更改，后者表示标志可以为编译器工程师提供诊断输出，以了解编译器如何运行。

另一种针对运行中的应用程序查看此类信息的方式是使用 jinfo。它的优势是允许在应用程序执行过程中更改标志的值。

以下命令可以获得进程中所有标志的值：

% jinfo -flags process_id

加上 -flags 选项，jinfo 会提供所有标志的信息，否则只会输出命令行设定的标志。这两种命令的输出都不如 -XX:+PrintFlagsFinal的输出更易读，但是 jinfo有其他重要的特性。

jinfo可以检查单个标志的值：

% jinfo -flag PrintGCDetails process_id

-XX:+PrintGCDetails

jinfo 本身并没有显示标志是否可操纵（manageable），但是可操纵（正如 PrintFlagsFinal输出中所展示的那样）的标志可以通过 jinfo 打开或者关闭。

% jinfo -flag -PrintGCDetails process_id # 关闭PrintGCDetails

% jinfo -flag PrintGCDetails process_id

-XX:-PrintGCDetails

# 信息太多？

PrintFlagsFinal 命令会输出几百个可用的 JVM 调优标志（例如 JDK 8u202 有 729 个可用的标志）。这些标志中的绝大多数是为了支持工程师从正在运行（运行异常）的应用程序中收集更多的信息。当你知道一个叫作 AllocatePrefetchLines 的标志（默认值是 3）后，你可能很想通过改变这个值让预读指令在特定处理器上运行得更好，但是这种随便的优化并不值得。如果没有令人信服的理由，那么任何一个标志都不应该更改。对 AllocatePrefetchLines 标志来说，需要了解应用程序的预读性能、运行应用程序的 CPU的特点以及改变这个数字会对 JVM代码本身有什么影响。

请注意，在 JDK 8 中，jinfo 可以更改任一标志的值，但这并不意味着 JVM 会响应这个更改。比如，大多数影响 GC 算法行为的标志会在启动时决定垃圾回收器的行为方式，之后通过 jinfo 更改这些标志并不会改变 JVM 的行为，JVM 会根据初始的算法继续执行。所以这个方法只对 PrintFlagsFinal 命令的输出结果中标记为 manageable 的标志起作用。在JDK 11 中，jinfo 会在你尝试修改不可变标志的值时报告错误。

# 3.3 性能分析工具

性能分析器是性能分析师工具箱中最重要的工具。Java 有很多性能分析器，每个都各有优缺点。性能分析常常需要使用各种工具——特别是采样分析器。采样分析器往往会以不同的方式展示问题。因此，有的分析器在某些应用程序上能很好地确定性能问题，而在其他应用程序上就不能。

很多常用的 Java 性能分析工具本身也是用 Java 写的，它们在工作时要“附加”到被分析的应用程序上。这种“附加”是通过 socket 或者被称为 JVM 工具接口（JVM ToolInterface，JVMTI）的原生 Java 接口进行的。然后，目标应用程序和性能分析工具开始交换关于目标应用程序行为的信息。

这意味着你必须注意对性能分析工具进行优化，和优化任何其他 Java 应用程序一样。尤其是应用程序很大的时候，它可能会传递大量数据到性能分析工具，所以性能分析工具必须

有足够大的堆来处理数据。运行性能分析工具时开启并发GC 算法通常是个好主意，性能分析工具不合时宜的Full GC暂停会导致缓冲区的数据溢出。

# 3.3.1 采样分析器

性能分析有两种模式：采样模式和探查模式。采样模式是性能分析的基本模式，造成的系统开销很小。这一点很重要，因为性能分析的一个缺陷就是在应用程序中引入测试会改变其性能1。限制使用性能分析可以让结果更接近通常情况下应用程序的行为。

不幸的是，采样分析器可能会出现各种误差。定时器会定期触发采样分析器，以检查每个线程并确定正在执行的方法。我们可以这样认为：这个方法一旦被定时器触发就会一直运行。

最常见的误差如图 3-1 所示。线程正在交替执行方法 A（阴影部分）和方法 B（空白部分）。如果定时器恰好只在线程执行方法 B 的时候触发，那么分析器会报告所有的时间都在执行方法B。事实上，方法 A花费的时间更长。

![](images/e739ea7edcfbe1ee2f42324238b347f60f7510b28c4b72fa7c4b64d44e436dc6.jpg)  
图3-1：方法交替执行

这是最常见的采样误差，但绝不是唯一的一个。想要减小误差，就要延长采样周期并减小采样间隔。减小采样间隔和“最小化性能分析的影响”这个目标相反，我们需要在这二者之间寻找一个平衡点。不同的性能分析工具处理这个平衡点的方式不同，这就是各种工具报告的数据有很大不同的原因之一。

这是所有采样分析器都固有的误差，只是对于很多 Java 性能分析器（特别是比较旧的分析器）会更糟。这是由安全点偏差（safepoint bias）造成的。在常用的性能分析器的 Java 接口中，性能分析器只有在线程到安全点之后才可以获取线程的栈轨迹（stack trace）。以下情况中，线程会自动进入安全点：

• 在同步锁上阻塞  
• 等待 I/O 时阻塞   
• 等待管程时阻塞   
• 线程挂起   
• 正在执行 Java 本地接口（Java Native Interface，JNI）代码（执行 GC 锁定函数除外）

另外，JVM 可以通过设置一个标志，让线程进入安全点。检查该标志（必要时进入安全点）的代码会被插入到 JVM 代码的关键位置，譬如某些内存分配过程中以及切换编译后

的（循环和方法）代码中。何时进行安全点检查并没有具体的规范，并且不同版本之间也存在差异。

安全点偏差对采样分析器的影响可能是深远的。因为栈只有达到安全点才会被采样，所以采样过程就不那么可靠了。在图 3-1 中，如果没有安全点偏差，那么一个随机采样分析器不太可能只在方法 B 执行期间触发线程采样。但是有了安全点偏差，又很容易看到这样的情况：方法A 永远也跑不到安全点，所有的时间都算到了方法B上。

Java 8为工具提供了获取栈轨迹的不同方式。（这是比较旧的工具有安全点偏差，而比较新的工具往往没有安全点偏差的原因之一，不过这需要新工具用新的机制重写。）在程序层面，这是通过 AsyncGetCallTrace 接口实现的。使用了这个接口的性能分析器往往被称为异步分析器（async profiler）。这里的异步是指 JVM 提供栈信息的方式，而不是指性能分析工具的工作方式。之所以被称为异步，是因为 JVM 可以在任何时间点提供栈信息，而无须等待线程到达（同步）安全点。

因此，使用这个异步接口的分析器，采样的失真就比其他采样分析器少一些（不过它们仍会有图 3-1所示的误差）。异步接口在 Java 8 被公开，在此之前它是私有接口。

所以，性能分析数据最上面的方法可以指示应该在哪个区域寻找可优化的地方。性能工程师不会试图让JVM的方法更快，而会搞清楚如何在总体上加快对象序列化。

还有另外两种方法来可视化采样输出，它们都可以直观地显示调用栈。最新的一种方法叫作火焰图（flame graph），它是一个应用程序调用栈的交互式图表。

# 3.3.2 探查分析器

探查分析器相对于采样分析器更有侵入性，但是它能提供更多关于应用程序内部正在发生什么的有益信息。

探查分析器会在类加载时更改字节码序列（插入统计调用次数的代码等）。与采样分析器相比，探查分析器更可能在应用程序中引入性能偏差。例如，JVM 会内联小方法（见第 4章），这样，执行小方法的代码时就不会产生方法调用。编译器是基于代码的大小来决定是否内联的，而根据代码探查的方式，方法可能不再符合内联的条件。这可能导致探查分析器高估某些方法对性能的影响。内联仅仅是一个例子，用于说明编译器会基于代码布局做决定。总体来说，探查（改变）的代码越多，性能分析数据就越可能改变。

因为探查引入了代码上的改变，所以最好限制个别类使用探查分析器。这意味着探查分析器最好用于二级分析：采样分析器将性能问题指向某个包或某段代码，然后探查分析器在需要时深入研究此代码。

图3-5 使用了探查分析器（没有使用异步接口）来查看REST样例服务器。

![](images/65ad965592d891585f00452906b76da63c60586537a8663f8140837acfc02034.jpg)  
图3-5：探查分析器

这个分析器的分析结果与事实有些不同。首先，主要的时间算在了 writeObject() 方法上，而不是 writeObject0() 方法。这是因为私有方法被排除在探查范围之外。其次，实体管理器出现了一个新的方法。它之前没有出现是因为在采样分析的例子里，它已经被内联到构造方法里了。

这种性能分析更重要的步骤是查看调用计数（invocation count）。调用计数显示，我们对那

个实体管理器的方法调用了 3300 多万次，对计算随机数的方法调用了 1.66 亿次。我们应该通过减少调用这些方法的次数，来对性能产生更大的影响，而不是优化它们的实现的速度。如果没有调用计数，我们就不一定会知道这一点。

这种性能分析比之前的采样分析更好么？这要分情况。在给定的情况下，我们无法得知哪种性能分析更准确。探查分析的调用计数肯定是准确的，这些额外的信息往往有助于确定代码在哪些地方花费了更多的时间，以及优化哪些方面更有效。

在这个例子中，探查分析器和采样分析器的结果都指向同一代码区域：对象序列化。在实践中，不同的性能分析器可能指向完全不同的代码区域。性能分析器可以用来很好地估计，但仅仅是估计而已，它有时候也会出错。

![](images/e7e6d61df44a40c4955d2ff2cb2757b61114372acd963dd1cf86ca9bf0629b6b.jpg)

# 快速小结

• 探查分析器能够给出更多的应用程序信息，但相对于采样分析器，它对应用程序的改动也更大。  
• 探查分析器应当用于探查一小部分代码——几个类或包。这有助于限制对应用程序性能的影响。

# 3.4 JFR

Java 飞行记录器（Java Flight Recorder，JFR）是 JVM 的一项功能，它可以对正在运行的应用程序进行轻量级性能分析。JFR 数据是 JVM 中事件的历史记录，可以用来诊断 JVM过去的性能和操作。

JFR 最初是 BEA 公司的 JRockit JVM 的功能，最终它被合并到了 Oracle 公司的 HotSpotJVM 中。在 JDK 8 中，只有 Oracle JVM 支持 JFR（而且只授权给 Oracle 客户使用）。但在JDK 11 中，JFR 可以在开源 JVM（包括 AdoptOpenJDK JVM）中使用。因此，它能以开源的形式向后移植到 JDK 8 中。也许有朝一日，AdoptOpenJDK 和其他版本的 JDK 8 都会支持JFR（虽然到 ${ \tt R u 2 3 2 }$ 为止，都还不支持）。

JFR 的基本操作是启用一系列事件（例如，线程因为等待锁而被阻塞的事件），每次选定的事件发生后，事件的相关数据就会保存下来（在内存或者文件中）。因为数据流被保存在一个循环缓冲区中，所以保存下来的只有最近的事件。之后你可以使用工具展示这些事件——实时从 JVM 中获取或者从保存的文件中获取——并分析这些事件以诊断性能问题。

事件的类型、循环缓冲区的大小、数据的存储位置等都可以在应用程序运行时通过 JVM的各种参数以及包括 jcmd 命令在内的工具来控制。在默认设置下，JFR 的系统开销非常低：对应用程序性能的影响小于 $1 \%$ 。但随着启用的事件增加、触发事件的阈值改变等，开销也会发生变化。本节稍后会讨论所有的配置细节，不过我们首先来看看展示的这些事件是什么样的，这样可以更轻松地了解JFR的工作原理。

# 3.4.1 Java Mission Control

至于管程阻塞事件，正如第 9 章讨论的那样，锁的竞争有两个级别，其中一个是线程自旋等待锁，然后它使用（进程中称为锁膨胀）一些特定的 CPU 或者 OS 代码等待锁。标准的分析器会提示这种情况，因为自旋时间包含在方法的 CPU 时间中。原生分析器可以给出锁膨胀的信息，但这是不确定的。而JVM可以直接向JFR提供所有的这些数据。

• GC时间（包括单个阶段）；代的大小  
• JFR只是统一了几个工具的信息

• 探查分析和采样分析  
• 没有你从真正的性能分析器中获得的信息多，但 JFR 的性能分析提供了很多高层次的概要信息

# 3.4.3 开启JFR

JFR 最初是关闭的，开启它需要在应用程序的命令行中添加标志-XX:+FlightRecorder。这就开启了 JFR 功能，但是直到记录过程本身开启之后才会有记录。开启记录可以通过GUI或者命令行。

在 Oracle 公 司 的 JDK 8 中， 你 必 须 在 -XX:+FlightRecorder 标 志 之 前 设 定 标 志：-XX:+UnlockCommercialFeatures（默认值为 false）。

如果忘记加上这些标志，那么你还可以用jinfo改变它们的值并开启JFR。如果你使用 jmc启动一次记录，那么它会在必要时自动改变这些标志在目标JVM中的值。

# 1. 通过 jmc 开启 JFR

在本地应用程序中开启记录最简单的方式是通过 Java Mission Control GUI（jmc）。当 jmc启动时，它会显示当前系统上所有的 JVM 进程。JVM 进程在树形节点配置中展示，打开飞行记录器标签下的节点会弹出图3-15所示的飞行记录器窗口。

Start Flight Recording

![](images/89564f479390177e23170ca25c73e56914974cc97d82f54088db1d3a3b4003e2.jpg)  
图 3-15：JFR 启动飞行记录器的窗口

飞行记录可以有两种模式：固定时间（图中为 1 分钟）模式或者持续记录模式。持续记录模式需要使用循环缓冲区。缓冲区会包含最近的事件，这些事件在预期的持续时间和大小之内。

为了执行主动分析，就是启动记录后做一些事情或者在 JVM 预热后的负载测试期间启动记录，应当使用固定时间模式记录。这些记录会指示在测试期间JVM是如何响应的。

持续记录最适合响应式分析（reactive analysis）。这会让 JVM 保存最近的事件，然后在某个事件发生时转储记录。例如，WebLogic应用程序服务器可以在服务器内发生异常（例如处理超过 5 分钟的请求）时触发记录转储。你也可以将你的监控工具设置为在某一事件发生时转储记录。

# 2. 通过命令行开启 JFR

在开启 JFR 之后（使用 -XX:+FlightRecorder 标志），有多种方式来控制如何以及在何时开始记录。

在 JDK 8 中，使用 -XX:FlightRecorderOptions=string 参数来启动 JVM 时，可以控制默认的记录参数。这对响应式记录最有用。参数中的 string 是一组逗号分隔的名 - 值对，有以下选项。

name=name

用来标识记录的名字。

```txt
defaultrecoding=<true/false> 
```

是否在初始时开启记录。默认值是 false。对于响应式分析，这个值应该设为 true。

settings $\equiv$ path

包含 JFR设置的文件名（见下一节）。

delay $=$ time

记录开始前的时间（例如，30s、1h）。

duration $=$ time

记录持续的时间。

filename $\equiv$ path

写入记录的文件名。

compress $=$ <true/false>

是否压缩记录（用 gzip）。默认值是 false。

maxage $=$ time

在循环缓冲区中保存记录数据的最长时间。

maxsize $=$ size

在循环缓冲区中保存记录数据的最大大小（例如，1024K、1M）。

-XX:FlightRecorderOptions 只可以设置选项的默认值。单次记录可以覆盖这些值。

在 JDK 8 和 JDK 11 中，通过 -XX:StartFlightRecording=string 标志以及类似的一组以逗号分隔的命令开始运行程序时，你都可以启动 JFR 记录。

在某些情况下设置默认记录（如下）很有用。但是为了更灵活，可以在运行期间通过 jcmd控制所有的命令。

开启飞行记录：

```txt
% jcmd process_id JFR.start [options_list] 
```

options_list 是一系列以逗号分隔的名 - 值对，用来控制记录方式。可用的选项和在命令行中使用 -XX:FlightRecorderOptions=string 标志时的选项完全相同。

如果启用了持续记录，那么当前循环缓冲区中的数据可以在任何时间通过下面的命令转储到文件中：

```txt
% jcmd process_id JFR.dump [options_list] 
```

选项列表如下所述。

```gitattributes
name=name 
```

已经开始的记录名（见下一个 JFR.check 的例子）。

filename $\equiv$ path

转储文件的位置。

给定的进程可能启动了多个 JFR 记录。通过以下命令查看可用记录：

% jcmd 21532 JFR.check [verbose]

21532:

Recording 1: name=1 maxsize=250.0MB (running)

Recording 2: name $^ { \circ 2 }$ maxsize=250.0MB (running)

在这个例子中，ID 为 21532 的进程有两个活跃的JFR 记录，名字分别是 1 和 2。名字可以在其他 jcmd命令中用作识别记录的标识。

最后，终止正在进行的记录：

% jcmd process_id JFR.stop [options_list]

这个命令有以下选项。

name=name

要终止的记录名。

discard=boolean

如果是 true，则丢弃数据，而不是写入之前提供的文件中（如果有的话）。

filename=path

将数据写入给定的路径。

在自动化性能测试系统中，当需要检查运行中是否存在性能倒退时，运行这些命令行工具而产生的记录会非常有用。

# 3.4.4 选择JFR事件

正如之前所述，JFR可以监控多个事件。通常，这些事件是周期性的，它们每隔几毫秒发生一次（例如，基于采样的性能分析事件）。其他事件仅在事件的持续时间超过特定阈值时才触发（例如，只有当read()方法耗费的时间超过设定时间时，读取文件的事件才会触发）。

# 其他 JFR 事件

JFR 是可扩展的：应用程序可以定义自己的事件。因此，你的 JFR 实现可能会显示更多可用的事件类型，这取决于测试的应用程序。例如，WebLogic 应用程序服务器会开启多个应用程序服务器事件：JDBC 操作、HTTP 操作等。这些事件的处理方式与这里讨论的其他 JFR 事件一样：它们可以单独开启，可能有与之关联的阈值，等等。同样，JVM的后期版本可能会有更多的事件，这里不做讨论。

请查阅最新的产品文档，了解最详细的信息。

回收事件自然会涉及系统开销。回收事件的阈值对启用JFR记录所带来的开销也有作用，因为它会增加事件的数量。在默认记录中，并不是所有的事件都会回收（6个最消耗系统资

源的事件没有开启），基于时间的事件阈值会比较高。这让默认配置的系统开销小于 $1 \%$ 。

有时候额外的开销是值得的。例如，查看 TLAB 事件可以帮助你判断对象是否直接分配到老年代上，但是这些事件的记录默认是不开启的。同样，性能分析事件记录默认是开启的，但也仅仅是每 20毫秒触发一次——这可以给出适当的概览，但也会导致采样误差 4。

JFR 需要捕获的事件（和事件的阈值）定义在模板中（模板通过命令行的settings 命令选中）。JFR 自带了两个模板：默认模板（限制事件，让系统开销小于 $1 \%$ ）和性能分析模板（将大多数基于阈值的事件设置为每 10 毫秒触发一次）。性能分析模板的开销大约为 $2 \%$ （但是你的情况可能不同，通常开销会比这个值低）。

模板由 jmc 模板管理器管理。你可能已经注意到图 3-15 里有个按钮可以启动模板管理器。模板存储在两个位置：$HOME/.jmc/ 目录下（用户本地）和 $JAVA_HOME/jre/lib/jfr 目录下（JVM 全局）。模板管理器让你能够选择全局模板（可以说全局模板“位于服务器”）、本地模板或者定义新的模板。要定义模板，请浏览可用的事件后根据需要开启（或关闭）它们，还可以选择设置事件触发的阈值。

图 3-16 中显示为开启阈值为 15 毫秒的文件读取（File Read）事件：文件读取时间超过15 毫秒就会触发事件。这个文件读取事件配置为在触发时生成栈轨迹。这增加了系统开销——这就是获取事件的栈轨迹是可配置选项的原因。

# 快速小结

• JFR 可以尽最大可能查看 JVM 内部信息，因为它内建于 JVM 中。  
• 像所有工具一样，JFR给应用程序引入了一些系统开销。日常使用时，JFR能以较低的开销收集大量信息。  
• JFR 在性能分析中非常有用，在生产环境开启后也同样有用，因为可以检查影响性能的事件。

# 3.5 小结

好的工具是做好性能分析的关键。本章只是对工具能告诉我们的信息进行了简单的介绍。以下是需要牢记的关键事项。

• 没有完美的工具，每种工具都有其优势。X 分析器可能适合很多应用程序，但在某些情况下，它会遗漏 Y分析器可以明确指出的信息。一定要灵活使用这些工具。  
• 命令行监控工具可以自动收集重要数据。在自动化性能测试中，最好使用命令行监控工具收集监控数据。  
工具发展很迅速。本章提到的某些工具可能已经过时了（或至少已经被更新、更优秀的工具所取代）。与时俱进在这个领域很重要。

# 使用即时编译器

即时编译器（just-in-time compiler，简称 JIT compiler）是 Java 虚拟机的核心。没有什么比即时编译器更能影响应用程序的性能了。

本章会深入介绍编译器，首先介绍编译器的工作原理，然后讨论使用即时编译器的优缺点。在 JDK 8 出现之前，你必须在两个 Java 编译器之间做出选择。如今，这两个编译器仍然存在，但彼此协同工作，不过在极少数情况下还是有必要选择其中一个。最后，我们会查看编译器的中级和高级优化方法。如果应用程序在没有任何明显原因的情况下运行缓慢，本章内容会帮助你判断编译器是否有问题。

# 4.1 即时编译器：概览

我们将从一些介绍性的材料开始，如果你已经了解了即时编译的基础知识，可以随时跳过。

计算机，更确切地说是 CPU，只能执行相对较少的特定指令，这些指令被称为机器码。因此，CPU执行的所有程序都必须翻译成这些指令。

像 $\mathrm { C } { + + }$ 和 Fortran 这样的语言被称为编译型语言，因为它们的程序是以二进制（编译后的）代码的形式发布的：程序编写完成后，由静态编译器生成二进制文件。该二进制文件中的汇编代码是针对特定 CPU 的。兼容的 CPU 可以执行同一个二进制文件。例如，AMD 和Intel 的 CPU 共享一个基本的、常用的汇编语言指令集，新版本的CPU 几乎总是可以执行和该CPU 先前版本相同的指令集。反之则不然，新版本的CPU 往往会引入旧版本CPU 无法运行的指令。

像 PHP 和 Perl 这样的语言是解释型语言。只要机器上有合适的解释器（就是 php 程序和perl 程序），同样的程序源代码就可以在任何CPU 上运行。在这些程序代码运行时，解释器会将每一行代码都翻译成二进制代码。

每种语言类型都有优缺点。用解释型语言编写的程序是可移植的。把同样的代码放在任何有合适解释器的机器上，它都可以运行，只是可能会运行得很慢。举个简单的例子，想一下在循环中会发生什么：在循环中执行时，解释器会重新翻译每一行代码。而编译后的代码不需要重复翻译。

好的编译器会在生成二进制文件时考虑多种因素。一个简单的例子是二进制语句的顺序：并非所有的汇编语言指令都需要相同的时间来执行。将存储在两个寄存器中的值相加的语句可能只需要一个周期，而检索（从主内存）加法所需的值可能需要多个周期。

因此，好的编译器生成的二进制文件会先执行加载数据的语句，然后执行其他指令，最后在数据可用时执行加法。每次只查看一行代码的解释器则没有足够的信息生成这样的代码。它会从内存请求数据，等到数据可用时再执行加法。顺便说一下，糟糕的编译器也会如此，而且即使是最优秀的编译器，偶尔也有等待指令完成的时候。

由于这些（和其他）原因，解释型代码几乎总是比编译型代码慢得多。编译器有足够的程序信息，可以对二进制代码进行优化，这是解释器根本无法完成的。

解释型代码也有优势，即可移植。为 ARM CPU 编译的二进制文件显然不能在 Intel CPU上运行。使用了 Intel Sandy Bridge 处理器最新 AVX 指令的二进制文件，也无法在旧版的Intel 处理器上运行。因此，商业软件常常对相当旧的处理器版本进行编译，而且不会利用最新的指令。针对这个问题有很多技巧，比如在发布二进制文件时附带多个共享库，这些共享库执行对性能敏感的代码，并对各种类型的CPU提供不同的版本。

Java试图找到一个中间地带。Java应用程序需要编译，但不是针对特定CPU编译为特定的二进制文件，而是编译为一种中间的低级语言。这种语言（称为Java字节码）可以用java命令运行（和 php 命令运行解释型 PHP脚本的方式一样）。这使得 Java具有解释型语言的平台独立性。因为它执行的是理想化的二进制代码，所以java 程序能够在代码执行时将其编译为平台相关的二进制代码。这种编译是在程序执行时发生的，所以它是“即时”的。

这种编译仍然受制于平台。例如，JDK 8 无法为 Intel 的 Skylake 处理器的最新指令集生成代码，不过 JDK 11可以。关于这一点，我将在 4.4节详细介绍。

Java虚拟机在执行时编译代码的方式是本章的重点。

# 热点编译

正如第 1 章所讨论的那样，本书讨论的 Java 实现是 Oracle 的 HotSpot JVM。HotSpot 这个名字来自于它编译代码的方式。在典型的程序中，只有一小部分代码会频繁执行，应用程序的性能则主要取决于这些代码执行的快慢。这些关键代码被称为应用程序的热点，这部分代码执行得越多，就说这部分代码越热。

因此，当 JVM 执行代码时，并不是立即开始编译代码。这有两个基本的原因。第一个原因是，如果代码只执行一次，那编译它实际上是白费力气。相对于编译后只执行一次的代码，解释执行Java字节码会更快一点。

但如果代码是频繁调用的方法或者迭代多次的循环，编译它就是值得的。编译后的代码运行得更快，多次执行后节省的时间比编译代码所需的周期长。这种权衡是编译器首先解释

执行代码的原因之一— 编译器能分清楚哪些方法调用得足够频繁，可以进行编译。

第二个原因和优化有关。JVM 执行特定方法和循环的次数越多，它就能掌握越多的代码信息。这让JVM在编译代码时可以做大量优化。

这些优化（以及影响它们的方法）会在本章的后面讨论。先来看一个简单的例子：equals() 方法。这个方法存在于每个 Java 对象中（因为它们都继承 Object 类），常常会被重写。当解释器遇到语句 b $=$ obj1.equals(obj2) 时，它必须查找 obj1 的类型（类）以得知 equals() 具体执行的是哪个方法。这种动态查找可能会有些耗时。

随着时间的推移，JVM 注意到每次执行这条语句时，obj1 的类型都是 java.lang.String。然后 JVM 会生成编译后的代码，直接调用String.equals() 方法。现在代码就更快了，不仅仅是因为它被编译了，还因为它可以跳过动态查找方法的过程。

然而事情没那么简单，可能下次代码执行时 obj1 引用的类就不是 String 了。JVM 会为这种可能性创建编译后的代码，其中会涉及代码的逆优化和重新优化（你会在 4.3.4 节看到例子）。尽管如此，编译后的代码总体上还是更快一些（至少只要 obj1 持续引用 String 对象），因为它跳过了动态查找方法的过程。这种类型的优化只有在代码运行了一段时间并确定它的作用之后才能进行。这是即时编译器等待编译代码的第二个原因。

# 寄存器和主内存

编译器可以进行的最重要的一种优化，涉及何时使用主内存中的值以及何时将值存储到寄存器中。考虑下面的代码：

```java
public class RegisterTest { private int sum; public void calculateSum(int n) { for (int i = 0; i < n; i++) { sum += i; } } 
```

在某些时候，sum 实例变量必须驻留在主内存中，但是从主内存获取值是非常消耗资源的操作，完成它需要消耗多个周期。如果 sum 的值在每次循环迭代时都从主内存获取（或存储），那性能会很糟糕。有编译器就不一样了，编译器会将sum 的值加载到寄存器中作为初始值，执行循环时使用寄存器中的值，然后（在不确定的时间点）将最终的结果从寄存器存回主内存。

这种优化非常有效，但同时也意味着线程同步语义（见第 9 章）对应用程序的行为至关重要。一个线程无法看见其他线程所用的寄存器中存储的值，同步使得我们可以准确地知道寄存器何时被存储到主内存供其他线程使用。

寄存器的使用是编译器普遍采用的一种优化方法。通常即时编译器会频繁地使用寄存器。我们将在 4.4.4节中深入讨论这个问题。

# 快速小结

• Java的设计利用了脚本语言的平台独立性和编译型语言的原生性能。  
• Java 类文件会被编译成中间语言（Java 字节码），然后由 JVM 进一步编译成汇编语言。  
• 在将字节码编译成汇编语言的过程中进行的优化，能极大地提高性能。

# 4.2 分层编译

很久以前，即时编译器有两种类型，你必须根据你想使用的编译器安装不同版本的 JDK。这两种编译器就是 client 编译器和 server 编译器。在 1996 年，这是一个重要的区别；到了2020 年，情况就不一样了。如今，所有发布的 JVM 中都包含这两种编译器（尽管通常它们都被称为 server JVM）。

# 编译器标志

在旧版本的 Java 中，你可以通过一个标志设定你想使用的编译器，这个标志并不遵循 JVM 标志的常规惯例。你可以用 -client 来表示 client 编译器，使用 -server 或者-d64 来表示 server 编译器。

因为开发人员不会对脚本进行不必要的修改，所以你一定会遇到设定了 -client 或-server 的脚本和其他命令。但是，自 JDK 8 之后，这些标志就没有任何作用了。许多早期的 JDK 版本也是如此：如果你为一个只支持 server 编译器的 JVM 设定了-client，你还是会得到 server 编译器。

另外请注意，旧的标志 -d64（本质上是-server的别名）已经从JDK 11中移除，使用它会导致错误。在JDK 8中使用这个标志则不会有任何操作。

尽管被称为 server JVM，但 client 编译器和 server 编译器的区别依然存在，这两种编译器都可用而且已经被JVM使用。所以了解这种区别对于理解编译器的工作原理很重要。

曾经，JVM 开发人员（和一些工具）有时将这两种编译器简称为 C1（编译器 1，client 编译器）和C2（编译器2，server编译器）。现在，这些名称会比较贴切，因为客户端和服务器计算机之间的区别早已不复存在，所以我们之后将使用这些名称。

这两种编译器的主要区别在于它们编译代码的时机。C1 编译器比 C2 编译器更早开始编译。这意味着在代码执行的开始阶段，C1 编译器的速度更快，因为与 C2 编译器相比，它编译的代码更多。

此处工程上的权衡是，C2编译器在等待时获得了信息，这些信息让C2编译器能够对编译后的代码进行更好的优化。最终，C2 编译器生成的代码会比C1 编译器生成的代码更快。从用户的角度来看，这种权衡的好处基于程序会运行多长时间和程序的启动时间有多重要。

当这两种编译器不能兼容的时候，显而易见的问题是为什么需要做出选择：JVM 不能从C1 编译器开始，然后随着代码越来越热开始使用 C2 编译器吗？这种技术被称为分层编

译，它是现在所有 JVM 都在使用的技术，可以用 -XX:-TieredCompilation 标志（默认值是 true）关闭它。在 4.4节中，我们将讨论这样做的后果。

# 4.3.1 优化代码缓存

当 JVM 编译代码时，它会在代码缓存中保存一系列汇编语言指令。代码缓存的大小是固定的，一旦它被填满，JVM就不再编译代码了。

如果代码缓存太小，很容易就能看出潜在的问题。一些热点方法会被编译，另一些则不会被编译，应用程序最终会解释执行大量代码（非常慢）。

当代码缓存填满时，JVM 会发出如下警告：

```txt
Java HotSpot(TM) 64-Bit Server VM warning: CodeCache is full. Compiler has been disabled.  
Java HotSpot(TM) 64-Bit Server VM warning: Try increasing the code cache size using -XX:ReservedCodeCacheSize= 
```

有时候很容易错过这条消息。另一种判断编译器是否已经停止编译代码的方式是跟踪编译日志输出，这将在本节后面讨论。

目前，没有一种很好的机制来计算一个特定应用程序需要多少代码缓存，所以当你需要增加代码缓存的大小时，恰好设置成所需的大小可能就得碰运气了。典型的选择是，简单地将其设置为默认值的2倍或者4倍。

代码缓存的最大值可以通过 -XX:ReservedCodeCacheSize=N 标志设置（ $N$ 是刚刚提到的特定编译器的默认值）。代码缓存和 JVM 中大部分内存的管理方式一样：它有初始大小（通过-XX:InitialCodeCacheSize=N 设置）。代码缓存大小的分配从初始大小开始，并随着缓存填满而增加。代码缓存的初始大小为 2496 KB，默认的最大值为 $2 4 0 ~ \mathrm { M B }$ 。缓存大小的调整是在后台发生的，不会真正影响性能，所以一般设置为 ReservedCodeCacheSize（设置代码缓存的最大值）就可以了。

为代码缓存的最大值设定一个非常大的值，使空间永远不会被耗尽，有什么缺点吗？这取决于目标机器上的可用资源。如果设定了 1 GB 的代码缓存大小，JVM 会保留 1 GB 的原生内存。该内存直到有需要时才会被分配，但它仍然被保留，这意味着你的机器上必须有足够的虚拟内存用来保留原生内存。

# 保留的内存和分配的内存

理解 JVM 如何保留内存和如何分配内存之间的差别很重要。这种差别适用于代码缓存、Java堆和JVM的其他各种原生内存结构。

关于这个主题的详细信息，请参见8.1节。

此外，如果你还有一台使用 32 位 JVM 的旧 Windows 机器，那么总的进程大小不能超过4 GB。这4 GB包括Java堆、JVM 本身所有代码占用的空间（包括原生库和线程栈）和应用程序分配的任何原生内存（直接分配的或者通过 New I/O分配的），当然还有代码缓存。

这些都是代码缓存受限制的原因，也是有时需要对大型应用程序进行优化的原因。在内存充足的 64 位机器上，将代码缓存设置得很大不太可能对应用程序产生实际影响。这是因为应用程序不会用完进程空间的内存，而且额外保留的内存一般会被操作系统接受。

在 Java 11 中，代码缓存被分成 3 段：

• 非方法代码（nonmethod code）  
• 性能分析代码（profiled code）  
• 非性能分析代码（nonprofiled code）

默认情况下，代码缓存的大小是一样的（最大 240 MB），你仍然可以使用 ReservedCodeCacheSize标志来调整代码缓存的总大小。在这种情况下，非方法代码段根据编译器线程的数量分配空间（参见 4.4.2 节），在 4 个 CPU 的机器上大约是 $5 . 5 \ \mathrm { M B }$ 。其他两段平均分配剩余的代码缓存。例如，4个CPU的机器上每段大约是117.2 MB（总计是 $2 4 0 \mathrm { M B }$ ）。

你基本上不需要单独优化这些段，但如果需要，可以使用这些标志：

• -XX:NonMethodCodeHeapSize=N，用于调整非方法代码   
• -XX:ProfiledCodeHeapSize=N，用于调整性能分析代码  
• -XX:NonProfiledCodeHeapSize=N，用于调整非性能分析代码

使用 jconsole 标志并在内存面板中选中内存池代码缓存图表，可以实时监控代码缓存（以及JDK 11中的段）的大小。你还可以开启Java 的原生内存跟踪特性，如第8章所述。

# 4.3.2 检查编译过程

第二个标志本身不是用来优化的，它不会提升应用程序的性能。确切地说，这个标志，即-XX:+PrintCompilation 标志（默认为 false），可以让我们看到编译器的工作情况（尽管我们也会查看提供类似信息的工具）。

如果开启了 PrintCompilation 标志，每编译一个方法（或循环）后，JVM 都会打印出关于刚刚编译了什么的信息。

大部分编译日志采用以下格式：

timestamp compilation_id attributes (tiered_level) method_name size deopt

这里的 timestamp 指的是编译完成时的时间（相对于 JVM 启动的时间）。

compilation_id 是一个内部任务 ID。通常情况下，这个数字是单调递增的，但有时你可能会看到乱序的编译 ID。这种情况最常发生在有多个编译线程的时候，而且表明编译线程的运行速度相对其他线程较快或较慢。但不要认为某个特定的编译任务是因为某种原因而变得非常慢。这通常是线程调度造成的。

attributes 字段由 5 个字符构成，表示正在编译的代码状态。如果某个特定的属性适用于给定的编译，以下列表中对应的字符就会打印出来。否则，这个属性会打印一个空格。因此，这些属性字符可能同时出现两个或者多个，用空格分开。各属性如下：

%

OSR 编译

s

方法是同步的

!

方法有异常处理器

b

在阻塞模式下发生的编译

n

原生方法封装时发生的编译

这些属性中的第一个表示栈上替换（on-stack replacement，OSR）。即时编译是个异步的过程。当 JVM 认为某个方法应该被编译时，该方法就会被放在一个队列中。JVM 会继续解释执行该方法，而不是等待编译完成。下一次调用该方法时，JVM 才会执行它的编译版本（当然，假设编译已经完成）。

试想一个长期运行的循环。JVM 会注意到该循环应该被编译，并将代码放入队列等待编译。但这还不够。JVM 必须有能力在循环还在运行的时候就执行循环的编译版本——如果等循环和包含循环的方法运行完（这可能根本不会发生），效率会很低。因此，当循环的代码完成编译后，JVM 会替换代码（在栈上），循环的下一次迭代会执行速度快得多的编译版本的代码。这就是OSR。

接下来的两个属性应该是不言自明的。在当前的 Java 版本中，默认情况下 b 属性永远不会被打印出来，b 表示编译没有发生在后台（详见 4.4.2 节）。n 属性表示 JVM 生成了编译后的代码，用来加快对原生方法的调用。

如果分层编译被禁用了，下一个字段 tiered_level 将会是空白。否则，它会是一个数字，表示编译是在哪个层级完成的。

接下来是被编译的方法名（或者正在进行 OSR 编译的循环所在的方法名），打印格式为ClassName::method。

再接下来是被编译的方法的大小（以字节为单位）。这是Java 字节码的大小，而不是编译后的代码的大小（所以不幸的是，这不能用来预估代码缓存的大小）。

最后，在某些情况下，编译日志行尾会有信息表明发生了某种逆优化，这些信息通常是made not entrant 或 made zombie。详见 4.3.4 节。

# 用 jstat 检查编译

要想查看编译日志，需要程序以 -XX:+PrintCompilation 标志启动。如果程序启动时没有该标志，你可以通过 jstat 有限地了解编译器内部的工作情况。

jstat 有两个选项提供关于编译器的信息。-compiler 标志提供了关于编译方法数量的概要信息（这里的5003是所检查程序的进程ID）。

```txt
% jstat -compiler 5003  
Compiled Failed Invalid Time FailedType FailedMethod  
206 0 0 1.97 0 
```

请注意，这里还列出了编译失败的方法数量和最后一个编译失败的方法名。如果性能分析和其他信息让你怀疑某个方法因为没有被编译而变得很慢，这是验证假设的一种简单方式。

另外，你也可以使用 -printcompilation 标志来获取最后一个被编译的方法的信息。因为 jstat 用一个可选参数来重复这个操作，所以你可以看到随着时间的推移，哪些方法被编译了。在下面这个例子中，jstat 每隔一秒（1000 毫秒）输出一次 ID 为 5003的进程信息。

```txt
% jstat -printcompilation 5003 1000  
Compiled Size Type Method
```

# 4.3.3 分层编译级别

程序使用分层编译的时候，编译日志会输出每个方法在哪个层级被编译。在前面的样例输出中，代码的编译级别为 3 或 4。目前我们讨论了两个编译器和一个解释器，其中 C1 编译器有3个级别，所有总共有5个编译级别：

解释代码

简单C1编译代码

受限C1编译代码

完全 C1 编译代码

C2编译代码

典型的编译日志显示，大多数方法第一次编译的级别是 3：完全 C1 编译。（当然，所有方法都是从级别 0 开始的，只是在日志中不显示。）如果一个方法运行得很频繁，它就会到编译级别 4（级别 3 的代码会被丢弃）。最常见的编译路径是：C1 编译器得到代码是如何使用的信息之后，会利用这些信息进行优化，然后才开始编译。

如果 C2 编译器队列已满，那么队列中的方法会被取出，在级别 2 上编译。在这个级别上，C1 编译器会使用方法调用计数器和回边计数器（但不需要性能分析反馈）。这会让方法编译得更快。在 C1 编译器收集了性能分析信息之后，方法会在级别 3 上编译。最后，当 C2编译器队列不忙时，方法会在级别4上编译。

另外，如果 C1 编译器队列已满，那么计划在级别 3 上编译的方法在等待编译的同时，可以进行级别4的编译。这时，它会被快速编译到级别2，然后转到级别4。

不重要的方法可以从级别 2 或者级别 3 开始编译。因为它们本质上不那么重要，所以会回到级别 1。如果 C2 编译器因为某种原因不能编译代码，那么方法也会回到级别 1 编译。如果代码发生了逆优化，就会回到级别0。

调优标志可以控制部分编译级别转换行为，而且这种级别的优化可以得到乐观的结果。如果方法按预期编译，即编译从级别 0 开始，到级别 3，再到级别 4，则性能可以达到最佳状态。如果方法被频繁编译到级别 2，并且有额外的 CPU 周期可用，那就增加编译器线程的数量，这会缩短 C2 编译器队列的长度。如果没有额外的 CPU 周期可用，你能做的就是尽量缩减应用程序的大小。

# 4.3.4 逆优化

在之前讨论 PrintCompilation 标志的输出时，我们提到了两种编译器会逆优化代码的情况。逆优化（deoptimization）意味着编译器不得不“撤销”之前的编译。结果就是应用程序的性能降低，至少到编译器重新编译相关代码为止。

逆优化会在两种情况下发生：代码被丢弃（made not entrant）时和产生僵尸代码（madezombie）时。

# 1. 代码被丢弃

有两个原因会导致代码被丢弃：一个是类和接口的工作方式，另一个是分层编译的实现细节。

我们来看第一个原因。回忆一下，股票应用程序有一个 StockPriceHistory 接口。在样例代码中，这个接口有两个实现：一个是基本的实现（StockPriceHistoryImpl），另一个是给每个操作都添加日志的实现（StockPriceHistoryLogger）。在 REST 代码中，具体用哪个实现由 URL 的 log 参数决定：

StockPriceHistory sph;   
String log $=$ request.getParameter("log");   
if (log != null && log.equals("true")) { sph $=$ new StockPriceHistoryLogger(...);   
}   
else{ sph $=$ new StockPriceHistoryImpl(...);   
}   
//然后由JSP调用 sph.HighPrice(); sph.getStdDev(); //等等

如果对 http://localhost:8080/StockServlet（此时没有 log 参数）进行一组调用，那么编译器会看到 sph 的实际类型是 StockPriceHistoryImpl。然后它会内联代码，并据此进行其他优化。

之后调用 http://localhost:8080/StockServlet?log=true，会发现编译器对 sph 对象的类型所做的假设是不正确的，之前的优化不再有效。这就会产生逆优化陷阱（deoptimization trap），之前的优化被丢弃。如果有大量开启了日志的额外调用，那么 JVM 会快速编译这些代码并进行新的优化。

此时的编译日志将包含以下信息：

```batch

841113 937 s net.sdo.StockPriceHistoryImpl::process (248 bytes) made not entrant  
1322722 25 % net.sdo.StockPriceHistoryImpl::<init> @ -2 (156 bytes) made zombie  
1322722 937 s net.sdo.StockPriceHistoryImpl::process (248 bytes) made zombie 
```

请注意，OSR编译的构造方法和标准编译的方法都已经被标记为 made not entrant，一段时间后，它们会被标记为 made zombie。

至少在性能方面，逆优化听起来像是一件坏事，但事实未必如此。表 4-1 展示了在逆优化场景下REST服务器完成的每秒操作数。

第二个导致代码被丢弃的原因是分层编译的工作方式。当代码被 C2 编译器编译时，JVM必须替换已经被 C1 编译器编译的代码。它将旧代码标记为丢弃，并用逆优化机制替换为新编译的（更高效的）代码。因此，程序使用分层编译时，编译日志会显示大量方法被标记为丢弃。不要惊慌，这种“逆优化”实际上会让代码变得更快。

# 2. 逆优化僵尸代码

当编译日志显示产生了僵尸代码时，这表示之前标记为丢弃的代码已经被回收。在前面的例子中，使用 StockPriceHistoryLogger 实现的测试运行后，StockPriceHistoryImpl 类的代码就被标记为丢弃。但 StockPriceHistoryImpl 类的对象仍然存在。最后所有的对象都被GC回收。全部回收之后，编译器会注意到这个类的方法已经可以被标记为僵尸代码。

从性能角度来看，这是一件好事。试想一下，编译后的代码保存在固定大小的代码缓存中，当发现僵尸方法时，相关的代码可以从代码缓存中移除，为其他类的编译腾出空间（或者限制 JVM之后需要分配的内存量）。

可能的缺点是，如果类的代码被标记为僵尸代码之后重新加载并频繁使用，就需要 JVM重新编译并重新优化代码。就像前面的场景中发生的那样，测试运行时，一会儿没有日志，一会儿有日志，一会儿又没有日志了。这种情况下性能并没有受到明显影响。总的来说，僵尸代码的小规模重新编译，不会对大多数应用程序产生显著的影响。

# 4.4 高级编译器标志

本节介绍了一些影响编译器的其他标志，主要是为了让你进一步了解编译器的工作原理。这些标志一般不应被使用。介绍这些标志的另一个原因是，它们曾经很常见，并被广泛使用，所以如果你遇到了它们，想知道它们做了什么，本节会回答这些问题。

# 4.4.1 编译阈值

本章已经粗略地介绍过触发代码编译的条件。触发的主要因素是代码的执行频率，一旦代码执行到一定次数，就达到了它的编译阈值，编译器就会认为它有足够的信息来编译代码。

优化会影响这些阈值。但是本节的目的其实是让你更好地了解编译器的工作原理（并介绍一些术语）。在当前的JVM中，优化阈值的意义不大。

编译基于 JVM 中的两个计数器，分别用来统计方法被调用的次数，以及方法中任何循环的分支返回次数。分支返回（branching back）次数实际上可以看作一个循环完成执行的次数。无论是执行到了循环本身的结尾，还是执行了像 continue 这样的分支语句，都属于完成执行。

当 JVM 执行一个 Java 方法时，它会检查这两个计数器的总和，并判断这个 Java 方法是否可以编译。如果可以，方法就会被放到编译队列中（关于队列的更多细节，详见 4.4.2节）。这种编译没有正式的名称，通常被称为标准编译。

同样，每当一个循环完成一次执行时，分支计数器都会增加计数并进行检查。如果分支计数器超过了阈值，那么这个循环（而不是整个方法）就可以被编译。

再次提醒，优化会影响这些阈值。当禁用分层编译时，标准编译会由-XX:CompileThreshold=N的值触发， $N$ 的默认值是10 000。更改CompileThreshold标志的值会导致编译器比正常情况下

更早（或更晚）编译代码。请注意，尽管有这个标志，但其中的阈值计算仍是用回边计数器加上方法调用计数器。

你经常可以看到更改 CompileThreshold 标志的建议。一些公开的 Java基准测试使用了这个标志（经常被设置为 8000），一些应用程序在发布时也默认使用了这个标志。

但请记住，我提到过这个标志在禁用分层编译时才会起作用，也就是说，当分层编译开启时（正常情况下是这样），这个标志什么都不做。这个标志的使用实际上是从 JDK 7 以及更早期遗留下来的。

过去常常推荐使用这个标志，有两个原因。首先，降低这个标志的值可以改善使用 C2 编译器的应用程序的启动时间，因为代码会编译得更快、更有效。其次，它可能会导致一些方法被编译，而这些方法本来是永远都不会被编译的。

这一点有点反常：如果一个程序永远在执行，所有的代码最终都会被编译吗？事实并非如此。这是因为虽然编译器使用的计数器会随着方法和循环的执行增加计数，但是它们也会随着时间的推移而减少。每个计数器的值都会周期性地减少（特别是当 JVM 到达安全点时）。

实际上，这意味着计数器只是方法或循环最近热度的相对度量。这会产生一个副作用，那就是有些频繁执行的代码可能永远不会被 C2 编译器编译，即使这个程序永远运行。这些方法有时被形容为温热（与热相对）。在分层编译之前，这种情况下降低编译阈值是有益的。

然而在今天，即使是温热的方法也会被编译。如果我们能让它们被 C2 编译器而不是 C1编译器编译，也许可以稍微改进它们。这几乎没有什么实际的好处，如果你真的对此感兴趣，可以尝试改变标志 -XX:Tier3InvocationThreshold=N（默认值 200），让 C1 编译器更快地编译方法，或更改标志 -XX:Tier4InvocationThreshold=N（默认值 5000），让 C2 编译器更快地编译方法。回边阈值也有类似的标志可用。

# 4.4.2 编译线程

4.4.1 节提到过，当方法（或循环）可以被编译时，它会被放到编译队列中。这个队列由一个或多个后台线程处理。

这些队列并不是严格的先进先出队列，调用次数更多的方法有更高的优先级。所以即使在程序开始执行时有大量代码需要编译，这种优先级顺序仍然可以确保最重要的方法先被编译。（这也是 PrintCompilation输出中编译ID乱序的另一个原因。）

C1 编译器和 C2 编译器有不同的队列，每个队列由（可能有多个）不同的线程处理。线程

如果分层编译被禁用，则只启动给定数量的C2 编译器线程。

什么时候可以调整这个值？因为默认值是基于 CPU 数量设定的，所以在 Docker 容器中运行旧版本的 JDK 8 会导致自动优化出问题。在这种情况下，你需要手动将此标志设置为需要的值（用表 4-2 中的值作为参考，根据分配给 Docker 容器的 CPU 数量来确定需要的值）。

如果一个程序在单 CPU 的虚拟机上运行，只拥有一个编译器线程可能会略有好处。在很多情况下，可用的 CPU 有限，较少的线程竞争系统资源对性能有益。但这种优势只限于最初的预热期，在此之后，可以编译的方法数量不会真正引起 CPU 资源竞争。如果股票批处理应用程序在单 CPU 机器上运行，且编译器线程的数量被限制为 1 个，那么初始计算大约会快 $1 0 \%$ （因为不必经常争夺CPU 资源）。运行的迭代次数越多，这种初始收益的总体效果就越小，直到所有热点方法都被编译之后，收益就消失了。

另外，线程数量很容易让系统不堪重负，特别是同时运行多个 JVM 时（每个 JVM 都会启动多个编译线程）。在这种情况下，减少线程数量可以帮助提升整体吞吐量（不过，可能需要付出的代价是，预热期会持续更长时间）。

同样，如果有大量额外 CPU 周期可用，那么理论上，当编译器线程数量增加时，程序（至少在预热期）会受益。在现实生活中，很难获得这种好处。进一步来说，如果所有多余的 CPU 都是可用的，那么在整个应用程序执行期间，你最好尝试利用可用的 CPU 周期（而不仅仅是在开始时进行更快的编译）。

还有一个适用于编译线程的设置是 -XX:+BackgroundCompilation 标志，其默认值为 true，这个设置意味着队列会像刚才解释的那样进行异步处理。这个标志可以设置为 false，这种情况下，一旦方法可以被编译，想要执行它的代码就会等待，直到方法实际上完成编译（而不是继续在解释器中执行）。使用 -Xbatch标志也可以禁用后台编译。

# 快速小结

• 对于放在编译队列中的方法，编译是异步进行的。  
•	编译队列没有严格的顺序，热点方法会在队列中的其他方法之前编译。这是编译日志中编译ID出现乱序的原因之一。

# 4.4.3 内联

内联方法是编译器最重要的优化之一。良好地遵循面向对象设计的代码经常包含一些需要通过 getter（也许还有 setter）访问的属性：

public class Point { private int x, y; public void getX() { return x; } public void setX(int i) { $\mathbf{x} = \mathbf{i}$ ; }

调用这种方法的系统开销是相当大的，尤其是相对于方法中的代码量而言。实际上，在Java 的早期版本中，考虑到所有这些方法调用对性能的影响，一些性能优化技巧常常反对这种封装。幸好，如今的 JVM 已经可以对这种类型的方法进行常规的代码内联。因此，你可以编写如下代码：

Point $p =$ getPoint();  
p.setX(p.getX() * 2);

编译后的代码本质上会这样执行：

Point $p =$ getPoint();  
p.x = p.x * 2;

内联是默认启用的。可以通过 -XX:-Inline 标志禁用它，不过内联对性能提升非常重要，所以实际上你永远不会这么做（例如，禁用内联会使股票批处理测试的性能降低 $50 \%$ 以上）。也正因为内联如此重要，而且还有其他许多调优标志，所以还会有关于优化JVM 内联行为的建议。

不幸的是，基本没办法查看 JVM 是如何内联代码的。（如果你从源码编译 JVM，那么可以生成一个包含 -XX:+PrintInlining 标志的调试版本。该标志提供了编译器做出的内联决策的各种信息。）你能做的就是查看代码的性能分析结果，如果其顶部有任何简单方法看起来应该被内联，那就可以用内联标志进行试验。

一个方法是否可以被内联取决于它的热度和大小。JVM 根据内部计算来判断一个方法是否是热点（即频繁调用），它不直接受任何优化参数的影响。如果一个方法因为调用得很频繁而可以被内联，那么只有在它的字节码小于 325 字节（或通过 -XX:MaxFreqInlineSize=N标志设置的任何值）时，它才会被内联。否则，只有在字节码小于 35 字节（或通过-XX:MaxInlineSize=N 设置的任何值）时，方法才会被内联。

有时你会看到建议说，应该增大 MaxInlineSize 标志的值，以便内联更多方法。这种关系

常常被忽略的一个方面是，将 MaxInlineSize 的值设置为大于 35 字节，意味着一个方法在首次调用时可能就被内联了。但是，如果方法经常被调用，就说明它的性能重要得多，它总归是会被内联的（假设它小于 325 字节）。否则，优化 MaxInlineSize 标志的最终效果是，它可能会缩短测试所需的预热时间，但是不太可能对一个长期运行的应用程序产生很大影响。

# 快速小结

• 内联是编译器能做的最有利的优化，特别是对于属性封装良好的面向对象代码。  
• 内联标志很少需要优化。优化内联标志的建议往往没有考虑到正常内联和频繁内联之间的关系。在考察内联的效果时，一定要考虑这两种情况。

# 4.4.4 逃逸分析

如果开启逃逸分析（通过 -XX:+DoEscapeAnalysis 标志，默认值是 true），那么 C2 编译器会进行大幅度的优化。例如，试想以下计算阶乘的类：

```txt
public class Factorial { private AtomicInteger factorial; private int n; public Factorial(int n) { this.n = n; } public synchronized AtomicInteger getFactorial() { if (factorial == null) factorial = ...; return factorial; } 
```

如果想在数组中保存前 100 个阶乘的值，那么可以用以下代码：

```java
ArrayList<BigInt> list = new ArrayList<BigInt>();  
for (int i = 0; i < 100; i++) {  
    Factorial factorial = new Factorial(i);  
    list.add(factorial.getFactorial());  
} 
```

factorial 对象只在循环内部引用，没有其他代码可以访问这个对象。因此，JVM 可以自由地对该对象进行优化。

• 当调用 getFactorial() 方法时没必要获取同步锁。  
• 没有必要将字段 n 保存在内存中，可以将这个值保存在寄存器中。同样，也可以在寄存器中保存 factorial 对象的引用。  
• 事实上，根本不需要分配一个实际的 factorial 对象，只需要跟踪对象的各个字段。

这种优化很复杂。它在这个例子中非常简单，但更复杂的代码也可以进行这些优化。根据代码的使用情况，选择合适的优化策略，并不是所有的优化策略都适用。逃逸分析可以决

定哪些优化是可行的，并会在编译后的代码中进行必要的修改。

逃逸分析默认是开启的。在极少数情况下，它会出错。这通常不太可能发生，在当前的JVM 版本中，这种情况非常少见。不过，因为曾经有一些广为人知的bug，所以你有时会看到禁用逃逸分析的建议。这些建议很可能已经不再适用，不过就像所有激进的编译器优化一样，禁用这个特性可能会带来更稳定的代码，这也不是不可能。如果你发现了这种情况，那么简化相关代码是最好的做法：更简单的代码会编译得更好。（然而这是个 bug，应该被报告。）

# 快速小结

逃逸分析是编译器能够进行的最复杂的优化。这种优化经常导致微基准测试出错。

# 4.4.5 CPU相关代码

前面提到过，即时编译器的一个优点是它可以根据运行环境为不同的处理器生成代码。当然，这假定JVM是根据较新的处理器信息构建的。

这正是编译器为Intel 芯片所做的。2013 年，Intel 为 Haswell 以及之后的芯片引入了AVX2（Advanced Vector Extensions 2，高级矢量扩展 2）。JVM 马上跟进支持了这些指令。2016年，Intel 又引入了 AVX-512 指令，这些指令存在于 Knights Landing 和后续的芯片上。JDK 8不支持这些指令，但是JDK 11支持。

通常情况下，你不需要关心这个特性。JVM 会检测正在运行的 CPU，并选择合适的指令集。但就像所有的新特性一样，它有时会出错。

JDK 9 首次引入了对 AVX-512 指令的支持，不过默认情况下没有开启。对它的支持在开始时失败了几次，它先是被默认开启，然后又被默认禁用。例如，在 JDK 11 中，这些指令是默认开启的，但从 JDK 11.0.6 开始，这些指令再次被默认禁用。因此，即使在 JDK 11中，对它的支持仍然是一项正在进行的工作。（顺便说一句，这并不是 Java 独有的，许多程序都在努力让AVX-512 指令得到正确的支持。）

所以，在某些较新的 Intel 硬件上运行某些程序，你可能会发现早期的指令集运行得更好。这种受益于新指令集的应用程序一般比常见的Java应用程序涉及更多的科学计算。

上述指令集可以用 -XX:UseAVX=N 标志选择，其中 $N$ 的值如下。

0

不使用 AVX 指令

1

使用 Intel AVX 1 指令（对于 Sandy Bridge 和之后的处理器）

2

使用 Intel AVX 2 指令（对于 Haswell 和之后的处理器）

3

使用 Intel AVX-512 指令（对于 Knights Landing 和之后的处理器）

这个标志的默认值取决于运行JVM的处理器，JVM会检测CPU并选择它能支持的最高值。Java 8不支持级别3，所以你会看到大多数处理器上使用的值是2。在较新的Intel处理器上运行Java 11时，在11.0.5及之前的版本上默认值是3，在之后的版本上默认值是2。

这是第 1 章中提过的最好使用最新版本的 Java 8 或 Java 11 的原因之一，因为像这样的重要修复都包含在最新版本中。如果你必须在最新的 Intel 处理器上使用早期版本的 Java 11，那么可以设置 -XX:UseAVX=2 标志。在很多情况下，这将提升性能。

说到代码成熟度，为完整起见，还需要提一下 -XX:UseSSE=N 标志支持 Intel 流 SIMD 扩展（Streaming SIMD Extensions，SSE）1 到 4。这些扩展是针对奔腾系列处理器的。在 2010年优化这个标志有一定的意义，因为它所有的使用方式都在开发之中。今天，我们一般可以信任这个标志的稳健性。

# 4.5 分层编译的权衡

当在内存受限的环境中运行时，也许有理由关闭它。当然，你的 64 位机器有大量的内存，但你的应用程序可能运行在内存限制得很小的 Docker 容器中，或者在一个内存不太大的云虚拟机中。又或者你在大型机器上运行了几十个 JVM。在这些情况下，你可能希望减少应用程序的内存占用。

# javac 编译器

在性能方面，编译实际上说的是 JVM 内置的即时编译。不过请回想一下，Java 代码首先会被编译为字节码，这是通过 javac 进程发生的。所以我们在本节的最后，说几个关于它的要点。

关于 javac 编译器，最重要的一点是它完全不会影响性能（但有一个例外）。

• 包含额外调试信息的-g 标志不会影响性能。  
• 在 Java应用程序中使用 final 关键字并不能更快地编译代码。  
• 用较新的javac 版本重新编译通常不会使应用程序更快。

这三点是多年来沿用的建议，直到 JDK 11 出现。JDK 11 引入了一种新的字符串连接方式，可以比以前的版本更快，但需要重新编译代码才能用。这是上面规则的一个例外。总的来说，你永远不需要通过重新编译字节码来使用新的特性。更多关于这方面的详细信息会在12.1节中给出。

# 4.6 GraalVM

GraalVM 是一个新的虚拟机。它提供了一种运行 Java 代码的方法，不过它也可以运行许多其他语言的代码。这个通用虚拟机还可以运行 JavaScript、Python、Ruby、R，以及由Java 和其他可编译为 JVM 字节码的语言（如 Scala、Kotlin 等）编译出来的传统 JVM 字节码。Graal 有两个版本：完全开源的社区版（Community Edition，CE）和商用的企业版（Enterprise Edition，EE）。每个版本都有支持 Java 8 或 Java 11 的二进制文件。

GraalVM 对 JVM 性能有两个重要贡献。首先，插件技术允许 GraalVM 生成完全原生的二进制文件，我们将在下一节研究这个问题。

其次，GraalVM 可以以常规 JVM 的模式运行，只是它包含了一个新的 C2 编译器实现。这个编译器是用Java写的（而传统的 C2编译器是用 $\mathrm { C } { + + }$ 写的）。

传统的 JVM 包含了 GraalVM 的即时编译器的某个版本，具体是哪一版取决于 JVM 的构建时间。这些即时编译器都来自GraalVM的社区版，社区版比企业版慢。和你可以直接下载的GraalVM版本相比，它们通常也过时了。

在 JVM 中，Graal 编译器的使用是实验性的，所以要开启它，你需要提供这些标志：-XX:+UnlockExperimentalVMOptions、-XX:+EnableJVMCI 和 -XX:+UseJVMCICompiler。这些标志的默认值是 false。

# 4.7 预编译

本章开始时，我们讨论了即时编译器背后的理念。虽然它有自己的优势，但代码在编译执行之前仍需要预热期。如果在我们的运行环境中传统编译模型可以更好地工作，我们该怎么办？例如，嵌入式系统没有额外的内存可以满足即时编译的需要，或者程序在有机会预热之前就已经完成运行了。

在本节，我们将分析针对这种情况的两个实验特性。提前编译是标准JDK 11 的实验特性，生成完全原生的二进制文件是GraalVM 的特性。

# 4.7.1 提前编译

提前编译（ahead-of-time compilation，简称 AOT compilation）最初仅在 JDK 9 的 Linux 版本中可用，到 JDK 11 时所有平台都可以用了。从性能的角度来看，它仍然在发展过程中，

但我们将在本节中先睹为快 1。

提前编译允许你在运行应用程序之前先编译应用程序的一部分（或全部）。编译后的代码会形成一个共享库，供 JVM 启动应用程序时使用。理论上，这意味着至少在应用程序的启动阶段不需要引入即时编译：你的代码最初应该至少和 C1 编译器编译的代码运行得一样好，且不必等待代码被编译。

在实践中，则有一点不同：共享库的大小（也就是共享库加载到 JVM 中的时间）对应用程序的启动时间影响很大。这意味着像“Hello, world”一样简单的应用程序在使用提前编译后不会运行得更快（根据预编译共享库时做出的选择，实际上它可能运行得更慢）。提前编译针对的是像 REST 服务器这样启动时间相对比较长的程序。这样一来，加载共享库的时间就被较长的启动时间抵消了，提前编译就更有优势。但也要记住，提
