## Page 13

兰ava深试技术
对于一个从C++/C 语言转别Java语言的程序员来说。线程堆栈（thread trace
stack)或者线程转储（thread dump）技术应波不陌生，但对于原生的Java 程序员来
说。却有很多人不清楚这项能门绝技。线程堆栈也标线程调用堆栈。ava线程堆栈是虚拟
机中线程（包括锁）状态的一个最间快照，即系统在某个时刻所有线程的运行状态。包括
每一个线程的调用维栈、锁的持有情况等信息。每一种Java虚拟机（SUNJVM、IBM
JVM、JRockit.GNU JVM等）都提供了线程转储的后门，通过这个后门可以将那个时
刻的线程堆栈打印出来。虽然各种Java虚拟机在线程堆栈的输出格式上有一些不同，但
是线程维找的信息都包含了以下内容。
（1）线程的名字、ID、数量等。
（2）线程的适行状态。锁的状态（锁被哪个线程持有，哪个线程在等待锁等）。
（3）调用维栈（函数的调用属次关系）。调用堆栈包含完整的类名、执行的方法，以
及源代码的行号。
打印的堆栈信息内容的多少依赖于用户系统的复杂程度，通常有几十行到上万行不等。
借助线程堆栈的内容可以分析出许多问题，如线程死膜、锁争用/锁竞争、死循环、识别
耗时操作等。对于在多线程场合下的稳定性和可靠性问题，利用线程堆栈分析是最有效的
方法，在多数情况下甚至无须了解系统就可以进行相应的分析。
由于线程堆栈是系统某个时刻的线程运行状况(最间快照)，对于已经消失且没值痕迹
的信息，线程堆栈是无法进行历史追踪的。这种情况下，只能结合日志进行分析，如连接
池中的连接被哪些线程使用目没有被释放等。尽管如此，线程堆栈仍是多线程类应用程序
非功能型问题定位的最有效手段。线程堆栈善于分析的问题类型如下。
·CPU的使用率无缘放过高。
·系统挂起，无响应。
·系统运行速度题来越
●性能脂膜（如无法充分利用CPU等）。
线程死损、死疆环等。
●由于线程数量太多导数系统运行失效（如无法创建线程等）。
倡助线程堆栈可迅速接小问题的范围，找到突破口，直接命中目标。本章将对线程堆
栈进行详细介绍，包括内容如下。

1.1打印线程堆栈
Java虚拟机提供了线程转储的后门，通过这个后门就可以将线程维栈打印出来，向
Java 进程发适一个 QUIT 值号。Java 虚拟机收到该信号之后,可将系统当前的 Java 线
程调用堆栈打印出来。有的虚拟机（如 SUNJDK）能将维栈值息打印在屏幕上。而有的
虚浆机（如 IBM JDK)可直接将线程堆栈打印到一个文件中,从当前的运行目录下可以找
到波文件。当JDK将线程堆栈打印在屏幕上时，由于调用的堆栈信息量太大（一般可能有
几千行或者几万行）经常会超出控制台缓冲区的最大行数限制，从而造成信息丢失，因此
好手工将其重定向到一个文件中。在Windows 和 UNIX / Linux 下,可向 Java 进程请

1.4.2Java代码死猫环等导致CPU使用率过高的分析
当系统负载大时，CPU的使用率会较高，但是不正确的代码也会导致CPU使用率过
高，如死猫环。当发生CPU使用率过高的问题时。就需要分析其原因。既然CPU使用率
过离可能是死福环导致的，那么如何从线程堆栈中找到死福环的线程呢？死循环的线程有
一个特征就是，它永远执行不完。如果多次打印堆栈，通过前后堆栈对比我到一直在持续
运行的线程，这些线程就是可疑的线程。定位CPU使用率过高的本质是要找到这查可疑
的线程，其体步如下。
（1）通过前面介绍的堆找获取方法获取第一次堆找信息。
（2）等持一定的时间，再获取第二次堆找信息。
（3）预处理两次堆栈信息，先去掉处于 sleeping或者waing 状态的线程，因为这种
线程是不消耗CPU的。

2.3性能瓶颈分析的手段和工具
一个系统的性能瓶领分析过程大数如下
（1）进行单流握的性能瓶题分析，让单流程的性能达到最优，如采用更加简单的算法等。
单流照的性能分析可以借助:Opimizet或通过增加时间截等方式进行代码片断的分析”，分析
哪股代码期时比较多。这种分析不需要太多的技巧，因此本章不多做介绍。
（2）对整体进行性能照颜分析。因为单流程性能最优并不能保证整个系统性能就高。

2.3.4性能调优的终结条件
性能调优的过程总要有一个止点，那么需要满足什么条件，就能说明系统已经没有优
化的空间了呢？总的原刻是系统中算法已经足够简化，即从算法的角度已无法提升性能时，
增加压力。CPU 使用率随着压力的增加能趋近100%。则说明系统的性能已经榨干，性能
调优即告结束。系统需满定条件如下。
（1）算法足够优化。
（2）没有线程/资源的使用不当而导致的CPU利用不足。
如果达到上面的条件，性能仍然无法满足应用的要求。则只能通过考虑购买更好的机
器，或者集群来实现更大的容量支持。

3.2.1发生OOM（OutOfMemroy，内存泄漏）问题的原因
Java使用的内存种类包含三种类型，它们都可能发生内存进漏
(1) 堆内存不足。 如累 JVM 不能在 Java 维中获得更多内存来分配更多的 Java 对
象，则会据出 Java 难内存不足 (Java OOM) 错误。如果Java 堆充满了活动对象，并
且 JVM无法再扩展 Java 堆,都么它将不能分配更多 Java 对象。
（2）本地内存不足。如果JVM无法获得更多的本地内存，它将抛出本地内存不足（本
地 OOM)错误。当造程用划的内存到达强作系统的最大展值，或者当计算机用宽RAM 和

着所有可调度和不可调度的实体都可能被优化，以便用最佳状态执行，但是执行时间不可
预测。
在一个硬实时系统中，必须遵守时间期限，否则计算结果就会无效。例如，汽车中的
嵌入式系统，如果在加速时电子加速器的响应都延迟了，那么就会产生无法预料的行为。

4.10Java线程的优先级
Java线程模型支持线程优先级。本质上，线程的优先级是从1到10之间的一个数字，
数字越大表明任务越家急。JVM首先调用优先级较高的线程，然后才调用优先级较低的
线程。但是，该标准对具有相同优先级的线程的处理是随机的。
如何处理这整线程取决于基显的操作系统策略。在一些情况下。优先级相风的线程分
时运行：在另一些情况下，线程将一直运行到结束。请记住，Java支持10个优先级，基
显操作系统支持的优先级可能要少得多，这样就会造成一些混乱。因此，只能将优先级作
为一种很相略的工具使用。通常情况下，请不要依靠线程优先级来控制线程的状态。
但是，不要瘤望利用线程优先级来解决系统中优先级高低的问题。当系统负荷不高的
时候，设置优先级基本上没有什么价值：在负荷比较高的情况下，设置优先级基本上解决
不了任何问题。实际上，Java的GC会对实时性造成极大的剂弱，通过控制线程优先缓
即使能获得一点点的实时好处，也被GC特性给大大削弱了。因此在Java中，线程优先
