<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">A program must be</span><span style="font-family:'CMTI10';font-size:10.91pt;font-style:italic;color:#000000"> correctly synchronized</span><span style="font-family:'CMR10';font-size:10.91pt;color:#000000"> to avoid the kinds of counterintuitive behaviors that</span>

<span style="font-family:'CMBX12';font-size:11.95pt;font-weight:bold;color:#000000">3.1</span>
<span style="font-family:'CMBX12';font-size:11.95pt;font-weight:bold;color:#000000">Sequential Consistency</span>
<span style="font-family:'CMTI10';font-size:10.91pt;font-style:italic;color:#000000">Sequential consistency</span><span style="font-family:'CMR10';font-size:10.91pt;color:#000000"> is a very strong guarantee that is made about visibility and ordering in an</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">execution of a program. Within a sequentially consistent execution, there is a total order over all</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">individual actions (such as reads and writes) which is consistent with the order of the program.</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">Each individual action is atomic and is immediately visible to every thread. If a program has</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">no data races, then all executions of the program will appear to be sequentially consistent.</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">As</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">noted before, sequential consistency and/or freedom from data races still allows errors arising from</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">groups of operations that need to be perceived atomically and are not.</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">If we were to use sequential consistency as our memory model, many of the compiler and</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">processor optimizations that we have discussed would be illegal. For example, in Figure 2, as soon</span>

<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">Having discussed sequential consistency, we can use it to provide an important clariﬁcation</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">regarding data races and correctly synchronized programs. A data race occurs in an execution of a</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">program if there are conﬂicting actions in that execution that are not ordered by synchronization.</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">A program is correctly synchronized if and only if all sequentially consistent executions are free of</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">data races. Programmers therefore only need to reason about sequentially consistent executions to</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">determine if their programs are correctly synchronized.</span>

<span style="font-family:'CMBX12';font-size:14.35pt;font-weight:bold;color:#000000">What is a Memory Model?</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">A</span><span style="font-family:'CMTI10';font-size:10.91pt;font-style:italic;color:#000000"> memory model</span><span style="font-family:'CMR10';font-size:10.91pt;color:#000000"> describes, given a program and an execution trace of that program, whether</span>

<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">memory model works by examining each read in an execution trace and checking that the write</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">observed by that read is valid according to certain rules.</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">The memory model describes possible behaviors of a program. An implementation is free to</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">produce any code it likes, as long as all resulting executions of a program produce a result that</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">can be predicted by the memory model.</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">This provides a great deal of freedom to perform a</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">myriad of code transformations, including the reordering of actions and removal of unnecessary</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">synchronization.</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">A high level, informal overview of the memory model shows it to be a set of rules for when</span>

<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">When we use the term “read” in this memory model, we are only referring to actions that</span>

<span style="font-family:'CMBX10';font-size:10.91pt;font-weight:bold;color:#000000">Shared variables/Heap memory</span>
<span style="font-family:'CMR10';font-size:10.91pt;color:#000000">Memory that can be shared between threads is called</span><span style="font-family:'CMTI10';font-size:10.91pt;font-style:italic;color:#000000"> shared</span>

<span style="font-family:'CMTI10';font-size:10.91pt;font-style:italic;color:#000000">divergence</span><span style="font-family:'CMR10';font-size:10.91pt;color:#000000"> actions). For more information on these actions, consult Section 7.1.</span>
