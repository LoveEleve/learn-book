<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">我们用一个传统的电子表格文件作为示例，来说明</span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">MySQL</span><span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">如何处理同一组数据上的并发工作。电子表格</span>

<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">为只有你可以对该文件进行更改。现在，想象一下你需要与一位同事合作制作电子表格，文件存放在一</span>

<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">一整个团队的人积极地尝试编辑、添加和删除这个电子表格中的单元格内容，那会怎么样呢？我们可以</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">说他们应该轮流修改，但这样效率是极低的。我们需要一种允许并发访问大容量电子表格的方法。</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">读写锁</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">从电子表格中读取数据不会有什么麻烦，即使很多人同时读取也不会有问题。因为读取操作不会修改数</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">据，所以不太可能出错。如果有人试图删除</span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">A</span><span style="font-family:'Symbol';font-size:12.00pt;color:#000000">25</span><span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">单元格中的内容，而其他人正在读取电子表格，会发生</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">什么情况？这要视情况而定，读取者可能遇到报错退出，也可能得到不一致的数据视图。为安全起见，</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">即使是从电子表格中读取数据也需要特别小心。</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">如果把上述电子表格看作数据库中的表，很容易发现也会有同样的问题。从很多方面来说，电子表格就</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">是一张简单的数据库表。修改数据库表中的记录，和删除或者修改电子表格文件中的单元格内容十分类</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">似。</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">并发控制这一经典问题的解决方案相当简单。处理并发读</span><span style="font-family:'Symbol';font-size:12.00pt;color:#000000">/</span><span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">写访问的系统通常实现一个由两种锁类型组成</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">的锁系统。这两种锁通常被称为共享锁（</span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">shared</span><span style="font-family:'Symbol';font-size:12.00pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">lock</span><span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">）和排他锁（</span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">exclusive</span><span style="font-family:'Symbol';font-size:12.00pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">lock</span><span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">），也叫读锁（</span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">read</span>
<span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">lock</span><span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">）和写锁（</span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">write</span><span style="font-family:'Symbol';font-size:12.00pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">lock</span><span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">）。</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">先不考虑具体的锁定机制，锁的概念可以如下描述：资源上的读锁是共享的，或者说是相互不阻塞的。</span>

<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">在实际的数据库系统中，每时每刻都在发生锁定：当某个客户端在修改某一部分数据时，</span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">MySQL</span><span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">会通过</span>

<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">锁的粒度</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">一种提高共享资源并发性的方式就是让锁定对象更有选择性。尽量只锁定包含需要修改的部分数据，而</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">不是所有的资源。更理想的方式是，只对需要修改的数据片段进行精确的锁定。任何时候，让锁定的数</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">据量最小化，理论上就能保证在给定资源上同时进行更改操作，只要被修改的数据彼此不冲突即可。</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">问题是加锁也需要消耗资源。锁的各种操作，包括获取锁、检查锁是否空闲、释放锁等，都会增加系统</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">的开销。如果系统花费大量的时间来管理锁，而不是存取数据，那么系统的性能可能会受影响。</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">锁定策略是锁开销和数据安全性之间的平衡，这种平衡会影响性能。大多数商业数据库系统没有提供太</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">多的选择，一般都是在表中施加行级锁（</span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">row</span><span style="font-family:'Symbol';font-size:12.00pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">level</span><span style="font-family:'Symbol';font-size:12.00pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">lock</span><span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">），为了在锁比较多的情况下尽可能地提供更好</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">的性能，锁的实现方式非常复杂。锁是数据库实现一致性保证的方法。数据库操作专家必须深入源代</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">码，才能确定合适的配置，以优化速度与数据安全之间的平衡。</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">而</span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">MySQL</span><span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">则提供了多种选择。每种</span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">MySQL</span><span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">存储引擎都可以实现自己的锁策略和锁粒度。在设计存储引</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">擎时，锁的管理是一个非常重要的决定。将锁粒度固定在某个级别，可以提高某些应用场景下的性能，</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">但同时会使其不适合另外一些应用场景。好在</span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">MySQL</span><span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">提供了多种存储引擎，而不是单一的通用解决方</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">案。下面让我们来看两种最重要的锁策略。</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">表锁</span>

<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">在理解事务的概念之前，接触数据库系统的其他高级特性还言之过早。事务就是一组</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">SQL</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">语句，作为一</span>

<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">单纯的事务概念并不是故事的全部。试想一下，如果数据库在执行第</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">4</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">条语句时崩溃了，会发生什么？天</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">知道，客户可能会损失</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">200</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">美元。再假如，在执行到第</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">3</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">条语句和第</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">4</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">条语句之间时，另外一个进程要删除</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">支票账户的所有余额会发生什么？结果可能是银行不知不觉白白给了</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">Jane</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000"> 200</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">美元。</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">在这一系列操作中，有更多的失败可能性。连接可能会断开、会超时，甚至数据库服务器在操作执行过</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">程中会崩溃。这就是为什么存在高度复杂且缓慢的两阶段提交系统的典型原因：为了应对各种失败场</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">景。</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">除非系统通过严格的</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">ACID</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">测试，否则空谈事务的概念是不够的。</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">ACID</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">代表原子性（</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">atomicity</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">）、一致性</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">（</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">consistency</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">）、隔离性（</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">isolation</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">）和持久性（</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">durability</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">）。一个确保数据安全的事务处理系统，必</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">须满足这些密切相关的标准。</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">原子性（</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">atomicity</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">）</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">一个事务必须被视为一个不可分割的工作单元，整个事务中的所有操作要么全部提交成功，要么全</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">部失败回滚。对于一个事务来说，不可能只执行其中的一部分操作，这就是事务的原子性。</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">一致性（</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">consistency</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">）</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">数据库总是从一个一致性状态转换到下一个一致性状态。在前面的例子中，一致性确保了，即使在</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">执行第</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">3</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">、</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">4</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">条语句之间时系统崩溃，支票账户中也不会损失</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">200</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">美元。如果事务最终没有提交，该事</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">务所做的任何修改都不会被保存到数据库中。</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">隔离性（</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">isolation</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">）</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">通常来说，一个事务所做的修改在最终提交以前，对其他事务是不可见的，这就是隔离性带来的结</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">果。在前面的例子中，当执行完第</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">3</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">条语句、第</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">4</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">条语句还未开始时，此时有另外一个账户汇总程序</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">开始运行，其看到的支票账户的余额并没有被减去</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">200</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">美元。后面我们讨论隔离级别（</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">isolation</span>
<span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">level</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">）的时候，会发现为什么我们要说“通常来说”是不可见的。</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">持久性（</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">durability</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">）</span>

<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">一旦提交，事务所做的修改就会被永久保存到数据库中。此时即使系统崩溃，数据也不会丢失。持</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">久性是一个有点模糊的概念，实际上持久性也分很多不同的级别。有些持久性策略能够提供非常强</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">的安全保障，而有些则未必。而且不可能有</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">100%</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">的持久性保障（如果数据库本身就能做到真正的持</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">久性，那么备份又怎么能增加持久性呢？）。</span>
<span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">ACID</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">事务和</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">InnoDB</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">引擎提供的保证是</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">MySQL</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">中最强大、最成熟的特性之一。虽然它们在吞吐量方面做</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">了一定的权衡，但如果应用得当，就可以避免在应用层实现大量复杂逻辑。</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">隔离级别</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">隔离性在实际操作中比看起来复杂得多。</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">ANSI</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">SQL</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">标准定义了</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">4</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">种隔离级别。如果你是数据库领域的新</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">手，我们强烈建议你在阅读特定的</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">MySQL</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">实现之前先熟悉</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">ANSI</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">SQL</span><span style="font-family:'Symbol';font-size:8.92pt;color:#0000ee">[6]</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">的通用标准。这个通用标准的目标</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">是定义在事务内外可见和不可见的更改的规则。较低的隔离级别通常允许更高的并发性，并且开销也更</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">低。</span>

<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">每种存储引擎实现的隔离级别都不尽相同。如果你熟悉其他的数据库产品，可能会发现</span>

<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">某些特性和你期望的会有些不一样（因此本节不打算讨论更详细的内容）。读者可以根据所选择的</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">存储引擎，查阅相关的手册。</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">下面简单地介绍一下这</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">4</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">种隔离级别。</span>
<span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">READ</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">UNCOMMITTED</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">（未提交读）</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">在</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">READ</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">UNCOMMITTED</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">级别，在事务中可以查看其他事务中还没有提交的修改。这个隔离级别会</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">导致很多问题，从性能上来说，</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">READ</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">UNCOMMITTED</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">不会比其他级别好太多，却缺乏其他级别的</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">很多好处，除非有非常必要的理由，在实际应用中一般很少使用。</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">读取未提交的数据，也称为脏读（</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">dirty</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">read</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">）。</span>
<span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">READ</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">COMMITTED</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">（提交读）</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">大多数数据库系统的默认隔离级别是</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">READ</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">COMMITTED</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">（但</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">MySQL</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">不是）。</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">READ</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">COMMITTED</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">满足前面提到的隔离性的简单定义：一个事务可以看到其他事务在它开始之后提交的修改，但在该</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">事务提交之前，其所做的任何修改对其他事务都是不可见的。这个级别仍然允许不可重复读</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">（</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">nonrepeatable</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">read</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">），这意味着同一事务中两次执行相同语句，可能会看到不同的数据结果。</span>
<span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">REPEATABLE</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">READ</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">（可重复读）</span>
<span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">REPEATABLE</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">READ</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">解决了</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">READ</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">COMMITTED</span><span style="font-family:'Symbol';font-size:8.92pt;color:#0000ee">[7]</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">级别的不可重复读问题，保证了在同一个事务中</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">多次读取相同行数据的结果是一样的。但是理论上，可重复读隔离级别还是无法解决另外一个幻读</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">（</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">phantom</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">read</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">）的问题。所谓幻读，指的是当某个事务在读取某个范围内的记录时，另外一个事</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">务又在该范围内插入了新的记录，当之前的事务再次读取该范围的记录时，会产生幻行（</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">phantom</span>
<span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">row</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000"> ）。</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">InnoDB</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000"> 和</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">XtraDB</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000"> 存储引擎通过多版本并发控制（</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">MVCC</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000"> ，</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">Multiversion</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">Concurrency</span>
<span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">Control</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">）解决了幻读的问题。本章稍后会对此做进一步讨论。</span>
<span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">REPEATABLE</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">READ</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">是</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">MySQL</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">默认的事务隔离级别。</span>
<span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">SERIALIZABLE</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">（可串行化）</span>
<span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">SERIALIZABLE</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">是最高的隔离级别。该级别通过强制事务按序执行，使不同事务之间不可能产生冲</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">突，从而解决了前面说的幻读问题。简单来说，</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">SERIALIZABLE</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">会在读取的每一行数据上都加锁，</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">所以可能导致大量的超时和锁争用的问题。实际应用中很少用到这个隔离级别，除非需要严格确保</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">数据安全且可以接受并发性能下降的结果。</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">表</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">1</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">-</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">1</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">概述了这几种隔离级别之间的利与弊。</span>

<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">死锁</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">死锁是指两个或多个事务相互持有和请求相同资源上的锁，产生了循环依赖。当多个事务试图以不同的</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">顺序锁定资源时会导致死锁。当多个事务锁定相同的资源时，也可能会发生死锁。例如，设想运行下面</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">两个针对主键为（</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">stock</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">_</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">id</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">，</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">date</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">）的</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">StockPrice</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">表的事务：</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">事务</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">1</span>

<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">事务</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">2</span>

<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">每个事务都开始执行第一个查询，在处理过程中会更新一行数据，同时在主键索引和其他唯一索引中将</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">该行锁定。然后，每个事务将在第二个查询中尝试更新第二行数据，却发现该行已经被锁定。这两个事</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">务将永远等待对方完成，除非有其他因素介入解除死锁。我们将在第</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">7</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">章进一步介绍索引如何随着模式的</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">发展而影响或破坏查询的性能。</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">为了解决这个问题，数据库系统实现了各种死锁检测和锁超时机制。更复杂的系统，比如</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">InnoDB</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">存储引</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">擎，检测到循环依赖后会立即返回一个错误信息。这可能是一件好事——否则，死锁将表现为非常缓慢</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">的查询。还有一种方式，当超过锁等待超时的时间限制后直接终止查询，这样做通常来说不太好。</span>
<span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">InnoDB</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">目前处理死锁的方式是将持有最少行级排他锁的事务回滚（这是一种最容易回滚的近似算法）。</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">锁的行为和顺序是和存储引擎相关的。同样的一系列查询语句，有些存储引擎会产生死锁，有些则不</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">会。死锁的产生有双重原因：有些是因为真正的数据冲突，这种情况通常很难避免，但有些则完全是由</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">于存储引擎的实现方式导致的。</span><span style="font-family:'Symbol';font-size:8.92pt;color:#0000ee">[8]</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">一旦发生死锁，如果不回滚其中一个事务（部分或全部），就无法打破死锁。对于事务型的系统，这是</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">无法避免的，所以应用程序在设计时必须考虑如何处理死锁。大多数情况下只需要重新从头开始执行被</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">回滚的事务即可，除非又遇到另一个死锁。</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">事务日志</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">事务日志有助于提高事务的效率。存储引擎只需要更改内存中的数据副本，而不用每次修改磁盘中的</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">表，这会非常快。然后再把更改的记录写入事务日志中，事务日志会被持久化保存在硬盘上。因为事务</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">日志采用的是追加写操作，是在硬盘中一小块区域内的顺序</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">I</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">/</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">O</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">，而不是需要写多个地方的随机</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">I</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000">/</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">O</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">，所以</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">写入事务日志是一种相对较快的操作。最后会有一个后台进程在某个时间去更新硬盘中的表。因此，大</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">多数使用这种技术（</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">write-ahead</span><span style="font-family:'Symbol';font-size:11.89pt;color:#000000"> </span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">logging</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">，预写式日志）的存储引擎修改数据最终需要写入磁盘两次。</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">如果修改操作已经写入事务日志，那么即使系统在数据本身写入硬盘之前发生崩溃，存储引擎仍可在重</span>

<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">中，直到发出</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">COMMIT</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">或者</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">ROLLBACK</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">，然后</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">MySQL</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">会立即启动一个新的事务。此外，当启用</span>

<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">准的隔离级别，</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">InnoDB</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">也支持这些隔离级别。</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">在事务中混合使用存储引擎</span>
<span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">MySQL</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">不在服务器层管理事务，事务是由下层的存储引擎实现的。所以在同一个事务中，混合使用多种</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">存储引擎是不可靠的。</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">假设在事务中混合使用事务表和非事务表（例如，</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">InnoDB</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">和</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">MyISAM</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">表），如果一切顺利，事务将正常</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">工作。如果需要回滚，则无法撤销对非事务表的更改。这会使数据库处于不一致的状态，可能难以恢</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">复，并使整个事务问题变得毫无意义。所以，为每张表选择合适的存储引擎，并不惜一切代价避免在应</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">用中混合使用存储引擎是非常重要的。</span>
<span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">在非事务表中执行事务相关操作的时候，</span><span style="font-family:'Helvetica';font-size:11.89pt;color:#000000">MySQL</span><span style="font-family:'Unnamed-T3';font-size:11.89pt;color:#000000">通常不会发出提醒，也不会报错。有时候只有回滚的时</span>

<span style="font-family:'Unnamed-T3';font-size:17.00pt;color:#000000">小结</span>
<span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">MySQL</span><span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">拥有分层的架构，上层是服务器级别的服务和查询执行层，下层则是存储引擎。虽然有很多不同</span>
<span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">作用的插件</span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">API</span><span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">，但存储引擎</span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">API</span><span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">是最重要的。</span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">MySQL</span><span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">通过</span><span style="font-family:'Helvetica';font-size:12.00pt;color:#000000">API</span><span style="font-family:'Unnamed-T3';font-size:12.00pt;color:#000000">来与存储引擎交互需要处理的数据行，理</span>
