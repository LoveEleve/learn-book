前言

MySQL是世界上最流行的数据库之一，具有开源、快速、灵活、易用、稳定等特征，
广受用户欢避，MySQL起源于瑞典，最初只是一个用于报表系统的底层存储引擎，在使
用的过程中。随若数据量越来越大，系统复余度越来越高，最初的存储引擎已经无法满足
使用需求。因此，开发人员重新开发了一套类似的存储引擎ISAM（Indexed Sequential
Access Method)。2000 年，MySQL AB 公司成立，开发了支持事务处理的 Berkeley DB 引
擎，2005年发布了具有里程碑意文的MySQL5.0，明确地表现出迈向高性能数据库的发展
步伐。
经过20多年的发展，MySQL已经成为开源领域排名第一的数据库，实现了它的耶景
“世界上最受欢迎的数据库”。MySQL的使用领城非常广泛，包括互联网、初创企业、软
件集成商及云厂商，用户依赖 MySQL案节省时间和金钱，为他们的高容量网站、关键业
务系统和软件提供动力。
随着用户在应用程序中越来越多地使用MySQL，关键业务及生产系统对数据库的要
求越来越严格，如保证数据安全、数据零丢失、防止泄露敏感信息，以及系统具有高可用
性等，本书将重点介绍MySQL实现高可用的多种解决方案，用户可以根据不同的系统环
境选择合适的高可用解决方案。以实现系统要求提供的服务级别。
本书将从高可用的概念开始介绍，其中包括可靠性、恢复、元余等基础概念和MySQL
高可用选项。进面介绍MySQL高可用的演进，其中包括主从异步复制、半网步复制、组
复制及InmoDB Cluster 的简介。书中的各个章节将详细介绍各种解决方案的具体内容，并
提供演示，以帮助读者快速理解相关内容。

1.1.2恢复
实现可靠性最简单的方法是从故障中恢复。IT系统环境中有很多组件可能会发生放

·逻辑备份。逻辑备份通过通历数据，逐行地复制数据，将数据从二进制形式转换
为 SQL语句，并将数据转健为数据文件。恢复时，通过执行 SQL语句读取数据文
件，将数据导入数据库。逻辑备份的优点是灵活，数据可以在恢复时进行修改，
并且用户可以读取数据的内容。缺点是在进行大量数据备份时，速度慢，并且会
占用更多的存储空间。
物理备份。物理备份是磁盘存储层的二进制副本。可以简单地理解为操作系统层

用同一应用程序进行恢复。物理备份的优点是速度快，利用空间小，并且支持一
些高级特性。例如增量备份、差异备份等。
如果系统的数据量规模比较小，则可以使用逻辑备份。如果数据量较大，则推荐用户

2.2组复制
MySQL的复制功能具有很多优点，但随着系统的高可用要求和写入要求不断增加。
MySQL复制逐渐无法满足更高级别的可用性和多点写入等要求。因此，MySQL推出了组
复制。
组复制是个MySQL服务器插件,可以创建具有弹性、高可用性和容错的复制拓扑。
组复制基于Paxos 协议（Mencius）实现，支持多点写入，具有冲突检测和解决机制，它
允许应用程序写入的数据在网一组内的所有服务器上保持一致，组复制内置的模块支持跨
平台的分布式恢复，并通过内置的故障转移机制实现容错，组复制是一种非共享复制模式，
每台服务器都有完整的数据副本。
2.2.1组复制实现的理论
创建容错系统的最常见方法是使用元余，当其中的组件发生故障时，系统能够继续按
预期运行。这种要求为用户管来了一系列携成，将系统的复条性提高至与单一组件完全不
同的水平。使用元余的数据库必须维护和管理多台服务器，面不仅仅是一台服务器，对于
多台服务器共同协作、创建组，用户还必须解决分布式系统所面鉴的经典问题，如网络分
区、“脑题等。
组复制需要面临的问题是如何将数据库、数据复制的逻辑。以及协调多台服务器达成
致的逻辑三者融合在一起，让多台服务器对系统的状态和系统所经历的每一次数据更改
都达成一致，可以概括为让服务器对每个数据库的状志转换都达成一致，以便将它们整体
作为一个数据库进行操作。这意味着它们需要作为状志机（分布式）进行操作。
MySQL组复制提供了服务器之间强大的协调能力。当服务器属于同一组时，它们会
自动进行协调。该组可以在单主模式下运行，仅一台服务器作为主要成员接收数据更新，
其他服务器可以提供数据读取，并其有自动选举主要成员的能力，此外，组复制还可以在
多主模式下运行，组中所有服务器都可以进行数据更新和读取。使用多主模式时会有一些
使用限制，应用程序必须在使用前解决这些限别。

3.2.2使用GTID进行复制
GTID（Global Transaction ID）为全局事务提供了唯一标识。使用 GTID 时，每个事务
都可以在主服务器和从服务器上进行标识和跟踪。这意味着在配置新的从服务器或将故障

使用GTID 进行复制时，MySQL将按照如下过程进行。

（2)如果事务分配了 GTID,部么在提文时,GTID 将在事务开始时写入二进制日志中,
井且 GTID 将添加到系统变量@@GLOBAL-gtid_executed 中的 GTID 集合上。GTID 集合

满要读取并检查复制的事务的 GTID，在处理事务本身之前，从服务器不仪保证已应用的

（@@GLOBALgtid_owned)显示从服务器中当前使用的每个 GTID及拥有它的线程的 ID,

3.2.3配置半同步复制
复制有两种类型：异步复制和半同步复制异步复制是 MySQL默认使用的复制类型
在异步复制的情况下，主服务器产生二进制日志更改后。直接将日志内容发送给从服务
器，并提文存储引擎。在极确情况下，当日志发送后，由于网络异常等原因，从服务器未
接收日志，与此同时，主服务器发生不可恢复的故障，造成这部分传输数据丢失，为了通
免这种情况，MySQL推出了半网步复制功能，能够避免数据丢失，实现 RPO0.
读者可能会有疑问，为什么不使用完全网步复制呢？如果使用完全同步复制，则主服
务器需要等待全部从服务器接收日志，并提交到存储引擎，会产生很大的延迟，半网步复
制的开销介于并多复制与完全同步复制之间，主服务器等到从服务器收到日志的应答之
后，将事务提交到存储引擎。因此，半网步复制能够保证，即使在主服务器别溃后，已经
提交的事务也能够通过日志传输到从服务器。相比较于异步复制，半同步复制能够保证数
据的完整性：相比较于全网步复制，半同步复制延迟低、速度快。
MySQL的半同步复制功能通过插件来实现，在使用半同步复制的主从服务器上一定
要安装井启用半同步复制插件，如果主从服务器没有网时开启半网步复制功能，期执行异
步复制。半网步复制在执行过程中会遵照下面的步骤进行。
第一步，检查是否启用半网步。
第二步，如果已启用半同步插件，则在主服务器上进行的提交将等待从服务器的应答
或发生超时。
第三步，当发生超时后，半网步复制将恢复为异步复制，直至有一台从服务器赶上主
服务器的执行进度，异步复制将返国半同步复制。

4.1什么是组复制
组复制在 2016 年12月以 GA（General Available）的形式发布（从 5.7.17 版本开始）。

并以插件的形式绑定到MySQL服务器上。
传统的MySQL复制功能是异步的复制（或者是半同步复制），面组复制实现了完全同
步的复制。因此，组复制提供了更强的数据一致性（数据在所有服务器上都可用，不会因
为等待副本的追赶面造成延迟）。组复制和复制之间的另一个重要区别是，组中的所有服
务器都可以参与数据更新，并自动解决数据冲突。以往需要用户在应用程序端解决冲突间
题，以便向特定的服务器发运写（更新）请求。使用组复制则无须考虑这个问题，此外，
用户可以对组复制进行配置，只允许一台服务器（主要成员）进行更新，面其他服务器作
为次要成员或副本（用于放障转移）。
组复制的数据一致性可以通过分布式状态机实现，分布式状志机在一组服务器之间具
有强大的协调能力，这种通信允许服务器在组内自动协调复制，组自动维护成员关系，服

4.2.1.2主要成员的选举算法
组复制在进行主要成员的自动选举时，每个成员都需要查看组的最新视图，对潜在的
主要成员进行排序，并选择最合适的成员，每个成员根据主要成员选举算法在本地做出自
已的决定。所有成员必须达成一致的决定才能够成功选举主要成员。如果组内的 MySQL
服务器版本不同，那么组成员将调整它们的选举算法，与组内MySQL服务器版本最低的
成员相网。
主要成员的选举算法如下：
考虑MySQL的版本。组将查找运行最低MySQL服务器版本的成员。如果所有组
成员都运行MySQL8.0.17或更高版本，那么成员将根据其发布版本的补丁版本进
行排序。如果成员运行的是MySQL 5.7或MySQL 8.0.16 及更低的版本，那么成员
将按其发行版的主版本进行排序，忽略补丁版本。
·考虑每个成员的成员权重，权重由每个成员的group_replication_member_weight
系统变量指定。group_replication_member_weight 系统变量的取值范围是0~100.
所有成员的默认权重为50，设置低于50 的权重可以降低优先级，设置高于 50的
权重可以提高优先级。用户可以使用这个权重方法来确定使用服务器的优先级。
如果该组内有成员运行 MySQL 5.7, 并且未开启这个系统变量,期会忽略该因素。
·考虑每个成员的UUID，即系统指定的变量server_wuid。按照字典排序，在服务器
上UUID 最低的成员被选择为主要成员，该因素起到了保证和可预测的作用，所
有成员都会做出相网的决定。

员已经发生故障，并从组中驱逐。被驱逐的成员将从成员资格列表中删除，该列表可以被
其他成员看到，但是被驱逐的成员不知道白己已经从该组中被驱逐出案，并认为自己是在
线的，面其他成员无法访网。
如果一个成员与组中的其他成员形成隔高，则它怀疑其他成员全部发生故障。该成员
将无法与组达成共识（因为它无法达到共识要求，需要超过半数成员），因此，它的怀疑
不会对组产生任何影响，当成员通过这种方式与组隔高时，它将无法执行任何本地事务。
如果组复制使用的网络不稳定。并且成员经常以不网的组合丢失或重新建立彼此的通

并需要重新创建。因此，组复制的组通信系统（GCS）会跟踪标记为被驱逐的组成员（从
MySQL8.0.20开始），在决定是否存在大多数成员时，将它们视为被怀疑成员中的一员。
这样可以确保至少有一个成员保留在组中，使该组可以继续存在。当一个被驱逐的成员已
经被移出组时，GCS将删除该成员被驱逐的记录。以便该成员之后能够重新加入组

4.5组复制的事务一致性
组复制是一个分布式系统，作为一个分布式系统，如何保证事务一致性非常重要。在
对组复制配置事务一致性之前，读者首先需要了解组复制如何保证事务一致性。
4.5.1组复制的一致性相关事件
组复制是一个最终一致性的分布式系统，无论在正意操作还是故障恢复操作中，一且
传入组的数据流量变慢或停止，所有组成员就会具有相同的数据内容。与一致性相关的事
件可以分为控制操作和数据流操作
4.5.1.1主要成员故障转移的控制操作（单主模式）
在单主模式下，当发生数障转移时，组复制将次要成员提升为主要成员有以下两种
方法。
新的主要成员立即接管应用程序的流量，无论是否存在还未进行应用的事务。
组复制在主要成员发生故障后，会选择一个新的主要成员，并允许应用程序立即访问
数据（可能存在部分尚未进行复制应用的事务）以尽可能少的时间来确保组成员的稳定。
写一致性能够得到保证，但是会出现暂时读取过期数据的现象，新的主要成员在这段时间
进行复制应用。例如，如果客户端CI在旧的主要成员发生故障之前将A-1的数据更新为
A-2.当客户端C1重新连接新的主要成员时，可能读到A-1，该值将保持到新的主要成
员赶上旧的主要成员离开之前的状态。
·在复制过程中尚未进行应用的事务全部应用结束后，允许应用程序进行访间。
组复制在主要成员发生故障后，选择一个新的主要成员，在这种情况下，离要等到新
的主要成员将商未进行复制应用的事务应用之后才允许应用程序进行访间，以确保应用程
序读取最新的数据，在第一种方法握述的情况下，当客户端C1重新连接到新的主要成员
时，它能够读取A=2、但是，用户需要权衡故障转移所置的时间与尚未进行应用的事务
比例。

4.5.1.2数据流操作
数据流操作适用于组复制的两种模式：单主模式和多主模式。为了让读者更加容易理
解，在这里仅限于单主模式，通常情况下，数据的写入和读取会被路由到组的不同成员，
写入路由到主要成员，读取均匀分布到次要成员，为了保证一致性，组复制的行为应该作
为一个单一的实体来实现，因此。用户希望主要成员上的写入操作结果在次要成员上即时
可用。虽然组复制是使用基于Paxos算法的组通信系统（GCS）协议实现的，但实际上，
组复制的某些部分仍然是异步的，数据异步应用到次要成员，这意味看客户端在主要成员
上写入 B=2WHERE B-1，然后立即连接到次要成员进行读取，读取的结果可能是 B-1,
原因是次要成员尚末应用复制的事务
为避免发生上述不一致的问题，用户可以在组范围内基于网步事务的点配置组的一致
性保证。如果数据在读取时网步，那么当前客户端会话将等特一个给定的时间点（之前全
部更新事务都已应用的时间），然后才可以开始执行。在使用这种方法时，只有该会话受
到影响，其他所有并发数据操作不受影响。
如果数据在写入时网步，那么写入会话就会等特，直到全部次要成员都写入它们的数
据，组复制在写操作上使用一个总顺序，这意味着写入会话要等待次要成员的队列中同步
点之前的写入事务全部应用结束。
无论使用哪种方案，在上例握述的情况下，用户希望客户端能够在连接到次要成员
时，也总是读取B-2。每种方案都有其优缺点，它们与系统的工作负载直接相关。下面举
例说明一下，对于不同类型的工作场景，用户可以根据需要选择适合的同步点。
场景一：写入少于读取，对读取进行负载平衡，并且不希望在服务器上增加额外的限
制，以避免读取过期的数据。
场景二：以只读为主，一且提交了读写事务就立刻在所有的成员应用，以便后续的读
取操作基于最新的数据（包括最新的写操作）进行，确保用户不需要为每个只读事务消耗
同步成本，面只需要为读写事务消耗同步成本。
对于场景一和场景二，用户应该选择在写入时进行同步。
场景三：对读取进行负载平衡，不需要在服务器上增加额外的限制，以避免读取过期
