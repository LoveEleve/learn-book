## 1. 引言："信任程序员"的代价

前面三章讲 C 的设计哲学、内存模型、类型系统，建立了一个完整的心智模型。本章是这个模型的反面——当你理解了 C 的强大之后，你必须理解同样的哲学带来了什么。

C 的第一哲学是"程序员至上"——C 假定程序员知道自己在做什么。`strcpy` 不检查边界、`printf` 不验证格式串、`free` 不清空指针。这些"信任"成就了 C 的零开销性能，也在数十年间累积了成千上万的 CVE（Common Vulnerabilities and Exposures，通用漏洞披露）。

据统计，C 语言相关弱点在 MITRE CWE（Common Weakness Enumeration，通用缺陷枚举）Top 25（2023）中占据约 6 席——包括越界写入（CWE-787，排名 #1）、释放后使用（CWE-416，#4）、越界读取（CWE-125，#7）、空指针解引用（CWE-476，#12）、整数溢出（CWE-190，#14）——这些全部源于"信任程序员"哲学。格式化字符串（CWE-134）虽未进入 Top 25，但仍是 C 独有的漏洞类型（Java/Python 不存在 `printf` 由用户控制格式串的攻击面）。

技术专家不是"会写 C"的程序员。技术专家是"知道每一个 `strcpy` 调用潜藏着什么，并且知道如何防御"的工程师。本章教你第二种知识。

## 2. 五大漏洞类型

### 2.1 缓冲区溢出（Buffer Overflow / CWE-120）

**根因**：写入缓冲区时超出分配的大小。

```c
char buf[10];
strcpy(buf, "hello world, this is too long");  // buf 只有 10 字节！
// 写入 12 个字符后开始覆写栈上的其他数据
```

**Heartbleed 越界读复现**（CVE-2014-0160）：

```bash
cat > heartbleed.c << 'EOF'
#include <stdio.h>
#include <string.h>
// 简化版 TLS 心跳——OpenSSL 的 memcpy(bp, pl, payload) 没有检查 payload
char heap[1024];   // 模拟 OpenSSL 的堆缓冲区

void tls1_process_heartbeat(unsigned short payload_len) {
    char request[32] = "HELLO";      // 心跳请求——实际只有 5 字节
    char response[65536];
    // BUG: payload_len 来自网络报文——攻击者设为 65535
    memcpy(response, request, payload_len);  // 越界读！请求后 65535 字节被复制到响应
    printf("Copied %d bytes (leaked heap data)\n", payload_len);
}
int main() { tls1_process_heartbeat(65535); return 0; }
EOF
gcc -Wall heartbleed.c -o heartbleed && ./heartbleed
```

**经典 CVE**：Morris 蠕虫（1988，第一个 internet 蠕虫）。利用 `fingerd` 的 `gets()` 缓冲区溢出覆盖栈上的返回地址（另有 `sendmail` 的 DEBUG 命令注入作为辅助传播途径），跳转到恶意代码。

**实验——亲眼看栈被覆写**：

```bash
cat > overflow.c << 'EOF'
#include <stdio.h>
#include <string.h>

void vulnerable(char *input) {
    char buf[16];
    int  canary = 0xDEADBEEF;       // 栈金丝雀——在缓冲区旁边放一个已知值
    strcpy(buf, input);              // 不检查边界！
    printf("canary = 0x%08X (expect DEADBEEF)\n", canary);
    if (canary != 0xDEADBEEF) {
        printf("CANARY DEAD! 栈被覆写了！\n");
    }
}

int main() {
    vulnerable("AAAA");              // 4 字节——安全
    vulnerable("AAAAAAAAAAAAAAAA");   // 16 字节——刚好填满缓冲区
    vulnerable("AAAAAAAAAAAAAAAAAAAAAAAA"); // 24 字节——超出 8 字节，覆写 canary
    return 0;
}
EOF
gcc -fno-stack-protector -O0 overflow.c -o overflow && ./overflow
# 最后一个调用应该显示 "CANARY DEAD!"
```

**防御手段**（从最弱到最强）：

| 手段 | 原理 | 何时用 |
|---|---|---|
| `strncpy` | 带长度参数的复制 | 知道目标大小时。**陷阱**：当 src 长度 >= n 时，目标数组不会被 \0 终结——后续 `strlen` 越界读取 |
| `snprintf` | 格式化到带长度限制的缓冲区 | 构造字符串时使用。**陷阱**：返回值是"可能需要的字节数"不是"实际写入的字节数"——当返回值 >= n 时，输出被截断且**未 \0 终结** |
| `strlcpy`/`strlcat`（BSD） | 带边界检查且保证 `\0` 终结 | Linux/BSD 系统编程（**glibc 不提供，需 libbsd** — 非标准 C） |
| `-fstack-protector-strong` | 栈金丝雀：编译器在函数入口的栈帧中放一个随机值，函数返回前检查 | **生产必开** |
| `-D_FORTIFY_SOURCE=2` | libc 函数边界检查：编译时+运行时检测 `memcpy`/`sprintf` 等溢出 | **生产必开**，需 `-O1+` |

### 2.2 格式化字符串漏洞（Format String Vulnerability / CWE-134）

**根因**：用户输入被直接当做 `printf` 的格式串参数。

```c
printf(user_input);   // 危险！用户可放 %x/%s/%n
// 应改为：
printf("%s", user_input);  // 安全
```

**为什么危险**：
- `%x` 可以泄露栈上的数据（之后的 4/8 字节被解释为十六进制整数）
- `%s` 可以读取任意内存地址（如果攻击者能控制栈上的值）
- `%n` 可以向内存写入已输出字节数——这是唯一的"写"原语

**实验**：

```bash
cat > fmtstr.c << 'EOF'
#include <stdio.h>

void unsafe() {
    char buf[32];
    printf("Enter your name: ");
    fgets(buf, sizeof(buf), stdin);
    printf("Hello, ");
    printf(buf);              // 危险！
}

void safe() {
    char buf[32];
    printf("Enter your name: ");
    fgets(buf, sizeof(buf), stdin);
    printf("Hello, %s", buf); // 安全
}

int main() {
    printf("=== Unsafe version ===\n");
    unsafe();
    // 输入: %x.%x.%x.%x.%x 试试看能不能泄露栈数据
    return 0;
}
EOF
gcc -Wall fmtstr.c -o fmtstr
# 输入 %x.%x.%x.%x 观察栈数据泄露
```

**防御**：**永远**写 `printf("%s", user_input)`，不用 `printf(user_input)`。启用 `-Wformat -Wformat-security`（GCC/Clang 默认警告 `printf(user_input)` 的使用）。

### 2.3 整数溢出（Integer Overflow / CWE-190）

**根因**：有符号整数溢出 = UB；即使无符号溢出产生回绕，也可能在 `malloc` 中产生严重后果。

**有符号溢出 = UB**：

```c
int x = INT_MAX;
int y = x + 1;       // UB! 编译器可消除包含此语句的代码路径
if (y < x) { ... }   // 编译器可优化为 if (0) —— 参见 Ch1 §4.3
```

**无符号溢出 = Wrapping（模运算）**：

```c
size_t n = 0xFFFFFFFF;   // 假设 32 位
size_t sz = n * sizeof(char); // = 0xFFFFFFFF × 1 = 0xFFFFFFFF ≈ 4GB（大概率不是有意）
```

**malloc 乘法溢出**（经典 CVE 场景）：

```c
// 经典代码模式（错误的）：
struct item *items = malloc(count * sizeof(struct item));
// 如果 count 很大，count * sizeof(struct item) 可能溢出为一个较小的值
// malloc 分配不足的缓冲区 → 后续写入越界
```

**防御**：

```c
// 正确方式：分配前检查
if (count > SIZE_MAX / sizeof(struct item)) {
    return NULL;   // 溢出，拒绝
}
struct item *items = malloc(count * sizeof(struct item));
```

**OpenSSH 2002 案例**（CVE-2002-0390）：`n = nresp * sizeof(char *)` — `nresp` 来自网络报文，攻击者发送超大的 `nresp` 使 `nresp * 8` 回绕为 0，`malloc(0)` 返回一个有效指针，后续向"0 字节"缓冲区写入数据 → 堆溢出。这个 bug 在 OpenSSH 中从 2.0（2000 年）存在到 3.3/3.4（2002 年）修复。

**编译期检测**：`-fsanitize=undefined`（UBSan — Undefined Behavior Sanitizer，未定义行为消毒器）可在运行时检测有符号整数溢出，报告发生位置。

### 2.4 释放后使用（Use-After-Free / UAF / CWE-416）

**根因**：`free(p)` 后指针 `p` 仍持有旧地址，后续通过 `p` 访问已释放的内存。

```c
int *p = malloc(sizeof(int));
*p = 42;
free(p);
*p = 100;            // UAF! p 指向已释放的 chunk
// 此内存可能已被 ptmalloc 分配给另一个线程的 malloc 调用
// *p = 100 写入的内容发生不可预测的交叉
```

**为什么 UAF 不立即崩溃（Ch2 补充）**：`free` 将 chunk 归还 ptmalloc 的空闲链表，但进程的页表映射仍在——OS 没有回收物理页面。访问 `*p` 只是读到被其他代码复用的物理内存，或者旧值还没被覆盖。

**Linux kernel SLUB allocator UAF 案例**（CVE-2023-2235 等）：内核中的 slab/slub 分配器与 glibc ptmalloc 类似——`kmem_cache_free` 后缓存块被归还给 slab 的 per-CPU 缓存，下次 `kmem_cache_alloc` 把同一块物理内存重新分配给可能完全不同的内核对象。

**防御**：
1. `free(p); p = NULL;` — 免费预防（对非 copy 单次所有权有效）
2. `-fsanitize=address`（ASan — Address Sanitizer，地址消毒器）：运行时检测 UAF，报告时显示 free 和 use 两个调用栈（2 倍性能开销）
3. 引用计数（如 Redis `shared` 中的 `refcount`）— 在有多个所有者时必需

### 2.5 双重释放（Double-Free / CWE-415）

**根因**：同一块内存被 `free` 两次。后果不是"浪费了调用"——而是将同一块物理内存分配给两个不相关的指针。

```c
int *a = malloc(sizeof(int));
int *b = a;
free(a);
free(b);             // 双重释放！b 指向已释放的块
```

**为什么危险（Ch2 §6.5 重温）**：glibc 2.26+ 的 tcache（per-thread cache）有空闲链表——double-free 导致同一个块在空闲链表中出现两次。下一次 `malloc` 从链表取走这个块分配给线程 A，下下次再将同一块分配给线程 B——两个线程持有同一个物理地址，互相覆写。

**防御**：
1. `free(p); p = NULL;` — 简单但最有效
2. `-fsanitize=address`（ASan）：检测双重释放并显示两个 free 位置
3. glibc 2.29+ 在 tcache 中加入简单的 key 检查（检测同一个 chunk 是否已在空闲列表中），但仅对最明显的案例有效

## 3. 编译期加固：生产基线配置

> **适用前提**：以下配置适用于 Linux 用户态 C 项目（嵌入式/bare-metal 项目需调整）。8 个选项分 4 组：优化、边界检查、栈保护、地址随机化与指针保护。

```bash
# 1. 优化级别
-O2                                      # 标准优化（D_FORTIFY_SOURCE 需要至少 -O1）

# 2. libc 边界检查
-D_FORTIFY_SOURCE=2                      # 检测 memcpy/sprintf/strcpy 等越界写+读（需 -O1+）

# 3. 栈保护
-fstack-protector-strong                 # 栈金丝雀—比 -fstack-protector 覆盖更全（保护局部数组和栈地址引用）

# 4-5. ASLR
-fPIE -pie                               # 位置无关可执行文件—地址空间布局随机化

# 6-7. 指针保护（Full RELRO）
-Wl,-z,relro,-z,now                      # Full RELRO：GOT 和 GOT.PLT 均只读
                                         # 代价：启动时解析所有动态符号（延迟绑定禁用），启动时间增加

# 8. 栈不可执行（Linux 用户态）
-Wl,-z,noexecstack                       # NX（No-eXecute）— 栈不可执行，防 shellcode
                                         # macOS 自 10.5 起默认开启，Windows 用 /NXCOMPAT

# 8. 控制流完整性
-fcf-protection                          # Intel CET—仅 Intel Tiger Lake 11th Gen+ / AMD Zen 3+
                                         # 旧 CPU 上触发 SIGILL！部署前确认目标平台硬件支持
```

**每个选项的"为什么需要"**：

| 选项 | 防御的漏洞 | 不开启的代价 |
|---|---|---|
| `-D_FORTIFY_SOURCE=2` | buffer overflow + **read overrun** | **Heartbleed 是 read overrun（CWE-125），注意区分** — buffer overflow 是越界写入，read overrun 是越界读取。Heartbleed 中 OpenSSL 使用自身分配器（`OPENSSL_malloc`）而非 glibc malloc，glibc fortify 机制无法介入 |
| `-fstack-protector-strong` | 栈 buffer overflow | Morris 蠕虫的栈返回地址覆写无法被检测 |
| `-fPIE -pie` | 代码重用攻击（ROP） | 栈溢出后跳转到固定地址的 libc 函数仍然可行 |
| `-Wl,-z,relro,-z,now` | GOT 覆写 | 攻击者可通过 buffer overflow 修改 GOT 中的 `free` 地址指向 `system()` |
| `-Wl,-z,noexecstack` | 栈代码执行 | 将 shellcode 放在栈上并跳转执行 |

## 4. 运行时检测：ASan / UBSan 与 Valgrind

### 4.1 Sanitizer 家族

| 工具 | 检测目标 | 编译选项 | 性能开销 | 生产可用？ |
|---|---|---|---|---|
| ASan（AddressSanitizer） | UAF / 缓冲区溢出 / double-free / 栈越界 | `-fsanitize=address` | ~2x | ❌（性能代价太高） |
| UBSan（UndefinedBehaviorSanitizer） | 有符号溢出 / 空指针 / 非法 shift 等 | `-fsanitize=undefined` | 几乎免费 | ✅（近乎零开销） |
| TSan（ThreadSanitizer） | 数据竞争 | `-fsanitize=thread` | 5-15x | ❌（性能代价太高） |
| LeakSanitizer | 内存泄漏 | 随 ASan 自动开启 | 几乎免费 | ✅（可作为 ASan 子集） |

### 4.2 ASan 示例

> **前提**：Ubuntu 需安装 `libasan`。执行 `sudo apt install libasan8` 或编译时加 `-static-libasan`。

```bash
cat > uaf.c << 'EOF'
#include <stdio.h>
#include <stdlib.h>
int main() {
    int *p = malloc(sizeof(int));
    *p = 42;
    free(p);
    printf("%d\n", *p);   // UAF!
    return 0;
}
EOF
gcc -g -fsanitize=address uaf.c -o uaf
./uaf
# ASan 输出：报告 UAF 的发生行、free 的行、堆追踪——完整的 bug 上下文
```

### 4.3 Valgrind

Valgrind 模拟 CPU 执行（非编译插桩——不需要修改源码），10-20x 性能开销，但检测范围广（未初始化读、内存泄漏、竞争条件）。不适合日常开发，但适合 CI 中定期运行。

```bash
gcc -g -O0 uaf.c -o uaf
valgrind --leak-check=full ./uaf
# 报告：Invalid read of size 4 at <UAF line> — Address ... is 0 bytes inside a block of size 4 free'd
```

### 4.4 生产环境替代方案（低开销 Profiler）

| 工具 | 检测目标 | 开销 |
|---|---|---|
| jemalloc profiling | 内存分配统计 | ~5% |
| tcmalloc heap profiler | 内存分配统计 | ~5% |
| perf record | CPU profiling | <1% |
| eBPF / bpftrace | 系统级追踪 | <1% |

## 5. 最佳实践总结

### 5.1 代码层面

1. **永远用 `size_t` 做大小和索引**，不用 `int`（避免溢出和 sign 比较）
2. **永远用 `<stdint.h>` 的精确宽度类型**处理协议/文件格式数据
3. `free(p); p = NULL;` — 免费获得 UAF 防御（对单一所有权场景）
4. malloc 前检查溢出：`if (count > SIZE_MAX / sizeof(*ptr)) return NULL;`
5. 所有用户输入先验证再使用 — 不信任任何外部数据
6. 错误路径必须处理 — `malloc` 返回 NULL、`open` 返回 -1 等所有可能失败的调用

### 5.2 编译层面

```bash
# 至少开着这些（任何项目，无一例外）
gcc -O2 \
    -D_FORTIFY_SOURCE=2 \
    -fstack-protector-strong \
    -fPIE -pie \
    -Wl,-z,relro,-z,now \
    -Wl,-z,noexecstack \
    -Wall -Wextra -Wpedantic -Werror \
    -Wformat=2 -Wconversion -Wsign-compare \
    source.c -o program
```

### 5.3 CI 层面

- 每个提交在 CI 中运行 ASan + UBSan 变体
- 定期运行 Valgrind（至少 release 前）
- 集成静态分析：clang-tidy / cppcheck / Coverity
- 集成 Fuzzing：libFuzzer / AFL（对解析输入的函数）

## 6. 总结

C 安全编程的本质不是"遵循什么规则"——而是**理解每一行代码在系统层面的后果，并将这个后果对齐到安全目标**。

| 漏洞类型 | 根因 | 后果 | 一级防御 |
|---|---|---|---|
| 缓冲区溢出 | `strcpy` 不检查边界 | 栈返回地址覆写 → 代码注入 | `-D_FORTIFY_SOURCE=2` + `-fstack-protector-strong` |
| 格式化字符串 | 用户输入当格式串 | 栈泄露 + 内存写入 | 永远写 `printf("%s", str)` + `-Wformat-security` |
| 整数溢出 | 有符号溢出 = UB / 乘法回绕 | malloc 分配不足 | 分配前检查 `count > SIZE_MAX/sizeof` + UBSan |
| UAF | free 不修改指针值 | 另一线程分配到同一内存 → 交叉 | `p = NULL` after free + ASan |
| Double-Free | 同一块释放两次 | 同一物理地址分给两个指针 | `p = NULL` after free + ASan |

**核心观点**：C 的"程序员至上"哲学是武器也是伤口。技术专家不是"不用这些危险函数"，而是"知道每一个危险函数的防御手段，并将它们作为编译配置永久固化在项目中"。
