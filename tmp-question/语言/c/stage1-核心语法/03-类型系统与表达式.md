> **对应 Layer**：Layer 2（类型系统与表达式），参考 C-C++技术专家学习路线.md § Layer 2  
> **参考书章节**：极致C 第1章（基本特性）；Fluent C 第3章（指针与值）  
> **主线标准**：C11/C17（本章不涉及 C23 新特性如 `nullptr`/`typeof`/`_BitInt`，这些在生产代码中尚未广泛使用）

## 1. 引言：类型系统的两重面孔

Java 的类型系统干净、一致、安全：`int` 是 32 位有符号整数，在所有 JVM 上都一样；`String` 是不可变对象，基于 Unicode；基本类型自动装箱、泛型提供编译时类型检查。C 的类型系统完全不同。

C 的类型系统看似简单——基本类型（`char` / `short` / `int` / `long` / `float` / `double`）加上派生类型（指针 / 数组 / 函数 / struct）。但在这个简单外表下，藏着两个会咬人的陷阱：

1. **整型提升（Integer Promotion）**：`char` 和 `short` 在表达式中被自动提升为 `int`，出现隐式类型转换
2. **有符号/无符号混用**：`int`（有符号）和 `size_t`（无符号）混用时，标准规定的转换方向常常和直觉相反

这两个陷阱导致了 C 代码中最隐蔽的 bug——不是编译错误、不是运行时崩溃，而是计算出了错误的值，只有检查结果才知道错了。本章的目标是让你理解类型系统的这两重面孔，以及如何在实际代码中避免它们的坑。

## 2. 整型提升（Integer Promotion）

### 2.1 根本规则

C 规定：表达式中任何**小于 `int` 的整数类型**（`char`、`signed char`、`unsigned char`、`short`、`unsigned short`）在参与算术运算或比较之前，会自动提升为 `int`（或 `unsigned int`，如果 `int` 装不下原始类型的所有值）。

这个规则是 C89 标准化的，原因是 1980 年代的 CPU 只能高效处理 16/32 位整数值——把 8 位的 `char` 提升为 16 位的 `int` 再做运算，比直接对 8 位值做运算要快。今天这个规则仍然存在，不是因为 CPU 的限制，而是因为改变它会破坏 30+ 年的 C 代码。

### 2.2 标准案例

> **警告**：`char` 是否有符号是**实现定义**的（C11 §6.2.5p15）。x86 平台（GCC/Clang）默认 `char` = `signed char`，但 ARM 平台（包括 Apple Silicon、大多数嵌入式 MCU）默认 `char` = `unsigned char`。以下示例以 x86 平台（signed char）为前提，**生产代码中应显式用 `signed char` 或 `unsigned char`，不依赖 `char` 的默认符号**。

```c
char a = 0x80;     // x86上: 有符号 char，值 = -128。ARM上: 无符号 char，值 = 128
char b = 0x7F;     // 有符号或无符号都不影响，值 = 127
int  c = a + b;    // x86: int(-128) + int(127) = -1 ✓
                   // ARM: int(128) + int(127) = 255（不是 -1！）
```

表面正确——但这是巧合。有符号 char 的提升保持了符号（sign extension），所以 -128 提升后仍是 -128。

### 2.3 一个会出错的例子

```c
unsigned char ua = 0x80;    // 128
unsigned char ub = 0x80;    // 128
unsigned char uc = ua + ub; // 表面上 128 + 128 = 256 → 截断为 unsigned char → 0

// 实际发生的事：
// 1. ua 提升为 int (128)，ub 提升为 int (128)
// 2. 相加 = 256（int，没有溢出——因为 int 至少是 16 位）
// 3. 赋值回 unsigned char uc，256 截断为 0（256 % 256 = 0）✓
```

这里没问题——但注意结果是 `int` 而不是 `unsigned char`。这在更大的上下文中会变得重要。

### 2.4 真正的坑：位运算中的提升

这是 C 中最隐蔽的 bug 之一：

```c
unsigned char flag = 0xFF;          // 255
if (~flag == 0xFE) { ... }          // 结果？

// 实际发生的事：
// 1. flag 提升为 int (255)，即 0x000000FF
// 2. ~flag = 0xFFFFFF00（按位取反整个 int）
// 3. 0xFFFFFF00 != 0xFE，条件为假！
```

解决办法：将结果显式截断为原来的类型：

```c
if ((unsigned char)~flag == 0xFE) { ... }  // ✓
```

### 2.5 自测

```c
unsigned char x = 0xFF;
printf("%d\n", x << 24);   // 你觉得输出多少？
```

答案：`x`（`unsigned char`）先被整型提升为 `int`（255），255 << 24 = 4278190080，超过 INT_MAX(2147483647)，触发有符号左移溢出 = **UB**（Undefined Behavior，未定义行为——整个程序没有保证，编译器可执行任意操作）。正确写法：`(unsigned int)x << 24`（先转无符号避免溢出），或 `0xFFu << 24`。

## 3. 有符号与无符号

### 3.1 混合运算的转换规则

**注意**：上述规则要求两操作数的整数转换 rank 相同（如 `int` vs `unsigned int`）。当 `unsigned short` 和 `int` 混用时，`unsigned short` 先被整型提升为 `int`（如果 int 能装下），最终两边都是 `int`，**不触发**有符号转无符号。

```c
int a = -1;
unsigned int b = 1;
if (a < b) { ... }   // 结果？

// a 被转换为 unsigned int（4294967295），4294967295 > 1，条件为假
```

这个陷阱在生产代码中几乎无法通过阅读发现——你需要对每个比较做"这个变量可能为负吗？另一个是无符号吗？"的检查。`-Wsign-compare`（GCC/Clang）会警告这种情况，**生产代码必须开启**。

### 3.2 size_t——无符号的存在哲学

`size_t` 是 C 标准定义的无符号整数类型（通常 `unsigned long` 或 `unsigned long long`，64 位平台 8 字节）。用于表示对象大小、数组索引、内存计数等。

**为什么是无符号？**
- 对象大小不可能为负——作为"大小"概念，无符号是有意义的选择
- 64 位下最大可寻址内存为 2^64 - 1 字节（约 18 EB），有符号只有 2^63 - 1（约 9 EB）——无符号给了两倍的地址范围

### 3.3 size_t 的循环陷阱

```c
// 倒序遍历数组——这个版本有 bug：
for (size_t i = n - 1; i >= 0; i--) { ... }   // 死循环！

// 为什么？i 是无符号——当 i = 0 时，i-- 变成 4294967295（64位：18446744073709551615）
// 条件 i >= 0 永远为真
```

正确的写法：

```c
// 版本 1：在递减前检查
for (size_t i = n; i-- > 0; ) { ... }

// 版本 2：用有符号索引
for (ssize_t i = n - 1; i >= 0; i--) { ... }   // ssize_t 有符号版本（POSIX 扩展，MSVC 不提供；跨平台首选 ptrdiff_t）

// 版本 3：用 ptrdiff_t（标准 C，<stddef.h> — 跨平台首选）
for (ptrdiff_t i = n - 1; i >= 0; i--) { ... } // 标准 C 的有符号索引
```

### 3.4 生产建议

1. 用 `size_t` 表示大小和索引——不要用 `int`（避免溢出和 sign 比较）
2. 倒序遍历用 `ssize_t` 或 `for (size_t i = n; i-- > 0;)` 模式
3. 启用 `-Wsign-compare -Wconversion` 编译器警告，把每个警告当错误处理
4. 网络协议 / 文件格式代码中用精确宽度类型（`uint32_t` / `int64_t`），不用 `int`

## 4. 位运算：C 的独门武器

### 4.1 为什么 C 工程师需要位运算？

位运算直接操作寄存器和标志位——这是网络协议解析、加密算法、压缩、序列化、硬件驱动、嵌入式固件的核心操作。Java 也有位运算——但大多 Java 工程师不做这些、也不需要理解物理层的限制。C 工程师必须理解。

### 4.2 常用模式

**flags 管理**：

```c
#define READ   (1u << 0)    // unsigned 后缀避免有符号溢出
#define WRITE  (1u << 1)
#define EXEC   (1u << 2)

int perm = READ | WRITE;           // 设置多个位
if (perm & READ) { ... }           // 测试
perm &= ~WRITE;                     // 清除
perm ^= EXEC;                       // 翻转
```

**提取位域**：

```c
uint32_t x = 0xDEADBEEF;
uint8_t byte1 = (x >> 24) & 0xFF;   // 0xDE
uint8_t byte2 = (x >> 16) & 0xFF;   // 0xAD
```

**边界对齐**：

```c
size_t align_up(size_t n, size_t alignment) {
    // alignment 必须是 2 的幂
    return (n + alignment - 1) & ~(alignment - 1);
}
// align_up(37, 16) = 48   37 → 最近大于等于的 16 倍数
```

**2 的幂检查**：

```c
bool is_power_of_2(unsigned int n) {
    return n && !(n & (n - 1));
}
// n = 8: 8 & 7 = 1000 & 0111 = 0 → true ✓
// n = 6: 6 & 5 = 0110 & 0101 = 0100 → false ✓
```

### 4.3 位域（bitfield）——仅用于节省内存，不用于网络协议

```c
struct Flags {
    unsigned int is_ready : 1;
    unsigned int mode     : 2;    // 可存 0-3
    unsigned int reserved : 5;
};
// sizeof(struct Flags) = 1 x sizeof(unsigned int) = 4 字节
// 三个字段打包在一个 int 里
```

**关键限制**：位域的内存布局是**实现定义**的——`mode` 是占 int 的低位还是高位，不同编译器不同处理。这意味着你不能用位域解析网络协议数据——网络协议必须用显式的移位和掩码，这也是大多数网络层代码不出现位域的原因。

### 4.4 位运算实验

```bash
cat > bitops.c << 'EOF'
#include <stdio.h>
#include <stdint.h>
int main() {
    // flags 管理
    enum { READ=1, WRITE=2, EXEC=4 };
    int p = READ | WRITE;
    printf("READ|WRITE = %d (expect 3)\n", p);

    // 提取位域
    uint32_t v = 0xDEADBEEF;
    printf("byte 3 = 0x%02X (expect DE)\n", (v >> 24) & 0xFF);
    printf("byte 2 = 0x%02X (expect AD)\n", (v >> 16) & 0xFF);

    // 对齐
    printf("align 37 to 16 = %zu (expect 48)\n",
           ((size_t)37 + 15) & ~(size_t)15);
    return 0;
}
EOF
gcc bitops.c -o bitops && ./bitops
```

## 5. 函数指针：回调与表驱动

### 5.1 为什么需要函数指针？

C 没有多态的语法支持——`struct` 不能动态绑定方法。函数指针是在编译时为 C 提供"运行时行为绑定"的唯一机制。

### 5.2 回调模式

```c
// 排序——qsort 标准库模式
int cmp_int(const void *a, const void *b) {
    return *(const int *)a - *(const int *)b;
}
int arr[] = {3, 1, 4, 1, 5, 9};
qsort(arr, 6, sizeof(int), cmp_int);

// 通用回调——自定义过滤函数
void filter(int *in, int n, int *out, int *nout,
            int (*pred)(int)) {
    *nout = 0;
    for (int i = 0; i < n; i++) {
        if (pred(in[i])) out[(*nout)++] = in[i];
    }
}
```

### 5.3 表驱动编程

```c
typedef void (*handler_t)(request_t *);

handler_t handlers[] = {
    handle_get,
    handle_post,
    handle_put,
    handle_delete
};

// 根据 method 分派——一行映射替代 switch/if-else 链
handlers[req.method](req);
```

### 5.4 声明技巧

C 的函数指针声明对新手来说难以解析。使用 typedef 让声明变得可读：

```c
// 原始声明——难以阅读
void (*signal(int sig, void (*handler)(int)))(int);

// 用 typedef 拆解
typedef void (*sig_handler_t)(int);
sig_handler_t old_handler = signal(SIGINT, new_handler);
```

## 6. C 的受限泛型

### 6.1 为什么没有完整的泛型？

C 没有完整的泛型系统。这不是设计缺陷——完整泛型系统要么需要类型擦除（Java）要么需要代码膨胀（C++/Rust monomorphization）。C 的目标平台包括 64KB RAM 的 MCU（微控制器），这两种都不是可行的选择。

### 6.2 泛型的三种 C 方式

**方式 1：`void *` + 函数指针**（最常用、最不安全）：

```c
// 通用 linked list
typedef struct node {
    void *data;
    struct node *next;
} node_t;

void list_foreach(node_t *head, void (*fn)(void *)) {
    for (node_t *n = head; n; n = n->next) fn(n->data);
}
```

**方式 2：宏**（编译时类型替换，但难调试）：

```c
#define DEFINE_VECTOR(T, name)              \
    typedef struct {                        \
        T *data;                            \
        size_t size, capacity;              \
    } name;                                 \
    void name##_push(name *v, T val) { ... }

DEFINE_VECTOR(int, IntVec);
DEFINE_VECTOR(char*, StrVec);
// 展开为两个完整的类型和函数定义
```

**方式 3：`_Generic`（C11 引入，编译时类型分选择 — 有限但类型安全）**：

```c
#define print_val(x) _Generic((x),          \
    int:    printf("int: %d\n", x),          \
    double: printf("double: %f\n", x),      \
    char*:  printf("str: %s\n", x),         \
    default: printf("unknown type\n")       \
)

print_val(42);      // 输出 "int: 42"
print_val("hello"); // 输出 "str: hello"
```

### 6.3 `_Generic` 的限制

`_Generic` 只是编译时类型分选择——它不能创建新的类型，不能做运算符重载，不能模拟 C++ 模板或 Java 泛型。它只能让你在调用时根据参数的实际类型选择不同的实现函数。大多数 C 项目仍用 `void *` + 函数指针。

{% note %}
**生产建议**：能用 `_Generic` 就用 `_Generic`（编译时类型安全 + 零运行时开销）。退而求其次用宏。最后一招用 `void *`——至少加上运行时类型标记（如 enum 标记 + assert）。`void *` 没有类型信息，这是 libc qsort 的核心设计缺陷——把类型错误暴发给运行时而不是编译时。
{% endnote %}

## 7. 表达式与副作用：序列点

### 7.1 什么是序列点？

序列点（sequence point）是程序中"所有之前的副作用已经完成、之后的副作用还没开始"的点。在之前的序列点和之后的序列点之间，同一个对象最多被修改一次。

| 序列点 | 例子 |
|---|---|
| 每个完整表达式末尾的分号 | `x++;` 后 |
| `&&` / `||` / `?:` / `,` 的第一操作数之后 | `a && b`：计算完 a 之后、b 之前 |
| 函数实参全部求值后、函数进入之前 | `foo(a++, b--)`：两个传参的增量/减量都已完成、foo 执行前 |

### 7.2 序列点违规

```c
// 这两行在两个序列点之间修改 x 两次 → UB
int x = 0;
int y = x++ + x++;      // UB!

// 读取和修改同一次序点之间 → UB
int a[10];
int i = 0;
a[i++] = i;             // UB!
```

### 7.3 实验

```bash
cat > seqpt.c << 'EOF'
#include <stdio.h>
int main() {
    int x = 0;
    int y = x++ + x++;   // UB — 同一对象在无序列点间隔内被修改两次
    // 一旦 UB 发生，整个程序没有任何保证——x 不保证是 2，
    // 编译器可消除这段代码、令后续代码崩溃、产生任意输出
    printf("x=%d y=%d\n", x, y);
    return 0;
}
EOF
# 不同编译器和优化级别输出完全不同——
# 这就是为什么"我跑过了"不能证明正确
gcc -O0 -Wall seqpt.c -o seqpt && ./seqpt
gcc -O2 -Wall seqpt.c -o seqpt && ./seqpt
```

## 8. 总结：类型系统的三重核心

1. **整型提升**：`char` / `short` 在表达式中自动提升为 `int`——位运算后不截断会得到错误结果。解决方案：运算后显式转换回原类型
2. **有符号与无符号**：混用时标准规则是"有符号转无符号"——直觉相反。解决方案：启用 `-Wsign-compare -Wconversion`，把每个警告当错误处理；倒序循环用 `ssize_t` 或 `for (size_t i = n; i-- > 0;)`
3. **位运算与函数指针**：C 提供了直接的内存级操作能力——理解位域的内存布局限制（实现定义、只能用于内存打包不能用于网络协议）、理解函数指针的 typedef 技巧和 `_Generic` 的编译时类型安全限制
