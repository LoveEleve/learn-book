## 2. 宏：强大的文本替换器（但不是编译器）

### 2.1 三个经典陷阱

预处理器只做文本替换——不知道优先级、不知道参数类型、不知道副作用。这三个陷阱覆盖了 90% 以上的宏 bug。

**陷阱 1：优先级**

```c
#define DOUBLE(x) x + x

int y = DOUBLE(3) * 2;
// 展开为：3 + 3 * 2 = 9（不是 12！）
// 乘法优先级更高 → 3 + (3 * 2) = 9
```

**正确写法**：`#define DOUBLE(x) ((x) + (x))` —— 每个参数加括号，整体也加括号。

**陷阱 2：副作用**

```c
#define SQUARE(x) ((x) * (x))

int i = 3;
int y = SQUARE(i++);
// 展开为：((i++) * (i++))
// i 自增两次，编译器不知道哪个先执行 → UB（Undefined Behavior，未定义行为）
```

**为什么无法修复**：无论怎么给 `x` 加括号，`i++` 的副作用都会在宏展开中产生两次。这是宏的固有问题——唯一避免的方法是**不把有副作用的表达式传给宏**。

**陷阱 3：字符串化与拼接**

```c
#define STR(x) #x          // # 将参数转为字符串
#define CAT(a, b) a ## b   // ## 将两个符号拼接为一个符号

STR(hello)    → "hello"
CAT(foo, bar) → foobar     // 如果 foobar 是一个已定义的函数/变量名
```

**生产用例**：`#` 常用于日志宏（`LOG(fmt, ...) printf("[%s:%d] " fmt, __FILE__, __LINE__, ##__VA_ARGS__)`）。`##` 常用于泛型容器生成（`DEFINE_VECTOR(int) → IntVec_push`）。

### 2.2 宏 vs 内联函数

| 维度 | 宏 | `static inline` 函数 |
|---|---|---|
| 类型检查 | 无（纯文本替换） | 有（编译器做类型检查） |
| 调试 | 难（GDB 无法在宏内部设断点） | 易（正常断点和单步调试） |
| 副作用 | 严重（参数被多次求值） | 无（参数只求值一次，然后传值） |
| 代码膨胀 | 每次展开产生副本 | 编译器决定是否内联（`inline` 是建议） |
| 泛型 | 支持（通过 `#`/`##` 和 `_Generic`） | 不支持（C 没有函数重载） |

**生产建议**：能用 `static inline` 就用 `static inline`。只有在需要泛型（多种类型的容器）或需要 `#`/`##` 操作符时才用宏。

### 2.3 预处理器实验

```bash
cat > macro_test.c << 'EOF'
#include <stdio.h>

// 陷阱 1：优先级
#define BAD_DOUBLE(x) x + x
#define OK_DOUBLE(x)  ((x) + (x))

// 陷阱 2：副作用
#define SQUARE(x) ((x) * (x))

// 陷阱 3：特殊操作符
#define STR(x)  #x
#define CAT(a,b) a ## b

int CAT(my_, function)(void) {   // 展开为 my_function
    return 42;
}

int main() {
    printf("BAD_DOUBLE(3)*2 = %d (expect 9, not 12)\n", BAD_DOUBLE(3) * 2);
    printf("OK_DOUBLE(3)*2  = %d (expect 12)\n", OK_DOUBLE(3) * 2);

    int i = 3;
    printf("SQUARE(i++) = %d, i = %d (i 变了两次——UB!)\n",
           SQUARE(i++), i);

    printf("STR(hello) = %s\n", STR(hello));
    printf("my_function() = %d\n", my_function());
    return 0;
}
EOF
gcc -Wall macro_test.c -o macro_test && ./macro_test
```

### 2.4 条件编译：预处理器最强大的功能

条件编译（`#ifdef`/`#if`/`#elif`/`#else`/`#endif`）是跨平台 C 代码的基础。Java 用反射和配置文件做条件行为；C 用预处理器。

**调试开关**：

```c
#ifdef DEBUG
    printf("x = %d, line = %d\n", x, __LINE__);
#endif
// 编译时：gcc -DDEBUG → 输出调试信息；不传 -DDEBUG → 静默
```

**平台适配**：

```c
#ifdef _WIN32
    #include <windows.h>
    #define SLEEP_MS(ms) Sleep(ms)
#elif defined(__linux__)
    #include <unistd.h>
    #define SLEEP_MS(ms) usleep((ms) * 1000)
#else
    #error "Unsupported platform"
#endif
```

**特性检测**（检测编译器 C 标准版本）：

```c
#if __STDC_VERSION__ >= 201112L
    // C11+：可以用 _Generic / <stdatomic.h>
#elif __STDC_VERSION__ >= 199901L
    // C99：可以用 // 注释 / <stdint.h>
#else
    #error "Requires at least C99"
#endif
```

**断言控制**：

```c
#include <assert.h>
// assert(expr) 只在 NDEBUG 未定义时生效
// Release 构建：gcc -DNDEBUG → 所有 assert 被预处理器移除
```

### 2.5 工程化宏模式

**`do { ... } while(0)` — 多语句宏的标准写法**：

```c
// 错误写法——在 if-else 中悬挂
#define SWAP(a, b)  int tmp = a; a = b; b = tmp;
if (cond)
    SWAP(x, y);     // 展开后只有 int tmp = a; 在 if 内！
else
    foo();

// 正确写法——永远安全
#define SWAP(a, b) do { int tmp = a; a = b; b = tmp; } while(0)
```

**X-macros — 数据驱动的代码生成**：

```c
// 定义一个"数据表"（不参与编译，仅被 #include 展开）
// colors.def:
//   X(RED,   0xFF0000)
//   X(GREEN, 0x00FF00)
//   X(BLUE,  0x0000FF)

#define X(name, value) name = value,
enum Color { 
    #include "colors.def"
};   // → RED = 0xFF0000, GREEN = 0x00FF00, BLUE = 0x0000FF

#undef X
#define X(name, value) [name] = #name,
const char *color_names[] = {
    #include "colors.def"
};   // → ["RED", "GREEN", "BLUE"]
```

### 2.6 预定义宏与诊断宏

| 宏 | 含义 | 典型用途 |
|---|---|---|
| `__FILE__` | 当前源文件名（字符串） | 错误日志携带位置 |
| `__LINE__` | 当前行号（整数） | 同上 |
| `__func__` | 当前函数名（C99，不同于 GCC 扩展 `__FUNCTION__`） | 函数级诊断 |
| `__DATE__` / `__TIME__` | 编译时间 | 版本信息嵌入 |
| `__STDC_VERSION__` | C 标准版本（C94=199409L, C99=199901L, C11=201112L, C17=201710L） | 条件编译特性检测 |

**`#error` — 编译时强制报错**：

```c
#if !defined(__STDC_VERSION__) || __STDC_VERSION__ < 201112L
    #error "This project requires at least C11"
#endif
```

**`#warning` — 编译时警告**（GCC/Clang 支持，非标准 C）：

```c
#if !defined(USE_SAFE_ALLOCATOR)
    #warning "Using default allocator — consider -DUSE_SAFE_ALLOCATOR"
#endif
```

**`-dM -E` — 查看所有预定义宏**（调试预处理的必备命令）：

```bash
gcc -dM -E - < /dev/null | head -20
# 输出：
# #define __STDC_VERSION__ 201710L
# #define __x86_64__ 1
# #define __linux__ 1
# ...
```

## 3. 头文件：接口与实现的边界

### 3.1 Include Guard（头文件保护）

防止同一个头文件在一个翻译单元中被包含两次：

```c
// foo.h — 传统方式
#ifndef FOO_H
#define FOO_H
// ... 头文件内容 ...
#endif
```

或：

```c
// foo.h — 更简洁的现代方式
#pragma once
```

**`#ifndef` vs `#pragma once`**：

| 维度 | `#ifndef` | `#pragma once` |
|---|---|---|
| 标准性 | C89 起标准支持 | 非标准（但 GCC/Clang/MSVC 全部支持） |
| 保护范围 | 基于唯一的宏名 | 基于文件路径/文件系统 inode |
| 跨文件系统 | 安全 | 软链接/硬链接可能误判为同一文件 |
| 名字冲突 | 手动选择唯一的宏名 | 无（不需要名字） |

**生产建议**：`#pragma once` 在所有主流编译器上可用且更简洁，但跨平台开源项目应同时提供两种保护（`#ifndef` + `#pragma once`）。

### 3.2 Include 顺序

标准化的 include 顺序可以避免隐式依赖和顺序敏感 bug：

```
#include "foo.h"        // 1. 对应的头文件（自给自足检查）
#include <stdio.h>      // 2. 标准库
#include <libxml/foo.h> // 3. 第三方库
#include "project/bar.h" // 4. 项目内其他头文件
```

**为什么 1 号位（自给自足检查）最重要**：如果 `foo.h` 需要 `<stddef.h>` 中的 `size_t` 但不自己 include，`foo.c` 的 include 顺序会掩盖问题——只有 `foo.h` 在其他 `.c` 中被独立 include 时才会暴露。把对应的头文件放在首位，能在编译时立即发现头文件是否自给自足。

### 3.3 头文件中应该放什么

| 应该放 | 不应该放 |
|---|---|
| 函数声明（`void foo(int);`） | 函数**定义**（`void foo(int) { ... }` — 多次定义 = 链接错误） |
| `extern` 变量声明 | 变量**定义**（`int x;` — 每个包含的 .c 都产生符号） |
| 类型定义（`typedef` / `struct` / `enum`） | 静态数据的大块定义 |
| `static inline` 函数（每个 TU 一份副本） | 非 inline 函数定义 |
| 宏 | 裸的 `using` 或项目配置代码 |

### 3.4 避免循环包含

```c
// a.h — 包含 b.h
#include "b.h"

// b.h — 包含 a.h（循环！）
#include "a.h"   // a.h 的 include guard 防止无限递归，但语义混乱
```

**解决方案**：前向声明 + 在 .c 中 include。

```c
// a.h
typedef struct b_t b_t;   // 前向声明——不需要知道 b_t 的成员
void foo(b_t *b);         // 用指针参数不需要完整类型

// a.c
#include "a.h"
#include "b.h"            // 这里才需要完整类型
void foo(b_t *b) { ... }
```

### 3.5 隐式依赖检测

当你 `#include "foo.h"` 时，你不应该需要手动 include `"bar.h"` 才能使用 `foo.h` 中的声明。这是"头文件自给自足"原则——将对应的头文件放在 include 列表首位可以在编译时立即暴露问题。生产环境中可以用 IWYU（Include What You Use，Google 的 clang 基工具）自动检测隐式依赖。

## 4. 编译单元（Translation Unit / TU）

### 4.1 什么是编译单元

一个编译单元（TU）是一个 `.c` 文件加上它 `#include` 的所有头文件展开后的完整内容。编译器一次处理一个 TU，**看不到**其他 TU 的代码——这就是 `static` 和 `inline` 可见性规则的基础。

```
foo.c = foo.c + foo.h 展开 + <stdio.h> 展开 + <stdlib.h> 展开
bar.c = bar.c + bar.h 展开 + <stdio.h> 展开
```

这两个 TU 被**独立**编译。编译器不知道 `foo.c` 中定义了什么函数，也不知道 `bar.c` 中声明了什么类型——它们只通过头文件了解彼此的存在。

### 4.2 `static`：TU 内可见

```c
// foo.c
static int private_counter = 0;     // 仅 foo.o 可见
static void private_helper(void) {  // 仅 foo.o 可调用
    private_counter++;
}

// bar.c — 即使声明一个同名的 private_counter，也是完全不同的变量
static int private_counter = 0;     // 独立！与 foo.c 的版本无关
```

**`static` 的两种含义**：
1. **函数/全局变量中的 `static`**：只在当前 TU 可见。= "私有"（private）���—这是 C 的模块化核心机制
2. **局部变量中的 `static`**：在函数调用之间保持值，但仅当前函数可见。= "持久局部变量"

### 4.3 `inline`：建议编译器内联

```c
// foo.h
static inline int add(int a, int b) {
    return a + b;
}
```

**`inline` 的微妙之处**：
- C 标准中的 `inline` 是**建议**（编译器可以忽略），`static inline` 保证每个 TU 有自己的副本（避免链接时多重定义）
- 不加 `static` 的 `inline` 函数需要在一个 `.c` 中提供外部定义——否则链接时可能找不到符号
- `gcc -O2` 会自动内联简单的 static 函数，即使没有 `inline` 关键字

### 4.4 TU 实验

```bash
cat > tu_a.c << 'EOF'
#include <stdio.h>
static int counter = 0;      // tu_a 的私有 counter
void inc_a(void) { counter++; printf("a: %d\n", counter); }
EOF

cat > tu_b.c << 'EOF'
#include <stdio.h>
static int counter = 100;    // tu_b 的私有 counter（独立的！）
void inc_b(void) { counter++; printf("b: %d\n", counter); }
EOF

cat > tu_main.c << 'EOF'
void inc_a(void);   // 声明（非 static → 可以被外部访问）
void inc_b(void);   // 声明（在 tu_b.c 中有 static 版本，这里只声明）
int main() {
    inc_a(); inc_a();   // "a: 1"  "a: 2"
    inc_b(); inc_b();   // "b: 101"  "b: 102"
    return 0;
}
EOF

gcc -Wall tu_main.c tu_a.c tu_b.c -o tu_test && ./tu_test
# 证明：tu_a 和 tu_b 中的 static counter 是完全独立的变量
```

## 5. 总结

| 概念 | 一句话 |
|---|---|
| 预处理器 | 只做文本替换——不知道类型、优先级、副作用 |
| 宏 | 优先级问题（加括号）+ 副作用问题（避免宏参数有 `++`/`--`）→ `static inline` 优先，泛型场景才用宏 |
| 条件编译 | `#ifdef`/`#if`/`#elif`/`#else`/`#endif` — 平台适配、调试开关、特性检测的基础 |
| 多语句宏 | `do { ... } while(0)` — 在任何上下文中安全的标准写法 |
| 预定义宏 | `__FILE__`/`__LINE__`/`__func__`/`__STDC_VERSION__` — 诊断、条件编译、版本检测 |
| `#error`/`#warning` | 编译时诊断：阻止不兼容配置、提示过期 API |
| Include Guard | `#pragma once` 简洁，`#ifndef` 标准——跨平台项目两者都提供 |
| Include 顺序 | 1. 对应头文件（自给自足检查） / 2. 标准库 / 3. 第三方 / 4. 项目内 |
| 编译单元 | `.c` + 所有 `#include` = TU → 编译器一次处理一个 TU → `static` 函数仅本 TU 可见 |
| `static` | 全局/函数级 = "私有"（TU 内可见）；局部变量 = 持久状态 |
| `inline` | 建议编译器内联 → 用 `static inline` 避免链接多重定义 → 大函数不用 `inline`（避免 I-cache 膨胀） |
| 循环包含 | 前向声明 + 在 `.c` 中 include → 头文件只放声明 |
| IWYU | Include What You Use — Google 的 clang 基工具，自动检测隐式依赖 |
