## 1. 引言：从"未定义引用"到符号表

每个 C 工程师都见过这个错误：

```
undefined reference to `foo'
```

这是链接器在说：我找了所有的 `.o` 文件，没有一个里面有 `foo` 的定义。但链接器怎么知道哪个 `.o` 有 `foo`、哪个没有？答案在**符号表**中。

本章从符号表的视角理解链接——符号是什么、强符号和弱符号的区别、链接器如何解析符号、以及最常见的链接陷阱。理解符号，你才能回答："为什么头文件中放了函数定义会多重定义错误？""为什么两个 `.c` 中的 `int x;` 不会报重复定义错误？""为什么 `-lm` 要放在编译命令的最后？"

## 2. 符号：链接器眼中的名字

### 2.1 什么是符号

符号（Symbol）是链接器对函数和全局变量的内部表示。对于一个 `.o` 文件，链接器只关心三件事：

| 符号类型 | 含义 | 例子 |
|---|---|---|
| **导出符号**（defined/exported） | 这个 `.o` 提供了这个函数/变量的定义 | `main` 在 `main.o` 中是 T（Text section） |
| **未定符号**（undefined） | 这个 `.o` 使用了但未提供这个函数/变量 | `printf` 在 `main.o` 中是 U（Undefined） |
| **局部符号**（local） | `static` 函数/变量 — 链接器看不到它们 | `static int counter` 在 `nm` 中是小写 t |

**实验——直接看符号表**：

```bash
cat > sym_test.c << 'EOF'
#include <stdio.h>
static int private_counter = 0;
int global_counter = 42;
void increment(void) {
    private_counter++;
    global_counter++;
    printf("incremented\n");
}
EOF

gcc -c sym_test.c -o sym_test.o
nm sym_test.o
# 输出解读：
# T increment          — T = 在 .text 节（代码），全局可见
# D global_counter     — D = 在 .data 节（已初始化数据），全局可见
# U printf             — U = 未定义（需要其他 .o 提供）
# b private_counter    — 小写 b = 在 .bss 节，static 局部可见（值为 0 视为 bss）
```

### 2.2 符号的节归属

`nm` 输出的第二列字母告诉你符号所在的节：

| 字母 | 节 | 含义 |
|---|---|---|
| T / t | .text | 代码（大写=全局，小写=局部 static） |
| D / d | .data | 已初始化全局数据 |
| B / b | .bss | 未初始化全局数据（文件中不占空间） |
| R / r | .rodata | 只读数据（const 全局/字符串字面量） |
| U | — | 未定义（需要链接时从其他 .o 解析） |
| W / w | — | 弱符号（Weak symbol — 详见 §3） |

**实验——对比不同节**：

```bash
cat > sections.c << 'EOF'
int global_init = 42;          // .data
int global_uninit;             // .bss
const int ro = 100;            // .rodata

int main(void) {
    static int local_static = 0;  // .bss（值为 0 视为 bss）
    return 0;
}
EOF

gcc -c sections.c -o sections.o
nm sections.o
# 验证：B global_uninit, D global_init, R ro, T main, b local_static
```

## 3. 强符号与弱符号

### 3.1 规则

这是 C 链接模型中最容易出错的机制。符号有强弱之分：

| 符号类型 | 条件 | 例子 |
|---|---|---|
| **强符号**（Strong） | 已初始化的全局变量 或 函数定义 | `int x = 42;` / `void foo(void) { }` |
| **弱符号**（Weak） | 未初始化的全局变量（tentative definition） | `int x;` — 在多个 `.o` 中出现不会报错 |

**链接规则**（当多个 `.o` 中有同名符号时）：
1. **多个强符号同名** → 链接错误（`multiple definition of 'foo'`）
2. **一个强符号 + 多个弱符号** → 选强符号
3. **全部是弱符号** → 选其中任意一个（编译器/链接器自行决定，通常选第一个）

### 3.2 规则 1：多个强符号 → 错误

```c
// a.c
int x = 1;   // 强符号 — 已初始化

// b.c
int x = 2;   // 强符号 — 已初始化（另一个！）
```

编译通过。链接时：
```
gcc -c a.c -o a.o
gcc -c b.c -o b.o
gcc a.o b.o -o prog
# 链接错误: multiple definition of 'x'
```

这就是为什么函数**定义**不能放在头文件中——每个 `#include` 的 `.c` 都会生成一个强符号，链接时冲突。

### 3.3 规则 2：强 + 弱 → 选强

```c
// a.c
int x = 42;   // 强符号—已初始化

// b.c
int x;        // 弱符号—未初始化
```

链接后 `x` 只有一个，值 = 42。规则 2 让两个文件中的 `x` 变成同一个变量——这可以是"共享全局变量"的诡计，也可能是意外的关联。

### 3.4 规则 3：全为弱 → 陷阱

```c
// a.c
int x;   // 弱符号

// b.c
int x;   // 弱符号
```

链接后 `x` 只有一个，初始值 = 0。两个文件共享同一个变量——但如果两个文件各自期望 `x` 是独立的，这就形成了一个难以发现的共享状态 bug。

**这就是为什么未初始化全局变量能在多个 `.c` 中定义而不会在链接时报错**——它们是弱符号，链接器默默地合并它们。

### 3.5 弱符号实验

```bash
cat > weak_a.c << 'EOF'
#include <stdio.h>
int x = 42;              // 强符号
void print_a(void) { printf("a: x=%d\n", x); }
EOF

cat > weak_b.c << 'EOF'
#include <stdio.h>
int x;                   // 弱符号
void print_b(void) { printf("b: x=%d\n", x); }
EOF

cat > weak_main.c << 'EOF'
void print_a(void);
void print_b(void);
int main() {
    print_a();   // "a: x=42"
    print_b();   // "b: x=42" — 同一个 x！
    return 0;
}
EOF
gcc -Wall weak_a.c weak_b.c weak_main.c -o weak_test && ./weak_test
```

## 4. 符号解析：链接器的三步工作

> **注**：本节聚焦符号表层面的解析流程。ELF 文件的内部结构（节头/段头/GOT/PLT/动态链接的懒惰绑定）详见 S2Ch03。符号解析是"链接器做什么"，ELF 是"链接产物长什么样"—两章是同一硬币的两面。

### 4.1 全景

```
阶段 1: 收集
  链接器读取所有 .o 和 .a/.so 的符号表
  → 列出：每个 .o 导出了哪些符号（T/D/B），需要哪些符号���U）

阶段 2: 查找
  对每个 U（未定义符号），在所有 .o 和库中查找对应的 T/D/B
  → 找到 → 好。没找到 → 这些是"undefined reference"的来源

阶段 3: 重定位
  把代码中的"占位符地址"替换为实际地址
  → call printf → call <printf 的实际地址>
    mov x, <x 的实际地址>
```

### 4.2 库查找顺序

这是 `gcc -lm` 必须放在最后的根本原因：

```bash
gcc main.o -lm        # ✓ — 先读 main.o，发现未定义符号（sin, cos 等），在 -lm 中找
gcc -lm main.o        # ✗ — 先处理 -lm，此时还没有未定义符号需要解析，跳过 libm
                      #     然后处理 main.o，发现需要 sin, cos，但已经没有库可查了
```

**规则**：库（`.a`/`.so`）按命令行顺序处理——每个库只为**之前**出现的未定义符号提供解析。这就是为什么：
- 自己的 `.o` 放在最前面
- 项目内的库按依赖顺序（底层库在后面）
- 系统库（`-lm` / `-lpthread`）放在最后

### 4.3 实验

```bash
# 制造 undefined reference 错误
cat > ref.c << 'EOF'
void foo(void);   // 声明但不定义
int main() {
    foo();
    return 0;
}
EOF
gcc ref.c -o ref    # 报错: undefined reference to `foo'
```

## 5. 常见链接陷阱

### 5.1 头文件中的函数定义

```c
// bad.h — 不要这样做
void helper(void) {   // 函数定义在头文件中！
    printf("helper\n");
}

// a.c: #include "bad.h"  → 产生强符号 helper
// b.c: #include "bad.h"  → 产生另一个强符号 helper
// 链接: multiple definition of 'helper'
```

**正确方式**：
```c
// good.h — 声明（无函数体）
void helper(void);

// good.c — 定义（函数体）
#include "good.h"
void helper(void) { printf("helper\n"); }
```

### 5.2 `static inline` 的安全性

```c
// header.h
static inline int add(int a, int b) { return a + b; }
// static + inline 在每个 TU 内产生局部副本，不会出现多重定义
```

### 5.3 两个 C 文件中的同名全局变量

```c
// a.c: int x;   — 弱符号
// b.c: int x;   — 弱符号
// 链接: 两个 x 被合并为一个变量（两个文件共享同一个 x）
// 不是链接错误，但可能是设计 bug
```

**解决方案**：给每个全局变量值（`int x = 0;`），让它成为强符号——这会激化同名冲突，在链接时就暴露问题。

### 5.4 `extern` 的正确用法

```c
// a.c
int counter = 0;     // 定义（强符号）

// b.c
extern int counter;   // 声明——不产生新符号，引用 a.c 的 counter
```

`extern` 不产生符号——它只是告诉编译器"这个变量在其他地方有定义，不要在本 TU 中分配存储"。

## 6. 总结

| 概念 | 一句话 |
|---|---|
| 符号 | 链接器眼中的函数和全局变量名 — 每个 .o 的符号表记录导出/未定义/局部三种符号 |
| 强符号 | 已初始化全局变量 + 函数定义 — 多个强符号同名 = 链接错误 |
| 弱符号 | 未初始化全局变量 — 多个弱符号同名不报错，被链接器合并 |
| 符号解析 | 收集 → 查找 → 重定位 — 库按命令行顺序处理 |
| 头文件规则 | 只放声明不放定义（`static inline` 除外） |
| `extern` | 声明不产生符号 — "本变量在其他地方有定义" |
| `nm` | 查看符号表的命令行工具 — 大写字母 = 全局，小写 = 局部 static |

**核心观点**：链接器不关心你的源代码怎么组织——它只关心符号表。如果你的头文件中放了函数定义，每个 `#include` 的 `.c` 都导出同一个强符号——这就是链接错误的根源。用 `nm` 看符号表，每次编译如果出现链接错误，第一件事是检查符号表，不是改代码。
