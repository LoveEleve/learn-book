## 1. 引言：锁之外的世界

前 3 章讲了 mutex/spinlock（S3Ch01）、原子操作（S3Ch02）、内存模型（S3Ch03）。这些是锁的底层实现。但如果**不用锁**，只用原子操作来搭建数据结构——会怎样？

**锁的代价**：
- 线程抢不到锁 → `futex` 进入睡眠 → 上下文切换（1-2μs）
- sleep 的线程唤醒后重新抢锁 → 可能再次失败（thundering herd）
- 持有锁的线程做了 I/O 或换页（page fault）→ 所有等待者排队等它回来

**无锁的意义**：不是"没有竞争"——而是"在竞争时也不阻塞"。线程可能短暂重试 CAS 循环（浪费几个 CPU 周期），但不进入内核睡眠。适合**高并发、短操作**的场景：计数器、无锁队列、RCU 读锁。

## 2. CAS——无锁算法的原子原语

### 2.1 CAS 的 C11 形式

S3Ch02 §2.1 介绍了 CAS。这里重温实际使用模式：

```c
#include <stdatomic.h>

// CAS——Compare-And-Swap
int expected = atomic_load(&var);
do {
    int new_val = expected * 2;                        // 计算新值
} while (!atomic_compare_exchange_weak(&var, &expected, new_val));
// 如果 var == expected → var = new_val → 返回 true
// 如果 var != expected → expected = var → 返回 false → 重试
```

**`strong` vs `weak`**：
- `_strong`：只在 var != expected 时失败——保证最多一次循环
- `_weak`：可能**虚假失败**（即使 var == expected 也返回 false）——在某些 ARM 平台上更高效（无需重试 LL/SC 的完整循环）

**生产建议**：当 CAS 在循环中用时，用 `_weak`（更快）；当 CAS 只执行一次时，用 `_strong`。

### 2.2 CAS 的硬件实现

| 架构 | 指令 | 机制 |
|---|---|---|
| x86 | `lock cmpxchg [addr], new_val` | 隐式全屏障——与 MESI 协议配合，缓存行锁定 |
| ARM | `LDREX` + `STREX` 循环 | 独占监视器——如果 STREX 失败（有其他 CPU 修改了 addr），跳回 LDREX |

## 3. ABA 问题——CAS 的盲区

### 3.1 ABA 是什么

CAS 只比较值——不比较"这个值是不是曾经被修改过"：

```
线程 1: 读到 var=A
线程 2: var=A→B→A（把 A 改成了 B，又改回了 A）
线程 1: CAS(var, A, new_val) → 成功！
```

CAS 认为 var 没变——但它在 CAS 期间经历了 A→B→A 的变更。如果 var 是指向堆内存的指针，ABA 可能导致 UAF：

```
线程 1: 读 head → 节点 X
线程 2: pop 掉 X → free(X) → push 新节点 Y（恰好分配到 X 的地址）
线程 1: CAS(head, X, X→next) → 成功！但 X 已经被 free 了 → UAF
```

### 3.2 ABA 的解决方案

| 方案 | 原理 | 开销 | 适用场景 |
|---|---|---|---|
| **标记指针**（tagged pointer） | 指针高位存版本号，CAS 同时比较指针+版本号 | 低（需要 128 位 CAS，x86 `cmpxchg16b`） | x86 可用 |
| **hazard pointer**（危险指针） | 延迟回收：读者声明"我在读 X"，写者暂不释放 X | 中（每个线程维护 retire 列表） | 通用，多线程友好 |
| **epoch-based reclamation**（EBR） | RCU 简化版：按"时代"分组延迟回收 | 低（单计数器） | 读多写少 |

## 4. MPSC 队列——多生产者单消费者

### 4.1 Vyukov 算法

Dmitry Vyukov 的 MPSC 队列是最经典的无锁队列之一——多个生产者线程安全地 push，单个消费者线程无锁地 pop：

```c
#include <stdatomic.h>
#include <stdlib.h>

typedef struct mpsc_node {
    void               *data;
    struct mpsc_node   *next;
} mpsc_node_t;

typedef struct {
    _Atomic(mpsc_node_t *) head;   // 生产者 push 到这里
    mpsc_node_t           *tail;   // 消费者 pop 从这里
} mpsc_queue_t;

void mpsc_init(mpsc_queue_t *q) {
    // 🔑 必须初始化哨兵节点——否则首次 enqueue 的 prev=NULL 解引用崩溃
    mpsc_node_t *sentinel = calloc(1, sizeof(mpsc_node_t));
    atomic_store_explicit(&q->head, sentinel, memory_order_relaxed);
    q->tail = sentinel;
}

void mpsc_enqueue(mpsc_queue_t *q, void *data) {
    mpsc_node_t *n = calloc(1, sizeof(mpsc_node_t));
    n->data = data;
    // 原子地将新节点放到队列头部——返回旧头部
    mpsc_node_t *prev = atomic_exchange_explicit(&q->head, n, memory_order_acq_rel);
    // 🔑 关键窗口：这里 exchange 已完成，但 prev->next = n 还没执行
    // 在另一 CPU 上的消费者可能看到 prev->next == NULL → 返回空或 spin-wait
    atomic_store_explicit(&prev->next, n, memory_order_release);
}

void *mpsc_dequeue(mpsc_queue_t *q) {
    mpsc_node_t *tail = q->tail;
    mpsc_node_t *next = atomic_load_explicit(&tail->next, memory_order_acquire);
    if (next == NULL) return NULL;  // 队列空（或生产者还在设置 prev->next）
    q->tail = next;
    void *data = next->data;
    free(tail);
    return data;
}
```

**关键陷阱**：

1. **哨兵节点必须有**——如果忘了初始化，首次 `enqueue` 的 `atomic_exchange` 返回 `NULL`，`prev->next = n` 解引用 NULL → SIGSEGV
2. **dequeue 可能短暂看到 `next == NULL`**——因为 `enqueue` 在 `exchange` 和 `prev->next = n` 之间有空窗期。dequeue 在这个空窗期中会返回 NULL（消费者需要重试或等待）
3. **内存回收**——`free(tail)` 后，如果当前消费者释放的旧哨兵还在被其他线程引用 → UAF。hazard pointer 或 epoch-based reclamation 来解决

### 4.2 MPSC 实验

```bash
cat > mpsc_test.c << 'EOF'
#include <stdio.h>
#include <pthread.h>
#include <stdlib.h>
#include <stdatomic.h>
// ... (上面 mpsc_* 代码) ...

#define N 100000
mpsc_queue_t q;
_Atomic int produced = 0;

void *producer(void *arg) {
    for (int i = 0; i < N; i++) {
        mpsc_enqueue(&q, (void *)(intptr_t)i);
        atomic_fetch_add(&produced, 1);
    }
    return NULL;
}

void *consumer(void *arg) {
    int consumed = 0;
    while (consumed < N * 2) {       // 2 个生产者
        void *v = mpsc_dequeue(&q);
        if (v != NULL) consumed++;
    }
    printf("Consumed: %d\n", consumed);
    return NULL;
}

int main() {
    mpsc_init(&q);
    pthread_t p1, p2, c;
    pthread_create(&p1, NULL, producer, NULL);
    pthread_create(&p2, NULL, producer, NULL);
    pthread_create(&c,  NULL, consumer, NULL);
    pthread_join(p1, NULL); pthread_join(p2, NULL);
    pthread_join(c,  NULL);
    printf("Done\n");
    return 0;
}
EOF
gcc -O2 -Wall mpsc_test.c -lpthread -o mpsc_test && ./mpsc_test
```

## 5. Treiber 栈——最简单的无锁容器

Treiber 栈（R. Kent Treiber, 1986）是无锁 LIFO 栈的实现——比 MPSC 队列简单，因为 pop 操作不需要前向指针：

```c
typedef struct treiber_node {
    void *data;
    struct treiber_node *next;
} treiber_node_t;

typedef struct {
    _Atomic(treiber_node_t *) head;
} treiber_stack_t;

void treiber_push(treiber_stack_t *s, void *data) {
    treiber_node_t *n = malloc(sizeof(treiber_node_t));
    n->data = data;
    do {
        n->next = atomic_load(&s->head);
    } while (!atomic_compare_exchange_weak(&s->head, &n->next, n));
}

void *treiber_pop(treiber_stack_t *s) {
    treiber_node_t *head;
    do {
        head = atomic_load(&s->head);
        if (head == NULL) return NULL;
    } while (!atomic_compare_exchange_weak(&s->head, &head, head->next));
    void *data = head->data;
    // 🔑 不能在这里 free(head) —— 可能有其他线程仍持有 head
    // 需要用 hazard pointer 或 EBR 延迟回收
    return data;
}
```

**为什么 pop 不能立即 free(head)**：线程 A 读到 head=X，线程 B 也读到 head=X。线程 A CAS 成功（head=X→X→next），释放 X。线程 B 还在持有 `head=X`——CAS 时会失败（因为 head 已经是 X→next），但 free 后访问 X 已经是 UAF。ABA 问题 + 内存回收问题交织。

## 6. 内存回收：hazard pointer 与 EBR

### 6.1 Hazard Pointer

每个线程维护一个 **hazard pointer** 列表——声明"我正在读这些指针"。回收线程检查这些指针，如果发现有指向正要回收的节点，暂不释放：

```
线程 A: hazard[0] = pop_head（声明："我在读这个节点"）
线程 B: CAS 成功，B 想要 free(old_head)
        → 检查所有线程的 hazard pointer → 发现 A 的 hazard[0] == old_head
        → 不释放！将 old_head 放入退休列表，等 A 清空 hazard[0]
```

**适用场景**：多线程都能 pop——需要协调多个线程之间的回收。开销中等（每次读需要写一次 hazard list）。

### 6.2 Epoch-Based Reclamation（EBR）

RCU 的简化版——按"时代"分组：

```
全局 epoch 计数器（3 个 epoch: 0/1/2）
线程 A: 进入读侧临界区 → 记录当前 epoch
线程 B: 释放节点 → 放入当前 epoch 的退休列表
        → 当所有线程都不在某个 epoch 时 → 释放该 epoch 的退休列表
```

**对比 RCU**：RCU 还提供 `synchronize_rcu()` 等待所有读者完成。EBR 没有等待——通过 epoch 翻转找到安全时刻。EBR 的临界区更轻量（只读一个计数器）。

### 6.3 什么时候选哪种

| 方案 | 内存开销 | CPU 开销 | 延迟 | 适用 |
|---|---|---|---|
| tagged pointer | 0 | 最低 | — | x86（仅当数据适合 64 位指针 + 标签） |
| hazard pointer | 高（每线程 N 个指针） | 中（写 hazard list） | 释放延迟 | 多线程 pop 场景 |
| EBR | 低（一个全局计数器） | 最低（临界区仅读计数器） | 释放延迟 | **读多写少**——Linux 内核 RCU 同样的设计哲学 |

## 7. Linux 内核的无锁数据结构

### 7.1 kfifo——无锁环形缓冲区

S3Ch03 §5.2 提到了 kfifo。这里的核心是：**单生产者 + 单消费者不需要 CAS**——用 `smp_wmb()`/`smp_rmb()` 即可保证顺序：

```c
// 内核 kfifo 的简化实现
unsigned int kfifo_put(struct kfifo *fifo, const unsigned char *buffer, unsigned int len) {
    unsigned int l;
    len = min(len, fifo->size - fifo->in + fifo->out);
    l = min(len, fifo->size - (fifo->in & (fifo->size - 1)));
    memcpy(fifo->buffer + (fifo->in & (fifo->size - 1)), buffer, l);
    memcpy(fifo->buffer, buffer + l, len - l);
    smp_wmb();  // 保证数据拷贝在索引更新之前对其他 CPU 可见
    fifo->in += len;
    return len;
}
```

**为什么不用 CAS**：当只有一个写者时，不需要原子比较——只需要保证写入顺序对其他 CPU 可见（`smp_wmb()` 就够）。CAS 的开销远大于写屏障——在内核中节省每个周期都极其重要。

### 7.2 RCU——读者完全无锁

RCU（Read-Copy-Update）是 Linux 内核最著名的高性能读方案——读者**不需要硬件内存屏障**：

```c
// 读者——x86/ARM 均不需要 DMB/mfence（仅需 READ_ONCE 编译器屏障）
rcu_read_lock();                              // 非抢占内核中编译为空——零 CPU 指令
node = rcu_dereference(global_ptr);           // 展开为 READ_ONCE(global_ptr) — 仅编译器屏障
read_data = node->data;
rcu_read_unlock();                            // 非抢占内核中编译为空

```c
// 读者——零屏障！
rcu_read_lock();
node = rcu_dereference(global_ptr);
read_data = node->data;
rcu_read_unlock();

// 写者——复制旧数据、修改、原子替换、等待所有读者完成
new_node = kmalloc(sizeof(*new_node), GFP_KERNEL);
*new_node = *old_node;
new_node->data = new_value;
rcu_assign_pointer(global_ptr, new_node);  // release store — 原子替换指针
synchronize_rcu();                           // 等待所有读者退出 rcu_read_lock
kfree(old_node);                             // 安全释放——已经没有读者持有旧指针
```

**为什么读者不需要屏障**？读者在 `rcu_read_lock()` 和 `rcu_read_unlock()` 之间不允许睡眠——内核以"读者是否正在执行"为判断标准。`synchronize_rcu()` 等待所有 CPU 都经历一次上下文切换（这保证之前的读者已退出），然后释放旧内存。

## 8. Lock-free vs Mutex 性能对比

```bash
cat > bench.c << 'EOF'
#include <stdio.h>
#include <pthread.h>
#include <time.h>
#include <stdatomic.h>

#define N_OPS 10000000
#define N_THREADS 4

// Mutex 版本
pthread_mutex_t mutex = PTHREAD_MUTEX_INITIALIZER;
long mutex_counter = 0;

void *mutex_worker(void *arg) {
    for (int i = 0; i < N_OPS / N_THREADS; i++) {
        pthread_mutex_lock(&mutex);
        mutex_counter++;
        pthread_mutex_unlock(&mutex);
    }
    return NULL;
}

// Lock-free 版本
_Atomic long atomic_counter = 0;

void *atomic_worker(void *arg) {
    for (int i = 0; i < N_OPS / N_THREADS; i++) {
        atomic_fetch_add_explicit(&atomic_counter, 1, memory_order_relaxed);
    }
    return NULL;
}

void run_test(const char *name, void *(*worker)(void *)) {
    pthread_t threads[N_THREADS];
    struct timespec start, end;
    clock_gettime(CLOCK_MONOTONIC, &start);
    for (int i = 0; i < N_THREADS; i++) pthread_create(&threads[i], NULL, worker, NULL);
    for (int i = 0; i < N_THREADS; i++) pthread_join(threads[i], NULL);
    clock_gettime(CLOCK_MONOTONIC, &end);
    double elapsed = (end.tv_sec - start.tv_sec) * 1e3 +
                     (end.tv_nsec - start.tv_nsec) / 1e6;
    printf("%s: %.1f ms (%.0f ops/μs)\n", name, elapsed, N_OPS / (elapsed * 1e3));
}

int main() {
    run_test("Mutex  ", mutex_worker);   // 预期：100-300 ms（大量 futex 内核调用）
    run_test("Atomic ", atomic_worker);  // 预期：10-50 ms（纯用户态 CAS）
    return 0;
}
EOF
gcc -O2 -Wall bench.c -lpthread -o bench && ./bench
# 典型结果：Mutex 150ms vs Atomic 35ms — lock-free 快 4-5 倍
# 注意：这是在 N_THREADS=4、无其他负载的情况——实际生产收益取决于争用
```

## 9. 总结

| 概念 | 一句话 |
|---|---|
| CAS | 原子比较并交换——所有无锁算法的原子原语 |
| ABA | CAS 只看值、不看历史——用 tagged pointer/hazard pointer/EBR 解决 |
| MPSC | 多生产者单消费者——用 `exch` + `prev->next` 配对，哨兵节点必须初始化 |
| Treiber 栈 | 最简无锁 CAS 循环——但 `pop` 不能立即 `free`（ABA） |
| hazard pointer | 读者声明"我在读 X"——写者暂不释放 X |
| EBR | 按"时代"回收——临界区极短（只读计数器），读多写少 |
| kfifo | 内核无锁环形缓冲区——单生产者不需要 CAS，`smp_wmb` 就够了 |
| RCU | 读者零屏障——`synchronize_rcu` 等待所有读者退出，释放旧内存 |
| Lock-free vs Mutex | 无竞争时 mutex 快（一条 CAS），竞争时无锁快（不需要休眠/唤醒） |

**核心观点**：无锁不是银弹——它只在"高争用 + 短操作"场景中比 mutex 更快。如果你不知道争用程度，用 mutex（安全且简单）。如果你确认这是性能瓶颈（且每个操作只有几纳秒到微秒级），再用无锁方案——但要准备好处理 ABA 和内存回收问题。

> **Benchmark 注意事项**：§8 的测试中 mutex 临界区仅 `counter++`（1 条指令），lock-free 用 `relaxed fetch_add`（1 条原子指令），且只有 4 线程。这偏向无锁方案。真实临界区通常 100ns+（几百条指令），此时 mutex 和 lock-free 的性能差距缩小。公平测试应在临界区内做模拟工作（如 `for(volatile int x=0;x<1000;x++)`），对比 `acq_rel fetch_add` vs `mutex`。结论**不具一般性**——仅说明"极短临界区 + 适度争用"场景中无锁的优势。

**与 Stage 2 的关系**：Stage 2 讲了链接/ELF/跨平台——Stage 3 的 `<stdatomic.h>` 和 `CAS` 依赖编译器生成正确的 `lock cmpxchg` 指令（Ch02）和 `LDREX/STREX`（Ch03）。没有 Stage 2 的 ABI 知识，你无法理解为什么同一份 C11 原子代码在 x86 和 ARM 上生成不同的机器指令。

**与 Stage 4 的展望**：Stage 4 进入系统编程——你将用 `mmap`/`epoll` 写高性能服务器。无锁 MPSC 队列将成为生产者（accept 新连接）和消费者（处理 I/O 事件）之间的核心管道。本章的 CAS 和内存回收知识将直接服务于无锁队列的系统级设计和信号驱动架构。
