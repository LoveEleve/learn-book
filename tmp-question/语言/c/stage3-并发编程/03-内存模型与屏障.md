## 1. 引言：为什么同一份代码在 x86 上跑对了，ARM 上崩了？

S3Ch02 用 C11 的内存序（`relaxed`/`acquire`/`release`/`seq_cst`）来编程。但不同 CPU 架构对这些内存序的实现方式完全不同——在 x86 上，`memory_order_release` 的 store 编译为一条普通的 `mov` 指令（零成本），而在 ARM 上，同样的 store 需要一条 `DMB ISH` 内存屏障（每条数纳秒）。

这不是"ARM 比 x86 慢"——这是 x86 在硬件层面提供了更强的内存一致性保证（TSO），付出的是功耗和复杂度的代价。本章的目��是让你理解**每个 CPU 架构在硬件层面做出了什么权衡，以及这些权衡如何反映到你写的 C 代码中**。

## 2. x86-TSO：强大但非完美

### 2.1 TSO 的 4 条规则

x86 的 TSO（Total Store Order，全存储序）给出了 4 条硬件保证：

| 规则 | 含义 | 允许的重排 |
|---|---|---|
| **Store→Store** | 所有 CPU 看到相同的 store 提交顺序（FIFO store buffer） | ❌ 不重排 |
| **Load→Load** | load 按程序顺序执行（但可被乱序执行窗口隐藏） | ❌ 不重排（x86 保证） |
| **Load→Store** | 先读后写——先执行的 load 不能重排到后执行的 store 之后 | ❌ 不重排 |
| **Store→Load** | 先写后读——store buffer 中的 store 还没来得及对其他 CPU 可见，CPU 已经执行了后续的 load | ✅ **可以重排！** |

**第四条是 x86-TSO 唯一的弱点**——也是为什么 x86 仍然需要 `mfence` 和 `lock` 前缀指令的地方。这四种情况在 Dekker 算法（两个线程各自写 flag 然后读对方的 flag）中暴露。

### 2.2 x86-TSO 的硬件实现

x86 用三样东西实现 TSO：

**x86 的三种屏障指令**：

| 指令 | 全称 | 效果 | 开销 |
|---|---|---|---|
| `mfence` | Memory Fence | 全屏障——所有之前的 load+store 在所有之后的 load+store 之前完成 | ~33 cycles |
| `sfence` | Store Fence | 只保证 store-store 顺序——只能和 store 指令配合，不能阻止 load 重排 | ~5 cycles |
| `lfence` | Load Fence | 只保证 load-load 顺序 + 序列化指令流（所有后续指令在 lfence 之后才执行） | ~5 cycles |

`sfence` 用于非临��存储（如写入 MMIO 寄存器后），`lfence` 用于防止 Spectre 变种（序列化指令流）。`mfence` 是通用的全屏障——最贵。

**1. Store Buffer（写入缓冲区）**：每个 CPU 核心有一个 FIFO store buffer。store 指令先在 store buffer 中排队、不立即提交到缓存行。后续 load 指令会先查 store buffer——如果有同地址的 store 还没提交，直接从前面的 store 读值（store-to-load forwarding），不等到提交。

**2. MESI（Modified/Exclusive/Shared/Invalid — 四种缓存行状态）缓存一致性协议**：当 store 从 store buffer 提交到缓存行时，MESI 协议向其他 CPU 发送 Invalidate 消息。接收 CPU 将自己的缓存行置为 I（Invalid，无效）状态，下次读该地址时发起缓存缺失→从其他 CPU 或内存读取新值。

**3. x86 不保证 Store→Load**：CPU0 的 store 在 store buffer 中，CPU1 还没收到 Invalidate——CPU0 已经执行了后续 load 看到 CPU1 还没有更新的值。这就是 Store→Load 重排的来源。

### 2.3 在 x86-TSO 下写 C 代码的含义

| C11 内存序 | x86 指令 | 开销 | 是否需要额外屏障？ |
|---|---|---|---|
| `relaxed` store | `mov` | ~1 cycle | ❌ |
| `relaxed` load | `mov` | ~1 cycle | ❌ |
| `release` store | `mov` | ~1 cycle | ❌ — x86 Store→Store 已保证 |
| `acquire` load | `mov` | ~1 cycle | ❌ — x86 Load→Load 已保证 |
| `acq_rel` RMW | `lock xadd`/`lock cmpxchg` | ~20-33 cycles | ✅ `lock` prefix 隐式全屏障 |
| `seq_cst` store | `xchg`（带隐式 lock）/ `mov;mfence` | ~20-33 cycles | ✅ 必须阻止 Store→Load 重排 |
| `seq_cst` load | `mov` | ~1 cycle | ❌ |

**关键洞察**：x86 上 `acquire` 和 `release` 几乎免费——不是 C 标准好心，而是 x86 硬件已经提供了这些保证。`seq_cst` store 的 20-33 cycles 代价来自阻止 Store→Load 重排（这是 x86-TSO 唯一不保证的），需要昂贵的全屏障。

> **现在来看 ARM 如何走到另一个极端**——放弃所有硬件保证以换取更低功耗和更高的乱序执行自由度。

## 3. ARM 弱内存模型

### 3.1 ARM 做了什么不同的选择

ARM 做了一个与 x86 相反的设计决策：**不保证任何 load/store 顺序**。所有 load 和 store 都可以被 CPU 内部乱序执行——包括 Store→Store、Load→Load、Load→Store、Store→Load。这给了 ARM CPU 极大的自由来最大化 IPC（Instructions Per Cycle）。

**为什么 ARM 选了这个？** 移动设备（手机/平板）的功耗是第一位的。弱一致性让 CPU 可以更激进地乱序执行、合并 store、延迟 cache 操作——每减少一条缓存操作就省一点电。在移动设备上，每天少 5% 功耗意味着电池多用 30 分钟。

### 3.2 ARM 的内存屏障指令

因为 ARM 不提供任何硬件保证，所有顺序必须通过显式的内存屏障指令来建立：

| 指令 | 全称 | 效果 | 开销 | 何时用 |
|---|---|---|---|---|
| **`DMB ISH`** | Data Memory Barrier — Inner Shareable | 内部可共享域的所有数据访问在此指令之前完成 | ~10-30 cycles | 多核同步（同簇 CPU 间） |
| **`DMB SY`** | Data Memory Barrier — SYstem | 全系统域（包括外设） | ~30-50 cycles | 跨簇、外设 DMA |
| **`DSB`** | Data Synchronization Barrier | 所有在 DSB 之前的内存访问**和缓存操作**必须完成，之后才能执行后续指令 | ~50-100 cycles | 更新页表、切换 MMU |
| **`ISB`** | Instruction Synchronization Barrier | 刷新流水线，确保所有上下文变更在 ISB 之后的指令中可见 | ~10-20 cycles | 修改系统控制寄存器后 |

**C11 内存序在 ARM 上的映射**：

> **ARMv8-A 起**，引入了专有的 acquire/release 指令：`LDAR`（Load-Acquire）和 `STLR`（Store-Release）——比传统的 `DMB ISH`+`LDR`/`STR` 快约 30%。编译器在 ARMv8+ 上实际生成 `LDAR`/`STLR` 而非 DMB 方案。理解 DMB 语义仍然重要——`seq_cst` 和全屏障仍需要 DMB。

| C11 内存序 | ARM 指令序列（ARMv8+） | ARMv7 传统方案 | 开销 |
|---|---|---|---|
| `relaxed` store/load | `STR`/`LDR` | 同 | ~1 cycle |
| `acquire` load | `LDAR` | `LDR`; `DMB ISH` | ~10-25 cycles |
| `release` store | `STLR` | `DMB ISH`; `STR` | ~10-25 cycles |
| `acq_rel` RMW | `LDREX`/`STREX` 循环 + `DMB ISH` | 同 | ~25-50 cycles |
| `seq_cst` store | `STLR`; `DMB ISH` | `DMB ISH`; `STR`; `DMB ISH` | ~30-60 cycles |

**ARM vs x86 成本对比**：x86 上 acquire/release 有零成本（mov），ARMv8+ 上 LDAR/STLR 需要 10-25 cycles——接近 x86 的 seq_cst store（~20-33 cycles）。这就是为什么在 ARM 上选择正确的内存序比 x86 上重要得多。

> **弱序模型的多样性**：ARM 是弱序模型的代表（为功耗优化），但不是唯一的。POWER 架构（IBM）用 `lwsync`/`hwsync`/`sync` 三级屏障体系，比 ARM 更细粒度。RISC-V 的 RVWMO（RISC-V Weak Memory Ordering）用 `fence rw,rw` 等指令——设计更精简。本章聚焦 x86 vs ARM，但技术专家应知道弱序模型内部存在多样性——不同架构的"弱"不是同一种"弱"。

## 4. 经典算法在不同架构上的行为

### 4.1 Dekker 算法——两个线程互相同步

Dekker 是两个线程各自设置自己的 flag 然后读对方的 flag——测试 Store→Load 重排：

```
线程0: flag0 = 1      线程1: flag1 = 1
         r0 = flag1             r1 = flag0
```

**在 x86-TSO 上**：可能 `r0=0, r1=0`。为什么？`flag0 = 1` 在 store buffer 中，CPU0 在 store 提交前执行了 `r0 = flag1`（Store→Load 重排）。如果 CPU1 同时做了同样的事→双方都看到对方的 flag 为 0。需要 `mfence` 阻止 Store→Load 重排。

**在 ARM 上**：更可能 `r0=0, r1=0`。因为 ARM 连 Store→Store 也不保证——`flag0 = 1` 可能被乱序执行到 `r0 = flag1` 之后。需要 `DMB` 在 store 和 load 之间插入全屏障。

**C11 修复**：两个 flag 都用 `seq_cst`——x86 上用 `xchg`/`mfence`，ARM 上用 `DMB`——保证不会同时看到对方为 0。

### 4.2 Peterson 锁——两个线程互斥

Peterson 锁是两个线程通过共享 flag + turn 变量仲裁谁进入临界区的算法：

```c
// 线程0
flag[0] = 1;                              // 声明意图
turn = 1;                                 // 让给线程1
while (flag[1] && turn == 1);             // 如果线程1不在临界区、或轮到我 → 进入

// 线程1 — 对称
flag[1] = 1;
turn = 0;
while (flag[0] && turn == 0);
```

**x86-TSO 上**：Peterson 锁**不工作**——`flag[0] = 1` 在 store buffer 中还没对其他 CPU 可见时，线程0 读到了 `flag[1]`（Store→Load 重排）→ 可能两个线程都在临界区。需要 `mfence` 在 store 和 load 之间。

**ARM 上**：同样不工作，需要 `DMB`。x86 的故障模式是 Store→Load 重排，ARM 还可能加上 Store→Store 重排。

**C11 实现**——Peterson 需要阻止 **Store→Load 重排**（flag store 必须在 flag load 之前对其他核可见）。`release` 只保证前面的 store 不重排到后面，`acquire` 只保证后面的 load 不重排到前面，**二者之间没有自动的全屏障**。Peterson 的正确 C11 实现需要 `seq_cst`（提供 StoreLoad 序）：

```c
// 线程0 — seq_cst 保证 store 在 load 之前完成
atomic_store_explicit(&flag[0], 1, memory_order_seq_cst);
atomic_store_explicit(&turn, 0, memory_order_seq_cst);
while (atomic_load_explicit(&flag[1], memory_order_seq_cst) == 1 &&
       atomic_load_explicit(&turn, memory_order_seq_cst) == 0);
// 线程1 — 对称，flag[1]=1, turn=0

## 5. Linux 内核的内存屏障

> **从用户态到内核态**：以上是用户态用 C11 原子操作的视角。但在 Linux 内核中，C 标准库不可用（内核直到 2022 年才迁 C11），内核维护了自己的内存屏障体系——这是理解"为什么内核能用 C89 写出 lock-free 代码"的关键。

### 5.1 内核屏障 API

S3Ch02 §5.3 提到内核用 `atomic_t` 和 `smp_mb()`。内核提供三套屏障：

| 屏障 | 含义 | x86 实现 | ARM 实现 |
|---|---|---|---|
| `smp_mb()` | 全内存屏障 | `mfence` 或 `lock; addl $0, (%rsp)` | `DMB ISH` |
| `smp_wmb()` | **写**屏障（仅保证 store 的顺序） | 编译器屏障（`barrier()`）—x86 Store→Store 早已保证 | `DMB ISHST`（store-only） |
| `smp_rmb()` | **读**屏障（仅保证 load 的顺序） | 编译器屏障—x86 Load→Load 早已保证 | `DMB ISHLD`（load-only） |

**x86 上 wmb/rmb 为什么是空的？** x86-TSO 已经保证 Store→Store 和 Load→Load 不重排，所以内核的 `smp_wmb()` 和 `smp_rmb()` 在 x86 上只是编译器屏障——防止编译重排但不需要 CPU 指令。

**ARM 上不能省**——`smp_wmb()` 生成 `DMB ISHST`，`smp_rmb()` 生成 `DMB ISHLD`——每个都有 10-20 cycles 的开销。

### 5.2 内核中内存屏障的使用场景

**场景 1：无锁环形缓冲区**——内核的 `kfifo`（内核 FIFO）实现中：

```c
// 生产者：写入数据后更新锁
buffer[in] = data;
smp_wmb();                        // 保证数据写入在 in 更新之前对其他 CPU 可见
WRITE_ONCE(in, (in + 1) % SIZE);  // 原子更新写入索引

// 消费者：读取 in 后读数据
unsigned int local_in = READ_ONCE(in);
smp_rmb();                        // 保证读 in 在数据读取之前
consume(buffer[local_out]);
```

**场景 2：RCU（Read-Copy-Update）— 读者无锁/写者锁**：

```c
// 写者（更新一个链表节点）：
new_node->data = value;
rcu_assign_pointer(head->next, new_node);   // publish——内含 release 语义

// 读者（遍历链表）：
rcu_read_lock();
node = rcu_dereference(head->next);         // consume——内含 acquire 语义
read_data = node->data;
rcu_read_unlock();
```

RCU 的 `rcu_dereference` 在 Linux 内核中依赖 `READ_ONCE` + `smp_read_barrier_depends()`——在 ARM 上是真正的读屏障，在 x86 上是编译器屏障。

## 6. Apple Silicon 迁移：为什么软件出 bug？

2020 年 Apple 从 Intel x86 迁移到 Apple Silicon（ARM）。macOS 上运行的大多数 x86 软件通过 Rosetta 2 转译运行，但一部分原生 ARM 的软件出现了大量数据竞争 bug。

**为什么**：这些软件在开发时只在 x86 上测试——acquire/release 被编译为免费的 `mov` 指令（x86-TSO 已保证），开发者甚至不知道自己在用什么内存序。当同样的代码在 ARM 上编译时，acquire/release 变成了带 `DMB` 屏障的指令序列——但有些变量可能根本没标注内存序（直接用普通变量），x86 的强一致性掩盖了真正的数据竞争。一旦 ARM 上执行，store buffer 的激进重排暴露了之前的隐藏 bug。

**教训**：
1. 开发多线程代码时**永远在 ARM 上测试**——x86-TSO 太强，会掩盖大量内存序 bug
2. 所有线程间共享的变量必须用 `_Atomic` 类型或显式内存屏障——不是"x86 上能用就行"
3. C11 `<stdatomic.h>` 的 `seq_cst` 默认是正确的——只在确认性能瓶颈时有针对性降级

## 7. Litmus 测试——亲手验证内存模���

Litmus 测试是小型的"多线程竞争实验"（Litmus 原意是石蕊试纸——"一测就知道结果"），用于验证 CPU 是否允许某种重排行为。常见的标准 Litmus 测试名：SB（Store Buffering，即 Dekker）、MP（Message Passing）、LB（Load Buffering）。

**Dekker Litmus 测试**：

```bash
cat > dekker.c << 'EOF'
#include <stdatomic.h>
#include <stdio.h>
#include <pthread.h>

atomic_int flag0 = 0, flag1 = 0;
int r0 = -1, r1 = -1;

void *thread0(void *arg) {
    atomic_store_explicit(&flag0, 1, memory_order_relaxed);
    r0 = atomic_load_explicit(&flag1, memory_order_relaxed);
    return NULL;
}

void *thread1(void *arg) {
    atomic_store_explicit(&flag1, 1, memory_order_relaxed);
    r1 = atomic_load_explicit(&flag0, memory_order_relaxed);
    return NULL;
}

int main() {
    int both_zero = 0;
    for (int i = 0; i < 1000000; i++) {
        flag0 = flag1 = 0;
        pthread_t t0, t1;
        pthread_create(&t0, NULL, thread0, NULL);
        pthread_create(&t1, NULL, thread1, NULL);
        pthread_join(t0, NULL); pthread_join(t1, NULL);
        if (r0 == 0 && r1 == 0) both_zero++;
    }
    printf("Both saw 0: %d / 1000000 (%.3f%%)\n", both_zero, 100.0 * both_zero / 1000000);
    return 0;
}
EOF
gcc -O2 -Wall dekker.c -lpthread -o dekker && ./dekker
# x86-TSO 上可能显示 "Both saw 0: 1 / 1000000"（极罕见——Store→Load 重排极小概率）
# ARM 上可能显示 "Both saw 0: 500 / 1000000"（显著——Store→Store 重排高频）
```

**将 `relaxed` 改为 `seq_cst` 再跑一次——应该显示 0/1000000。**

## 8. 总结

| 概念 | 一句话 |
|---|---|
| x86-TSO | Store/Store 和 Load/Load 永不重排，Store→Load 可以——seq_cst 的代价全在这里 |
| ARM Weak | 不保证任何顺序——所有顺序通过显式 `DMB` 屏障指令建立 |
| Store Buffer | x86 上 每个 CPU 核心的写入暂存区——Store→Load 重排的物理根源 |
| MESI | x86 的缓存一致性协议——Invalidate 延迟可能让不同 CPU 看到不同值 |
| `mfence` | x86 全屏障——阻止 Store→Load 重排，seq_cst 的昂贵来源 |
| `DMB ISH` | ARM 内部可共享域屏障——保证前后访问在域内有序 |
| Dekker | 两个线程写 flag 后读对方——暴露 Store→Load 重排的经典测试 |
| RCU | Linux 内核的读者无锁机制——依赖 `READ_ONCE` + 依赖序（`consume`） |
| Apple Silicon | x86→ARM 迁移暴露大量隐藏数据竞争——弱一致性是天然的质量检测器 |
| Litmus 测试 | 百万次重反复现平台特有的内存重排行为 |

**核心观点**：CPU 的内存模型不是一成不变的规则——它是硬件设计权衡的结果。x86 选择强一致性（TSO）但增加了功耗和复杂度，ARM 选择弱一致性但省电。C11 的内存序是架在不同 CPU 之间的桥梁——你的代码不只需要知道"怎么写 acquire/release"，更要知道"这条 release 在 x86 上是免费的 mov，在 ARM 上是一条 20-cycle 的 DMB"。技术专家不是"知道规范就够了"，而是"知道规范在每种 CPU 上的硬件代价"。
