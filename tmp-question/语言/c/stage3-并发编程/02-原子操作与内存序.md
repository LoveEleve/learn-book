## 1. 引言：一行 `counter++` 的内在秘密

S3Ch01 用 mutex 保护了 `counter++`。但 mutex 本身就是由更基础的**原子操作**构建的。这一章向下探索——CPU 指令层面发生了什么，以及 C 语言如何���过 `<stdatomic.h>` 让你直接控制这些指令。

一行 `counter++` 在机器码中是**三条指令**：

```
mov eax, [counter]     # 1. 读 counter 的值
add eax, 1             # 2. 加 1
mov [counter], eax     # 3. 写回
```

两个线程同时执行这三条指令——线程 A 读到 counter=0、线程 B 也读到 counter=0、两者都加 1、两者都写回 1 → 两次操作后 counter=1 而不是 2。这就是**数据竞争**。原子操作用**一条 CPU 指令**完成"读-改-写"三步，消除竞争窗口。

## 2. C11 `<stdatomic.h>` 基础

### 2.1 原子类型

```c
#include <stdatomic.h>

atomic_int counter = 0;          // 等价于 _Atomic int counter
atomic_uint flag = 0;            // _Atomic unsigned int
atomic_bool ready = false;
atomic_uintptr_t ptr = 0;        // 原子指针

// 基本操作
atomic_store(&flag, 1);          // 原子写
int v = atomic_load(&counter);   // 原子读
atomic_fetch_add(&counter, 1);   // 原子加——返回旧值
int old = atomic_exchange(&flag, 0); // 原子交换——返回旧值

// Compare-And-Swap——无锁算法的核心
int expected = 0;
if (atomic_compare_exchange_strong(&counter, &expected, 42)) {
    // counter 原来是 0 → 现在是 42
} else {
    // counter 不是 0 → expected 被更新为 counter 的实际值
}
```

`_Atomic` 关键字确保对变量的每一次读写都是原子的——编译器不能将一次 64 位读分解为两次 32 位读、不能将变量缓存在寄存器中。

### 2.2 原子操作 vs 非原子操作

`int counter`（非原子）= 编译器可以自由优化：
- 可以缓存值到寄存器，连续两次 `counter++` 可能被合并
- 可能被分解为多字节读（非对齐访问）
- 编译器可能重排序与其他内存操作的顺序

`atomic_int counter`（原子）= 编译器保证：
- 每次都从内存读写（不缓存到寄存器）
- 64 位平台上保证单指令读/写（不对齐访问仍可能不原子）
- **默认 `memory_order_seq_cst`**—最强的顺序保证

## 3. 五种内存序

### 3.0 为什么 CPU 会重排指令？

内存序是 CPU 的 store buffer（写入缓冲区）和 invalidate queue（失效队列）的产物。理解这两个硬件机制是理解"为什么需要 acquire/release"的前提。

**Store Buffer**：CPU 发出 store 指令时，不等到数据写入缓存行——先放入 store buffer，CPU 继续执行后续指令。store buffer 将 store 对其他 CPU 可见的时刻推迟——其他 CPU 在 store 从 buffer 提交到缓存行之前看不到新值。

**由此产生的问题**——考虑两个 CPU 各自维护一个 flag：

```
CPU0: x = 1          CPU1: y = 1
       flag1 = 1              flag0 = 1
  ↓ store buffer 延迟  ↓ store buffer 延迟
```

在没有内存屏障的情况下，CPU0 的 store buffer 可能延迟 `flag1 = 1` 的提交，而在此间隙 CPU1 已经看到了 `flag0 = 1`。结果：CP0 看到 flag0=1 但没看到 y=1，CP1 看到 flag1=1 但没看到 x=1——两个 CPU 对"谁先写"看到不同的顺序→内存一致性的崩溃。

**这就是为什么 x86 需要 TSO（Total Store Order）**：x86 的所有 CPU 在"store buffer 中的 store 对其他 CPU 可见的顺序"上达成一致——store buffer 是 FIFO，所有 CPU 看到相同的 store 提交顺序。但 store-load 仍可重排：CPU 先 store 到 buffer，该 store 还没对其他 CPU 可见时，CPU 已经执行了后续的 load。

**这就是为什么 ARM 选择弱一致性**：ARM 的 store buffer 更激进——不仅 store-load 可以重排，store-store 也可以重排。这给了 CPU 更大的乱序执行空间（功耗更低、IPC 更高），但代价是需要显式的内存屏障（`DMB`）来保证顺序。

**如果让我来设计内存一致性模型**：

1. 最激进的方案——无顺序保证（`relaxed`）：只保证原子性，最快，但程序员用着难受
2. 适度方案——提供 acquire/release 配对：CPU 可以在 store buffer 和 invalidate queue 上乱序执行，但在程序员指定的点加屏障——性能与安全性的平衡
3. 最保守方案——所有操作顺序一致（`seq_cst`）：最简单，但在有 store buffer 的 CPU 上极慢

C11 提供了这三种方案的全部——让程序员根据场景选择。

---

C11 定义了 5 种内存序（memory order），从最弱到最强：

```
relaxed  ─── 只保证原子性，不保证顺序
    ↓
acquire  ── load 获取——后续读写不能重排到此操作之前
release  ── store 释放——之前的读写不能重排到此操作之后
    ↓
acq_rel  ── RMW 操作——既有 acquire 又有 release
    ↓
seq_cst  ── 全局单一顺序——所有线程看到相同的操作顺序
```

### 3.1 `memory_order_relaxed` ——计数器专用

**语义**：只保证原子性，不保证顺序。对其他变量的访问可以自由重排到此原子操作之前或之后。

```c
atomic_int counter = 0;

void increment(void) {
    atomic_fetch_add_explicit(&counter, 1, memory_order_relaxed);
}
```

**适用场景**：唯一的约束是原子性——计数器、���计、引用计数（当释放的顺序不重要时）。如果计数器结果用于决定"是否释放内存"——不，此时需要 acquire/release 确保数值和可见性同步。

**为什么需要 relaxed？** 这是最快的内存序——x86 上编译为单条 `lock xadd` 指令，不需要额外的内存屏障。ARM 上也编译为 `LDREX/STREX` 循环，同样无屏障开销。

### 3.2 `memory_order_acquire` ——读 + 检查

**语义**：load 操作——**后续**的读写不能重排到此 load 之前。

```
线程1（生产者）            线程2（消费者）
   data = 42;                 ↓ 必须从这里之后读 data
   flag.store(1, release); →  →  flag.load(acquire);
                                  assert(data == 42);  ← 仅当 flag==1 时
```

**为什么**：需要确保读到 flag=1 时，生产者写入 flag 之前的所有写操作（包括 data=42）都对消费者可见。acquire 和 release 配对建立 **happens-before** 关系。

**x86 实现**：acquire load = 普通 load 指令（`mov`）—零额外成本。x86-TSO 本身就保证 load 不被重排序到��续指令之前。

**ARM 实现**：acquire load = `LDR` + `DMB ISH`（数据内存屏障，内部可共享域）—在 load 指令后插入轻量屏障。

### 3.3 `memory_order_release` ——写 + 发布

**语义**：store 操作——**之前**的读写不能重排到此 store 之后。

```c
data = 42;                               // 必须在这里之前写完
atomic_store_explicit(&flag, 1, memory_order_release); // 发布信号
```

**x86 实现**：release store = 普通 store 指令（`mov`）—零额外成本。x86-TSO 保证 store-store 不变序（store 不会重排到后续 store 之后）。

**ARM 实现**：release store = `DMB ISH` + `STR` —在 store 指令前插入轻量屏障。

### 3.4 acquire + release 配对——同步模式

```c
// 生产者
data = 42;
atomic_store_explicit(&flag, 1, memory_order_release); // 所有之前的写不能被移到后面

// 消费者
while (atomic_load_explicit(&flag, memory_order_acquire) != 1); // 所有之后的读不能被移到前面
assert(data == 42);                         // OK
```

这个配对建立了 **synchronizes-with** 关系：flag 的 release store 与 flag 的 acquire load 同步——生产者对 data 的写入在消费者对 data 的读取之前发生（happens-before）���

### 3.5 `memory_order_acq_rel` ——Read-Modify-Write 操作

**语义**：RMW（Read-Modify-Write——读-修改-写）操作既有 acquire 又有 release。读取之前的值提供 acquire 语义（后续的读写不能被重排到此 RMW 之前），写入新值提供 release 语义（之前的读写不能被重排到此 RMW 之后）。

```c
int old = atomic_fetch_add_explicit(&counter, 1, memory_order_acq_rel);
// 既有 acquire（读取 counter 旧值）又有 release（写入 counter 新值）
// RMW 操作——读取和写入是原子的、不可分的
```

**使用场景**：引用计数释放。你需要先减少引用计数（RMW），然后如果减到 0，释放对象——在释放之前必须看到所有之前持有者对对象的修改（acquire），在释放之前的所有写必须对其他线程可见（release）。

```c
if (atomic_fetch_sub_explicit(&refcount, 1, memory_order_acq_rel) == 1) {
    free(obj);  // 保证能看到所有指针使用者的写操作
}
```

**x86 实现**：所有 `lock` 前缀的指令隐式提供全屏障（full barrier）。`lock xadd`、`lock cmpxchg` 等 RMW 指令在 x86 上的 seq_cst 和 acq_rel 生成相同的机器码——两者开销基本相同（~20-33 cycles）。

**ARM 实现**：`LDREX` + `STREX` 循环，在成功写入后需要 `DMB ISH` 保证 release 语义。开销略高于单纯的 relaxed RMW。

### 3.6 `memory_order_consume` ——依赖序

C11 的 `memory_order_consume` 比 `acquire` 更弱——只保证与加载结果直接依赖的后续操作不会被重排。大部分编译器（GCC/Clang）将 `consume` 降级为 `acquire`（因为正确实现 consume 极其困难），但 Linux 内核的 RCU（Read-Copy-Update）机制依赖于 consume 语义。

### 3.7 `atomic_thread_fence` ——独立内存屏障

`atomic_thread_fence` 是独立的内存屏障——不伴随内存访问，纯控制指令重排：

```c
data = 42;
atomic_thread_fence(memory_order_release);  // 之前的写不能重排到此 fence 之后
atomic_store_explicit(&flag, 1, memory_order_relaxed);
```

**与操作级 ordering 的区别**：
- `atomic_store(&flag, 1, release)` = store + 隐藏的屏障——屏障和 store 是一体的
- `fence(release)` + `atomic_store(&flag, 1, relaxed)` = 显式分离的屏障——barrier 位置灵活控制

### 3.8 `memory_order_seq_cst` ——默认最安全

**语义**：所有 seq_cst 操作在全局有一个统一的顺序——所有线程看到相同的执行顺序。这是默认选择，也是最安全的。

```c
atomic_int x(0), y(0);

// 线程1
atomic_store(&x, 1);
int r1 = atomic_load(&y);

// 线程2
atomic_store(&y, 1);
int r2 = atomic_load(&x);

// seq_cst 保证：不能同时出现 r1=0 且 r2=0
// relaxed 下可能同时为 0（store 和 load 可被重排序）
```

**x86 代价**：seq_cst store = `xchg` 或 `mov; mfence` —~20-40 cycles（比普通 store 慢约 10-20 倍）。acquire/release 在 x86 上几乎免费。

**ARM 代价**：seq_cst = `DMB ISH` 全屏障，比 acquire/release 的轻量屏障更贵。

### 3.7 五种内存序速查表

| 内存序 | 含义 | x86 指令 | x86 操作类型基线 | 适用场景 |
|---|---|---|---|---|
| `relaxed` | 只保证原子性 | `lock xadd`（RMW） | RMW 基线 | 计数器、统计 |
| `acquire` | load—后续不重排到前面 | `mov` | **Load 基线（免费）** | 读 flag/spin_lock |
| `release` | store—前面不重排到后面 | `mov` | **Store 基线（免费）** | 写 flag/spin_unlock |
| `acq_rel` | RMW—既有 acquire 又有 release | `lock xadd`（同 relaxed） | RMW 基线 | RMW 操作 |
| `consume` | 依赖序（弱 acquire） | 同 acquire（编译器降级） | — | RCU（内核） |
| `seq_cst` | 全局单一顺序（默认） | `xchg`/`mov;mfence` | **RMW 10-20x**（相对于 relaxed） | 默认安全 |

**选择原则**：
- 默认 `seq_cst`——正确性优先
- 计数器/统计 → `relaxed`
- flag 生产者/消费者 → `release` 写 + `acquire` 读
- RMW → `acq_rel`

## 4. 经典模式

### 4.1 自旋锁——从零实现

S3Ch01 介绍了 `pthread_spinlock_t`。现在用 C11 原子操作从零实现一个：

```c
#include <stdatomic.h>

typedef atomic_int spinlock_t;
#define SPINLOCK_INIT 0

void spinlock_lock(spinlock_t *lock) {
    while (atomic_exchange_explicit(lock, 1, memory_order_acquire) == 1) {
        // 忙等——反复尝试获取锁
        // __builtin_ia32_pause();  // x86: 提示 CPU 这是自旋循环
    }
}

void spinlock_unlock(spinlock_t *lock) {
    atomic_store_explicit(lock, 0, memory_order_release);
}
```

**为什么 `lock=acquire`、`unlock=release`**？获取锁后需要看到之前持有者写的所有数据—这些数据在 unlock(release) 时保证可见，在 lock(acquire) 时保证对当前线程可见。

### 4.2 无锁计数器——relaxed 的正确用法

```c
atomic_uint request_count = 0;

void handle_request(void) {
    uint32_t seq = atomic_fetch_add_explicit(&request_count, 1, memory_order_relaxed);
    log_request(seq);  // seq 单调递增，不依赖其他变量的顺序
}
```

这里 `request_count` 只是单调递增的序列号——不需要 acquire/release 因为不看其他变量的值。

### 4.3 flag 通知——acquire/release 的典型

```c
char shared_data[1024];
atomic_bool data_ready = false;

void producer(void) {
    prepare_data(shared_data);
    atomic_store_explicit(&data_ready, true, memory_order_release);
}

void consumer(void) {
    while (!atomic_load_explicit(&data_ready, memory_order_acquire));
    process_data(shared_data);  // 保证能看到 producer 写入的数据
}
```

## 5. 原子操作的内核实现

### 5.1 x86 的 `lock` 前缀

x86 硬件提供 `lock` 前缀指令——在前面的指令执行期间锁定内存总线（或缓存行），防止其他 CPU 同时修改同一内存位置：

```
lock cmpxchg [addr], new_val    # CAS—比��并交换
lock xadd   [addr], inc         # 原子加
lock inc    [addr]              # 原子递增
```

**机制**：不是"锁总线"——现代 x86 使用 **MESI 缓存一致性协议** + `lock` 前缀的缓存行锁定。`lock` 指令只锁定目标缓存行（64 字节），不阻塞对其他位置的访问。

### 5.2 ARM 的 LL/SC 模式

ARM 没有 `lock` 前缀——它用 LL/SC（Load-Linked / Store-Conditional）模式：

```
LDREX r0, [addr]      # 1. Load-Linked: 读值 + 标记 addr 为"独占监视"
... (其他指令)
STREX r1, r0, [addr]  # 2. Store-Conditional: 仅当 addr 没有被其他 CPU 修改时才写
                      #    r1=0 成功, r1=1 失败 → 需要循环重试
```

**为什么 ARM 用 LL/SC**：没有硬件的缓存行锁定——依赖"监视独占"机制。设备简单、低功耗，但 CAS 循环需要��式写（C 代码中看不到，编译器自动生成 `LDREX/STREX` 循环重试）。

### 5.3 Linux 内核的原子操作

内核有自己的原子操作 API—不使用 C11 `<stdatomic.h>`（内核直到 2022 年才升级到 C11）：

```c
#include <linux/atomic.h>

atomic_t counter = ATOMIC_INIT(0);
atomic_inc(&counter);                    // 原子++
int v = atomic_read(&counter);          // 原子读
int old = atomic_cmpxchg(&counter, 0, 1); // CAS

// 内核的隐性内存屏障
smp_mb();  // 全内存屏障
smp_wmb(); // write barrier
smp_rmb(); // read barrier
```

**与 C11 的对应**：`READ_ONCE`/`WRITE_ONCE` ≈ `memory_order_relaxed`（仅原子性，无顺序保证）。`smp_store_release`/`smp_load_acquire` ≈ `memory_order_release`/`acquire`。

## 6. 跨语言对照：C11 atomic vs JMM vs Rust

| 概念 | C11 `<stdatomic.h>` | Java `java.util.concurrent.atomic` | Rust `std::sync::atomic` |
|---|---|---|---|
| 原子类型 | `atomic_int` | `AtomicInteger` | `AtomicI32` |
| CAS | `atomic_compare_exchange_strong` | `compareAndSet` | `compare_exchange` |
| 计数 | `atomic_fetch_add` | `getAndAdd` | `fetch_add` |
| acquire | `memory_order_acquire` | `volatile` read（`get()` on `AtomicX`） | `Ordering::Acquire` |
| release | `memory_order_release` | `volatile` write（`set()` on `AtomicX`） | `Ordering::Release` |
| seq_cst | `memory_order_seq_cst`（默认） | `volatile` + synchronized（组合效果） | `Ordering::SeqCst`（默认） |

**关键差异**：C11 没有"happens-before"在 volatile 上——Java 的 `volatile` 提供 acquire/release 语义，但 C 的 `volatile` 不提供任何线程间可见性保证（参见 S1Ch02 §9）。这是 Java 工程师转 C 并发编程时最常见的错误认知。

## 7. 总结

| 概念 | 一句话 |
|---|---|
| `_Atomic` | 编译器保证每次访问是原子的 + 默认 seq_cst |
| CAS | `compare_exchange_strong`——无锁算法的原子原语 |
| relaxed | 只保证原子性——计数器专用，0 顺序保证 |
| acquire/release | load/store 配对——生产者-消费者的同步协议 |
| seq_cst | 全局单一顺序（默认）——x86 上有 10-20x 性能代价 |
| x86 `lock` prefix | 缓存行锁定——MESI 协议保证原子性 |
| ARM LL/SC | `LDREX`/`STREX` 循环——依赖独占监视器 |
| 内核 atomic | `atomic_t` + `smp_mb()`——不使用 C11 接口但语义等价 |
| JMM vs C11 | Java `volatile`=acquire/release，C `volatile`=无同步语义 |

**核心观点**：原子操作用"一条 CPU 指令"消除 race condition——不是在软件层面加锁，而是利用硬件提供的原子指令（x86 `lock`、ARM `LDREX/STREX`）。内存序决定了编译器可以如何重排指令——它是 CPU 内存模型和 C 标准之间的桥梁。技术专家不是"知道 API 怎么用"，而是"知道 seq_cst 在 x86 上有多贵、知道 relaxed 为什么对计数器是安全的"。
