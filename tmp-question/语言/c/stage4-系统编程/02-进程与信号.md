## 1. 引言：为什么 Java 没有 `fork`？

Java 创建一个子线程只消耗 ~1MB 栈空间和几微秒——线程和主线程共享所有变量、文件描述符、信号处理。C 用 `fork` 创建子进程——它完整复制父进程的地址空间、文件描述符表和信号处理——代价是微秒级到毫秒级（取决于父进程内存占用）。

**为什么 C 用 fork 而非 thread？** Unix 的历史路径：1970 年 Unix 设计时没有"线程"的概念——fork 是唯一的"新执行流"机制。线程（pthread）直到 1995 年 POSIX 标准化时才有。至今 fork 仍是 Linux 上创建进程的标准方式。理解 fork 才能理解 Linux 的进程模型——每个进程都是一个独立的 task_struct，通过页表隔离。

## 2. `fork`：进程的克隆

### 2.1 基本机制

```c
#include <unistd.h>
#include <stdio.h>

pid_t pid = fork();
if (pid == 0) {
    // 子进程——pid==0
    printf("Child:  pid=%d, ppid=%d\n", getpid(), getppid());
} else if (pid > 0) {
    // 父进程——pid==子进程的 PID
    printf("Parent: pid=%d, child=%d\n", getpid(), pid);
} else {
    perror("fork failed");
}
```

**fork 返回两次**：在父进程中返回子进程 PID，在子进程中返回 0——这是 Unix 独特的"调用一次、返回两次"语义。

### 2.2 fork 的低层实现：COW

`fork` 不会立即复制整个父进程的地址空间——它用 **COW（Copy-On-Write，写时复制）**：父子进程共享同一份物理内存页，标记为"只读"。当任一进程写入某页时，CPU 的 MMU 触发**页错误**→内核分配新页→复制原始页→更新页表——这一过程对用户态完全透明。

```c
int x = 42;
if (fork() == 0) {
    x = 100;   // 触发 COW 页错误——子进程获得 x 的独立副本
    printf("Child: x=%d (addr=%p)\n", x, &x);
} else {
    sleep(1);
    printf("Parent: x=%d (addr=%p)\n", x, &x);  // x=42——未修改
    // 注意：两个进程中的 &x 显示的虚拟地址相同——物理地址不同！
}
```

**重要**：COW 使得 fork 在"父进程内存大、子进程大多是只读访问"场景中极为高效——只复制实际被写入的页。

### 2.3 fork 后子进程继承什么

| 继承 | 不继承 |
|---|---|
| 打开的文件描述符（共享偏移量—见 S4Ch01 §4.2） | 父进程 PID（`getppid()` 在子进程中返回父进程 PID） |
| 信号处理函数（`SIGINT` handler 等） | 未决信号（pending signals—被清除） |
| 当前工作目录 | 文件锁（`flock` —由父进程持有，子进程不继承） |
| 环境变量 | 进程计时器（CPU usage —被重置） |
| 内存映射（`mmap`—COW 共享） | 线程—fork 只复制**调用 fork 的那个线程** |

**经典陷阱**：`fork` 在多线程程序中只复制**调用线程**——其他线程在子进程中不存在。如果被"蒸发"的线程持有 mutex，子进程永远拿不到那把锁→死锁。解决方案：`pthread_atfork` 注册 fork 前后的 handler。

## 3. `exec`：替换进程镜像

### 3.1 基本用法

fork 创建新进程，`exec` 将新进程的代码替换为另一个可执行文件——fork+exec 是 Unix 创建新程序的经典配方：

```c
pid_t pid = fork();
if (pid == 0) {
    // 子进程
    execlp("ls", "ls", "-l", NULL);  // 替换为 ls 程序
    perror("exec failed");             // 只有 exec 失败才会执行到这里
    _exit(127);
}
// 父进程继续
waitpid(pid, NULL, 0);
```

**六种 exec 变体**：

| 函数 | 路径解析 | 参数传递 |
|---|---|---|
| `execl` | 绝对路径 | `execl("/bin/ls", "ls", "-l", NULL)` — 列表形式 |
| `execv` | 绝对路径 | `execv("/bin/ls", argv)` — 数组形式 |
| `execle` | 绝对路径 | `execle(...)` — 列表+自定义环境变量 |
| `execlp` | `$PATH` 搜索 | 同上列表，自动找可执行文件 |
| `execvp` | `$PATH` 搜索 | 同上数组 |
| `execvpe` | `$PATH` 搜索 | 数组+自定义环境 |

### 3.2 exec 后子进程保留什么

| 保留 | 被重置 |
|---|---|
| **进程 PID**（不���） | 信号处理函数 — 全部重置为 `SIG_DFL`（默认处理） |
| 打开的**文件描述符**（除非 `FD_CLOEXEC` 被设置） | **`SIG_IGN` 信号保留** — 被忽略的信号仍被忽略 |
| 当前工作目录 | 内存映射（`mmap`—全部被清除） |
| 进程组 ID、会话 ID | 线程—全部被清除 |

**关键**：`SIG_IGN` 被保留是个微妙但重要的细节。如果父进程调用了 `signal(SIGCHLD, SIG_IGN)`，子进程在 exec 之后这个设置不变。但 `SIG_DFL` 的手动 handler 会被清除。

## 4. `wait`：回收子进程

### 4.1 基本用法

```c
pid_t pid = fork();
if (pid == 0) { /* child works */ _exit(42); }

int status;
waitpid(pid, &status, 0);

if (WIFEXITED(status))
    printf("Exit code: %d\n", WEXITSTATUS(status));  // 42
if (WIFSIGNALED(status))
    printf("Killed by signal: %d\n", WTERMSIG(status));
```

**不 wait 的后果**：子进程退出后变成**僵尸进程（zombie）**——task_struct 仍保留在内核中（占用 PID 和少量内存），直到父进程 `wait` 回收。如果父进程长期不回收→僵尸进程积压→PID 耗尽。

### 4.2 防止僵尸的好方法

```c
// 方法 1：忽略 SIGCHLD—内核自动回收
signal(SIGCHLD, SIG_IGN);

// 方法 2：双 fork—孙进程被 init（PID 1）收养
if (fork() == 0) {
    if (fork() == 0) {
        // 孙进程—做实际工作
        sleep(10);
        _exit(0);
    }
    _exit(0);  // 子进程立即退出，init 收养孙进程
}
wait(NULL);  // 父进程只等子进程（已退出）

// 方法 3：循环 waitpid + WNOHANG 非阻塞回收
while (waitpid(-1, NULL, WNOHANG) > 0);
```

## 5. 信号：异步中断

### 5.1 信号的本质

信号是内核向进程发送的**异步通知**——进程不需要轮询，由内核在"合适的时刻"（从内核态返回用户态时）投递。

```c
#include <signal.h>

void handler(int sig) {
    // 🔑 只能做"异步信号安全"的操作
    write(1, "Got signal!\n", 12);  // ✓ write 是异步信号安全的
    // printf("Got signal!\n");      // ✗ printf 不是！
}

signal(SIGINT, handler);   // Ctrl+C → handler
signal(SIGTERM, handler);  // kill 默认信号
```

### 5.2 异步信号安全函数

**只有 ~110 个 POSIX 函数是异步信号安全的**。常见的**不安全**函数：

| 不安全函数 | 为什么 | 替代 |
|---|---|---|
| `printf` / `fprintf` | stdio 缓冲区可能被信号中断时持有锁 | `write(1, ...)` |
| `malloc` / `free` | 堆分配器内部锁 | 预分配缓冲区 |
| `pthread_mutex_lock` | 死锁——信号可能打断了持有同一锁的代码 | 用 `sig_atomic_t` flag |

**信号处理器中只能做的事**：
1. 设置 `volatile sig_atomic_t flag = 1`（通知主循环）
2. 调用 `write` / `read` / `open` / `close`（少数 syscall 是安全的）
3. 调用 `_exit(0)`（立即退出）

### 5.3 关键信号的语义

| 信号 | 默认行为 | 可捕获？ | 典型用途 |
|---|---|---|---|
| `SIGINT`（Ctrl+C） | 终止 | 可 | 用户中断 |
| `SIGTERM`（kill 默认） | 终止 | 可 | 优雅关闭 |
| `SIGKILL`（kill -9） | 终止 | **不可** | 强制杀死（最后手段） |
| `SIGSTOP`（Ctrl+Z） | 暂停 | **不可** | 作业控制 |
| `SIGCHLD` | 忽略 | 可 | 子进程退出通知 |
| `SIGPIPE` | 终止 | 可 | 写已关闭的 pipe—忽略防崩溃 |
| `SIGHUP` | 终止 | 可 | 终端关闭 / 重载配置文件 |
| `SIGSEGV` | 终止+core dump | 可 | 段错误—调试用 handler |

### 5.4 sigaction——signal 的现代替代

`signal(2)` 在不同 Unix 变体上行为不一致（System V vs BSD 语义）。用 `sigaction` 替代：

```c
struct sigaction sa = {
    .sa_handler = handler,
    .sa_flags   = SA_RESTART     // 自动重试被中断的 read/write
};
sigemptyset(&sa.sa_mask);        // 处理期间不阻塞额外信号
sigaction(SIGTERM, &sa, NULL);
```

`SA_RESTART` 标志的关键作用：被信号中断的 `read` 不会返回 `EINTR`——内核自动重试。但**仅对 syscall 有效**，不影响 `poll`/`select`。

### 5.5 阻塞与等待信号：`sigprocmask` + `sigsuspend`

**阻塞信号**（不同于忽略信号——阻塞的信号被推迟，忽略的信号被丢弃）：

```c
sigset_t mask;
sigemptyset(&mask);
sigaddset(&mask, SIGINT);
sigprocmask(SIG_BLOCK, &mask, NULL);
// SIGINT 在此期间被阻塞——Ctrl+C 无效果
sigprocmask(SIG_UNBLOCK, &mask, NULL);
// 阻塞期间到达的 SIGINT 此时被投递
```

**等待信号——sigsuspend**：原子的"解除阻塞+等待信号"。没有 sigsuspend 的方案（先解除再 pause）有竞态条件——信号在 sigprocmask 和 pause 之间到达→丢失。

```c
sigset_t old_mask, wait_mask;
sigprocmask(SIG_BLOCK, &mask, &old_mask);  // 阻塞 SIGINT
while (!flag) {
    sigsuspend(&old_mask);  // 等待 SIGINT——被信号唤醒后重新回到 old_mask
}
```

### 5.6 `kill(2)`——不只是"发信号给一个进程"

```c
kill(pid, sig);
```

| pid 值 | 含义 |
|---|---|
| `pid > 0` | 发送信号给 PID=pid 的进程 |
| `pid = 0` | 发送信号给**同一进程组**的所有进程 |
| `pid = -1` | 发送信号给**所有有权限**的进程（广播，仅 root/CAP_KILL） |
| `pid < -1` | 发送信号给进程组 ID = -pid 的所有进程 |

**权限检查**：发送者必须是 root，或 uid == 目标的 uid，或 sender uid == target euid（有效 UID）。

### 5.7 实时信号：排队 + 携带数据

标准信号（SIGINT 等）不排队——同一信号在未处理期间再次到达→只记录一次。实时信号（`SIGRTMIN` ~ `SIGRTMAX`，共 32 个）排队且携带附加数据：

```c
union sigval val = { .sival_int = 42 };
sigqueue(target_pid, SIGRTMIN, val);  // 携带一个 int 数据
// 接收端：siginfo_t 中包含 si_value
```

## 6. 进程组与会话

进程组是 shell 作业控制的基础——Ctrl+C 发送 `SIGINT` 给**前台进程组的所有进程**，Ctrl+Z 发送 `SIGSTOP`。

```c
setpgid(pid, pgid);     // 将 pid 加入进程组 pgid
setsid();               // 创建新会话——新进程组+脱离终端
tcsetpgrp(fd, pgid);    // 设置终端的���前台进程组
```

**会话**：多个进程组的集合——会话 leader 控制终端（tty）。当终端连接断开时，内核发送 `SIGHUP` 给前端进程组——这正是 daemon 需要用 `setsid` 脱离原会话的原因。

## 7. 进程资源限制

生产环境中必须限制进程的资源使用——避免单个进程耗尽全系统的 fd 或内存：

```c
#include <sys/resource.h>
struct rlimit lim = { .rlim_cur = 1024, .rlim_max = 4096 };
setrlimit(RLIMIT_NOFILE, &lim);    // 限制打开文件数——防止 fd 泄漏耗尽系统
setrlimit(RLIMIT_CPU, &lim);       // 限制 CPU 时间——防止死循环
setrlimit(RLIMIT_AS, &lim);        // 限制地址空间——防止 OOM
```

查看当前限制：`getrlimit` 或 `ulimit -a`（shell 命令）。SIGXCPU 信号在 CPU 限制超时后发送。

## 9. 总结

| 概念 | 一句话 |
|---|---|
| `fork` | 复制当前进程—返回两次（父=子PID、子=0），COW 避免不必要的物理内存复制 |
| `exec` | 替换当前进程的代码和数据—保留 PID、fd（除非 FD_CLOEXEC） |
| `waitpid` | 回收子进程—防僵尸进程 |
| 僵尸进程 | 已退出但父进程未 wait——占用 PID，可用 SIGCHLD+IGN 自动回收 |
| 信号 | 异步中断—处理器只能做异步信号安全的操作（~110 个 syscall） |
| `sigprocmask` | 阻塞信号—推迟处理，不同于 SIG_IGN 忽略 |
| `kill(2)` | 四种 pid 语义—不只是一个进程 |
| 实时信号 | 排队+携带数据—SIGRTMIN ~ SIGRTMAX |
| 孤儿进程 | 父进程先退出—init 收养，自动回收 |
| daemon | fork+setsid+chdir+close+umask—标准 6 步创建守护进程 |
| 进程组/会话 | shell 作业控制基础—Ctrl+C 发信号给前台进程组所有成员 |
| rlimit | 限制进程资源—NOFILE/CPU/AS 防耗尽 |
| `sigaction` | signal 的现代替代—SA_RESTART 自动重试被中断的 syscall |

**核心观点**：进程是 Linux 的基本隔离单元——fork 复制、exec 替换、wait 回收。信号是唯一的进程间异步通信机制——因为异步所以危险：信号处理器中的代码受到严格约束。技术专家不是"知道 fork 怎么用"，而是"知道多线程下 fork 可能死锁（被蒸发的线程持有的锁）、知道 exec 后 SIG_IGN 被保留但 handler 被清除、知道信号处理器中不能调 printf"。
