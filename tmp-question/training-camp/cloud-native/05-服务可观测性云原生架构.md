## 一、Logging：统一抽象 + 生产实践

### 1.1 训练营方向：Microsphere Logging 统一 API

训练营设计了跨 JDK/Log4j2/Logback 的统一 Logger 抽象：

```
io.microsphere.logging.Logger          — 统一 API（类似 SLF4J）
  └── LoggerFactory（SPI）              — 工厂，按优先级选择实现
       ├── SLF4jLoggerFactory          — 最高优先级（多数场景命中）
       ├── ACLLoggerFactory            — 中间优先级
       └── JDKLoggerFactory            — 最低优先级（兜底）
```

**设计价值**：当一个库需要日志输出时，不硬编码依赖 SLF4J 或 Log4j2——通过 SPI 自动适配运行环境中的日志框架。

### 1.2 日志过滤器 SPI

```
io.microsphere.logging.filter.Filter
  → 使用场景1：内容替换（脱敏：手机号 → 138****1234）
  → 使用场景2：内容删除（过滤心跳日志）
  → 使用场景3：内容补充（自动注入 userId/traceId）
```

### 1.3 JMX LoggingMXBean 多框架适配

Java 标准 `java.lang.management.PlatformLoggingMXBean` 仅适配 JDK Logger。Microsphere 扩展了 logback/log4j2/log4j 的 JMX 管理：

| 框架 | ObjectName domain | type 属性 |
|---|---|---|
| JDK Logger | `java.util.logging` | `Logging` |
| Logback | `ch.qos.logback.classic` | `Logger` |
| Log4j2 | `org.apache.logging.log4j` | `Logger` |
| Log4j | `org.apache.log4j` | `Logger` |

**核心操作**：`getLoggerLevel(name)` / `setLoggerLevel(name, level)` —— 运行时动态调整日志级别，无需重启。

### 1.4 生产级日志架构

```
应用（Logback JSON Encoder）
  → Kafka（缓冲层，防止 Logstash 背压）
    → Logstash（解析+过滤+富化）
      → ElasticSearch（热数据 7 天）
        → ES Snapshot → S3（冷数据长期存储）
    → Kibana（可视化查询）

或更轻量的替代：
应用（Logback JSON Encoder）
  → Promtail（K8s DaemonSet 自动采集）
    → Loki（标签索引 + 内容压缩）
      → Grafana（统一查询+告警）
```

**Kafka 作为缓冲层的价值**：
- Logstash 处理慢时，Kafka 缓冲不丢日志
- 日志突发峰值时，Kafka 吸收峰值
- Logstash 升级/重启时，Kafka 保留未处理日志

### 1.5 结构化日志完整配置

```xml
<!-- logback-spring.xml -->
<appender name="JSON" class="ch.qos.logback.core.ConsoleAppender">
    <encoder class="net.logstash.logback.encoder.LogstashEncoder">
        <includeMdcKeyName>traceId</includeMdcKeyName>
        <includeMdcKeyName>spanId</includeMdcKeyName>
        <includeMdcKeyName>userId</includeMdcKeyName>
    </encoder>
</appender>
```

```json
{"timestamp":"2025-01-15T10:23:45.123Z","level":"INFO","message":"订单创建","traceId":"a1b2","userId":"123"}
```

## 二、Metrics：Micrometer 核心 + SLO

### 2.1 Micrometer 四种计量器

| 类型 | 场景 | API |
|---|---|---|
| Counter | 错误数、订单总量 | `registry.counter("orders.total").increment()` |
| Timer | HTTP RT、DB 耗时百分位 | `Timer.builder("http.requests").publishPercentiles(0.5,0.95,0.99).register(registry)` |
| Gauge | 线程池活跃数、连接池占用 | `Gauge.builder("pool.active", pool, Pool::getActive).register(registry)` |
| DistributionSummary | 请求 body 大小 | `summary.record(size)` |

### 2.2 Observation API（Spring Boot 3+）

Micrometer 1.10+ 引入 Observation API，统一 metrics + tracing：

```java
Observation.createNotStarted("order.create", registry)
    .lowCardinalityKeyValue("status", "success")   // 低基数 tag（指标用）
    .highCardinalityKeyValue("orderId", "123")     // 高基数 tag（追踪用）
    .observe(() -> orderService.create(order));
```

一个 API 同时输出 Counter/Timer（给 Prometheus）和 Span（给 Zipkin/Tempo）。

### 2.3 SLO 烧钱率

| 概念 | 示例 | 公式 |
|---|---|---|
| SLI | P99 = 850ms | 实际测量 |
| SLO | P99 < 1000ms | 目标 |
| Error Budget | 每月允许 43m 不达标 | (1 - SLO%) × 30 天 |

**从 0 到 1：为 order-service 定义 SLO 的完整步骤**

**第一步：确定"用户体感"**：用户最在乎的是什么？订单创建成功？还是页面加载速度？用户不在乎你内部某个方法的耗时。选对了指标，SLO 才有意义。

**第二步：选定 SLI（指标）**：

| SLI 指标 | 采集方式 | 衡量什么 |
|---|---|---|
| **可用性** = 成功请求 / 总请求 | `http_server_requests_total{status!~"5.."}` ÷ `http_server_requests_total` | "服务活着吗" |
| **P99 延迟** = 最慢 1% 请求的耗时 | `histogram_quantile(0.99, rate(http_server_requests_seconds_bucket[5m]))` | "慢不慢" |
| **错误率** = 错误请求 / 总请求 | `rate(http_server_requests_total{status=~"5.."}[5m])` | "有没有出错" |

**第三步：看历史数据定 SLO**：SLI 不是拍脑袋定的，是从实际数据里来的：

```promql
# 查过去 30 天的 P99 延迟
histogram_quantile(0.99, rate(http_server_requests_seconds_bucket[30d]))
# 结果：P99 = 850ms
# 定 SLO：P99 < 1000ms（留 15% Buffer）
```

**第四步：配 Misometer + Prometheus**：

```yaml
management:
  metrics:
    distribution:
      slo:
        http.server.requests: 100ms, 500ms, 1000ms, 2000ms
      percentiles-histogram:
        http.server.requests: true
```

**第五步：配告警**（什么时候该紧张）：

```yaml
- alert: SLOBurningFast
  expr: |
    (1 - sum(rate(http_server_requests_total{status!~"5.."}[1h])) 
    / sum(rate(http_server_requests_total[1h]))) 
    > 0.01 * 14.4  # 1 小时内烧了 1% 月度预算的 14.4 倍 → 紧急
  labels: {severity: critical}
```

**关键**：SLO 不是"100% 可用"，是"允许失败但控制在可接受范围内"。99.9% = 每月允许 43 分钟不可用。用这个数据决定"要不要半夜叫醒工程师"。

**Prometheus 规则**：
```yaml
management.metrics.distribution.slo.http.server.requests: 100ms, 500ms, 1000ms, 2000ms
```
查询：`histogram_quantile(0.99, rate(http_server_requests_seconds_bucket[5m]))`

## 三、Tracing：SkyWalking + OpenTelemetry

### 3.1 SkyWalking Agent 字节码增强

SkyWalking 通过 Java Agent + Byte Buddy 在**不修改应用代码**的情况下实现链路追踪：

```
类加载时 → Byte Buddy 拦截目标方法
  → 创建 EntrySpan（入口）
    → 注入 traceId 到 MDC
      → 传播到下游（HTTP Header / Dubbo Attachment）
        → 创建 ExitSpan（出口）
          → 上报到 OAP Server
```

**为什么 SkyWalking 在生产中流行**：
- 零代码改动（比 Sleuth 的 `@NewSpan` 注解方案更无侵入）
- 内置性能分析（线程 dump、慢 SQL 检测）
- 拓扑自动发现（不依赖人工配置）

### 3.2 OpenTelemetry：统一标准

```
应用(OTel SDK)
  → OTel Collector（接收/处理/导出）
    → Traces → Jaeger/Tempo
    → Metrics → Prometheus
    → Logs → Loki/ES
```

**2025 年选择**：已有 SkyWalking 保留；新项目用 **OTel + Grafana**（Tempo+Loki+Prometheus 一站式）。

### 3.3 采样策略

| 策略 | 方式 | 适用场景 |
|---|---|---|
| **头采样**（Head-based） | 入口决定是否采样（固定比例 10%） | 低流量，简单 |
| **尾采样**（Tail-based） | 全部采集，存储时保留错误+高延迟 | 高流量，不丢异常 |
| **自适应采样** | 正常请求 1%、错误请求 100% | 生产推荐 |

## 五、告警体系

### 5.1 Prometheus Alertmanager 分级

```yaml
groups:
- name: order-service
  rules:
  - alert: HighErrorRate
    expr: rate(http_server_requests_total{status=~"5.."}[5m]) > 0.01
    for: 5m
    labels: {severity: critical}
  - alert: HighP99Latency
    expr: histogram_quantile(0.99, rate(http_server_requests_seconds_bucket[5m])) > 2
    for: 10m
    labels: {severity: warning}
```

### 5.2 告警抑制

`critical` 触发时自动抑制同服务 `warning`。防止一个 DB 故障产生 50 条告警。

### 5.3 通知渠道

Alertmanager → Webhook → 企业微信/钉钉/飞书/邮件/PagerDuty。

## 附：查漏补缺

### 高基数指标 —— Prometheus 的头号敌人

Prometheus 每个 **指标名 + 标签组合** = 一个时间序列。每个时间序列消耗内存。而标签组合的数量 = 标签值的笛卡尔积。

**高基数陷阱**：

```java
// ❌ 致命错误：userId 作为标签
Counter.builder("orders.total")
    .tag("userId", userId)   // 用户有百万级！百万级时间序列！
    .register(registry);
```

一个 `userId` 标签 = 百万级时间序列 = Prometheus 内存爆炸 + 查询超时。

**正确做法**：
```java
// ✅ userId 作为 highCardinalityKeyValue（只出现在 tracing，不产生时间序列）
Observation.createNotStarted("order.create", registry)
    .lowCardinalityKeyValue("status", "success")     // 做 Metric 标签
    .highCardinalityKeyValue("userId", "123")        // 只做 Tracing 标签
    .observe(() -> orderService.create(order));
```

**经验规则**：
- 标签值 < 100 种 → 可以做标签（`status=success/fail`，`method=GET/POST`）
- 标签值 100-1000 种 → 谨慎使用（`instance`，`pod`）
- 标签值 > 1000 种 → **禁止做标签**（userId，orderId，traceId）

**为什么这个知识点重要**：这是 Prometheus 在生产环境最常见的故障原因。"昨天加了 userId 标签做监控，今天 Prometheus 挂了"——几乎每个团队都经历过。

### 业务指标 vs 技术指标

运维看技术指标（QPS/P99/错误率），产品和业务看业务指标（订单量/支付转化率/用户活跃数）。两者不能混用，但需要关联。

**清晰的分层**：

| 层 | 指标示例 | 谁看 | 存储 |
|---|---|---|---|
| **业务层** | 订单量、GMV、支付转化率、DAU | PM/运营/老板 | 数据仓库（ClickHouse/Hive） |
| **应用层** | 下单接口 QPS、P99 延迟、错误率 | 研发/运维 | Prometheus + Grafana |
| **基础设施层** | CPU/Memory、Pod 重启次数、网络流量 | 运维/SRE | Prometheus Node Exporter + kube-state-metrics |

**关联方式**：当业务指标异常时（如下单量下降 30%），通过 traceId 关联到应用层——"是哪个服务慢了？哪条 SQL 慢了？"

**为什么需要分层**：
- 数据仓库不适合存储秒级监控数据（成本高）
- Prometheus 不适合存储月度/年度趋势数据（数据过期清理）
- 不能把"用户活跃数"和"JVM GC 次数"放在同一个月仪表板上——不同角色看不同的东西
