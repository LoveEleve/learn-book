## 一、Sentinel 核心架构

### 1.1 Slot 链机制

```
SphU.entry("resourceName")
  → NodeSelectorSlot    — 构建资源调用树
  → ClusterBuilderSlot  — 统计信息（RT/QPS/线程数）
  → StatisticSlot       — LeapArray 滑动窗口实时指标
  → FlowSlot            — QPS/线程数/关联/链路 流控
  → DegradeSlot         — 慢调用/异常比例/异常数 熔断
  → SystemSlot          — Load/CPU/RT 系统自适应保护
  → 业务逻辑
```

### 1.2 滑动窗口（LeapArray）

`AtomicReferenceArray<WindowWrap<MetricBucket>>` — `sampleCount × intervalInMs` 窗口大小，原子替换过期窗口。

### 1.3 Sentinel 框架扩展点

| 框架 | 扩展接口 | resourceName 生成规则 |
|---|---|---|
| JDBC | P6Spy `JdbcEventListener` / Druid `FilterAdapter` | 表名 + DML 操作类型 |
| MyBatis | `Interceptor.plugin()` 包装 `Executor` | Mapper 方法全限定名 |
| Hibernate | `Interceptor` + `DelegatingInterceptor` 装饰器 | Entity + 操作类型 |
| Dubbo | `Filter` SPI（接口级） | 服务接口 + 方法名 |
| Feign | `RequestInterceptor` + `Client` + `FeignDecorator` | 请求 URL |

## 二、Resilience4j：区别于 Sentinel

### 2.1 6 模块 + Registry 模式

```
CircuitBreaker    — 慢调用/失败率，HALF_OPEN 探测恢复
Bulkhead          — 信号量（SemaphoreBulkhead）vs 线程池（FixedThreadPoolBulkhead）
RateLimiter       — 令牌桶，AtomicReference<ActiveLimiter> 原子补充
TimeLimiter       — CompletableFuture.timeout()
Retry             — 指数退避 IntervalFunction.ofExponentialBackoff
Cache             — 函数结果缓存

Registry<Entry, Config> — 存储 Entry 实例（computeIfAbsent 惰性创建）
  └── InMemoryCircuitBreakerRegistry 等 6 个内建实现
```

### 2.2 Sentinel vs Resilience4j

| 维度 | Sentinel | Resilience4j |
|---|---|---|
| 流控 | QPS/线程数/关联/链路 | RateLimiter 令牌桶 |
| 系统保护 | ✅ Load/CPU/RT | ❌ |
| 集群限流 | ✅ Token Server | ❌ |
| Bulkhead 线程池隔离 | ❌ | ✅ FixedThreadPoolBulkhead |
| Retry 重试 | ❌ | ✅ IntervalFunction |
| 控制台 | Dashboard | 需自建（如 Microsphere DevOps） |
| 2025 推荐场景 | SCA 生态，需要全局限流 | 独立项目，需要线程池隔离+重试 |

### 2.3 FeignDecorator 整合模式

```java
FeignDecorators.builder()
    .addCircuitBreaker(circuitBreaker)
    .addRateLimiter(rateLimiter)
    .addBulkhead(bulkhead)
    .build()
    → 每个 FeignDecorator 包裹前一个 CheckedFunction → 洋葱模型责任链
```

**多扩展共存问题**：`Resilience4jFeign`、`SentinelFeign`、`FeignCircuitBreaker` 三者无法共存——每个 Builder 创建独立的 Feign 对象。

### 2.4 Resilience4j → Spring 事件桥接

```
Resilience4j EventConsumer
  → Resilience4jEventApplicationEventPublisher
    → Spring ApplicationEventPublisher
      → @EventListener 监听
```

内建 5 个 Publisher：CircuitBreaker / Bulkhead / RateLimiter / TimeLimiter / Retry。

## 附：深度展开

### 一、Sentinel Slot 链逐个解析

训练营第 3 篇详细讲了 Slot 链。每个 Slot 的源码职责：

| Slot | 源码位置 | 核心方法 | 作用 |
|---|---|---|---|
| `NodeSelectorSlot` | `sentinel-core/.../slots/NodeSelectorSlot.java` | `entry()` → 构建调用树（DefaultNode→ClusterNode） | 资源路径统计 |
| `ClusterBuilderSlot` | `sentinel-core/.../slots/ClusterBuilderSlot.java` | `entry()` → 创建/更新 ClusterNode | 集群级别指标 |
| `StatisticSlot` | `sentinel-core/.../slots/StatisticSlot.java` | `entry()` 记录 Pass/Block/RT | 实时指标采集 |
| `FlowSlot` | `sentinel-core/.../slots/block/flow/FlowSlot.java` | `checkFlow()` → 查 FlowRuleManager | 流控判断 |
| `DegradeSlot` | `sentinel-core/.../slots/block/degrade/DegradeSlot.java` | `entry()` → 查 CircuitBreaker 状态 | 熔断判断 |
| `SystemSlot` | `sentinel-core/.../slots/system/SystemSlot.java` | `entry()` → 查 SystemStatusListener | 系统自适应 |

**Slot 链构建机制**：`SpiLoader.load(ProcessorSlotChain.class)` → `DefaultSlotChainBuilder.build()` → 按 `@Spi(order=-10000)` 排序加载，用户可通过 SPI 插入自定义 Slot。

### 二、Resilience4j Registry 模式深度

```
Registry<E, C>  — 存储 Entry 实例和配置
├── find(name)              — 查找（不创建）
├── computeIfAbsent(name)   — 不存在时创建（惰性，返回已存在或新建）
├── addConfiguration(name, config) — 注册配置
└── getDefaultConfig()      — 获取默认配置（name="default"）

AbstractRegistry
├── ConcurrentHashMap<String, E> entryMap     — Entry 存储
├── ConcurrentHashMap<String, C> configs      — 配置存储
└── RegistryEventConsumer                    — 事件通知
```

**6 个内建实现**：`InMemoryCircuitBreakerRegistry` / `InMemoryBulkheadRegistry` / `InMemoryThreadPoolBulkheadRegistry` / `InMemoryRateLimiterRegistry` / `InMemoryTimeLimiterRegistry` / `InMemoryRetryRegistry`。

**为什么全部是 InMemory**：Resilience4j 设计哲学——Registry 只负责运行时管理，持久化通过 `RegistryEventConsumer` 事件外挂。状态恢复（如重启后重置 CircuitBreaker 状态）由外部系统负责。

### 三、FeignDecorator 洋葱模型

```java
// 构建层次
FeignDecorators.builder()
    .addCircuitBreaker(cb)    // 最外层：熔断保护
    .addRateLimiter(rl)       // 第二层：限流
    .addBulkhead(bh)          // 第三层：并发控制
    .addRetry(retry)           // 最内层：重试
    .build()
```

**执行顺序**（从外到内）：请求到达 → CircuitBreaker.isCallPermitted() → RateLimiter.acquirePermission() → Bulkhead.tryAcquire() → Retry → 实际调用。

**回调链**（从内到外）：调用完成 → Retry.onResult() → Bulkhead.onComplete() → RateLimiter.onComplete() → CircuitBreaker.onSuccess()/onError()。

### 四、Bulkhead 线程池隔离 vs Sentinel 信号量隔离

| 维度 | Resilience4j FixedThreadPoolBulkhead | Sentinel 并行流控 |
|---|---|---|
| 隔离粒度 | **物理隔离**：独立线程池，故障不传播 | **逻辑隔离**：同一线程池，信号量限制 |
| 适用场景 | 下游服务不稳定，需要物理隔离防止线程池耗尽 | 下游服务稳定，防止瞬时并发冲高 |
| 线程模型 | 调用线程 ≠ 执行线程 | 调用线程 = 执行线程 |
| ThreadLocal | ⚠️ 无法传递（跨线程） | ✅ 可以传递（同线程） |
| 配置 | `maxThreadPoolSize` / `coreThreadPoolSize` / `queueCapacity` | `maxConcurrency`（线程数上限） |
| 选择建议 | 需要强隔离场景（如第三方不可控服务） | 需要 ThreadLocal 传递（如 TraceId） |

**为什么 Sentinel 选信号量而非线程池？**

Sentinel 的定位是**入口流量的轻量级拦截**，不是复杂的故障隔离。三个核心原因：

1. **ThreadLocal 传递**：分布式追踪的 traceId 存 ThreadLocal 里。如果 Sentinel 切线程，traceId 丢失——下游所有 Span 会断链。国内微服务基本都用 SkyWalking，信号量保证 traceId 不丢。

2. **性能**：信号量 `tryAcquire()/release()` 是纯内存操作，耗时 < 1μs。线程池需要切换线程、传递上下文，耗时 10-50μs。Sentinel 作为每条请求都要经过的拦截层，每微秒都重要。

3. **协议设计**：Sentinel 的设计哲学是"在同一个线程上做检查和统计"。`SphU.entry()` → `Entry.exit()` 必须成对出现在同一线程。线程池隔离违反了这个契约。

**那什么时候用线程池隔离？** 当被调方不可控时。比如调用一个第三方 HTTP API，它可能响应极慢、可能假死不释放连接。这时需要物理隔离——把第三方调用扔到独立线程池，即使线程池满了也不影响主业务。这就是 Sentinel 该让位给 Resilience4j FixedThreadPoolBulkhead 的时候。

### 五、拦截模式 — 环绕 vs 前后

训练营第 10 篇总结了容错框架的拦截模式分类：

| 模式 | 接口形态 | 典型实现 |
|---|---|---|
| **环绕拦截** | `invoke(proceed)` — 包裹整个调用 | FeignDecorator, Dubbo Filter, Sentinel Entry |
| **前置拦截** | `before()` — 调用前检查 | Bulkhead.tryAcquire(), FlowSlot.checkFlow() |
| **后置拦截** | `after()` — 调用后处理 | onSuccess()/onError() 事件回调 |

**上下文传递设计**：前置和后置拦截通常需要传递中间状态（如开始时间、尝试次数），常见方案是 `ThreadLocal` 存储 `Context` 对象。Spring WebMVC `HandlerInterceptor` 的 `preHandle() → postHandle() → afterCompletion()` 是最标准的"前后拦截+上下文传递"实现。

## 附二：查漏补缺

### 限流算法深度对比

Sentinel 和 Resilience4j 用了不同的限流算法。理解算法差异才能选对方案：

| 算法 | 原理 | 优势 | 劣势 | 谁在用 |
|---|---|---|---|---|
| **令牌桶** | 固定速率放令牌，请求须拿到令牌才能通过 | 允许突发流量（桶满了可一次性取出） | 实现复杂 | Resilience4j RateLimiter |
| **漏桶** | 请求进桶，固定速率流出 | 流量绝对平滑 | 不容忍突发 | Nginx limit_req |
| **滑动窗口计数** | 将时间片切分为小窗口（如 1s 切为 2 个 500ms），统计各窗口请求数 | 精度可调节（窗口越小越精确） | 内存开销 = 窗口数 × 计数器 | Sentinel LeapArray |
| **滑动窗口日志** | 记录每个请求的时间戳，统计窗口内记录数 | 最精确 | 内存开销大（存储每个请求的时间戳） | Redis Sorted Set 实现 |

**为什么 Sentinel 选滑动窗口计数**：

Sentinel 的 `LeapArray` 使用 `AtomicReferenceArray<WindowWrap<MetricBucket>>` 实现。每个窗口一个 `MetricBucket`（long 类型计数器 — 极低内存开销）。1000ms 切为 2 个窗口 = 每个窗口 500ms = 只需 2 个 MetricBucket。内存占用：2 × 8 字节 = 16 字节。

对比滑动窗口日志：如果 QPS = 10000，每秒需要存储 10000 个时间戳。1 个时间戳 = 8 字节 → 80KB/秒 → 4.8MB/分钟 → 内存爆炸。

**所以 Sentinel 的选择是：用精度换内存**。2 个窗口意味着流量曲线是阶梯式的（不是平滑曲线），但内存开销极低（16 字节），适合高 QPS 场景。

### Sentinel Slot 链的性能开销

每条请求都要经过 6 个 Slot。每个 Slot 的性能开销是多少？

| Slot | 操作 | 耗时（估算） |
|---|---|---|
| NodeSelectorSlot | 树节点查找 + 创建 | ~1μs |
| ClusterBuilderSlot | ClusterNode 创建/更新 | ~1μs |
| StatisticSlot | LeapArray.currentWindow() 原子操作 | ~2μs |
| FlowSlot | FlowRuleManager 查规则（Map get） | ~0.5μs |
| DegradeSlot | CircuitBreaker 状态检查 | ~1μs |
| SystemSlot | SystemStatusListener 当前值读取 | ~0.5μs |
| **总计** | | **~6μs/请求** |

**6μs 意味着什么？** 每条请求 6μs 的额外开销。如果服务本身的业务逻辑耗时 1ms，Sentinel 开销占比 0.6%。如果是纯内存计算耗时 100μs，占比 6%。这是 Sentinel 被大规模采用的原因之一——性能代价几乎可以忽略。

**但要注意**：以上是单 Slot 的统计开销。当规则数据量大时（如 1000 条流控规则），FlowRuleManager 的规则匹配变成 O(n) → 1ms 级别。解决方案：规则按资源名哈希存储（`Map<String, FlowRule>`），O(1) 查找。
