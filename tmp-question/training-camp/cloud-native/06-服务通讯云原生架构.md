## 一、服务间通讯的本质问题

微服务拆开了，问题来了：**服务之间怎么说话？**

这不像单体应用里直接调用同一个 JVM 里的方法——0 延迟、0 网络开销、0 序列化。微服务环境下，每次调用都变成了网络请求，带了三个新问题：

1. **怎么找到对方？**（服务发现 — 02/04 已讲）
2. **怎么说对方能听懂？**（序列化协议 — 本文讲）
3. **失败了怎么办？**（容错 — 04 已讲）

"服务通讯"解决的就是第 2 个问题，但这不只是"选哪个框架"的问题。在选框架之前，需要先理解三种根本不同的通讯模式。

## 二、REST vs RPC vs 消息：三种通讯模式的本质差异

微服务体系里同时存在三种通讯模式，它们的适用场景完全不同：

| 维度 | REST | RPC | 消息 |
|---|---|---|---|
| 通讯模型 | 请求-响应（同步） | 请求-响应（同步） | 发布-订阅（异步） |
| 协议 | HTTP/1.1 | HTTP/2 / TCP 自定义 | TCP 自定义 |
| 序列化 | JSON（文本） | Protobuf/Hessian（二进制） | JSON/Protobuf（二进制） |
| 调用方式 | `GET /orders/123` | `orderService.getOrder(123)` | `producer.send(message)` |
| 耦合度 | 低（URI + HTTP Method 约束） | 中（接口定义耦合） | 极低（完全解耦） |
| 性能 | 中 | **高**（2-3x REST） | 中（异步削峰） |
| 跨语言 | **最好**（curl 就能调） | 中（需 ProtoBuf 或跨语言 SDK） | 中（需消息协议一致） |
| 适用场景 | 对外 API、跨团队 | 内部高频调用、服务治理 | 异步解耦、削峰填谷 |

**一个真实系统的通讯地图**：

```
外部客户端 → REST（HTTP/1.1 + JSON）
  → Gateway（统一入口）
    → order-service → inventory-service：Dubbo Triple（高频同步调用）
    → order-service → notification-service：RocketMQ 消息（异步通知）
    → order-service → payment-service：Dubbo Triple（需要同步确认支付结果）
    → 跨部门风控服务：REST（双方团队解耦）
```

REST/RPC/消息不是三选一——**它们共存于同一个系统**。REST 用于对外和跨部门（低耦合需求），RPC 用于内部高频调用（性能需求），消息用于异步解耦（最终一致性需求）。

## 三、为什么 Dubbo 3.3 把默认协议从 dubbo:// 切到了 triple://

### 3.1 dubbo:// 的问题

Dubbo 2.x 的 `dubbo://` 是自定义二进制协议，设计于 2011 年，服务于完全由 Java 组成的阿里内部微服务体系。它有 3 个当代环境下不可接受的缺陷：

1. **不通**：只能 Java 调 Java。前段收到"看不懂"的二进制数据，网关无法解析。
2. **不合规**：企业网关/防火墙只放行 HTTP/HTTPS 流量，自定义 TCP 协议被拦截。
3. **不可迁移**：要迁移到 K8s + Service Mesh，Sidecar（Envoy）只理解 HTTP/1.1、HTTP/2、gRPC，不理解 dubbo:// 协议。

### 3.2 为什么是 HTTP/2 + Protobuf

Triple = HTTP/2 传输层 + Protobuf 序列化。选 HTTP/2 不是跟风——是因为它解决了 dubbo:// 的三个问题：

**通**：curl 能发，Postman 能测，前段能看，网关能转发。Dubbo 服务变成了标准的 HTTP 服务——这是从"自研协议孤岛"走到"互联网标准公民"的关键一步。

**合规**：HTTP/2 是标准协议，企业防火墙放行。

**可迁移**：Envoy 原生理解 HTTP/2。Triple 可以直接接入 Service Mesh，不需要协议转换。

**为什么不是 HTTP/1.1**：HTTP/1.1 有队头阻塞（一个请求卡住，后续排队），多路复用是 HTTP/2 的核心优势——同一连接上可以同时发送多个请求。

**为什么不是 gRPC 呢**：gRPC 也是 HTTP/2 + Protobuf。但 Dubbo 选 Triple 而不是直接用 gRPC，原因很简单——**生态存量**。Dubbo 3.3 需要兼容 Dubbo 2.x 的服务治理体系（Router/Filter/Registry/LoadBalance）。gRPC 只有 RPC，没有服务治理。

### 3.3 Triple 的三次序列化策略

Triple 默认 Protobuf，但也兼容 Hessian2 + JSON：

```
Protobuf（新服务，跨语言）→ 最小、最快、强类型
Hessian2（Dubbo 2.x 存量）→ 兼容迁移路径
JSON（调试期）→ 人类可读
```

这为 Dubbo 2.x 的老用户提供了渐进式迁移路径——不需要一次性把所有接口定义改为 .proto，先升级协议（用 JSON），再按需迁移到 Protobuf。

## 四、Dubbo 3.3 SPI：插件系统的设计哲学

Dubbo 是一个"微内核"架构**。内核只做最核心的事——服务导出（export）和服务引用（refer）。其他一切（序列化怎么选、负载均衡怎么算、注册中心连哪个）全部通过 SPI 插件化。

### 4.1 为什么选择"微内核 + 插件化"

一个反例：Sentinel 的流控规则是硬编码在 FlowSlot 里的。如果你想要一个"按用户 ID 限流"的新规则，需要修改 Sentinel 源码。

Dubbo 的设计取舍完全相反——所有组件都可替换：

```
Dubbo 内核（不可变的部分）
├── 服务导出（export）       — 把本地对象变成网络可访问的服务
├── 服务引用（refer）         — 把远程服务变成本地代理对象
└── 调用链（invoker.invoke()） — 执行请求

插件化（可变的部分，全部 SPI）
├── 协议（dubbo:// / triple:// / rest:// / grpc://）
├── 注册中心（Nacos / ZK / Consul / K8s API）
├── 序列化（Protobuf / Hessian2 / JSON / Kryo）
├── 负载均衡（Random / RoundRobin / LeastActive / ConsistentHash）
├── 路由（条件路由 / 标签路由 / Mesh 路由）
├── 过滤器（Provider Filter / Consumer Filter）
└── 集群容错（Failover / Failfast / Failsafe）
```

### 4.2 为什么不学 Spring Boot 全部自动装配

Dubbo SPI 和 Spring Boot 自动装配的哲学差异：

| | Dubbo SPI | Spring Boot 自动装配 |
|---|---|---|
| 扩展方式 | 写一个类 + 放在 META-INF/dubbo/ 下 | 写一个类 + @Configuration + META-INF/spring.factories |
| 植入方式 | 运行时通过 ExtensionLoader 加载 | 启动时通过 AutoConfiguration 加载 |
| 替换方式 | 覆盖原 SPI 实现（名字相同覆盖） | 排除 Bean + 注册替代实现 |
| 适用场景 | RPC 框架插件 | 应用自动配置 |

Dubbo 选择 SPI 而非 Spring Boot 自动装配的原因：Dubbo 可以在不依赖 Spring 的环境下运行。SPI 是纯 Java 的，不绑定任何框架。

## 五、gRPC vs Dubbo Triple：不是二选一

| 维度 | gRPC | Dubbo Triple |
|---|---|---|
| 跨语言 | ⭐⭐⭐⭐⭐ 一流 | ⭐⭐⭐ 不如 gRPC 广 |
| Java 服务治理 | ❌ 无（需自己建） | ⭐⭐⭐⭐⭐ Dubbo 全家桶 |
| 性能 | 高 | 高（与 gRPC 相当） |
| 与 Spring Cloud 整合 | 中（需 grpc-spring-boot-starter） | 高（dubbo-spring-boot-starter） |
| 协议 | HTTP/2 | HTTP/2（可以与 gRPC 互通） |
| 生态 | Google 维护，多语言广泛 | 阿里主导，Java 生态为主 |

**2025 年生产选择**：

- **Java 为主的微服务体系** → Dubbo Triple。保留 Dubbo 的 Router/Filter/Registry 全套治理能力，同时通过 Triple 获得 HTTP/2 的标准通讯能力。

- **Java + Go + Python 平等的多语言体系** → gRPC。跨语言生态最完整，Google 长期维护。

- **存量 Dubbo 2.x** → 升级到 Triple。不需要换框架，只需要换协议——代码改动最小。

## 六、Proxyless Mesh：RPC 框架在 Service Mesh 时代的生存之道

### 6.1 Sidecar 的代价

传统 Service Mesh 在每个 Pod 旁注入一个 Envoy Sidecar：

```
应用 → localhost:15001（Envoy 出站）→ 网络 → Envoy 入站 → 应用
```

这一跳的代价：
- **延迟增加 5-10ms**（Envoy 内部处理）
- **资源消耗**：每个 Sidecar 占用 100-200MB 内存
- **运维复杂度**：Sidecar 升级、配置变更需要协调

### 6.2 Proxyless：应用内嵌 xDS

Dubbo 3.3 的 Proxyless Mesh：应用进程内直接与 Istio 控制面（Pilot）通信：

```
Dubbo 应用（内嵌 xDS 客户端）
  → 向 Istio Pilot 订阅 xDS（服务发现+路由规则）
    → 绕过 Envoy Sidecar
      → 直接与服务通信
```

**什么时候选择 Proxyless**：
- 所有服务都是 Dubbo 3.3+（整个调用链无需 Sidecar）
- 对延迟敏感（如交易链路）
- 不想维护 Sidecar 集群

**什么时候仍需要 Sidecar**：
- 有非 Dubbo 服务（Python/Node.js）
- 需要统一的安全策略（mTLS）
- 需要服务级别的限流（Sidecar 层，而非应用层）

## 七、性能对比：不同协议的真实差异

不是"哪个更快"的简单问题——性能差异取决于报文大小和调用模式。

| 场景 | 最佳选择 | 原因 |
|---|---|---|
| 1KB 请求/响应，单连接反复调用 | Triple/gRPC | HTTP/2 多路复用避免连接建立开销 |
| 100KB 大报文 | Triple（Protobuf） | Protobuf 比 JSON 小 3-10x，大报文差异明显 |
| 调试/排错阶段 | REST（JSON） | 人类可读，curl 直接看 |
| 冷启动（只有 1-2 次调用） | REST | RPC 连接建立的固定开销 > 1-2 次调用的收益 |
| 异步事件（发完不管结果） | RocketMQ | 不需要同步响应，消息最合适 |

## 总结

服务通讯不是"选 Dubbo 还是 gRPC"——是要理解三种模式的适用边界，然后按场景混合使用。

7 个核心认知：
1. REST/RPC/消息三者共存于同一系统——不是选择题，是场景匹配
2. Dubbo 3.3 切到 HTTP/2 是从"自研协议孤岛"走向"互联网标准公民"
3. 微内核 + SPI = 不可变的内核 + 全可变的插件
4. gRPC vs Triple 不是二选一——Java 为主选 Triple，多语言平等选 gRPC
5. Proxyless Mesh = 内嵌 xDS 替代 Sidecar，减少一跳延迟
6. 协议选型取决于报文大小和调用模式——不是"哪个更快"的简单答案
7. Triple 的三次序列化 = Protobuf（新服务）+ Hessian2（存量迁移）+ JSON（调试）

## 查漏补缺

### Dubbo 2.x → 3.x 的真实迁移路径

不是一次升级全部服务，而是渐进式：

```
阶段 1：升级协议（不改接口）
  dubbo:// → triple://（协议切换，仍用 Hessian2）
  → 不需要定义 .proto 文件
  → 代码改动：改一行配置

阶段 2：渐进式改为 Protobuf（新接口优先）
  新服务直接 Protobuf，老服务保持 Hessian2
  → Triple 同时支持两者
  → 没有"一次性全部重构"的压力

阶段 3：全量迁移（按节奏）
  老服务重构/下线时自然切换到 Protobuf
```

Dubbo 2.x 存量用户的最大阻力是"需要全部改 .proto"。Triple 的三次序列化策略解决了这个问题——**可以先享受 HTTP/2 的好处，再按需迁到 Protobuf**。

### Native Image 对 Dubbo 的影响

GraalVM 把 Java 应用编译成本地二进制，启动时间从秒级降到毫秒级。但 Dubbo 大量使用反射和动态代理——这正是 Native Image 的死穴。

**受影响的 Dubbo 机制**：

| 机制 | Native Image 需做什么 |
|---|---|
| SPI（ExtensionLoader 扫描 META-INF/dubbo/） | `reflect-config.json` 声明所有 SPI 实现类 |
| 动态代理（Proxy.newProxyInstance） | `proxy-config.json` 声明所有代理接口组合 |
| 序列化（Hessian2 反射读写字段） | `serialization-config.json` 声明所有序列化类型 |
| Javassist 字节码生成 | 替换为静态编译生成 |

**Dubbo 3.3 的应对**：`dubbo-native-plugin` 自动生成以上 metadata。但 Tritple + Protobuf 是最佳路径——Protobuf 是静态生成的 Stub，不依赖反射，天然适合 Native Image。

### 服务通讯的常见反模式

1. **"全 REST"症**：所有服务间调用都用 REST，包括高频内部调用。结果：JSON 序列化开销 = 30% CPU 浪费。内部高频调用用 RPC（Triple/gRPC）。

2. **"全 RPC"症**：所有服务间都用 Dubbo，包括对外 API。结果：外部客户端无法调用（没有 Java SDK）。对外 API 必须用 REST。

3. **"同步 RPC 替代异步"症**：下单后直接同步调用通知服务——通知服务挂了，下单也失败。应该用消息（RocketMQ）：下单成功→发消息→通知服务异步消费。

4. **协议混用不隔离**：同一个接口同时支持 dubbo:// 和 triple://。结果：部分消费者用旧协议，故障排查时需要同时检查两个协议的日志。应该统一协议版本。

5. **过度 Proxyless**：把所有服务都改为 Proxyless Mesh，但忽略了部分服务需要 mTLS。结果：安全漏洞。Proxyless 不适合需要 Sidecar 做安全控制的场景。
