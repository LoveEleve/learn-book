## 一、网关的哲学命题：前端该知道多少后端信息？

一个真实的问题：App 的首页需要商品列表（product-service）、用户头像（user-service）、推荐内容（recommendation-service）。没有网关时，App 需要知道**3 个服务的地址 + 3 种不同的认证方式 + 3 种不同的数据格式**。

有网关后：App 只知道 `gateway.example.com`。

网关是一个**信息屏蔽层**——前端不需要知道后端有多少个服务、它们在哪、用的什么协议、谁在维护。前端只需要知道网关。这就是网关存在的根本原因，不是"可以加限流很方便"，而是**没有网关，客户端就耦合了后端的物理部署**。

## 二、Spring Cloud Gateway 的过滤器链：不只是功能列表

### 2.1 路由匹配的代价

Gateway 的路由匹配不是 O(1) 的查找——它是**遍历所有 Route，逐一尝试 Predicate 匹配**：

```java
// RoutePredicateHandlerMapping.lookupRoute()
for (Route route : routes) {
    if (route.getPredicate().test(exchange)) {
        return route;  // 第一个匹配的 Route
    }
}
```

**当路由表很大时（100+ 条规则），每次请求的匹配开销不可忽略**。优化策略：
- 高频路由放前面（越早匹配，越少遍历）
- 使用明确的 Path 谓词（避免 `Path=/**` 作为通配兜底）
- 考虑 `CachingRouteDefinitionLocator`（缓存路由解析结果）

### 2.2 每次请求的 3 次数组复制

`FilteringWebHandler` 每次处理请求时创建新的过滤器链，涉及三次 `Arrays.copyOf`：

```java
// 每次请求都要做的：
List<GatewayFilter> combined = new ArrayList<>();
combined.addAll(globalFilters);    // copyOf #1
combined.addAll(routeFilters);     // copyOf #2 + ArrayList 扩容 = copyOf #3
combined.sort(comparator);
// 对列表中 N 个 filter 依次执行 filter()
```

**10000 QPS → 每秒 30000 次数组复制**。优化方案：`CachingFilteringWebHandler` 预先缓存合并后的过滤器链，路由变更时通过 `RefreshRoutesResultEvent` 失效缓存。省掉每次请求的 3 次复制。

### 2.3 Route/Predicate/Filter 三组件为何分离

```
Route = id + uri + predicates + filters
Predicate: 这个请求该不该走这条路？（匹配逻辑）
Filter: 走这条路时做哪些处理？（处理逻辑）
```

**如果 Predicate 和 Filter 绑定在一起**：新增一种匹配方式（如按 Header 匹配灰度标签）需要修改 Filter 代码。分离后：加一个 `HeaderRoutePredicateFactory` 即可，Filter 不变。两者独立变化——这是 Strategy 模式在网关中的应用。

## 三、四合一网关：不只是"四种能力"，是架构的必然

### 3.1 为什么需要四合一

Spring Cloud 生态下，一个完整的微服务系统需要四类网关：

```
1. API 网关       — 外部请求 → 内部 REST 服务（Spring Cloud Gateway 原生）
2. 配置网关       — 统一入口访问 Nacos/Apollo/etcd 配置
3. 服务注册发现网关— 统一入口查询多个注册中心的服务实例
4. RPC 网关       — REST → Dubbo/gRPC 的协议转换
```

**如果没有四合一**：运维需要维护 4 个独立的网关组件→4 套部署→4 套监控→4 套安全策略。**四合一的核心价值是"统一治理"**——一套安全策略、一套限流规则、一套日志格式、一套监控面板——而不是节省服务器成本。

### 3.2 协议转换：REST → Dubbo 泛化调用

外部客户端只能发 REST 请求，但内部服务用 Dubbo。网关作为翻译层：

```
外部：POST /orders {"userId": 123}
  → Gateway 解析 URL 和 Body
    → 泛化调用：GenericService orderService.$invoke("createOrder", ...)
      → Dubbo Provider 处理 → 返回结果
        → Gateway 将结果转为 JSON → 返回给客户端
```

**泛化调用的关键**：网关不 import `order-service-api.jar`——它不知道 `CreateOrderRequest` 类的定义。通过运行时反射加载服务接口和参数类型：

```java
GenericService genericService = ApplicationModel.defaultModel()
    .getServiceRepository()
    .lookupService("com.example.OrderService");
Object result = genericService.$invoke(
    "createOrder",                          // 方法名
    new String[]{"com.example.CreateOrderRequest"},  // 参数类型
    new Object[]{requestMap}                // 参数值（从 JSON 反序列化得到）
);
```

这就是网关和业务服务之间的**编译时解耦**——业务服务加一个新方法和一个新 DTO，网关不用重新编译和部署。

### 3.3 服务发现网关：CompositeDiscoveryClient 的短路问题

Gateway 查询服务实例时，用 `DiscoveryClient.getInstances(serviceId)`。如果配置了多个注册中心（如 Nacos+Eureka），Spring Cloud 的 `CompositeDiscoveryClient` 有一个关键缺陷：

```java
// CompositeDiscoveryClient.getInstances
for (DiscoveryClient client : clients) {
    List<ServiceInstance> instances = client.getInstances(serviceId);
    if (!instances.isEmpty()) return instances;  // 短路！第一个非空就返回
}
// 如果 service-a 在 Nacos 有实例，但 Nacos 排第二
// → 不会查到（短路在第一个注册中心就返回了空）
```

**Microsphere 的方案**：`UnionDiscoveryClient` 合并所有注册中心的结果，而不是短路返回。这是网关集成多注册中心的关键能力。

## 四、灰度路由：网关层的金丝雀发布

网关是灰度发布的最自然地——所有流量都经过网关，这里是做**流量染色和路由**的最佳位置。

### 4.1 基于 Header 的灰度路由

```yaml
spring:
  cloud:
    gateway:
      routes:
      - id: order-route-gray
        uri: lb://order-service
        predicates:
        - Path=/orders/**
        - Header=X-Version, v2        # 带 v2 Header 的请求
        filters:
        - AddRequestHeader=X-Color, gray
```

**流程**：发布新版本（v2）→ 部署 1 个 v2 实例 → 网关配置灰度路由 → 内部测试人员带 `X-Version: v2` → 路由到新版本 → 验证通过 → 滚动升级全量。

### 4.2 灰度路由的局限

Header 路由的问题是：**谁来决定哪些请求去灰度**？

- 内部测试人员 → 手动加 Header → 简单但不适合真实用户分流
- 按流量比例（10%）→ 需要自定义 LoadBalancer + 权重
- 按用户 ID（userId % 10 < 1）→ 需要自定义 GatewayFilter 解析请求

Gateway 不原生支持按流量比例灰度——这是"四合一网关"为什么需要自定义 LoadBalancer 的原因。

## 五、WebSocket 在网关中的透传模式

### 5.1 为什么不能和 HTTP 一样处理

HTTP 是短连接——请求→路由→转发→返回→断开。WebSocket 是长连接——建立→保持→双向推送→一方断开。

Gateway 对 HTTP 的处理模式（解析 Headers → 修改 → 转发）不能直接用于 WebSocket——解析帧会造成巨大的性能开销（每个 WebSocket 帧都解析）。

### 5.2 透传模式

Gateway 检测到 WebSocket 升级请求（`Upgrade: websocket` Header）后：
- **握手阶段**：正常转发 HTTP Upgrade 请求
- **连接建立后**：进入透传模式——双向转发字节流，不解析帧

**与 HTTP/2 Server Push 的区别**：WebSocket 是独立的连接（有自己的 Keep-Alive、心跳），Server Push 复用同一个 HTTP/2 连接。

## 六、Spring Cloud Gateway vs K8s Ingress/Gateway API

| 维度 | Spring Cloud Gateway | K8s Ingress | K8s Gateway API |
|---|---|---|---|
| 路由粒度 | URL/Header/参数/权重 | Host + Path | HTTP/G RPC/TCP/UDP |
| 动态路由 | ✅ 运行时更新 | ❌ 需 reload Nginx | ✅ 通过 Controller |
| 与注册中心集成 | ✅ Native（lb://） | ❌ 需自己维护 upstream | ❌ |
| 限流 | ✅ Redis RateLimiter | ❌ 需额外组件 | ✅ RateLimitPolicy |
| 编程性 | Java 代码（GateFilter） | YAML 配置 | YAML 配置 |
| 适用场景 | Spring Cloud 微服务 | 简单的 Host/Path 路由 | K8s 原生、多协议 |

**选择策略**：
- 已有 Spring Cloud + Nacos → Sping Cloud Gateway（与注册中心集成最紧密）
- 全 K8s 部署 + 简单路由 → K8s Gateway API（云原生，标准化的未来方向）
- 混合架构（部分 K8s + 部分裸机） → Spring Cloud Gateway（统一入口）

## 七、网关的常见反模式

1. **"胖网关"**：在网关层做复杂的业务逻辑（数据转换、校验、聚合）。网关的定位是**路由 + 横切关注点**（认证/限流/日志），业务逻辑属于后端服务。

2. **"不设限"**：网关不做限流。结果一个服务超时，Gateway 线程池被打满→所有服务不可用。Gateway 的线程池大小必须有限制。

3. **"全走网关"**：内部服务间调用也经过网关（A→Gateway→B）。多一跳网络延迟 5-10ms。**内部调用直接用 RPC，不要绕经网关**。

4. **"单点网关"**：只部署一个 Gateway 实例——挂了全站不可用。Gateway 本身也是一微服务→多实例 + Nacos 注册 + 客户端负载均衡。

## 查漏补缺：真正影响生产的问题

### 网关的线程模型：为什么"线程池满"是网关的头号事故

Spring Cloud Gateway 基于 Netty，但**不是完全不阻塞**。真相是：Netty I/O 线程（Boss + Worker）处理网络读写（不阻塞），但底层的 HTTP Client（转发请求到后端服务）会在 Worker 线程上执行。如果后端服务响应慢，Worker 线程被占用→新的网络读写无法处理→网关"假死"。

**生产环境的线程池配置**：

```yaml
spring:
  cloud:
    gateway:
      httpclient:
        pool:
          type: elastic        # 连接池类型
          max-idle-time: 60s   # 空闲连接保留时间
          max-life-time: 300s  # 连接最大存活时间
          max-connections: 1000 # 每个后端服务的最大连接数
```

**为什么默认 1000 不够**：后端有 20 个微服务，每个 1000 连接 = 2 万连接池。当流量爆发时，线程池满→Gateway 拒绝新建连接→客户端看到 503。解决方案：
- 增大 `max-connections`（但内存会增长）
- 加降级逻辑（CircuitBreaker + fallback）
- 横向扩展（多 Gateway 实例 + Nginx 负载均衡）

### Filter 链的顺序陷阱

Gateway 的 Filter 通过 `@Order` 或 `Ordered` 接口控制执行顺序。**顺序错了，功能失效**——而且不抛异常，只静默地坏掉。

**正确的执行顺序**（从外到内）：

```
NettyRoutingFilter（路由转发，最外层，最后执行）
  ← CircuitBreaker（熔断，在路由之前判断）
    ← RequestRateLimiter（限流，在熔断之前判断）
      ← RemoveHopByHopHeadersFilter（安全头，在限流之前处理）
        ← AdaptCachedBodyGlobalFilter（缓存 Body，在所有需要读 Body 的 Filter 之前）
          ← 实际请求进来
```

**为什么 AdaptCachedBodyGlobalFilter 必须排在前**：Gateway 的 `ServerHttpRequest.getBody()` 返回一个 Stream——只能读一次。如果限流 Filter 先读了 Body（检查是否有某个字段），后续 Filter 再读就空了。AdaptCachedBody 把 Body 缓存起来，后续 Filter 可以重复读。

### 请求体缓存问题是生产故障的常见根源

```
请求到达 Gateway
  → Filter A 读取 Body 做签名校验 → read() → Body 空了
    → Filter B 尝试读取 Body 做日志记录 → read() → 什么都没有！
      → 日志里 Body=null
        → 排查故障时："请求 Body 到底是什么？？"
```

**解决**：加 `spring.cloud.gateway.filter.adapt-cached-body.enabled=true`，或者在自定义 Filter 之前确保 Body 已被缓存。

### 多协议实时通信对比：不仅是"都支持双向"

| 协议 | 握手 | 帧格式 | 背压支持 | 浏览器支持 | 适用场景 |
|---|---|---|---|---|---|
| **WebSocket** | HTTP Upgrade（101） | 纯二进制（Gateway 透传） | ❌ | ✅ 原生 | 聊天、实时通知、K线图 |
| **SSE** | 普通 HTTP 长连接 | 纯文本 | ❌ | ✅ EventSource API | 单向服务端推送（如状态更新） |
| **RSocket** | 自定义（或 WebSocket 之上） | 二进制（帧长度+类型+数据） | ✅ 原生 | ❌ | 微服务间高频流、有背压需求 |
| **gRPC Streaming** | HTTP/2 | Protobuf 编码帧 | ✅ HTTP/2 flow control | ❌（需 gRPC-Web） | 跨语言、需要类型安全 |

**为什么 Gateway 只对 WebSocket 有原生支持**：WebSocket 是 HTTP 协议的自然扩展（通过 Upgrade 头升级）。RSocket 和 gRPC 需要专门的协议处理——它们不是 HTTP，Gateway 不能直接转发。这就是四合一网关需要"RPC 网关"的原因——做协议转换。

### 跨域 CORS 的完整流程

不是"加个配置就完了"，要理解为什么需要这些步骤：

```
浏览器发起跨域请求
  → 浏览器先发 OPTIONS（preflight 预检）
    → 请求头包含 Origin: https://app.example.com
      → 服务器返回 Access-Control-Allow-Origin: https://app.example.com
        → 浏览器检查：Origin 匹配？→ 是 → 允许
          → 再发真正的 POST 请求
```

**生产配置的陷阱**：`allowedOrigins: "*"` 不能用——因为它和 `allowCredentials: true` 互斥（浏览器安全策略）。带 Cookie/Token 的跨域请求必须指定明确的 Origin 列表。

### 网关的链路追踪

Gateway 是分布式链路追踪的**关键断点**——如果 traceId 在这里丢了，整个调用链就会断裂。

Gateway 自动传播 traceId 的机制：

```
客户端请求 → Gateway
  → 从 Header 读取 traceparent（W3C 标准）
    → MDC 注入 traceId
      → 转发请求到后端时，自动追加 traceparent Header
        → 后端服务从 Header 读取 traceId → 继续传播
```

**���见的故障模式**：自定义 Filter 在转发之前清空了 Header（如 `request.mutate().headers(h -> h.clear())`）→ traceId 丢失。自定义 Filter 必须保留 traceId Header。

## 附二：网关安全

### CSRF 防护

```yaml
spring:
  cloud:
    gateway:
      filter:
        secure-headers:
          csrf: enabled
```

Gateway 生成 CSRF Token → 存入 Cookie → 前端每次 POST 请求必须带上 Token → Gateway 校验。

### HSTS（HTTP Strict Transport Security）

强制客户端只能通过 HTTPS 访问：

```
Strict-Transport-Security: max-age=31536000; includeSubDomains
```

### CSP（Content Security Policy）

限制页面可以加载哪些资源：

```
Content-Security-Policy: default-src 'self'; script-src 'self' https://trusted-cdn.com
```

**这些在 Spring Cloud Gateway 中通过 SecureHeaders 自动注入**。不需要手动写 Filter。

### X-Forwarded-For 头的风险

Gateway 作为反向代理，会自动追加 `X-Forwarded-For` 记录请求来源 IP。但如果攻击者伪造了这个头怎么办？

```
攻击者请求：X-Forwarded-For: 10.0.0.1（伪造内部 IP）
  → Gateway 追加真实 IP → X-Forwarded-For: 10.0.0.1, 真实 IP
    → 后端服务从右边取最后一个 IP（真实 IP）
```

**为什么不是从左取**：左边是客户端声称的 IP（不可信），右边是 Gateway 追加的真实 IP（可信）。后端服务必须取**最右边**的 IP 才能拿到真实来源。
