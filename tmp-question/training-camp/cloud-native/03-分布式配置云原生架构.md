## 一、配置加密

生产环境的数据库密码、API Key、Token 不能明文存在配置中心。三种方案：

### 1.1 Jasypt（应用层加密，轻量）

```yaml
# application.yml
spring:
  datasource:
    password: ENC(加密后的密文)
jasypt:
  encryptor:
    password: ${JASYPT_ENCRYPTOR_PASSWORD}  # 解密密钥从环境变量注入
```

**原理**：Spring Boot 启动时，`jasypt-spring-boot-starter` 拦截 `PropertySource`，自动解密 `ENC(...)` 包裹的值。

**适用**：中小规模，不需要独立密钥管理服务。

**缺陷**：密钥还是存在某个地方（环境变量/启动参数），没有从根本上解决密钥管理问题——只是把明文密码换成了明文密钥。

### 1.2 Vault（独立密钥管理，企业级）

```
应用启动 → 向 Vault 认证（AppRole/K8s ServiceAccount）
  → Vault 返回临时数据库凭据（动态生成，自动过期）
    → 应用用临时凭据连接数据库
```

**优势**：
- **动态凭据**：每次请求生成新凭据，用完自动回收（彻底消除硬编码）
- **审计日志**：谁在什么时候获取了什么凭据——满足合规要求
- **自动轮转**：凭据定期自动更换，无需人工介入

**适用**：中大型企业，有合规要求的场景。

### 1.3 云厂商 KMS（托管方案）

| 云厂商 | 服务 | 集成方式 |
|---|---|---|
| 阿里云 | KMS + 凭据管家 | `aliyun-spring-boot-starter-kms` |
| AWS | KMS + Secrets Manager | `spring-cloud-aws-secrets-manager` |
| 腾讯云 | KMS + SSM | `tencentcloud-ssm-spring-boot-starter` |

**优势**：零运维，和云上其他服务（RDS/Redis）无缝集成。

## 二、灰度配置

不能一改配置就全量生效——万一配置错了，所有实例都受影响。

### 2.1 Nacos 灰度配置（原生支持）

```yaml
# 灰度配置：仅对带有 label=gray 的实例生效
config:
  data-id: order-service.yaml
  group: DEFAULT_GROUP
  gray:
    enabled: true
    gray-rule:
      - type: label
        value: gray
        ratio: 10%   # 先 10% 实例生效
```

**流程**：修改配置 → 勾选灰度发布 → 选择目标实例（按标签/IP/比例）→ 先推送到灰度实例 → 观察 5-10 分钟 → 确认无误 → 全量推送。

### 2.2 Apollo 灰度配置

Apollo 天然支持"主版本 + 灰度版本"双重配置：
- 主版本：全量生效（稳定版本）
- 灰度版本：仅指定实例生效（如 10.0.1.1, 10.0.1.2）

**发布流程**：新建灰度版本 → 选择灰度实例 → 发布 → 观察 → 全量覆盖主版本。

## 三、配置审计

### 3.1 Nacos 内置审计日志

Nacos 2.x 支持记录每次配置变更：
```
[2025-01-15 10:23:45] user=admin, action=UPDATE, dataId=order-service.yaml, 
  group=DEFAULT_GROUP, from=version-15, to=version-16
```

### 3.2 配置变更通知

当配置被修改时，自动通知相关责任人：
```
Nacos Config Change Event
  → Webhook (企业微信/钉钉/飞书)
    → "@张三 修改了 order-service.yaml 的 timeout 从 5000 改为 3000"
```

### 3.3 配置变更 Diff

每次变更记录 `before` 和 `after` 的完整内容，可以在管理界面看到 Diff：
```diff
- timeout: 5000
+ timeout: 3000
```

## 四、配置回滚

### 4.1 Nacos 配置回滚

```
点击"历史版本"→ 选择目标版本（如 version-15）→ 点击"回滚"
  → Nacos 自动将当前配置替换为 version-15 的内容
    → 发布新版本（version-*=17），内容是 version-15 的内容
```

**注意**：回滚不是"恢复"旧版本号，而是"用旧内容创建新版本"——保持版本历史完整。

### 4.2 自动回滚策略

- **健康检查触发回滚**：新配置推送后，监控到错误率上升 → 自动回滚到上一个稳定版本
- **灰度验证未通过**：灰度实例的错误率超过阈值 → 自动取消灰度发布

## 五、多环境隔离

### 5.1 Nacos Namespace 隔离

```
Namespace: dev       → 开发环境配置（数据库连接池 2、日志 DEBUG）
Namespace: staging   → 预发布环境配置（数据库连接池 10、日志 INFO）
Namespace: prod      → 生产环境配置（数据库连接池 50、日志 WARN）
```

**关键**：不同 Namespace 之间**完全隔离**——即使配置名相同（如 `timeout=5000`），dev 和 prod 的值互不影响。

### 5.2 环境迁移策略

```yaml
# 开发环境
spring:
  profiles: dev
  cloud:
    nacos:
      config:
        namespace: dev-namespace
        group: DEFAULT_GROUP

# 生产环境
spring:
  profiles: prod
  cloud:
    nacos:
      config:
        namespace: prod-namespace
        group: PROD_GROUP
```

## 附：训练营补充细节

### 一、Apollo 配置中心机制（子页面 6）

训练营在 Microsphere Configuration 整合中介绍了 Apollo 集成。Apollo 与 Nacos 的核心差异：

| 维度 | Nacos | Apollo |
|---|---|---|
| 配置变更推送 | 长轮询 + gRPC（2.x） | HTTP 长轮询 |
| 灰度发布 | 支持（标签/比例/IP） | 原生支持（主版本+灰度版本） |
| 部署复杂度 | 集群部署（CP+AP 双模式） | 需要 Config/Admin/Portal 三服务 + Eureka |
| 社区活跃度 | 阿里主导，国内主流 | 携程主导，企业用户多 |
| 2025 年推荐 | 新项目首选 | 已有 Apollo 存量可继续用 |

### 二、Spring Boot 2.x ConfigurationPropertySource 新 API（子页面 10）

Spring Boot 2.0 重新设计了配置抽象层：

```
ConfigurationPropertySource          — 替代 PropertySource 的配置来源抽象
  ├── getConfigurationProperty(name) — 查找配置（支持松散绑定）
  └── filter(predicate)              — 按前缀过滤子集

ConfigurationPropertyName            — 配置属性名（支持 kebab/camel/underscore/UPPERCASE 互转）
ConfigurationProperty                — name + value + Origin 追踪
PropertyMapper                       — 松散绑定映射器
  ├── DefaultPropertyMapper           — 默认（kebab→camel 等）
  └── SystemEnvironmentPropertyMapper — OS 环境变量（UPPERCASE_UNDERSCORE 识别）
```

**Origin 追踪的价值**：配置来自哪个文件、哪一行——排查配置问题时直接定位来源。

### 三、配置元数据处理的价值（子页面 9-12）

Microsphere 的 `ListenableConfigurableEnvironment` + `CollectingConfigurationPropertyListener` 实现配置可观测性：

```
Environment.getProperty("server.port") → 拦截 → 记录到 ConfigurationPropertyRepository
@Value("${my.key}")                    → 拦截 → 记录到 ConfigurationPropertyRepository
${my.placeholder}                      → 拦截 → 记录到 ConfigurationPropertyRepository
```

**最终产出**：完整的配置引用地图——运维知道"server.port 被 5 个 Bean 引用、my.key 只被 1 个 @Value 引用"。

**Actuator 集成**：`ConfigurationMetadataEndpoint` 暴露配置地图到 JMX/Web + 24 小时缓存优化（`microsphere.cache.long-long-time-to-live`）。
