## 一、服务注册发现：Spring Cloud 抽象 + 5 种实现

### 1.1 三大核心接口

```
DiscoveryClient                        — 服务发现（消费者侧）
├── getServices()       → 列出所有服务
├── getInstances(id)    → 获取实例列表
├── getOrder()          → 多 DiscoveryClient 时的优先级

ServiceRegistry<Registration>          — 服务注册（提供者侧）
├── register(reg)       → 注册实例
├── deregister(reg)     → 注销
├── setStatus(reg, s)   → UP/DOWN

Registration                           — 服务实例模型
├── getServiceId()      → 服务名
├── getHost()/getPort() → 网络地址
└── getMetadata()       → 扩展元信息
```

### 1.2 设计模式 1：Composite vs Union — 多注册中心聚合的两种策略

Spring Cloud 的 `CompositeDiscoveryClient` 采用**短路模式**：
```java
for (DiscoveryClient client : discoveryClients) {
    List<ServiceInstance> instances = client.getInstances(serviceId);
    if (!instances.isEmpty()) return instances;  // 只要第一个非空就返回
}
```

**问题**：一个服务只在 Nacos-2 注册，但 Nacos-1 排在第一 → 查询不到。

**Microsphere 的 UnionDiscoveryClient 解决方案**：
```java
for (DiscoveryClient client : discoveryClients) {
    instances.addAll(client.getInstances(serviceId));  // 全部合并
}
return instances;  // 返回所有注册中心的实例
```

**两种模式的选择**：
| 模式 | 适用场景 | 代表 |
|---|---|---|
| **短路（Composite）** | 副本模式：多注册中心互为备份 | Spring Cloud 默认 |
| **合并（Union）** | 聚合模式：不同服务在不同注册中心 | Microsphere Spring Cloud |

### 1.3 设计模式 2：ServiceRegistry 单注册约束

Spring Cloud 默认**不允许**多注册。原因：

`ServiceRegistryEndpoint`（Actuator Endpoint）构造器注入**单个** `ServiceRegistry` Bean：
```java
public ServiceRegistryEndpoint(ServiceRegistry<?> serviceRegistry) {
    // 只有一个 ServiceRegistry，多注册中心时抛出 NoUniqueBeanDefinitionException
}
```

要支持多注册，需要：
1. 禁用默认的 `ServiceRegistryEndpoint`
2. 实现 `MultipleAutoServiceRegistration`（Microsphere 方案）
3. 每个注册中心对应一个 `ServiceRegistry` + `Registration` 对

### 1.4 训练营展示的 5 种注册中心实现对比

| 注册中心 | DiscoveryClient 实现 | ServiceRegistry 实现 | 注册模型 | 长轮询机制 |
|---|---|---|---|---|
| **Eureka** | EurekaDiscoveryClient | EurekaServiceRegistry (EurekaRegistration) | AP | 无长轮询，客户端 30s 定时全量拉取 |
| **Consul** | ConsulDiscoveryClient | ConsulServiceRegistry (ConsulRegistration) | CP (Raft) | Blocking Query (wait=2s + index) |
| **ZooKeeper** | ZookeeperDiscoveryClient | ZookeeperServiceRegistry (ZookeeperRegistration) | CP (ZAB) | Watcher（一次性，需重注册） |
| **Nacos** | NacosDiscoveryClient | NacosServiceRegistry (NacosRegistration) | CP+AP | 1.x: 客户端定时拉取 / 2.x: gRPC 双向流 |
| **K8s** | KubernetesInformerDiscoveryClient | **无注册接口** | 最终一致 | SharedInformer（etcd Watch 底层） |

**K8s Service 的独特之处**：它是唯一**不需要应用主动注册**的方案——Label Selector 自动匹配 Pod，无需 `ServiceRegistry` 接口。

### 1.5 注册中心优化技术（时间无关）

**Eureka 优化**：
- 全量：启动时 Fetch 相邻节点数据
- 增量：JGroup 广播同步节点间状态（延迟毫秒级）

**ZooKeeper 优化**：
- `WatchManager` → `WatchManagerOptimized`：HashMap 替换为 **BitMap**，大规模场景内存节省显著

**Consul 优化**：
- Blocking Query 参数：`wait=2s` + `index` 版本号，相比 Eureka 30s 全量拉取，延迟降低 15 倍

## 二、设计模式 3：NamedContextFactory — Spring Cloud 多租户核心

`NamedContextFactory` 是 Spring Cloud 中最重要但最少被讨论的设计模式。

### 2.1 应用场景

| 组件 | 使用方式 | 原因 |
|---|---|---|
| **Ribbon** | `@RibbonClient(name="service-a")` | 每个服务独立配置（超时/重试/LB 策略） |
| **LoadBalancer** | `@LoadBalancerClient(name="service-a")` | 同上 |
| **OpenFeign** | `@FeignClient(name="service-a")` | 每个 Feign Client 独立 Encoder/Decoder/Logger |

### 2.2 原理

每个 `name` 创建一个**独立的 Spring 子 ApplicationContext**：
- 独立的 PropertySource（配置隔离）
- 独立的 Bean 生命周期（互不干扰）
- 父上下文提供共享 Bean（如公共配置）

### 2.3 在 DevOps 多集群管理中的应用

```
Spring 父 ApplicationContext（DevOps 平台）
├── Nacos-北京 子上下文（nc-beijing）
│   ├── ConfigClient bean（独立 Nacos Server 地址）
│   └── NamingClient bean
├── Nacos-上海 子上下文（nc-shanghai）
│   ├── ConfigClient bean（独立 Nacos Server 地址）
│   └── NamingClient bean
└── K8s-生产 子上下文（k8s-prod）
    └── ApiClient bean（独立 K8s API Server）
```

**配置驱动（而非注解驱动）**：
```properties
# 北京集群
microsphere.nacos.clients.nc-beijing.server-address = 10.0.1.1:8848

# 上海集群
microsphere.nacos.clients.nc-shanghai.server-address = 10.0.2.1:8848
```

每个集群对应一个子上下文，配置完全隔离，互不影响。

## 三、设计模式 4：优雅停机生命周期

### 3.1 Spring Boot Actuator ShutdownEndpoint 全链路

```
ShutdownEndpoint (HTTP POST /actuator/shutdown)
  → ConfigurableApplicationContext.close()
    → AbstractApplicationContext.doClose()
      → LifecycleProcessor.onClose()
        → WebServerGracefulShutdownLifecycle.stop(Runnable)
          → TomcatWebServer.shutDownGracefully()
            → GracefulShutdown.shutDownGracefully()
              → 逐个 Connector 关闭 + 等待 Context 不再活跃
```

### 3.2 优雅停机需要额外处理的 4 个风险点

**1. 定时任务**：
- 使用 Spring `ThreadPoolTaskScheduler` + 实现 `Lifecycle` 接口，确保任务先于 Web 连接池关闭前停止
- JDK 原生 `ScheduledThreadPoolExecutor` 需要主动关联 `Lifecycle`

**2. 异步任务**：同定时任务——使用 Spring `@Async` + `ThreadPoolTaskExecutor`

**3. MQ 消费**：
- JMS API 原生支持生命周期管理
- Kafka/RocketMQ 需要 `@PreDestroy` 中调用 `consumer.close()`

**4. RPC 积压**：
- Dubbo 的优雅停机：先摘除注册表 → 处理完积压请求 → 关闭

```
完整停机时序（三层）：
K8s preStop hook → Nacos deregister（摘除注册表）
  → terminationGracePeriodSeconds=30s（等存量请求完成）
    → Spring graceful shutdown（Web → 定时任务 → MQ → RPC）
      → SIGTERM → 进程退出
```

## 四、设计模式 5：Open API 设计通用方法论

训练营第 8 篇梳理了 Open API 设计的 9 个通用问题（跨语言适用）：

| 问题 | 解决方案 |
|---|---|
| API 稳定性 | 版本化 URL（`/v1/...` / `/v2/...`）+ 向后兼容 |
| HTTP Status × API Status | 分开输出：HTTP 2xx/4xx/5xx + API Status Code（枚举） |
| 多端适配 | REST 端 / SDK 端（gRPC/TCP）/ CLI 端 |
| 缓存策略 | 强一致性（长连接）vs 弱一致性（定时器 + 短连） |
| 查询设计 | 全量 + 增量（长轮询/SSE/HTTP2 Push） |
| 健康检查 | 多样性（HTTP/TCP/gRPC）+ 指标可降级（防系统性风险） |
| 版本兼容 | 反模式：控制台版本 vs API 版本不匹配 |
| 模型设计 | 反模式：绑定某类序列化实现 / 文档与数据不一致 |
| 安全性 | HTTP Basic / Token（JWT + OAuth2）/ mTLS |

**长轮询的 3 种实现方式**（跨所有注册中心通用）：

| 方案 | 实现 | 适用场景 |
|---|---|---|
| HTTP 长轮询 | Consul Blocking Query、Apollo Config 监听、Eureka Delta 更新 | HTTP REST 客户端 |
| SSE（Server-Sent Events） | 浏览器端 EventSource API | Web 前端实时推送 |
| gRPC 双向流 | Nacos 2.x、etcd v3 Watch | 服务端主动推送、低延迟 |

## 总结：5 个时间无关的设计模式

1. **NamedContextFactory**：Spring Cloud 多租户核心——每个"name"一个独立子上下文
2. **Composite vs Union**：多注册中心聚合——短路（备份）vs 合并（异构）
3. **ServiceRegistry 单注册约束**：ServiceRegistryEndpoint 注入导致——破除需要 MultipleAutoServiceRegistration
4. **优雅停机生命周期**：ShutdownEndpoint → Connector → 定时任务/RPC/MQ 级联——每层独立处理
5. **Open API 长轮询**：HTTP 长轮询 / SSE / gRPC 双向流——跨所有注册中心通用

这 5 个模式不绑定任何框架——Nacos、K8s、etcd、Consul 只是它们的实现载体。

### 一、服务网格 vs 传统注册中心

**两条不同的路径**：

| 维度 | SDK 模式（Nacos/Sentinel） | Sidecar 模式（Istio/Envoy） |
|---|---|---|
| 服务发现 | 应用主动注册到 Nacos | Envoy 自动发现（xDS 协议） |
| 负载均衡 | 客户端 LoadBalancer | Envoy 代理层 |
| 流量控制 | Sentinel 注解/规则 | VirtualService + DestinationRule |
| mTLS | 应用自行实现 | Envoy 自动注入，应用无感知 |
| 升级成本 | 代码改动（引入 SDK） | 部署 Sidecar（无代码改动） |

**2025 年选择建议**：
- 已有 Spring Cloud 生态 → SDK 模式（Nacos + Sentinel），渐进迁移
- 新项目且全 K8s 部署 → Sidecar 模式（Istio），一次投入
- 混合架构（部分裸机+部分 K8s） → SDK 模式为主，K8s 内部用 Service 替代

### 二、健康检查三层探针

K8s 提供三种探针，每种用途不同：

```
Startup Probe               — 应用启动完成前不检查其他探针
  ↓ 通过后
Liveness Probe              — 应用是否还活着（死 → 重启）
Readiness Probe             — 应用能否接收流量（否 → 摘除）
```

**典型配置**：

```yaml
startupProbe:
  httpGet:
    path: /actuator/health
    port: 8080
  initialDelaySeconds: 0        # 立即开始
  periodSeconds: 5              # 每 5s 检查
  failureThreshold: 30          # 允许 30 次失败 = 150s 启动窗口

livenessProbe:
  httpGet:
    path: /actuator/health/liveness
    port: 8080
  initialDelaySeconds: 0        # startupProbe 通过后才开始
  periodSeconds: 15             # 每 15s
  failureThreshold: 3           # 3 次失败 → 重启

readinessProbe:
  httpGet:
    path: /actuator/health/readiness
    port: 8080
  periodSeconds: 5              # 每 5s
  failureThreshold: 2           # 2 次失败 → 摘除流量
```

**关键参数选择**：
- `failureThreshold * periodSeconds < terminationGracePeriodSeconds` — 确保摘除先于强制杀死
- Startup Probe 的 `failureThreshold` = `(启动时间 / periodSeconds) + buffer`

### 三、流量治理在注册层的配合

**金丝雀发布**（注册中心侧的实现）：

```
正常实例：version=v1（metadata）
金丝雀实例：version=v2, label=canary（metadata）

网关/消费者侧：
  根据 Header x-canary=true → 选择 version=v2 实例（10%）
  无 Header → 选择 version=v1 实例（90%）
```

**关键**：金丝雀实例必须注册到**同一个服务名**下，用 metadata 区分版本。不能用不同服务名——那样就变成了两个独立服务，失去了"渐进切流"的意义。

### 四、多集群服务发现

**场景**：北京 K8s 集群 + 上海 K8s 集群 + 裸机服务器

Nacos Server（虚拟集群）
├── 北京 K8s 集群（namespace=beijing）

- 配合 Spring Cloud LoadBalancer 的 `ZonePreferenceServiceInstanceListSupplier`

### 五、mTLS 服务间通信

零信任网络下，服务间调用需要双向 TLS 认证：

```
order-service → inventory-service
  → order-service 出示证书（我是 order-service）
    → inventory-service 验证证书（CA 签名有效？CN 匹配？）
      → inventory-service 出示证书（我是 inventory-service）
        → order-service 验证证书
          → 双向认证通过 → 建立加密连接
```

**实现方案**：

| 方案 | 侵入性 | 适用场景 |
|---|---|---|
| **Istio 自动 mTLS** | 无（Sidecar 处理） | 已有 Service Mesh |
| Spring Cloud Gateway + cert-manager | 网关层 | API 网关统一入口 |
| 应用层（gRPC + TLS） | 代码改动 | 无 Service Mesh 时 |

**2025 年推荐**：如果只用 K8s，优先 Istio 自动 mTLS（零代码改动）。如果有裸机节点，应用层 gRPC + TLS 更可控。

## 附二：训练营补充细节

### 一、GoLang 多语言（子页面 5-8）

**为什么训练营花 4 篇讲 Go**：云原生核心组件（K8s API/etcd/Prometheus/Istio）的客户端基本都是 Go 写的。Java 工程师不需要精通 Go，但需要能看懂和参与。

**Goroutine 并发模型**：M:N 调度——N 个 goroutine 映射到 M 个 OS 线程，不同于 Java 的 1:1 线程模型。内存开销 ~2KB/goroutine，可以轻松启动百万级协程。

**Channel 通信**：无缓冲 Channel 同步阻塞、有缓冲 Channel 异步非阻塞。Go 的 CSP 模型不同于 Java 的共享内存+Lock 模型。

### 二、Open API 设计的 9 个通用问题（子页面 8）

| # | 问题 | 解决方案 |
|---|---|---|
| 1 | API 稳定性 | 版本化 URL（/v1/... / /v2/...）+ 向后兼容 |
| 2 | HTTP Status × API Status | 分开输出：HTTP 2xx/4xx/5xx + API Status Code 枚举 |
| 3 | 多端设计 | REST / SDK（gRPC/TCP）/ CLI |
| 4 | 缓存策略 | 强一致（长连接） vs 弱一致（定时器+短连接） |
| 5 | 查询设计 | 全量 vs 增量（长轮询/SSE/HTTP2 Push） |
| 6 | 健康检查 | HTTP/TCP/gRPC/UDP + 指标可降级 |
| 7 | 模型设计反模式 | 绑定序列化实现 / 文档与代码不一致 |

### 三、Jenkins Managed Files 模板化（子页面 12-13）

**核心思想**：不重复写 Dockerfile 和 K8s YAML——用参数化模板（`${APP_NAME}` / `${JAVA_OPTS}` / `${CLUSTER_ENV}`）。

**2025 年等价方案**：Kustomize（overlay 模式）+ Helm Chart（values.yaml 模板化）。

### 四、GoLang vs Java 并发模型对比

| 维度 | Java 线程（1.x-21） | Go Goroutine | Java 虚拟线程（21+ Loom） |
|---|---|---|---|
| 调度 | OS 线程 1:1 | Go Runtime M:N | JVM 管理 M:N |
| 内存 | ~1MB/线程 | ~2KB/goroutine | ~1KB（栈可动态伸缩） |
| 通信 | 共享内存+Lock | Channel（CSP） | 共享内存+Lock |
| 创建 | `new Thread()` | `go func()` | `Thread.startVirtualThread()` |
