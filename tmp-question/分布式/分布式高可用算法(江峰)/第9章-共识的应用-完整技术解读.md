## 9.1 全序广播——共识 + 广播

### 9.1.1 从共识到全序广播

第6章讲了广播的层次——从尽力到因果。但**没有一种广播保证全局顺序**。全序广播（Total Order Broadcast）填补了这个空缺。

**定义**：全序广播是一个广播协议，满足：
1. **可靠性**：正确的发送方的消息最终被所有正确的进程交付
2. **全序**：所有正确的进程以**相同的顺序**交付消息

### 9.1.2 用共识构建全序广播

多轮序列共识（第8章 8.6 节）正好提供了这个机制：

```
算法——基于序列共识的全序广播：
  状态：
    seq = []                    // 已达成共识的消息序列（初始为空）
    pending = {}                // 待共识的消息集合

  upon Bcast(m):
    pending.add(m)               // 消息加入待共识集合
    
  持续共识循环：
    在 pending 中选择一个消息 m（或"Noop"如果为空）
    运行 序列共识.Propose(m)     // 等待共识轮次
    共识决定后：
      seq.append(m)               // 消息加入已共识序列
      if m ≠ "Noop":
        输出 Deliver(p, m)        // 交付给上层——全局唯一顺序！
```

**关键**：共识协议保证了所有进程在 seq 的第 i 个位置一致。所以所有进程以**相同的顺序**交付消息——这就是全序广播的实现。

**为什么需要 Noop？** 如果没有待共识的消息，序列共识"空缺"的位置必须填补——否则后续的共识轮次无法推进。Noop 填补空缺，保持序列的连续性。

## 9.2 复制状态机——共识 + 状态机

### 9.2.1 从全序广播到复制状态机

第1章介绍了状态机——确定性函数：给定相同的初始状态和相同的输入序列，所有状态机到达相同的状态。

**复制状态机（State Machine Replication, SMR）** 是全序广播 + 状态机的组合：

```
算法——基于全序广播的复制状态机：
  状态：
    state = init       // 当前状态（初始为初始状态）

  upon 全序广播.Deliver(p, cmd):
    state = apply(state, cmd)    // 确定性变换——所有进程相同
```

**正确性**：全序广播保证所有进程以相同顺序交付命令，状态机保证相同输入产生相同状态——所以所有进程的状态一致。

### 9.2.2 SMR 的工程对应

| 理论抽象 | 工程实现 |
|---------|---------|
| SMR = 全序广播 + 状态机 | etcd = Raft + 内存键值树（bbolt持久化） |
| SMR = 全序广播 + 状态机 | ZooKeeper = ZAB + znode树 |
| SMR = 全序广播 + 状态机 | TiKV = Multi-Raft + Region状态 |

### 9.2.3 SMR 的两类状态转移——读和写

**写（Write）** 必须经过全序广播——确保一致。
**读（Read）** 不一定需要——取决于一致性要求：
- **线性一致性读**：必须先执行一次全序（Read 插入到"序列中的某个位置"）——确保读的是最新状态
- **顺序一致性读**：从本地状态机读取（不需要全局序）——但可能读到旧状态

## 9.3 原子提交——共识 + 事务

### 9.3.1 从共识到原子提交

第3章释慧利详细讲了 2PC 的两阶段提交——协调者 + 参与者，原子提交的直觉算法。现在用共识重新构建原子提交。

**原子提交的需求**：
- 一组进程执行各自的本地事务
- 共识决定：全体提交 or 全体回滚——不能部分提交

### 9.3.2 基于序列共识的原子提交

```
算法——基于序列共识的原子提交：
  状态：
    tx_status = PENDING        // 事务状态：PENDING/COMMITTED/ABORTED
    votes = {}                  // 各参与者的投票

  upon 协调者：广播 CommitRequest(tx_id, operations) 给所有参与者

  upon 参与者：执行本地事务→ 决定投票
    如果本地事务成功：vote = YES；否则 vote = NO
    广播 (tx_id, vote) 给所有进程

  共识阶段：
    所有进程收集投票 → 序列共识的共识输入 = {YES, NO}
    如果所有投票都是 YES → 共识决定 COMMIT
    否则 → 共识决定 ABORT

  upon 共识决定：
    if COMMIT：执行本地提交
    if ABORT：执行本地回滚
```

**为什么共识保证原子性？** 共识的 Agreement 性质保证了所有进程看到相同的共识决定——要么全体 COMMIT，要么全体 ABORT。2PC 的"协调者单点故障"问题被共识的"多数派容忍"解决——只需要多数派投票 YES 即可提交，不需要等待所有参与者。

## 9.4 组成员关系——共识 + 成员变更

### 9.4.1 定义

**组成员关系（Group Membership）** 管理分布式系统的节点集合——哪些进程在线、哪些已退出、哪些新加入。

| 性质 | 含义 |
|------|------|
| **全局一致** | 所有进程对成员集合的看法相同 |
| **动态性** | 成员可以随时加入和离开 |

### 9.4.2 基于共识的组成员变更

```
算法——基于序列共识的组成员变更：
  状态：
    members = {p1, p2, ..., pN}  // 当前成员集合

  upon 新进程请求 Join：
    序列共识.Propose(Join(q))   // 共识决定 Join 后，添加到 members

  upon 进程 q 被检测为崩溃（失败检测器◇P输出）：
    序列共识.Propose(Leave(q))  // 共识决定 Leave 后，从 members 移除

  upon 共识决定成员变更：
    更新 members = members ± q
```

**为什么需要共识？** 如果两个节点各自认为成员集合不同——一个认为有 5 个节点，一个认为只有 4 个——它们对"多数派"的认知不同——可能导致两个多数派重叠但各自选定不同的值（脑裂）。

**工程的优化**：第6章释慧利的 Raft Joint Consensus——通过两阶段过渡（Cold,new→Cnew）实现成员变更。第5章释慧利的 ZooKeeper——通过 ZAB 的 epoch 机制实现动态成员管理。

## 9.5 四个抽象的关系——共识是共同的根

```
共识（Consensus）—— 分布式系统的"活水之源"
    │
    ├──► 全序广播（Total Order Broadcast）—— 保证消息顺序
    │       │
    │       └──► 复制状态机（State Machine Replication）—— 保证状态一致
    │
    ├──► 原子提交（Atomic Commit）—— 保证事务全或无
    │
    └──► 组成员关系（Group Membership）—— 保证成员一致
```

**每一个上层抽象都依赖于共识的 Agreement 性质**——保证所有正确进程看到的相同决策。没有共识，就没有全序、没有一致状态、没有全或无事务、没有一致成员。

## 9.6 本章总结

### 3 个核心要点

1. **全序广播 = 共识 + 广播**——通过序列共识的每轮选择消息（或用 Noop 填补），所有进程以相同顺序交付消息。这是保证全局顺序的关键机制。

2. **复制状态机 = 全序广播 + 状态机**——全序保证命令序列一致，状态机保证从命令到达相同状态。etcd/ZooKeeper/TiKV 都是 SMR 的工程实例。

3. **原子提交和组成员关系**同样建立在共识之上——共识的 Agreement 性质保证了所有进程对事务结果和成员集合的看法一致。
