## 2.1 为什么需要形式化模型？

### 2.1.1 自然语言的模糊性

考虑以下用自然语言描述的 Leader 选举算法：

> "每个进程周期性地检查是否有 ID 比自己大的进程存活。如果没有，就宣布自己是 Leader。"

这个描述看起来很清晰，但实际上有大量模糊之处：

- "周期性地"——周期多长？10ms？10s？如果周期太短，网络可能爆炸；如果周期太长，故障检测太慢
- "检查是否有 ID 比自己大的进程存活"——怎么检查？发心跳？发 ping？如果消息延迟怎么办？
- "宣布自己是 Leader"——如果两个进程同时宣布怎么办？其他进程认不认？

**自然语言的模糊性导致"看似正确，实则千疮百孔"的算法实现**。这就是需要形式化模型的根本原因。

### 2.1.2 伪代码的局限性

伪代码比自然语言更好——它定义了变量、条件和循环，更精确。但伪代码也有局限：

1. **无法证明并发正确性**：伪代码执行是顺序的——但分布式系统中多个进程并发执行。伪代码无法显式描述"多个进程同时执行时能否保证一致性"
2. **无法证明活性**：伪代码能描述"如果收到消息 A，则执行 B"，但不能形式化描述"消息 A 最终一定能到达"
3. **无法区分假设和保证**：伪代码混在一起写——哪些是协议自身的保证？哪些是环境提供的假设？不清晰

### 2.1.3 I/O 自动机——为分布式系统设计的形式化模型

I/O 自动机（Input/Output Automaton）是 Nancy Lynch 和 Mark Tuttle 在 1987 年的论文中首次提出的，专门为异步并行系统（包括分布式系统）的建模而设计。

**I/O 自动机的核心思想**：
- 每个进程看作一个自动机——有自己的状态和状态转移规则
- 分布式系统 = 多个自动机的**组合**
- 自动机通过消息（输入/输出事件）交互
- 自动机的行为由它的**迹**（执行事件序列）定义

**I/O 自动机能回答什么问题？**
- **安全性**：自动机的所有迹都满足某个性质吗？（例如：永远不会有两个不同的 Leader）
- **活性**：在公平调度下，某个好的事件最终一定会发生吗？（例如：最终一定会选出一个 Leader）
- **组合正确性**：多个自动机组合后，系统行为仍然满足某些性质吗？

> **跨书对照**：释慧利用伪代码和 ASCII 时序图描述 Paxos/Raft——直观但不够严格。江峰用 I/O 自动机描述同样的算法——严格但抽象。两者互补：释慧利告诉你"算法怎么执行"，江峰告诉你"算法为什么正确"。

## 2.2 I/O 自动机的基本模型

### 2.2.1 状态（State）

一个 I/O 自动机有一组状态变量，共同构成自动机的**状态**。所有可能的状态的集合称为**状态空间**。

**例子——计数器自动机**：

```typescript
// 计数器自动机的状态
counter: number;  // 当前计数值，初始为 0
```

**例子——Leader 选举自动机**：

```typescript
// Leader 选举自动机的状态
leader: process_id | null;  // 当前 Leader，初始为 null
alive: Set<process_id>;     // 已知存活的进程集合，初始为空
```

**关键特性**：
- 状态空间可以是无穷的（如计数器可以无限增长）
- 状态在任意时刻都是**确定的**——没有"介于两个状态之间"
- 状态转移是**原子的**——要么完成，要么没发生

### 2.2.2 操作（Action）——三类事件

自动机的行为由**操作**（Action，也叫事件 Event）驱动。操作分为三类：

| 操作类型 | 谁触发 | 自动机能拒绝吗 | 对外可见吗 |
|---------|--------|-------------|----------|
| **输入操作（Input）** | 运行时环境（外部） | 不能拒绝——输入在任何状态下都"激活" | 是（外部事件） |
| **输出操作（Output）** | 自动机自己 | 可以控制——只在特定状态下触发 | 是（外部事件） |
| **内部操作（Internal）** | 自动机自己 | 可以控制 | 否（仅自动机内部可见） |

**签名的形式化定义**：

自动机 A 的签名（Signature）是所有可能操作的集合，分为三个不相交的子集：

```
acts(A) = in(A) ∪ out(A) ∪ int(A)
```

- `in(A)`：输入操作集合——自动机从环境接收
- `out(A)`：输出操作集合——自动机向环境发送
- `int(A)`：内部操作集合——自动机内部执行

**例子——计数器自动机的签名**：

```
in(A)  = {Increment, Reset}       // 接收"加1"或"重置"请求
out(A) = {Ack}                     // 回复"完成"
int(A) = {}                        // 没有内部操作
```

**例子——Leader 选举自动机的签名**：

```
in(A)  = {Heartbeat(from, pid), VoteRequest(from, pid)}
out(A) = {Elect(pid), VoteReply(to, vote)}
int(A) = {Timeout}                 // 内部超时触发选举
```

**关键规则**：
1. 输入操作在任何状态下都是激活的——自动机不能拒绝接收输入
2. 输出操作和内部操作只在特定状态下激活——自动机可以选择何时触发
3. 每一步（Step）是原子的——一个自动机一次只能执行一步，不能同时执行两步

### 2.2.3 状态转移（Transition）

状态转移定义了"自动机在状态 s 下收到/触发操作 e，会变成什么新状态 s'"：

```
s ──e──► s'
```

状态转移是确定性的——给定状态 s 和操作 e，新状态 s' 是唯一确定的。

**例子——计数器自动机的状态转移**：

```
// s: counter = k, e: Increment
s' → counter = k + 1

// s: counter = k, e: Reset  
s' → counter = 0

// 任何状态下收到 Increment 或 Reset
// 自动机必须接受（输入操作在任何状态下激活）
```

**例子——Leader 选举自动机的状态转移**：

```
// s: leader = null, e: Timeout（内部超时触发）
s' → 广播 VoteRequest(self) 给所有进程

// s: leader = null, e: VoteReply(to=self, vote=yes)
if 收到多数派 yes 投票 → s': leader = self

// s: leader = self, e: Heartbeat(from=p3, pid=p3)
s': alive += p3  // 记录 p3 存活
```

### 2.2.4 迹（Trace）

自动机的**迹**（Trace）是执行过程中所有**外部操作**（输入 + 输出）的序列。

自动机执行的一串操作（输入、输出、内部操作）称为一个**执行**（Execution）。去掉内部操作后，剩下的外部操作序列就是**迹**。

```
执行:  Increment → (内部: 超时) → VoteRequest → VoteReply → Ack
迹:    Increment → VoteRequest → VoteReply → Ack
                                 （去掉了"超时"——内部操作对外不可见）
```

**迹为什么重要？** 因为外部观察者只能看到外部操作——不能看到自动机的内部状态。两个不同的自动机可能有相同的迹，但内部状态不同。迹定义了自动机的"外部行为"。

**迹是分析正确性的关键**：
- 安全性：自动机的所有迹都满足某个性质（例如：任何迹中都不会出现两个不同的 Elect(pid) 操作）
- 活性：在公平调度下，任何迹的有限前缀最终都会包含某个好的操作（例如：最终出现一个 Elect(pid) 操作）

## 2.3 输入、输出和内部操作的深入理解

### 2.3.1 为什么输入操作不能被拒绝？

**直觉**：输入操作代表环境向自动机"投递"信息——自动机无法阻止环境向它发送消息。

**例子**：链路自动机收到 `Send(m)`——它必须接受这个消息。如果链路自动机可以拒绝接收消息，它就不是"输入"了——它是在做"协议决策"。

**设计原则**：
- **输入 = 环境赋予的义务**——自动机必须处理
- **输出 = 自动机对环境的要求**——自动机可以选择何时发出

### 2.3.2 输出操作和内部操作的区别

两者都是自动机自己触发的，区别在于**可见性**：
- 输出操作对**运行时环境**可见——被看做自动机与外部环境通信的手段
- 内部操作只对**自动机自身**可见——外部无法观察

**为什么需要内部操作？**
- 自动机的某些行为不应该暴露给外部——例如 Leader 选举的"检查超时"是内部操作
- 隐藏实现细节——类似于面向对象编程的 `private` 方法

### 2.3.3 输入激活——在任何状态下都能接收

每个 I/O 自动机的输入操作在**所有状态**下都是激活的——这是 I/O 自动机和其他状态机模型的本质区别。

**含义**：环境随时可以向自动机发送输入消息——自动机不能假设"输入不会在某个状态到来"。如果输入在该状态下没有定义转移规则，说明自动机的状态空间设计有缺陷。

**例子**——Leader 选举自动机必须处理"在选举过程中收到旧 Leader 的心跳"：

```
状态: leader=self, 正在选举中
输入: Heartbeat(from=p3, pid=p1)  // p3 说 p1 是 Leader

转移: 如果 self > p1 → 忽略（我的 ID 更大，继续当 Leader）
      如果 self < p1 → leader = p1（p1 的 ID 更大，退位）
```

如果自动机没有为此输入定义转移规则——在选举过程中收到 Heartbeat 就会卡住——算法设计有漏洞。

## 2.4 组合模型——从单个自动机到分布式系统

### 2.4.1 并发组合（Parallel Composition）

分布式系统 = 多个自动机的组合。N 个进程 = N 个自动机同时运行。

**组合的基本规则**：
1. 合成自动机的状态 = **所有部件自动机状态的元组**——`(s₁, s₂, ..., s_N)`
2. 当一个部件自动机执行输出操作 e（同时 e 是另一个部件自动机的输入操作）——两个自动机**同时**进行状态转移（共享同一个操作 e）
3. 当一个部件自动机执行内部操作——只有它自己进行状态转移
4. 当一个部件自动机执行输出操作 e（e 不是任何部件自动机的输入操作）——合成自动机向环境输出 e

**例子——Leader 选举的并发组合**：

```
进程 p1 的自动机: leader=null, 超时触发 → 输出 Elect(p1)
                                ↓
进程 p2 的自动机: 输入 Elect(p1) → 比较 ID → 
                   如果 p1.id > p2.id → 更新 leader=p1
                   如果 p1.id < p2.id → 忽略（自己的 ID 更大）
```

**关键规则**：当 p1 的输出操作 Elect(p1) 同时是 p2 和 p3 的输入操作时——p1、p2、p3 同时进行状态转移。这模拟了"广播"——一条消息被多个接收者同时处理。

### 2.4.2 兼容条件

两个自动机可以组合的**前提条件**：
1. A 的内部操作不在 B 的签名中（内部操作不对外可见）
2. A 和 B 没有同名输出操作（两个自动机不能输出相同的事件——否则无法区分是谁触发的）

### 2.4.3 隐藏操作（Hiding）

组合后的自动机可以**隐藏**（Hiding）部分输出操作——将其变为内部操作。隐藏后的操作对外部环境不可见。

**为什么需要隐藏？** 模块化——底层操作不需要暴露给上层。

**例子**——加密通信的隐藏：

```
加密自动机: 输入 Read(m) → 内部加密 → 输出 Send(m')
发送链路自动机: 输入 Send(m') → 放入发送队列

组合后:
  输出 Send(m') 被隐藏——外部只看到 Read(m) 和 Receive(m)（解密后）
  加密过程完全透明
```

这与软件工程中的"抽象层"相同——底层的实现细节通过隐藏操作封装。

## 2.5 用迹定义正确性——安全性和活性

### 2.5.1 迹集（Trace Set）

自动机 A 的所有可能的外部操作序列的集合称为**迹集**（Traces of A），记为 `traces(A)`。

**迹集不是固定不变的**——它取决于环境如何与自动机交互。不同的环境产生不同的迹集。但自动机必须保证**所有可能的迹**都满足安全性和活性属性。

### 2.5.2 用迹集定义安全性

**安全性在 I/O 自动机中的形式化定义**：

自动机 A 满足安全性属性 P，当且仅当 `traces(A) ⊆ P`。

也就是说——对于自动机的所有可能的迹，P 都成立。不存在任何迹使得 P 不成立。

**例子**——Leader 选举的安全性：

```
安全性属性: "任何迹中都不会出现两个不同的 Elect(pid) 操作"

traces(A) 中所有迹都满足这个性质 → A 满足安全性 ✓
如果存在一条迹出现 Elect(p1) 和 Elect(p2) → A 不满足安全性 ✗
```

### 2.5.3 用公平性定义活性

活性不能只用迹集定义——还需要**公平性**（Fairness）假设。

**公平性的直觉**：如果自动机持续处于"可以触发操作 e"的状态，环境最终会让 e 执行。

**公平性条件**（弱公平性，Weak Fairness）：
- 如果操作 e 在某个时刻激活（enable），并且此后一直保持激活，则最终 e 会被执行
- 如果操作 e 在某个时刻激活，但随后变为未激活状态——不需要保证 e 被执第

**强公平性（Strong Fairness）**：
- 如果操作 e 在无限次变为激活状态，则最终 e 会被无限次执行

**活性在 I/O 自动机中的定义**：

在公平调度下，对于迹的任何有限前缀，都存在一个扩展使得最终某个"好"的操作出现。

**例子**——Leader 选举的活性：

```
活性属性: "最终会选出一个 Leader"

在公平调度下：
  每个选举超时后自动机触发 Elect 操作
  Elect 操作被接受或拒绝
  最终有一个进程获得多数派投票
  → 迹的有限前缀后必然出现 Elect(pid) 操作 ✓
```

## 2.6 抽象与实现——自动机之间的关系

### 2.6.1 实现关系

**定义**：自动机 B **实现**了自动机 A，当且仅当：

1. `in(A) ⊆ in(B)`——B 接受所有 A 的输入（可能接受更多）
2. `out(B) ⊆ out(A)`——B 的所有输出都是 A 的合法输出（不会输出 A 没有的操作）
3. `traces(B) ⊆ traces(A)`——B 的所有迹都是 A 的合法迹

**直觉**：B 是 A 的"细化"——B 的行为 ⊆ A 的行为。

**与软件工程的类比**：B 实现了接口 A。B 可以处理更多的输入（更宽容），但不会产生接口 A 未定义的输出（不引入新行为）。

### 2.6.2 为什么需要抽象？

I/O 自动机的"抽象→实现"关系是分布式算法设计的核心方法：

1. 先定义**理想化抽象**（高层自动机）——描述"系统应该做什么"
2. 再设计**具体实现**（底层自动机）——描述"系统在故障假设下怎么做"
3. 证明实现自动机的迹 ⊆ 抽象自动机的迹——即"实现不会产生抽象没有的坏行为"

**例子**——可靠链路的抽象和实现：

```
抽象自动机（可靠链路）:
  input: Send(m)
  output: Receive(m)
  迹性质: 每条 Send(m) 最终对应唯一的 Receive(m)，不丢不重复

实现自动机（公平丢包链路 + 重传）:
  input: Send(m)
  output: Receive(m), Ack(n), Retransmit(m)
  (多了 Ack 和 Retransmit 操作——隐藏后与抽象等价)
```

**证明思路**：
1. 隐藏实现自动机的 Ack 和 Retransmit 操作
2. 证明隐藏后的迹集 ⊆ 抽象自动机的迹集
3. → 实现自动机"实现了"可靠链路抽象 ✓

## 2.7 案例：用 I/O 自动机建模 Paxos Acceptor——形式化如何发现漏洞

前面讲了 I/O 自动机的概念，但你可能会问：**这套形式化工具到底有什么用？** 这一节用一个具体的例子回答：用 I/O 自动机建模 Paxos Acceptor（第4章的核心角色），展示形式化分析如何**在设计阶段就发现算法的潜在漏洞**。

### 2.7.1 直觉实现——感觉上没问题

Paxos Acceptor（第4章 4.3 节已学过）的直觉规则：

1. 收到 Prepare(n)：如果 n > promised_n，承诺不再接受编号 < n 的提案，返回 Promise(n, accepted_proposal)；否则拒绝
2. 收到 Accept(n, v)：如果 n >= promised_n，接受提案，持久化；否则拒绝

**直觉实现**（伪代码）：

```python
promised_n = 0
accepted_proposal = None

def on_prepare(n):
    if n > promised_n:
        promised_n = n
        return Promise(n, accepted_proposal)
    return Nack

def on_accept(n, v):
    if n >= promised_n:
        accepted_proposal = (n, v)
        return Accepted(n)
    return Nack
```

看起来没问题——大于和大于等于的逻辑是对的。但请停下来想一想：**你能保证这个实现满足 Paxos 的 Safety 吗？**

### 2.7.2 用 I/O 自动机建模——发现漏洞

**状态定义**：

```typescript
// Acceptor 自动机的状态
state = {
    promised_n: 0 | 1 | 2 | ...          // 已承诺的最大编号（初始为 0）
    accepted_proposal: null | (n, v)       // 已接受的提案（初始为 null）
}
```

初始状态：(promised_n=0, accepted_proposal=null)

**签名（三类操作）**：

```
in(Acceptor)  = {Prepare(n), Accept(n, v)}     // 从 Proposer 接收
out(Acceptor) = {Promise(n, prev), Accepted(n)} // 向 Proposer 发送
int(Acceptor) = {}                              // 没有内部操作
```

**状态转移规则**——从直觉实现翻译为 I/O 自动机：

| 当前状态 | 输入操作 | 新状态 | 输出操作 |
|---------|---------|--------|---------|
| promised_n=m, any acc | Prepare(n), n > m | promised_n=n | Promise(n, accepted_proposal) |
| promised_n=m, any acc | Prepare(n), n <= m | **不变** | Nack |
| promised_n=m, any acc | Accept(n,v), n >= m | accepted_proposal=(n,v) | Accepted(n) |
| promised_n=m, any acc | Accept(n,v), n < m | **不变** | Nack |

### 2.7.3 分析迹——发现一个关键问题

考虑以下执行迹（p=Proposer, a=Acceptor）：

```
迹 1:
  Prepare(5) → Promise(5, null) → Accept(5, v₁) → Accepted(5)
  
迹 2（不同 Proposer 尝试"跳过"Promise）:
  Accept(3, v₂) → Accepted(3)
  
问题：迹 2 中，Acceptor 接受了 (3, v₂)
  但 promised_n 没有被 Prepare(3) 设置——promised_n 仍然是 0！
```

**迹 2 的具体执行**：

```
s₀: (promised_n=0, acc=null)
  e₁: Accept(3, v₂)  // n=3 >= promised_n=0 → 接受
s₁: (promised_n=0, acc=(3, v₂))
  e₂: Accepted(3)
```

**这个迹说明什么？** Acceptor 在没有收到 Prepare(3) 的情况下，**直接接受了 Accept(3)**——因为 `n >= promised_n` 检查通过了（3 >= 0 = True）。

**这有问题吗？** 取决于算法假设。如果 Proposer 按照 Paxos 规范**必须先 Prepare 再 Accept**，那 Acceptor 接受"未经 Prepare 的 Accept"不算 Bug——Proposer 不会发这样的请求。但如果系统被攻击或被有缺陷的 Proposer 调用，Acceptor 接受了"野 Accept"——这就是安全性问题。

### 2.7.4 形式化如何指导设计——补上漏洞

通过迹分析发现：Promised_n 的初始值为 0 导致 Acceptor 可能接受编号很小的"野 Accept"。修复方案：

**方案一**：让 Acceptor 也检查"Accept 必须对应一个已完成的 Prepare"

```python
def on_accept(n, v):
    if n == promised_n:  # 严格等于——必须是当前承诺的编号
        accepted_proposal = (n, v)
        return Accepted(n)
    return Nack
```

这样 Accept 只接受"当前 Prepare 批次的提案"——不接受更小或更大的提案。（注意：这改变了 Paxos 的语义——需要评估性能影响）

**方案二**：Promised_n 初始值设为 -1，确保第一个 Prepare 必须是 >= 1

```python
promised_n = -1  # 初始值 < 0，确保任何合法提案 n >= 0 都 > promised_n
```

### 2.7.5 这个案例的启示

**没有形式化分析时**：你看伪代码——感觉没问题——写代码——测试通过——上线——偶尔出现不一致。

**有形式化分析时**：你建模 Acceptor 为 I/O 自动机 → 检查所有可能的状态转移序列（迹）→ 发现"接受野 Accept"的漏洞 → 在设计阶段修复。

**这个模式在后续章节中反复出现**：
- 第4章：用 I/O 自动机定义"公平丢包链路"，证明重传算法实现可靠链路
- 第5章：用迹集定义"完美失败检测"，证明超时算法实现最终失败检测
- 第8章：用迹集定义"正则共识"，证明 Paxos 实现正则共识

**形式化的价值不是一个抽象概念**——它是**在设计阶段抓出并发 Bug** 的工具。就像单元测试抓出代码 Bug，形式化分析抓出算法 Bug。

## 2.8 本章总结

### 5 个核心要点

1. **I/O 自动机是分布式算法的形式化模型**——用状态 + 操作 + 转移规则精确定义进程行为。相比自然语言和伪代码，它可以被严格证明。

2. **三类操作**：输入（环境投递，不可拒绝）、输出（向环境发送，可控制）、内部（自动机内部，对外不可见）。签名 = 输入 ∪ 输出 ∪ 内部。

3. **组合模型**：多个自动机通过共享输出/输入操作并发执行。合成自动机的状态是所有部件状态的元组。隐藏操作封装底层实现细节。

4. **用迹集定义正确性**：安全性 = 所有迹满足某个性质。活性 = 在公平调度下，好事最终发生。

5. **抽象→实现**：高层自动机定义"系统应该做什么"，底层自动机定义"系统怎么做"。实现正确性 = 实现自动机的迹 ⊆ 抽象自动机的迹。
