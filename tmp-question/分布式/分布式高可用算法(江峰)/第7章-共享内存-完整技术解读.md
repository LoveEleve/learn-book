## 7.1 共享内存的挑战——从单机到分布式

### 7.1.1 单机的共享变量——你已经习以为常

在单机程序中，一个变量被多个线程读写：

```java
int counter = 0;  // 全局变量
// 线程1: counter++;
// 线程2: int x = counter;
```

你通过 `volatile` 或 `synchronized` 保证一致性。这些机制依赖**共享内存**——所有线程访问同一块物理内存。

### 7.1.2 分布式的共享变量——没有共享内存

在分布式系统中，没有物理共享内存。你的`counter`需要被复制到多个节点上。如果多个节点同时读写——

```
时间 T0: p1 写 counter=5
时间 T1: p2 写 counter=3
时间 T2: p3 读 counter——应该读到什么？
```

**没有全局时钟，你无法判断 T1 和 T2 的先后顺序**。p3 读到 5 或 3 都可能是"正确的"，取决于你定义的一致性模型。

### 7.1.3 注册器——分布式的共享变量抽象

**注册器（Register）** 是一个分布式共享变量。它有两个接口：

```
input:  Write(v)      // 写进程调用：将值 v 写入注册器
output: Read() → v    // 读进程调用：从注册器读取值，返回 v
```

注册器的正确性由**一致性模型**定义——它规定了"在什么情况下，读操作应该返回什么值"。

**注册器的层次结构**（从最弱到最强）：

```
正则注册器（最弱：读返回最近写或初始值）
    │
    ├──► 原子注册器（线性化：读写的全序）
    │
    ├──► 顺序注册器（顺序一致：按进程顺序）
    │
    ├──► 因果注册器（因果顺序）
    │
    └──► FIFO 注册器（同一写进程的顺序不动）
```

**每一层定义了"更严格"的读写一致性保证**——你需要根据业务需求选择最合适的注册器类型。

> **跨书对照**：释慧利第1章 1.6.4 节定义了一致性谱系（线性→顺序→因果→最终）。江峰第7章的注册器层次就是一致性谱系在"共享变量"层面的具体实例化——释慧利讲"一致性是什么"，江峰讲"一致性在共享变量中怎么实现"。

## 7.2 如果你来设计——从最简单的注册器开始

### 7.2.1 单写单读——最简单的场景

先考虑最简单的情况：只有一个写进程（p_w）和一个读进程（p_r）。

**直觉方案**：写进程发送新值给读进程，读进程接收后更新本地副本。

```
p_w: 每次 Write(v) → 可靠链路.Send(p_r, v)
p_r: 每次 可靠链路.Receive(p_w, v) → 本地副本 = v
     每次 Read() → 返回本地副本
```

**这个方案有什么问题？** 如果读进程在 Read() 时，写进程正在发送新值——但 Read() 会立即返回"当时的旧值"——这不违反任何性质（因为单写单读，只需要返回最近写或初始值）。

**这个最简单的注册器已经满足了一个性质**——读返回的值一定是**写进程在过去某个时刻写的值**，而不是凭空捏造的值。

### 7.2.2 多写单读——需要写顺序

现在增加了一个写进程 p_w2。两个写进程都能写。读进程从两个来源接收 Write 消息。

**新增问题**：读进程收到了 p_w1 的 Write(5) 和 p_w2 的 Write(3)。它应该以什么顺序应用这两个写操作？如果先后收到 (5) 然后 (3)——读进程的本地副本应该是 3。但反过来——(3) 先到、(5) 后到——应该是 5。

**读进程看到的顺序取决于消息到达的顺序**——这未必是写操作发生的真实顺序。这就是注册器需要处理的"并发写"问题。

## 7.3 正则注册器——最基础的正确性

### 7.3.1 定义

**正则注册器（Regular Register）** 是最基础的注册器。它满足：

| 性质 | 含义 |
|------|------|
| **有效性** | 读返回的值是某个写操作写入的值（或者是初始值）——不会凭空产生值 |
| **正则性** | 如果读操作 R 在写操作 W 完成之后**开始**，R 必须返回 W 的值或比 W 更新的值 |

### 7.3.2 时间线解释

```
写 W1: Write(5) ──开始───完成──
                               读 R1: Read() ──开始──► 返回 ≥5（W1 已在 R1 之前完成）
写 W2: Write(3) ──开始──（并发，与 R1 重叠）
```

- **R1 必须返回 ≥5**——因为 W1 在 R1 开始之前已完成
- **R1 可以返回 5 或 3**——因为 W2 与 R1 是并发的（重叠执行）

### 7.3.3 正则注册器不能做什么

```
写 W1: Write(5) ──开始───────完成──
写 W2: Write(3)          ──开始───完成──
                                      读 R1: Read() 开始 → 返回 ???
```

如果 W1 和 W2 都与 R1 不重叠（都在 R1 开始之前完成了），正则注册器**不保证**R1 必须返回最新的值（W2的值=3）。R1 可能返回 W1 的值（5），即使 W2 在 W1 之后完成。这是因为**重叠写的不确定性**——没有规定"哪个是最后一个完成的写"。

## 7.4 原子注册器——线性化

### 7.4.1 定义

**原子注册器（Atomic Register）**——即**线性化注册器**——是所有注册器中最强的。它满足：

| 性质 | 含义 |
|------|------|
| **线性化** | 所有读写操作可以被排序为一个全局全序，使得：① 如果操作 A 在操作 B 开始之前已完成，则 A 排在 B 之前（实时序）；② 读返回的值等于全序中最后apply的写操作的值 |

### 7.4.2 与正则注册器的区别

**正则注册器**：R1 只需要返回"≥ W1"——不指定 W1 和 W2 谁是最新的

**原子注册器**：R1 必须返回**线性化顺序**中最后apply的写

```
写 W1(5): ═══开始═══════完成═══
写 W2(3):      ═══开始════完成══
                                读 R1: ═开始═► 返回 ??

正则注册器：R1 ≥ W1 或 R1 ≥ W2 —— 返回 5 或 3 都合法
原子注册器：R1 必须在全序中反映最后完成的写操作 —— 返回 3（假设 W2 在 W1 之后完成）
```

### 7.4.3 为什么原子注册器这么难实现？

你要构建一个原子注册器——读操作必须返回全局全序中最新的写结果。但分布式系统中**没有全局时钟**——你怎么判断 W1 和 W2 哪个是"最后完成的"？

**需要共识**：要确定多个写操作的全序，你要让所有节点就这个全序**达成一致**——这就是共识算法（Paxos/Raft）解决的问题。原子注册器需要在底层的写操作上执行一次共识——这是它的性能代价。

**这也是为什么第8章的共识如此重要**——它提供了原子注册器的实现基础。

## 7.5 顺序注册器

### 7.5.1 定义

**顺序注册器（Sequential Register）** 的要求比原子注册器弱：

| 性质 | 含义 |
|------|------|
| **顺序一致** | 所有操作可以被排序为一个全序，但**不需要**尊重实时先后——只需要每个进程看到的顺序一致 |

### 7.5.2 与原子注册器的区别

```
写 W1(5): p_w ═══开始════完成═══
写 W2(3): p_w ════开始═════════完成═══
读 R1:    p_r                ═开始═► 返回 ??

原子注册器：必须返回 3（W2 是 p_w 最后完成的写）——因为 p_w 在分布式时间线上最后写的是 3
顺序注册器：可以返回 5，只要 p_r 之后的所有读操作也看到一致的顺序
```

**顺序注册器为什么可能返回 5？** 因为顺序一致性不约束"实时先后"——它只约束"同一个进程的写入顺序"。p_w 的两次写入的顺序是(5)然后(3)——但顺序注册器允许其他进程**以不同的顺序感知**，只要这个感知在所有进程间是一致的。

### 7.5.3 工程实践——ZooKeeper 的顺序一致性

ZooKeeper 的写操作通过 ZAB 保证 primary order——所有 Follower 看到相同的操作顺序。这是标准的**顺序一致性**（Sequential Consistency）。第5章释慧利正是这样评价的——ZooKeeper 提供顺序一致性，非线性一致性。

## 7.6 因果注册器与 FIFO 注册器

### 7.6.1 因果注册器

比顺序注册器更弱——只保持**因果顺序**，不保证全局全序。

有因果关系的写操作——例如 W1(5) 触发了 W2(3)——必须被所有进程按因果顺序感知：
- 如果 p_r 先看到 W2(3)（果），必须也先看到 W1(5)（因）
- 因果顺序通过第6章的**向量时钟**检测

### 7.6.2 FIFO 注册器

最弱的顺序——只保证**同一个写进程的写入顺序**不被打破。如果 p_w 先 Write(5) 再 Write(3)，所有读进程看到的是 (5)→(3)，不能是 (3)→(5)。

但不同写进程的写入——p_w1 Write(5), p_w2 Write(3)——可以以任意顺序呈现。

## 7.7 注册器的形式化——用 I/O Automata 定义

第2章学了 I/O Automata。现在用它来精确描述注册器的性质——而不是依赖直觉。

### 7.7.1 正则注册器的 I/O Automata

```
自动机定义（RegularRegister[p_w→p_r]，p_w=p_r 也是合法的）：
  状态：
    val = v0                 // 注册器当前值（初始为 v0）
    writing = None           // 正在进行的写操作（或 None）

  签名：
    input = {Write(v) | v ∈ V}   // 来自写进程 p_w 的写操作
    output = {Read() → v}        // 读进程 p_r 的读操作，返回 v

  转移规则：
    写操作 Write(v)：
      val = v
      writing = v
    
    读操作 Read()：
      if writing != None：
        返回 writing    // 正在写的值优先
      else：
        返回 val        // 返回最近一次完成的写的值
```

**关键点**：正则注册器保证"如果写操作 W 在读 R 开始之前已完成，R 返回 W 的值"。这通过 writing 的状态来实现——正在进行的写不会影响读结果，已完成的写会。

### 7.7.2 原子注册器的完整推演

假设 3 个进程：p_w（写进程），p_r1 和 p_r2（读进程）。

```
T0: p_w 发起 Write(5)
T1: Write(5) 到达 p_r2 → p_r2 更新本地副本 = 5
T2: p_w 发起 Write(3)
T3: Write(3) 到达 p_r1 → p_r1 更新本地副本 = 3
T4: p_r2 执行 Read() → 返回 5（因为 p_r2 还没收到 Write(3)）
T5: Write(3) 到达 p_r2 → p_r2 更新本地副本 = 3
T6: p_r1 执行 Read() → 返回 3

p_r2 看到：Write(5) → Read=5 → Write(3)（不一致！-- 正常注册器允许）
p_r1 看到：Write(3) → Read=3 → Write(5)（迟迟的 5）
```

**这种乱序是正则注册器的合法行为——但不满足原子注册器**。

**原子注册器要求**：存在一个全局全序，使得所有进程的观察都一致。在这个例子中，全序可能是：
```
Write(5) → Write(3) → Read(p_r2)=3 → Read(p_r1)=3
```
（但 p_r2 读到的却是 5，不是 3——违反全序！）

### 7.7.3 为什么原子注册器需要共识？

要让 p_r2 在 T4 时读到 3（全局全序要求），p_r2 在 T4 时必须**等待**确认 p_w 是否发出了 Write(3)——这需要跨节点协调。

而跨节点的"值确认"正是共识（Paxos）要做的事——让所有节点就"哪个值是最近的写入"达成一致。

**共识为原子注册器提供"全序"——注册器提供分布式存储**。两者组合构成复制状态机的基础。

## 7.8 CAP 理论——注册器的终极约束

### 7.7.1 注册器 VS CAP

第1章的 CAP 理论告诉你：一致性（C）、可用性（A）、分区容错（P）不能三者兼得。现在在注册器的框架下，你看到 CAP 的具体实例：

| 注册器类型 | CAP 选择 | 所需机制 | 性能代价 |
|-----------|---------|---------|---------|
| 原子注册器 | CP（一致性） | 需要共识（Paxos/Raft） | 最高延迟 |
| 顺序注册器 | CP（一致性） | 需要 Leader 保序 | 较高延迟 |
| 因果注册器 | AP（可用性） | 向量时钟 | 中等延迟 |
| FIFO 注册器 | AP（可用性） | 简单序列号 | 低延迟 |
| 正则注册器 | AP（可用性） | 无需共识 | 最低延迟 |

**你选择哪种注册器 = 你选择哪个 CAP 分支**。

### 7.7.2 CAP 的形式化证明——注册器框架

用注册器重新证明 CAP——比第1章的直觉证明更严谨：

**反证法**：假设存在一个满足一致性（C）和可用性（A）的原子注册器，在发生网络分区时仍能正常工作。

考虑两个节点 N1 和 N2。网络分区——N1 和 N2 之间的通道断开。

```
T0: p1（写入方，连接 N1）发出 Write(1) 给 N1
T1: p2（读取方，连接 N2）发出 Read() 给 N2

一致性要求：Read 返回全序中最后 apply 的 Write 的结果 → 必须返回 1
可用性要求：N2 必须响应 Read() —— 不能阻塞等待（因为 N1 分区了）

N2 无法区分 "N1 还活着，只是分区了" 还是 "N1 崩溃了"
→ N2 不能响应 Read() 为 1（不知道 N1 是否有未完成的 Write）
→ 违反可用性（必须响应）！
```

**这个矛盾证明**：无法同时满足一致性（C）和可用性（A）在分区发生时。两个选择：
- 选择 C（一致性）：拒绝 Read() → 违反 A（不响应），但返回正确的值
- 选择 A（可用性）：返回任意值 → 违反 C（返回的可能是旧值），但立即响应

### 7.7.3 工程启示

- **并发写少、读多** → 因果注册器（AP）——最大吞吐
- **需要全局全序** → 原子注册器（CP）——最高一致性
- **生产系统通常在中部选择**：ZooKeeper（顺序注册器），etcd（原子注册器）

## 7.8 本章总结

### 5 个核心要点

1. **注册器是分布式共享变量的抽象**——从单机的共享内存到分布式的消息传递，注册器的接口（Write/Read）统一抽象了"多进程读写一致性"的问题。

2. **注册器的层次结构**：正则（最弱）→ 原子（线性化，最强）→ 顺序 → 因果 → FIFO。每一层定义了更严格的读写一致性保证——你根据需要选择。

3. **原子注册器的实现需要共识**——要在多个写操作中确定全局全序，你需要共识算法（Paxos/Raft）。这是分布式系统从"注册器"到"共识"的自然过渡（第8章）。

4. **CAP 理论在注册器框架下的证明**：原子注册器（CP）在分区时无法同时满足一致性和可用性。你的注册器选择 = CAP 分支选择。

5. **工程实践**：ZooKeeper = 顺序注册器（ZAB 保证 primary order），etcd = 原子注册器（Raft 保证线性一致性）。
