## 5.1 如果你来设计——分布式系统的"瞎子"困境

### 5.1.1 场景：3 节点集群，Leader 可能崩溃

你有一个 3 节点的集群，进程 q 是 Leader。其他进程通过消息传递与 q 通信——没有共享内存，没有全局时钟。有一天，你发现 q 不再发送心跳了。

**你的判断应该是什么？**

选项 A：q 崩溃了——发起选举，选新 Leader
选项 B：q 还活着，只是网络延迟——继续等待

**问题**：只凭"心跳停了几秒"——你无法区分 A 和 B。这就是分布式系统的**"瞎子"困境**——你只能看到本地状态和收到的消息，看不到全局。

### 5.1.2 直觉方案：超时判断

最直觉的方案——设定一个超时 T：

```
如果 now - last_heartbeat[q] > T：认为 q 崩溃了
否则：q 存活
```

**问题**：T 设多大？

```
如果 T = 100ms：在正常网络中正确，但网络偶尔抖动（RTT>300ms）
                → 误判 q 崩溃 → 错误地选举新 Leader → 两个 Leader

如果 T = 5s：即使网络偶尔抖动也不会误判，但 Leader 真的崩溃了
              → 需要等 5s 才能选新 Leader → 5s 内集群不可用
```

**这就是失败检测的根本权衡**：超时太小 → 假阳性（误判）；超时太大 → 检测太慢。

### 5.1.3 形式化定义——把直觉转换为数学

**失败检测器（Failure Detector）** 是一个分布式模块。每个进程 p 本地运行一个实例，它监听其他进程发送的心跳，输出一个"被怀疑已崩溃"的进程集合：

```
suspects(D, p, t) = {q | p 的检测器在时刻 t 怀疑 q 已崩溃}
```

**两类错误**：

| 错误 | 直觉含义 | 后果 |
|------|---------|------|
| **假阳性（误判）** | 认为 q 崩溃，但 q 还活着 | p 错误地发起 Leader 选举 → 两个 Leader |
| **假阴性（漏判）** | 认为 q 活着，但 q 已崩溃 | 集群没有 Leader → 不可用 |

**你的超时方案能保证哪种错误不发生？** 答案是——**哪种都不能保证**，因为它依赖一个你不知道的值：网络的最大延迟。这就是形式化的价值——它迫使你明确"在什么假设下保证什么"。

## 5.2 完美失败检测——如果你知道消息延迟上界

### 5.2.1 假设：同步系统

先考虑一个理想状态——假设你知道消息延迟的上界 D。这是第3章 3.6.1 节定义的**同步系统**：

- 每条消息在 ≤ D 时间内到达
- 每个正确进程在 ≥ S 时间内完成一步

### 5.2.2 完美失败检测器 P 的定义

在同步假设下，你可以设计一个**完美失败检测器（Perfect Failure Detector, P）**。它满足两个性质：

| 性质 | 你翻译成自己的话 | 形式化定义 |
|------|----------------|-----------|
| **强完备性** | "漏不掉——崩溃的最终都会被检测到" | ∀崩溃的 q，∀正确的 p：∃T 使得 ∀t>T，q ∈ suspects(D,p,t) |
| **强准确性** | "不冤枉——没崩溃的绝不怀疑" | ∀正确的 q，∀正确的 p：∀t，q ∉ suspects(D,p,t) |

**为什么完美？** 不漏判 + 不误判。这是你理想中的"百分百正确"。

### 5.2.3 算法——基于心跳 + 超时

```
算法（p 的完美失败检测器，假设同步系统）：

状态：
  alive = {}              // 已知存活的进程
  last_hb[q] = T0         // 上次收到 q 的心跳的时刻（初始为当前时间）
  timeout = D + δ         // 超时阈值：已知延迟上限 D + 安全冗余 δ

操作：
  每 heartbeat_interval 时间：
    广播 Heartbeat(p, now)

  upon 收到 Heartbeat(q):
    last_hb[q] = now
    if q 之前被怀疑了：
      alive.add(q)

  持续检测（后台循环）：
    if now - last_hb[q] > timeout：
      alive.remove(q)
      suspects.add(q)
```

**为什么这个算法是完美的？** 分两步证明：

**不误判的证明**：
- 假设 q 正确。p 收到 q 的 Heartbeat 后（last_hb[q] = now），直到 next Heartbeat 发出后（≤ heartbeat_interval 后），p 才会检测超时
- 新的 Heartbeat 在 ≤ D 时间内到达（同步假设保证）
- 所以 last_hb[q] 被更新后，p 必须在 ≤ heartbeat_interval + D 时间内再次收到 Heartbeat
- 超时阈值 = D + δ（合理设置，δ 是冗余）→ 永远不会有 now - last_hb[q] > D + δ
- 所以 q 永远不会被误判 ✓

**不漏判的证明**：
- 假设 q 崩溃。最后一次 Heartbeat(q) 发出后（时间为 H），q 不再发送任何消息
- 对于任何正确的进程 p：在 H 之后的所有时刻，p 的 last_hb[q] = H
- 在经过 timeout 后（H + D + δ），now - last_hb[q] > D + δ → p 判定 q 崩溃
- q 被添加到 suspects，且永远不会移除 ✓

### 5.2.4 完美失败检测需要同步系统——从证明中看到本质

上一步的证明依赖于一个核心假设：**消息在 ≤ D 时间内到达**。如果去掉了这个假设（变成异步系统），不误判的证明就无法成立——p 永远不能区分"Heartbeat 在 D+1 时间后到达"和"q 真的崩溃了"。

**FLP 定理的关系**（第10章）：完美失败检测在异步系统中不可能实现——因为异步系统没有消息延迟上界，完美失败检测所需的确定性推断无法完成。

### 5.2.5 时间推演——完美检测如何工作

```
T0: p 收到 q 的 Heartbeat
    last_hb[q] = T0

T10: p 发送 Heartbeat → q 也应该在发

T110: D（网络延迟上界，假设 D=100ms）已过
     → p 应该收到 q 的 Heartbeat 了
       但 p 没收到 → p 判断：q 要么崩溃，要么消息丢包太多
       但公平丢包链路的假设下，"无限发送最终到达" → 确认：q 一定崩溃了
       p 将 q 标记为 suspects

T110+：q 在 suspects 中——永远不会移除（直到 q 恢复并重新发送 Heartbeat）
```

## 5.3 最终完美失败检测——如果你不知道消息延迟上界

### 5.3.1 假设：部分同步系统

第3章 3.6.3 节定义了部分同步系统：延迟上界存在，但你不知道具体值。在部分同步下，你无法构建完美检测（因为你不知道 D），但可以构建**最终完美检测器（Eventually Perfect, ◇P）**。

**为什么叫"最终"？** 因为检测器需要时间"学习"实际网络的延迟。在"学习"期间，可能有些误判；但"学习完成后"，不再误判。

### 5.3.2 ◇P 的两个性质

| 性质 | 你的翻译 | 形式化定义 | 与P的差异 |
|------|---------|-----------|---------|
| **强完备性** | "漏不掉" | 同 P | **相同** |
| **最终强准确性** | "最终不冤枉——之前误判的会纠正" | ∃T：∀t>T，∀正确的 p,q：q ∉ suspects(D,p,t) | **不同**——允许在 T 之前有有限次误判 |

### 5.3.3 算法——自适应超时

◇P 的关键创新：**不再需要固定的超时值——而是让检测器"学习"网络的实际延迟**。

```
算法（p 的自适应超时检测器）：

状态：
  timeout[q] = T0（初始值，如 100ms）
  last_hb[q] = 收到 Heartbeat(q) 的时刻

upon 收到 Heartbeat(q)：
  last_hb[q] = now
  if q 之前被标记为 suspects：
    suspects.remove(q)   // 误判被纠正
    timeout[q] *= 2      // 不是 100% → 扩大超时 ← 关键！

upon 检测超时（后台循环）：
  if now - last_hb[q] > timeout[q]：
    suspects.add(q)
    timeout[q] *= 2      // 继续扩大——让下一次更宽容

upon suspects 变化 → 主动通知上层（选主算法）
```

### 5.3.4 为什么自适应超时"最终"收敛？

你可能会想：每次都加倍超时——最终不是变成了"无限超时"——检测太慢了！

**但答案是——在网络**最终稳定后**，超时**不会再增加了**。

在部分同步下，存在一个最终的最大延迟 D。检测器每误判一次，超时翻倍——**最终**会超过 D——之后不再误判，超时停止增加。

**收敛轮数的数学分析**：

```
从初始超时 T0 到最终超时 D 需要多少次加倍？

T0 → T0×2 → T0×4 → ... → T0×2^k > D

2^k > D/T0
k > log₂(D/T0)
```

**具体例子——T0=100ms, D=1600ms**：

```
k > log₂(1600/100) = log₂(16) = 4 轮
每轮至少需要一次误判触发超时翻倍 → 至少需要 4 次误判
如果网络波动周期为 1分钟，需要约 4分钟达到收敛
```

### 5.3.5 完整时间推演——从误判到收敛

```
T0: 网络正常——RTT=1ms，timeout=100ms
    p 检测正确——没误判

T100: 网络突然严重拥塞——RTT 从 1ms 涨到 300ms
T200: p 发 Heartbeat，q 回复但延迟了（~300ms）
T300: timeout=100ms 到期——p 没收到回复 → suspects.add(q)
      p 还加倍超时：timeout *= 2 = 200ms

T310: q 的延迟回复到达——p 发现："我冤枉 q 了！"
      suspects.remove(q)
      但 timeout 已经被加倍为 200ms

T400: 网络再次拥塞——RTT 增加到 500ms
T500: timeout=200ms 到期——再次误判
      timeout *= 2 = 400ms

T510: q 的回复到达——再次纠正

T600: 网络第三次拥塞——RTT 增加到 900ms
T700: timeout=400ms 到期——第三次误判
      timeout *= 2 = 800ms

T800: 网络终于稳定——RTT 回到正常 1ms
      但 timeout 已经是 800ms——即使网络正常，timeout 也不会再减少（只增不减）
      未来的检测不会再误判了！
```

**从这个时间推演中，你学到了什么？**
- timeout **只增不减**——一旦增加到能覆盖网络的最坏延迟，就保证了"最终正确"
- 收敛时间取决于**网络波动的频率和幅度**——越不稳定，需要的收敛轮次越多
- 最终 timeout 可能是初始值的数倍——初始 100ms 最终变成 800ms → **检测崩溃的延迟也变为 800ms**

## 5.4 Chandra-Toueg 失败检测器——完整分类

### 5.4.1 四种检测器的谱系

Chandra 和 Toueg（1996）的论文《Unreliable Failure Detectors for Reliable Distributed Systems》对失败检测器做了完整分类。基于两个维度——**完备性（漏判吗？）** 和**准确性（误判吗？）**：

| 检测器 | 完备性 | 准确性 | 同步要求 | 你能做什么 |
|--------|--------|--------|---------|-----------|
| **P（完美）** | 强完备 | 强准确 | 同步系统 | Leader 选举、互斥、原子提交——所有 |
| **S（强）** | 强完备 | 弱准确 | — | 大多数操作——但可能误判导致短暂不可用 |
| **◇P（最终完美）** | 强完备 | 最终强准确 | 部分同步 | Leader 选举（允许短暂双Leader，最终收敛） |
| **◇S（最终强）** | 强完备 | 最终弱准确 | — | 共识（需要至少一个正确节点不被误判） |

### 5.4.2 每种检测器的形式化性质

**P（完美）**：
```
∀崩溃进程 q：∃T 使得 ∀t>T，∀正确的 p：q ∈ suspects(D,p,t)
∀正确进程 q：∀t，∀正确的 p：q ∉ suspects(D,p,t)
```
→ 不漏判 + 不误判。

**S（强）**：
```
强完备：同 P
弱准确：∃正确进程 q：∀t，∀正确的 p：q ∉ suspects(D,p,t)
```
→ 不漏判，但只保证**至少一个正确进程**不被误判。

**◇P（最终完美）**：
```
强完备：同 P
最终强准确：∃T：∀t>T，∀正确的 p,q：q ∉ suspects(D,p,t)
```
→ 不漏判，T 之后没有任何误判。

**◇S（最终强）**：
```
强完备：同 P
最终弱准确：∃T：∀t>T，∃正确的 q：(∀正确的 p: q ∉ suspects(D,p,t))
```
→ 不漏判，T 之后**至少一个正确进程**不被误判。

### 5.4.3 检测器的层次关系

```
P ⊂ ◇P ⊂ S ⊂ ◇S
```

**证明——P ⇒ ◇P**：P 要求"永远不误判"，这自然蕴含"最终不误判"。◇P = P 弱化到"最终"——P 更强。

**证明——◇P ⇒ S**：◇P 的最终强准确 ⇒ 至少在 T 之后没有正确进程被误判 ⇒ 至少有一个正确进程不被误判 = S 的弱准确（S 只要求"至少一个"不被误判）。

**为什么 S 比 ◇P 弱？** S 只要求"至少一个正确进程不被误判"，但可以**永远** 误判其他所有正确进程。◇P 要求"最终没有正确进程被误判"——远更强。

### 5.4.4 哪种检测器"够用"——工程选择

| 需要做什么 | 需要什么检测器 | 原因 |
|-----------|-------------|------|
| Leader 选举（不允许双Leader） | P | 需要绝对可靠——不误判、不漏判 |
| Leader 选举（允许短暂双Leader） | ◇P | 允许短暂冲突，最终收敛 |
| 共识（Paxos只保证Safety） | ◇S | 只需要"至少一个正确进程不被误判"来保证Leader可选 |
| 共识（需要Termination） | ◇P | 需要最终完美——所有正确进程最终都齐备 |

**为什么 Raft/ZAB 没有显式使用 Chandra-Toueng 检测器？** 它们通过**内置超时+选举机制**隐式实现了 ◇P——Leader 心跳超时→发起选举。这个过程等价于"最终完美检测+霸道选举"。

## 5.5 Leader 选举——基于失败检测的选主

### 5.5.1 定义

**Leader 选举（Leader Election）** 在进程集合中选出唯一的 Leader。

| 性质 | 形式化定义 |
|------|-----------|
| **安全性** | 在所有正确进程中，最多一个被认为是当前 Leader |
| **活性** | 最终会选出 Leader，且如果 Leader 崩溃，最终会选出新 Leader |

### 5.5.2 基于 P 的绝对选主——不容犯错

```
算法（基于 P 的霸道选举）：

  状态：leader = None

  操作：
    alive = 所有进程 - suspects  // P 输出：哪些进程活着
    leader = max({pid 在 alive 中})  // 选 ID 最大的存活进程
    
    if leader == self：
      广播 "我是 Leader" 给所有进程
```

**为什么需要 P？** 因为要确保"不会有两个 Leader"——如果检测器误判最大 ID 的进程（错误的漏掉了它），self 会错误地宣布自己是 Leader。只有 P 保证不误判。

### 5.5.3 基于 ◇P 的最终选主——允许短暂双Leader

```
算法（基于 ◇P 的最终选举）：

  同 5.5.2 基础，额外增加：
  
  upon 收到 "我是 Leader" 消息 from 进程 q：
    if q.id > self.leader.id：
      self.leader = q  // 向更高 ID 的 Leader 退位
    else if q.id == self.leader.id：
      // 无变化
    else if q.id < self.leader.id：
      回复 "我已认定某更高ID的进程" 给 q
```

**时间线——双Leader的发生和收敛**：

```
T0: q(pid=3) 是 Leader → 正常

T1: p(pid=2) 的 ◇P 在"学习期间"误判 q 崩溃
    p 执行选主→ leader = p(pid=2)
    现在有**两个 Leader**：q(旧) + p(新)

T2: 两个 Leader 同时发送心跳
    其他进程收到 q 和 p 的心跳
    但 q.id > p.id → 新 Leader 被否决 → p 退位
    p 将 leader 设为 q

T3: p 的 ◇P 纠正了误判→ suspects.remove(q)
    → leader 唯一：q(pid=3)
    双Leader 收敛 ✓
```

**这就是 ◇P 的"最终安全"——短期可能双Leader，但最终收敛**。◇P 不能保证 Safety 永远满足（因为学习期间不满足），但保证 Liveness 和学习后的 Safety。

## 5.6 工程实践——ZooKeeper vs etcd vs 人工巡检

### 5.6.1 ZooKeeper 的失败检测

ZooKeeper 通过 ZAB 的 Leader-Follower 心跳检测节点存活：

```
tickTime = 2000ms            // 心跳间隔
syncLimit = 5               // Follower 容忍的丢失心跳数
超时 = tickTime × syncLimit = 10s
```

**为什么设 10s？** ZooKeeper 偏向**稳定**——宁可慢检，不要误判。10s 对大部分网络抖动来说是足够的冗余。

### 5.6.2 etcd 的失败检测

etcd 的 Raft 通过周期性心跳 + 随机选举超时检测 Leader：

```
heartbeat-interval = 100ms     // 心跳间隔
election-timeout = 1000ms       // 选举超时
（随机范围 [1000, 2000]ms）
```

**为什么设 1s？** etcd 偏向**快速恢复**——宁可误判一次，不能让集群 5s 无 Leader。1s 超时 + 随机范围避免并发选举。

### 5.6.3 对比表

| 系统 | 超时 | 哲学 | ◇P 还是 P？ |
|------|------|------|-----------|
| ZooKeeper | 10s | 稳定第一——慢检、少误判 | ◇P（最终完美，大数据安全） |
| etcd | 1s | 快速恢复——快检、容忍误判 | ◇P（最终完美，小数据快速） |
| 人工巡检 | 分钟级 | 运维介入——慢速、高精度 | P（完美，因为人类能判断） |

## 5.7 本章总结

### 6 个核心要点

1. **失败检测器解决"瞎子困境"**——在不可靠链路上判断进程是否存活。你设计的超时方案本质上是一个失败检测器——它的正确性依赖于时间假设。

2. **完美失败检测 P**——在同步系统（已知延迟上界 D）下满足：不漏判（所有崩溃都被检测）和不误判（所有正确进程都不被怀疑）。这是最强保证——但需要有已知延迟上界。

3. **最终完美失败检测 ◇P**——在部分同步系统（延迟上限未知）下满足：不漏判 + 学习后不再误判。通过**自适应超时**实现——每次误判后加倍超时，最终超过网络的最大延迟。

4. **Chandra-Toueng 四种检测器**：P ⊂ ◇P ⊂ S ⊂ ◇S。P 是最强的（同步要求），◇S 是最弱的（异步也可行）。大多数分布式应用至少需要 ◇P 或 ◇S。

5. **Leader 选举基于失败检测**：P 支持绝对选主（不会双Leader），◇P 支持最终选主（允许短双Leader，收敛后唯一）。

6. **实际工程**：ZooKeeper（tickTime×syncLimit=10s，稳定优先）vs etcd（election-timeout=1s，快速恢复优先）——两者都是隐式的 ◇P 实现。
