## 3.1 系统模型——为什么需要明确定义假设？

### 3.1.1 一个算法，两种结论

考虑一个简单的 Leader 选举算法：

> "每个进程定期发送心跳。如果 3 秒内没收到 Leader 的心跳，发起选举。"

**在局域网中**：心跳丢失的概率很低——3 秒超时可能是合理的。算法大概率正确。

**在跨大洲网络中**：心跳 RTT 可能 300ms，加上抖动——3 秒内没收到心跳**不能说明 Leader 宕机**。算法会频繁触发假选举。

**同一个算法，在不同环境下正确性不同。** 系统模型的作用就是**明确定义"算法在什么环境下运行"**——进程如何失败、时钟如何行为、网络延迟有上界吗。

### 3.1.2 系统模型的三个维度

系统模型由三个维度组成：

| 维度 | 定义的内容 | 典型假设 |
|------|----------|---------|
| **进程模型** | 进程如何启动、执行、失败 | 崩溃式失败（最常用） |
| **时钟模型** | 时钟的精度和行为 | 本地时钟（每个进程有自己的时钟） |
| **时间假设** | 网络延迟和进程速度的有界性 | 部分同步（最接近真实网络） |

> **跨书对照**：释慧利第1章 1.1.2 节从"单机确定性的消失"直观感受切入——没有共享内存、没有全局时钟、没有瞬时故障感知。江峰第3章将这些感受**形式化**为进程模型、时钟模型、时间假设——释慧利讲"你感觉到了什么"，江峰讲"数学模型如何描述这些感觉"。

## 3.2 进程模型

### 3.2.1 进程的形式化定义

**进程（Process）** 是分布式系统中执行算法的基本单元。形式化定义：

```
进程 p_i = {
    状态空间 S_i：所有可能的状态值的集合
    初始状态 init_i ∈ S_i
    转移函数 δ_i: S_i × Acts → S_i  // 状态 × 接收到的操作 → 新状态
    赋值：
        如果 e ∈ in(p_i) → p_i 接收 e 并执行转移
        如果 e ∈ out(p_i) ∪ int(p_i) → p_i 触发 e 并执行转移
}
```

**关键属性**：
1. **步不可拆分**：一步（Step）是原子的——不能同时执行两步。第2章 I/O 自动机规则 5 保证了这一点
2. **无共享内存**：进程 p_i 不能直接访问进程 p_j 的状态
3. **通信唯一途径**：消息传递

### 3.2.2 进程执行模型

**异步执行**：进程以自己的速度执行——没有全局同步时钟来协调各进程的步调。进程的"一步"意味着：

- 一次状态转移（接收一个输入或触发一个输出/内部操作）
- 执行的时间**不设上限**——可以花任意长时间
- 不同的进程可以在不同时间点执行

**并发 vs 并行**：
- **并发**：多个进程看似同时执行（通过交错执行）
- **并行**：多个进程真正同时执行（多核）

分布式算法关注**并发**——因为进程事件可以按任意顺序交错，无需真并行。

### 3.2.3 进程数量假设

| 假设 | 含义 | 典型场景 |
|------|------|---------|
| N 固定 | 进程数量在算法执行期间不变 | 多数共识算法（Paxos/Raft/ZAB） |
| N 可变 | 进程可以动态加入/离开 | 组成员关系（第9章） |

## 3.3 消息模型

### 3.3.1 消息的定义

**消息** 是进程之间通信的唯一媒介。形式化定义：

```
消息 m = {
    发送方 src(m)：进程 ID
    接收方 dst(m)：进程 ID
    内容 data(m)：任意的应用消息体
}
```

**消息传递** 是独立的事件——一条消息的 Send 和 Receive 是两个独立的自动机事件（第2章），不保证同步。

### 3.3.2 消息传递的关键假设

| 假设 | 含义 | 形式化定义 |
|------|------|-----------|
| **无篡改** | 消息内容在传输过程中不被修改 | data(m_received) = data(m_sent) |
| **无伪造** | 只有发送方能生成消息——接收方无法假冒发送方 | src(m) 是真实的 |
| **可能丢失** | 消息可能在网络中丢失（取决于链路模型） | 有些链路模型允许丢，有些不允许 |

### 3.3.3 链路模型的层次

链路定义了消息传输的可靠性（第4章详述）：

```
公平丢包 → 顽固 → 可靠 → FIFO 可靠 → 日志可靠
```

**目前我们假设**：进程之间存在双向公平丢包链路——如果无限次重发，每条消息最终会到达。

## 3.4 进程失败——四种模型

### 3.4.1 问题：进程如何失败？

进程的失败方式是系统模型最核心的假设之一。不同的失败模型对应**不同的容错能力和算法复杂度**。

### 3.4.2 四种失败模型

| 失败模型 | 行为描述 | 形式化定义 | 典型算法 |
|---------|---------|-----------|---------|
| **崩溃式（Crash-Stop）** | 进程停止执行所有操作，不再恢复 | 在某个时刻后，进程不再有任何转移 | Paxos/Acceptor (Basic) |
| **遗漏式（Omission）** | 进程运行正常，但某些消息不发送或不接收 | 某些输入/输出事件的转移被"跳过" | 失败检测器 (Chandra-Toueg) |
| **崩溃恢复式（Crash-Recovery）** | 进程崩溃后恢复，但丢失内存状态 | 在某个时刻后停止，之后重新开始——但状态重置为持久化子集 | Paxos Acceptor + 持久化 |
| **拜占庭式（Byzantine）** | 进程可以偏离算法规定——包括伪造消息 | 进程的状态转移可以违反算法规定 | PBFT |

### 3.4.3 四种模型的关系

```
崩溃式 ⊂ 遗漏式 ⊂ 崩溃恢复式 ⊂ 拜占庭式
```

**证明**：
- 崩溃式 ⊆ 遗漏式：崩溃的进程自然遗漏所有消息——所以崩溃进程是"遗漏所有消息"的遗漏式进程
- 遗漏式 ⊆ 崩溃恢复式：遗漏式进程不崩溃——可以看作"崩溃后立即恢复，没有丢失内存状态"
- 崩溃恢复式 ⊆ 拜占庭式：崩溃恢复进程的行为是确定性的——可以看作"行为在某种程度上受限"的拜占庭进程

**为什么需要不同的模型？**

- **崩溃式是最简单的**——多数算法（Paxos/Raft/ZAB）默认假设崩溃式。实现简单、容错需要 2f+1
- **遗漏式更接近真实网络**——网络拥塞、GC 停顿、磁盘 IO 抖动都可以看作"消息遗漏"
- **崩溃恢复式是生产常态**——节点崩溃后重启是常规运维操作
- **拜占庭式是公网环境**——后端的 Tls/mTLS 在传输层将拜占庭退化为崩溃式

**韧性**（f）—— 算法可以容忍的不超过f个进程同时失败的最大数量：
- 崩溃式 + 多数派：f < N/2 → 需要 2f+1 个进程容忍 f 个崩溃
- 拜占庭式：f < N/3 → 需要 3f+1 个进程容忍 f 个叛徒

> **跨书对照**：释慧利第1章 1.6.2-1.6.3 节详细讨论了 Crash-Stop vs Crash-Recovery vs Omission，并给出了 2F+1 和 3F+1 的数学推导。江峰第3章的贡献在于**证明了四种模型之间的层次关系**——释慧利没给出这个层次证明。

## 3.5 时钟模型

### 3.5.1 为什么分布式系统中时钟不可靠？

**物理限制**：
1. **石英晶振的抖动**：每个 CPU 的晶振精度不同——两块相同型号的 CPU 计数速率可能有 ~100ppm 的误差（百万分之一百）。这意味着两个同时启动的时钟，在 1 秒后可能偏差 0.1 毫秒，1 小时后可能偏差 360 毫秒
2. **NTP 同步误差**：网络延迟使得 NTP 同步精度在毫秒级别——两个节点的 NTP 同步时间可能偏差 10-100ms
3. **时钟跳变**：系统管理员手动调整时间、闰秒、VM 迁移——时钟可能向前跳、向后跳

### 3.5.2 时钟模型的层次

| 模型 | 性质 | 工程实现 |
|------|------|---------|
| **本地时钟** | 每个进程有本地硬件时钟——无同步保证 | 任何不需要时钟同步的算法适用 |
| **同步时钟** | 所有进程的时钟差异有上界 ε | 原子钟 + GPS（Spanner TrueTime） |
| **逻辑时钟（Lamport 1978）** | 不测量物理时间，只保证因果序 | Lamport Clock, Vector Clock |
| **混合逻辑时钟（HLC）** | 物理时间 + 逻辑计数器 | CockroachDB 的 HLC |

### 3.5.3 因果顺序不变——为什么逻辑时钟有效？

**Lamport 1978 的核心洞察**：在分布式系统中，你不需要全局物理时钟来排序事件——你只需要保证**因果顺序**。如果事件 A 通过消息影响了事件 B（即 A 发生在 B 的"因"之前），则 A 的时钟必须 < B 的时钟。无因果关系的"A 和 B 谁先发生"——不可知，也不需要知。

**实现**：每个进程维护一个本地计数器。发送消息时计数+1，接收消息时将自己的计数器更新为 `max(自己的计数, 消息中的计数) + 1`。

**与 Paxos 提案编号的关系**：Paxos 的提案编号 n 本质上是 Lamport 时钟——它不需要物理时钟，只需要保证"编号在全局单调递增"。这就是为什么 Paxos 在异步系统下也能保证 Safety。

## 3.6 时间假设——同步、异步、部分同步

### 3.6.1 三种时间假设

| 假设 | 网络延迟上界 | 进程速度下界 | 形式化定义 |
|------|------------|------------|-----------|
| **同步（Synchronous）** | 已知上界 D | 已知下界 S | ∀消息 m 在时间 D 内到达；∀进程在 S 时间内完成一步 |
| **异步（Asynchronous）** | 无上界 | 无下界 | 延迟和速度可以任意大 |
| **部分同步（Partial Synchronous）** | 存在上界但未知；或上界只在最终成立 | 存在下界但未知 | ∃D 使得最终所有消息在 D 内到达，但 D 的值在算法开始前未知 |

### 3.6.2 为什么部分同步是最接近真实的模型？

**互联网不是完全异步的**：虽然消息延迟可能很高，但不可能无限高——总有一个"实际最大值"。如果延迟超过 30 秒，TCP 连接通常已经断开了。

**但延迟的确切值是无法预测的**：网络条件变化——拥塞、路由变更、机房故障——使得"今天 1ms"不代表"明天也是 1ms"。

**部分同步**模型捕捉了这种不确定性：存在一个延迟上界（网络不会永远挂掉），但这个上界的值是不可提前预知的。它允许算法主动探测（例如发送心跳），根据实际情况自适应调整超时值。

### 3.6.3 不同时间假设下共识的可解性

| 时间假设 | 失败模型 | 共识可解？ | 代表性算法 |
|---------|---------|:---:|---------|
| 同步 | 崩溃式 | ✅ | Paxos（恢复速度更快） |
| 同步 | 拜占庭式 | ✅ | PBFT |
| 异步 | 崩溃式 | ❌ 不可能！(FLP 1985) | —— |
| 异步 | 崩溃式 + 随机化 | ✅ 概率正确 | Ben-Or (1983) |
| 部分同步 | 崩溃式 | ✅ | Paxos / Raft / ZAB |

**关键洞察**：大部分实用共识算法运行在**部分同步 + 崩溃式**假设下——部分同步保证 Liveness（Leader + 超时打破活锁，等待最终稳定），崩溃式保证 Safety（多数派不包含过多失败节点）。

## 3.7 安全性和活性——形式化定义（扩展）

### 3.7.1 回顾第1章的直觉定义

第1章 1.4.4 节讨论了：
- **Safety**："不好的事情永远不会发生"
- **Liveness**："好的事情最终会发生"

现在在第2章 I/O 自动机的框架下，用**迹集（Traces）** 给出严格的形式化定义。

### 3.7.2 安全的迹集定义

**定义**：自动机 A 满足安全性属性 P，当且仅当 `traces(A) ⊆ P`。

**说明**：
- P 是一个**迹的集合**——代表"好"的外部行为模式
- 自动机 A 必须保证：不管环境和故障如何，A 产生的外部操作序列（迹）**一定**属于 P
- 如果你能找到一个迹 t ∈ traces(A) 且 t ∉ P——自动机不满足安全性

**例子**：Leader 选举的安全性

```
P_election = {
    t 是外部操作序列 |
    t 包含的 "Elect(pid)" 操作中，最多有一个不同的 pid 值
}
```

- 迹 t₁ = "Elect(1) Elect(1) Elect(1)" → t₁ ∈ P ✅（只有一个不同的 pid）
- 迹 t₂ = "Elect(1) Elect(2)" → t₂ ∉ P ❌（出现两个不同的 pid）

### 3.7.3 活性的踪迹定义

活性不能用迹集直接定义（因为活性关注"最终是否发生"），而需要**公平性执行**作为中介。

**公平性执行**：如果在某个状态下操作 e 是激活的，并且此后一直保持激活，那么 e 最终必须被执行。

**定义**：自动机 A 满足活性属性 L，当且仅当在所有的公平性执行中，无论执行可以多长，都最终会包含某个"好"的操作。

**例子**：Leader 选举的活性

```
L_election: 最终会出现一个 Elect(pid) 操作
```

- 公平性执行：每个超时后自动机触发选举 → 最终某进程获得多数派 → Elect(pid) 出现 ✅
- 不公平执行：进程被无限阻塞 → Elect(pid) 永远不出现 —— 不是公平执行，不违反活性

### 3.7.4 Safety 和 Liveness 的组合——FLP 的关键

FLP 不可能定理（第10章）的核心是：在异步系统下，**无法同时满足 Safety 和 Liveness**。因此：

- **Paxos**：Safety 永远保证（在纯异步下也成立）；Liveness 只在部分同步下保证（Leader + 超时打破活锁）
- **2PC**：Safety 保证（原子提交）；Liveness 可能受影响（协调者阻塞导致无限等待）
- **随机算法**：Safety 不变；Liveness 概率性（随机超时/随机提案编号）

## 3.8 多数派——一个初等的集合论事实

### 3.8.1 定理：任意两个多数派必有交集

**形式化定义**：在 N 个进程的集合中，多数派 Q 是满足 |Q| > N/2 的子集（即 |Q| > ⌊N/2⌋）。

**定理**：对于任意两个多数派 Q₁ 和 Q₂，必有 |Q₁ ∩ Q₂| ∉ ∅ 为空集。

**证明**（鸽笼原理）：

设 |Q₁| ≥ ⌊N/2⌋+1，|Q₂| ≥ ⌊N/2⌋+1。

如果 Q₁ 和 Q₂ 没有交集，那么 |Q₁ ∪ Q₂| = |Q₁| + |Q₂| ≥ 2⌊N/2⌋+2。

当 N 为奇数时：⌊N/2⌋ = (N-1)/2，所以 2⌊N/2⌋+2 = N+1 > N——矛盾！
当 N 为偶数时：⌊N/2⌋ = N/2，所以 2⌊N/2⌋+2 = N+2 > N——矛盾！

所以 Q₁ 和 Q₂ 不可能没有交集——必有至少一个共同进程。

### 3.8.2 多数派定理的工程含义

**它是共识算法的数学基础**——一个被多数派 Q₁ 接受了的值，会被所有后续多数派 Q₂ 中的至少一个节点持有。因此，当系统的状态需要从一代传到下一代时，多数派机制能保证信息的传递。

具体应用：
- Paxos/Raft/ZAB 都使用多数派进行提案和决策
- 一个被多数派确认的 Leader 不会被另一个多数派否决
- 两个不同的多数派不可能选择两个不同的值

## 3.9 实例分析——同一个算法在三种时间假设下

理论讲了这么多，用一个实例把它们结合起来。考虑以下 Leader 选举算法：

> "每个进程每 100ms 发送一次心跳。如果 300ms 内没收到Leader的心跳，发起选举。"

在**三种时间假设**下，同一个算法有不同的正确性分析：

### 同步系统（已知延迟上界 D=50ms）

```
假设：D=50ms（网络延迟<=50ms），心跳间隔=100ms，超时=300ms

分析：心跳在 50ms 内到达 → 3个心跳间隔=300ms >> 50ms
      → 超时一定意味着 Leader 真的崩溃了
      → 算法正确：漏判概率 0，误判概率 0
      → 这是"完美失败检测 + 霸道选举"——满足 Safety 和 Liveness ✓
```

### 异步系统（延迟无上界）

```
假设：延迟无任何上限——可能突然延迟 10s、1分钟、1小时

分析：心跳可能因为延迟未被收到 → 300ms超时可能是网络延迟，不是Leader崩溃
      → p 错误地发起选举 → 双 Leader → 违反 Safety ✗
      → 但在无限时间后——如果延迟无限长——算法永远无法恢复
      → FLP 说：异步系统下无法保证 Termination ✓
```

### 部分同步系统（延迟上界存在但未知）

```
假设：存在延迟上界 D，但 D 的值在算法开始时未知（如 D=500ms，但算法不知道）

分析：初始阶段，超时=300ms < 500ms → 可能误判 Leader 崩溃（假阳性）
      但最终网络稳定——300ms 内 Leader 回复到达 → 不再误判
      → 这是"最终完美检测 + 最终选主"——短期可能双Leader，最终收敛
      → 满足 Safety（最终）和 Liveness ✓
```

**关键洞察**：同一个算法→三种假设→三种结果是**不相同的**——这就是为什么在第1章开头定义系统模型如此关键。你不能说"Leader 选举算法是安全可靠的"——你必须说"在部分同步假设下，该算法满足最终 Safety 和 Liveness"。

## 3.10 Safety 和 Liveness 的具体推演——以 Paxos Acceptor 的 Promise 为例

第2章 2.7 节用 I/O 自动机建模了 Paxos Acceptor。现在我们在系统模型的框架下，用 Safety 和 Liveness 分析它的正确性。

**Paxos Acceptor 的两个关键性质**：

| 性质 | 形式化表述 | Safety 还是 Liveness？ |
|------|-----------|---------------------|
| 不变式1：一个Acceptor不会对编号n做Promise，然后接受编号<n的Accept | ∀执行序列，这个性质成立 | Safety（有限前缀可判） |
| 不变式2：如果某个提案被多数派接受，后续任何选定的值都等于该提案的值 | ∀可能的选择序列，这个性质成立 | Safety（有限前缀可判——一旦发现两个不同提案被多数派接受） |
| FLP 下的状态 | 在纯异步时：两个属性都 Safety（Paxos 在异步下 Safety 不变) | 但 Liveness 不能保证——多个 Proposer 互相抢占产生活锁 |

**为什么 Paxos 在异步下 Safety 不变？** 因为 Safety 属性**不依赖时间假设**——它们只依赖于不变式1/2在**任何**执行序列中都成立。时间假设影响的是能否在有限时间内完成共识（Liveness），而不是是否保持一致性（Safety）。

**这就是为什么第3章的时间假设如此关键**：
- Safety 需要**不同时间假设下的证明**是否相同
- Liveness 直接依赖于时间假设——在同步或部分同步下保证了 Liveness，在异步下可能丢失

## 3.11 本章总结

### 6 个核心要点

1. **系统模型定义了算法的假设前提**——同一个算法在不同假设下可能正确也可能不正确。系统模型的三个维度：进程模型、时钟模型、时间假设。

2. **四种进程失败模型**：崩溃式（最常用，2f+1）⊂ 遗漏式（更接近真实网络）⊂ 崩溃恢复式（生产常态）⊂ 拜占庭式（3f+1，用于公网场景）。不同模型对应不同的容错能力和算法复杂度。

3. **三种时间假设**：同步（已知延迟上界）——共识可行；异步（无延迟上界）——FLP 证明共识不可行；部分同步（延迟上界存在但未知）——Paxos/Raft/ZAB 的实际工作假设。

4. **时钟模型的层次**：本地时钟（无同步）→ 同步时钟（Spanner TrueTime，δ 界定）→ 逻辑时钟（Lamport 1978，因果序）→ 混合逻辑时钟（CockroachDB HLC，物理+逻辑）。

5. **安全性 = 所有可能的迹都属于某个迹集**（traces(A) ⊆ P，差的行为不在集合内）。**活性 = 在所有公平性执行中最终出现好操作**（需要公平性作为判断活性是否违反的条件）。

6. **多数派定理**：任意两个多数派必有交集（鸽笼原理证明）。多数派交集 = 信息可以从一代多数派传到下一代 = 共识算法的数学基础。

### 第4章预告

第4章将开始构建通信抽象——从最基础的公平丢包链路开始，依次构建顽固链路、可靠链路、FIFO 可靠链路。每个抽象层次都通过"更低层的链路 + 算法逻辑"实现更高层的保证——这是分布式算法"分层设计"的首次实战。
