# 第6章 可靠广播

> **技术专家重写**
>
> 原书第6章介绍了从尽力广播到因果广播的七种广播类型。本重写版聚焦：**在已有链路抽象的基础上，如何构建"将消息可靠地广播给一组进程"的高层抽象？** 展示分布式算法的"分层设计"在广播层面的延续。

## 6.1 为什么需要广播抽象？

### 6.1.1 链路 vs 广播——一对一 vs 一对多

第4章的链路抽象是**一对一**的——进程 p 向进程 q 发送一条消息。广播是**一对多**的——一个进程向**一组**进程发送消息。

**广播需要解决的问题**：
1. **交付保证**：消息是否被所有（或多数）进程接收？是否可能丢失、重复、乱序？
2. **顺序**：不同进程收到消息的顺序是否一致？

### 6.1.2 广播的层次结构

和第4章的链路分层一样，广播也有分层——每一层在下一层的基础上增加保证：

```
尽力广播（最弱：可能丢、可能乱）
    │
    ├──► 正则可靠广播（不丢：如果发送方正确，所有正确进程都收到）
    │       │
    │       ├──► 统一可靠广播（统一性：发送方崩溃后，要么所有正确进程都收到，要么都没收到）
    │       │
    │       ├──► 顽固广播（持久：发送方持续重发直到所有正确进程确认）
    │       │
    │       ├──► FIFO广播（顺序：同一发送方的消息按发送顺序接收）
    │       │
    │       └──► 因果广播（因果序：有因果关系的消息保持顺序）
    │
    └──► 概率广播（Gossip）：大规模系统中的概率性交付
```

**各层广播的接口统一为**：

```
input:  Bcast(m)         // 发送方调用：广播消息 m
output: Deliver(p, m)    // 接收方感知：从进程 p 交付消息 m
```

### 6.1.3 如果你来设计广播——你能做到多可靠？

你已经有了第4章的链路抽象（一对一可靠传输）。现在你想把它升级为一对多广播。直觉方案：

**方案A：发送方循环发送**

```
for each q in 所有进程：
  可靠链路.Send(q, m)
```

**问题**：如果发送方在发送到一半时崩溃——q1 和 q2 收到了 m，q3 没收到。这就是尽力广播能给你的——它不保证"全员收到"。

**你想改进**：即使发送方崩溃，也要保证 m 到达所有正确进程。

**方案B：接收方转发**

```
收到 m 的进程 → 转发 m 给其他所有进程
```

**改进效果**：如果发送方崩溃前至少送到了 1 个正确的进程——那个进程会转发→覆盖所有。

**你能做到什么？** 这就是本章的核心：从"尽力"（可能丢）到"因果"（保持顺序），每一层都在下层的基础上增加保证。

## 6.2 尽力广播——最基础的假设

### 6.2.1 定义

**尽力广播（Best-Effort Broadcast）** 是最弱的广播——它只做"尽力而为"的交付：

| 性质 | 含义 |
|------|------|
| **有效性** | 如果发送方 p 正确且接收方 q 正确，p 广播的消息 m 最终会被 q 交付 |
| **无伪造** | 如果 q 交付了消息 m，m 一定是某些进程广播的 |
| **无重复** | 同一条消息最多被交付一次 |

**关键**：尽力广播**不保证**所有正确的进程都收到消息——如果发送方 p 在广播过程中崩溃，某些进程可能收到 m，某些没收到。

### 6.2.2 算法——基于可靠链路的简单广播

```
算法（基于可靠链路）：

  upon Bcast(m):
    for each 进程 q in 所有进程:
      可靠链路.Send(q, m)  // 用第4章的可靠链路一对一发送

  upon 可靠链路.Receive(p, m):
    输出 Deliver(p, m)    // 交付给上层
```

**为什么基于可靠链路？** 可靠链路保证一对一不丢——发送方正确时每条消息都会被接收。广播只需对每个接收方都调用一次可靠链路。

**尽力广播的"尽力"体现在哪里？** 如果发送方在发送到一半时崩溃——已经通过可靠链路发送给 q1 和 q2 的消息会被交付，但尚未发送给 q3 的消息永远不会到达 q3。这就是"尽力"——发送方尽力发送了，但没保证完成。

**尽力广播的"不保证"示例**：

```
p 广播 m 给 {q1, q2, q3}：
  T0: p 开始 Bcast(m)
  T1: p 调用 Link.Send(q1, m) —— 成功
  T2: p 调用 Link.Send(q2, m) —— 成功
  T3: p 崩溃! 还没调用 Link.Send(q3, m)

结果：q1 和 q2 交付了 m；q3 永远不会收到 m
```

## 6.3 正则可靠广播——发送方正确时保证全员送达

### 6.3.1 定义

**正则可靠广播（Regular Reliable Broadcast）** 在尽力广播的基础上增加：

| 附加性质 | 含义 |
|---------|------|
| **一致性** | 如果发送方 p 正确，p 广播的消息 m **最终被所有正确的进程交付** |

**与尽力广播的区别**：尽力广播的"有效性"只说"正确的接收方最终收到"——但不保证 **所有** 正确的接收方都收到。正则可靠广播增加了一致性——发送方正确 + 任何正确的接收方收到 → **所有** 正确的接收方都会收到。

### 6.3.2 算法——接收方转发（Eager Reliable Broadcast）

**核心思想**：当 q 第一次从 p 收到消息 m 时，q **转发** m 给所有其他进程——即使发送方 p 在广播过程中崩溃，被转发的消息仍然覆盖所有正确的接收方。

```
算法（基于尽力广播）：

  upon Bcast(m):
    尽力广播.Bcast(m)  // 通过尽力广播发送给所有进程

  upon 首次从 p 收到 m（尽力广播.Deliver(p, m) 触发）:
    尽力广播.Bcast(m)  // 转发：让其他进程也能收到
    输出 Deliver(p, m) // 交付给上层
```

**时间线——发送方崩溃时的覆盖**：

```
p 广播 m：
  T0: p 调用 尽力广播.Bcast(m) → 发送给 q1
  T1: p 崩溃! 还没发送给 q2 和 q3
  T2: q1 收到 m→ 首次交付 → 尽力广播.Bcast(m)（转发给 q2, q3）
  T3: q2 和 q3 收到 q1 转发的 m → 交付
结果：所有正确的进程最终都交付了 m ✓
```

**为什么"转发"能保证一致性？** 只要至少有一个正确的进程（本例中的 q1）收到了 m——它就会转发给其他进程——其他正确的进程最终都会收到。这就是"击中一个正确的节点，就能传播到全部"的思想。

## 6.4 统一可靠广播——发送方崩溃也保证统一

### 6.4.1 正则可靠广播的局限

正则可靠广播要求**发送方正确**。如果发送方在广播过程中崩溃，一致性不能保证。

**统一可靠广播（Uniform Reliable Broadcast）** 更强——即使发送方崩溃，也要保证**所有正确的进程都收到消息**，或者**所有正确的进程都没收到**。

| 附加性质 | 含义 |
|---------|------|
| **统一一致性** | 如果任何进程（正确的或崩溃的）交付了消息 m，那么 m 最终会被所有正确的进程交付 |

**为什么需要这种保证？** 在需要"全或无"的场景（如原子提交、数据库的分布式事务）中——所有参与者必须同意"提交"或"回滚"。统一可靠广播保证不会出现"有些参与者认为提交，有些认为回滚"的不一致。

### 6.4.2 算法——先确认多数派，再交付

**核心改进**：将接收方的交付延迟到**确认多数派已收到**后——只有被大多数正确进程确认的消息，才被视为"统一确定"。

```
算法（基于可靠链路 + 确认机制）：

  upon Bcast(m):
    可靠链路.Send(每个进程 q, m, seq)
    等待多数派 ACK
    if 收到多数派 ACK:
      输出 Commit(m)  // 确认消息已统一确定

  upon 可靠链路.Receive(p, m, seq):
    保存 (p, m, seq) 到本地缓冲区
    可靠链路.Send(p, ACK(m, seq))  // 确认收到
    
  upon 收到 Commit(m):
    输出 Deliver(p, m)
```

**关键机制**——多数派确认：消息 m 只有收到多数派的接收确认后，才能被"交付"给上层。这是因为：如果多数派收到了 m，即使发送方崩溃，剩下正确的节点中也至少有一个节点有 m 的副本——它可以通过转发机制覆盖其余节点。

### 6.4.3 多数派确认的完整推演

假设 5 个进程 p1-p5，majority = 3。发送方 p1 广播消息 m。

```
T0: p1 广播 m——同时发送给 p2, p3, p4, p5
T1: p2 收到 m → 回复 ACK(m)
T2: p3 收到 m → 回复 ACK(m)
    p1 收到 p2 和 p3 的 ACK = 2 个确认（不足以达到 majority=3）
    
T3: p4 收到 m → 回复 ACK(m)
    p1 收到 p4 的 ACK = 3 个确认 >= majority → 发送 Commit(m)

T4: p1 崩溃！但 m 已经被 3 个进程确认（p2, p3, p4 持有 m 的副本）
    虽然 p5 没收到 m——但有足够的节点可以转发

T5: p2 检测到 p1 崩溃→ 从本地缓冲区取出 m → 转发给 p5
    p5 收到 m → 回复 ACK → p5 也收到了 m ✓
```

**关键观察**：
- 即使发送方在 T4 崩溃——majority=3 的确认保证了副本数≥3——足以覆盖剩余的 2 个未确认进程
- **为什么 majority 是黄金数字？** 因为 (N - majority) = 少数派——如果 majority 持有副本，其他节点的数量不足以超过它。即使所有少派都还没收到 m——只要有 1 个 majority 中的节点还可以转发——m 最终到达全部

### 6.4.4 统一可靠广播与 Paxos Prepare 的关系——哪里不同？

| 维度 | 统一可靠广播 | Paxos Prepare |
|------|-----------|--------------|
| **目标** | 消息 m 被全体或全不接收（全或无） | 选定的值 v 被全体 Acceptor 同意 |
| **多数派的作用** | 确认 m 至少存在一个正确的节点中（转发覆盖） | 确认 v 被选中——如果有冲突，准备继承 |
| **发送方崩溃后** | 不需要恢复——剩下节点有副本，可以转发 | 需要 Prepare 重新探测选定值 |

**这是两种不同层次的多数派使用**——广播用多数派保证"不会丢失"，共识用多数派保证"会一致"。

## 6.5 顽固广播——确保全员确认

### 6.5.1 定义

**顽固广播（Stubborn Broadcast）** 继承了统一可靠广播——但要求发送方或转发方**持续重复发送**直到**所有正确的进程**都确认接收。

**与统一可靠广播的区别**：统一可靠广播只需要"多数派确认"——即便有少数进程没收到，也不影响效果。顽固广播要保证"每一个正确的进程"都确认——即使只有一个进程没收到，也要持续重发。

### 6.5.2 为什么需要顽固广播？

在成员关系管理（Group Membership）中——新节点加入时，必须可靠地通知**每一个**现有节点。这不是"多数派"的问题——漏掉任何一个节点，该节点可能永远不知道新节点加入。

**算法**

每个进程维护一个"已确认"的列表——记录哪些进程已对该消息确认。后台循环持续向未确认的进程重发消息，直到所有进程都确认。

## 6.6 概率广播——Gossip 协议

### 6.6.1 定义

**概率广播（Probabilistic Broadcast）**——即 **Gossip 协议**——放弃确定性保证，使用概率性交付。

| 性质 | 含义 |
|------|------|
| **概率性交付** | 每个正确的进程以高概率最终收到消息——不是 100% 保证 |
| **可扩展性** | 适用于大规模系统（数万节点）——确定性广播的消息复杂度 O(N²)，Gossip 是 O(N × log N) |

### 6.6.2 算法——随机选择节点推送

```
算法（Eager Push）：

  upon Bcast(m):
    local_store.add(m)
    for each round:  // 周期性执行
      随机选择 k 个进程（k 是扇出因子，如 k=3）
      向这 k 个进程发送 m
```

**收敛分析——N=10000 节点需要多少轮？**

设扇出因子 k=3（每个节点每轮随机选 3 个推送）。消息传播的数学和传染病模型相同——最初有 1 个感染者（发送方），每轮每个感染者感染 k 个新节点。

```
轮次 0：1 个节点有消息（发送方）
轮次 1：1 + 3 = 4 个节点有消息
轮次 2：4 × 3 = 12 → 共 4 + 12 = 16 个节点
轮次 3：16 × 3 = 48 → 共 16 + 48 = 64 个节点
...
```

**为什么要取对数？** 每轮感染倍数 = 扇出因子+1。经过 r 轮后，感染总数 ≈ (扇出因子+1)^r。要感染 N 个节点，需要 r ≈ log_{扇出因子+1}(N) 轮。

**具体例子——N=10000，k=3**：

```
log_4(10000) ≈ 6.64 → 约 7 轮
```

所以 10000 个节点在经过 7 轮后就几乎全部收到了消息。

**为什么实际收敛更快？** 在后期，感染节点越多，选到未被感染的节点的概率越大——传染效率反而**加速**——类似于"指数增长的一半是中后期的加速阶段"。

**Gossip 的容错性**：即使某些节点崩溃或消息被丢弃，其他节点持续选择不同节点推送——最终覆盖率仍然很高。这就是"容错"的来源。

### 6.6.3 Gossip 的收敛性量化分析

**为什么有效？** 每个节点每次都随机选择 k 个节点推送——经过 log_k(N) 轮后，几乎所有节点都会收到消息（类似传染病传播模型）。

**概率性**：不保证 100% 的节点都收到——但通过增加扇出因子 k 或增加推送轮数，可以把"收不到"的概率降低到任意小（但永远不是 0）。

**与 TCP 可靠性的对比**：TCP 是确定性可靠（通过 ACK + 重传），Gossip 是概率性可靠（通过随机推送 + 多次迭代）。在大规模系统中（>10000 节点），Gossip 的消息效率远高于确定性广播——因为不需要每个节点向每个其他节点发送确认。

**Gossip 的应用**：Cassandra 的成员发现（Gossip 协议检测节点上下线）、Consul 的跨数据中心服务发现、DynamoDB 的副本同步。

## 6.7 因果广播——保持因果顺序

### 6.7.1 为什么需要因果顺序？

**例子**——聊天应用：

```
p1 广播 "你好"
p2 收到 "你好"，然后广播 "你好，我是 p2"

p3 的顺序：
  正确顺序: Deliver("你好") → Deliver("你好，我是 p2")
  因果顺序: 先收 "你好" 再收 "你好，我是 p2" 才是符合因果的
  错误顺序（FIFO不够）: Deliver("你好，我是 p2") → Deliver("你好") ← 违反因果！
```

**因果顺序定义**：如果事件 A 通过消息影响了事件 B（A 发送 → B 接收 → B 响应），则 A 的交付必须先于 B。

### 6.7.2 算法——向量时钟

**因果广播（Causal Broadcast）** 通过**向量时钟（Vector Clock）** 实现：

每个进程维护一个维度为 N 的向量 `VC[1..N]`：
- VC[p] = p 已发送的消息数
- VC[q] = p 最后从 q 接收的消息的编号

**交付规则**——p 收到 q 的消息 m 携带向量 `m.VC`：

- 条件 1：`m.VC[q] == VC[q] + 1`（q 的消息是按序到达的——FIFO）
- 条件 2：对所有 k ≠ q，`m.VC[k] ≤ VC[k]`（发送方 q 在发送 m 之前，已经看到了 p 所看到的所有消息——因果前提满足）

**只有两个条件都满足，才能交付消息**——否则消息被缓冲，直到条件满足。

**向量时钟和 Lamport 时钟的关系**：Lamport 时钟是标量——只保证"如果 A 影响 B，则 C(A) < C(B)"。向量时钟是向量——还保证"如果 VC(A) < VC(B)（逐元素比较），则 A 是 B 的因果前提"。标量时钟不能用于因果广播——需要向量时钟来检测因果前置条件。

### 6.7.3 向量时钟的完整示例

假设 3 个进程 p1, p2, p3。设初始向量时钟全部为 (0,0,0)。

```
T0: p1 广播 "hello"
    p1.VC = (1,0,0) -> 发送 m₁("hello", VC=(1,0,0))

T1: p2 收到 m₁
    检查：m₁.VC[1] = 1 == p2.VC[1]+1 = 1 ✓（FIFO条件）
          m₁.VC[2] = 0 == p2.VC[2] = 0 ✓（m₁ 中 p2 的计数不变）
          m₁.VC[3] = 0 == p2.VC[3] = 0 ✓
    → 条件满足！交付 m₁
    p2.VC 更新为 max(p2.VC, m₁.VC) = (1,0,0)

T2: p2 广播 "hello, I'm p2"（响应消息）
    p2.VC[2] += 1 → p2.VC = (1,1,0)
    发送 m₂("hello, I'm p2", VC=(1,1,0))

T3: p3 先收到 m₂（乱序！因为发送顺序是 m₁ 先于 m₂）
    m₂.VC = (1,1,0)
    检查：m₂.VC[2] = 1 == p3.VC[2]+1 = 1 ✓（FIFO条件）
          m₂.VC[1] = 1 ≤ p3.VC[1] = 0? ❌ 不满足！
          → p3 还没有收到 m₁（p3 的 VC[1] 仍是 0，但 m₂ 知道 p1 已发了一条消息）
          → m₂ 被缓冲！等待 m₁ 到达

T4: p3 收到 m₁("hello", VC=(1,0,0))
    检查：m₁.VC[1] = 1 == p3.VC[1]+1 = 1 ✓
          m₁.VC[2] = 0 == p3.VC[2] = 0 ✓
    → 条件满足！交付 m₁
    p3.VC 更新为 (1,0,0)

T5: p3 重新检查缓冲的 m₂
    m₂.VC = (1,1,0)
    现在的 p3.VC = (1,0,0)
    检查：m₂.VC[1] = 1 ≤ p3.VC[1] = 1 ✓（现在 p3 已经有 m₁ 了！）
          m₂.VC[2] = 1 == p3.VC[2]+1 = 1 ✓
    → 条件满足！交付 m₂

p3 的最终交付顺序：m₁("hello") → m₂("hello, I'm p2") ✓ 正确因果顺序！
```

**关键观察**：
- **T4 之前的缓冲**——m₂ 依赖于 m₁ 的因果前提（p2 先收到 m₁ 才发 m₂），p3 检测到因果前置条件不满足，自动缓冲 m₂
- **因果依赖链**——m₁ → m₂（m₁ 是 m₂ 的因），p3 在交付 m₂ 之前必须确认已经"看到"了 m₁ 的因果前提
- **Lamport 时钟无法做到这件事**——Lamport 时钟只保证 C(m₁) < C(m₂)，但不能告诉 p3 "你还没收到 m₁"

**向量时钟的局限**：维度为 N（进程数量）。在大规模系统中（N>1000），每个消息都携带 N 维向量是不可接受的。**区间向量时钟（Interval Vector Clock）** 是优化——将多个进程的时钟合并为区间，减少维度。

## 6.8 各层广播的工程选择

| 业务场景 | 推荐广播 | 理由 |
|---------|---------|------|
| Leader选举的心跳 | 尽力广播 | 丢失后重发即可 |
| 配置更新通知 | 正则可靠广播 | 所有节点必须收到 |
| 分布式事务的提交 | 统一可靠广播 | 所有参与者必须一致 |
| 新节点加入集群 | 顽固广播 | 每个现有节点都必须知道 |
| 大规模集群成员发现 | 概率广播（Gossip） | 数万节点——确定性广播不可扩展 |
| 聊天应用 | 因果广播 | 保持对话的因果顺序 |
| 复制状态机的日志 | FIFO广播 | 保持操作顺序一致 |

## 6.9 本章总结

### 5 个核心要点

1. **广播抽象逐层构建**——和最底层的尽力广播出发，根据需要增加 一致性（正则）、统一性（统一）、确定性（顽固）、顺序（FIFO/因果）、概率性（Gossip）等保证。

2. **正则可靠广播通过"接收方转发"实现一致性**——即使发送方崩溃，只要至少一个正确的节点收到了消息，它就会转发给其他节点。

3. **统一可靠广播通过"多数派确认"实现统一性**——只有被多数派确认的消息才能被交付，保证了崩溃后消息的"全或无"性质。

4. **Gossip 协议用概率性替代确定性**——在大规模系统中，通过随机推送实现高效传播，消息复杂度 O(N×logN) 远低于确定性广播的 O(N²)。

5. **因果广播通过向量时钟保持因果顺序**——消息遵循"因果前置"条件才能交付。Lamport 时钟不能替代向量时钟进行因果检测。
