## 1.1 什么是分布式系统——形式化定义

### 1.1.1 从直觉到精确定义

**直觉定义**：分布式系统是由多个通过网络连接的独立计算机组成的系统，这些计算机通过消息传递协同工作，对外表现为一个统一的系统。

但直觉定义无法用来证明算法的正确性。形式化分析需要更精确的定义：

**形式化定义**：一个分布式系统由以下元素构成：
- **进程集合**（Process Set）：N 个进程 P = {p₁, p₂, ..., p_N}，每个进程有自己的本地状态和本地时钟
- **消息**（Message）：进程之间的唯一通信方式——没有共享内存，没有全局时钟
- **事件**（Event）：进程状态的改变由事件驱动——接收消息、超时、本地计算

**关键约束**：
1. **无共享内存**：进程不能直接读写其他进程的状态
2. **无全局时钟**：每个进程有自己的本地时钟，时钟可能不同步
3. **消息延迟无上界**（异步模型下）：消息可能被延迟任意长时间，可能丢失，可能乱序
4. **部分失效**：某些进程可能崩溃，而其他进程继续运行

**这个定义排除了什么**：
- 多线程程序（有共享内存）——不算分布式系统
- 主备数据库同步（有共享存储）——如果依赖共享存储同步，不算严格意义上的分布式系统

> **跨书对照**：释慧利第1章 1.1 节从"单机确定性消失"的直观感受切入。江峰本书从形式化定义切入——两者互补：释慧利给直觉，江峰给数学。

### 1.1.2 为什么需要形式化？

考虑一个简单的问题：设计一个算法，让 3 个进程就一个值达成一致。

**直觉做法**：投票——每进程广播自己的值，选择多数值。

但"多数值"在以下情况下可能不存在：
- 进程 p₁ 提议 v=0，但消息丢失，其他进程不知道
- 进程 p₂ 提议 v=1，进程 p₃ 提议 v=1——多数是 v=1
- 但 p₁ 不知道，p₁ 看到的"多数"可能是 0

**如果没有形式化**，你会花大量时间 debug"为什么偶尔不一致"——但你永远找不到根本原因，因为根本原因是**算法在理论上就不正确**。

**形式化的价值**：在写代码之前，用数学证明算法是否正确。如果数学证明无法通过，代码写得再好也没用——分布式系统的错误往往不是"代码有 bug"，而是"算法有漏洞"。

## 1.2 分布式算法的意义

### 1.2.1 分布式算法解决什么问题？

分布式算法解决三类问题：

| 问题类别 | 描述 | 典型算法 |
|---------|------|---------|
| **选主（Leader Election）** | 多个进程中选出唯一的 Leader | 霸道算法（Bully）、Ring 算法、Chandra-Toueg 失败检测器（第5章） |
| **广播（Broadcast）** | 将消息可靠地传递给所有（或多数）进程 | 尽力广播→可靠广播→原子广播（第6章） |
| **共识（Consensus）** | 多个进程就一个值达成一致 | Paxos、Raft、ZAB（第8章） |

### 1.2.2 为什么需要形式化分析？

**例子**：设计一个"霸道算法"（Bully Algorithm）——进程 ID 最大的当 Leader。

直觉上很简单：每个进程检查是否有比自己 ID 更大的进程存活，如果没有就当 Leader。

但在以下故障场景下：
- p₃（最大 ID）发起选举，消息发给 p₁、p₂
- p₁ 收到，但 p₁ 和 p₃ 之间的网络随后断开
- p₁ 不知道 p₃ 是否存活——p₁ 应该发起选举吗？

**形式化分析的回答**：这取决于系统的**时间假设**（同步/异步/部分同步）和**故障模型**（崩溃/遗漏/拜占庭）。不同的假设下，同样的算法可能是正确的、也可能是不正确的。

## 1.3 "两将军"问题——分布式不可能性的经典

### 1.3.1 问题描述

两支军队驻扎在两个山头上，中间隔着敌人。两将军通过通信兵传递消息，约定进攻时间。**问题是：两将军如何保证同时进攻？**

```
将军 A ──信使(消息可能被俘)──► 将军 B
        ◄──信使(消息可能被俘)──
```

### 1.3.2 直觉方案——为什么三层确认也不行

**方案一：发送一次消息**

```
A → B: "明天 6:00 进攻"
```

问题：如果消息丢失，B 不知道要进攻——A 进攻了，B 没进攻 → 战败。

**方案二：要求确认**

```
A → B: "明天 6:00 进攻"
B → A: "收到，我明天 6:00 进攻"
```

问题：B 不知道 A 是否收到了自己的确认。从 B 的视角：
- 如果 A 收到了确认 → A 会进攻，B 也会进攻 → 胜利
- 如果 A 没收到确认 → A 可能不进攻，B 进攻 → 战败

**方案三：确认的确认**

```
A → B: "明天 6:00 进攻"
B → A: "收到"（第一层确认）
A → B: "收到你的确认"（第二层确认——确认的确认）
```

问题：A 不知道 B 是否收到了"确认的确认"。从 A 的视角：
- 如果 B 收到了"确认的确认" → B 会进攻，A 也会进攻 → 胜利
- 如果 B 没收到 → B 可能不进攻 → 战败

**方案 N：第 N 层确认**

每一层确认都需要下一层确认来保证"对方知道我知道"。第 N 层的发送方永远不知道第 N 层是否被接收——因为需要第 N+1 层来确认，这就进入了**无穷递归**。

**关键洞察**：在任意有限层确认后，最后一层的发送方永远无法确定接收方是否收到了最后一条消息。因此，不存在一个有限消息序列能保证两将军同时进攻。

### 1.3.3 形式化证明——为什么无解？

不严格的直觉推导还不够——我们需要严格证明。

**定义**：
- 设有一个确定性协议 P，两将军通过有限消息交换达成一致
- "达成一致" = 两将军做出相同的决策（都进攻或都不进攻）
- 信道不可靠：消息可能丢失，但消息内容不被篡改（不丢的消息是正确的）

**定理**：在不可靠信道上，不存在有限消息的确定性协议能保证两将军总是达成一致。

**证明**（反证法）：

假设存在这样的协议 P。考虑 P 的执行历史 H——从初始状态到决策状态的消息序列。

设 H 中最短的可能执行序列（消息最少）有 m 条消息。不失一般性，假设最后一条消息 m 是 A 发给 B 的。

考虑执行序列 H'——与 H 完全相同，但第 m 条消息丢失。此时：
- A 的视角：A 发送了第 m 条消息后，不知道 B 是否收到。在 H 中，A 发送第 m 条消息后做出了决策 D_A。在 H' 中，A 的状态与 H 中完全相同（A 无法区分 H 和 H'）——所以 A 同样做出决策 D_A。
- B 的视角：在 H' 中，B 收到的消息序列比 H 少 1 条（少收到第 m 条）。B 的状态与 H 中不同——B 可能做出与 H 中不同的决策 D'_B。

如果 D_A ≠ D'_B，则协议 P 无法保证一致性 → 矛盾。

如果 D_A = D'_B，则 H' 也是一个"达成一致"的有效执行序列，且消息数为 m-1。但我们假设 m 是最短的"必然达成一致"的消息数 → m-1 < m，矛盾。

**因此，不存在有限消息的确定性协议** ✓

### 1.3.4 证明的关键——无法区分

这个证明的核心是一个重要思想：**进程无法区分不同的执行历史**。

在证明中，A 无法区分 H（第 m 条消息到达 B）和 H'（第 m 条消息丢失）。因为 A 的本地状态在两种情况下完全相同，A 必须做出相同的决策。任何要求 A 在 H 和 H' 中做出不同决策的协议必然违反"确定性"要求。

**无法区分（Indistinguishability）** 是分布式计算的形式化理论中最基本的概念：
- 两个执行历史对进程 p 来说无法区分，当且仅当 p 在两者中经历了完全相同的事件序列
- 由于分布式系统中进程只能依赖本地信息，如果两个历史对进程来说无法区分，进程必须在两者中做出相同的行为

这个概念在后续章节中反复出现——第3章的系统模型、第8章的共识不可能性证明（FLP）都依赖它。

### 1.3.5 两将军问题的工程含义

两将军问题告诉我们：**在不可靠的通信链路上，两个进程无法通过有限消息交换对共同行动达成确定性的一致。**

**工程绕行方案**：

1. **引入超时 + 重试**：在"部分同步"假设下（网络最终会恢复），通过超时和重试可以概率性地达成一致（虽然不是 100% 保证）。TCP 的三次握手就是这种思路——两将军问题在理论上无解，但 TCP 在工程上"足够可靠"
2. **引入随机化**：用随机退避（Random Backoff）降低冲突概率。不是确定性的保证，但概率上可行
3. **使用可靠信道**：如果信道本身保证可靠（如 TCP over 局域网），两将军问题退化为"可靠信道上的协调"——这是另一个问题

**与共识算法的关系**：两将军问题是 2 进程的协调问题。共识问题是 N 进程的值一致问题。两者在本质上不同——两将军问题中不存在值分歧（两将军都同意要进攻），只需要对"何时进攻"达成时间协调；共识问题中进程可能提出不同的值。

> **跨书对照**：释慧利第1章 1.4.3 节讲的是拜占庭将军问题（可靠通道 + 不可靠行为），江峰本书讲的是两将军问题（不可靠通道 + 可靠行为）。两者是完全不同的问题，但都揭示了分布式的根本困难。释慧利的拜占庭将军有解（需要 3f+1 个将军容忍 f 个叛徒），江峰的两将军在有限消息下无解（但可以通过 TCP 重传在工程上绕过）。

## 1.4 设计分布式算法的主要挑战

### 1.4.1 并发执行

**问题**：多个进程并发执行，没有全局顺序。

**例子**：两个进程 p₁ 和 p₂ 同时增加同一个计数器。

```
p₁: 读 counter=0 → 计算 counter+1 → 写 counter=1
p₂: 读 counter=0 → 计算 counter+1 → 写 counter=1

结果：counter=1（应该是 2）
```

**形式化处理**：分布式算法必须显式处理并发——用锁（分布式锁）、用仲裁（Leader 串行化）、用冲突检测（向量时钟/版本号）。

### 1.4.2 进程失败

进程可能以多种方式失败。不同的失败模型对应不同的算法假设——一个算法在崩溃式失败模型下是正确的，在遗漏式失败模型下可能不正确。

| 失败模型 | 行为 | 检测难度 | 形式化定义 |
|---------|------|---------|-----------|
| **崩溃式（Crash-Stop）** | 进程停止执行，不再恢复，停止前后行为正确 | 简单（不响应 = 崩溃？还是消息延迟？） | 进程按照算法正确执行每一步，直到某个时刻后停止执行所有步骤 |
| **遗漏式（Omission）** | 进程可能不发送或不接收某些消息，但计算正确 | 困难（不响应可能是遗漏也可能是延迟） | 进程按照算法正确计算，但某些消息的发送或接收事件"遗漏"了 |
| **崩溃恢复式（Crash-Recovery）** | 进程崩溃后恢复，可能丢失内存状态 | 困难（需要持久化 + 状态恢复） | 进程可能停止执行，之后重新开始，但丢失了上一次运行的内存状态 |
| **拜占庭式（Byzantine）** | 进程可能执行任意操作，包括发送矛盾消息 | 极困难（需要加密验证） | 进程的行为可以偏离算法的任何规定——包括发送伪造消息 |

**四种失败模型的层次关系**：

```
崩溃式 ⊂ 遗漏式 ⊂ 崩溃恢复式 ⊂ 拜占庭式
（最弱）                                      （最强）
```

"⊂"表示：如果一个算法在更弱的失败模型下正确，它在更强的失败模型下**不一定**正确。例如：Paxos 在崩溃式模型下正确，但在拜占庭模型下不正确（因为拜占庭进程可能伪造消息）。

**崩溃式失败——最简单的模型**：

这是大多数实用共识算法（Paxos/Raft/ZAB）的默认假设。为什么选择崩溃式？
1. **可控环境**：后端中间件部署在局域网/私有云——不存在恶意进程
2. **TLS/mTLS 解决消息篡改**：加密通道保证了"消息不被篡改"——在传输层解决了拜占庭问题
3. **复杂度大幅降低**：崩溃式容错需要 2f+1 个进程，拜占庭容错需要 3f+1——实现复杂度和性能差距巨大

**遗漏式失败——更接近真实网络**：

崩溃式假设"进程正常执行直到崩溃"。但真实系统中，进程可能因为网络拥塞、磁盘 IO 抖动、GC 停顿等原因"暂时性无法发送消息"——这既不是崩溃（进程还在运行），也不是完全正常（消息发送遗漏了）。

遗漏式失败模型更接近真实网络环境——但处理也更复杂。Chandra-Toueg 失败检测器（第5章）就是为遗漏式失败设计的。

> **跨书对照**：释慧利第1章 1.6.2 节讨论了 Crash-Stop vs Crash-Recovery vs Omission vs Timing。江峰本书在第3章进一步形式化这些失败模型，并证明它们之间的层次关系（例如，"遗漏式失败可实现崩溃式失败检测"）。

### 1.4.3 链路失败

链路（进程之间的通信通道）同样可能以多种方式失败。分布式算法的一个重要技巧是**在弱链路上构建强抽象**——用算法逻辑弥补链路的不确定性。

| 链路模型 | 行为 | 需要算法做什么 | 形式化定义 |
|---------|------|--------------|-----------|
| **公平丢包** | 如果无限次重发，每条消息最终都会到达 | 重传 + 去重 | 对于每条消息，如果发送方无限次重发，接收方最终会收到至少一次 |
| **顽固链路** | 公平丢包 + 发送方持续重复发送直到接收方确认 | 发送方需实现"持续重传直到确认" | 基于公平丢包链路，发送方永远重复发送，直到收到确认 |
| **可靠链路** | 消息不丢失（每条消息恰好被接收一次） | 无需重传 | 每条消息如果被发送，最终恰好被接收一次，不重复 |
| **FIFO 可靠链路** | 可靠 + 顺序保证 | 无需排序 | 消息按发送顺序接收 |
| **日志可靠链路** | FIFO + 崩溃恢复后可恢复之前的链路状态 | 记录日志 + 恢复 | FIFO 可靠 + 进程崩溃后重启，链路状态（如已接收的消息）不丢失 |

**链路抽象的层次结构**（第4章详述）：

```
公平丢包 ──(重传去重)──► 顽固链路 ──(持续确认)──► 可靠链路 ──(序号)──► FIFO ──(持久化)──► 日志可靠
```

每一层都在下层的基础上增加保证——这是分布式算法的"分层设计"哲学：不假设底层提供所有保证，而是逐层构建。

**为什么中间有"顽固链路"？**

顽固链路看起来很低效——发送方永远重复发送直到确认。但它是从公平丢包到可靠的关键过渡：公平丢包说"无限次重发最终会到达"，但没说"谁来重发"。顽固链路的发送方承担了"无限重发"的责任，从而构建了"最终不丢"的保证。

这与 TCP 的累积确认（Cumulative ACK）+ 重传机制在概念上同源——不同的是，TCP 在传输层实现，顽固定链路在分布式算法层实现。

### 1.4.4 Safety 与 Liveness——分布式算法的两个维度

Safety 和 Liveness 是分布式算法形式化分析的核心框架（Alpern & Schneider, 1985）。几乎所有分布式算法的正确性都归结为证明这两个属性。

**Safety（安全性）**：不好的事情**永远不会发生**。

- 形式化定义：对于所有可能的执行序列 σ，属性 P 在 σ 的每一个前缀上都成立。也就是说——如果 P 在某个时刻被违反，它就无法再被"修复"。
- 违反 Safety 的例子：共识算法产生两个不同的值——一旦出现两个不同的输出，即使后续更正，Safety 已经违反了（"坏事情已经发生"）。
- 有限前缀判据：Safety 可以在有限时间内判断是否被违反——如果看到两个不同的输出值，立刻知道 Safety 被违反，不需要等待。

**Liveness（活性）**：好的事情**最终会发生**。

- 形式化定义：对于每个有限的执行前缀 α，存在一个扩展（后续的步骤序列）使得属性 P 最终成立。也就是说——无论当前看起来多么糟糕，只要你对未来"永远抱有希望"地继续执行，最终会有好事发生。
- 违反 Liveness 的例子：系统永远选不出 Leader——无论多长时间，好事都没发生。但如果系统在 T₁→T₂→T₃ 都没有选出 Leader，却在 T₄ 时刻选出了——Liveness 没有被违反（"好事虽然迟到了，但最终发生了"）。
- 无限后缀判据：Liveness 只能在**无限执行序列**上判断是否被违反——你永远不能说"系统现在还没选出 Leader，所以 Liveness 被违反了"——因为它可能在下一秒选出。只有系统**无限期**没有选出 Leader，你才能在"看完"无限长的执行后判断 Liveness 被违反。

**Safety 和 Liveness 的直观对比**：

| 属性 | 判据 | 可以"修复"吗？ | 例子 |
|------|------|-------------|------|
| Safety | 有限前缀 | 不能——一旦违反，就永远违反了 | 两个不同的共识值、两个 Leader 同时存在、消息被篡改 |
| Liveness | 无限后缀 | 可以——迟到不算违反 | 最终选出 Leader、最终所有消息都被投递、最终值被选定 |

**每个 Safety 属性都有一个对应的 Liveness 属性**——它们是分布式算法的"阴阳两面"：

| 分布式算法 | Safety（不会发生坏事） | Liveness（好事最终发生） |
|-----------|---------------------|----------------------|
| 共识（Paxos） | 只有一个值被选定 | 如果所有正确进程提出提案，最终有值被选定 |
| 选主 | 同一任期最多一个 Leader | 最终会选出 Leader |
| 可靠广播 | 所有正确进程收到相同的消息 | 如果发送方正确，最终所有正确进程都会收到消息 |
| 原子广播 | 所有进程按相同顺序收到消息 | 如果发送方正确，最终所有正确进程都会收到消息 |

**为什么需要两个维度？**

FLP 不可能定理（第10章）的核心结论是：在异步系统中，不可能同时保证 Safety 和 Liveness。因此，每个实用分布式协议都在二者之间做了取舍：
- Paxos：Safety 永远保证（在异步模型下），Liveness 只在部分同步模型下保证（Leader + 超时打破活锁）
- 2PC（两阶段提交）：Safety 保证（原子提交），但 Liveness 有缺陷（协调者崩溃导致阻塞）

### 1.4.5 性能度量——不止是时间复杂度

分布式算法的性能度量与传统算法不同——消息复杂度往往比时间复杂度更重要。

**三个关键度量**：

| 度量 | 含义 | 典型分析 |
|------|------|---------|
| **消息复杂度** | 执行过程中发送的消息总数 | Paxos 的 Prepare 阶段 = O(N) 消息（向所有 Acceptor 发送） |
| **时间复杂度（通信轮次）** | 从发起到完成所需的"消息往返次数" | Paxos 需要 2 轮 RTT（Prepare + Accept），Multi-Paxos 只需 1 轮 |
| **空间复杂度（持久化状态）** | 每个进程需要持久化的数据量 | Paxos Acceptor 需要持久化 promised_n + accepted_proposal |

**为什么消息复杂度更重要？** 因为网络 I/O 是分布式系统的瓶颈——CPU 计算相对便宜。

**例子**：Paxos 的 Prepare 阶段向所有 Acceptor 发送 Prepare 消息 = O(N) 消息。如果集群有 100 个节点，每个 Prepare 阶段需要 100 条消息。而 Paxos 的具体计算（比较编号、检查承诺）是 O(1) 的——消息开销远超计算开销。

### 1.5.1 从"是什么"到"为什么对"

本书的核心目标是回答"为什么分布式算法正确"，而不仅仅是"分布式算法做什么"。

**框架**：
1. **算法描述**：用准代码描述算法（类似伪代码，但更严谨）

3. **正确性证明**：证明算法满足 Safety 和 Liveness
4. **性能分析**：消息复杂度 + 时间复杂度

## 1.6 本章总结

### 4 个核心要点

1. **分布式系统的形式化定义**：进程集合 + 消息传递 + 无共享内存/全局时钟 + 部分失效。形式化的价值：在设计阶段证明算法正确性，避免"看似正确但偶尔出错"的工程悲剧。

2. **两将军问题**：在不可靠链路上，两方通过有限消息交换无法保证同时进攻——揭示了分布式协调的本质不可能性。这是比 FLP 更简单的"分布式不可能性"入门。

3. **三种失败模型**：进程失败（崩溃→遗漏→恢复→拜占庭）、链路失败（丢包→可靠→FIFO→日志）、并发执行。不同假设下，同一算法可能正确也可能不正确。

4. **Safety 与 Liveness**：Safety 在每一时刻都可能违反（"坏事永远不会发生"），Liveness 只能在无限时间后判断（"好事最终发生"）。这是本书证明所有算法正确性的核心框架。

### 第2章预告

第2章将介绍 **I/O 自动机**——分布式算法的形式化描述工具。I/O 自动机是 Nancy Lynch 提出的数学模型，用状态机 + 标记转移精确描述算法行为。所有后续章节的算法都将用 I/O 自动机描述和证明。
