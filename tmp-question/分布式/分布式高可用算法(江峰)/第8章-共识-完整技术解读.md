## 8.1 共识——分布式系统的皇冠明珠

### 8.1.1 前7章的积累——你能用 I/O Automata 做什么了？

回顾前7章的构建过程：

```
第4章（链路）→ 一对一可靠传输
第6章（广播）→ 一对多可靠交付
第7章（注册器）→ 多读多写共享变量
第5章（失败检测）→ 判断进程存活
```

**这些是你设计共识算法的工具**。第8章的目标是——用这些工具构建**共识**：在一个可能崩溃的分布式系统中，让所有正确进程就一个共同的值达成一致。

### 8.1.2 共识的直觉定义——三个核心性质

回忆释慧利第1章的形式化定义：

| 性质 | 含义 | 你的翻译 |
|------|------|---------|
| **Agreement（一致性）** | 所有正确进程决定的值相同 | "不能有两个不同的结果" |
| **Validity（有效性）** | 决定的值必须是某个进程提出的值 | "不能凭空捏造结果" |
| **Termination（终止性）** | 每个正确的进程最终都会决定 | "不能永远悬而未决" |

**这三个性质就是共识的全部要求**。但第1章的定义是直觉的——现在用 I/O Automata 给出精确定义。

### 8.1.3 用 I/O Automata 定义共识

```
共识自动机 Consensus[N]：

  每个进程 p_i：
    状态：
      proposed_i = None | v_i     // p_i 提出的值（初始为 None）
      decided_i = None | v_i      // p_i 决定的值（初始为 None）
    
    签名：
      input = {Propose(v)}        // 进程 p_i 提出值 v
      output = {Decide(v)}        // 进程 p_i 决定值 v

  性质（用迹集定义）：
    Agreement:  ∀p_i, p_j：如果 p_i 的迹包含 Decide(v1)，p_j 的迹包含 Decide(v2)
               → v1 = v2
    Validity:  ∀p_i：如果迹包含 Decide(v)，则 ∃p_j 使得迹包含 Propose(v)
    Termination: ∀正确的 p_i：迹最终包含某个 Decide(v)
```

**关键**：Agreement 和 Validity 是 Safety 性质（有限前缀可判）。Termination 是 Liveness 性质（公平性+终判）。

> **跨书对照**：释慧利第1章 1.4 节定义了共识的这三个性质，第4章展示了 Paxos 如何满足它们。江峰第8章用 I/O Automata 的迹集精确刻画——差异在于"工程直觉 vs 形式化严谨"。

## 8.2 共识的不可解性——FLP 的直觉重温

### 8.2.1 在什么假设下共识可解？

不是所有假设下共识都可解。第3章的时间假设决定了答案：

| 失败模型 | 时间假设 | 共识可解？ | 原因 |
|---------|---------|:---:|------|
| 崩溃式 | 同步 | ✅ | 已知延迟上界——可区分崩溃和延迟 |
| 崩溃式 | 异步 | ❌ | FLP 定理——无法在有限时间内终止 |
| 崩溃式 | 部分同步 | ✅ | Paxos/Raft/ZAB——概率性终止 |
| 拜占庭 | 异步 | ❌ | 即使允许无限时间也无法——拜占庭进程可能伪造消息 |
| 崩溃式+随机化 | 异步 | ✅（概率） | Ben-Or 1983——概率性协议 |

### 8.2.2 FLP 的直觉——为什么异步共识不可能？

第1章的两将军问题给了第一步直觉。FLP 定理（Fischer/Lynch/Paterson, 1985）是它的泛化——在 N 进程的异步系统中，只要有一个进程可能崩溃——共识不可能在有限时间终止。

**恶意调度器的游戏**：
1. 系统从初始双值状态（既可能决定 0 也可能决定 1）
2. 调度器控制消息的延迟顺序——永远让系统保持双值
3. 调度器可以无限延迟一个进程的消息——其他进程无法判断是"延迟"还是"崩溃"
4. 系统永远无法进入单值状态 → 永远无法决定 → 违反 Termination

**这就是为什么所有实用共识算法都需要部分同步假设**——超时打破了调度器的"恶意"。

## 8.3 正则共识——Paxos 的目标

### 8.3.1 定义——不同于统一共识

共识有两种形式：

| 形式 | 统一性要求 | 含义 |
|------|-----------|------|
| **正则共识** | 不需要统一 | 正确进程决定的值相同——但崩溃的进程可能决定不同的值（在它们崩溃的瞬间） |
| **统一共识** | 需要统一 | 即使崩溃的进程——也必须在崩溃的瞬间做出与正确进程相同的决定 |

**为什么需要区分？** Paxos 的 Accept 阶段是"多数派确认"——如果进程在多数派确认后崩溃了，它的决定可能和正确的进程不同。所以 Paxos 是**正则共识**，不是统一共识。

### 8.3.2 正则共识的形式化定义（I/O Automata）

```
性质——正则共识 RegularConsensus[N]:

  Agreement: 同 8.1.3
  Validity:  同 8.1.3
  Termination: 同 8.1.3
  正则性:     不需要统一——崩溃进程可能决定了与正确进程不同的值
```

### 8.3.3 Paxos 是正则共识的实例

Paxos 满足正则共识的所有性质：

| Paxos 的保证 | 对应正则共识的哪个性质 | 如何保证 |
|-------------|---------------------|---------|
| 只有一个值被选定 | Agreement | Prepare/Promise 探测并发→多数派交集保证 |
| 选定的值是某个提案的值 | Validity | Proposer 只提已准备的值 |
| 最终有值被选定 | Termination | Leader + 超时打破活锁 |

**但 Paxos 不是统一共识**——如果一个进程在 Accept 阶段接受了提案但崩溃了，它可能没收到 Commit——它的决定值可能不同于正确的进程。

### 8.3.4 证明草图——Paxos 满足正则共识（I/O Automata 迹集分析）

用第2章学的 I/O Automata 和迹集，给一个形式化证明的缩略版——让你看到"形式化如何工作"。

**要证明**：Paxos 的 Acceptor+Proposer 组合自动机的所有迹都满足正则共识的 Agreement 和 Validity 性质。

**证明 Agreement（不可能有两个不同值被选定）**：

回顾 Paxos 的第4章关键不变式：
```
Invariant 1: 如果一个提案 (n₁, v₁) 被多数派 Q₁ 接受，且 (n₂, v₂) 被多数派 Q₂ 接受
→ 如果 n₂ > n₁ → v₂ = v₁
```

**用迹集证明**：
- 假设 Paxos 的迹集中存在一条迹 t，其中出现了两个不同值 v₁ ≠ v₂
- 在 t 中，v₁ 在轮次 n₁ 被多数派 Q₁ 接受，v₂ 在轮次 n₂ 被多数派 Q₂ 接受
- 不失一般性，设 n₁ < n₂
- 当 Q₂ 中的 Acceptor 接收到 Prepare(n₂) 时，它们会报告轮次 < n₂ 的已接受提案
- Q₁ 和 Q₂ 有交集（鸽笼原理）→ 交集节点在 Q₂ 的 Prepare 中报告了 v₁
- Proposer 发出 Accept(n₂) 时，必须采用 v₁ 作为提案值（第4章不变式2）→ v₂ = v₁
- **矛盾！** 所以不存在 v₁ ≠ v₂——Agreement 满足 ✓

**证明 Validity（选定的值一定是某个进程提出的值）**：
- 从迹集中看，Proposer 只在 Accept 阶段提出值
- 每个 Accept 的值要么是 Proposer 自己的值（如果 Prepare 时没有已接受提案），要么是继承的值（第4章不变式2）
- 继承的值 = 之前某个 Proposer 的值——所以最终选定的值一定是某个进程提出的值 — Validity 满足 ✓

**为什么不能在异步下完成？**
- Agreement 和 Validity 的证明不依赖于时间假设 — Safety 在异步下成立
- Termination 需要"最终有值被选定"——在异步下可能被活锁阻止（多 Proposer 竞争）
- 活锁突破需要同步假设（Leader + 超时 + 随机退避）→ Paxos 的 Termination 只在部分同步下成立

### 8.3.5 正则共识的能力与局限——小结

正则共识（Paxos）的能力：保证正确进程的值相同 + 值来自某个提议。局限：崩溃进程可能不一致。统一共识（8.4）补上这个局限。序列共识（8.6）将单值扩展到序列。

## 8.4 统一共识——更强的保证

### 8.4.1 定义

**统一共识（Uniform Consensus）** 在正则共识的基础上增加：

| 附加性质 | 含义 |
|---------|------|
| **统一一致性** | 任何进程（包括崩溃的）的决定值都相同——不会出现"崩溃的进程决定 0，正确的进程决定 1" |

**如何实现？** 需要保证：在决定值之前，**所有进程**（包括崩溃的）都必须确认接收。这在工程实践中等价于"全确认"协议——类似第6章的统一可靠广播的"全部进程都确认"。

### 8.4.2 Paxos + Commit 确认 = 近似统一共识

Paxos 加上**Commit 确认机制**可以近似统一共识：
1. Accept 阶段选定值后——发送 Commit 给每个进程
2. 进程收到 Commit → 决定值
3. 如果某个进程崩溃——没收到 Commit → 没决定值
4. 重试 Commit 直到所有进程确认（类似顽固广播）

## 8.5 随机共识——异步系统下的概率解决

### 8.5.1 FLP 不允许确定性共识——但概率共识可以

FLP 定理证明**确定性**共识在异步下不可能。但 **Ben-Or（1983）** 的随机化协议证明：如果允许随机化（算法使用随机源），共识在异步下是**概率性**可行的——最终以概率 1 达成一致。

### 8.5.2 Ben-Or 概率共识——直觉算法

```
算法（Ben-Or，简化版——每个进程 p）：
  状态：
    value = 0 或 1（初始为 p 提出的值）
    round = 0

  while not decided：
    round += 1
    
    阶段一（投票）：
      广播 (round, value) 给所有进程
      等待 majority 回复
      if majority 的所有回复 value 都相同：
        value = 那个共同的值  // 多数派同意
      else：
        value = None  // 没有多数派

    阶段二（决定）：
      广播 (round, value) 给所有进程
      等待 majority 回复
      if majority 的所有回复 value 相同且 ≠ None：
        decided = value  // 多数派都同意非 None 值——决定！
      elif 存在至少一个回复 value ≠ None：
        value = 那个非 None 的值
      else：
        value = 随机取值——0 或 1（各 50% 概率）  ← **关键：随机性**！
```

**为什么需要随机性？** 如果所有进程永远选择 0——可能永远无法达成多数派的非 None 值（如果初始值不同）。随机取值打破了对称性——以一定的概率让系统进入"所有人都同意"的状态。

**收敛概率**：每轮随机选择的概率 ≥ 1/2。N 轮后收敛概率 ≥ 1 - (1/2)^N。

**这就是为什么 FLP 在异步系统下不是"共识不可能"——它是"确定性共识不可能"**。通过引入随机源，可以绕过 FLP 的限制。

### 8.5.3 Ben-Or 完整推演——随机性如何打破对称

设 3 个进程 p1（初始v=0），p2（初始v=0），p3（初始v=1）。

**第一轮**：

```
p1（v=0）：阶段一 → 广播(round=1, v=0)
p2（v=0）：阶段一 → 广播(round=1, v=0)  
p3（v=1）：阶段一 → 广播(round=1, v=1)

majority：p1和p2的值=0，p3的值=1 —— 不同！→ value=None

阶段二：广播(round=1, value=None)
majority：所有人都 None → 无决策
p1, p2, p3 的随机取值：
  p1: 随机 → 0（概率 50%）
  p2: 随机 → 0（概率 50%）
  p3: 随机 → 1（概率 50%）
  这轮没达到收敛。
```

**第二轮**（假设随机取值后 p1=0, p2=0, p3=1——与第一轮相同）：

```
同样流程 → 收敛概率 = 随机取值全部相同的概率 = (1/2)^3 = 1/8
```

**第三轮**：

```
p1（v=0），p2（v=0），p3（v=1）
阶段一：广播(round=3, v=0/0/1) → 不同的值 → None
阶段二：广播(round=3, None) → 如果没有非 None 值 → 随机取值

关键：如果 p1 和 p2 随机都取 0——概率 (1/2)^2 = 1/4
      p3 无论取什么——p1+p2 已经是 majority
      阶段一 majority 的值为 0（共同值）→ 阶段二 majority 也为 0 → decided=0！

收敛概率：每轮 ≥ (1/2)^(N-f) = (1/2)^2 = 1/4（f=1 崩溃容忍）
  4 轮后：≥ 1 - (3/4)^4 = 1 - 0.316 = 68.4%
  10 轮后：≥ 1 - (3/4)^10 = 94.4%
```

**为什么收敛？** 因为即使初始值不同——也有的概率某个轮次随机取值"凑巧"让大多数人一致——从此那个值就传播开来——永不改变。随机性不是"不可靠"——它是"最终可靠"。

## 8.6 序列共识——从单值到序列

### 8.6.1 定义

**序列共识（Sequence Consensus）** 扩展正则共识——从单个值到序列：

```
正则共识：Propose(v) → 选定一个值 v
序列共识：Propose([v1, v2, ..., vk]) → 选定一个序列 [v1, v2, ..., vk]
```

**与 Paxos 的关系**：Basic Paxos 每次运行一次正则共识——选择一个值。Multi-Paxos 将多次 Basic Paxos 串联为序列共识——日志的每个 entry 对应一次正则共识。

### 8.6.2 Multi-Paxos 和 Raft 的共识类型

| 算法 | 共识类型 | 如何处理序列 |
|------|---------|------------|
| Basic Paxos | 正则共识 | 每运行一次——选定一个值 |
| Multi-Paxos | 序列共识 | 通过多次正则共识串联——每个 entry 一次 |
| Raft | 序列共识 | Leader 串行 AppendEntries——每个 entry 加入日志序列 |
| ZAB | 序列共识 | Leader 串行 PROPOSAL——每个事务有唯一的 zxid |

**对比**：Multi-Paxos 的序列是通过"独立的正则共识"组合成的——序列中不同的 entry 对应**不同轮次**的 Paxos 运行。Raft 的序列是自然的——Leader 按 entry index 串行追加。Raft 更简单，Multi-Paxos 更灵活。

## 8.7 共识与其他抽象的关系

### 8.7.1 共识可以构建其他所有抽象

共识是分布式系统中最基础的抽象——可以构建：

```
共识（Consensus）
    ├──► 原子广播（第6章的统一可靠广播）
    ├──► Leader 选举（第5章——选主就是共识：选定 Leader ID）
    ├──► 分布式锁（共识：谁来持有锁）
    ├──► 原子注册器（第7章——共识确定写顺序）
    └──► 复制状态机（共识确定操作序列）
```

### 8.7.2 共识等价于全序广播

**全序广播（Total Order Broadcast）** 是一组进程就消息序列达成一致——这是**共识+广播**的组合。

共识 = 在 N 个进程中就一个值达成一致。全序广播 = 在 N 个进程中就消息序列的顺序达成一致。两者在形式化上是**等价的**——共识可以构建全序广播，全序广播可以构建共识。

## 8.8 本章总结

### 5 个核心要点

1. **共识是分布式系统皇冠上的明珠**——所有正确进程就一个值达成一致。三个性质：Agreement（一致）、Validity（有效）、Termination（终止）。用 I/O Automata 的迹集精确定义。

2. **共识的可解性依赖于假设**——在同步系统下可解、异步系统下 FLP 证明不可解、部分同步下可解（Paxos/Raft/ZAB）。不同失败模型对应不同的共识强度要求。

3. **正则共识 vs 统一共识**——前者允许崩溃进程决定不同值（Paxos 的多数派确认不需要全员确认），后者要求所有进程（包括崩溃的）决定相同值。

4. **随机共识绕过 FLP**——Ben-Or 1983 的概率协议在异步系统下以概率 1 达成一致。FLP 只适用于**确定性**共识——引入随机源后异步共识可行。

5. **序列共识通过多轮正则共识串联**——Multi-Paxos（独立的正则共识每次选择一个 entry），Raft（自然的串行序列），ZAB（unique zxid 保证顺序）。
