## 4.1 为什么需要链路抽象？

### 4.1.1 真实网络的本质

物理网络是复杂的：网线、交换机、WiFi、光缆、卫星——它们有完全不同的丢包率、延迟和故障模式。

**分布式算法的正确性不能依赖于物理层的具体实现**——如果算法假设"网络不丢包"，那么在 WiFi 环境（丢包率 <1%）中是正确的，但在卫星通信环境下（丢包率 >10%）就会失败。

### 4.1.2 分层设计——在弱假设上构建强保证

分布式算法的解决方案不是"用更好的网络"，而是**在弱假设上逐层构建强保证**——

```
公平丢包链路（最弱假设：无限重发最终会到达）
    │
    ├──(重传 + 去重)──► 可靠链路（不丢 / 不重复）
    │                       │
    │                       ├──(序号)──► FIFO 可靠链路（保持顺序）
    │                       │               │
    │                       │               └──(持久化日志)──► 日志可靠链路（崩溃恢复）
    │                       │
    │                       └──(持续确认)──► 顽固链路（确保对方收到）
    │
    └──(直接使用)──► 尽力广播（第6章）
```

每一层都是**抽象（Abstraction）**——对外暴露统一的接口（Send/Receive），隐藏内部实现（重传、去重、保序、恢复）。

### 4.1.3 所有链路的统一接口

无论哪一层，链路对外暴露的接口是相同的：

```
input:  Send(q, m)     // 向进程 q 发送消息 m
output: Receive(p, m)  // 从进程 p 接收消息 m
```

**抽象的价值**：上层算法（如 Leader 选举、广播、共识）只依赖 Send/Receive 接口，不关心底层是公平丢包链路还是日志可靠链路。你可以随时替换底层实现，而不需要修改上层算法。

> **跨书对照**：TCP 在传输层做了类似的分层设计——不丢不重复 + 保序。但分布式算法的链路抽象是在**应用层**重新实现的——因为不能假设底层 TCP 覆盖所有情况（例如进程崩溃恢复后 TCP 连接的状态丢失）。这就是为什么需要"日志可靠链路"和"顽固链路"——它们在 TCP 之上补充了崩溃恢复和持续确认的保证。

## 4.2 公平丢包链路——最基础的假设

### 4.2.1 定义

**公平丢包链路**（Fair-Loss Link）是所有链路抽象的基础。它定义了两个性质：

| 性质 | 形式化描述 | 含义 |
|------|-----------|------|
| **公平性（Fairness）** | 如果进程 p 无限次向 q 发送消息 m，且 q 是正确的（未崩溃），那么 q 最终会至少接收一次 m | 不会"永远丢包"——只要坚持重发，消息最终会到达 |
| **无伪造（No Creation）** | 如果 q 接收到了消息 m，那么 m 一定曾被某个进程 p 发送给 q | 不会凭空出现消息 |
| **无重复（No Duplication）** | 每一条消息最多被接收一次 | 一条消息不会"凭空多出来一份"——但如果链路层重复发送，会导致"重复接收"（这是链路本身的特性，不是消息被复制） |

**关键**：公平丢包链路**不清楚**消息何时会到达、经过几次重传——它只保证"最终"会到达。这是最弱的链路假设，也是与现实世界最接近的模型。

### 4.2.3 时间线推演——公平丢包到底有多"公平"？

假设 p 向 q 发送 1 条消息 m₁。网络有 30% 的丢包率。让我们跟踪 sent 和 rcvd 计数器：

```
T0: p 发送 m₁（第1次发送）
    sent = 1, rcvd = 0
T1: 消息丢失（30% 概率）——q 没收到

T2: p 发送 m₁（第2次发送）
    sent = 2, rcvd = 0
T3: 消息丢失（30% 概率）——q 没收到

T4: p 发送 m₁（第3次发送）
    sent = 3, rcvd = 0
T5: 消息成功到达 q（70% 概率）
    rcvd = 1 ← 终于收到了！

T6: q 确认后，p 停止发送 m₁
    消息 m₁ 成功投递——经过 3 次重传
```

**关键观察**：sent 从 1 增长到 3，rcvd 从 0 跳到 1。到达的时机**不可预测**——可能是第 1 次（如果没丢），可能是第 5 次（如果运气不好）。但公平性保证：**只要 sent 无限增长，rcvd 终将等于 1**——不会出现"永远丢包"的情况。

**如果丢包率是 100% 呢？**（即物理链路完全断开）

公平丢包链路的公平性条件**要求丢包率 < 100%**。如果链路完全断开（例如光纤被切断），sent → ∞ 但 rcvd = 0——公平性被违反。这不是公平丢包链路的假设——公平丢包假设"总有一定的概率成功"。在实际系统中，你需要让算法在**一段时间**后检测到"链路可能已经完全断开"，并进入故障状态处理——这正是第5章的失败检测器要解决的问题。

### 4.2.2 形式化定义（I/O 自动机）

```
抽象4-1：公平丢包链路

自动机：FairLossLink[p→q]（进程 p 到进程 q 的单向链路）

签名：
  in  = {Send(q, m) | m ∈ M}               // p 调用发送
  out = {Receive(p, m) | m ∈ M}             // q 感知接收
  int = {}

状态：
  sent[p, q, m] = 0  // p 发送 m 给 q 的次数（初始为 0）
  rcvd[p, q, m] = 0  // q 从 p 接收 m 的次数（初始为 0）

转移规则：
  input Send(q, m):
     sent[p, q, m] += 1
  output Receive(p, m):
     pre: rcvd[p, q, m] < sent[p, q, m]    // 不能接收未发送的消息
     rcvd[p, q, m] += 1                    // 不能重复接收同一条消息
```

**公平性条件（Fairness Condition）**：
- 如果 sent[p, q, m] → ∞（无限次发送），且 q 是正确的（未崩溃），则 rcvd[p, q, m] → ∞（无限次接收）

这确保了：只要坚持发送，消息最终一定会被接收——不是概率性的，而是**确定性的保证**。

### 4.2.3 公平丢包与伪消息的关系

公平丢包链路的"无伪造"性质保证了**链路不会凭空编造消息**——但它不保证"不会丢失消息"。丢失是指 sent > rcvd 的情况——这是链路特性允许的，算法需要通过重试来处理。

**丢包的两种原因**：
1. **缓冲区溢出**：发送方或接收方的缓冲区满，新消息被丢弃
2. **物理丢包**：无线干扰、交换机故障、光纤损坏

**公平性为什么成立？** 即使物理丢包率为 p（0 < p < 1），在无限次发送中，至少一次成功的概率趋近于 1。公平丢包不是"不丢包"——它是"丢失的概率趋近于 0"（当重试次数趋近于无穷时）。

### 4.2.4 为什么不能直接从公平丢包跳到可靠？

如果你自己在设计链路抽象——你可能想："公平丢包链路已经保证'最终会到达'，那我直接在它上面加序列号去重，不就能实现可靠链路了吗？为什么中间还需要顽固链路？"

**答案**：因为**公平丢包链路本身不负责重发**——它只说"如果无限次发送，最终会到达"，但没说"谁负责发送"。如果你直接在公平丢包上构建可靠链路，你的可靠链路算法**每次都必须自己实现重发逻辑**——这意味着每一个使用可靠链路的算法都把重发逻辑复制了一遍。

**顽固链路的作用**：**将重发逻辑集中到链路层本身**——上层算法不再需要关心"是否重发、多久重发一次"。上层只需调用 Send(m)，顽固链路保证最终送达。

**为什么 TCP 不需要顽固链路？** 因为 TCP 本身就在传输层做了重传——它相当于把公平丢包+重传整合成可靠链路。但在分布式算法中，你不能假设底层一定用 TCP——也可能用 UDP。所以需要在应用层重新实现重传逻辑。

**与网络协议栈的类比**：

| 网络层 | 分布式算法层 | 作用 |
|--------|------------|------|
| 物理介质（以太网/WiFi） | 公平丢包链路 | 最基础的消息传输 |
| IP 层 | 顽固链路 | 负责重传（但 IP 不保证可靠性） |
| TCP 层 | 可靠链路 + FIFO | 保证可靠+顺序 |
| — | 日志可靠链路 | TCP 没有的：崩溃恢复 |

## 4.3 顽固链路——确保对方收到

### 4.3.1 公平丢包的局限

公平丢包链路说"无限重发最终会到达"。但它没有说**谁负责重发**——是发送方自己重发？还是链路层自动重发？公平丢包链路本身不提供自动重发机制。

### 4.3.2 顽固链路的定义

**顽固链路（Stubborn Link）** 是在公平丢包链路之上，由**发送方主动承担重发责任**而构建的抽象。

| 性质 | 形式化描述 |
|------|-----------|
| **最终送达** | 如果发送方正确（未崩溃），且接收方正确，那么每条被发送的消息最终都会被接收方至少接收一次（可能有重复） |
| **无伪造** | 同公平丢包链路 |

### 4.3.3 算法——发送方持续重复发送

```
算法 4-1：顽固链路的发送方（基于公平丢包链路）

状态：
  pending = {(q, m)}  // 待发送队列——消息+目标进程

操作：
  upon Send(q, m) 发生:
    pending.add((q, m))  // 加入待发送队列

  持续执行（后台重发循环）:
    for each (q, m) in pending:
      调用 FairLossLink.Send(q, m)  // 通过公平丢包链路的 Send 操作发送
      // 注意：这里调用了多次 Send——公平丢包链路保证最终至少一次到达
```

**为什么叫"顽固"？** 因为发送方永远不放弃——只要 pending 队列里有消息，它就持续发送，直到消息被确认（但顽固链路不定义确认机制——确认在可靠链路中引入）。

**重发的代价**：网络带宽被占用——每条消息可能被发送成千上万次。这就是为什么后续的"可靠链路"进一步优化（引入确认机制，停止不必要的重发）。

**与 TCP 的对比**：TCP 的重传机制本质上也是"顽固"的——发送方重传直到收到 ACK。区别在于 TCP 在传输层实现，顽固链路在分布式算法层实现。

## 4.4 可靠链路——不丢不重复

### 4.4.1 顽固链路的局限

顽固链路保证"每条消息最终至少被接收一次"。但有两个问题：

1. **消息可能被重复接收**：公平丢包链路的重发可能导致同一条消息被接收多次
2. **发送方永远不停止重发**：没有确认机制，浪费网络带宽

### 4.4.2 可靠链路的定义

**可靠链路（Reliable Link）** 是最常用的链路抽象。它保证：

| 性质 | 形式化描述 |
|------|-----------|
| **可靠性（Validity）** | 如果发送方正确且接收方正确，每条被发送的消息最终被**恰好一次**接收（不丢不重复） |
| **无伪造** | 同前 |

### 4.4.3 算法——ACK确认 + 去重

```
算法 4-2：可靠链路（基于公平丢包链路）

发送方状态：
  pending = {(q, m, seq)}  // 待发送消息 + 序列号
  seq = 0                  // 序列号计数器

接收方状态：
  received_seq = 自发送方接收到的最大 seq  // 用于去重

操作：
  upon Send(q, m) 发生:
    seq += 1
    pending.add((q, m, seq))
    FairLossLink.Send(q, (DATA, seq, m))

  upon FairLossLink.Receive(p, (ACK, seq)) 发生:  // 收到确认
    if (p, _, seq) in pending:
      pending.remove((p, _, seq))  // 消息已被成功接收——停止重发

  upon FairLossLink.Receive(p, (DATA, seq, m)) 发生:  // 收到数据
    if seq > received_seq[p]:  // 新消息——去重
      received_seq[p] = seq
      输出 Receive(p, m)  // 交付给上层
    FairLossLink.Send(p, (ACK, seq))  // 发送确认

  持续执行（后台重发）:
    for each (p, m, seq) in pending:
      FairLossLink.Send(p, (DATA, seq, m))  // 重发未确认的消息
```

**关键机制**：
- **序列号去重**：接收方只接受 seq > received_seq 的消息——如果同一条消息被重传多次，只有第一个到达的被接收
- **ACK 确认**：接收方收到数据后立即发送 ACK——发送方收到 ACK 后停止重发。这解决了顽固链路"永远重发"的问题
- **不丢保证**：如果消息丢失（公平丢包链路丢包），发送方会重发（后台循环）→ 公平链路保证最终到达

### 4.4.4 完整时间线推演——发 3 次、丢 1 次、ACK 丢 1 次

假设 p 向 q 发送 1 条消息 m₁。seq=1。网络有 30% 丢包率。

```
T0: p 发送 (DATA, seq=1, m₁) 到 q（第1次发送）
T1: 消息丢失（30% 概率）——q 没收到
    p 的 pending 中 m₁ 未确认

T2: p 发送 (DATA, seq=1, m₁) 到 q（第2次重发，后台循环触发）
T3: q 收到 (DATA, seq=1, m₁)
    q 检查：seq=1 > received_seq[p]=0 → 新消息！
    q 记录：received_seq[p] = 1
    q 输出 Receive(p, m₁) 给上层
    q 发送 (ACK, seq=1) 到 p
T4: ACK 消息丢失（30% 概率）——p 没收到

T5: p 发送 (DATA, seq=1, m₁) 到 q（第3次重发——因为没收到 ACK）
T6: q 收到 (DATA, seq=1, m₁)
    q 检查：seq=1 > received_seq[p]=0 → 不对！seq == received_seq[p]
    → 不是新消息，是重复的！
    q 忽略 DATA，但仍发送 (ACK, seq=1) 到 p（帮助 p 停止重发）
T7: p 收到 (ACK, seq=1)
    p 删除 pending 中的 m₁——停止重发
    m₁ 投递完成 ✓
```

**关键观察**：
- T1 的消息丢失→ T2 重发——正如公平丢包保证的那样
- T3 q 接收了消息，但 T4 ACK 丢失→ T5 重发——即使 ACK 丢失，重发继续
- T6 q 检测到重复（seq 不增加）→ 忽略 DATA 但仍发 ACK——**帮助 p 停止重发**
- 从 p 的角度：消息丢失和 ACK 丢失都导致"重发"——直到 ACK 到达才停止（恰好一次接收）

**这个时间线展示了什么**：
- 可靠链路的"不丢"依赖于**发送方持续重发未确认的消息**（公平丢包保证最终到达）
- 可靠链路的"不重复"依赖于**接收方按 seq 去重**（即使消息被多次重发，也只接受一次）
- ACK 机制不仅确认接收，更重要的是**让发送方停止重发**——从"永远重发"变为"ACK后停止"

### 4.4.5 可靠链路的安全性证明

**要证明**：可靠链路满足"不丢不重复"的 Validity 性质。

**不重复证明**：接收方检查 seq > received_seq——如果同一条消息因公平丢包的重发而多次到达接收方，seq 相等条件阻止重复接受。

**不丢证明**：发送方持续重复发送未确认消息——公平丢包的公平性保证最终会有一条 (DATA, seq, m) 到达接收方——接收方检查 seq > received_seq → 接受 → 发��� ACK → 发送方停止重发。消息被恰好一次接收。

## 4.5 FIFO 可靠链路——保持顺序

### 4.5.1 可靠链路的局限

可靠链路保证"不丢不重复"——但**不保证顺序**。如果发送方发送 m1 然后 m2，可靠链路可能让 m2 先于 m1 到达（因为公平丢包可能导致 m1 丢失后重传，m2 却成功送达）。

### 4.5.2 FIFO 可靠链路的定义

**FIFO 可靠链路（FIFO Reliable Link）** 在可靠链路的基础上增加：

| 附加性质 | 形式化描述 |
|---------|-----------|
| **FIFO 顺序** | 如果发送方先发送 m1 再发送 m2，接收方会先接收 m1 再接收 m2 |

### 4.5.3 算法——按 seq 顺序交付

接收方不仅检查 seq > received_seq，还检查 **seq == received_seq + 1**——必须按严格升序接收。

```
接收方额外状态：
  next = 1  // 下一个期望的 seq

upon FairLossLink.Receive(p, (DATA, seq, m)):
  if seq == next:  // 恰好是下一个期望的序号——按序到达
    next += 1
    输出 Receive(p, m)
    发送 ACK(p, seq)
  else if seq > next:  // 序号高于期望——中间有消息丢失
    // 不接收——等待中间的消息（发送方会重发）
    发送 ACK(p, next - 1)  // 告诉发送方"我还在等 next，请重发之前的消息"
```

**为什么不像 TCP 一样缓冲？** TCP 的接收缓冲区可以存储 out-of-order 的消息（seq 大于期望值），等缺失的消息到达后重新排序。但在分布式算法中，如果消息长时间缺失（发送方故障），接收方不能无限等待——通过丢掉 out-of-order 消息，避免了"旧消息永远被阻塞"的问题。这是 FIFO 链路的工程选择。

## 4.6 日志可靠链路——容忍崩溃恢复

### 4.6.1 FIFO 链路的局限

如果接收方崩溃后恢复——它在崩溃期间缺失了哪些消息？FIFO 链路无法处理这种情况，因为它的 seq 状态（next 和 received_seq）在崩溃时丢失。

### 4.6.2 日志可靠链路的定义

**日志可靠链路（Logged Reliable Link）** 在 FIFO 可靠链路的基础上增加：

| 附加性质 | 形式化描述 |
|---------|-----------|
| **崩溃恢复** | 如果接收方崩溃后恢复，它仍然能接收到崩溃期间发送方发送的所有消息（按序，不丢不重复） |

### 4.6.3 算法——持久化 seq + 日志恢复

**关键改进**：接收方将 seq 状态（received_seq, next）持久化到**日志**（WAL）。崩溃后恢复时，从日志中恢复 seq 状态，然后发送方可以从崩溃前的断点继续发送。

```
接收方：
  upon 每次接收消息后:
    received_seq = seq
    next = seq + 1
    持久化到日志（WAL）包括: {received_seq, next, m}  // m 是接收的消息

  upon 崩溃恢复后:
    从日志恢复 {received_seq, next}
    向发送方发送 恢复通知(p, next)  // 告诉发送方 "从 next 开始重发"
```

**为什么需要持久化**？第3章 3.4.3 节讨论了崩溃恢复式失败——进程重启后丢失内存状态。如果不持久化 seq，接收方可能接收同一条消息两次或接受乱序消息——违反可靠性和 FIFO 属性。

## 4.7 各层链路的工程选择——什么时候用哪一层？

你作为系统设计者，需要根据业务场景选择最合适的链路抽象。用得太弱——上层算法无法正常工作。用得太强——浪费性能和复杂度。

| 业务场景 | 推荐链路 | 理由 |
|---------|---------|------|
| Leader 选举的心跳 | 公平丢包链路 | 心跳丢失后会自动重发——不需要确认机制加重负载 |
| 配置更新通知 | 可靠链路 | 配置必须被送达——不丢不重复；Acks 的开销可接受 |
| 分布式锁的锁请求 | FIFO 可靠链路 | 锁请求必须按序处理——先请求者先获得锁 |
| 数据库的同步写入 | 日志可靠链路 | 写入操作必须被持久化——即使接收方崩溃后也能恢复 |
| 尽力广播（Gossip 成员协议） | 公平丢包链路 | 广播丢失后由 Gossip 的周期性推送弥补——不需要可靠性 |
| 复制状态机的日志条目 | FIFO 可靠链路 | 日志必须按序、不丢不重复被复制——保证一致性 |

**选择原则**：只使用**你需要的保证**——不要为不需要的保证付出代价（消息延迟、内存开销、持久化 I/O）。

**与 TCP 的对应**：

| TCP 保证 | 分布式算法中的对应层 | 为什么要重新实现 |
|---------|-------------------|----------------|
| 可靠传输（不丢） | 可靠链路 | TCP 是进程间（IP:port），不是"分布式进程"间——进程崩溃后 TCP 连接断开 |
| 保序（FIFO） | FIFO 可靠链路 | 同上 |
| 流控/拥塞控制 | 不在链路层——在通信层 | TCP 处理了 |
| 崩溃恢复 | **日志可靠链路** | TCP**不处理**崩溃恢复——进程重启后 TCP 连接状态丢失，需要上层 WAL 恢复 |

## 4.8 本章总结

### 5 个核心要点

1. **分层设计是分布式算法的核心方法论**——在弱假设（公平丢包）上逐层构建强保证（不丢不重复 → 顺序 → 崩溃恢复）。每一层对外暴露相同的接口（Send/Receive），隐藏内部实现。

2. **公平丢包链路**是最基础的假设——无限重发最终会到达。这是最接近现实世界的模型（物理丢包率 >0，但不能永远丢包）。

3. **可靠链路 = 公平丢包 + 重传 + ACK确认 + 序列号去重**。TCP 在传输层做了同样的事，但分布式算法在应用层重新实现——因为需要处理 TCP 覆盖不到的场景（崩溃恢复、进程级确认）。

4. **FIFO 可靠链路 = 可靠链路 + 严格按序号接收**。接收方丢掉落序的乱序消息（不缓冲），让发送方重发——避免了"旧消息阻塞"的问题。

5. **日志可靠链路 = FIFO 可靠链路 + 持久化状态（WAL）**。崩溃恢复后从日志恢复 seq 继续接收——保证不丢、不重复、不逆序、不怕重启。

### 第5章预告

第5章将链路抽象应用到实际问题：**失败检测和选主**。失败检测器（Failure Detector）是在链路上的更高层抽象——它判断"进程是否存活"。基于失败检测，实现 Leader 选举。我们将看到：链路的时序假设（同步/异步/部分同步）如何决定失败检测的"完美性"。
