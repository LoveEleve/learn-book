## 1. 为什么需要容器化与 K8s？

### 1.1 手工运维的痛点

第一篇讲完架构演进后，部署了多节点集群、配置了负载均衡和健康检查。但这些都是手工操作——每加一个节点，运维人员需要手动配置一遍。

**手工运维的四大痛点**：

| 痛点 | 手工现状 | 期望 |
|------|---------|------|
| **部署** | 手动 SSH + 安装依赖 + 启动 | 一键部署 |
| **扩容** | 手动加机器 + 配置 LB | 自动扩容 |
| **故障恢复** | 人工发现 + 手动重启 | 自动检测 + 自动重启 |
| **环境一致性** | 开发环境 ≠ 测试环境 ≠ 生产环境 | 完全一致 |

### 1.2 容器化的解决思路

**容器的本质**：把应用及其所有依赖打包成一个可移植的镜像。在任何安装了 Docker 的机器上运行，环境完全一致。

```
传统部署：
  机器1（Ubuntu 18.04 + JDK 8）
  机器2（CentOS 7 + JDK 11）
   → 环境差异导致"在我机器上能跑"

容器化部署：
  所有机器：Docker
  容器内：Ubuntu 22.04 + JDK 17（完全一致）
   → 环境一致，100% 可复现
```

### 1.3 K8s 的解决思路

**K8s 的本质**：容器的编排平台。管理容器的部署、扩缩容、负载均衡、自愈。

**K8s 实现了第一篇和第二篇讲的所有机制——而且是自动化的**：

| 机制 | 手工实现（第一/二篇） | K8s 自动化 |
|------|---------------------|----------|
| 多副本部署 | 手动配置 Nginx upstream | Deployment replicas |
| 负载均衡 | 手动配置 LB | Service（自动） |
| 健康检查 | 手动配置健康检查脚本 | liveness/readiness probe |
| 故障恢复 | 人工发现+手动重启 | 自动重启故障 Pod |
| 滚动更新 | 手动替换节点+改配置 | Deployment rolling update |
| 灰度发布 | 手动改权重 | K8s + Istio（自动） |
| 扩容 | 手动加机器+配 LB | HPA 自动扩缩 |

## 2. Docker 核心概念

### 2.1 镜像（Image）

**Docker 镜像** 是应用及其所有依赖的只读模板。从镜像启动的实例称为容器。

**Dockerfile 示例**（Spring Boot 应用）：
```dockerfile
FROM eclipse-temurin:17-jre
COPY target/app.jar /app.jar
EXPOSE 8080
ENTRYPOINT ["java", "-jar", "/app.jar"]
```

构建：`docker build -t myapp:v1.0 .`

### 2.2 容器（Container）

**容器** 是镜像的运行实例。每个容器有独立的文件系统、网络和进程空间。

```
docker run -d -p 8080:8080 --name myapp myapp:v1.0
```

### 2.3 容器 vs 虚拟机

| 维度 | 虚拟机 | 容器 |
|------|--------|------|
| 启动速度 | 分钟级 | 秒级 |
| 资源占用 | GB 级（Guest OS） | MB 级（共享 Host OS 内核） |
| 隔离性 | 强（Hypervisor） | 中（Namespace + Cgroup） |
| 可移植性 | 需考虑 Hypervisor 兼容 | 任何 Docker 环境 |

### 2.4 容器网络

**Bridge 模式**：默认，容器通过虚拟网桥（docker0）与宿主机通信。

**Host 模式**：容器直接使用宿主机网络栈（`--network host`），性能最高，但端口不能冲突。

**多容器通信**：`docker-compose` 定义多个容器和服务依赖。

```yaml
# docker-compose.yml
version: '3'
services:
  app:
    build: .
    ports: ["8080:8080"]
    depends_on:
      - redis
  redis:
    image: redis:7
    ports: ["6379:6379"]
```

### 2.5 容器存储

**Volume**：宿主机目录挂载到容器内，容器删除后数据保留。

```
docker run -v /data/mysql:/var/lib/mysql mysql:8
```

## 3. K8s 架构与核心组件

### 3.1 Master 节点（控制平面）

K8s 集群的控制中心，管理所有 Node 和 Pod。

```
┌──────────────────────────────┐
│         K8s Master            │
│  ┌────────────────────────┐  │
│  │  API Server（REST API） │  │ ← 所有操作的入口
│  ├────────────────────────┤  │
│  │  etcd（集群状态存储）    │  │ ← 基于 Raft 的强一致 KV
│  ├────────────────────────┤  │
│  │  Controller Manager     │  │ ← 控制循环（保持期望状态）
│  ├────────────────────────┤  │
│  │  Scheduler              │  │ ← 调度 Pod 到 Node
│  └────────────────────────┘  │
└──────────────────────────────┘
```

**关键组件**：

| 组件 | 作用 | 与前面书的关联 |
|------|------|--------------|
| **API Server** | 提供 REST API，所有操作通过它 | — |
| **etcd** | 存储集群所有状态（基于 Raft） | 释慧利第 6 章 Raft |
| **Controller Manager** | 控制循环，确保实际状态 = 期望状态 | — |
| **Scheduler** | 把 Pod 调度到合适的 Node | 负载均衡的思路 |

### 3.2 Node 节点（工作节点）

```
┌──────────────────────────────┐
│         K8s Node              │
│  ┌────────────────────────┐  │
│  │  kubelet（节点代理）     │  │ ← 管理本节点 Pod
│  ├────────────────────────┤  │
│  │  kube-proxy（网络代理） │  │ ← 管理 Service 负载均衡
│  ├────────────────────────┤  │
│  │  Container Runtime     │  │ ← Docker/containerd
│  └────────────────────────┘  │
└──────────────────────────────┘
```

### 3.3 Pod

**Pod** 是 K8s 的最小调度单位。一个 Pod 包含一个或多个共享网络和存储的容器。

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: myapp
spec:
  containers:
  - name: app
    image: myapp:v1.0
    ports:
    - containerPort: 8080
```

**Pod 的生命周期**：
- Pending → Running → Succeeded / Failed
- Pod 不是长期稳定的——Node 故障 → Pod 被重新调度到健康 Node

### 3.4 Deployment

**Deployment** 管理 Pod 的副本数、滚动更新和回滚。

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 3                    # 3 副本（第一篇讲的冗余）
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: app
        image: myapp:v1.0
        ports:
        - containerPort: 8080
```

**Deployment 实现了第一篇讲的核心机制**：
- `replicas: 3` → 冗余（第一篇 第 2 节多副本）
- 滚动更新 → 灰度发布（第一篇 第 7 节）
- 自愈 → 故障恢复（第一篇 第 3 节熔断的上游）

## 4. K8s 高可用机制

### 4.1 自愈（Self-Healing）

**K8s 的容错本质**：不是"保证 Pod 不故障"，而是"Pod 故障后自动恢复"。

**自动重启**（liveness probe）：
```yaml
livenessProbe:
  httpGet:
    path: /health
    port: 8080
  initialDelaySeconds: 30   # 容器启动后 30 秒开始检查
  periodSeconds: 10          # 每 10 秒检查一次
```

Pod 健康检查失败 → K8s 自动重启容器。

**就绪检查**（readiness probe）：
```yaml
readinessProbe:
  httpGet:
    path: /ready
    port: 8080
  initialDelaySeconds: 10
  periodSeconds: 5
```

Pod 就绪检查通过后，才加入 Service 的负载均衡。

**Node 故障自动迁移**：
Node 宕机 → Master 标记 Node 不可用 → 其上的 Pod 被调度到健康 Node。

### 4.2 滚动更新（Rolling Update）

**Deployment 更新策略**：逐步替换旧版本 Pod 为新版本。

```yaml
spec:
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1          # 最大超出 replicas 的 Pod 数
      maxUnavailable: 0    # 最大不可用 Pod 数（0=全部可用期间更新）
```

**更新过程**：3 副本，`maxSurge=1, maxUnavailable=0`
```
步骤 1：创建 1 个新 Pod（总数 4）  → 等待新 Pod Ready
步骤 2：删除 1 个旧 Pod（总数 3）   → 等待
步骤 3：创建 1 个新 Pod（总数 4）   → 等待新 Pod Ready
步骤 4：删除 1 个旧 Pod（总数 3）   → 等待
步骤 5：创建 1 个新 Pod（总数 4）   → 等待新 Pod Ready
步骤 6：删除 1 个旧 Pod（总数 3）   → 完成
```

**对比手工运维**：手工替换需要人工监控旧节点、配 LB、验证新节点。K8s 全部自动化。

### 4.3 自动扩缩容（HPA）

**HPA**（Horizontal Pod Autoscaler）：根据 CPU/内存使用率自动调整 Pod 数量。

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 60
```

**HPA 的自动扩缩逻辑**：
```
CPU 使用率 > 60% → metrics-server 报告 → HPA 触发扩容
  → Deployment replicas 从 3 增加到 4
  → Scheduler 分配新 Pod 到 Node
  → Service 自动加入新 Pod

CPU 使用率 < 30% → HPA 触发缩容
  → Deployment replicas 从 4 减少到 3
  → 随机终止一个 Pod
```

**对比手工扩容**：手工扩容需要人工监控 → 判断要不要加 → 加机器 → 配 LB → 验证。K8s HPA 全程自动化。

### 4.4 调度策略

**K8s Scheduler 的决策**：新 Pod 应该放到哪个 Node 上？

**调度考量**：
- 资源：Node 有足够的 CPU/内存？
- 亲和性：Pod 倾向和哪些 Pod 在一起（或分开）？
- 污点：Node 是否打上了"不接受普通 Pod"的标记？

**跨可用区调度**（高可用核心）：
```yaml
affinity:
  podAntiAffinity:
    requiredDuringSchedulingIgnoredDuringExecution:
    - labelSelector:
        matchLabels:
          app: myapp
      topologyKey: topology.kubernetes.io/zone
```
强制 Pod 分布到不同可用区（AZ）。避免所有 Pod 都在同一个 AZ。

## 5. 服务发现与负载均衡

### 5.1 Service

**Service** 为 Pod 提供稳定的访问入口。Pod IP 会变（重启/调度后），Service IP 保持不变。

```yaml
apiVersion: v1
kind: Service
metadata:
  name: myapp
spec:
  selector:
    app: myapp
  ports:
  - port: 8080
    targetPort: 8080
  type: ClusterIP
```

**Service 类型**：

| 类型 | 访问范围 | 用途 |
|------|---------|------|
| **ClusterIP** | 集群内部 | 服务间调用 |
| **NodePort** | 节点 IP:端口 | 外部访问（测试） |
| **LoadBalancer** | 云厂商 LB | 生产外部访问 |

**Service 实现了第一篇和第二篇的核心机制**：
- Service 选择器 = 健康检查后自动加入的节点（第一篇 第 3 节 健康检查）
- Service IP 不变 = 服务发现（第一篇 第 2 节 LB）
- 负载均衡分发 = 第二篇 第 2 节（负载均衡算法，kube-proxy 默认 iptables 随机）

### 5.2 Ingress

**Ingress** 为 HTTP/HTTPS 流量提供路由规则。

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: myapp-ingress
spec:
  rules:
  - host: myapp.example.com
    http:
      paths:
      - path: /api
        pathType: Prefix
        backend:
          service:
            name: myapp
            port:
              number: 8080
```

**Ingress vs Service**：
- Service：L4 负载均衡（TCP/UDP）
- Ingress：L7 路由（HTTP/HTTPS 路径/域名）

### 5.3 K8s DNS

K8s 内置 DNS 服务（CoreDNS），为 Service 提供域名解析。

```
Service Name: myapp
Namespace: default
DNS：myapp.default.svc.cluster.local → Service ClusterIP
```

Pod 内只需访问 `myapp` 即可——K8s DNS 自动补齐 `default.svc.cluster.local`。

## 6. 灰度与金丝雀发布

### 6.1 K8s 原生灰度

**方式 1：多 Deployment + 共享 Service**
```yaml
# 稳定版本
Deployment: myapp-stable (replicas: 9)

# 灰度版本
Deployment: myapp-canary (replicas: 1)

# 共享 Service
Service: myapp (selector: app=myapp)  # 匹配两个 Deployment
```
10 个 Pod：9 个稳定版本 + 1 个灰度版本 → 10% 流量到灰度。

**方式 2：Ingress 注解实现灰度**
```yaml
nginx.ingress.kubernetes.io/canary: "true"
nginx.ingress.kubernetes.io/canary-weight: "10"
```
10% 流量到灰度 Service。

### 6.2 Istio 灰度

**Istio** 是 Service Mesh 的实现。通过 sidecar 代理（Envoy）管理所有流量。

**Istio VirtualService 灰度**：
```yaml
apiVersion: networking.istio.io/v1beta1
kind: VirtualService
spec:
  http:
  - route:
    - destination:
        host: myapp
        subset: v1
      weight: 90              # 90% 流量到 v1
    - destination:
        host: myapp
        subset: v2
      weight: 10              # 10% 流量到 v2
```

**Istio 灰度的优势**：
- 基于 Header（如 VIP 用户先体验新版本）
- 基于比例（不用改 Deployment replicas）
- 自动回滚（错误率超阈值 → 自动切回旧版本）

**K8s 原生灰度 vs Istio 灰度**：
| 维度 | K8s 原生 | Istio |
|------|---------|-------|
| 灰度粒度 | Pod 数量比例 | 流量比例 |
| 条件路由 | 支持（Ingress canary 注解，需 NGINX Ingress Controller） | 支持（基于 Header/Cookie/Query） |
| 自动回滚 | 不支持 | 支持（错误率监控） |
| 复杂度 | 低 | 高（需部署 Istio） |

## 结语：K8s 是自动化的高可用

回顾本篇，K8s 把前两篇的手工运维全面自动化：

**第一篇（架构演进+网络）→ K8s 自动化**：
- 多副本部署：`replicas: 3`
- 健康检查：`livenessProbe` + `readinessProbe`
- 负载均衡：Service + Ingress
- 故障恢复：自动重启 + Node 迁移

**第二篇（负载均衡+锁+选主）→ K8s 自动化**：
- 负载均衡：kube-proxy（iptables/IPVS）
- 分布式选主：etcd Leader Election（K8s 自身就基于 etcd）
- 自动扩缩容：HPA

**本篇的核心论断**：K8s 不是新的理论——它是第一篇和第二篇所有高可用机制的自动化实现。理解了前两篇的手工运维，再学 K8s 就会明白"K8s 替我们做了什么"。云原生的本质是：**用声明式配置替代手工运维**——告诉 K8s "我要 3 个副本"（期望状态）而不是"先 SSH 到机器 1，启动，再 SSH 到机器 2..."（操作步骤）。K8s 的 Controller Manager 持续对比实际状态与期望状态，自动修复所有偏差。
