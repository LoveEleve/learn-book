## 7. K8s 架构——高可用的设计哲学

### 7.1 从人工运维到自动化高可用

第1-6章的高可用方案有一个前提：**需要人为介入**。负载均衡故障 → 需要运维切流量。应用崩溃 → 需要重启。这些操作可以自动化——这就是 K8s 的设计动机。

**K8s 的核心哲学——声明式 + 控制器**：

```
用户定义"期望状态"（YAML）→ K8s 控制器持续调谐 → 实际状态 = 期望状态
```

你不再需要手动重启崩溃的 Pod——K8s 的 ReplicaSet 控制器检测到 Pod 数量不足，自动创建新 Pod。这是"自动化高可用"的本质。

### 7.2 控制平面——高可用的"大脑"

```
┌─────────────────────────────────────────────────────────┐
│                Control Plane (控制平面)                    │
│  ┌────────┐  ┌──────────┐  ┌──────────┐  ┌──────────┐  │
│  │  API   │  │Controller│  │Scheduler │  │   etcd   │  │
│  │ Server │  │ Manager  │  │          │  │  (Raft)  │  │
│  └────────┘  └──────────┘  └──────────┘  └──────────┘  │
└─────────────────────────────────────────────────────────┘
                          │
┌─────────────────────────────────────────────────────────┐
│                 Worker Nodes (工作节点)                   │
│  ┌──────────┐  ┌──────────┐  ┌──────────┐              │
│  │  kubelet │  │  kubelet │  │  kubelet │              │
│  │  Pod×N   │  │  Pod×N   │  │  Pod×N   │              │
│  └──────────┘  └──────────┘  └──────────┘              │
└─────────────────────────────────────────────────────────┘
```

**控制平面组件如何保证自身高可用**：
- **API Server**：无状态 → 多副本 + 负载均衡
- **Controller Manager / Scheduler**：通过 Lease 机制选主——同一时刻只有一个 Active，故障后自动切换
- **etcd**：Raft 共识——3 或 5 节点集群，任何少数节点故障不影响

### 7.3 控制器模式——"调谐循环"如何实现自愈

```
Reconcile Loop (调谐循环):

  for {
    当前状态 = 获取集群中实际运行的 Pod 数量和状态
    期望状态 = 读取 Deployment YAML 中的 replicas 值
    
    if 当前状态 ≠ 期望状态:
      执行操作（创建 Pod、删除多余 Pod、重启不健康 Pod）
    
    sleep(控制循环间隔)
  }
```

**四种关键控制器**：

| 控制器 | 管理什么 | 高可用保证 |
|--------|---------|-----------|
| **ReplicaSet** | Pod 数量 = replicas 值 | 崩溃 Pod 自动重建 |
| **Deployment** | 滚动更新（旧 Pod → 新 Pod） | 升级过程中保证可用 Pod≥replicas |
| **StatefulSet** | 有状态 Pod（固定名称、持久化卷） | Pod 重启后数据不丢失 |
| **DaemonSet** | 每个节点一个 Pod（如日志采集） | 新节点自动部署 |

## 8. Pod 的高可用——从启动到自愈

### 8.1 Pod 的生命周期

```
Pending → Running → Succeeded/Failed
              │
              ▼
         CrashLoopBackOff (崩溃循环退避)
```

**自愈机制**——三种探针：

| 探针 | 检查什么 | 失败后果 |
|------|---------|---------|
| **livenessProbe** | "容器还活着吗？" | 失败 → 重启容器 |
| **readinessProbe** | "容器准备好接收流量了吗？" | 失败 → 暂时从 Service 摘除 |
| **startupProbe** | "容器启动完成了吗？" | 失败 → 等待，不触发 livenessProbe |

**区别——为什么需要三个探针？**

- 启动阶段容器需要不同判定标准（JVM 预热、DB 连接池建立）——startupProbe 隔离了启动慢于存活检查的冲突
- 一个容器可以存活但不健康的业务状态（如内存高水位、线程池满）——readinessProbe 主动摘除避免雪崩
- livenessProbe 只判定最严重的故障（进程僵死、无响应）

### 8.2 资源限制——防止雪崩

```
resources:
  requests:      # 最低保证资源（调度依据）
    cpu: 500m
    memory: 512Mi
  limits:        # 最高限制（防止失控）
    cpu: 1000m
    memory: 1Gi
```

**关键**：如果没有 limits，一个 Pod 的内存泄漏可以 OOM Kill 节点上的所有 Pod——雪崩。

**三种 QoS 级别**：

| QoS | 条件 | OOM 优先级 |
|-----|------|-----------|
| Guaranteed | requests = limits | 最后被 Kill |
| Burstable | requests < limits | 中间 |
| BestEffort | 无 requests/limits | 最先被 Kill |

## 9. 服务发现——Pod 漂移后的自动路由

### 9.1 问题：Pod IP 不固定

Pod 重启后 IP 变化——你不能用固定 IP 访问服务。K8s Service 通过 Label Selector 自动汇聚 Pod：

```
Service: "user-service"  (固定 ClusterIP: 10.96.0.1)
  │
  ├── Pod(user-api-abc123, IP: 10.244.1.5) ← Label: app=user
  ├── Pod(user-api-def456, IP: 10.244.2.3) ← Label: app=user
  └── Pod(user-api-ghi789, IP: 10.244.3.7) ← Label: app=user
```

Service 自动检测 Label 匹配的 Pod → 新 Pod 加入，旧 Pod 摘除 → 客户端通过 ClusterIP 访问，无需关心 Pod IP。

### 9.2 kube-proxy——流量的自动调度

kube-proxy 是每个节点上的代理，负责将 Service 流量转发到后端 Pod：

```
用户代码 → Service ClusterIP (10.96.0.1) → kube-proxy iptables/IPVS → Pod IP (10.244.1.5)
```

**iptables 模式的局限**：O(N) 规则匹配（N = Service 数量 × 后端 Pod 数）。大规模集群（>1000 Service）性能下降。

**IPVS 模式**：Linux 内核的负载均衡模块——O(1) 哈希查找。从 K8s 1.11 开始可用，生产推荐。

## 10. 云原生的灰度与流量治理

### 10.1 Ingress——七层路由的入口

Service 是四层转发（不检查 HTTP 内容）。Ingress 是七层——按域名/路径路由：

```
api.example.com/api/user → user-service
api.example.com/api/order → order-service
```

**高可用保证**：Ingress Controller（如 Nginx Ingress）本身是一个 Deployment → 多副本 + 负载均衡 → 故障 Pod 自动替换。

### 10.2 灰度发布——控制风险

```
Deployment:
  replicas: 10
  新版本 v2: 2 个 Pod (20% 流量)
  旧版本 v1: 8 个 Pod (80% 流量)
```

逐步增加 v2 的比例 → 观察监控指标（错误率、延迟）→ 如果正常，继续扩大 v2 → 回滚到 v1 如果异常。

**为什么高可用？** 灰度发布将"全量升级风险"拆解为"逐步验证"——即使新版本有 bug，最多影响 20% 的用户。

## 10.3 本章总结

1. **K8s 的控制平面是高可用的设计**——API Server 无状态多副本，etcd 用 Raft 共识。
2. **控制器模式实现自愈**——ReplicaSet 保证 Pod 数量，Deployment 保证滚动更新。
3. **三种探针**：liveness (存活)、readiness (流量)、startup (启动)——各司其职，共同保证 Pod 的高可用。
4. **Service 自动路由**——Pod IP 变化不影响客户端访问。kube-proxy IPVS 模式是生产推荐。
5. **灰度发布**——逐步验证新版本，控制风险——是云原生高可用部署的基石。
