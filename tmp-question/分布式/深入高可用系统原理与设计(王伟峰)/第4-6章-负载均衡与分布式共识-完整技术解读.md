## 1. 为什么需要负载均衡？

### 1.1 从单点到多节点的必然

第一篇讲了架构演进：单体→集群→微服务。多节点的前提是"每个节点不能闲着"——必须有一种机制把请求均匀分发到各节点。这就是负载均衡。

**没有负载均衡时**：所有请求打到一个节点，其他节点闲着。从多节点的角度看，这是巨大的资源浪费。

**有负载均衡时**：请求根据策略分发，多节点并行处理，整体吞吐 = 单节点吞吐 × 节点数。

### 1.2 负载均衡的本质

负载均衡的回答是：**在请求到达多节点集群时，应该把每个请求分给哪个节点处理？**

但这个选择非常复杂——需要考虑的因素很多：
- 目标：各节点负载尽可能均匀
- 约束：同一用户的请求需要发到同一节点吗？
- 失败：某个节点故障怎么办？
- 权重：如果各节点的处理能力不同，权重如何分配？

### 1.3 负载均衡的位置

```
              ┌─────────────────┐
              │  全局负载均衡    │  ← 用户→最近的数据中心
              └────────┬────────┘
                       │
              ┌────────┴────────┐
              │  服务端负载均衡  │  ← 数据中心内各节点间的分发
              └────────┬────────┘
                       │
              ┌────────┴────────┐
              │  客户端负载均衡  │  ← 服务消费者侧的路由
              └─────────────────┘
```

**本文重点**：服务端负载均衡（数据中心内部的请求分发）。客户端负载均衡（如 Spring Cloud LoadBalancer）和全局负载均衡（如 DNS GTM）不在本文范围。

## 2. 负载均衡算法

### 2.1 轮询（Round Robin）

**核心思想**：依次把请求循环分配到各节点。

```
请求1 → 节点1
请求2 → 节点2
请求3 → 节点3
请求4 → 节点1
...
```

**优势**：实现最简单，各节点的请求数严格相等。

**劣势**：假设所有节点的处理能力相同。如果节点 1 处理能力差但请求数一样多，它会成为瓶颈。

**工程实现**（Nginx 默认）：
```nginx
upstream backend {
    server 192.168.1.1:8080;
    server 192.168.1.2:8080;
    server 192.168.1.3:8080;
}
```

### 2.2 加权轮询（Weighted Round Robin）

**核心思想**：给不同节点分配不同权重，能力强的节点分配更多请求。

```
节点1: weight=5
节点2: weight=3
节点3: weight=2

10 个请求：节点1(5)+节点2(3)+节点3(2)
```

**优势**：各节点按权重分配请求，充分发挥异构硬件的能力。

**劣势**：权重是静态的（不会根据当前负载动态调整）。

**工程实现**（Nginx）：
```nginx
upstream backend {
    server 192.168.1.1:8080 weight=5;
    server 192.168.1.2:8080 weight=3;
    server 192.168.1.3:8080 weight=2;
}
```

**权重的另一种实现**（加权随机）：
- Alibaba Dubbo 和 Spring Cloud 的随机负载均衡通常支持加权
- 各节点被选中的概率与其权重成比例

### 2.3 最少连接（Least Connections）

**核心思想**：选择当前活跃连接数最少的节点——连接数可以反映当前负载。

```
请求到达 → 检查各节点当前连接数 → 选连接数最少的
```

**优势**：动态根据当前负载分配，对长连接场景效果好。

**劣势**：每个请求都需要查询连接数，有一定开销。对短连接场景无明显优势。

**工程实现**（Nginx）：
```nginx
upstream backend {
    least_conn;
    server 192.168.1.1:8080;
    server 192.168.1.2:8080;
    server 192.168.1.3:8080;
}
```

### 2.4 一致性哈希（Consistent Hashing）

**核心思想**：对请求的某个 key（如用户 ID）做哈希，映射到哈希环上的节点。同一 key 的请求总是路由到同一节点。

```
请求：hash(user_id) % 2^32  →  在哈希环上找到顺时针第一个节点
```

**为什么需要一致性哈希？**
- 场景：用户的 WebSocket 长连接绑定到特定节点
- 如果用户断了重连，路由到其他节点，之前的连接状态丢失
- 一致性哈希保证用户总是连接同一个节点（除非该节点故障）

**虚拟节点**：为每个物理节点创建多个虚拟节点，避免哈希环上的分布不均。

**优势**：
- 同一用户总到同一节点（适合有状态的场景）
- 节点变动时只有少量 key 重新映射

**劣势**：
- 不适合无状态的短连接场景（不需要 sticky session）
- 初始时可能分布不均（虚拟节点可缓解）

### 2.5 权重响应时间（Weighted Response Time）

**核心思想**：根据各节点的响应时间动态调整权重——响应快的节点分配更多请求。

```
初始权重：全部相等
定期更新：weight_new = weight_old × (avg_rt / node_rt)
  - node_rt 低于平均 → weight 增加（这个节点快，分配更多请求）
  - node_rt 高于平均 → weight 减少（这个节点慢，分流到其他节点）
```

**优势**：自动适应节点性能变化（GC 暂停/慢查询导致某节点变慢时自动降低权重）。

**劣势**：实现复杂，需要持续监控各节点的响应时间。

**Nginx Plus 的动态权重更新**：
```nginx
upstream backend {
    zone backend 64k;
    server 192.168.1.1:8080 weight=5;
    server 192.168.1.2:8080 weight=3;
    server 192.168.1.3:8080 weight=2;
}
```
- `zone` 定义共享内存，多个 worker 进程共享连接数统计

**历史参考（Netflix Ribbon，已过时）**：
- `WeightedResponseTimeRule` 是 Ribbon 中的算法——每 30 秒根据各节点平均响应时间重新计算权重
- **注**：Spring Cloud 已移除 Ribbon，当前替代为 Spring Cloud LoadBalancer（默认提供 RoundRobin 和 Random）

### 2.6 五者对比

| 算法 | 公平性 | 适合场景 | 复杂度 | 对有状态支持 |
|------|--------|---------|--------|------------|
| 轮询 | 绝对公平（请求数） | 无状态短连接 | 最低 | 无 |
| 加权轮询 | 权重公平 | 异构硬件 | 低 | 无 |
| 最少连接 | 连接数公平 | 长连接 | 中 | 部分 |
| 一致性哈希 | 哈希分布 | 有状态/缓存命中 | 高 | 完全 |
| 权重响应时间 | 性能公平 | 节点性能波动 | 最高 | 无 |

### 2.7 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 第 3 章 5.4 节 一致性哈希——负载均衡与分区的数学基础 |
| **Unmesh** | 模式 | 第二篇 一致性哈希模式是负载均衡中"最小迁移"的核心思想 |
| **释慧利** | 算法 | — |

### 2.8 本节小结

五种负载均衡算法：
- **轮询/加权轮询**：最简单，适合无状态服务
- **最少连接**：长连接场景效果好
- **一致性哈希**：有状态场景（WebSocket/本地缓存）的选择
- **权重响应时间**：节点性能波动时的自动调权

## 3. 分布式锁

### 3.1 模式概述

**核心问题**：多个节点同时竞争一个资源时，如何保证同一时刻只有一个节点能操作？

**场景**：
- 定时任务：多个节点同时执行定时任务，应只有一个节点真正执行
- 库存扣减：防止超卖
- 幂等性：防止重复处理

**单机锁的局限**：单机的 `synchronized` / `ReentrantLock` 无法跨 JVM。

### 3.2 分布式锁的四个要求

**要求 1：互斥性**
- 同一时刻只有一个客户端能持有锁
- 其他客户端尝试获取锁时返回失败或等待

**要求 2：防死锁**
- 客户端崩溃时锁必须能释放
- 通常用超时机制（锁有 TTL）
- 获取锁的客户端如果崩溃，锁在 TTL 后自动释放

**要求 3：可重入**
- 同一客户端在持有锁期间可以再次获取
- 类似 Java 的 `ReentrantLock`

**要求 4：高可用**
- 锁服务本身不能是单点
- Redis 哨兵/集群、ZooKeeper 集群、etcd 集群都可以作为锁服务

### 3.3 Redisson 分布式锁（基于 Redis）

**Redisson** 是 Java 最常用的分布式锁实现，基于 Redis。

**获取锁**：
```java
RLock lock = redisson.getLock("order:lock:123");
lock.lock();          // 阻塞等待，直到获取锁
// 业务逻辑
lock.unlock();
```

**底层实现**（Lua 脚本保证原子性）：
```lua
-- 如果锁不存在，创建锁并设置持有者
if (redis.call('exists', KEYS[1]) == 0) then
    redis.call('hincrby', KEYS[1], ARGV[2], 1);
    redis.call('pexpire', KEYS[1], ARGV[1]);
    return nil;
end;
-- 如果锁存在且属于当前线程，可重入计数+1
if (redis.call('hexists', KEYS[1], ARGV[2]) == 1) then
    redis.call('hincrby', KEYS[1], ARGV[2], 1);
    redis.call('pexpire', KEYS[1], ARGV[1]);
    return nil;
end;
-- 否则返回锁的剩余TTL
return redis.call('pttl', KEYS[1]);
```

**Redisson 的锁续期（Watchdog）**：
- 锁默认 TTL = 30 秒
- 如果业务逻辑执行超过 30 秒，Watchdog 自动续期（每 10 秒续到 30 秒）
- 业务逻辑完成 → `unlock()` → 停止续期
- 客户端崩溃 → Watchdog 停止 → 锁在 30 秒后自动释放

**Redisson 的可重入性**：
- Redisson 锁用 Redis Hash 实现
- Hash Key = 锁名，Hash Field = 线程 ID，Hash Value = 重入计数
- 同一线程再次 `lock()` → 计数+1
- `unlock()` → 计数-1，计数为 0 时真正释放锁

### 3.4 ZooKeeper 分布式锁

**核心思想**：利用 ZooKeeper 的临时顺序节点实现锁。

**获取锁的流程**：
```
1. 每个客户端在 /locks 下创建一个临时顺序节点
   client1 → /locks/lock-0000000001
   client2 → /locks/lock-0000000002
   client3 → /locks/lock-0000000003

2. 客户端获取 /locks 下所有子节点，按序号排序
3. 如果自己的节点序号最小 → 获取锁成功
4. 否则 → 对前一个节点的删除事件设置 Watch
   如 client3 Watch lock-0000000002 的删除事件
5. 前一个节点删除（释放锁）→ Watch 触发 → client3 成为序号最小 → 获取锁
```

**ZooKeeper 锁的优势**：
- 自动释放：客户端断开连接 → 临时节点自动删除 → 锁自动释放
- 公平锁：按节点序号排队，先到先得
- 通知机制：Watch 触发，不轮询

**ZooKeeper 锁的劣势**：
- 性能低：每次获取锁需要操作 ZK（ZAB 多数派写）
- 写吞吐低：ZK 写吞吐远低于 Redis
- 适合低并发场景（如分布式选主），不适合高并发锁（如库存扣减）

### 3.5 etcd 分布式锁

**实现原理类似 ZooKeeper**：
```java
// etcd lease + transaction
Lease lease = client.createLease(30);  // TTL=30s
client.lock("order:lock:123", lease);  // 如果锁存在则阻塞等待
// 业务逻辑
client.unlock("order:lock:123");
```

**etcd 与 ZooKeeper 锁的对比**：
| 维度 | ZooKeeper | etcd |
|------|-----------|------|
| 锁机制 | 临时顺序节点 + Watch | 租约 + 事务 |
| 续期 | — | KeepAlive（续租） |
| 性能 | 低（ZK 写吞吐低） | 低（Raft 多数派） |
| 生态 | Curator Recipe 封装 | jetcd 库 |

### 3.6 Redisson vs ZooKeeper vs etcd 锁选型

| 维度 | Redisson（Redis） | ZooKeeper | etcd |
|------|-----------------|-----------|------|
| 性能 | 高（单机 10 万 QPS） | 低（数千 QPS） | 低（数千 QPS） |
| 一致性 | AP（Redis 哨兵模式） | CP | CP |
| 可靠性 | 中（Redis 异步复制可能丢锁） | 高（ZAB 多数派） | 高（Raft 多数派） |
| 自动释放 | TTL + Watchdog | 临时节点 | 租约 |
| 适用 | 高并发（Redis Cluster） | 低并发（选主） | 低并发（K8s 生态） |

**生产选型原则**：
- **高并发库存扣减**：Redisson（Redis Cluster）
- **分布式定时任务**：Redisson 或 XXL-Job 内置锁
- **高可靠性选主**：ZooKeeper 或 etcd
- **K8s 环境**：etcd（原生支持）

### 3.7 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | — |
| **释慧利** | 算法 | ZK 临时顺序节点 + Watch 的锁机制是 ZAB 共识的工程应用 |
| **Unmesh** | 模式 | — |

### 3.8 本节小结

分布式锁的三种实现：
- **Redisson**（Redis）：高性能，AP，适合高并发
- **ZooKeeper**：高可靠，CP，适合低并发高可靠
- **etcd**：高可靠，CP，K8s 生态首选

## 4. 分布式选主

### 4.1 模式概述

**核心问题**：多节点集群中，如何选出一个 Leader 负责协调所有节点？

**场景**：
- 分布式定时任务：多节点部署时，定时任务只应在一个节点上执行
- 主从架构：数据库主从复制的主节点选举
- 协调者：分布式事务 2PC 的协调者选举

**分布式选主与分布式锁的区别**：
- 分布式锁：短暂的互斥访问（持有锁→操作→释放锁）
- 分布式选主：长期的角色分配（选举→持有 Leader 角色→心跳维持→故障切换）

### 4.2 ZooKeeper 选主

**核心思想**：利用 ZooKeeper 的临时顺序节点实现选主——和分布式锁几乎相同的机制，但 Leader 持有节点的时间更长，通过心跳维持。

```
1. 每个候选者在 /election 下创建临时顺序节点
   node1 → /election/leader-0000000001
   node2 → /election/leader-0000000002
   node3 → /election/leader-0000000003

2. 序号最小的节点成为 Leader
3. Leader 维持心跳（ZooKeeper 会话）
4. Leader 故障 → 临时节点删除 → Watch 通知 → 序号次小的节点成为新 Leader
```

**Curator LeaderSelector 示例**：
```java
LeaderSelectorListener listener = new LeaderSelectorListenerAdapter() {
    public void takeLeadership(CuratorFramework client) throws Exception {
        // 当前节点成为 Leader
        while (true) {
            doLeaderWork();      // 执行 Leader 工作
            // 注：心跳由 Curator 框架的 ZooKeeper 会话自动管理，sleep 仅为限速示例
            Thread.sleep(1000);
        }
    }
};
LeaderSelector selector = new LeaderSelector(client, "/election", listener);
selector.autoRequeue();  // 失去 Leader 后自动重新参与选举
selector.start();
```

### 4.3 etcd 选主

**核心思想**：利用 etcd 的租约 + 事务实现选主。

```java
// etcd election
Election election = client.createElection("election");
election.campaign("node1");  // 参与选举，选出后阻塞直到成为 Leader
// 成为 Leader 后执行业务逻辑
election.resign();           // 主动退出
```

**etcd 选主的特点**：
- 租约 TTL 控制心跳间隔（类似 ZooKeeper 的会话超时）
- Leader 定期续租（KeepAlive）
- Leader 故障 → 租约过期 → 重新选举

### 4.4 基于数据库的选主

**适用场景**：没有 ZooKeeper/etcd 等专用协调服务时，可以用数据库作为过渡方案。

**实现**：利用数据库的 `INSERT` + 唯一约束。

```sql
-- 候选者尝试插入自己的记录
INSERT INTO leader_election (lock_name, leader_id, expire_time)
VALUES ('order-scheduler', 'node1', NOW() + INTERVAL 30 SECOND)
ON DUPLICATE KEY UPDATE
    leader_id = IF(expire_time < NOW(), VALUES(leader_id), leader_id),
    expire_time = IF(expire_time < NOW(), VALUES(expire_time), expire_time);
```

**原理**：
1. `INSERT` 到有唯一约束的表
2. 插入成功 → 获得 Leader 角色
3. Leader 定期更新 `expire_time`（续期）
4. Leader 故障 → `expire_time` 过期 → 新节点可以抢占

**优势**：不引入额外组件
**劣势**：性能低，依赖数据库的可用性

### 4.5 选主方案的选型

| 方案 | 性能 | 可靠性 | 复杂度 | 适用 |
|------|------|--------|--------|------|
| ZooKeeper | 中 | 高（CP） | 中 | 已有 ZK 的生态 |
| etcd | 中 | 高（CP） | 中 | K8s 环境 |
| 数据库 | 低 | 中（依赖 DB） | 低 | 没有 ZK/etcd 的过渡 |
| Redisson | 高 | 中（AP） | 低 | 对一致性要求不高的场景 |

### 4.6 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 第 4 章 共识算法——选主是共识的基础，共识算法天然支持选主 |
| **释慧利** | 算法 | Raft 的 RequestVote + 多数派选主、ZAB 的 lastZxid 选主——选主的形式化 |
| **Unmesh** | 模式 | 第三篇 Leader 选举模式——Bully/Ring/Raft 三种选主 |

### 4.7 本节小结

分布式选主的三条路径：
- **ZooKeeper/etcd**：高可靠 CP 选主（分布式定时任务、协调者选举）
- **数据库**：过渡方案（没有 ZK/etcd 的应急方案）
- **Redisson**：高性能 AP 选主（库存扣减等一致性要求不高的场景）

## 结语：负载均衡、锁与选主的三角关系

回顾本篇，三个主题形成一个完整的协调体系：

- **负载均衡**：把请求均匀分发——"谁来处理"
- **分布式锁**：保证互斥访问——"谁先处理"
- **分布式选主**：选出唯一协调者——"谁负责协调"

**负载均衡算法的演进**：轮询→加权→最少连接→一致性哈希→权重响应时间——从"公平"到"智能"的进化。

**分布式锁的选型**：高并发用 Redis（Redisson），高可靠用 ZooKeeper/etcd。

**分布式选主的本质**：一次"长期持有"的分布式锁，区别在于锁的生命周期——锁是秒级，选主是分钟/小时级。

**本篇的核心论断**：负载均衡是流量的"交警"，分布式锁是资源的"门卫"，分布式选主是集群的"班长"。三者共同构成了分布式系统的协调层。
