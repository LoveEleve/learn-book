## 4. 负载均衡——高可用的入口

### 4.1 为什么负载均衡是高可用的基石？

**问题**：第1章架构演进中，你从单机走到了集群——多台应用服务器。但用户的请求如何分发到这些服务器？如果一台宕机了怎么办？

**负载均衡解决两个核心问题**：
1. **流量分发**：请求均匀（或按策略）分配到多台后端服务器
2. **故障转移**：后端服务器宕机后自动摘除，请求不再转发到故障节点

### 4.2 四层 vs 七层负载均衡

| 维度 | 四层（L4，传输层） | 七层（L7，应用层） |
|------|------------------|------------------|
| **工作层** | TCP/UDP 层，基于 IP:Port 转发 | HTTP/HTTPS 层，基于 URL/Header/Cookie 转发 |
| **性能** | 高（只解析到传输层，1-10μs） | 较低（需要解析 HTTP，100-500μs） |
| **灵活性** | 低（不能按 URL/Cookie 路由） | 高（可按路径/Header/Cookie 路由） |
| **典型实现** | LVS（Linux Virtual Server）、F5 硬件 | Nginx、HAProxy、Spring Cloud Gateway |
| **适用场景** | 后端服务统一入口 | 微服务网关、API 路由、灰度发布 |

**生产架构——四层 + 七层组合**：

```
Client → DNS → LVS(L4) → Nginx(L7) → 后端服务集群
                   │            │
              (高性能分发)   (按URL/Cookie路由,
                             灰度发布, 限流)
```

**为什么需要两层？** L4 性能高但不灵活，L7 灵活但性能低。L4 做第一层"粗粒度分发"（把所有 `/api/` 流量导向 Nginx 集群），L7 做第二层"细粒度路由"（`/api/order` → 订单服务，`/api/user` → 用户服务）。

### 4.3 四种经典负载均衡算法

#### 4.3.1 轮询（Round Robin）——均等分配

```
请求序列: R1 R2 R3 R4 R5 R6
分发结果: S1 S2 S3 S1 S2 S3  (三个服务器轮流)
```

**优点**：简单、无状态。**缺点**：假设所有服务器处理能力相同——不适用于异构集群。

#### 4.3.2 加权轮询（Weighted Round Robin）

```
服务器   权重   分发比例
S1(4C8G)  1      1/6
S2(8C16G) 2      2/6
S3(16C32G)3      3/6
```

**适用**：异构集群（不同配置的服务器）。Nginx 的 `weight` 参数就是加权轮询。

#### 4.3.3 最少连接（Least Connections）

将请求分发给**当前活跃连接数最少**的服务器。

**为什么比轮询好？** 轮询假设每个请求耗时相同——但实际请求耗时差异大。长连接（WebSocket）会严重占用轮询分配不均（一台服务器 100 个长连接，另一台 10 个——轮询仍均分）。最少连接动态感知负载。

**适用**：长连接场景（WebSocket、gRPC streaming）、请求耗时差异大。

#### 4.3.4 一致性哈希（Consistent Hash）

将请求按**哈希环**映射到服务器——相同 key 的请求始终路由到同一台服务器。

```
哈希环: 0 ──── 2^32-1
         S1  S3  S2
        /    \/    \
       /     /\     \
      /     /  \     \
```

**关键特性**——**添加/移除节点时，只影响哈希环上相邻的节点**：
- 添加 S4：S4 落在 S1-S3 之间 → 只有 S1-S3 范围内的 key 需要重新映射（约 1/N 的 key）
- 移除 S2：只有 S2-S3 范围内的 key 需要重新映射（约 1/N 的 key）
- 对比普通哈希 `hash(key) % N`：添加/移除节点 → **所有** key 重新映射（100%）

**适用**：缓存集群（Redis Cluster、Memcached）。相同 key 路由到相同节点 = 命中率高。

### 4.4 健康检查——负载均衡的"眼睛"

负载均衡器必须知道后端服务器是否存活。三类健康检查：

| 类型 | 检查方式 | 速度 | 准确性 |
|------|---------|------|--------|
| **TCP 连接** | 尝试建立 TCP 连接 | 快（1-10ms） | 低（只检测端口开，不检测应用层） |
| **HTTP 健康检查** | GET /health → 检查 HTTP 2xx | 中（10-50ms） | 中（检查 HTTP 层，不检查 DB 连接） |
| **深度健康检查** | 由应用暴露 /health，返回 DB/Redis/MQ 连接状态 | 慢（50-200ms） | 高（完整的应用健康状况） |

**故障转移机制**：
- 连续 N 次检查失败 → 标记为 DOWN（摘除）
- 连续 M 次检查成功 → 标记为 UP（恢复）
- N > M（摘除更敏感，恢复更谨慎）——防止频繁抖动

## 5. 分布式共识——高可用的灵魂

### 5.1 高可用为什么需要共识？

**场景**：3 节点集群，Leader 负责写入。Leader 宕机后——谁当新 Leader？

**没有共识**：节点各自判断 → 两个节点宣布自己是 Leader → 脑裂。
**有共识**：Paxos/Raft/ZAB 的多数派投票 → 唯一的新 Leader 被选出。

**在高可用系统中，共识保证了两件事**：
1. **选主**：唯一 Leader 被选出（或确认不存在）
2. **日志一致性**：所有节点的数据是一致的

> **跨书深度导航**：释慧利第4/5/6章详细推导了 Paxos/ZAB/Raft 的完整算法。江峰第8章用 I/O Automata 证明了三者的安全性。本篇从**高可用架构师**的视角——"什么场景选什么共识协议"。

### 5.2 三协议选型——高可用的工程决策

| 协议 | 典型系统 | 选型理由 |
|------|---------|---------|
| **Paxos** | Google Chubby/Spanner、Cassandra LWT、MySQL MGR | 灵活性最高，但实现复杂——适合"我需要定制共识流程"的场景 |
| **Raft** | etcd、TiKV、Consul、Kafka KRaft | 可理解性最高——新系统首选，"我需要在半年内稳定上线" |
| **ZAB** | ZooKeeper | 主备复制 + primary order——"我需要协调服务" |

**决策树**：

```
需要共识？
  ├─ 需要现成的协调服务 → ZooKeeper（ZAB）
  ├─ 新系统，需要简单可靠 → etcd（Raft）
  ├─ 需要全球数据库 → Spanner（Paxos + TrueTime）
  └─ 需要开源全球数据库 → CockroachDB（Raft + HLC）
```

### 5.3 高可用集群的规模规划

| 节点数 | 容忍故障数 | 推荐场景 |
|--------|----------|---------|
| 3 | 1 | 小规模生产——etcd/ZooKeeper 标准配置 |
| 5 | 2 | 需要更高容错——容忍两个节点同时故障 |
| 7 | 3 | 大规模跨地域——3 个地域各部署一些节点 |

**关键约束**：节点越多 → 多数派越大 → 写延迟越高。3 节点（多数派=2）的写延迟远低于 7 节点（多数派=4）。**不要为了"更多容错"盲目增加节点**——3 节点足够大多数场景。

### 6.2 典型高可用架构全景

```
                    DNS (多A记录)
                         │
                    ┌────┴────┐
                    │  LVS×2  │  (L4 负载均衡，主备)
                    └────┬────┘
                         │
                    ┌────┴────┐
                    │ Nginx×3 │  (L7 负载均衡集群)
                    └────┬────┘
                         │
          ┌──────────────┼──────────────┐
     ┌────▼────┐    ┌────▼────┐    ┌────▼────┐
     │ Web ×3  │    │ API ×3  │    │ Wx ×3   │  (应用层集群)
     └────┬────┘    └────┬────┘    └────┬────┘
          │              │              │
     ┌────▼────┐    ┌────▼────┐    ┌────▼────┐
     │  Redis  │    │  etcd   │    │  MQ     │  (存储/中间件层)
     │ Cluster │    │  (Raft) │    │ Cluster │
     └─────────┘    └─────────┘    └─────────┘
```

**每一层的高可用保证**：
- **DNS**：多 A 记录（域名解析到多个 IP），客户端自动切换
- **LVS**：主备模式（keepalived），故障自动切换 VIP
- **Nginx**：集群 + 健康检查，故障节点自动摘除
- **应用层**：集群 + 负载均衡，单节点故障不影响整体
- **存储/中间件层**：共识（Raft/Paxos）保证数据一致性

## 6.3 本章总结

1. **负载均衡是高可用入口**——四层（LVS，高性能）+ 七层（Nginx，灵活路由）组合。
2. **四种负载均衡算法**：轮询（均分）、加权（异构集群）、最少连接（长连接感知）、一致性哈希（缓存命中）。
3. **共识是高可用的灵魂**——选主和日志一致性由 Paxos/Raft/ZAB 保证。3 节点（容忍 1 故障）标准配置。
4. **完整高可用架构**：DNS → L4 → L7 → 应用 → 存储——每一层都有对应的高可用保证。
