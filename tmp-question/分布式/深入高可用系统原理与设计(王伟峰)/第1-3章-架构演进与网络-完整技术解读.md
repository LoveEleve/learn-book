## 1. 架构演进：从单点到高可用

### 1.1 单体架构：高可用的起点

**架构**：一个应用进程 + 一个数据库。

```
┌──────────────┐     ┌──────────┐
│  Web/API/DB  │ ←→ │  数据库   │
└──────────────┘     └──────────┘
    单机部署
```

**优点**：简单、开发快、部署快。适合初创项目（QPS < 100）。

**高可用问题**：
- 应用故障 → 全站不可用
- 数据库故障 → 所有数据不可访问
- 没有故障切换能力

### 1.2 集群架构：加机器解决单点

**架构**：多台服务器部署相同应用，负载均衡分发流量。

```
          用户
           │
    ┌──────┴──────┐
    │  负载均衡    │
    └──────┬──────┘
    ┌──────┼──────┐
    ↓      ↓      ↓
  节点1   节点2   节点3
    └──────┼──────┘
           │
    ┌──────┴──────┐
    │  数据库主从  │
    └─────────────┘
```

**高可用改进**：
- 应用层多节点 → 单节点故障不影响服务
- 负载均衡 → 流量自动分发到健康节点
- 数据库主从 → 从库可分担读压力，主库故障可手动切换

**新问题**：
- 负载均衡本身是单点 → 下一步：多 LB + 健康检查
- 数据库主库是单点 → 下一步：MGR/分布式数据库
- 用户 Session 跨节点不共享 → 下一步：集中 Session 存储

**Session 集中存储方案**：
```
方案 1（粘性会话）：
  负载均衡配置 Sticky Session → 同一用户始终路由到同一节点
  缺点：节点故障会丢失该节点上的所有 Session

方案 2（集中存储）：
  Session 存 Redis → 所有节点都能读写 Session
  Redis 本身需要高可用（Sentinel 主从）
```

### 1.3 微服务架构：拆分成独立服务

**架构**：按业务拆分，每个服务独立部署和扩展。

```
          用户
           │
    ┌──────┴──────┐
    │  API 网关    │
    └──────┬──────┘
    ┌──────┼──────────────────┐
    ↓      ↓                  ↓
  用户服务  订单服务           商品服务
  (独立)   (独立)             (独立)
    │       │                  │
    └───────┼──────────────────┘
            │
    分布式数据库 + 缓存 + MQ
```

**高可用改进**：
- 单个服务故障不影响其他服务（隔离）
- 核心服务可以独立扩容（不浪费非核心服务的资源）
- 每个服务都可以独立部署多节点

**新问题**：
- 服务间调用链路变长 → 需要链路追踪
- 分布式事务 → 需要 Seata/消息事务
- 配置管理 → 需要配置中心

### 1.4 云原生架构

**架构**：容器化 + K8s 编排 + Service Mesh + CI/CD。

```
         ┌──────────────────────────┐
         │       K8s 集群            │
         │  ┌────────────────────┐  │
         │  │    Service Mesh     │  │
         │  │  (Istio/Linkerd)    │  │
         │  └──┬──────┬──────┬───┘  │
         │     ↓      ↓      ↓      │
         │   Pod1   Pod2   Pod3     │
         │  ┌────┐ ┌────┐ ┌────┐   │
         │  │容器│ │容器│ │容器│   │
         │  └────┘ └────┘ └────┘   │
         └──────────────────────────┘
```

**高可用特性**：
- Pod 自动重启（liveness probe 失败 → K8s 自动重启）
- Pod 自动扩容（HPA，CPU/内存超阈值 → 自动加 Pod）
- 节点故障 → Pod 自动调度到健康节点
- 滚动更新 → 新版本逐步替换旧版本（类似灰度）

### 1.5 架构演进的代价

| 架构 | 复杂度 | 运维成本 | 可用性 | 扩展性 |
|------|--------|---------|--------|--------|
| 单体 | 低 | 低 | 低（单点） | 低 |
| 集群 | 中 | 中 | 中（应用层多节点） | 中 |
| 微服务 | 高 | 高 | 高（隔离） | 高（独立扩容） |
| 云原生 | 最高 | 中（自动化提效） | 最高（自动恢复） | 最高（自动扩缩） |

**选型原则**：**不要为了架构而架构**——QPS < 1000 时，单体 + 定时备份就够了。过度架构会让团队被复杂度拖垮。

### 1.6 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 架构演进是"部分失败"难题驱动的——单点→集群解决节点故障，微服务→云原生解决规模故障 |
| **唐扬** | 工程 | 冗余与分片是架构演进的核心手段——加机器 + 拆服务 |
| **Unmesh** | 模式 | 集群管理/故障检测/负载均衡是架构演进的模式层 |

### 1.7 本节小结

架构演进的核心驱动力是"高可用"：
- **单体→集群**：解决单点故障
- **集群→微服务**：解决规模瓶颈
- **微服务→云原生**：解决运维复杂性
- **选择匹配 QPS 量级的架构**——不要过度设计

## 2. 网络高可用

### 2.1 网络层是高可用的第一道关口

所有流量进入系统前，必须先经过网络层：

```
用户 → DNS → CDN → 负载均衡 → API 网关 → 应用
```

网络层的每一层都可能成为瓶颈或单点。网络高可用的核心是：**在每一层做冗余 + 健康检查 + 自动故障切换**。

### 2.2 DNS 高可用

**DNS 的作用**：将域名解析为 IP 地址。

**单 DNS 的风险**：
- DNS 服务器故障 → 域名无法解析 → 全站不可用
- 案例：2021 年 Akamai DNS 故障导致全球大量网站不可用 1 小时

**DNS 高可用方案**：
```
方案 1：多 DNS 厂商
  主：Cloudflare DNS
  备：阿里云 DNS
  域名注册商配置多个 NS 记录

方案 2：DNS 负载均衡（GTM）
  根据用户地理位置返回最近的 IP
  故障时自动切换到备用 IP
```

**DNS 切换的局限**：
- TTL 影响切换速度（TTL=60s → 最长 60 秒完成全量切换）
- 部分 ISP 会忽略 TTL，强制缓存更长时间
- 对实时性要求极高的场景，DNS 切换不够快

### 2.3 负载均衡高可用

**负载均衡的作用**：将流量分发到多个后端节点。

**单 LB 的风险**：LB 本身故障 → 虽然后端节点都健康，但流量进不来。

**LB 高可用方案**：

**方案 1：Keepalived + VIP（主备）**
```
         VIP（虚拟 IP）
             │
    ┌────────┴────────┐
    │                 │
  LB Master ←→ LB Backup
（主）        （备，通过 VRRP 协议检测 Master 健康）
```
- 优势：成熟稳定
- 劣势：备 LB 空闲浪费；切换有短暂间隔

**方案 2：多 LB + DNS 轮询**
```
用户 → DNS 返回 LB1 或 LB2 的 IP
         ┌──────┴──────┐
         ↓              ↓
       LB1            LB2
```
- 优势：两个 LB 都工作，无资源浪费
- 劣势：DNS 缓存影响切换速度

**方案 3：云厂商 LB（NAT 模式）**
```
用户 → 云 LB（如 AWS ALB / 阿里云 SLB）
         → 自动健康检查
         → 自动扩缩容
```
- 优势：云厂商保证 LB 自身高可用
- 劣势：成本较高，定制性低

### 2.4 反向代理高可用

**反向代理的作用**：接收客户端请求，转发到后端服务。

**Nginx 高可用方案**：
```
方案 1：Keepalived + VIP（Nginx 主备）

方案 2：Nginx + Nacos 动态配置
  Nacos 配置中心存储上游服务列表
  Nginx 通过 Lua 从 Nacos 拉取配置
  上游服务变更时，Nacos 推送更新
```

**Nginx 的健康检查**：
```nginx
upstream backend {
    server 192.168.1.1:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.2:8080 max_fails=3 fail_timeout=30s;
}
```
- `max_fails=3`：连续 3 次失败后标记为不可用
- `fail_timeout=30s`：30 秒后重新探测

**Nginx Plus（商业版）**支持主动健康检查：
```nginx
upstream backend {
    zone backend 64k;
    server 192.168.1.1:8080;
    server 192.168.1.2:8080;
}
match health_check {
    status 200;
    body ~ "OK";
}
server {
    location / {
        proxy_pass http://backend;
        health_check interval=5s fails=2 passes=3 uri=/health;
    }
}
```

### 2.5 CDN 与边缘节点

**CDN 的作用**：将静态资源缓存到离用户最近的节点，加速访问。

**CDN 的高可用特性**：
- 多节点冗余（全球数千节点）
- 自动故障切换（节点不可用 → 回源到源站）
- 自动负载均衡（用户就近访问）

**CDN 回源策略**：
```
用户请求 → CDN 边缘节点
  ├─ 缓存命中 → 直接返回
  └─ 缓存未命中 → 回源到源站 → 缓存到边缘节点 → 返回
```

### 2.6 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 网络层不可靠是 4 大难题之一——DNS/LB/反向代理/CDN 各层都要冗余 |
| **唐扬** | 工程 | 多 LB + 健康检查是网络高可用的工程实践 |
| **Unmesh** | 模式 | 心跳/故障检测——实现健康检查的基础 |

### 2.7 本节小结

网络高可用的核心原则：**每一层都有冗余 + 自动故障切换**：
- **DNS**：多厂商 + GTM
- **LB**：主备（Keepalived）/ 多 LB + DNS / 云 LB
- **反向代理**：Keepalived + Nacos 动态配置 + 健康检查
- **CDN**：全球多节点 + 自动回源

## 3. 网络层的容灾设计

### 3.1 多活入口

**核心问题**：如何让用户在所有入口都故障的情况下仍能访问？

**方案 1：多 DNS 解析**
- 主域名解析到主机房 IP
- 备用域名（或同一域名的多个 A 记录）解析到备机房
- 主 DNS 不可用 → 用户可尝试备用域名

**方案 2：Anycast（任播）**
- 同一 IP 在多个机房同时发布路由
- 用户访问该 IP → BGP 自动选择最近的机房
- 一个机房故障 → BGP 自动撤销该机房路由 → 流量切换到其他机房

### 3.2 故障检测与切换

**三板斧：探测、摘除、恢复**：
1. **探测**（健康检查）：定期探测后端节点是否健康
2. **摘除**（故障隔离）：探测失败 → 从负载列表移除
3. **恢复**（重新加入）：探测成功 → 重新加入负载列表

**健康检查的方式**：
| 方式 | 说明 | 适用 |
|------|------|------|
| **TCP** | 检测端口是否可连接 | 通用 |
| **HTTP** | 检测返回码（200/302） | Web 服务 |
| **自定义** | 执行业务逻辑检查（如查询 DB） | 深度健康检查 |

### 3.3 生产案例

**案例 1：多机房多活入口**
```
用户 → 全局 DNS（GTM 智能解析）
  ├─ 华北用户 → 北京机房（VIP1）
  │     └─ Keepalived 主备 LB（VIP1）
  ├─ 华东用户 → 上海机房（VIP2）
  │     └─ Keepalived 主备 LB（VIP2）
  └─ 华南用户 → 深圳机房（VIP3）
        └─ Keepalived 主备 LB（VIP3）
```
任何机房故障 → GTM 自动切换到其他机房的 VIP。

**案例 2：CDN 容灾**
```
用户请求静态资源
  ├─ CDN 节点 1（上海）→ 故障（回源超时）
  ├─ CDN 自动重试 → CDN 节点 2（杭州）→ 命中缓存
  └─ 如果所有 CDN 节点都未命中 → CDN 节点 2 回源到源站
```

### 3.4 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 网络层容灾是"网络不可靠"的工程应对——多层冗余让单点失效不致命 |
| **唐扬** | 工程 | 多活入口 + 健康检查 + 自动切换——第一篇容错手段的工程实现 |
| **Unmesh** | 模式 | 故障检测（Failure Detector）+ 心跳——网络层容灾的基础模式 |

### 3.5 本节小结

网络层容灾的核心：
- **多活入口**：多 DNS / Anycast
- **故障切换三步**：探测 → 摘除 → 恢复
- **健康检查**：TCP / HTTP / 自定义三级

## 结语：从架构演进到网络高可用

回顾本篇，我们完成了架构演进与网络高可用的推导：

- **第 1 节**：架构演进——单体→集群→微服务→云原生，每一步都是"高可用"驱动
- **第 2 节**：网络高可用——DNS/LB/反向代理/CDN 四层，每层都要冗余
- **第 3 节**：网络容灾——多活入口 + 健康检查 + 故障切换

**第一篇的核心论断**：架构演进的核心驱动力是"高可用"——从单点到集群解决节点故障，从集群到微服务解决规模瓶颈，从微服务到云原生解决运维复杂性。网络层是流量的第一道关口——DNS/LB/反向代理/CDN 每一层都要冗余 + 自动故障切换。**选择匹配 QPS 量级的架构——不要过度设计**。
