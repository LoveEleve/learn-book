## 1. 大型互联网后台的基础架构

### 1.1 从单机到亿级流量的架构全景

一个能扛住亿级流量的互联网后台，不是某个单一技术的堆砌，而是**分层架构**的产物。每一层解决一类问题，层与层之间通过标准协议交互。

```
┌─────────────────────────────────────────┐
│              接入层                       │
│   DNS → CDN → 反向代理 → API 网关         │
├─────────────────────────────────────────┤
│              服务层                       │
│   微服务集群（注册发现 + RPC + 配置中心）   │
│   业务逻辑：用户/订单/商品/内容/社交         │
├─────────────────────────────────────────┤
│              中间件层                     │
│   消息队列 / 缓存 / 搜索引擎 / 定时任务     │
├─────────────────────────────────────────┤
│              数据层                       │
│   关系数据库（分库分表）/ KV 存储 / 对象存储 │
├─────────────────────────────────────────┤
│              基础设施层                   │
│   K8s / 容器 / 监控 / 日志 / 链路追踪      │
└─────────────────────────────────────────┘
```

**五层架构的职责**：

| 层 | 解决的问题 | 核心组件 | 典型 QPS 上限 |
|---|----------|---------|-------------|
| 接入层 | 流量入口、安全防护、协议转换 | Nginx/Spring Cloud Gateway/DNS/CDN | 百万级 |
| 服务层 | 业务逻辑、服务治理 | 微服务集群（Nacos+Sentinel+Dubbo） | 十万级 |
| 中间件层 | 异步解耦、加速读写 | RocketMQ/Redis/Elasticsearch | 十万级 |
| 数据层 | 持久化存储、事务保证 | MySQL（分库分表）/TiDB/HBase | 万级 |
| 基础设施层 | 部署、监控、运维 | K8s/Prometheus/SkyWalking/ELK | — |

**关键认知**：每一层都有自己的 QPS 上限。系统整体的 QPS 上限由**最慢的层**决定（短板效应）。高并发设计的本质是**找到每一层的瓶颈并突破它**。

### 1.2 大厂基础组件全景

一个成熟的互联网后台需要的基础组件：

| 组件类型 | 具体组件 | 职责 | 与前面书的关联 |
|---------|---------|------|-------------|

### 1.3 机房部署架构

**同城双活 + 异地灾备**（中型公司的标配）：

```
       用户（全国）
          │
   ┌──────┴──────┐
   │  DNS 智能解析 │
   └──────┬──────┘
   ┌──────┼──────────────┐
   ↓      ↓              ↓
北京双活              上海灾备
┌─────────┐         ┌─────────┐
│ AZ-A(主)│         │ 冷备     │
│ AZ-B(主)│         │ (异步同步)│
└─────────┘         └─────────┘
```

**同城双活**（北京两个 AZ）：流量 50:50 分配，两个 AZ 都提供完整服务。
**异地灾备**（上海）：异步同步数据，北京全宕时手动切换到上海（数据可能丢失，恢复时间分钟级）。

**同城双活的技术要求**（王伟峰第一篇已讲过，本节聚焦具体参数）：
- 跨 AZ 延迟：< 5ms
- 数据库：TiDB 5 副本（AZ-A:2 + AZ-B:2 + learner:1）或 MySQL MGR 单主模式（AZ-A:3，AZ-B 异步复制备集群；MGR 3 副本 2+1 分布下 AZ-A 故障后无法形成多数派）
- 缓存：Redis Sentinel 跨 AZ 主从 + 双写失效通知
- 消息队列：RocketMQ 双主双从跨 AZ

### 1.4 架构演进的决策框架

**不是越大越好——匹配规模选架构**（唐扬和王伟峰都强调的原则）：

| QPS | 数据量 | 推荐架构 | 关键组件 |
|-----|--------|---------|---------|
| <1000 | <100GB | 单机房单活 | Nginx + Spring Boot + MySQL 主从 + Redis 单机 |
| 1000-5000 | <500GB | 同机房 3 副本 | 加 Sentinel 熔断限流 + RocketMQ 异步 |
| 5000-20000 | <2TB | 同城双活 | 加 Nacos 注册配置 + 分库分表 |
| 20000-100000 | <10TB | 同城双活+分片 | 加 Elasticsearch + TiDB/CockroachDB |
| >100000 | >10TB | 异地多活+单元化 | 加 CDN + 多机房 + 单元化路由 |

### 1.5 本节小结

互联网后台五层架构：接入→服务→中间件→数据→基础设施。每一层有 QPS 上限，短板决定系统整体性能。**匹配规模选架构**——不是越大越好。

## 2. 通用的高并发架构设计

### 2.1 高并发场景的分类

高并发场景不是单一的——读场景和写场景有本质区别。

| 场景 | 典型 QPS | 特点 | 核心手段 |
|------|---------|------|---------|
| **高并发读** | 100K-1M QPS | 读多写少，数据可缓存 | 缓存、读写分离、CDN |
| **高并发写** | 1K-100K TPS | 写多读少，数据不可缓存 | 分库分表、异步写、写聚合 |
| **混合读写** | 读写均衡 | 读写都高 | 读写分离 + 异步写 + 缓存 |

**读场景**（如商品详情页）：用户看多买少，读 QPS 远超写 QPS。优化重点：缓存、CDN、静态化。

**写场景**（如秒杀下单）：瞬时写 QPS 极高，不能依赖缓存（写操作不能"缓存"）。优化重点：限流、异步写、库存 Redis 预扣。

### 2.2 高并发读：读写分离

**核心思想**：主库处理写，从库处理读。读 QPS 通过加从库横向扩展。

```
写请求 → 主库（1 台）
读请求 → 从库（N 台，可横向扩展）
```

**读写分离的实现**（ShardingSphere-JDBC 4.x 语法）：
```yaml
sharding:
  masterSlaveRules:
    ds_0:
      masterDataSourceName: master_0
      slaveDataSourceNames:
        - slave_0_0
        - slave_0_1
      loadBalanceAlgorithmType: ROUND_ROBIN
```

**读写分离的坑**（从库延迟）：

| 延迟大小 | 影响 | 应对 |
|---------|------|------|
| < 100ms | 用户几乎无感知 | 无需处理 |
| 100ms-1s | 用户写后立即读可能看不到 | 写后读走主库 |
| > 1s | 明显的读滞后 | 监控告警 + 检查从库同步状态 |

**写后读走主库**（用户刚下单立即查询，必须读到最新数据）：
```java
@Transactional
public Order createOrder(OrderDTO dto) {
    Order order = orderMapper.insert(dto);  // 写主库
    return orderMapper.selectById(order.getId());  // 读主库（不走从库）
}
```

### 2.3 高并发读：本地缓存

**核心思想**：在服务实例的 JVM 内存中缓存热点数据，不经过网络调用——延迟从 ms 级降到 μs 级。

```
无缓存：  用户 → Nginx → 服务 → Redis（1ms）→ MySQL（10ms）
本地缓存：用户 → Nginx → 服务 → JVM 内存（1μs）
```

**本地缓存的适用场景**：
- 数据量小（< 10000 条）
- 变更频率低（分钟/小时级）
- 读 QPS 极高（> 10 万）

**Caffeine 本地缓存示例**：
```java
Cache<String, Product> cache = Caffeine.newBuilder()
    .maximumSize(10000)          // 最多 1 万条
    .expireAfterWrite(10, TimeUnit.MINUTES)  // 10 分钟过期
    .recordStats()               // 记录命中率
    .build();
```

**本地缓存的坑**（多实例不一致）：
- 实例 A 的缓存是旧版本，实例 B 的缓存已更新
- **解决方案**：短 TTL（1-5 分钟）+ 消息通知失效（RocketMQ 广播缓存清除消息）

### 2.4 高并发读：分布式缓存

**核心思想**：Redis 作为集中的缓存层，所有实例共享同一份缓存。

**缓存的热点问题**：
- 单个 key 被极高频率访问（如热门商品）
- Redis 单分片扛不住 → 需要**本地缓存 + 分布式缓存双层**

```
双层缓存架构：
  请求 → 本地缓存（Caffeine，1μs）
            ↓ miss
         Redis 分布式缓存（1ms）
            ↓ miss
         MySQL（10ms）
            ↓
         回填 Redis + 本地缓存
```

**缓存穿透/击穿/雪崩**（唐扬第二篇已提到，本节聚焦具体应对）：

| 问题 | 现象 | 应对 |
|------|------|------|
| **穿透** | 查询不存在的数据，每次都穿透到 DB | 布隆过滤器 / 空值缓存（TTL=60s） |
| **击穿** | 热点 key 过期，瞬时大量请求到 DB | 互斥锁（Redisson）+ 逻辑过期 |
| **雪崩** | 大量 key 同时过期，DB 瞬间被打爆 | TTL + 随机偏移（±30%） |

### 2.5 高并发读的统一视角：CQRS

**CQRS**（Command Query Responsibility Segregation）：读写分离的架构模式。不是数据库层面的主从分离，而是**应用层面的读写模型分离**。

```
Command 端（写）                     Query 端（读）
  用户下单                            用户查看订单
    ↓                                  ↓
  写模型（OrderWriteService）         读模型（OrderReadService）
    ↓                                  ↓
  写数据库（主库）                    读数据库（从库/ES/Redis）
    ↓                                  ↓
  发 MQ 消息 ─────→ 更新读模型 ────→ 读数据库同步
```

**CQRS 的价值**：
- 读写用不同的数据模型（写用范式化表、读用宽表/ES 文档）
- 读写独立扩展（写压力大扩写库、读压力大扩读库）
- 事件驱动同步（写模型变更通过 MQ 通知读模型更新）

**CQRS 的代价**：
- 数据一致性延迟（写模型更新后，读模型需要时间同步）
- 系统复杂度增加（两套模型 + 同步机制）

### 2.6 高并发写：数据分片

**核心思想**：把一张大表按 Sharding Key 拆成多张子表，分散到多个数据库实例。

**分库分表策略**（已由唐伟志第 3 章和 Unmesh 第二篇详讲，此处聚焦高并发写场景的选型）：

| 分片策略 | 适用 | 高并发写的优势 | 代价 |
|---------|------|-------------|------|
| **按 user_id 哈希** | 用户维度的操作 | 用户数据均匀分布 | 跨用户操作需分布式事务 |
| **按时间范围** | 时序数据（日志/订单） | 新数据写入最新分片（无热点） | 旧分片可能成为冷数据 |
| **按业务维度** | 不同业务分不同库 | 业务隔离，互不影响 | 跨业务查询复杂 |

**分库分表的热点应对**：
- 大 V 发微博，所有用户访问集中在一条数据
- **解决方案**：热点 key 探测 → 子 key 拆分 → 本地缓存

### 2.7 高并发写：异步写与写聚合

**异步写的核心**：把同步写入改为先发 MQ 再异步消费，减少用户请求的等待时间。

```
同步写：用户等待数据库写入完成（10-50ms）
异步写：用户请求 → 发 MQ → 立即返回（1ms）→ 消费者异步写入 DB
```

**写聚合**：把多个写操作合并成一次批量写入。

```
单次写入：100 个请求 → 100 次 DB 写入（100×10ms = 1000ms）
批量写入：100 个请求 → 聚合 → 1 次 DB 批量写入（10ms）
```

**异步写的局限**：
- 用户不知道写入是否成功（需要异步通知或查询确认）
- 只适合非核心链路（核心链路如支付必须同步确认）

### 2.8 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 第 3 章 一致性模型 + Quorum → 读写分离/缓存的一致性理论基础 |

### 2.9 本节小结

高并发读的三级缓存体系：本地缓存（Caffeine）→ 分布式缓存（Redis）→ 数据库（MySQL/TiDB）。CQRS 是读写分离的架构模式。高并发写用分库分表 + 异步写 + 写聚合。

**高并发优化提升了吞吐，但也放大了单个故障的爆炸半径**——一个服务挂了，瞬间积压的并发请求会把整条链路压垮。所以紧接着需要可用性治理。第 3 节展开。

## 3. 通用的服务可用性治理

### 3.1 可用性治理的三层架构

**一个 100 个微服务的系统，如果每个服务都没有治理手段，一个慢 SQL 就能拖垮整个链路**（雪崩级联）。为什么需要分层治理？因为每一层的失败模式不同——接入层是流量突发，服务层是下游故障，数据层是慢查询和连接池耗尽。分层治理 = 在每一层用最适合的手段拦截失败。

唐扬第一篇详讲了四种容错手段（熔断/降级/隔离/限流）。本节从**架构层**重新组织——不是单个服务的容错，而是**整个系统的可用性治理体系**。

```
                    ┌─────────────┐
                    │   接入层治理  │  ← 网关限流 + 入口熔断
                    └──────┬──────┘
                           │
                    ┌──────┴──────┐
                    │   服务层治理  │  ← Sentinel 服务治理
                    └──────┬──────┘
                           │
                    ┌──────┴──────┐
                    │   数据层治理  │  ← 数据库连接池 + 慢查询熔断
                    └─────────────┘
```

### 3.2 接入层治理

**Sentinel 网关限流**（通过 Sentinel Dashboard 或 API 注册规则，非 YAML 直接配置）：
```java
// 代码注册网关流控规则
GatewayFlowRule rule = new GatewayFlowRule()
    .setResource("order-service")
    .setGrade(RuleConstant.FLOW_GRADE_QPS)
    .setCount(5000);      // 网关层限制 5000 QPS
GatewayRuleManager.loadRules(Collections.singletonList(rule));
```

**网关的价值**：
- 统一入口，集中管控（不需要每个服务自己配限流）
- 前置拦截，减少无效流量进入服务层
- 路由 + 限流 + 认证一体化

### 3.3 服务层治理

**Sentinel 服务治理全景**（唐扬第一篇已讲四种模式，本节聚焦服务层组合）：

```
服务 A 调用服务 B：
  ├─ 限流：服务 A 对服务 B 的调用 QPS 限制（保护 B）
  ├─ 熔断：服务 B 的慢调用比例 > 50% → 熔断（保护 A 和整个链路）
  ├─ 降级：服务 B 熔断后，服务 A 返回降级结果
  └─ 隔离：核心链路独立线程池，不受非核心影响
```

**服务治理的监控**：
```java
// Sentinel Dashboard 监控的关键指标
- QPS：每秒请求数
- RT：平均响应时间 + P99
- 通过/拒绝 QPS：限流效果
- 熔断状态：OPEN/CLOSED/HALF-OPEN
```

### 3.4 数据层治理

**数据库连接池治理**：控制每个服务对数据库的并发连接数。

```yaml
# HikariCP 连接池配置
spring:
  datasource:
    hikari:
      maximum-pool-size: 20     # 最大连接数（默认 10）
      minimum-idle: 5            # 最小空闲连接
      connection-timeout: 3000   # 连接超时 3s
      idle-timeout: 600000      # 空闲回收 10min
      max-lifetime: 1800000     # 最大生命周期 30min
```

**连接池太大的风险**：
- 多个服务同时创建过多连接 → DB 连接数超限 → DB 拒绝新连接
- 微服务场景下，10 个服务 × 20 连接 = 200 连接，可能超过 DB 的 max_connections（默认 151）

**慢查询治理**：
- 慢查询监控（> 200ms 的 SQL 记录日志）
- 慢查询自动熔断（某 SQL 持续慢 → 对该 SQL 的调用熔断，避免拖垮 DB）

### 3.5 可用性治理的灰度发布

**可用性治理规则也需要灰度**：
- 新的限流阈值先在 1% 流量上验证
- 新的熔断规则先在非核心服务上验证
- 避免"一刀切"导致误杀正常流量

### 3.7 本节小结

可用性治理的三层体系：接入层（网关限流）→ 服务层（Sentinel 四大金刚）→ 数据层（连接池 + 慢查询熔断）。每一层都有独立的治理策略，互相配合形成纵深防御。

## 结语：从基础架构到业务服务

回顾本篇，我们建立了亿级流量系统的架构基础：

- **第 1 节**：五层架构 + 机房部署 + 决策框架——每一层都有 QPS 上限
- **第 2 节**：高并发读（读写分离 + 本地/分布式缓存 + CQRS）+ 高并发写（分库分表 + 异步写）
- **第 3 节**：可用性治理三层体系（接入层 + 服务层 + 数据层）

**第一篇的核心论断**：亿级流量系统不是某个单一技术的堆砌，而是**分层架构**的产物。接入→服务→中间件→数据→基础设施，每一层解决一类问题。高并发读用缓存体系（本地+分布式+CQRS），高并发写用分片+异步+聚合，可用性治理用纵深防御（接入层+服务层+数据层）。
