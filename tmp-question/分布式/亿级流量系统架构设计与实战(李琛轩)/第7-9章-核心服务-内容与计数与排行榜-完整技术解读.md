## 1. 内容发布系统

### 1.1 内容发布的完整链路

**用户点击"发布"按钮后发生了什么？**

```
客户端 → API 网关 → 内容服务
  ├─ 1. 敏感词过滤（DFA / AC 自动机）
  ├─ 2. 内容审核（机审 + 人审）
  ├─ 3. 内容存储（MySQL）
  ├─ 4. 内容分发（MQ Fanout）
  │    ├─ Feed 流服务（推送到粉丝的 Timeline）
  │    ├─ 搜索服务（ES 索引）
  │    ├─ 计数服务（用户发布数 +1）
  │    └─ 通知服务（@ 的用户收到通知）
  └─ 5. 返回发布成功
```

**为什么步骤 4 要异步？**
- 同步 Fanout 到百万粉丝的 Timeline：100ms+ 延迟，用户体验差
- 异步 Fanout：发 MQ 1ms + 粉丝异步拉取 → 用户感知延迟从 100ms 降到 10ms

### 1.2 敏感词过滤（DFA 算法）

**核心问题**：题目中包含敏感词（如广告/涉黄/政治），需要实时检测。

**为什么不用正则或逐字扫描？**

| 方案 | 时间复杂度 | 10000 条敏感词的表现 |
|------|----------|-------------------|
| 逐字扫描 | O(n × m) | 10000 条 × 500 字 = 500 万次比较 → 太慢 |
| 正则 | O(n × m) | 同上 |
| **DFA（确定有限自动机）** | O(n) | 与敏感词数量无关，只扫描文本一遍 |

**DFA 的核心**：把敏感词库构建成 Trie 树，一次扫描文本即可匹配所有敏感词。

```
Trie 树结构（敏感词：你好，你妹）：
        root
        /
       你
      /  \
     好   妹
     (终) (终)

文本扫描："你好啊" → 你→好→啊(不匹配)→ 找到"你好"
```

**工程实现要点**：
- 敏感词库热加载（更新词库不需要重启服务）
- 内存占用：10000 条敏感词 ≈ 2MB（Trie 树比 HashMap 省空间）
- 并发安全：volatile + ReadWriteLock（写锁更新词库时，读锁不阻塞）

### 1.3 内容审核流水线

**审核的三级体系**：

```
内容发布
  ├─ 1. 机审（实时，< 100ms）
  │    ├─ DFA 敏感词（违规 → 直接拒绝）
  │    ├─ 图片鉴黄（调用第三方 API / 本地模型）
  │    └─ 文本反垃圾（广告/灌水检测）
  │
  ├─ 2. 人审（异步，分钟~小时级）
  │    └─ 机审不确定的内容 → 进入审核队列 → 人工审核
  │
  └─ 3. 举报复审（用户举报触发）
       └─ 举报量超过阈值 → 自动下架 → 人工复核
```

**机审不确定的判断**：
- DFA 匹配到"低风险"敏感词（如某些可商用的广告词）
- 图片鉴黄 API 返回"疑似"（概率 50%~80%）
- → 进入人审队列，但不阻塞发布（先发后审模式）

### 1.4 内容分发的 Fanout 模式

**Push 模式**：发布者直接把内容推送到所有粉丝的 Feed 流。

```
发布者 → 查询粉丝列表（可能百万级）
  → 逐个推送到粉丝的 Feed Redis List
  → 如果粉丝太多 → Push 任务可能超时
```

**Pull 模式**：发布者只存内容，粉丝拉取时从关注列表聚合。

```
粉丝打开 App → 拉取关注列表 → 逐个查询最新内容 → 聚合排序
  → 关注 100 人 → 需要查询 100 次 → 可能较慢
```

**Push + Pull 混合**（大 V 场景）：
- 普通用户（粉丝 < 1000）：Push 模式（实时推送到粉丝 Feed）
- 大 V（粉丝 > 1000）：Pull 模式（粉丝拉取时从大 V 的主页拉）

**工程实现**：
- 判断粉丝数：`followerCount > 1000` → 标记为大 V
- 大 V 的内容只存到自己的发布列表
- 粉丝拉取 Feed 时：普通用户的 Push 内容 + 大 V 的 Pull 内容 → 合并排序

### 1.5 本节小结

内容发布的三核心：DFA 敏感词（O(n) 扫描 + 热加载）、审核流水线（机审→人审→举报复审）、Fanout 模式（普通用户 Push + 大 V Pull）。

## 2. 通用计数系统

### 2.1 计数系统的核心挑战

**计数的本质**：读多写多的高频操作。
- 点赞：写 QPS 可能达数万
- 阅读数：写 QPS 可能达数十万
- 粉丝数：写 QPS 较低，但读 QPS 极高

**直接用 MySQL 的问题**：
- `UPDATE SET count = count + 1` → 行锁竞争 → 高并发下性能急剧下降
- 即使分库分表，热门内容的行锁仍是瓶颈

### 2.2 Redis 原子计数

**Redis INCR** 是原子操作，单机可达 10 万 QPS。

```redis
INCR note:like:123      → 点赞数 +1
INCRBY note:view:123 10  → 阅读数 +10（批量累积）
```

**Redis 计数的持久化风险**：
- Redis 异步复制可能丢数据（AOF everysec 丢最多 1 秒）
- 如果 Redis 故障，计数回退

**解决方案**：Redis 计数 + MySQL 定期对账。

```
用户点赞：
  Redis INCR like:123           → 实时计数
  MQ 发送 like_event             → 异步持久化
  MySQL UPDATE COUNT + 1        → 最终一致

对账任务（每 10 分钟）：
  比较 Redis 和 MySQL 的计数
  不一致 → 用 Redis 的值更新 MySQL
```

### 2.3 写聚合：减少 Redis 写入次数

**问题**：阅读数每次 +1 太频繁，Redis 写 QPS 可能成为瓶颈。

**写聚合**：本地内存累积一段时间，再批量写入 Redis。

```java
// 本地聚合计数器
ConcurrentHashMap<String, AtomicInteger> localCounters = new ConcurrentHashMap<>();

public void incrView(String noteId) {
    localCounters.computeIfAbsent(noteId, k -> new AtomicInteger()).incrementAndGet();
}

// 定时任务：每 5 秒批量写入 Redis
@Scheduled(fixedDelay = 5000)
public void flushToRedis() {
    localCounters.forEach((noteId, counter) -> {
        int delta = counter.getAndSet(0);
        if (delta > 0) {
            try {
                redis.incrBy("note:view:" + noteId, delta);
            } catch (Exception e) {
                counter.addAndGet(delta);  // Redis 写入失败 → 加回本地计数器，下轮重试
            }
        }
        if (counter.get() == 0) {
            localCounters.remove(noteId);  // 清零后移除，避免内存泄漏
        }
    });
}
```

**写聚合的效果**：
- 阅读数 QPS=50000 → 5 秒聚合一次 → Redis 写 QPS=50（降低 1000 倍）
- **代价**：计数不是实时的（最多 5 秒延迟）

### 2.4 冷热分离

**核心思想**：热数据用 Redis（内存，贵），冷数据用 MySQL（磁盘，便宜）。

```
热数据（最近 7 天有更新的）→ Redis
冷数据（超过 7 天未更新的）→ MySQL
```

**热数据升冷策略**：
- Redis key 设置 TTL=7 天
- 每次写入时刷新 TTL
- 7 天内无写入 → key 过期 → 下次读取从 MySQL 加载（提升为热数据）

**冷数据的读取**：
- Redis 未命中 → 查 MySQL → 回写 Redis（TTL=7 天）

### 2.5 计数系统的"类 Reddit 赞踩"

**点赞/踩的互斥操作**：
- 用户点"赞" → 取消之前的"踩"
- 用户点"踩" → 取消之前的"赞"

**Redis Hash 存储**：
```
HSET note:like:123 userId1 1      # userId1 点赞
HSET note:like:123 userId2 -1     # userId2 点踩
HGET note:like:123 userId1        # 查询用户的态度

点赞数 = HSCAN 统计 value=1 的数量（或用额外的计数器维护）
点踩数 = HSCAN 统计 value=-1 的数量
```

**用独立的计数器维护赞/踩总数**（避免 HSCAN 大量成员时全量遍历）：
```
用户点赞：HSET + INCR like_count + (之前踩过 → DECR dislike_count)
用户取消赞：HDEL + DECR like_count
```
> 注：直接用 HSCAN 遍历 Hash 统计赞/踩数在大成员量时效率极低，必须用独立计数器（like_count/dislike_count）维护。

### 2.6 本节小结

计数系统的核心手段：Redis 原子计数（INCR/INCRBY）+ 写聚合（本地累积 5s → 批量刷 Redis）+ 冷热分离（7 天 TTL → Redis/MySQL 双层）+ MySQL 定期对账（10 分钟）。

## 3. 排行榜服务

### 3.1 排行榜的核心需求

| 需求 | 说明 | 实现 |
|------|------|------|
| **实时更新** | 积分变化后排行榜立刻反映 | Redis Sorted Set ZINCRBY |
| **TopN 查询** | 查询前 K 名 | ZREVRANGE 0 K-1 |
| **排名查询** | 查询某个用户的排名 | ZREVRANK |
| **历史榜单** | 昨天的排行榜 | 定时固化到 MySQL |

### 3.2 Redis Sorted Set 实现实时排行榜

**添加/更新分数**：`ZADD leaderboard 100 user1`

```redis
ZADD leaderboard 100 user1    # user1 得分 100
ZADD leaderboard 200 user2    # user2 得分 200
ZADD leaderboard 150 user3    # user3 得分 150
```

**TopN 查询**：`ZREVRANGE leaderboard 0 9 WITHSCORES`

```redis
ZREVRANGE leaderboard 0 9 WITHSCORES
→ [user2, 200, user3, 150, user1, 100]
```

**排名查询**：`ZREVRANK leaderboard user1`

```redis
ZREVRANK leaderboard user1  → 2（第 3 名，排名从 0 开始）
```

### 3.3 排行榜的延时更新

**问题**：实时更新排行榜（用户每得 1 分就 ZINCRBY）的 Redis 写 QPS 可能很高（如 10 万 QPS）。

**延时更新方案**：
- 用户得分变更 → 先写本地聚合（每 10 秒批量 ZINCRBY）
- TopN 榜单有一定延时（10 秒）——可接受

**延时更新的分层**：
| 榜单类型 | 更新频率 | 延时 | 适用 |
|---------|---------|------|------|
| 实时榜 | 每次得分变更立即更新 | 0s | 小规模榜单（< 1 万用户） |
| 近实时榜 | 10s 聚合更新 | 10s | 中等规模（< 100 万用户） |
| 历史榜 | 每天固化一次 | 24h | 大规模/长期榜单 |

### 3.4 历史榜单的固化

**为什么要固化？** 昨天的排行榜不能依赖 Redis（可能过期或数据量太大）。

**固化流程**：
```
每天 00:00：
  1. Redis ZREVRANGE 导出现时 TopN
  2. 写入 MySQL 历史榜表
  3. Redis 清空当天榜单（或保留最近 N 天）
```

**MySQL 历史榜单表**：
```sql
CREATE TABLE leaderboard_history (
    id BIGINT PRIMARY KEY,
    date DATE NOT NULL,
    user_id BIGINT NOT NULL,
    score INT NOT NULL,
    rank INT NOT NULL,
    KEY idx_date (date)
);
```

### 3.5 性能优化

**ZREVRANGE 大榜单的性能问题**：
- Top 100 从 100 万用户的榜单：O(log(N)+M) ≈ O(17+100) ≈ 很快
- Top 10000 从 1000 万用户的榜单：O(24+10000) ≈ 仍可接受
- 全量导出时（ZREVRANGE 0 -1）：O(N) 遍历全部成员 → 需要分页

**大榜单的优化**：
- 分桶：把用户分成 100 个桶（按 score 范围），每个桶独立 Sorted Set
- 查询时先在最高分的桶里查，不够再查下一个桶

### 3.6 本节小结

排行榜的工程实现：Redis Sorted Set（实时）+ 本地聚合（延时 10s 降低写 QPS）+ MySQL 定时固化（历史榜）。大榜单用分桶优化。

## 结语：从基础服务到核心业务

回顾本篇，三个核心服务展示了**高频操作的工程优化**：

- **内容发布**：DFA O(n) 敏感词 + 审核三级 + Fanout（Push+大V Pull）
- **计数系统**：Redis INCR + 写聚合 + 冷热分离 + MySQL 对账
- **排行榜**：Redis Sorted Set + 延时聚合 + 定时固化 + 分桶

**第三篇的核心论断**：内容/计数/排行榜是 UGC 平台的三大核心服务。它们的共同模式是：**Redis 扛住高频读写 + 异步/聚合降低峰值 + MySQL 兜底保证持久**。亿级流量下，不是"Redis 够了"或"MySQL 够了"，而是两者分层配合。
