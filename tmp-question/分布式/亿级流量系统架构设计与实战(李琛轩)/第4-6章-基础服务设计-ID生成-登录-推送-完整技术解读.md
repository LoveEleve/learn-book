## 1. 唯一 ID 生成器

### 1.1 为什么需要分布式 ID 生成器？

单机数据库的自增主键（`AUTO_INCREMENT`）在分布式场景下失效——分库分表后，每个分片独立自增，ID 会冲突。

**分布式 ID 的四个硬性要求**：

| 要求 | 说明 | 反面案例 |
|------|------|---------|
| **全局唯一** | 所有分片、所有服务生成的 ID 不能重复 | 两个用户注册获得同一个 user_id |
| **趋势递增** | ID 按时间趋势递增（不要求严格递增） | MySQL InnoDB 主键乱序插入导致页分裂 |
| **高性能** | 生成 ID 不能成为瓶颈（> 10 万 QPS） | 每次生成 ID 查数据库 → 数据库成为瓶颈 |
| **高可用** | ID 生成服务不能成为单点 | ID 生成服务挂了 = 全站不可用 |

### 1.2 Snowflake 算法

**结构**（64 位整数）：

```
┌─────────────┬───────────┬──────────┐
│  timestamp  │ workerId  │ sequence │
│   41 bit    │  10 bit   │  12 bit  │
└─────────────┴───────────┴──────────┘
```

- **timestamp（41 bit）**：毫秒级时间戳，从自定义起始时间（epoch）算起。可以使用 69 年。
- **workerId（10 bit）**：机器 ID。需人工分配（避免重复）。最多 1024 台机器。
- **sequence（12 bit）**：同一毫秒内的计数器。每毫秒最多 4096 个 ID。

**QPS 上限**：4096 / ms = **409 万 QPS**（理论极限，实际受限于时钟精度和 workerId 分配）。

**Snowflake 的"时钟回拨"问题**（详见唐伟志第 6 章 物理时钟局限）：
- NTP 校时可能使时钟回拨（step 模式）
- 回拨后生成的 timestamp 可能小于之前的 → ID 重复
- **三种应对**：
  1. 短回拨（< 5ms）：等待追上（最多等 5ms）
  2. 长回拨：借用未来时间戳
  3. 拒绝服务：宁可不可用也不生成冲突 ID

**Snowflake 的工程实现**（百度 UidGenerator / 美团 Leaf / my-xhs Cosid）：
```java
public synchronized long nextId() {
    long currentTime = System.currentTimeMillis();
    if (currentTime < lastTimestamp) {
        long offset = lastTimestamp - currentTime;
        if (offset <= 5) {
            Thread.sleep(offset);  // 短回拨：等追上
            currentTime = System.currentTimeMillis();
        } else {
            throw new RuntimeException("Clock moved backwards!");  // 长回拨：拒绝
        }
    }
    if (currentTime == lastTimestamp) {
        sequence = (sequence + 1) & sequenceMask;
        if (sequence == 0) {
            currentTime = waitNextMillis(lastTimestamp);  // 序列号用完：等到下一 ms
        }
    } else {
        sequence = 0;
    }
    lastTimestamp = currentTime;
    return (currentTime - epoch) << timestampShift
         | workerId << workerIdShift
         | sequence;
}
```

### 1.3 号段模式（号段模式 Leaf）

**Snowflake 的问题**：强依赖机器时钟。时钟回拨即使在短时间被恢复，仍可能导致 ID 不严格趋势递增。

**号段模式**：从数据库批量获取一段 ID，本地缓冲。用完后再去数据库取下一段（批量获取，减少 DB 交互）。

```
数据库号段表（t_id_segment）：
┌──────────┬──────┬──────┐
│ biz_tag  │ max_id │ step │
├──────────┼──────┼──────┤
│ user     │ 10000 │ 1000 │  ← user 的号段：10001~11000
│ order    │ 20000 │ 1000 │  ← order 的号段：20001~21000
└──────────┴──────┴──────┘

获取号段：UPDATE t_id_segment SET max_id = max_id + step WHERE biz_tag = 'user'
返回：旧 max_id + 1 ~ 新 max_id（如 10001 ~ 11000）
```

**双 Buffer 优化**（美团 Leaf 的关键设计）：
- Buffer1：当前使用的号段（如 10001~11000）
- Buffer2：预加载的下一号段（如 11001~12000）
- Buffer1 用到 90% → 触发异步加载 Buffer3（如 12001~13000）
- Buffer1 用完 → 立刻切换到 Buffer2 → 无停顿

**Snowflake vs 号段模式**：

| 维度 | Snowflake | 号段模式 |
|------|-----------|---------|
| 唯一性 | workerId + sequence | DB 唯一约束保证 |
| 性能 | 极高（纯内存计算） | 高（内存分配 + DB 批量预取） |
| 趋势递增 | 近似递增（受时钟影响） | 严格递增（号段连续） |
| 时钟依赖 | **强依赖**（回拨导致问题） | **不依赖** |
| 实现复杂度 | 中（需处理回拨） | 中（需 DB + 双 Buffer） |
| QPS 上限 | 409 万/秒 | 取决于 step 大小（step=10000 → DB QPS = 实际 QPS / 10000） |

### 1.4 生产建议

- **QPS < 1 万**：Snowflake 足够（时钟回拨概率低，即使发生影响也小）
- **QPS 1 万~10 万**：Snowflake + 短回拨等待（容忍 < 5ms 回拨）
- **QPS > 10 万**：号段模式 + 双 Buffer（美团 Leaf）
- **跨机房部署**：号段模式更适合（不依赖本地时钟同步）

### 1.5 本节小结

分布式 ID 的两条路径：Snowflake（纯内存，性能最高但强依赖时钟）和号段模式（DB 批量分配，时钟无关但需要 DB 高可用）。选型取决于 QPS 级别和时钟可靠性。

## 2. 用户登录服务

### 2.1 登录服务的核心挑战

**挑战 1：密码安全**
- 绝对不能存明文密码
- 哈希 + 盐（Salt）防彩虹表攻击
- 自适应哈希函数（bcrypt/scrypt/Argon2）防暴力破解

**挑战 2：会话管理**
- HTTP 是无状态的——如何知道"这个请求来自已登录的用户"？
- JWT vs Session 的选型

**挑战 3：多设备登录**
- 同一用户在手机和电脑同时登录
- 修改密码后所有设备强制下线

**挑战 4：SSO 单点登录**
- 企业内多个系统共享登录状态
- 一个系统登录，其他系统自动识别

### 2.2 密码存储方案

**bcrypt 加盐哈希**（Spring Security 默认）：
```java
PasswordEncoder encoder = new BCryptPasswordEncoder();
String hashed = encoder.encode("userPassword123");
// $2a$10$xxxxxxxxxxxxx（自动包含随机 salt）
```

**为什么不能用 SHA-256 直接哈希？**
- SHA-256 是快速哈希，GPU 可以每秒计算数十亿次
- bcrypt 是慢哈希，故意减慢计算速度（cost factor=10 → 约 100ms 一次）
- 暴力破解难度：bcrypt 比 SHA-256 高 6 个数量级

### 2.3 JWT vs Session

| 维度 | Session | JWT |
|------|---------|-----|
| 存储 | 服务端（Redis/内存） | 客户端（HTTP Header） |
| 状态 | 有状态（服务端存 session） | 无状态（token 自包含） |
| 扩展性 | 需要共享 Session 存储 | 天然支持（任何节点可验签） |
| 注销难度 | 简单（删 session） | 困难（token 签发后无法主动失效） |
| 适合场景 | 单体/少量服务（< 10） | 微服务（> 10 个服务） |

**JWT 注销的解决方案**：
- Token 黑名单（Redis 存已注销的 token ID，TTL=token 过期时间）
- 短 TTL + Refresh Token（access token 5 分钟过期，用 refresh token 换新）

### 2.4 SSO 单点登录

**OAuth 2.0 授权码模式**（最安全的 SSO 方案）：

```
用户 → 浏览器 → 应用 A → 重定向到 SSO 登录页
  SSO 验证用户 → 返回授权码 → 应用 A 用授权码换 token
  应用 A 用 token 获取用户信息 → 完成登录

用户再访问应用 B：
  应用 B → 重定向到 SSO → SSO 发现用户已登录 → 直接返回授权码
  应用 B 用授权码换 token → 完成登录（用户无感知）
```

**SSO 会话同步**：用户在应用 A 修改密码 → SSO 通知所有已登录的应用"此用户需要重新认证"。

### 2.5 多设备管理

**同一用户多设备登录**：
- 每个设备独立签发 JWT（device_id 作为 token 的一部分）
- 用户可以在"设备管理"页面查看和踢出特定设备

**踢出设备的实现**：
- Redis 存 `user:device:123:status = ACTIVE`
- 用户踢出设备 123 → `user:device:123:status = KICKED`
- 设备 123 的下一次请求 → 网关检查 status → 发现 KICKED → 拒绝

### 2.6 本节小结

登录服务的核心：密码安全（bcrypt）、会话管理（JWT vs Session）、SSO 单点登录（OAuth 2.0 授权码模式）、多设备管理（按设备签发 + 黑名单踢出）。

## 3. 海量推送系统

### 3.1 推送系统的核心挑战

**挑战 1：海量长连接**
- 千万级设备同时在线
- 每台服务器能支撑的长连接数有限（C10K→C100K→C1000K）

**挑战 2：心跳与连接保活**
- 移动网络不稳定（WiFi↔4G 切换）
- NAT 超时会断开空闲连接

**挑战 3：在线/离线状态管理**
- 如何快速判断用户是否在线？
- 离线消息如何处理？

### 3.2 单机长连接上限

**为什么一台服务器不能撑无限连接？**

| 瓶颈 | C10K（1 万连接） | C100K（10 万连接） | C1000K（100 万连接） |
|------|----------------|-----------------|-------------------|
| 线程模型 | 必须用 NIO（不能每连接一个线程） | 必须用 Netty（EventLoop 模式） | Netty + 内核参数调优 |
| 内存 | ~100MB | ~1GB | ~10GB |
| 文件描述符 | 正常（默认 1024 需调高） | 需设置 `ulimit -n 200000` | 需 `ulimit -n 2000000` |
| 带宽 | 正常 | 需千兆网卡 | 需万兆网卡 |

**Netty 推送网关的内存计算**：
每个连接占用的内存 ≈ 1KB（SocketChannel + 缓冲区） + 业务开销（userId/设备信息等 ≈ 500B） = **~1.5KB**。100 万连接 × 1.5KB = **~1.5GB**。

### 3.3 心跳保活

**为什么需要心跳？**
- NAT 网关会清理长时间无数据的 TCP 连接（通常 2-5 分钟）
- 心跳确保连接不会被 NAT 清理

**心跳间隔的选择**：
| 间隔 | 保活效果 | 电量和带宽消耗 |
|------|---------|-------------|
| 30s | 好 | 较高 |
| 60s | 好 | 中等（推荐） |
| 120s | NAT 可能已清理 | 低 |
| 300s | NAT 大概率已清理 | 最低 |

**建议**：客户端心跳间隔 60s，服务端超时 180s（3 个心跳周期未收到 → 判定断线）。

**心跳优化**（降低服务器压力）：
- 不是每个连接独立计时——用时间轮（HashedWheelTimer）批量管理
- Netty `IdleStateHandler（180, 0, 0）` 自动检测读空闲

### 3.4 在线状态管理

**用户在线状态的存储**：
- Redis 存 `user:online:{userId} = {gateway_ip}:{channel_id}`
- TTL = 心跳超时（180s）
- 客户端每次心跳刷新 TTL

**判断用户是否在线**：
- `EXISTS user:online:{userId}` → 存在 = 在线，不存在 = 离线（或心跳超时）

**用户上下线通知**：
- 用户上线 → 通知其好友/关注者（"XXX 上线了"）
- 用户离线 → TTL 过期后通知（可能有 180s 延迟）

### 3.5 离线消息

**用户离线时的消息处理**：
1. 推送消息到达网关
2. 网关检查 Redis：`user:online:{userId}` 是否存在
3. 在线 → 直接推送
4. 离线 → 消息存入 `user:offline_msg:{userId}` 列表（Redis List）+ 发 MQ 延迟消息

**用户上线后的离线消息拉取**：
1. 建立连接 → 检查 `user:offline_msg:{userId}`
2. 有离线消息 → 批量推送 → 清空列表

### 3.6 千万级连接的架构

```
           移动设备（千万级）
                │
        ┌───────┴───────┐
        │   DNS / LVS   │  ← 四层负载均衡
        └───────┬───────┘
    ┌───────────┼───────────┐
    ↓           ↓           ↓
 网关1        网关2        网关3
(100万连接)  (100万连接)  (100万连接)
    │           │           │
    └───────────┼───────────┘
                │
        ┌───────┴───────┐
        │    Redis      │  ← 在线状态 + 离线消息
        └───────────────┘
```

**水平扩展的关键**：
- 每台网关 100 万连接，10 台网关 = 1000 万连接
- 一致性哈希确保同一用户总是连接到同一网关（减少状态迁移）
- 新网关加入时只影响部分用户

### 3.7 本节小结

推送系统三核心：长连接（Netty 单机百万连接）、心跳保活（客户端 60s / 服务端 180s 超时）、离线消息（Redis 存储 + 上线拉取）。水平扩展：一致性哈希 + 多网关。

## 结语：三个基础服务的共性

回顾本篇，三个基础服务展示了**场景驱动的架构设计**：

- **ID 生成器**：Snowflake（纯内存，强依赖时钟）vs 号段模式（DB 批量，时钟无关）
- **登录服务**：bcrypt 哈希 / JWT vs Session / OAuth 2.0 SSO / 多设备管理
- **推送系统**：Netty 百万连接 / 心跳保活 60s / 离线消息 / 水平扩展

**第二篇的核心论断**：基础服务是互联网后台的"共享基础设施"——它们不直接承载业务逻辑，但所有业务都依赖它们。每个基础服务的设计都围绕一个核心权衡：ID 生成器的"性能 vs 时钟依赖"，登录服务的"无状态 vs 注销便利"，推送系统的"连接数 vs 内存开销"。
