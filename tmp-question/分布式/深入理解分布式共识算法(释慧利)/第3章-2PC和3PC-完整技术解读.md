## 3.1 问题引入——从 ACID 崩塌到 2PC 的动机

### 3.1.1 第2章的结论：ACID 在分布式下崩塌

第2章我们展示了 ACID 的四个保证在分布式环境下逐一崩塌。其中最致命的是**原子性崩塌**——跨节点的转账事务中，Node-1 提交成功、Node-2 提交失败，导致"部分提交"——钱凭空消失或凭空产生。

这个问题的本质是：**多个节点没有共享的 undo log，无法像单机那样"要么全做要么全回滚"**。

### 3.1.2 直觉方案：选一个"总指挥"

既然多个节点各自为政会导致混乱，最直觉的方案是引入一个"总指挥"——**协调者（Coordinator）**：

- 协调者负责统一调度所有节点（**参与者 Participant**）的事务
- 参与者执行完毕后向协调者汇报
- 协调者根据所有参与者的汇报，决定"全局提交"还是"全局回滚"

这就是 **2PC（Two-Phase Commit，两阶段提交）** 的出发点。它最早由 Jim Gray 在 1978 年的论文《Notes on Data Base Operating Systems》中系统化描述，此后成为关系型数据库 XA 协议（XA = eXtended Architecture，X/Open DTP 标准定义的分布式事务协议，如 MySQL XA、Oracle XA）的理论基础。

### 3.1.3 2PC 的核心思想

2PC 的核心思想可以用一句话概括：**先问 everyone 能不能做，如果能就通知 everyone 做；如果有任何一个不能，就通知 everyone 别做**。

这看起来非常简单——就是"先投票再执行"。但分布式系统的故障（节点宕机、网络分区、消息丢失）会让这个简单流程变得异常复杂。让我们逐步拆解。

## 3.2 2PC 完整推演

### 3.2.1 角色定义

2PC 有两个角色：

| 角色 | 职责 | 生产对应 |
|------|------|---------|
| **协调者（Coordinator）** | 发起事务、收集投票、决定提交/回滚、驱动执行 | Seata TC（Transaction Coordinator）、XA 事务管理器 |
| **参与者（Participant）** | 执行分支事务、汇报状态、提交/回滚本地事务 | Seata RM（Resource Manager）、XA 资源管理器（如 MySQL InnoDB） |

### 3.2.2 阶段一：准备阶段（Prepare Phase）

准备阶段又称**投票阶段（Vote Request）**。协调者向所有参与者发送 Prepare 请求，询问"能否提交？"参与者执行本地事务但不提交，锁定资源后回复 Yes/No。

```
协调者                              参与者-1    参与者-2
   │                                   │           │
   │──── Prepare(事务内容) ───────────►│           │
   │──── Prepare(事务内容) ───────────────────────►│
   │                                   │           │
   │                        执行本地事务(不提交)   │
   │                        记录 undo/redo log      │
   │                        锁定资源                │
   │                                   │           │
   │◄──── Yes ────────────────────────│           │
   │◄──── Yes ────────────────────────────────────│
   │                                   │           │
   │ 收到所有 Yes → 进入阶段二         │           │
   │ 收到任一 No 或超时 → 中断事务     │           │
```

**参与者在准备阶段做了什么**：
1. 开启本地事务
2. 执行 SQL（如扣款、加款）
3. 记录 undo log（用于回滚）和 redo log（用于提交）
4. **锁定资源**（行锁、间隙锁等）——这是 2PC 性能问题的根源
5. **不提交**——保持事务在"准备"状态
6. 回复 Yes（可以提交）或 No（执行失败，不能提交）

**关键点**：参与者在回复 Yes 后，就进入了一种"待命"状态——它已经锁定了资源，但还不能提交，必须等待协调者的第二阶段指令。

### 3.2.3 阶段二：提交/回滚阶段（Commit/Abort Phase）

协调者根据阶段一的投票结果决定全局提交或全局回滚：

**情况一：所有参与者都回复 Yes → 全局提交**

```
协调者                              参与者-1    参与者-2
   │                                   │           │
   │──── Commit ─────────────────────►│           │
   │──── Commit ───────────────────────────────────►│
   │                                   │           │
   │                        提交本地事务             │
   │                        释放锁定资源             │
   │                                   │           │
   │◄──── Ack ────────────────────────│           │
   │◄──── Ack ────────────────────────────────────│
   │                                   │           │
   │ 事务完成                          │           │
```

**情况二：任一参与者回复 No 或超时 → 全局回滚**

```
协调者                              参与者-1    参与者-2
   │                                   │           │
   │──── Rollback ───────────────────►│           │
   │──── Rollback ───────────────────────────────────►│
   │                                   │           │
   │                        回滚本地事务(undo log)   │
   │                        释放锁定资源             │
   │                                   │           │
   │◄──── Ack ────────────────────────│           │
   │◄──── Ack ────────────────────────────────────│
   │                                   │           │
   │ 事务回滚完成                      │           │
```

### 3.2.4 正常流程的完整时间线

以跨节点转账（A 在 Node-1，B 在 Node-2）为例：

```
T0: Client 向协调者发起转账请求(A→B, 100元)
T1: 协调者向 Node-1、Node-2 发送 Prepare 请求
T2: Node-1 执行 "A 扣 100"（不提交，记录 undo log，锁定 A 的行）
T3: Node-2 执行 "B 加 100"（不提交，记录 undo log，锁定 B 的行）
T4: Node-1 回复 Yes
T5: Node-2 回复 Yes
T6: 协调者收到所有 Yes，决定全局提交
T7: 协调者向 Node-1、Node-2 发送 Commit
T8: Node-1 提交本地事务（A 余额正式扣减），释放锁
T9: Node-2 提交本地事务（B 余额正式增加），释放锁
T10: Node-1、Node-2 回复 Ack
T11: 协调者向 Client 返回"转账成功"
```

在无故障的情况下，2PC 是清晰且正确的。但分布式系统从不存在"无故障"——下一节我们推演四种故障场景。

## 3.3 2PC 故障分析——四种异常场景

### 3.3.1 场景一：参与者在收到 Commit 后、提交并 Ack 前崩溃

**时间线**：

```
T0-T5: 正常执行 Prepare 阶段，Node-1 和 Node-2 都回复 Yes
T6:   协调者收到所有 Yes，决定提交
T7:   协调者向 Node-1、Node-2 发送 Commit
T8:   Node-1 收到 Commit，提交本地事务，回复 Ack
T9:   Node-2 收到 Commit，但在此刻崩溃（未提交、未回复 Ack）
T10:  协调者等待 Node-2 的 Ack 超时
```

**问题**：协调者不知道 Node-2 是"已提交但 Ack 丢失"还是"未提交"。协调者只能**无限重试**发送 Commit 给 Node-2，直到 Node-2 恢复。

**Node-2 恢复后的行为**：Node-2 重启后，检查本地日志——发现自己处于"Prepare 完成、未 Commit"状态。此时 Node-2 **不知道该提交还是回滚**——它只能等待协调者重新发送 Commit（或其他指令）。

**关键问题**：Node-2 在恢复期间，它在 Prepare 阶段锁定的资源**一直无法释放**——这些行被锁定，其他事务无法访问。

### 3.3.2 场景二：协调者在收到所有 Yes 后、发送 Commit 前崩溃

这是 2PC 最致命的故障——**协调者崩溃导致所有参与者阻塞**。

**时间线**：

```
T0-T5: 正常执行 Prepare 阶段，所有参与者回复 Yes
T6:   协调者收到所有 Yes，决定提交——但在此刻崩溃
       （未来得及发送 Commit，也来不及记录"决定提交"到日志）
T7:   所有参与者等待 Commit 指令——永远等不到
T8:   参与者锁定的资源无法释放
T9:   其他事务被阻塞（等待这些资源）
T10:  系统局部甚至全局不可用
```

**问题根源**：参与者无法自行决定——它只知道"自己 Prepare 成功了"，但不知道"其他参与者是否 Prepare 成功"，更不知道"协调者是否决定了提交"。

**为什么参与者不能自己决定？**
- 如果参与者单方面提交：万一其他参与者收到的是 No，协调者本应回滚——此时数据不一致
- 如果参与者单方面回滚：万一其他参与者已经提交——此时数据不一致
- **参与者没有足够的信息做出安全决策**——这就是 2PC 的"阻塞"本质

**新协调者接管能解决吗？** 假设有一个备份协调者接管，它需要：
1. 读取原协调者的日志，看是否有"决定提交"或"决定回滚"的记录
2. 如果没有记录——新协调者也不知道该提交还是回滚
3. 新协调者可以询问所有参与者——但参与者只知道自己的状态，不知道全局决策
4. 如果所有参与者都说"Yes，我 Prepare 成功了"——新协调者能安全提交吗？不能！因为原协调者可能在崩溃前已经收到了某个 No 并决定了回滚，只是还没发出去

**这就是 2PC 的根本性阻塞问题**——协调者在"决定"和"发送"之间崩溃，会导致所有参与者无限期阻塞。

### 3.3.3 场景三：网络分区导致部分参与者收不到 Commit

**时间线**：

```
T0-T6: 正常执行 Prepare + 协调者决定提交
T7:   协调者发送 Commit 给 Node-1、Node-2、Node-3
T8:   Node-1 收到 Commit，提交，回复 Ack
T9:   网络分区，Node-2、Node-3 收不到 Commit
T10:  Node-1 已提交（A 余额扣减生效）
T11:  Node-2、Node-3 仍处于 Prepare 状态（资源锁定）
T12:  协调者等待 Node-2、Node-3 的 Ack 超时
T13:  协调者重试发送 Commit——但网络仍分区
```

**问题**：
- Node-1 已提交，数据变更对外可见
- Node-2、Node-3 未提交，数据变更不可见
- 从 Client 视角：转账"部分完成"——A 扣了钱，B 没收到
- 如果此时有其他事务读取数据，会看到不一致状态

**这就是 2PC 的"数据不一致"问题**——网络分区导致部分节点提交、部分节点未提交，全局数据不一致。

### 3.3.4 场景四：空回滚与防悬挂

这是一个更隐蔽的故障——消息乱序导致的异常。

**空回滚（Empty Rollback）**：

```
T0: 协调者发送 Prepare 给 Node-1
T1: Prepare 消息在网络中延迟/丢失
T2: 协调者超时，发送 Rollback 给 Node-1
T3: 网络恢复，Node-1 先收到 Rollback（但从未收到过 Prepare）
T4: Node-1 收到一个"没有对应 Prepare 的 Rollback"——该怎么办？
```

**问题**：Node-1 从未执行过 Prepare，现在收到 Rollback，如果直接报错或不做处理，可能导致事务状态混乱。

**解决方案**：引入全局唯一的事务 XID（Transaction ID）。协调者在每个消息中携带 XID，参与者在 Prepare 阶段记录 XID 到日志。收到 Rollback 时先查日志——如果 XID 不在日志中，说明是空回滚，直接返回 Ack 即可。

**防悬挂（Prevent Suspended）**：

```
T0: 协调者发送 Prepare 给 Node-1
T1: Prepare 消息在网络中严重延迟
T2: 协调者超时，发送 Rollback 给 Node-1
T3: Node-1 收到 Rollback，执行空回滚（记录 rollback-log）
T4: 延迟的 Prepare 终于到达 Node-1
T5: Node-1 收到一个"已经回滚过的 Prepare"——该怎么办？
```

**问题**：Node-1 已经对 XID 执行了回滚，现在又收到 Prepare——如果执行 Prepare，会锁定资源并等待 Commit，但协调者不会再发 Commit（它认为事务已回滚）。Node-1 会**永久阻塞**——这就是"悬挂"。

**解决方案**：参与者在收到 Prepare 时，先查 rollback-log——如果 XID 已在 rollback-log 中，说明事务已回滚，拒绝执行 Prepare。

> **跨书对照**：空回滚和防悬挂是 2PC 的工程实现细节。释慧利原书给出了伪代码。唐伟志《深入理解分布式系统》第5章从"分布式事务"角度讲解时，将这些归入"2PC 的工程化挑战"。Seata AT 模式通过 branch_table（分支事务表）和 global_table（全局事务表）解决了这些问题——我们将在 3.7 节展开。

## 3.4 2PC 的致命缺陷——为什么它不是终点

### 3.4.1 缺陷一：同步阻塞

**现象**：参与者在 Prepare 阶段锁定资源后，必须等待协调者的第二阶段指令才能释放。如果协调者宕机，资源**永久锁定**（直到协调者恢复或新协调者接管）。

**后果**：
- 被锁定的行无法被其他事务访问
- 如果锁定的是热点数据（如商品库存），整个业务的并发能力归零
- 如果协调者长时间不恢复，业务实质停摆

**根因**：2PC 用"阻塞"换"一致性"——参与者在不确定全局决策时，宁可阻塞也不做任何操作。这是安全的（不会数据不一致），但代价是可用性。

### 3.4.2 缺陷二：数据不一致

**现象**：协调者发送 Commit 后，部分参与者收到并提交，部分参与者因网络分区未收到。结果是部分节点已提交、部分节点未提交。

**后果**：
- Client 读到不一致的数据（部分节点是新值，部分是旧值）
- 如果读请求路由到已提交的节点，看到"转账成功"；路由到未提交的节点，看到"转账未完成"
- 违反了 ACID 的原子性——事务"部分提交"了

**根因**：2PC 的第二阶段不是原子的——Commit 消息的发送和参与者的执行是分步的，任何中间步骤的故障都会导致不一致。

### 3.4.3 缺陷三：单点问题

**现象**：协调者是单点——如果协调者宕机且没有备份，所有进行中的事务都会阻塞（参与者无法得到第二阶段指令）。

**后果**：
- 协调者宕机 = 系统不可用
- 即使有备份协调者，接管过程也需要时间（日志恢复、状态重建）
- 在接管期间，所有参与者阻塞

**根因**：2PC 的设计中心化——所有决策依赖协调者。这与分布式系统"去中心化"的初衷相悖。

### 3.4.4 缺陷四：性能问题

**现象**：2PC 的两阶段 + 资源锁定 = 极低的吞吐。

**具体代价**：
1. 阶段一：参与者执行事务 + 锁定资源（整个阶段资源被锁）
2. 阶段一→阶段二的网络往返（协调者收集所有 Yes）
3. 阶段二：协调者发送 Commit + 参与者提交 + Ack 返回
4. 整个过程中资源被锁定——其他事务等待

**性能估算**：
- 假设 3 个参与者，网络 RTT（Round-Trip Time，往返时延，一次请求+响应的耗时）1ms，事务执行 5ms（SQL 执行 + 锁获取，不含网络）
- 阶段一：5ms（执行）+ 1ms（回复）= 6ms
- 阶段一→二：1ms（协调者处理）
- 阶段二：1ms（发送 Commit）+ 2ms（提交）+ 1ms（Ack）= 4ms
- 总计：约 11ms/事务
- 资源锁定时间：约 10ms
- 对比单机事务：约 1ms（无网络开销，无跨节点协调）

**2PC 的 TPS（Transactions Per Second，每秒事务数）通常是单机事务的 1/10 到 1/100**——这就是为什么生产环境很少直接使用 XA 事务。

### 3.4.5 根本原因：2PC 是"原子提交协议"不是"共识协议"

2PC 的所有缺陷都指向一个根本原因：**2PC 用"全员一致"和"阻塞"来保证原子性，而不是用"多数派"来保证共识**。

| 维度 | 2PC | 共识算法（Paxos/Raft） |
|------|-----|----------------------|
| 决策方式 | 全员一致（所有参与者必须 Yes） | 多数派（>50% 同意即可） |
| 故障容忍 | 任一节点故障 = 阻塞 | 少数节点故障不影响 |
| 协调者角色 | 单点，崩溃导致阻塞 | Leader，崩溃可重新选举 |
| 资源锁定 | 整个两阶段都锁定 | 仅在应用阶段锁定 |
| 一致性保证 | 原子提交（全做或全不做） | 线性一致性（第1章 1.6.4 节已定义） |

**2PC 要求"所有人同意"才能推进——这在一个可能故障的分布式系统中是不可行的**。共识算法的突破在于：放弃"全员一致"，追求"多数派一致"——少数节点故障不影响全局决策。

这就是为什么 2PC 之后需要 Paxos/Raft——它们不是 2PC 的"改进版"，而是根本范式的转换。

> **跨书对照**：
> - **释慧利（本章）**：从"分布式事务"角度讲 2PC——2PC 是事务原子性的分布式扩展
> - **唐伟志《深入理解分布式系统》第5章**：从"原子提交"角度讲 2PC——将 2PC 与 Paxos Commit 对比，展示"原子提交协议"与"共识协议"的区别

## 3.5 3PC——三阶段提交协议的修复尝试

### 3.5.1 3PC 的设计动机

2PC 的最大问题是"协调者崩溃导致参与者阻塞"。3PC（Three-Phase Commit）的核心目标就是**消除阻塞**——让参与者在协调者宕机时能自行决策。

3PC 的思路是：在 2PC 的两个阶段之间插入一个"预提交"阶段，并引入**超时机制**——参与者在等待超时后可以自主决定提交或中断。

### 3.5.2 3PC 的三个阶段

**阶段一：CanCommit（询问阶段）**

协调者询问参与者"能否执行事务？"参与者做健康检查（连接是否正常、是否有冲突），但**不锁定资源**，只回复 Yes/No。

```
协调者 ──CanCommit──► 参与者
参与者: 健康检查(不锁资源)
参与者 ──Yes/No──► 协调者
```

**阶段二：PreCommit（预提交阶段）**

如果所有参与者回复 Yes，协调者发送 PreCommit。参与者执行事务、记录 undo/redo、锁定资源，但不提交——回复 Ack。

```
协调者 ──PreCommit──► 参与者
参与者: 执行事务 + 锁定资源 + 记录日志(不提交)
参与者 ──Ack──► 协调者
```

**阶段三：DoCommit（提交阶段）**

协调者收到所有 Ack 后发送 DoCommit。参与者提交事务、释放资源、回复 Ack。

```
协调者 ──DoCommit──► 参与者
参与者: 提交事务 + 释放资源
参与者 ──Ack──► 协调者
```

### 3.5.3 3PC 如何解决 2PC 的阻塞问题

3PC 的关键创新是**超时机制**：

| 阶段 | 参与者等待超时后的行为 | 理由 |
|------|---------------------|------|
| CanCommit 后等 PreCommit | **中断事务**（Abort） | 如果协调者没发 PreCommit，说明有参与者回复了 No 或协调者决定不继续——安全中断 |
| PreCommit 后等 DoCommit | **自动提交**（Commit） | 参与者已执行事务并锁定资源，如果协调者没发 DoCommit，可能是协调者崩溃——此时提交是"合理猜测" |

**为什么 PreCommit 后超时可以自动提交？**

因为到了 PreCommit 阶段，说明所有参与者在 CanCommit 阶段都回复了 Yes——全局事务"大概率"会提交。如果协调者在此后崩溃，参与者自行提交虽然不 100% 正确（协调者可能本想回滚），但在大多数情况下是正确的。

### 3.5.4 3PC 的致命问题——网络分区下的数据不一致

3PC 解决了协调者崩溃的阻塞问题，但引入了新的问题——**网络分区下的数据不一致**：

```
T0: 协调者发送 PreCommit 给 Node-1、Node-2
T1: Node-1 和 Node-2 都收到 PreCommit，执行事务，锁定资源，回复 Ack
    （两个参与者都已进入 PreCommit 状态）
T2: 协调者收到所有 Ack，决定提交——发送 DoCommit
T3: 网络分区，Node-1 收到 DoCommit，Node-2 收不到
T4: Node-1 提交事务，释放资源
T5: 协调者因某种原因（如 Node-2 的 Ack 丢失或其他故障）决定 Abort，发送 Abort
T6: 网络恢复，Node-1 收到 Abort——但 Node-1 已经提交了！
T7: 同时 Node-2 等待 DoCommit 超时 → 按 3PC 规则"PreCommit 后超时自动提交"
T8: Node-2 提交事务
T9: 结果：协调者发了 Abort，但 Node-1 和 Node-2 都提交了——数据不一致
```

**根因**：3PC 的超时机制假设"如果参与者进入了 PreCommit 状态，全局事务大概率会提交"。但在网络分区下，协调者可能改变决策（从提交变为 Abort），而参与者超时后自行提交——两者的决策矛盾，导致数据不一致。

### 3.5.5 3PC 的实际命运——为什么生产环境很少用

3PC 在理论上比 2PC 更好（消除了阻塞），但生产环境几乎没人用：

1. **网络分区问题无法解决**：3PC 在网络分区下仍会导致数据不一致——这比 2PC 的"阻塞"更危险（阻塞至少不会数据不一致）
2. **消息开销更大**：3 阶段比 2 阶段多一次网络往返，延迟更高
3. **工程实现更复杂**：超时机制的边界条件极难处理
4. **共识算法更好**：Paxos/Raft 在多数派模型下同时解决了阻塞和数据不一致——3PC 成了"被跳过的中间产物"

**生产环境的实际选择**：
- 需要强一致：用 Paxos/Raft（共识算法）
- 需要业务级强一致：用 TCC（Try/Confirm/Cancel，业务层面两阶段补偿，不依赖数据库锁）
- 需要最终一致：用 Saga（长事务拆分 + 补偿回滚）
- 需要兼容 XA：用 2PC（虽然阻塞，但简单可靠）
- **3PC 成了教科书上的过渡方案**——理论价值大于工程价值

> **跨书对照**：唐伟志《深入理解分布式系统》第5章将 2PC/3PC 归类为"原子提交协议"，与 Paxos Commit（共识算法的提交协议）对比。核心差异：原子提交协议要求"全员同意"，共识算法只要求"多数派同意"——这是从 2PC/3PC 到 Paxos 的根本跨越。

## 3.6 从 2PC/3PC 到共识算法——为什么需要 Paxos/Raft

### 3.6.1 2PC/3PC 的根本局限

通过前两节的分析，2PC/3PC 的根本局限可以归纳为一点：**它们要求"全员同意"才能推进**。

- 2PC：所有参与者必须回复 Yes 才能提交——任一参与者故障 = 阻塞
- 3PC：所有参与者必须回复 CanCommit Yes 才能进入 PreCommit——超时机制虽然缓解了阻塞，但引入了分区下的不一致

**"全员同意"在分布式系统中是不可行的**——因为节点可能宕机、网络可能分区，你永远无法保证"所有节点都在线并响应"。

### 3.6.2 共识算法的突破——多数派替代全员

共识算法（Paxos/Raft/ZAB）的突破在于：**用"多数派同意"替代"全员同意"**。

| 2PC/3PC | 共识算法 |
|---------|---------|
| 所有参与者必须 Yes | 多数派 Yes 即可 |
| 任一节点故障 = 阻塞 | 少数节点故障不影响 |
| 协调者 = 单点 | Leader = 可重新选举 |
| 全员提交才完成 | 多数派提交即完成 |

**为什么多数派可行？** 第1章 1.2.6 节已用鸽笼原理（Pigeonhole Principle：n 个鸽子放入 n-1 个笼子，必有至少一个笼子有 2 个鸽子）证明：3 个节点的两个多数派（各 2 个节点）必有 1 个交集节点——这个交集节点携带了已通过的决议，保证决议能被传递。

### 3.6.3 从"原子提交"到"共识"的范式转换

2PC/3PC 是**原子提交协议（Atomic Commit Protocol）**——解决"多个节点的事务要么全提交要么全回滚"。

共识算法是**共识协议（Consensus Protocol）**——解决"多个节点就某一个值达成一致"。

两者的关系：
- 原子提交可以视为共识的一个特例——"值"是"提交还是回滚"
- 但共识协议比原子提交更强——它不仅决定"提交/回滚"，还保证所有节点对操作顺序达成一致（线性一致性，第1章 1.6.4 节已定义）

**Paxos Commit**：Paxos 可以用于实现原子提交——将"是否提交"作为 Paxos 的提案。这样即使部分节点故障，多数派仍能做出决策——这就是 2PC 到 Paxos 的演进路径。

> **关键洞察**：2PC/3PC 的失败不是实现细节的问题，而是**设计范式的局限**。"全员同意"在故障环境中不可行——只有"多数派同意"才能在保证安全性的同时容忍故障。这就是第4章 Paxos 的起点。

## 3.7 生产实践：Seata AT 模式如何实现 2PC

### 3.7.1 Seata 的角色模型

Seata（Simple Extensible Autonomous Transaction Architecture，阿里开源的分布式事务框架）是 2PC 在生产环境的最重要实践之一。它定义了三个角色：

| 角色 | 全称 | 对应 2PC | 职责 |
|------|------|---------|------|
| **TC** | Transaction Coordinator（事务协调者） | 协调者 | 维护全局事务状态，驱动分支事务提交/回滚 |
| **TM** | Transaction Manager（事务管理者） | 客户端 | 定义全局事务的边界（开始/提交/回滚） |
| **RM** | Resource Manager（资源管理者） | 参与者 | 管理本地资源（如数据库），执行分支事务 |

### 3.7.2 AT 模式——2PC 的工程优化

Seata AT（Auto Transaction）模式是对 2PC 的工程优化，核心改进是**"一阶段本地提交 + 二阶段补偿（提交异步、回滚同步）"**，避免了传统 2PC 的长时间资源锁定。

**传统 2PC 的问题**：阶段一锁定资源直到阶段二——锁定时间 = 两个网络 RTT + 协调者处理时间。

**AT 模式的优化**：

**阶段一（执行 + 本地提交）**：
```
1. TM 向 TC 注册全局事务，获取 XID
2. RM 执行本地 SQL（如 UPDATE account SET balance = balance - 100）
3. RM 自动记录 undo_log（修改前后的镜像）
4. RM 本地事务提交——立即释放锁！
5. RM 向 TC 汇报分支事务状态
```

**关键差异**：AT 模式在阶段一就**提交本地事务并释放锁**——不像传统 2PC 那样保持锁定。这是通过 undo_log 实现的：如果全局事务需要回滚，Seata 根据 undo_log 反向补偿。

**undo_log 的具体结构**——作为技术专家，你需要理解 undo_log 不只是一个概念，它有具体的表结构和数据格式：

```sql
-- Seata AT 的 undo_log 表结构
CREATE TABLE undo_log (
    branch_id     BIGINT      NOT NULL COMMENT '分支事务ID',
    xid           VARCHAR(128) NOT NULL COMMENT '全局事务ID',
    context       VARCHAR(128) NOT NULL COMMENT '上下文（如序列化类型）',
    rollback_info LONGBLOB    NOT NULL COMMENT '回滚信息（修改前后镜像）',
    log_status    INT         NOT NULL COMMENT '状态：0-正常, 1-全局已完成',
    log_created   DATETIME    NOT NULL COMMENT '创建时间',
    log_modified  DATETIME    NOT NULL COMMENT '修改时间',
    UNIQUE KEY ux_undo_log (xid, branch_id)
);
```

**rollback_info 的 JSON 格式**（修改前后的数据镜像）：

```json
{
  "@class": "io.seata.rm.datasource.sql.struct.TableRecords",
  "tableName": "account",
  "rows": [{
    "fields": [
      {"name": "id", "value": 1},
      {"name": "balance", "value": 1000}  // beforeImage：修改前余额=1000
    ]
  }]
},
{
  "@class": "io.seata.rm.datasource.sql.struct.TableRecords",
  "tableName": "account",
  "rows": [{
    "fields": [
      {"name": "id", "value": 1},
      {"name": "balance", "value": 900}   // afterImage：修改后余额=900
    ]
  }]
}
```

**回滚时**：Seata 读取 beforeImage，生成反向 SQL（`UPDATE account SET balance = 1000 WHERE id = 1 AND balance = 900`），执行反向补偿。注意 WHERE 条件包含 `balance = 900`（修改后的值）——如果其他事务已经把 balance 改成了其他值，这个 WHERE 不匹配，说明发生了脏写，Seata 会抛出异常需要人工介入。

**全局锁机制——AT 模式如何实现写隔离**：

AT 模式在阶段一释放了数据库锁，但 Seata 通过 **lock_table**（全局锁表）实现写隔离：

```
1. RM 执行 UPDATE account SET balance = balance - 100 WHERE id = 1
2. RM 在本地事务提交前，向 TC 申请全局锁：lock_key = "account:1"
3. TC 检查 lock_table：
   - 如果 id=1 未被其他全局事务锁定 → 授予锁，记录到 lock_table
   - 如果 id=1 已被其他全局事务锁定 → RM 等待重试
4. RM 获得全局锁后，提交本地事务
5. 阶段二全局提交 → TC 释放全局锁
6. 阶段二全局回滚 → TC 释放全局锁（在 RM 执行完补偿后）
```

**全局锁 vs 数据库锁的区别**：
- 数据库锁：由 InnoDB 管理，事务结束后释放——AT 模式在阶段一就释放了
- 全局锁：由 TC 管理，全局事务结束后释放——保证两个全局事务不会同时修改同一行

**完整交互时序**（TC/TM/RM 三方）：

```
TM                    TC                    RM-1(库存)    RM-2(余额)
 │                     │                       │              │
 │── begin() ─────────►│                       │              │
 │◄── XID ────────────│                       │              │
 │                     │                       │              │
 │── 下单(扣库存+扣款) ──┼──► RM-1 执行          │              │
 │                     │                       │              │
 │                     │◄── branch register ───│              │
 │                     │── branch id ─────────►│              │
 │                     │                       │              │
 │                     │            RM-1 执行 SQL + 记录 undo_log
 │                     │            RM-1 申请全局锁
 │                     │◄── lock request ──────│              │
 │                     │── lock granted ──────►│              │
 │                     │                       │              │
 │                     │            RM-1 本地事务提交(释放 DB 锁)
 │                     │◄── branch status ─────│              │
 │                     │                       │              │
 │                     │              RM-2 执行 SQL + 记录 undo_log
 │                     │              RM-2 申请全局锁
 │                     │◄── lock request ──────┼──────────────│
 │                     │── lock granted ───────┼─────────────►│
 │                     │              RM-2 本地事务提交
 │                     │◄── branch status ─────┼──────────────│
 │                     │                       │              │
 │── commit(XID) ─────►│                       │              │
 │                     │── undo_log cleanup ──►│              │
 │                     │── undo_log cleanup ───┼─────────────►│
 │                     │── lock release ──────►│              │
 │                     │── lock release ───────┼─────────────►│
 │◄── committed ───────│                       │              │
```

**Seata 四种模式对比**——作为技术专家，你需要理解 AT 不是 Seata 的唯一模式：

| 维度 | AT | TCC | Saga | XA |
|------|-----|-----|------|-----|
| **侵入性** | 无（自动代理 SQL） | 高（需写 Try/Confirm/Cancel） | 中（需写补偿逻辑） | 无（依赖数据库 XA） |
| **一致性** | 最终一致 | 业务级强一致 | 最终一致 | 强一致 |
| **隔离性** | 默认读未提交（+全局锁=读已提交） | 业务层隔离 | 无 | 依赖数据库 |
| **性能** | 高 | 中 | 高 | 低 |
| **适用场景** | 电商、O2O | 金融核心 | 长事务（旅行预订） | 兼容已有 XA 系统 |
| **回滚机制** | undo_log 反向补偿 | Cancel 接口 | 补偿操作 | 数据库回滚 |

### 3.7.3 AT 模式 vs 传统 2PC 对比

| 维度 | 传统 2PC (XA) | Seata AT |
|------|-------------|----------|
| 阶段一 | 执行不提交，锁定资源 | 执行并提交，释放锁 |
| 锁定时间 | 整个两阶段 | 仅本地事务执行期间 |
| 回滚机制 | undo log（数据库层） | undo_log（Seata 层，JSON 格式） |
| 隔离性 | 依赖数据库锁 | 默认读未提交（需用 @GlobalLock 保证读已提交） |
| 数据一致性 | 强一致 | 最终一致（阶段一已提交，回滚靠补偿） |
| 性能 | 低（长时间锁定） | 高（短时间锁定） |

**AT 模式的代价**：
1. **隔离性降低**：阶段一已提交，其他事务能看到"未全局确认"的数据——默认是"读未提交"
2. **回滚可能失败**：如果阶段一提交后、阶段二回滚前，其他事务修改了同一行——回滚时发现数据已被覆盖，需要人工介入
3. **undo_log 占用空间**：每个分支事务都要记录修改前后的镜像

### 3.7.4 Seata 的工程启示

Seata AT 的设计体现了 BASE 理论（Basically Available 基本可用、Soft state 软状态、Eventually consistent 最终一致性，第2章 2.3 节已详述）的实践——**用"最终一致 + 补偿"替代"强一致 + 阻塞"**：

- 接受阶段一的"暂时不一致"（本地已提交但全局未确认）
- 通过 undo_log 保证"最终可回滚"
- 牺牲隔离性换取性能

这就是为什么 Seata AT 在电商、O2O、金融外围（如积分、通知）等场景被广泛使用——它在"一致性"和"性能"之间找到了一个工程平衡点。

> **跨书对照**：
> - **释慧利（本节）**：从 2PC 工程实现角度讲 Seata——AT 模式是 2PC 的优化变体
> - **唐伟志《深入理解分布式系统》第5章**：从分布式事务全景角度讲 Seata——将 AT/TCC/Saga/XA 放在同一框架下对比
> - **陈东明《分布式系统与一致性》**：从系统案例角度讲 Seata——与其他分布式事务方案（Spanner/Percolator）对比

## 3.8 本章总结

### 8 个核心要点

1. **2PC 的核心思想是"先投票再执行"**：阶段一（Prepare）收集所有参与者的 Yes/No，阶段二（Commit/Abort）根据投票结果全局提交或回滚。角色分为协调者和参与者。

2. **2PC 在无故障时是正确的**，但分布式系统从不存在"无故障"——四种故障场景（参与者崩溃、协调者崩溃、网络分区、消息乱序）暴露了 2PC 的脆弱性。

3. **2PC 最致命的故障是协调者在"决定"和"发送"之间崩溃**——所有参与者无限期阻塞，无法自行决策。这是 2PC 同步阻塞问题的根源。

4. **2PC 的四大缺陷**：同步阻塞（协调者宕机=资源永久锁定）、数据不一致（网络分区=部分提交）、单点问题（协调者是单点）、性能问题（两阶段+锁定=低吞吐）。

5. **2PC 的根本原因是"用阻塞换一致性"**——参与者在不确定时宁可阻塞也不做决策。这在"全员同意"模型下是必然的。

6. **3PC 通过"预提交阶段 + 超时机制"消除了阻塞**，但引入了网络分区下的数据不一致——参与者超时后自行提交，可能与协调者的实际决策矛盾。

7. **3PC 在生产环境几乎不用**——网络分区问题无法解决，消息开销更大，共识算法（Paxos/Raft）在多数派模型下同时解决了阻塞和不一致。

8. **从 2PC/3PC 到共识算法的根本跨越**：用"多数派同意"替代"全员同意"。2PC/3PC 是原子提交协议，共识算法是共识协议——后者用多数派容忍少数故障，这是第4章 Paxos 的起点。

### 生产实践要点

- **Seata AT** 是 2PC 的工程优化——"一阶段本地提交 + 二阶段异步补偿"，用 undo_log 替代长时间锁定，牺牲隔离性换性能
- **传统 XA 事务**仍在金融场景使用——强一致但低性能
- **3PC 几乎无人使用**——理论价值大于工程价值

### 第4章预告

第4章将进入本书的核心——Paxos 算法。Paxos 不是 2PC 的"改进版"，而是根本范式的转换：用"多数派"替代"全员"，用"共识"替代"原子提交"。我们将从 Lamport 1989/1990 年的原始论文出发，完整推导 Basic Paxos 的 Prepare/Promise → Propose/Accept 两阶段，以及 Multi-Paxos 的 Leader 优化。
