## 5.1 事件驱动——发布与订阅的魔法

### 5.1.1 为什么不用"直接调用"？

用户在商品页点赞。你需要：
1. 更新点赞数
2. 刷新推荐模型
3. 更新用户画像
4. 发送通知

**反模式——直接在点赞接口里调这些**：

```java
public void likePost(long userId, long postId) {
    likeMapper.insert(userId, postId);       // 1
    recommendService.refresh(userId);         // 2 ← 如果这个挂了？
    userProfileService.update(userId);        // 3 ← 如果这个慢？
    notificationService.send(userId, postId); // 4 ← 如果这个超时？
}
```

如果 `recommendService.refresh()` 超时 30 秒——用户点赞的响应变成 31 秒。而且推荐服务挂了 → 点赞功能也挂了。

### 5.1.2 事件驱动——发布后各管各的

```
用户点赞 → 发布 "UserLiked" 事件 → MQ
                │
    ┌───────────┼───────────┐
    ▼           ▼           ▼
 更新点赞数  刷新推荐    更新画像
 (本地事务)  (异步消费)  (异步消费)
```

**好处**：点赞接口只负责"更新点赞数"——然后发布事件——立刻返回。推荐和画像的更新是异步的——即使它们挂了——点赞本身不受影响。**降低耦合 = 提高系统韧性。**

## 5.2 消息队列的选择——RocketMQ vs Kafka

| 维度 | RocketMQ | Kafka |
|------|----------|-------|
| 事务消息 | ✅ 半消息+回查 | ❌（需要外部协调） |
| 顺序消息 | ✅ 分区有序 | ✅ 分区有序 |
| 延迟消息 | ✅ 18个级别 | ❌（需要外部实现） |
| 持久化 | ✅ CommitLog 顺序写 | ✅ Partition 分段 |
| 适用 | 金融支付、电商核心 | 日志、大数据、流处理 |

**你选 RocketMQ**——因为你需要"事务消息"来保证下单和发积分的原子性（第2章 2.5.3 节已述）。

## 5.3 可扩展的数据结构

**场景**：用户表有 `name, email, phone`。现在要加 `wechat_openid`。传统方式——`ALTER TABLE ADD COLUMN`——锁表数分钟。这在亿级数据表上是不可接受的。

**方案**：JSON 列或 KV 属性表。

```sql
-- 属性表
user_id | attr_name         | attr_value
1       | wechat_openid      | oxxxxxx
1       | alipay_openid      | axxxxxx
```

新增属性 → INSERT 一行——不影响现有查询。**不需要 ALTER TABLE**。

---

## 5.4 本章总结

1. **事件驱动 = 发布订阅**——发布者不依赖消费者。点赞不需要知道"谁会收到通知"。
2. **RocketMQ 事务消息**保证"本地事务+消息发送"的原子性。
3. **JSON/KV属性表**代替 ALTER TABLE——亿级表加列 = 灾难。
