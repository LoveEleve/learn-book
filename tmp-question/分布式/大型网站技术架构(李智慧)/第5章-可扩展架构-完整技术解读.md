## 5.1 事件驱动——发布与订阅的魔法

### 5.1.1 为什么不用"直接调用"？

用户在商品页点赞。你需要：
1. 更新点赞数
2. 刷新推荐模型
3. 更新用户画像
4. 发送通知

**反模式——直接在点赞接口里调这些**：

```java
public void likePost(long userId, long postId) {
    likeMapper.insert(userId, postId);       // 1
    recommendService.refresh(userId);         // 2 ← 如果这个挂了？
    userProfileService.update(userId);        // 3 ← 如果这个慢？
    notificationService.send(userId, postId); // 4 ← 如果这个超时？
}
```

如果 `recommendService.refresh()` 超时 30 秒——用户点赞的响应变成 31 秒。而且推荐服务挂了 → 点赞功能也挂了。

### 5.1.2 事件驱动——发布后各管各的

```
用户点赞 → 发布 "UserLiked" 事件 → MQ
                │
    ┌───────────┼───────────┐
    ▼           ▼           ▼
 更新点赞数  刷新推荐    更新画像
 (本地事务)  (异步消费)  (异步消费)
```

**好处**：点赞接口只负责"更新点赞数"——然后发布事件——立刻返回。推荐和画像的更新是异步的——即使它们挂了——点赞本身不受影响。**降低耦合 = 提高系统韧性。**

## 5.2 消息队列的选择——RocketMQ vs Kafka

| 维度 | RocketMQ | Kafka |
|------|----------|-------|
| 事务消息 | ✅ 半消息+回查 | ❌（需要外部协调） |
| 顺序消息 | ✅ 分区有序 | ✅ 分区有序 |
| 延迟消息 | ✅ 18个级别 | ❌（需要外部实现） |
| 持久化 | ✅ CommitLog 顺序写 | ✅ Partition 分段 |
| 适用 | 金融支付、电商核心 | 日志、大数据、流处理 |

**你选 RocketMQ**——因为你需要"事务消息"来保证下单和发积分的原子性（第2章 2.5.3 节已述）。

## 5.3 可扩展的数据结构

**场景**：用户表有 `name, email, phone`。现在要加 `wechat_openid`。传统方式——`ALTER TABLE ADD COLUMN`——锁表数分钟。这在亿级数据表上是不可接受的。

**方案**：JSON 列或 KV 属性表。

```sql
-- 属性表
user_id | attr_name         | attr_value
1       | wechat_openid      | oxxxxxx
1       | alipay_openid      | axxxxxx
```

新增属性 → INSERT 一行——不影响现有查询。**不需要 ALTER TABLE**。

## 5.4 分布式服务——从单体到微服务

### 5.4.1 Dubbo vs Spring Cloud——选哪个？

第1章的业务拆分成独立服务后，需要服务间的调用框架。

| 维度 | Dubbo | Spring Cloud |
|------|-------|-------------|
| 通信协议 | Dubbo/Triple（自定义二进制） | HTTP/REST |
| 序列化 | Hessian2/Protobuf | JSON |
| 性能 | 高（二进制+长连接） | 中（JSON+HTTP开销） |
| 跨语言 | Triple支持gRPC互通 | HTTP天然跨语言 |
| 服务治理 | 内置（注册/负载/容错/路由） | 依赖SC生态（Nacos/Sentinel/Gateway） |
| 适用 | Java微服务、高性能RPC | 多语言、HTTP API |

**你的选择**：纯 Java 项目 → Dubbo（性能高）；需要跨语言 → Spring Cloud（HTTP 天然跨语言）。**现代趋势**：Dubbo Triple 是 gRPC 兼容的——和 HTTP 一样天然跨语言——所以 Dubbo 的"Java only"不再是限制。

### 5.4.2 服务治理——注册、发现、负载、熔断

```
Provider(提供者) → 注册到 Nacos → Consumer(消费者)从 Nacos 拉取服务列表 → Dubbo 做负载均衡 → 调用
                                                             │
                                                         Consumer 缓存列表
                                                         Provider 下线 → Nacos 通知 → Consumer 更新列表
```

**注册**：Provider 启动 → 向 Nacos 注册自己的 IP+Port+服务名。**发现**：Consumer 启动 → 从 Nacos 拉取服务列表 → 本地缓存。Provider 变动 → Nacos 通知 Consumer。

**负载**：Dubbo 内置 5 种负载策略——随机、轮询、最少活跃调用、一致性哈希、最短响应时间。比 Nginx 的 upstream 更智能——因为有"最少活跃调用"和"最短响应时间"。

**熔断**：Dubbo 3.x 内置 Sentinel 集成——Provider 故障 → Consumer 自动熔断——保证 Consumer 不被拖垮。

### 5.4.3 服务拆分的度——什么时候拆，什么时候不拆

**拆的信号**：
- 一个服务有 3 个以上团队在改（冲突多）
- 一个服务的数据库表 > 50 个（职责边界模糊）
- 一个服务的部署频率和其他模块严重不同（热变更 vs 稳定不变）

**不拆的信号**：
- 两个"服务"之间的事务必须强一致（拆了就需要分布式事务——复杂度暴增）
- 日活 < 1000 的初创项目（微服务在 1000 日活时是过度设计）
- 团队只有 5 个人（维护 20 个微服务的成本比单体高 10 倍）


## 5.5 本章总结

1. **事件驱动 = 发布订阅**——发布者不依赖消费者。点赞不需要知道"谁会收到通知"。
2. **RocketMQ 事务消息**保证"本地事务+消息发送"的原子性。
3. **JSON/KV属性表**代替 ALTER TABLE——亿级表加列 = 灾难。
