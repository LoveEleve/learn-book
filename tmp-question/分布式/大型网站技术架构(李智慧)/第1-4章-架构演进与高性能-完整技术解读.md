## 1. 架构演进——从单机到分布式

### 1.1 单体架构：一切的起点

```
┌─────────────────────────────┐
│        单台服务器             │
│  ┌────────┐  ┌────────────┐ │
│  │  应用   │  │   数据库    │ │
│  └────────┘  └────────────┘ │
│   (同一进程或同一台机器)       │
└─────────────────────────────┘
     LAMP 架构：Linux + Apache + MySQL + PHP
```

**适用阶段**：日活 < 1 万，QPS < 100。

**优点**：开发快、部署快、维护简单。所有代码在一个工程里，新人半天就能上手。

**什么时候撑不住？** 三个信号：
1. **数据库 CPU > 80%**：SQL 查询吃满 CPU
2. **应用服务器内存 > 85%**：并发请求太多，JVM 堆不够
3. **用户感知延迟 > 2s**：单机处理能力达到极限

### 1.2 应用与数据分离

```
┌──────────────┐     ┌──────────────┐     ┌──────────────┐
│   应用服务器   │ ←→ │   数据库      │ ←→ │   文件服务器   │
│  (3台集群)    │     │  (主从)       │     │  (独立存储)   │
└──────────────┘     └──────────────┘     └──────────────┘
```

**核心收益**：
- 应用服务器可以独立扩容（加机器）
- 数据库可以独立优化（换 SSD、加内存）
- 文件（图片/视频）从数据库中分离——数据库不再被大文件拖慢

### 1.3 缓存——性能的第一道防线

```
用户请求 → CDN（静态资源）→ 反向代理缓存 → 本地缓存 → 分布式缓存 → 数据库
```

**缓存层次**：

| 层次 | 技术 | 命中延迟 | 适用 |
|------|------|---------|------|
| CDN | 阿里云 CDN/CloudFront | <10ms | 静态资源（图片/CSS/JS） |
| 反向代理缓存 | Nginx proxy_cache | <1ms（本地） | 页面片段、API 响应 |
| 本地缓存 | Caffeine/Guava Cache | <1μs（JVM 堆内） | 热点配置、字典数据 |
| 分布式缓存 | Redis/Memcached | <1ms（网络） | 用户 Session、商品详情 |

**缓存的核心问题——一致性**：
- Cache Aside：读缓存 → 缓存未命中 → 读 DB → 写缓存
- 更新：先写 DB → 再删缓存（而不是更新缓存）
- 为什么删而不是更新？并发场景下，两个写操作先后更新缓存，可能覆盖对方的写入

### 1.4 集群——应用服务器的横向扩展

```
                ┌──────────────┐
                │   负载均衡器   │
                │  (Nginx/LVS)  │
                └──────┬───────┘
           ┌───────────┼───────────┐
    ┌──────▼──────┐ ┌──▼──────────┐ ┌──▼──────────┐
    │  App Svr 1  │ │ App Svr 2   │ │ App Svr 3   │
    └─────────────┘ └─────────────┘ └─────────────┘
```

**集群带来的新问题**——**Session 共享**：

方案一：**Session 复制**（Tomcat 自带）——每台服务器广播 Session 给所有其他服务器 → 网络开销 O(N²)

方案二：**Session Sticky**（负载均衡器绑定）——同一用户的请求始终路由到同一台服务器 → 服务器宕机 Session 丢失

方案三：**集中式 Session**（Redis）——所有服务器从 Redis 读写 Session → **生产推荐**

### 1.5 读写分离——数据库的第一道防线

```
         ┌──────────┐
         │   主库    │ ← 所有写操作
         └────┬─────┘
              │ binlog 同步
    ┌─────────┼─────────┐
┌───▼───┐ ┌───▼───┐ ┌───▼───┐
│ 从库1  │ │ 从库2  │ │ 从库3  │ ← 所有读操作
└───────┘ └───────┘ └───────┘
```

**关键问题**——**读写延迟**：主库写入后，从库可能有几十毫秒的同步延迟。如果用户"刚发了一个帖子，立即刷新看不到"——这是读写分离的代价。

**解决方案**：关键业务（如支付）强制读主库——"写后立即读"场景走主库。

### 1.6 分库分表——数据库的终极扩展

当单表超过 2000 万行时，即使读写分离也扛不住。分库分表是最后的手段：

```
用户表 → 按 user_id % 16 分 16 个库
订单表 → 按 order_id 分 32 个表
```

**分库分表带来的代价**：
- 跨库 JOIN 不可用——需要应用层聚合
- 分布式事务——本地事务不可用
- 分页查询变复杂——需要从多个库取数据后合并排序

## 2. 架构模式——可复用的设计范式

### 2.1 分层架构

```
┌──────────────────────────┐
│        展示层              │  ← 前端页面、移动 App
├──────────────────────────┤
│        应用层              │  ← 业务逻辑、工作流
├──────────────────────────┤
│        服务层              │  ← 可复用的业务服务
├──────────────────────────┤
│        数据层              │  ← 数据库、缓存、文件存储
└──────────────────────────┘
```

**分层原则**：
- 每层只依赖它的下一层——展示层不能直接访问数据层
- 层间通过接口通信——接口是契约，实现可变
- 同一层可以横向扩展——应用层加机器，数据层分库

### 2.2 分割——按业务拆分

**垂直拆分**：一个大型网站不可能由一个团队维护所有功能。按业务域拆分：

```
淘宝 → 商品域 / 订单域 / 用户域 / 支付域 / 物流域
```

每个业务域有独立的团队、独立的代码仓库、独立的数据库。

### 2.3 异步——削峰填谷

```
同步模式：用户请求 → 扣库存 → 扣余额 → 发积分 → 返回（2s+）

异步模式：用户请求 → 扣库存 → 扣余额 → 返回（200ms）
                                      ↓
                                MQ → 发积分（异步）
                                      ↓
                                消费失败 → 重试 → 补偿
```

**消息队列的三大作用**：
1. **解耦**：生产者和消费者不直接依赖
2. **削峰**：高峰期消息堆积在 MQ 中，消费者按自己的速度处理
3. **异步**：非关键操作不阻塞主流程

### 2.4 分布式服务——从单体到微服务

```
ESB（企业服务总线，不推荐）→ Dubbo（RPC）→ Spring Cloud（HTTP+RPC混合）
                                                      │
                                                  云原生（K8s+Service Mesh）
```

**服务化带来的挑战**：
- 服务发现：Consumer 如何找到 Provider？
- 负载均衡：多个 Provider 实例如何分发流量？
- 容错：Provider 故障如何熔断降级？

这些问题的现代解法由 Spring Cloud 全家桶（Nacos + Sentinel + Gateway）提供。

## 3. 核心架构要素——五个维度评估架构

### 3.1 五个维度

| 维度 | 核心问题 | 关键指标 |
|------|---------|---------|
| **性能** | 系统响应多快？ | QPS、RT、TPS |
| **可用性** | 系统能正常工作多久？ | 99.9%（3个9）、99.99%（4个9） |
| **伸缩性** | 系统能否通过加机器提升性能？ | 线性扩展比 |
| **扩展性** | 系统能否方便地添加新功能？ | 新功能开发周期 |
| **安全性** | 系统能否抵御攻击？ | XSS/CSRF/SQL注入防御 |

### 3.2 性能 vs 可用性的权衡

| 场景 | 选择 | 理由 |
|------|------|------|
| 秒杀 | 性能优先 | 短暂不可用可接受——但秒杀期间不能慢 |
| 支付 | 可用性优先 | 慢几秒可以接受——但不能支付失败 |
| 社交动态 | 性能优先 | 延迟 > 1s 用户流失 |
| 配置下发 | 可用性优先 | 配置必须送达——等几秒没问题 |

**没有"完美架构"——只有"合适的权衡"**。

## 4. 高性能架构——从 CDN 到多级缓存

### 4.1 CDN——让用户就近获取静态资源

```
用户(北京) → 阿里云CDN(北京节点) → 命中静态资源 → <10ms 返回
                │ (未命中)
                └→ 源站(杭州) → 回源获取 → 缓存到北京节点
```

**CDN 缓存什么？** 静态资源（图片、CSS、JS、视频）。动态内容（用户信息、订单状态）不能缓存——每个用户不同。

### 4.2 页面静态化——动态内容的"伪静态"

对于"不经常变化但访问量大"的内容（如商品详情页），可以提前生成静态 HTML：

```
CMS 发布商品 → 模板引擎生成 product_12345.html → CDN 缓存
用户访问 → CDN 直接返回静态 HTML → 零后端压力
```

商品信息变更 → CMS 重新生成 HTML → 触发 CDN 刷新。

### 4.3 多级缓存策略

| 策略 | 读 | 写 | 适用 |
|------|----|----|------|
| **Cache Aside** | 读缓存→miss→读DB→写缓存 | 先写DB→删缓存 | 读多写少 |
| **Read/Write Through** | 缓存层代读DB | 缓存层代写DB | 缓存与DB紧耦合 |
| **Write Behind** | — | 先写缓存→异步写DB | 写入密集 |

**生产最常用 Cache Aside**——简单可靠。**注意**：不要先删缓存再写 DB（并发场景下可能导致缓存中是旧数据）。

### 4.4 池化技术——高性能的基础

**数据库连接池**（HikariCP）：连接复用，避免频繁创建/销毁连接。

**线程池**（Tomcat 200 + 业务 50）：限制并发线程数，避免内存溢出。

**对象池**（commons-pool2）：复用大对象（如 Elasticsearch Client），避免 GC 压力。

**池化原则**：
- 池大小 = (核数 × 2) + 磁盘数（DB 连接池）
- 池大小不是越大越好——太大导致上下文切换开销 > 收益
- 监控：HikariCP 的 `active connections` / `idle connections` / `pending threads`

## 本章总结

1. **架构演进是渐进式的**——从单体→分离→缓存→集群→读写分离→分库分表，每一步解决一个具体瓶颈。
2. **没有银弹**——每个架构决策都有代价。加缓存 → 一致性挑战。分库分表 → 跨库查询挑战。
3. **五个维度**评估架构：性能、可用性、伸缩性、扩展性、安全性——根据业务场景权衡。
4. **多级缓存**（CDN→代理→本地→分布式）是高性能的核心——每层缓存减少一层延迟。
5. **池化技术**（连接池/线程池/对象池）是高性能的基石——复用比创建快两个数量级。
