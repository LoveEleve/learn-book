## 3.1 高可用的度量——"9"的真正含义

### 3.1.1 几个9对应多少故障时间？

| 级别 | 年故障时间 | 日故障时间 | 你的系统承受的起吗 |
|------|-----------|-----------|-----------------|
| 99%（两个9） | 3.65 天 | 14 分钟/天 | ❌ 用户每天都有 14 分钟无法下单——不可接受 |
| 99.9%（三个9） | 8.76 小时 | 1.44 分钟/天 | ⚠️ 可接受——但会有零星投诉 |
| 99.99%（四个9） | 52.6 分钟 | 8.6 秒/天 | ✅ 支付核心标准——用户感知不到 |
| 99.999%（五个9） | 5.26 分钟 | 0.86 秒/天 | ✅ 电信级——你需要冗余的硬件和网络 |

**你的"趣买"电商需要几个9？**

支付核心——**四个9**。商品浏览——**三个9**。管理后台——**两个9**够了（只有运营人员用，故障期间没人报警）。不同服务有不同可用性要求——这就是**分级高可用**。

### 3.1.2 可用性不是"服务器不宕机"

可用性 = 系统正确响应请求的时间 / 总时间。**即使所有服务器都正常运行——如果数据库的读写分离导致用户看到过时的库存数量——这也是不可用的。**

第1章的架构演进中，每一步都提高了可用性：
- 应用集群 → 一台宕机不影响整体
- Session 集中存储 → 服务器切换不影响用户登录状态
- 数据库读写分离 → 主库故障可以手动切从库

## 3.2 负载均衡——故障的第一道防线

### 3.2.1 健康检查——你相信服务器还活着吗？

```
负载均衡器每 5 秒向 App1 发送 GET /health
App1 返回 200 → LB 标记 App1 为 UP
App1 连续 3 次返回 500 → LB 摘除 App1——新请求不再路由到它
```

**三层健康检查**：

| 检查方式 | 检测什么 | 速度 | 盲区 |
|---------|---------|------|------|
| TCP 连接 | 端口存活 | 1ms | 应用层挂了但进程还活着——端口还在监听 |
| HTTP /health | 应用层存活 | 10ms | 数据库连接池耗尽——/health 还返回 200 |
| 深度检查 | DB/Redis/MQ 连通 | 100ms | 无——最全面 |

**生产建议**：Nginx 用 TCP 检查（快速摘除），应用层的 Spring Boot Actuator `/health` 做深度检查。两者结合——**主检查（TCP）+ 辅助检查（深度）**。

### 3.2.2 心跳与超时的权衡——假阳性 vs 假阴性

Nginx 连续 3 次超时才摘除后端——每次超时 5 秒：

- 摘除太快（1 次超时）：网络偶尔抖动 → 假阳性——错误摘除正常服务器
- 摘除太慢（10 次超时）：服务器真的崩溃了 → 长达 50 秒内请求仍然路由到它——不可用

**3 次 × 5 秒 = 15 秒**是业界常用的"甜点"——15 秒的检测延迟对于三个9（年故障 8.76 小时）已经足够。

> **跨书对照**：江峰第5章详细讨论了失败检测器的两种错误——假阳性（误判崩溃）和假阴性（漏判崩溃）。负载均衡的健康检查等价于"最终完美失败检测"（◇P）——需要连续多次确认才摘除，避免误判。

### 3.2.3 四层 vs 七层——什么时候用 LVS，什么时候用 Nginx

```
Client → LVS(L4) → Nginx(L7) → App
          │            │
    (IP:Port层)  (HTTP层)
```

| 维度 | LVS | Nginx |
|------|-----|-------|
| 工作层 | TCP | HTTP |
| 转发速度 | 1-10μs | 100-500μs |
| 灵活路由 | ❌（只能按IP/Port） | ✅（按URL/Cookie/Header） |
| 硬件成本 | 需要额外的 DR 网络配置 | 不需要——标准 TCP |
| 适用 | 大流量统一入口 | 微服务网关、灰度发布 |

**生产架构**：LVS 做第一层（高性能分流），Nginx 做第二层（灵活路由）。LVS 的故障转移用 Keepalived 实现——主 LVS 宕机 → VIP 自动漂移到备 LVS。

## 3.3 灰度发布——低风险的变更

### 3.3.1 为什么不能直接全量发布？

**真实事故**：你发了新版本 v2.1。1 分钟后，所有用户突然无法下单——v2.1 在某个边缘情况下对 `null` 参数抛出了 NPE。你不得不回滚。从发现问题到完成回滚——5 分钟。5 分钟的全站停服对应了多少损失？

**灰度发布**：先放 10% 流量到新版本，观察 30 分钟——没问题再扩大。

```
Step 1: 1 台服务器部署 v2.1 → 10% 流量 → 观察 30 分钟
       监控：错误率 < 0.1%、P99 RT < 100ms → 正常
Step 2: 3 台服务器部署 v2.1 → 50% 流量 → 观察 30 分钟
       监控：错误率 < 0.1%、P99 RT < 100ms → 正常
Step 3: 全量 v2.1
```

**如果 Step 1 发现问题——回滚只需 1 分钟**（K8s 的 Deployment rollback 或 Docker 镜像版本回退）。这就是灰度发布的价值——把"全量发布风险"拆成"逐步验证"。

### 3.3.2 灰度发布的技术实现

**Nginx 层**：按 Cookie 或 Header 路由 10% 流量到灰度环境。

```
upstream app_v2 {
    server 10.0.0.1:8080 weight=1;  # v2.1
}
upstream app_v1 {
    server 10.0.0.2:8080 weight=9;  # v1.0 × 9台
}

# 按 userId 取模——确保同一用户的请求始终路由到同一版本
map $cookie_userId $backend {
    default "app_v1";                     # 90% 流量走 v1
    "~*^[0-9]*[02468]$" "app_v2";        # userId 末位偶数的 → v2（10%）
}
```

**Spring Cloud Gateway 层**：使用 `WeightRoutePredicateFactory` 配置权重路由——更灵活（不需要改 Nginx 配置）。

## 3.4 监控——高可用的"雷达"

### 3.4.1 你的监控盲区在哪里？

有一个周五晚上 10 点，你在家休息。数据库连接池用到了 95%——但没有任何告警。10:30pm，连接池满了——所有请求排队——用户开始报错——QQ/钉钉炸了——你凌晨 3 点被迫爬起来重启服务。

**如果你有三层监控——这个故障根本不会发生。**

| 层 | 监控什么 | 告警阈值 |
|----|---------|---------|
| **基础设施** | CPU > 80%、磁盘 > 85%、网络带宽 > 70% | 钉钉 + 短信（严重的） |
| **应用层** | HikariCP active > 80%、Redis 命中率 < 60%、接口 RT P99 > 500ms | 钉钉（重要的） |
| **业务层** | 下单成功率 < 99%、支付回调延迟 > 5s | 钉钉+电话（核心业务挂 == 损失钱） |

**HikariCP 80% 连接数的含义**：如果你的最大连接池是 20，15 个连接在用——还有 5 个冗余。此时告警——你可以同时排查原因（慢查询、死锁、外部 API timeout），但没有用户受到影响。**告警要在故障发生前触发，不是故障发生后——这是高可用监控的核心哲学。**

### 3.4.2 故障发生前——你的监控告诉你什么

Prometheus 在 22:05 发现了连接数上升到 12（60%）。这是正常范围内的波动——不告警。

22:10：连接数到 16（80%）→ 达到告警阈值。Prometheus 触发告警 → Alertmanager 通知你：**"HikariCP active connections > 80%"**。

你打开 Grafana——看到在 22:05-22:10 之间，有一个慢 SQL 执行了 50 次（平均 2s/次）——它占用了连接池超过正常水平。你找到了 SQL → 加索引 → 连接数降回 10（50%）。**故障没发生——你已经解了。**

## 3.5 限流、熔断、降级——三道防线防雪崩

### 3.5.1 为什么需要限流？

你的"趣买"电商双11大促。平时 QPS 2000，大促瞬间 QPS 暴涨到 20000——翻了 10 倍。如果所有请求都放过去——数据库、Redis、MQ 全部被打崩——不仅是秒杀商品打不开——连首页都打不开了。

**限流（Rate Limiting）**：设置系统的最大承载能力——超出限额的请求直接拒绝。虽然拒绝了部分用户——但**保住了大部分用户能访问首页**。

限流不是"拒绝服务"——是"保护服务不被完全冲垮"。

### 3.5.2 三种限流算法

| 算法 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 计数器 | 每秒重置计数器 | 简单 | 临界点瞬间双倍放行 |
| 滑动窗口 | 每次请求向前推一个窗口 | 平滑 | 内存开销高 |
| 令牌桶 | 固定速率往桶中放令牌 | 允许突发流量 | 实现比其他两种稍复杂 |

**令牌桶算法**——Sentinel 默认使用：

```
桶容量：1000 个令牌
速率：100 令牌/秒（即 QPS=100）

请求到来 → 从桶中取 1 个令牌
  - 有令牌 → 放行，桶中令牌数-1
  - 无令牌 → 拒绝（或排队等待）
```

速率 100 令牌/秒意味着每秒最多处理 100 个请求。桶容量 1000 意味着如果之前的请求没打满——最多可以积压 1000 个令牌——允许瞬时的 1000 QPS 突发（10 秒的余量）。

**为什么令牌桶允许突发？** 计数器"每秒 100 QPS"意味着如果前 100ms 来了 50 个请求——后 900ms 只能放 50 个。令牌桶在同一场景下——前 100ms 可以从累积的桶中一次取出 50 个令牌——不需要削峰。

### 3.5.3 熔断——当依赖方故障时保护自己

你的下单服务调用了"支付服务"。支付服务挂了。

**如果不用熔断**：下单服务的每次调用都会等 30 秒超时 → 下单服务的线程全部阻塞在等待支付服务 → 下单服务本身也挂了。

**熔断（Circuit Breaker）**：当支付服务的失败率超过阈值（如 50% 连续失败）→ 熔断器"跳闸"——后续请求不再调用支付服务——直接返回降级响应（"支付服务暂时不可用"）。这样下单服务不会因为调用支付而崩溃。

```
熔断器状态：
  CLOSED（正常）——请求正常通过
    │ 失败率 > 50%（连续 10 次）
    ▼
  OPEN（熔断）——直接返回降级响应，不调用支付服务
    │ 等待 30 秒
    ▼
  HALF-OPEN（试探）——放行少量请求试探支付是否恢复
    │ 成功 → 回到 CLOSED
    │ 失败 → 回到 OPEN
```

**与限流的区别**：限流是"我自己扛不住了"，熔断是"我的依赖方扛不住了"。两者都是高可用的防线。

## 3.6 数据备份——灾难恢复的最后防线

### 3.6.1 RPO 和 RTO——你的备份能承受多少损失

| 指标 | 含义 | 你的目标 |
|------|------|---------|
| **RPO（Recovery Point Objective）** | 数据丢失的最大容忍时间 | 支付服务：<1 分钟（binlog 实时同步） |
| **RTO（Recovery Time Objective）** | 从故障到恢复的最大容忍时间 | 支付服务：<5 分钟（从库提升为主库） |

**你的备份策略**：

```
每日全量备份：    凌晨 3 点（用户最少）
每小时增量备份：  binlog 持续同步
异地容灾：        备份复制到另一个机房的 S3/OSS
```

**灾难场景推演**——主库机房火灾：

```
T0: 主库机房火灾——数据不可访问
T1: (RTO=5min) DBA 将异地容灾实例提升为主库
    从全量备份 + binlog 恢复到火灾前的状态
T2: DNS 切换到新主库 → 应用恢复
```

**火灾前最后 1 小时的数据没有异地备份——这些数据的一部分在 binlog 中（如果实时同步到异地），剩下的丢失了。RPO 取决于 binlog 的异地同步频率——如果每 60 秒同步一次，RPO < 1 分钟。**

## 3.7 SLO/SLI/Error Budget——用数学量化高可用

### 3.7.1 三个核心概念

| 概念 | 含义 | 你的支付服务 |
|------|------|------------|
| **SLI（指标）** | 用什么衡量服务质量 | 支付成功率 |
| **SLO（目标）** | 应该达到什么水平 | 支付成功率 >= 99.99%（四个9） |
| **Error Budget（误差预算）** | 允许多少失败 | 0.01% 失败率 → 每月允许 ~4.3 分钟的故障 |

**为什么需要 Error Budget？** 如果你规定"支付成功率必须 100%"，团队永远不敢上线新版本——因为任何变化都有风险。Error Budget 告诉你：**每月 4 分钟以内的故障是可接受的**——在这个预算内，你可以大胆地上线、灰度发布、A/B 测试。

### 3.7.2 Error Budget 的实践

```
Error Budget = 1 - SLO
  = 1 - 0.9999 = 0.0001 (0.01%)

月度 Error Budget = 30 × 24 × 60 × 0.0001 = 4.32 分钟
```

**当 Error Budget 耗尽时**：
- 暂停所有生产变更（冻结发布）
- 全部团队优先修复可靠性问题
- Error Budget 下个月重置

**当 Error Budget 充足时**：
- 可以上线新功能
- 可以灰度发布
- 可以做 A/B 测试

这比"PM 说不能发"/"开发说可以发"的主观判断**客观**得多——数据说话。

## 3.5 本章总结——高可用的四条防线

| 防线 | 机制 | 故障检测时间 | 你能做什么 |
|------|------|------------|-----------|
| 负载均衡 | 健康检查 + 自动摘除 | 15 秒 | 不需要人工介入 |
| 灰度发布 | 逐步验证新版本 | 发布期间 | 1 分钟回滚 |
| 监控 | Prometheus + Grafana + 钉钉 | <5 分钟 | 提前发现、提前干预 |
| 数据备份 | 主从复制 + 定期快照 | 分钟级 | 故障后恢复数据 |

### 作为 CTO，你的高可用哲学

1. **不要等故障发生——要在故障的"风"吹来时就知道。** Prometheus 的告警阈值不是事后通知——是事前预警。
2. **不要一次改太多。** 灰度发布把"3 个月开发完的一次性上线"变成"每两周一小步的渐进上线"——每一小步都可以安全回滚。
3. **分级高可用——不是所有服务都需要四个9。** 支付核心：四个9。商品浏览：三个9。管理后台：两个9够了。
