# 第4章 伸缩性架构——加机器而不是换机器

> **技术专家重写——第一性原理版**
>
> 第1-3章建立了架构骨架、高性能和高可用。本章的核心问题：**你的日活再翻一倍——你买一台 256GB 内存的 IBM 小型机，还是加 20 台普通 PC 服务器？**

---

## 4.1 垂直伸缩 vs 水平伸缩

### 4.1.1 垂直伸缩——换更好的机器

```
4C8G → 16C32G → 64C256G
```

**优点**：简单——不需要改代码，不需要改架构。**缺点**：有物理极限——单台服务器最多几百个 CPU 核、几 TB 内存。而且越往上越贵——64C256G 的机器比 4×16C32G 贵 3-5 倍。

### 4.1.2 水平伸缩——加更多的机器

```
1 台 4C8G → 10 台 4C8G → 100 台 4C8G
```

**优点**：理论上无限扩展。用普通 PC 服务器——成本低。**缺点**：需要负载均衡、分布式 Session、数据分片——这些是第1章已经解决的。

**互联网公司选择水平伸缩**——不是因为它更高级，而是因为"加机器"比"换机器"更便宜。

## 4.2 负载均衡的六种方式

第3章讲了负载均衡的概念。现在从伸缩性的角度重新审视——每一种方式的**性能、成本和扩展上限**。

| 方式 | 工作层 | 性能 | 扩展上限 | 适用 |
|------|--------|------|---------|------|
| DNS 域名解析 | DNS | 高（客户端直接解析） | 无限 | 地域级负载（北京用户→北京机房） |
| HTTP 重定向 | HTTP | 低（多一次请求） | 有限 | 极少使用——用户体验差 |
| 反向代理（Nginx） | L7 | 中（10万QPS） | LB 成为瓶颈 | 大多数场景——灵活路由 |
| IP 负载均衡（LVS） | L4 | 高（百万QPS） | LB 可能瓶颈 | 大流量入口 |
| 数据链路层（LVS/DR） | L2 | 最高（千万QPS） | 几乎无限 | 超大规模入口 |
| 客户端负载均衡 | 应用层 | 最高 | 无限（真正去中心化） | 微服务间调用 |

### 4.2.1 为什么需要这么多方式？

**DNS 解析**：一台机器在北京、另一台在上海——DNS 根据用户 IP 返回最近的服务器。这是**地域级负载均衡**——Nginx 做不到（因为 Nginx 只能在同一机房内做均衡）。

**LVS/DR（直接路由）**：性能最高的方式。LVS 只修改 MAC 地址（不修改 IP）——返回包直接从后端服务器返回客户端——不经过 LVS。Nginx 和普通 LVS 都需要返回包也经过 LB——这是性能瓶颈的根本原因。

**客户端负载均衡**：微服务时代的产物。Ribbon（已过时）/Spring Cloud LoadBalancer 在客户端做负载均衡——从注册中心获取服务列表，客户端自己选择调用哪个实例。**这才是真正的去中心化——没有 LB 单点。**

### 4.2.2 你的架构需要哪几层？

```
用户 → DNS（地域级）
         │
    ┌────┴────┐
    │ LVS×2   │ (L4 级——VIP 漂移)
    └────┬────┘
         │
    ┌────┴────┐
    │ Nginx×3 │ (L7 级——灵活路由)
    └────┬────┘
         │
    ┌────┴────┐
    │ App ×10 │ (客户端负载均衡——已无中心)
    └─────────┘
```

**为什么 App 层可以没有 LB？** 因为 Spring Cloud LoadBalancer 在应用层做了负载均衡——每个微服务实例直接从 Nacos 获取其他服务的地址——自己选择调用哪个。不再需要经过 Nginx。性能最高、没有中心单点。

## 4.3 一致性哈希——缓存的伸缩性

### 4.3.1 问题：加一个 Redis 节点，所有缓存都失效

你有一个 3 节点的 Redis 集群。缓存 key 通过 `hash(key) % 3` 分配到不同节点：

```
key="product:123" → hash=45682 → 45682 % 3 = 1 → Redis-1
```

现在你加了第 4 个节点：

```
key="product:123" → hash=45682 → 45682 % 4 = 2 → Redis-2 ← 变了！
```

**3/4 的 key 重新映射到了不同的节点——缓存全部失效——所有请求打到数据库——数据库被打崩。**

### 4.3.2 一致性哈希如何解决

**一致性哈希**不直接对节点数取模，而是把所有节点映射到一个**哈希环**上（0 ~ 2^32-1）：

```
哈希环:
        0
    S1     S3
   /    \  /
  /      \/
  \      /\
   \    /  \
    S2     S4
       2^32-1
```

**key 的分配规则**：计算 `hash(key)`——在哈希环上顺时针找第一个节点——那就是这个 key 的存储节点。

**加一个新节点 S5**：S5 落在哈希环上 S1-S3 之间。只有原本映射到 S3 的 key 中、落在 S1-S5 之间的部分需要重新映射到 S5。其他 key 不受影响。**通常只有约 1/N 的 key 需要迁移**（N 为新增节点前的节点数）。

### 4.3.3 虚拟节点——解决数据倾斜

如果 S1 和 S2 在哈希环上距离很近，它们之间的 key 很少——大部分 key 落在离 S3 近的区域——S3 承担了不成比例的热数据。

**虚拟节点**：每个物理节点对应 100-200 个虚拟节点。虚拟节点均匀分布在哈希环上。通过增加虚拟节点密度——将物理节点的不均匀负载"抹平"到整个环上。

```
物理节点 S1 → 虚拟节点 S1-1, S1-2, ..., S1-150
物理节点 S2 → 虚拟节点 S2-1, S2-2, ..., S2-150
```

这样即使 S1 和 S2 物理位置不均——150 个虚拟节点的分布足够均匀——数据倾斜被消除。

## 4.4 分库分表——数据库的终极伸缩

### 4.4.1 什么时候必须分库分表？

单表超过 2000 万行、数据库 CPU 持续 > 70%、从库数量 > 5 台——这三个信号同时亮起——你必须分库分表。

第1章提到了分库分表的代价。现在深入具体的方案选择。

### 4.4.2 三种分片策略

| 策略 | 例子 | 优点 | 缺点 |
|------|------|------|------|
| **Hash 分库** | `user_id % 16` | 数据均匀分布 | 扩容需要大量迁移 |
| **范围分库** | `create_time` 按月 | 扩容简单（新月份新库） | 热点集中（最近 3 个月数据被频繁访问） |
| **一致性哈希分库** | 哈希环映射 | 扩缩容只影响相邻 | 实现复杂度稍高 |

**你选哪种？** 用户表用 Hash 分库（均匀）。订单表用范围分库（最近 3 个月的数据热点——放 SSD 节点，老数据放 HDD 节点）。没有"最佳策略"——只有"最合适的组合"。

### 4.4.3 分库后的"跨库"操作——怎么做

**分页查询**——`LIMIT 100, 20`：

- 每个分库取 120 条（100+20），在应用层合并排序
- 5 个分库 → 5 × 120 = 600 条合并 → 取前 20 条

**跨库 JOIN**——SELECT * FROM orders JOIN users：

- 应用层做两次查询：先从用户库查出 user_id → 再在订单库查各分库
- 不能依赖数据库 JOIN——因为数据在不同的数据库实例中
- 性能最低，但这是分库的代价

**分布式 ID 生成**——雪花算法（Snowflake）：

```
ID = 时间戳(41位) + 机器ID(10位) + 序列号(12位)
```

保证在分布式环境下生成全局唯一 ID——不依赖数据库的自增主键（分库后自增主键重复）。

### 4.4.4 雪花算法的时钟回拨问题

雪花算法依赖**机器时钟单调递增**。但如果 NTP 同步导致时钟回拨——生成的 ID 可能重复。

**三种解决方案**：

1. **抛异常**（美团 Leaf）：检测到回拨 → 直接拒绝服务 → 人工介入
2. **等待**（百度 UidGenerator）：检测到回拨 → sleep 等待时钟追上来再继续
3. **借用未来时间**：记录上次最大时间戳——回拨时直接使用上次时间戳+1

**生产选择**：方案1最简单可靠——出现回拨的窗口通常只有几毫秒（NTP 微调频率），拒绝几毫秒的请求影响极低。

### 4.4.5 分库分表的扩容实战——从 4 库到 8 库

第1章讲了"为什么要分库"。现在你真的要从 4 个库扩到 8 个库。你不能简单地停服迁移——你要让用户在这期间继续下单。

**双写过渡方案**：

```
阶段1（双写）：
  新数据 → 同时写入旧库(4库)和新库(8库)
  读操作 → 依然从旧库读

阶段2（迁移历史数据）：
  后台任务逐步将旧库的历史数据迁移到新库
  （按 user_id 范围或 create_time 分批次迁移）

阶段3（切读）：
  新库数据验证通过 → 读操作切换到新库
  读旧库的请求开始返回"404"时的回退

阶段4（下线旧库）：
  旧库的写入停止 → 下线旧库
  完成从 4 库到 8 库的无缝扩容
```

**关键防坑**：
- 阶段1双写期间——新库写入失败不能影响主流程（异步重试，不阻塞用户请求）
- 阶段2迁移历史数据——需要检查数据是否已在阶段1写入（INSERT ... ON DUPLICATE KEY 或 INSERT IGNORE）
- 阶段2的批量迁移需要限速——不能打满数据库 CPU——影响正常的业务请求

### 4.4.6 读写分离的中间件选择

第1章的读写分离是手动在代码中区分——用 `@Transactional(readOnly=true/false)` 注解选择数据源。但对于更复杂的分库分表+读写分离场景——需要中间件。

| 中间件 | 原理 | 优点 | 缺点 |
|--------|------|------|------|
| ShardingSphere-JDBC | JDBC 层代理（与应用集成） | 性能高、无额外维护 | 需要修改应用配置 |
| MyCat | 数据库代理（独立进程） | 应用无感知 | 多一跳网络延迟 |

**生产推荐**：ShardingSphere-JDBC——性能高、配置简单。不需要单独部署中间件——直接用 JDBC 连接串配置读写分离。

```yaml
spring:
  shardingsphere:
    datasource:
      names: master, slave0, slave1
      master:
        url: jdbc:mysql://10.0.0.1:3306/order
      slave0:
        url: jdbc:mysql://10.0.0.2:3306/order
      slave1:
        url: jdbc:mysql://10.0.0.3:3306/order
    rules:
      readwrite-splitting:
        data-sources:
          order_rw:
            write-data-source-name: master
            read-data-source-names: slave0, slave1
            load-balancer-name: round_robin
```

**一行配置替换了你之前几百行的 `@Transactional(readOnly=...)` 注解**。

## 4.5 本章总结

1. **水平伸缩 > 垂直伸缩**——加机器比换机器便宜，而且没有物理极限。
2. **一致性哈希**——扩容只影响 ~1/N 的 key，虚拟节点消除数据倾斜。
3. **分库分表**——Hash（均匀）+ 范围（灵活）+ 一致性哈希（扩容友好）。三种策略按业务混合使用。
4. **负载均衡的六种方式**——DNS（地域级）→ LVS（高性能）→ Nginx（灵活）→ 客户端LB（去中心化）。每层解决一个不同的伸缩问题。
