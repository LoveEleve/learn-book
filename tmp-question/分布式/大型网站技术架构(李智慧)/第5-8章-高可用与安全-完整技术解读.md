## 5. 高可用架构——万无一失的设计

### 5.1 可用性的度量

可用性用"几个9"衡量：

| 级别 | 可用性 | 年故障时间 | 典型场景 |
|------|--------|-----------|---------|
| 3个9 | 99.9% | ~8.8 小时 | 一般互联网业务 |
| 4个9 | 99.99% | ~52 分钟 | 支付、交易核心 |
| 5个9 | 99.999% | ~5 分钟 | 电信级、金融核心 |

**可用性 = 系统正常运行时间 / 总时间**。注意：计划内的维护窗口不算在"故障时间"里。

### 5.2 高可用的应用——无状态 + 负载均衡

**核心原则**：应用服务器必须无状态。Session 不存本地，存 Redis。

```
┌──────────────────┐
│    负载均衡器     │ ← 健康检查 + 故障转移
└────────┬─────────┘
    ┌────┼────┐
┌───▼──┐┌───▼──┐┌───▼──┐
│ App1 ││ App2 ││ App3 │  ← 全部无状态
└──┬───┘└──┬───┘└──┬───┘
   │       │       │
   └───────┼───────┘
           │
    ┌──────▼──────┐
    │ Redis(Session)│ ← 集中式 Session
    └─────────────┘
```

**故障转移时间线**：

```
T0: App2 宕机
T1: 负载均衡器第 1 次健康检查失败（1s）
T2: 负载均衡器第 2 次健康检查失败（2s）
T3: 负载均衡器第 3 次健康检查失败（3s）→ 确认故障
T4: 负载均衡器将 App2 摘除（<1ms）
T5: 新请求不再路由到 App2

总故障检测时间：约 3-5 秒（取决于健康检查间隔和重试次数）
```

### 5.3 高可用的数据——CAP 与数据备份

> **跨书深度导航**：释慧利+江峰详细推导了 CAP 和共识算法。本篇从**高可用工程师**的视角——"如何备份数据、如何在主库故障时切换"。

**数据高可用的三层保障**：

1. **主从复制**：主库写入后同步到从库。异步复制有延迟，同步复制降低性能
2. **数据备份**：定期全量备份 + 增量 binlog 备份。冷备存储在异地
3. **失效转移**：主库故障 → 从库提升为主库 → 应用切换连接

**主从切换的关键挑战**：

```
主库宕机 → 从库提升为主库

问题 1：新主库的数据是最新的吗？
  解决：半同步复制——主库等待至少 1 个从库 ACK 后才返回成功
  
问题 2：旧主库恢复后怎么办？
  解决：旧主库降级为从库，从新主库同步数据
  
问题 3：切换期间的应用请求怎么办？
  解决：应用重试 + 数据库中间件（ShardingSphere）自动切换
```

### 5.4 灰度发布——降低发布风险

```
Step 1: 发布 v2 到 1 台服务器（10% 流量）
        观察 30 分钟 → 错误率正常
Step 2: 发布 v2 到 5 台服务器（50% 流量）
        观察 30 分钟 → 错误率正常
Step 3: 全量发布 v2（100% 流量）
```

**灰度发布的回滚**：任何一步发现问题 → 立即回滚到 v1。关键：回滚操作必须能在**1 分钟内**完成（通过 K8s 的 Deployment rollback 或提前保留旧版本 Docker 镜像）。

### 5.5 监控——高可用的"雷达系统"

**三层监控**：

| 层次 | 监控什么 | 工具 |
|------|---------|------|
| **基础设施** | CPU、内存、磁盘、网络 | Prometheus + Node Exporter |
| **中间件** | Redis 命中率、DB 连接数、MQ 积压 | Prometheus + Exporter |
| **业务** | 订单量、支付成功率、用户注册量 | 自定义 Metrics + Grafana |

**关键告警阈值**（经验值）：

| 指标 | 告警 | 严重 |
|------|------|------|
| CPU | >70% | >90% |
| 内存 | >80% | >95% |
| DB 连接数 | >80% 池大小 | >95% 池大小 |
| 接口 RT | >P99 翻倍 | >P99 翻 3 倍 |
| 错误率 | >1% | >5% |

## 6. 伸缩性架构——应对流量增长

### 6.1 伸缩性的两个方向

| 方向 | 怎么做 | 极限 |
|------|--------|------|
| **垂直伸缩（Scale Up）** | 换更好的机器（更多 CPU、更多内存） | 物理极限（单机 CPU 最多几百核） |
| **水平伸缩（Scale Out）** | 加更多的机器 | 几乎无限 |

**互联网公司选择水平伸缩**——普通 PC 服务器 + 水平扩展，而不是昂贵的 IBM 小型机。

### 6.2 负载均衡的六种方式（从 DNS 到数据链路层）

| 方式 | 工作层 | 优点 | 缺点 |
|------|--------|------|------|
| **DNS 域名解析** | DNS | 简单、免费 | 有缓存延迟、不可控 |
| **HTTP 重定向** | HTTP | 简单 | 多一次请求、URL 变化 |
| **反向代理** | HTTP(L7) | 灵活路由 | 应用层，性能较低 |
| **IP 负载均衡** | TCP(L4) | 性能高 | 所有流量经过 LB |
| **数据链路层** | MAC(L2) | 最高性能 | 需要物理网络支持 |
| **一致性哈希** | 应用层 | 缓存友好 | 不适用于无状态服务 |

### 6.3 分布式缓存的伸缩——一致性哈希

添加/移除缓存节点时，**一致性哈希只影响哈希环上相邻节点的数据**——而不是所有数据重新分布。

```
扩容前：4 个 Redis 节点 → 每个节点 25% 的 key
扩容后：5 个 Redis 节点 → 只有 ~20% 的 key 需要迁移
```

**虚拟节点**：每个物理节点对应 100-200 个虚拟节点，减少数据倾斜。

### 6.4 数据存储的伸缩——分库分表

**分库分表的三种策略**：

| 策略 | 例子 | 优点 | 缺点 |
|------|------|------|------|
| **Hash 分库** | `user_id % 16` | 数据均匀分布 | 扩容需要大量迁移 |
| **范围分库** | `create_time 按月` | 扩容简单 | 热点数据集中 |
| **查找表** | 单独的"用户→分库"映射表 | 灵活、任意迁移 | 多一次查表 |

**推荐**：先用 Hash 分库（均匀），配合**一致性哈希**做扩容——减少迁移量。

## 7. 可扩展架构——降低系统耦合

### 7.1 事件驱动架构

```
用户下单 → 发布"订单创建"事件
              │
    ┌─────────┼─────────┐
    ▼         ▼         ▼
 发积分    更新统计   发送通知
```

**好处**：新增消费者不需要修改下单代码。解耦 = 可扩展。

### 7.2 分布式消息队列

| MQ | 适用场景 | 特点 |
|-----|---------|------|
| RocketMQ | 金融级可靠 | 事务消息、顺序消息 |
| Kafka | 大数据/日志 | 高吞吐、持久化 |
| RabbitMQ | 通用场景 | 路由灵活 |

**MQ 的三大模式**：
- **点对点**：一个消息被一个消费者处理（订单处理）
- **发布订阅**：一个消息被多个消费者处理（事件通知）
- **请求响应**：RPC over MQ（不推荐——直接用 HTTP/gRPC 更快）

### 7.3 分布式服务框架

**Dubbo vs Spring Cloud**：

| 维度 | Dubbo | Spring Cloud |
|------|-------|-------------|
| 通信协议 | Dubbo/Triple（自定义） | HTTP/REST |
| 注册中心 | Nacos/ZooKeeper | Nacos/Eureka/Consul |
| 服务治理 | 内置（负载均衡/熔断/限流） | 依赖 Spring Cloud 生态 |
| 适用 | Java 微服务 | 多语言、HTTP API |

**现代选择**：Dubbo Triple（gRPC 兼容）+ Nacos 注册。比 HTTP 性能高，比原生 Dubbo 协议更标准。

## 8. 安全架构——抵御攻击

### 8.1 XSS 跨站脚本攻击

```
攻击者提交评论：<script>document.cookie 发给攻击者服务器</script>
                网站直接渲染这段 HTML → 脚本执行 → Cookie 被盗
```

**防御**：所有用户输入**HTML 转义**（`<script>` → `&lt;script&gt;`）。现代前端框架（React/Vue）默认做了转义——但后端渲染（JSP/Thymeleaf）需要显式处理。

### 8.2 CSRF 跨站请求伪造

```
用户在 A 银行登录 → 浏览器保存 Cookie
用户浏览 B 网站 → B 网站嵌入了 <img src="http://A.bank/transfer?to=attacker&amount=10000">
浏览器自动发送请求（携带 A 银行的 Cookie）→ 攻击成功
```

**防御**：CSRF Token——每次表单提交携带唯一 Token，服务器验证。

### 8.3 SQL 注入

```
用户登录：用户名 = admin，密码 = ' OR '1'='1
SQL: SELECT * FROM users WHERE name='admin' AND password='' OR '1'='1'
     → '1'='1' 永远为真 → 绕过密码验证
```

**防御**：参数化查询（PreparedStatement）——SQL 结构与数据分离，攻击者无法修改 SQL 语义。

### 8.4 安全的三道防线

1. **输入校验**：所有用户输入必须过滤（HTML 转义、SQL 参数化）
2. **权限控制**：最小权限原则——每个服务只能访问它需要的数据
3. **加密传输**：HTTPS（TLS 1.3）——防止中间人攻击。内网服务间使用 mTLS

## 本章总结

1. **高可用用"9"度量**——支付核心需要 4 个 9（年故障 < 52 分钟）。三层保障：主从复制 + 备份 + 失效转移。
2. **伸缩性是水平扩展**——一致性哈希解决了"加节点不迁移全量数据"的问题。分库分表是数据库的终极扩展方案。
3. **可扩展靠解耦**——消息队列解耦生产者与消费者，分布式服务框架解耦业务模块。
4. **安全三道防线**——输入校验 + 权限控制 + 加密传输。XSS/CSRF/SQL 注入是永恒的经典攻击，防御方案也是永恒的经典：转义/Token/参数化。
