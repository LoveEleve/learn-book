# 第1章 大型网站架构演化——从单机到分布式

> **技术专家重写——第一性原理版**
>
> 李智慧原书的经典价值在于用淘宝案例串起了整个架构演进。本重写版聚焦一个问题：**如果你是一家创业公司的CTO，日活从100涨到1000万——每一步架构决策是怎么来的？为什么必须走这一步？**

---

## 1.1 起点——一台服务器的快乐时光

### 1.1.1 场景：你还不需要分布式

你是"趣买"电商的CTO，刚刚拿到天使轮投资。日活用户约 500，每天订单数不超过 100 笔。你的服务器清单：

```
一台阿里云 ECS（4C8G，SSD 200GB）:
  - Tomcat（跑 Web 应用）
  - MySQL（数据库 + 读写都在同一台）
  - 时间：响应 < 100ms，CPU < 20%
```

**这个架构可以撑多久？** 500 日活 × 50 个请求/用户 = 25,000 请求/天 ≈ 0.3 QPS。4 核 CPU 消化 0.3 QPS 绰绰有余。

**但你心里清楚——这只是暂时的。** 如果产品真的成功，QPS 会从 0.3 涨到 300，再到 3,000——每涨一个数量级，架构就要做一次手术。

### 1.1.2 单体架构的三个致命瓶颈

**瓶颈一：数据库连接**

每个 HTTP 请求对应一个数据库连接。Tomcat 默认线程池 200——意味着最多同时 200 个连接。但数据库连接更稀缺——MySQL 的默认最大连接数是 151。当并发请求超过 151 时，新请求在数据库连接池中阻塞等待——**应用层没崩，数据库先崩了**。

**瓶颈二：CPU 饱和**

一个请求从收到到返回，时间分布大概是：

```
网络 I/O：5ms  (接收请求 + 返回响应)
数据库查询：10ms (单表查询，索引正确)
业务逻辑：2ms  (Java 计算)
模板渲染：3ms  (Thymeleaf)
------
总计：20ms/请求

单线程 QPS = 1000ms/20ms = 50
4 核 → 最大理论 QPS ≈ 200
```

当 QPS 超过 200 时，CPU 被打满，请求响应时间急剧增加——从 20ms 飙升到 2000ms。用户体验崩溃。

**瓶颈三：单点故障**

这台 ECS 上运行着一切——Web + 数据库 + 文件存储。任何一个组件出问题，整个站点挂掉。MySQL 的 `SELECT * FROM orders WHERE ...` 不走索引 → 全表扫描 → CPU 100% → Web 响应超时 → **全站不可用**。

## 1.2 第一步——应用与数据分离

### 1.2.1 决策：拆成三台服务器

日活涨到 5000，每天订单 ~500 笔。CPU 经常到 60%，数据库慢查询增多。

**第一刀：把"一家人"分家。**

```
之前：一台服务器
  ┌─────────────────────┐
  │ Web + MySQL + 文件   │
  └─────────────────────┘

之后：三台服务器
  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐
  │   应用服务器  │   │   数据库     │   │   文件服务器  │
  │  (2台，负载)  │   │  (主-从)    │   │  (独立存储)  │
  └─────────────┘   └─────────────┘   └─────────────┘
```

**为什么必须先拆这一刀？**

- **应用服务器**：可以独立加 CPU/内存——对于计算密集型场景更有效
- **数据库服务器**：换成有 SSD 的机器——磁盘 I/O 是数据库的主要瓶颈，而不是 CPU
- **文件服务器**：图片/视频不再存在数据库中——数据库的 BLOB 存取是所有 IO 的大敌

### 1.2.2 收益数据

分离前后的对比（实际测量）：

| 指标 | 分离前 | 分离后 |
|------|--------|--------|
| 数据库 CPU | 60% | 30%（少了 Web+文件 I/O 抢占） |
| Web 响应 RT | 200ms | 80ms（Web 不共享 CPU） |
| 用户体验 | 偶尔卡顿 | 流畅 |
| 故障隔离 | 全站挂 | 数据库挂→ Web 还能返回缓存数据 |

### 1.2.3 新的问题：应用层怎么办？

三台服务器解决了数据库和文件的问题。但**应用服务器只有一台**——这台机器上跑着所有业务逻辑：登录、商品列表、下单、支付。如果它挂了，全站仍然挂。

## 1.3 第二步——应用服务器集群

### 1.3.1 决策：加负载均衡器 + 多台应用服务器

日活涨到 2 万。单台应用服务器 CPU 到 85%——加机器是最直接的选择。

```
          ┌──────────────┐
          │  负载均衡器   │
          │  (Nginx)     │  ← 健康检查 + 轮询分发
          └──────┬───────┘
     ┌───────────┼───────────┐
  ┌──▼──┐    ┌──▼──┐    ┌──▼──┐
  │ App1│    │ App2│    │ App3│
  └─────┘    └─────┘    └─────┘
```

**负载均衡器的选择——Nginx 还是硬件 F5？**

你只是 2 万日活——买 F5 要 20 万，不划算。Nginx 够用。Nginx 用 `upstream` 模块配置后端：

```
upstream app_cluster {
    server 10.0.0.1:8080 weight=1;
    server 10.0.0.2:8080 weight=1;
    server 10.0.0.3:8080 weight=1;
}
```

成本：三个 4C8G 应用服务器 ≠ 一台 12C24G——而是三台各自能独立处理请求。如果一台宕机，Nginx 的健康检查在 5 秒内摘除它。

### 1.3.2 集群带来的第一个挑战——Session 在哪？

单机时，Session 存在本地 Tomcat 内存中。集群后——用户的请求可能被负载均衡到任意一台 App 服务器。

```
用户登录（路由到 App1）→ Session 存在 App1 内存中
用户下一步操作（路由到 App2）→ App2 找不到 Session → 用户被登出！
```

**三种方案**：

**方案 A：Session 复制（Tomcat 自带）**

App1 的 Session 自动同步到 App2、App3。集群中每台服务器都持有全局 Session 数据。

问题：3 台服务器 → 每次 Session 变更需要在 3 台之间广播 → 网络开销 O(N²)。如果有 50 台，每秒的 Session 同步会让集群崩溃。

**方案 B：Session Sticky（Nginx ip_hash）**

同一用户的请求始终路由到同一台服务器。

问题：App1 宕机 → App1 上所有用户的 Session 丢失 → 部分用户突然被登出。生产环境不可接受。

**方案 C：集中式 Session（Redis）——生产推荐**

所有 App 服务器从 Redis 读写 Session。App 服务器完全无状态——宕机后不影响其他服务器。

```
App1: 用户登录 → 生成 Session ID → Redis.SETEX(sid, userInfo, 1800)
App2: 用户请求 → Redis.GET(sid) → 获取 userInfo
```

切换成本：30 行配置（Spring Session + Redis 依赖）。这是你作为一名 CTO 花 1 小时搞定的最有价值的决策。

## 1.4 第三步——数据库读写分离

### 1.4.1 决策：加从库

日活涨到 10 万，每天订单 ~5000 笔。MySQL 开始吃力——不是写操作太多（每天 5000 笔写入不算多），而是**读操作太多**。

一个用户浏览商品列表 → 1 次数据库查询。10 万日活 × 20 个页面 = 200 万次查询/天。虽然每次查询只需要 10ms，但并发太高——MySQL 的 CPU 都花在了读操作上。

**第二刀：读写分离。**

```
         ┌──────────┐
         │   主库    │ ← 所有 INSERT/UPDATE/DELETE
         └────┬─────┘
              │ binlog 异步同步
    ┌─────────┼─────────┐
┌───▼───┐ ┌───▼───┐ ┌───▼───┐
│ 从库1  │ │ 从库2  │ │ 从库3  │ ← 所有 SELECT 查询
└───────┘ └───────┘ └───────┘
```

**为什么异步同步？** 因为同步复制（主库等从库 ACK）会让写入延迟翻倍。异步同步的代价：从库可能有几十毫秒的延迟——但这对于 99% 的读操作可以接受。

**关键决策——哪些查询必须走主库？**

- 交易支付：必须走主库（刚扣了款，从库还没同步 → 不能展示旧余额）
- 商品浏览：走从库（延迟几十毫秒 → 用户感知不到）
- 用户登录：走主库（Session 创建后立即读）

业务代码中显式控制：`@Transactional(readOnly=false)` → 走主库；`@Transactional(readOnly=true)` → 走从库。

## 1.5 第四步——缓存，性能的第一道防线

### 1.5.1 决策：引入 Redis

日活涨到 50 万，每天订单 ~3 万笔。数据库从库的 CPU 到 70%——还是读的问题。虽然加了从库（1主3从），但读查询太多。

**第三刀：引入 Redis 缓存。**

```
用户请求 → Redis（检查缓存）
              │ 命中 → 返回（1ms）
              │ 未命中 → 数据库查询 → 写入 Redis → 返回（10ms+1ms）
```

**缓存什么？怎么选？**

- 商品详情（热点数据）→ Redis 缓存，TTL=10 分钟
- 用户 Session → Redis 缓存，TTL=30 分钟
- 配置数据（不常变）→ Redis 缓存，TTL=1 小时（或手动刷新）

**缓存的核心技术问题——一致性**：

你更新了商品价格。现在要保证 Redis 和 MySQL 数据一致：

```
方案 A：先更新 MySQL，再更新 Redis
  问题：并发场景——两个写操作同时更新 MySQL 和 Redis，可能覆盖

方案 B：先删除 Redis，再更新 MySQL
  问题：更新 MySQL 期间，另一个请求来读→ Redis 缓存未命中→ 读 MySQL（旧值）→ 写 Redis（旧值）→ Redis 中永远是旧值

方案 C：先更新 MySQL，再删除 Redis（推荐）
  问题：极端情况下，Redis 删除后、下一个请求来读 MySQL 前——读到旧值并写入 Redis
  但这个问题持续不到 1ms，可接受
```

你选了方案 C——简单、可靠。这是 CAP 理论的工程应用：优先保证一致性（旧值只在 <1ms 窗口内存在）。> **跨书对照**：释慧利第2章 2.3 节讨论了 BASE 理论——缓存更新策略是 BASE 的具体体现。江峰第7章用注册器定义了"读到的值是哪个写操作的值"——缓存一致性的本质是注册器在 Redis 层上的实现。

### 1.5.2 缓存雪崩与穿透

**缓存雪崩**：大量缓存同时过期 → 流量全部打到数据库 → 数据库崩溃。

预防：TTL 加上随机值——不是所有缓存在同一秒过期：`TTL = 600 + random(0, 60)` 秒。

**缓存穿透**：查询一个不存在的 key → Redis 未命中 → 每次都打到数据库 → 恶意攻击。

预防：布隆过滤器——对不存在的 key 快速判断为"不存在"，不查数据库。

## 1.6 第五步——分库分表，数据库的终极扩展

### 1.6.1 决策：不能再通过加从库解决了

日活涨到 200 万，每天订单 ~20 万笔。单表数据量超过 5000 万行。即使是走索引的查询也要 50-100ms。

**第四刀：分库分表。**

```
用户表（1亿行）→ 按 user_id % 16 分到 16 个库
订单表（5亿行）→ 按 order_id 分 32 个表
```

**分库后的代价**：

1. **跨库查询不可用**：`SELECT * FROM orders WHERE user_id = ? AND create_time > ?` 要做两次：先在用户库查 user_id → 确定在哪几个分库 → 再在每个分库查订单
2. **分布式事务**：创建订单需要操作两个不同的分库（用户库减余额 + 订单库创订单）→ 本地事务无效 → 需要 2PC 或 TCC
3. **分页查询复杂**：`LIMIT 100, 20` 需要从 16 个库各取 120 条，然后合并排序取前 20 条

**这个决定不是轻率的**——分库后，你永远失去了"关系数据库的灵活性"。但到了这个规模，要么分库，要么整体迁移到 NoSQL 数据库——而 NoSQL 也有自己的代价。

## 1.7 架构演进的本质——每一刀都留下疤痕

### 1.7.1 回顾：五刀手术

| 步骤 | 决策 | 解决什么 | 代价 |
|------|------|---------|------|
| 1 | 应用与数据分离 | 数据库 CPU 占用 | 多两台服务器 |
| 2 | 应用集群 + Session集中 | 单机 CPU 瓶颈 | Redis 成为新的单点 |
| 3 | 读写分离 | 数据库读瓶颈 | 主从延迟 |
| 4 | 缓存 | 读操作打垮数据库 | 一致性问题 |
| 5 | 分库分表 | 单表数据量爆炸 | 跨库查询复杂 |

### 1.7.2 原则：渐进式演进，而不是一步到位

**"没有 1000 万日活，不需要分库分表。"**

很多初创公司一开始就上微服务、分库分表、K8s——但日活只有 5000。这叫做**过度设计**。它的危害：

- 系统复杂度增加 → 开发效率降低
- 出问题时，排查链路变长 → 故障恢复变慢
- 招聘门槛提高 → 找不到合适的工程师

**架构演进的正确节奏**：每一步都是**被迫的**——不是"架构师提前规划"，而是"业务逼着你做"。

## 1.8 本章总结

### 五条核心决策的记录

作为"趣买"CTO，你在不同阶段做了五个关键决策：

| 日活 | 决策 | 如果不做会怎样 |
|------|------|--------------|
| 5,000 | 应用与数据分离 | 数据库 CPU 先被打满 → Web+DB 共同崩溃 → 全站宕机 |
| 20,000 | 应用集群+Session集中 | 单机 CPU 饱和 → 响应从 80ms 飙升到 2000ms → 用户流失 |
| 100,000 | 读写分离 | 读操作打满数据库 → 每次查询从 10ms 涨到 500ms → 首页加载 >3s |
| 500,000 | Redis 缓存 | 数据库从库 CPU 到 90% → 只能继续加从库 → 成本线性增长 |
| 2,000,000 | 分库分表 | 单表 5000 万行 → 索引查询也要 100ms → 数据量继续涨 → 整个 MySQL 不可用 |

### 如果再来一次，你会怎么判断

1. **不要过早优化**：日活 500 时不需要读写分离——维护成本 > 性能收益
2. **也不要过晚反应**：CPU 到 80% 就预警——不要等 95% 才动手，那时用户已经开始流失
3. **先监控，后动手**：用 Prometheus + Grafana 可视化系统瓶颈——不要凭"感觉"判断
4. **每一刀只解决一个问题**：不要为了"万一以后需要"而做——如果你现在就分库分表但日活只有 5 万，你在浪费团队的开发时间
