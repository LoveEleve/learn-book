## 2.1 一个请求的 500ms 去哪了——打开 Chrome DevTools

### 2.1.1 你没有优化前的时间线

你是"趣买"CTO。用户投诉"首页太慢了"。你打开 Chrome Network 面板，看到一个请求的完整耗时：

```
DNS 解析:        20ms   (域名 → IP)
TCP 握手:        30ms   (SYN → SYN-ACK → ACK)
TLS 握手:        50ms   (证书验证 + 密钥交换)
服务器处理:      200ms  (业务逻辑 + 数据库)
数据传输:        80ms   (HTML 大小 200KB，带宽 2.5MB/s)
浏览器渲染:      120ms  (解析 HTML + 加载 CSS/JS + 渲染)
────────────────
总计:            500ms
```

**关键发现**：服务器处理 200ms 只占 40%——剩下 300ms 花在了网络和浏览器上。**性能优化不是只优化后端——而是优化整条链路。**

### 2.1.2 你的优化目标

你想把这个 500ms 降到 50ms。但每毫秒的"砍法"不同：

| 阶段 | 当前耗时 | 目标耗时 | 怎么做 |
|------|---------|---------|--------|
| DNS + TCP + TLS | 100ms | 10ms | 长连接复用 + CDN 就近访问 |
| 服务器处理 | 200ms | 20ms | 缓存 + 异步 + 池化 |
| 数据传输 | 80ms | 5ms | 压缩 + CDN 边缘节点 |
| 浏览器渲染 | 120ms | 15ms | 页面静态化 + 骨架屏 |
| **总计** | **500ms** | **50ms** | **10 倍优化** |

**这个 10 倍优化不是靠"更好的代码"——而是靠架构层面的决策。**

## 2.2 CDN——让用户就近获取内容

### 2.2.1 问题：用户在北京，服务器在杭州

DNS 解析 20ms + TCP 30ms + TLS 50ms = 100ms。这些延迟是因为用户的请求必须跨越 1000 公里到达你的服务器。

**CDN（Content Delivery Network）的原理**：把你的静态资源（图片、CSS、JS、HTML）复制到全国各地的边缘节点。用户请求时，最近的节点响应。

```
用户(北京) → 请求 index.html → CDN 北京节点（缓存命中）→ 10ms 返回
                │ (未命中)
                └→ 回源到杭州服务器 → 获取 → 缓存到北京节点 → 下次命中
```

**CDN 缓存什么？**
- 静态资源：图片（JPEG/PNG/WebP）、CSS、JS 文件——**TTL=7 天**
- 半静态内容：商品详情页——**TTL=10 分钟**（商品信息不常变）
- 动态内容：用户的订单列表——**不缓存**（每个用户不同）

### 2.2.2 回源率——CDN 性能的核心指标

**回源率 = CDN 未命中的请求数 / 总请求数**。你的目标是把回源率压到 <5%。

怎么做低回源率？**预热**：发布新版本时，提前把热门商品的详情页 HTML 推送到 CDN 节点——上线瞬间就有缓存，不会打回源站。

### 2.2.3 CDN 的代价

CDN 不是免费的——每 GB 流量收费。你需要明确"哪些值得缓存"：
- 商品图片：每个用户访问都加载 → **必缓存**
- 管理后台页面：只有运营人员访问 → **不必缓存**（流量太小，不值得花 CDN 费用）

## 2.3 反向代理缓存——Nginx 层的第一道防线

### 2.3.1 问题：CDN 没命中，请求打到源站

CDN 的回源率 5%——每天还是有 25 万请求打到你的 Nginx。这些请求如果都穿透到 Tomcat + 数据库，你的应用服务器还是扛不住。

**反向代理缓存**：在 Nginx 层做一层缓存。CDN 未命中的请求 → Nginx 检查自己的缓存 → 命中直接返回（1ms）→ 不穿透到应用服务器。

```
用户 → CDN（命中率 95%）→ Nginx proxy_cache → 应用服务器（命中率 20%）→ 数据库
                      ← 95% 在这里结束                ← 20% 在这里结束
```

Nginx proxy_cache 配置：

```
proxy_cache_path /data/nginx/cache levels=1:2 keys_zone=page_cache:10m;
server {
    location / {
        proxy_cache page_cache;
        proxy_cache_valid 200 10m;  # 200 响应缓存 10 分钟
        proxy_cache_key "$uri$is_args$args";
    }
}
```

### 2.3.2 缓存策略——缓存什么，不缓存什么

| 内容 | 缓存吗 | 理由 |
|------|--------|------|
| 商品详情页 | 缓存 10 分钟 | 商品信息不常变——10 分钟延迟可接受 |
| 首页 | 缓存 1 分钟 | 变化较快——但不能每次都查数据库 |
| 搜索结果 | 不缓存 | 每次搜索条件不同——缓存命中率极低 |
| 个人订单 | 不缓存 | 每个用户不同——缓存命中率为 0 |

## 2.4 多级缓存——逐层减少数据库压力

### 2.4.1 从 CDN 到数据库——你建立了三层防线

```
第1层 CDN          (命中率 95%)
  │ 未命中（5%）→
第2层 Nginx 代理缓存 (命中率 20%)
  │ 未命中（80%）→
第3层 Redis 分布式缓存 (命中率 70%)
  │ 未命中（30%）→
第4层 本地缓存 Caffeine (命中率 40%)
  │ 未命中（60%）→
     数据库查询
```

**每一层的价值**：
- CDN 95% → 只打 5% 到源站
- Nginx 20% → 再拦截 20% 的源站请求
- Redis 70% → 再拦截 70% 的应用层请求
- Caffeine 40% → 再拦截 40% 的 Redis miss

**Caffeine 为什么比 Guava Cache 好？W-TinyLFU 算法**

你之前的项目用的是 Guava Cache——它用 LRU（最近最少使用）淘汰策略。但 LRU 有一个致命问题：**它只看"最近有没有被访问"，不看"历史上被访问了多少次"**。

例子：一个商品详情页因为一次大促被频繁访问了 1 分钟——LRU 把它保留在缓存中。大促结束后，这个商品不再被访问——但 LRU 不会淘汰它（因为"最近一次访问"是 1 分钟前，不算"很久没用了"）。同时，一个新上架的商品（还没有被访问过）想要进入缓存——但 LRU 里的旧数据不淘汰——新数据进不来。

**Caffeine 的 W-TinyLFU** 同时看**两个维度**：
- **频率（Frequency）**：这个 key 被访问了多少次——用 Count-Min Sketch 近似统计（O(1) 空间，允许小误差）
- **新近度（Recency）**：这个 key 最近什么时候被访问——用 Window-TinyLFU 的窗口机制处理突发热点

**淘汰决策**：Caffeine 比较"候选淘汰对象"和"新来对象"的频率——**只有新来对象的频率更高，才会替换旧的**。这就是为什么 Caffeine 在命中率上比 LRU 高 5-30%。

```
Guava LRU: 看"最近有没有被访问"
Caffeine W-TinyLFU: 看"历史上被访问了多少次"
  高频冷数据不会被热点驱逐（LRU做不到）← 这就是差距所在
```

**生产配置**：

```
Cache<String, Product> cache = Caffeine.newBuilder()
    .maximumSize(10_000)
    .expireAfterWrite(5, TimeUnit.MINUTES)
    .recordStats()             // 必须开——否则你不知道命中率
    .build();

// 监控
cache.stats().hitRate();       // 命中率——目标 >70%
cache.stats().evictionCount(); // 淘汰数——如果激增，说明 maxSize 太小
```

**最终打到数据库的请求比例** = 5% × 80% × 30% × 60% = **0.72%**。每 1000 个用户请求，只有 7 个真正查数据库。

### 2.4.2 本地缓存 Caffeine——最后一道"热屏障"

当 Redis 也没有命中时，应用服务器在自己的 JVM 内存中（Caffeine 缓存）做一次检查。虽然命中率只有 40%，但响应时间是 **1μs**——比 Redis（1ms）快 1000 倍。

**为什么命中率只有 40%？** 因为 Caffeine 的缓存只在本地 JVM 中——如果用户请求被负载均衡到另一台 App 服务器——那台服务器上没有这个缓存——需要重新从 Redis 加载。所以本地缓存不适合"需要全局一致性"的数据。

**Caffeine 配置要点**：

```
Cache<String, Product> cache = Caffeine.newBuilder()
    .maximumSize(10_000)      // 最多 1 万条缓存——超出用 LRU 淘汰
    .expireAfterWrite(5, TimeUnit.MINUTES) // 5 分钟后自动过期
    .recordStats();            // 监控命中率
```

**监控命中率**是本地缓存的核心运维手段——如果命中率 < 30%，说明你的缓存策略需要调整（TTL 太短、或 key 设计有问题）。

### 2.4.3 缓存穿透与缓存雪崩——两种经典故障

**缓存穿透**：查询一个不存在的 key → 每层缓存都没命中 → 每次都打到数据库。

攻击者可以故意请求 `product/-1`、`product/-2`、`product/-3`——全部不存在——全部穿透到数据库——数据库瞬间被打满。

**防御**：布隆过滤器。在 Redis 层预加载所有"合法 key"——对不合法的 key 直接返回空，不查数据库。

**缓存雪崩**：大量缓存同时过期 → 所有请求同时穿透到数据库。

例如：商品列表缓存 TTL=10 分钟。如果有 10 万个商品——它们被加载的时刻可能非常接近——导致这 10 万个缓存在同一时刻一起过期。

**防御**：TTL 加随机偏移——`TTL = 600 + random(0, 60)` 秒。10 万个缓存的过期时间散落在 600-660 秒区间——不会同一时刻全部过期。

## 2.5 异步化——把不紧急的事情往后挪

### 2.5.1 问题：一个请求在等你做一堆不紧急的事

用户下单。你的代码顺序执行：

```
1. 扣库存          (20ms)
2. 扣余额          (30ms)
3. 创建订单         (15ms)
4. 发积分           (50ms)  ← 不紧急！用户不需要立即看到积分变化
5. 发送通知         (80ms)  ← 不紧急！短信可以延迟 5 秒
6. 更新统计         (30ms)  ← 不紧急！可以等 1 分钟
────────────────
总计:              (225ms)
```

实际上**前 3 步是关键路径**（用户必须等待），后 3 步可以异步化。

### 2.5.2 异步化后的请求路径

```
1. 扣库存          (20ms)
2. 扣余额          (30ms)
3. 创建订单         (15ms)
────────────────
返回用户:          (65ms)  ← 降了 160ms！

异步任务:
4. 发送消息到 MQ → 发积分 (50ms, 异步)
5. 发送消息到 MQ → 发送通知 (80ms, 异步)
6. 发送消息到 MQ → 更新统计 (30ms, 异步)
```

**消息队列在这里起什么作用？** 它保证异步任务**最终会被执行**——即使应用服务器宕机，MQ 中的消息也不会丢失。这和第1章的架构演进一致——"不丢消息"依赖 MQ 的持久化和重试机制。

### 2.5.3 RocketMQ 事务消息——如何保证"本地事务和消息发送"的原子性？

异步化有一个致命问题：**创建订单和发送 MQ 消息是两步操作——如果第一步成功、第二步失败，消息丢了，积分永远不会发。**

```
// 错误写法——本地事务和 MQ 发送不是原子的
@Transactional
public void createOrder(Order order) {
    orderMapper.insert(order);        // 步骤1：插入订单
    rocketMQ.send(createOrderMsg);    // 步骤2：发送消息 → 如果这里抛异常，事务已提交了
}
```

**RocketMQ 事务消息**（半消息 + 回查机制）：

```
T0: Producer 发送半消息（half message）到 Broker
    → 半消息对 Consumer 不可见（标记为"待确认"）
T1: Producer 执行本地事务（如插入订单）
T2: Producer 根据本地事务结果：
    - 成功 → 发送 Commit 给 Broker → 半消息变为可见，Consumer 可以消费
    - 失败 → 发送 Rollback 给 Broker → 半消息被删除
T3: 如果 Broker 在超时内（默认 15s）未收到 Commit/Rollback：
    → Broker 主动回查 Producer："T1 的本地事务状态是什么？"
    → Producer 检查本地事务状态（如查订单表）→ 回复 Commit 或 Rollback
```

**为什么半消息+回查能保证最终一致性？**
- 如果 Producer 在 T1 执行本地事务成功后崩溃（还没发 Commit）——Broker 的回查会发现订单已存在——帮忙发 Commit——消息最终被投递
- 如果 Producer 在 T1 执行本地事务失败后崩溃——Broker 的回查会发现没有订单——帮忙发 Rollback——半消息被丢弃

**与本地消息表模式的关系**：

> **跨书对照**：释慧利第2章 2.3.5 节详细讨论了 BASE 理论的工程实现——本地消息表 + afterCommit 模式。RocketMQ 事务消息是本地消息表的工业实现——RocketMQ Broker 承担了"轮询数据库+重试发送"的职责。两者都是 BASE 理论（基本可用+最终一致）的具体实践。——差异在于：本地消息表是应用层实现，RocketMQ 事务消息是中间件层实现。

### 2.5.4 异步化的代价

**不是所有操作都可以异步化**——你需要明确区分"关键路径"和"非关键路径"：

| 操作 | 关键吗 | 为什么 |
|------|--------|--------|
| 扣库存 | ✅ 关键 | 不能超卖——必须同步确认 |
| 扣余额 | ✅ 关键 | 钱不能错——必须同步确认 |
| 发积分 | ❌ 非关键 | 用户不需要立即看到积分变化——延迟 5 秒可接受 |
| 发短信 | ❌ 非关键 | 短信延迟 30 秒——用户感知不到 |
| 更新推荐 | ❌ 非关键 | 推荐模型训练以分钟计——异步是自然的 |

**如果把"关键"操作异步化了——你可能会遇到什么？** 用户下单成功——但库存还没扣——钱还没减——用户以为下单成功，实际失败了。这就是异步化的工程代价——你必须保证"至少一次执行"和"最终一致性"。

## 2.6 池化技术——减少创建与销毁的开销

### 2.6.1 数据库连接池——HikariCP

每个 HTTP 请求创建一个数据库连接 → 执行查询 → 关闭连接。看起来没问题——但"创建连接"本身的开销被你忽略了：

```
创建 TCP 连接:  30ms (TCP 握手)
MySQL 握手:     10ms (认证 + 初始化)
执行查询:       10ms (SQL 执行)
────────────────
总计:           50ms  (创建连接占 80%！)
```

**HikariCP 连接池**：预先创建一批连接，请求复用它。创建连接的开销被摊销到每个请求中——几乎为零。

```yaml
spring:
  datasource:
    hikari:
      maximum-pool-size: 20      # 最大 20 个连接
      minimum-idle: 5            # 最少保留 5 个空闲连接
      connection-timeout: 30000  # 等待连接的超时（30s）
      idle-timeout: 600000       # 空闲连接的最大存活时间（10min）
```

**池大小公式**：`pool-size = (cpu-cores × 2) + disk-count`。例如 4 核 + 1 块 SSD → 建议池大小为 9 个。注意：**不是越大越好**——池太大 → 数据库端需要维护大量空闲连接 → 内存浪费 + 上下文切换。

### 2.6.2 线程池——Tomcat + 业务

**Tomcat 线程池**：处理 HTTP 请求的线程。默认最大 200 线程。

**为什么需要限制线程数？** 如果每个请求创建一个新线程 → 1000 QPS → 1000 个线程 → JVM 上下文切换开销爆炸 → CPU 浪费在"切换"而不是"处理"上。

**业务线程池**：异步任务的线程池。与 Tomcat 线程池分离——避免异步任务抢占 HTTP 请求的线程。

```java
@Bean("bizExecutor")
public ThreadPoolTaskExecutor bizExecutor() {
    ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
    executor.setCorePoolSize(10);     // 常驻 10 个线程
    executor.setMaxPoolSize(50);      // 最大 50 个线程
    executor.setQueueCapacity(500);   // 等待队列容量 500
    executor.setRejectedExecutionHandler(
        new CallerRunsPolicy());      // 队列满 → 由调用者线程执行
    return executor;
}
```

**拒绝策略**——CallerRunsPolicy：当队列满时，新任务由**提交任务的线程**（即 Tomcat 线程）执行。虽然会短暂降低 HTTP 处理能力，但不会丢失任务。

### 2.6.3 HikariCP 的生产关键参数

HikariCP 能成为 Spring Boot 2.x+ 默认连接池，不仅是"快"——而是一套完整的连接生命周期管理。以下是生产环境必须关注的四个参数：

| 参数 | 含义 | 默认值 | 生产建议 |
|------|------|--------|---------|
| `connection-test-query` | 在将连接交给应用前，发送这条测试 SQL 验证连接是否存活 | 无（依赖 JDBC4 的 `isValid()`） | MySQL：`SELECT 1`（比 `isValid()` 更可靠——`isValid()` 只测 TCP，不测 MySQL 层） |
| `leak-detection-threshold` | 连接借出超过这个时间还没归还 → 打印警告日志（连接泄漏检测） | 0（关闭） | 2000ms（发现泄漏时日志中包含调用栈，直接定位代码） |
| `keepalive-time` | 空闲连接超过这个时间 → HikariCP 发送 `SELECT 1` 保持连接存活 | 0（关闭） | 60000ms（防止 MySQL 的 `wait_timeout` 断开空闲连接——默认 8 小时，但很多 DBA 调成 600s） |
| `validation-timeout` | `connection-test-query` 执行的最大等待时间 | 5000ms | 1000ms（快速判定——如果 1 秒里连 `SELECT 1` 都没返回，这个连接大概率已经坏了） |

**连接泄漏检测的价值**：

你写了 100 行代码，结果忘了关数据库连接。HikariCP 在 `leak-detection-threshold=2000` 下会在 2 秒后打印：

```
[HikariPool-1] Connection leak detection triggered for conn1, 
 stack trace follows:
  at com.example.OrderService.createOrder(OrderService.java:42)
```

不需要看代码——直接定位到 `OrderService.java:42`——就是这个位置的代码忘了关连接。**这就是生产故障排查的救命稻草。**

### 2.6.4 Nginx 的 sendfile 和 gzip——你不花钱能做的最大优化

**sendfile：零拷贝传输静态文件**

普通方式：磁盘 → 内核缓冲区 → 用户缓冲区 → 内核发送缓冲区 → 网卡（4 次拷贝）。sendfile：磁盘 → 内核缓冲区 → 网卡（2 次拷贝）。少两次用户态切换。

```
sendfile on;  # 一行配置，静态文件传输性能 +50%
```

**gzip 压缩**

200KB 的 HTML → 经过 gzip 压缩 → 40KB。传输量减少 80%。

```
gzip on;
gzip_types text/plain text/css application/json application/javascript text/xml;
gzip_min_length 1000;  # 小于 1KB 的不压缩——压缩的 CPU 开销 > 传输收益
```

**收益数据**：

| 场景 | 优化前 | 优化后 | 节省 |
|------|--------|--------|------|
| 首页 HTML 200KB | 传输 200KB | 传输 40KB（gzip） | 80% |
| 商品列表 JSON 50KB | 3次拷贝（无sendfile） | 2次拷贝（sendfile on） | 33%延迟 |

**两行配置**——不花钱、不需要代码改动。这就是运维和架构的区别——架构师知道"哪里可以不花代码就能优化"。

## 2.7 压测验证——调优前后数据对比

作为 CTO，你需要用数据而不是"感觉"证明优化的效果。以下是使用 JMeter 对"商品详情页"接口的压测对比：

| 指标 | 优化前 | 优化后 |
|------|--------|--------|
| QPS | 200 | 1800（9x） |
| 平均 RT | 420ms | 45ms |
| P99 RT | 850ms | 90ms |
| 数据库 CPU | 65% | 8% |
| Redis 命中率 | 0%（未使用） | 73% |
| Caffeine 命中率 | 0%（未使用） | 38% |
| 缓存穿透率 | — | 布隆过滤器拦截了所有不法查询 |

**关键发现**——QPS 从 200 涨到 1800（9 倍提升），不是因为代码更快了——而是因为 92% 的请求根本没打到数据库。**高性能架构的核心不是"让数据库更快"，而是"让数据库不被访问"。**

## 2.8 本章总结——你的高性能工具箱

### 从 500ms 到 50ms 的完整路径

| 阶段 | 优化前 | 优化后 | 用的什么 |
|------|--------|--------|---------|
| DNS + TCP + TLS | 100ms | 10ms | CDN 就近访问 + HTTP/2 长连接 |
| 服务器处理 | 200ms | 20ms | 多级缓存 + 异步化 + 池化 |
| 数据传输 | 80ms | 5ms | Gzip 压缩 + CDN 边缘节点 |
| 浏览器渲染 | 120ms | 15ms | 页面静态化 + 骨架屏 |
| **总计** | **500ms** | **50ms** | **10 倍优化** |

### 高性能架构的核心原则

1. **优化整条链路，不是只优化代码**——网络 100ms + 浏览器 120ms 占了 44%——这部分不是你写更好的 Java 代码能解决的。CDN 和长连接是网络层的解法。
2. **建多层防线**——CDN(95%)→Nginx(20%)→Redis(70%)→Caffeine(40%)——每一层拦截一部分请求，打到数据库的只剩 0.72%。
3. **异步化非关键操作**——把积分、通知、统计从 225ms 的主路径中剔除 → 降到 65ms。消息队列保证异步任务最终执行。
4. **池化减少创建开销**——HikariCP 让"创建数据库连接"的时间从 40ms 降到几乎为零。线程池防止上下文切换爆炸。
5. **缓存是双刃剑**——命中率高 → 性能高。但穿透和雪崩可以瞬间打垮数据库。布隆防穿透，随机TTL防雪崩。
