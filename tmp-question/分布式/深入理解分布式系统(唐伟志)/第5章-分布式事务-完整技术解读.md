# 第5章 分布式事务

> **本章定位**：从第 4 章的"共识算法"到"业务层 ACID"之间的桥梁——分布式事务。
>
> 第 4 章的共识算法保证 CAP C（副本一致），但业务需要的是 ACID C（业务约束）。**分布式事务是把共识算法包装为业务可用的 ACID 保证**——跨节点的原子性、一致性、隔离性、持久性。
>
> 本章从"为什么单机 ACID 在分布式下失效"出发，按"经典理论（2PC/3PC）→ 工程模式（TCC/Saga）→ 大规模方案（Percolator）→ 工业实现（Seata）"的顺序展开，最后给出选型决策框架。
>
> **跨书视角声明**：

> - **唐伟志《深入理解分布式系统》第 5 章 分布式事务** 视角次之：理论 + Go 实现
> - **江峰《分布式高可用算法》第 8 章 共识** 视角给形式化基础（读者推导，江峰原书未讨论具体事务协议）
> - **陈东明《分布式系统与一致性》第 8 章 Spanner** 视角给工业案例（Spanner 的跨分片事务）
>

## 1. 为什么需要分布式事务？

### 1.1 单机事务的局限

**单机事务**通过 ACID 保证：
- **A**tomicity（原子性）：要么全做要么全不做（undo log + redo log）
- **C**onsistency（一致性）：事务前后数据库满足约束
- **I**solation（隔离性）：并发事务互不干扰（锁 / MVCC）
- **D**urability（持久性）：提交后永久保留（WAL + fsync）

单机事务的 ACID 建立在"单机 WAL"基础上——所有操作在同一个数据库实例内，崩溃恢复时通过 redo/undo log 保证原子性。

**但分布式场景下，单机 ACID 失效**：
- 数据分散在多节点（分库分表）
- 操作跨多个服务（微服务）
- 每个节点独立崩溃恢复

### 1.2 典型的分布式事务场景

**场景 1：跨库转账**
```
账户 A（库 1）扣 100 元
账户 B（库 2）加 100 元
```
两个库独立提交——如何保证"要么都成功，要么都失败"？

**场景 2：电商下单**
```
订单服务：创建订单
库存服务：扣减库存
优惠券服务：核销优惠券
积分服务：增加积分
```
四个微服务——如何保证"全部成功或全部回滚"？

**场景 3：跨表事务（分库分表）**
```
订单表（分片 1）：插入订单
订单详情表（分片 2）：插入详情
```
ShardingSphere 分片后，跨分片事务如何保证 ACID？

### 1.3 分布式事务的核心难题

**难题 1：原子性**
- 单机：undo log + redo log 保证
- 分布式：跨节点如何保证"全部 commit 或全部 abort"？

**难题 2：一致性**
- 单机：数据库约束 + 触发器
- 分布式：跨节点约束如何检查？

**难题 3：隔离性**
- 单机：锁 / MVCC
- 分布式：跨节点锁？分布式 MVCC？

**难题 4：持久性**
- 单机：WAL + fsync
- 分布式：每个节点独立 WAL，如何协调？

**难题 5：部分失败**
- 单机：要么全成功要么全失败（事务原子性）
- 分布式：节点 A 成功、节点 B 失败、节点 C 超时——如何处理？

### 1.4 分布式事务的分类

按一致性保证强度：

| 类型 | 一致性 | 典型方案 | 适用场景 |
|------|--------|---------|---------|
| **强一致** | 线性一致 | 2PC / 3PC / XA | 金融账户 |
| **业务级强一致** | 业务原子性 | TCC | 高并发 |
| **最终一致** | 最终一致 | Saga / 本地消息表 / Seata AT | 互联网高并发 |
| **快照隔离** | 快照一致 | Percolator / Spanner | 大规模分布式 |

按实现层：

| 层 | 方案 | 特点 |
|---|------|------|
| **数据库层** | 2PC / 3PC / XA | 数据库原生支持 |
| **业务层** | TCC / Saga | 业务补偿 |
| **基础设施层** | Percolator / Seata | 框架封装 |

## 2. ACID 在分布式下的挑战

### 2.1 原子性的挑战

**单机原子性**：通过 WAL 实现
```
BEGIN
  操作 1
  操作 2
  操作 3
COMMIT  ← 写 commit record，事务原子完成
```
崩溃恢复：有 commit record → redo；无 commit record → undo。

**分布式原子性**：操作跨多节点
```
BEGIN
  节点 A 操作
  节点 B 操作
  节点 C 操作
COMMIT  ← 如何让 A/B/C 全部 commit 或全部 abort？
```

**核心问题**：没有一个"全局 WAL"能记录跨节点事务的 commit record。

**解决方向**：
- 引入协调者（Coordinator）统一决定 commit/abort
- 协调者用共识算法保证自身高可用
- 这就是 2PC 的设计动机（第 3 节）

### 2.2 一致性的挑战

**单机一致性**：数据库约束（外键、唯一性、检查约束）

**分布式一致性**：跨节点约束
- 例：账户 A 在库 1，账户 B 在库 2，约束"A + B 总和不变"——跨库检查约束
- 单机数据库无法检查跨库约束

**解决方向**：
- 应用层检查约束（业务代码）
- 分布式数据库（TiDB / Spanner）原生支持跨分片约束

### 2.3 隔离性的挑战

**单机隔离性**：锁 / MVCC
- 锁：行锁 / 表锁
- MVCC：多版本并发控制

**分布式隔离性**：跨节点锁？分布式 MVCC？

**分布式锁的问题**：
- 跨节点加锁延迟高
- 死锁检测复杂

**分布式 MVCC 的问题**：
- 全局时间戳如何分配？（时间戳分配机制详见第 6 章，本章第 7 节 Percolator 会用到）
- 跨节点事务顺序如何确定？

**解决方向**：
- 全局时间戳分配（TSO / TrueTime / HLC）
- 快照隔离（Snapshot Isolation）
- 这就是 Percolator / Spanner 的设计动机（第 7 节）

### 2.4 持久性的挑战

**单机持久性**：WAL + fsync

**分布式持久性**：每个节点独立 WAL
- 节点 A commit 后崩溃——恢复时能否保留？
- 节点 B 未 commit 就崩溃——恢复时能否回滚？

**解决方向**：
- 协调者记录"全局 commit record"（2PC 的协调者日志）
- 每个参与者记录"prepare record"（2PC 的第一阶段）

### 2.5 部分失败的挑战

**部分失败**是分布式事务最棘手的问题：
```
节点 A：成功
节点 B：失败
节点 C：超时（未知状态）
```

**处理选择**：
- 如果回滚 A：A 的操作撤销
- 如果重试 B：B 的操作重做
- C 超时：无法判断 C 是否已执行

**核心难题**：调用方无法区分"超时"是哪种情况（第 1 章 3.1 节）。

**解决方向**：
- 幂等设计（重试不产生副作用）
- 补偿事务（反向操作）
- 这就是 TCC / Saga 的设计动机（第 5/6 节）

### 2.6 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | 第 3 章 2PC/3PC 解决原子性 + 持久性的跨节点协调问题 |
| **江峰** | 形式化 | ACID 在分布式下的挑战本质是"部分失败 + 网络不可靠"（读者推导，江峰原书未讨论具体事务协议） |
| **唐伟志** | 工程 | 工程上用"协调者 + 共识 + 补偿"组合解决 ACID 挑战 |
| **陈东明** | 案例 | Spanner 用 TrueTime + Paxos 解决跨分片 ACID（第 8 章） |

**4 视角差异与互补**：

- **互补关系**：释慧利给"算法方案"、江峰给"挑战本质"、唐伟志给"工程组合"、陈东明给"工业案例"

### 2.7 本节小结

ACID 在分布式下的挑战：
- **原子性**：无全局 WAL → 协调者 + 2PC
- **一致性**：跨节点约束难检查 → 应用层 / 分布式数据库
- **隔离性**：跨节点锁 / MVCC → 全局时间戳 + 快照隔离
- **持久性**：独立 WAL → 协调者日志 + 参与者 prepare
- **部分失败**：超时不可区分 → 幂等 + 补偿

带着这些挑战进入第 3 节——2PC。

## 3. 2PC（两阶段提交）

### 3.1 2PC 的核心问题

2PC 回答一个问题：**如何让多个节点就"commit 还是 abort"达成一致？**

### 3.2 2PC 的角色

**协调者（Coordinator）**：
- 决定 commit 还是 abort
- 通常由应用层或事务管理器担任

**参与者（Participant）**：
- 执行实际操作
- 通常由数据库担任

### 3.3 2PC 的两阶段

**阶段 1：Prepare（准备阶段）**
```
协调者 → 所有参与者：PREPARE
参与者：
  - 执行操作（但不 commit）
  - 写 prepare log（持久化）
  - 返回 YES（可以 commit）或 NO（不能 commit）
```

**阶段 2：Commit / Abort（提交/中止阶段）**
```
如果所有参与者都返回 YES：
  协调者 → 所有参与者：COMMIT
  参与者：
    - 写 commit log
    - 释放锁
    - 返回 ACK

如果有任何参与者返回 NO（或超时）：
  协调者 → 所有参与者：ABORT
  参与者：
    - 写 abort log
    - 回滚操作
    - 释放锁
    - 返回 ACK
```

### 3.4 2PC 的不变量

**不变量 1**：如果任何参与者 commit，则所有参与者都 prepare 过
- 协调者只在所有参与者 YES 后才发 COMMIT

**不变量 2**：如果任何参与者 abort，则没有任何参与者 commit
- 协调者只在有参与者 NO 后才发 ABORT

**这两个不变量保证原子性**——全部 commit 或全部 abort。

### 3.5 2PC 的故障场景

**场景 1：协调者在 Prepare 后崩溃**
- 参与者已 prepare，但未收到 COMMIT/ABORT
- 参与者**阻塞**——不知道该 commit 还是 abort
- 直到协调者恢复并重发决定

**场景 2：参与者在 Prepare 后崩溃**
- 协调者收不到该参与者的 YES
- 协调者决定 ABORT（超时）
- 其他参与者 abort
- 崩溃参与者恢复后，根据本地 log 判断状态——有 prepare log → 询问协调者 → ABORT

**场景 3：协调者在 Commit 后崩溃**
- 部分参与者收到 COMMIT，部分未收到
- 未收到的参与者**阻塞**
- 协调者恢复后重发 COMMIT

### 3.6 2PC 的阻塞问题

**2PC 是阻塞协议**——协调者崩溃时参与者可能无限阻塞。

**为什么阻塞？**
- 参与者 prepare 后，不能单方面决定 commit/abort
- 必须等协调者的决定
- 协调者崩溃 → 参与者等待

**阻塞的影响**：
- 参与者持有锁，其他事务等待
- 系统可用性下降
- 可能死锁（多个事务互相等待）

### 3.7 2PC 的工程实现

**XA 协议**（X/Open 标准）：
- XA 是 2PC 的工业标准
- MySQL / PostgreSQL / Oracle 都支持 XA
- Java JTA（Java Transaction API）基于 XA

**MySQL XA 示例**：
```sql
XA START 'xid1';
INSERT INTO account VALUES (...);
XA END 'xid1';
XA PREPARE 'xid1';  -- 阶段 1
XA COMMIT 'xid1';   -- 阶段 2
```

**XA 的问题**：
- 性能差（两阶段 + 锁持有时间长）
- 协调者单点（崩溃导致阻塞）
- 不适合高并发互联网场景

### 3.8 2PC 与共识的关系

**2PC 本身不是共识算法**：
- 2PC 的协调者是单点（不用共识）
- 2PC 不满足 Termination（协调者崩溃时阻塞）

**2PC + 共识 = 强一致分布式事务**：
- 协调者用共识算法保证高可用（如 Spanner 的协调者基于 Paxos）
- 参与者用共识保证副本一致
- 这是 Spanner / TiDB 的设计（第 7 节 + 第 4 章）

### 3.9 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | 第 3 章 2PC 的两阶段是"协调者统一决定"的最小设计——Prepare 保证参与者准备好，Commit/Abort 统一决定 |
| **江峰** | 形式化 | 2PC 不满足 Termination（协调者崩溃时阻塞），不是真正的共识算法（读者推导） |
| **唐伟志** | 工程 | 工程上 2PC 的协调者单点是核心问题——用共识算法保证协调者高可用 |
| **陈东明** | 案例 | MySQL XA / Spanner 的协调者都基于 2PC，但 Spanner 用 Paxos 保证协调者高可用 |

### 3.10 本节小结

2PC 是经典分布式事务方案：
- **两阶段**：Prepare + Commit/Abort
- **保证原子性**：协调者统一决定
- **阻塞问题**：协调者崩溃时参与者阻塞
- **工程实现**：XA 协议（MySQL / PostgreSQL）
- **与共识关系**：2PC 本身不是共识，但可与共识组合

带着 2PC 的阻塞问题进入第 4 节——3PC。

## 4. 3PC（三阶段提交）

### 4.1 3PC 的核心问题

3PC 回答一个问题：**如何解决 2PC 的阻塞问题？**

### 4.2 3PC 的设计思路

**3PC 的关键 insight**：2PC 阻塞的根因是"协调者崩溃后参与者不知道其他参与者的状态"。如果参与者能**超时自动决定**，就不必等协调者。

**3PC 的解决方案**：把 2PC 的 Commit 阶段拆成两步——CanCommit + PreCommit + DoCommit，让参与者在 PreCommit 后能超时自动 commit。

### 4.3 3PC 的三阶段

**阶段 1：CanCommit（询问阶段）**
```
协调者 → 所有参与者：CanCommit?
参与者：
  - 检查能否执行（不锁资源）
  - 返回 YES / NO
```

**阶段 2：PreCommit（预提交阶段）**
```
如果所有参与者返回 YES：
  协调者 → 所有参与者：PreCommit
  参与者：
    - 执行操作（但不 commit）
    - 写 prepare log
    - 返回 ACK
```

**阶段 3：DoCommit（提交阶段）**
```
协调者 → 所有参与者：DoCommit
参与者：
  - 写 commit log
  - 释放锁
  - 返回 ACK

如果**已进入 PreCommit 状态**的参与者超时未收到 DoCommit：
  - 参与者自动 commit（关键差异！）
  - 未进入 PreCommit 的参与者超时则自动 abort
```

### 4.4 3PC 的非阻塞特性

**3PC 的关键机制**：参与者超时后**自动 commit**。

**为什么能自动 commit？**
- 参与者收到 PreCommit，意味着所有参与者都 CanCommit=YES
- 如果协调者崩溃，参与者可以推断"既然到了 PreCommit，说明大家都准备好了"
- 超时后自动 commit 是安全的

**但有问题**：
- 网络分区时，部分参与者收到 PreCommit，部分未收到
- 收到 PreCommit 的参与者超时自动 commit
- 未收到 PreCommit 的参与者超时自动 abort
- **数据不一致**！

### 4.5 3PC 的局限

**局限 1：网络分区下不安全**
- 上述场景：部分 commit 部分 abort，违反原子性
- 3PC 假设"网络可靠或同步"——但第 1 章已说明网络不可靠

**局限 2：性能更差**
- 三阶段比两阶段多一次往返
- 锁持有时间更长

**局限 3：实际很少使用**
- 3PC 的非阻塞特性在网络分区下不成立
- 工程上很少直接用 3PC

### 4.6 3PC 的价值

虽然 3PC 实际很少使用，但它的设计思路有价值：
- **超时自动决定**：减少对协调者的依赖
- **阶段拆分**：把阻塞点分散

这些思路被后续方案借鉴（如 TCC 的 Try 阶段类似 CanCommit）。

### 4.7 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | 第 3 章 3PC 通过 PreCommit 阶段让参与者超时自动决定，解决 2PC 阻塞 |
| **江峰** | 形式化 | 3PC 在同步网络下非阻塞，但在异步网络（分区）下仍可能不一致（读者推导） |
| **唐伟志** | 工程 | 3PC 实际很少使用，但"超时自动决定"思路被 TCC 借鉴 |
| **陈东明** | 案例 | 工业上几乎无 3PC 实现——主流是 2PC（XA）或 TCC/Saga |

### 4.8 本节小结

3PC 解决 2PC 阻塞，但引入新问题：
- **三阶段**：CanCommit + PreCommit + DoCommit
- **非阻塞**：参与者超时自动 commit
- **网络分区下不安全**：部分 commit 部分 abort
- **实际很少使用**：主流是 2PC 或 TCC/Saga

带着"3PC 的超时自动决定"思路进入第 5 节——TCC。

## 5. TCC（Try-Confirm-Cancel）

### 5.1 TCC 的核心问题

TCC 回答一个问题：**如何在业务层实现分布式事务，避免 2PC 的阻塞？**

### 5.2 TCC 的设计思路

**TCC 的关键 insight**：2PC 的阻塞根因是"数据库层持锁等待"。如果把事务逻辑上移到业务层，每个阶段都执行业务代码（不持数据库锁），就能避免阻塞。

**TCC 的三个阶段**：
- **T**ry：预留资源（不实际执行）
- **C**onfirm：确认执行（实际提交）
- **C**ancel：取消预留（回滚）

### 5.3 TCC 的三阶段

**阶段 1：Try（尝试）**
```
Try-扣库存：冻结 100 件（不实际扣减）
Try-扣款：冻结 100 元（不实际扣减）
Try-创建订单：预创建（状态=待确认）
```

**阶段 2：Confirm（确认）**
```
如果所有 Try 成功：
  Confirm-扣库存：实际扣减 100 件（解冻 + 扣减）
  Confirm-扣款：实际扣减 100 元
  Confirm-创建订单：状态=已确认
```

**阶段 3：Cancel（取消）**
```
如果有任何 Try 失败：
  Cancel-扣库存：解冻 100 件
  Cancel-扣款：解冻 100 元
  Cancel-创建订单：状态=已取消
```

### 5.4 TCC 的关键特性

**特性 1：业务层事务**
- TCC 的每个阶段都是业务代码
- 不依赖数据库锁
- 不阻塞其他事务

**特性 2：幂等性**
- Confirm 和 Cancel 必须幂等
- 重试不产生副作用
- 网络超时后可安全重试

**特性 3：业务级强一致（中间状态可见）**
- Try 后，资源被冻结（中间状态对外可见，业务级弱一致）
- Confirm/Cancel 完成后，资源达到**业务级强一致**（Try/Confirm/Cancel 在业务层保证 all-or-nothing）
- 与 2PC 的差异：2PC 是数据库层强一致（持锁），TCC 是业务层强一致（无锁，但业务层原子）

### 5.5 TCC 的工程挑战

**挑战 1：业务侵入**
- 每个操作要实现 Try/Confirm/Cancel 三个接口
- 业务代码复杂度增加 3 倍

**挑战 2：幂等性保证**
- Confirm/Cancel 可能被重试
- 必须保证幂等（如用事务 ID 去重）

**挑战 3：空回滚**
- Try 未执行，但 Cancel 被调用
- 例：Try 超时，协调者发 Cancel，但 Try 实际未到达
- 解决：Cancel 时检查 Try 是否执行，未执行则空回滚

**挑战 4：悬挂**
- Cancel 先于 Try 到达
- 例：网络延迟，Cancel 先到，Try 后到
- 解决：Cancel 后记录状态，Try 时检查是否已 Cancel

### 5.6 TCC 的工业实现

**Seata TCC 模式**（第 8 节详解）：
- Seata 框架封装 TCC
- 业务实现 Try/Confirm/Cancel 接口
- Seata 协调者管理事务状态

**Hmily**（京东金融开源，作者喻海华，现归属 dromara 开源组织）：
- Java TCC 框架
- 支持注解式 TCC

**EasyTransaction**：
- 轻量级 TCC 框架

### 5.7 TCC vs 2PC

| 维度 | 2PC | TCC |
|------|-----|-----|
| 层 | 数据库层 | 业务层 |
| 锁 | 数据库锁（阻塞） | 无锁（业务冻结） |
| 一致性 | 数据库层强一致 | 业务层强一致 |
| 性能 | 低（锁等待） | 高（无锁） |
| 业务侵入 | 无 | 高（3 倍代码） |
| 适用场景 | 金融账户 | 互联网高并发 |

### 5.8 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | TCC 是 3PC 的业务层变种——Try≈CanCommit，Confirm≈DoCommit，Cancel≈Abort |
| **江峰** | 形式化 | TCC 的正确性依赖"幂等 + 空回滚 + 防悬挂"三个工程不变量（读者推导） |
| **唐伟志** | 工程 | TCC 的核心代价是业务侵入（3 倍代码），但性能远优于 2PC |
| **陈东明** | 案例 | Seata TCC 模式 / Hmily 是工业实现——封装了幂等/空回滚/悬挂处理 |

### 5.9 本节小结

TCC 是业务层分布式事务：
- **三阶段**：Try（预留）+ Confirm（确认）+ Cancel（取消）
- **无锁**：业务层冻结资源，不持数据库锁
- **最终一致**：中间状态可见，最终一致
- **工程挑战**：幂等 / 空回滚 / 悬挂
- **工业实现**：Seata TCC / Hmily

带着 TCC 的补偿思路进入第 6 节——Saga。

## 6. Saga（长事务补偿）

### 6.1 Saga 的核心问题

Saga 回答一个问题：**如何处理长事务（跨多个服务的业务流程），避免长时间持锁？**

### 6.2 Saga 的设计思路

**Saga 的关键 insight**：长事务（如订单流程：下单→支付→发货→收货）持续数小时甚至数天，不可能用 2PC 持锁。**把长事务拆成一系列短事务，每个短事务有对应的补偿事务**。

**Saga 的结构**：
```
T1 → T2 → T3 → ... → Tn（正向执行）
如果某步失败：
  C1 ← C2 ← C3 ← ... ← Cm（反向补偿）
```

### 6.3 Saga 的两种实现

**编排式（Choreography）**：
- 无中央协调者
- 每个服务监听事件，执行本地事务 + 发布新事件
- 例：订单服务发布"订单创建"事件 → 库存服务监听并扣库存 → 发布"库存扣减"事件 → ...

**协调式（Orchestration）**：
- 有中央协调者（Saga 协调器）
- 协调者依次调用各服务
- 失败时协调者调用补偿

### 6.4 Saga 的补偿示例

**正向流程**：
```
T1：创建订单（状态=待支付）
T2：扣减库存
T3：扣减积分
T4：订单状态=已支付
```

**补偿流程**（T3 失败）：
```
C2：恢复库存
C1：订单状态=已取消
（T3 的补偿 C3 不需要，因为 T3 失败了）
```

**关键**：补偿不是"回滚"，而是"反向操作"。例如 T2 扣减库存的补偿 C2 是增加库存。

### 6.5 Saga 的特性

**特性 1：最终一致**
- Saga 不保证强一致
- 中间状态可见（如订单已创建但库存未扣）
- 最终通过补偿达到一致

**特性 2：无锁**
- 每个短事务持锁时间短
- 不跨服务持锁

**特性 3：适合长流程**
- 订单 / 支付 / 物流等长流程
- 跨多个服务的业务

### 6.6 Saga 的工程挑战

**挑战 1：补偿的幂等性**
- 补偿可能被重试
- 必须幂等

**挑战 2：补偿的顺序**
- 补偿必须按反向顺序执行
- 例：T1→T2→T3 失败，补偿顺序 C2→C1（C3 不需要）

**挑战 3：不可补偿的操作**
- 某些操作不可补偿（如发邮件 / 短信）
- 解决：把不可补偿操作放在 Saga 末尾，或用"通知+确认"模式

**挑战 4：空补偿**
- 补偿执行了，但原事务实际未执行
- 解决：用事务 ID 去重

### 6.7 Saga 的工业实现

**Seata SAGA 模式**（第 8 节详解）：
- Seata 框架封装 Saga
- 状态机驱动（State Machine Engine）
- 支持编排式

**Camunda**：
- 工作流引擎，支持 Saga 模式

**Eventuate Tram Sagas**：
- 微服务 Saga 框架

### 6.8 Saga vs TCC

| 维度 | TCC | Saga |
|------|-----|------|
| 一致性 | 最终一致（Try 后冻结） | 最终一致（T 后可见） |
| 持锁 | Try 阶段持锁 | 短事务持锁 |
| 业务侵入 | 高（3 倍代码） | 中（每步需补偿） |
| 适用场景 | 短事务（秒级） | 长事务（分钟~天） |
| 补偿 | Cancel（反向） | C（反向） |

### 6.9 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | Saga 是"长事务 = 一系列短事务 + 补偿"的模式 |
| **江峰** | 形式化 | Saga 的正确性依赖"补偿幂等 + 反向顺序"（读者推导） |
| **唐伟志** | 工程 | Saga 适合长流程，但补偿的复杂性是核心挑战 |
| **陈东明** | 案例 | Seata SAGA / Camunda 是工业实现——状态机驱动补偿 |

### 6.10 本节小结

Saga 是长事务补偿方案：
- **拆分**：长事务 = 一系列短事务
- **补偿**：每步有对应补偿事务
- **最终一致**：中间状态可见，最终一致
- **两种实现**：编排式 / 协调式
- **工业实现**：Seata SAGA / Camunda

带着 Saga 的补偿思路进入第 7 节——Percolator。

## 7. Percolator（Google 快照隔离）

### 7.1 Percolator 的核心问题

Percolator 回答一个问题：**如何在大规模分布式系统（PB 级数据）中实现跨行事务 + 快照隔离？**

### 7.2 Percolator 的背景

**Percolator**（Google 2010 论文）为 Google 索引系统设计：
- 处理 PB 级数据
- 跨行事务（修改多行）
- 快照隔离（Snapshot Isolation）
- 基于 BigTable 存储

### 7.3 Percolator 的核心机制

**机制 1：时间戳**
- 每个事务分配一个全局唯一时间戳（Timestamp Oracle）
- 时间戳单调递增
- 用于快照隔离

**机制 2：锁**
- 事务修改数据时，写"锁"标记
- 锁包含：事务 ID + 时间戳 + 主键
- 其他事务看到锁，等待或清理

**机制 3：写**
- 数据有多个版本（按时间戳）
- 事务 commit 时，写入新版本（带 commit 时间戳）

**机制 4：通知**
- 事务 commit 后，触发通知（更新索引等）

### 7.4 Percolator 的事务流程

**阶段 1：Prewrite（预写）**
```
1. 获取时间戳 T1
2. 对所有要修改的行：
   - 检查是否有冲突（其他事务的锁或更新版本）
   - 写入锁（事务 ID + T1 + 主键）
   - 写入数据（版本 T1）
3. 选择一个主键，其他为从键
```

**阶段 2：Commit（提交）**
```
1. 获取时间戳 T2
2. 对主键：
   - 写入 commit 记录（版本 T2）
   - 清除锁
3. 对从键：
   - 异步写入 commit 记录
   - 清除锁
4. 事务完成
```

### 7.5 Percolator 的快照隔离

**快照隔离（Snapshot Isolation）**：
- 事务 T 在时间戳 T_start 读数据
- 读到的数据是 T_start 之前的最新已提交版本
- 写入的数据在 T_commit 时可见

**与线性一致性的差异**：
- 快照隔离不保证线性一致
- 但避免了"写偏斜"（Write Skew）等异常

### 7.6 Percolator 的故障恢复

**场景：事务在 Prewrite 后崩溃**
- 锁未清除
- 其他事务看到锁，尝试清理
- 清理规则：如果主键的 commit 记录存在，则 commit；否则 abort

**场景：事务在 Commit 中崩溃**
- 主键已 commit，从键未 commit
- 其他事务看到从键的锁，检查主键
- 主键已 commit → 从键也 commit（异步补齐）

### 7.7 Percolator 的工业影响

**TiDB**：PingCAP 的分布式数据库，基于 Percolator 事务模型
- TiKV 存储（多 Raft 组）
- PD 分配时间戳（TSO）
- Percolator 事务模型

**Spanner**：Google 的全球数据库
- 用 TrueTime 替代 TSO
- 用 Paxos 保证副本一致
- 事务模型类似 Percolator

### 7.8 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | Percolator 通过"时间戳 + 锁 + 多版本"实现跨行事务 + 快照隔离 |
| **江峰** | 形式化 | Percolator 的快照隔离是"Snapshot Isolation"级别，弱于线性一致（读者推导） |
| **唐伟志** | 工程 | TiDB 基于 Percolator 事务模型 + TiKV（多 Raft）+ PD（TSO） |
| **陈东明** | 案例 | Spanner 用 TrueTime 替代 TSO，实现全球快照隔离（第 8 章） |

### 7.9 本节小结

Percolator 是大规模分布式事务方案：
- **时间戳**：全局唯一单调递增（TSO）
- **锁**：跨行事务协调
- **多版本**：快照隔离
- **两阶段**：Prewrite + Commit
- **工业影响**：TiDB / Spanner

带着 Percolator 的事务模型进入第 8 节——Seata 工业实现。

## 8. Seata 工业实现

### 8.1 Seata 的核心问题

Seata 回答一个问题：**如何让业务开发者不用手写 2PC/TCC/Saga，用一个框架统一管理分布式事务？**

### 8.2 Seata 的架构

**Seata**（阿里开源，原名 Fescar）的核心组件：

```
┌─────────────────────────────────────┐
│  TC（Transaction Coordinator）       │ ← 事务协调者
│  - 维护全局事务状态                    │
│  - 决定 commit/rollback              │
└─────────────┬───────────────────────┘
              │
   ┌──────────┴──────────┐
   ↓                     ↓
┌──────────┐         ┌──────────┐
│   TM     │         │   RM     │
│ 事务管理器│         │ 资源管理器│
│ - 开启全局│         │ - 分支事务│
│   事务    │         │ - 上报状态│
│ - 决定提交│         │           │
│   /回滚   │         │           │
└──────────┘         └──────────┘
```

**三个角色**：
- **TC**（Transaction Coordinator）：事务协调者，独立部署
- **TM**（Transaction Manager）：事务管理器，发起全局事务
- **RM**（Resource Manager）：资源管理器，管理本地资源（数据库）

### 8.3 Seata 的四种模式

**AT 模式（Auto Transaction）**：
- 自动模式，业务无侵入
- 基于 SQL 解析 + undo log
- 两阶段：阶段 1 执行 SQL + 记录 undo log；阶段 2 异步删除 undo log（成功）/ 用 undo log 回滚（失败）
- **默认模式，最常用**

**TCC 模式**：
- 业务实现 Try/Confirm/Cancel
- Seata 协调者管理状态
- 适合高并发场景

**SAGA 模式**：
- 状态机驱动
- 支持长事务
- 业务实现正向 + 补偿

**XA 模式**：
- 基于数据库 XA 协议
- 强一致
- 性能较差

### 8.4 Seata AT 模式详解

**AT 模式的核心**：自动生成 undo log，业务无侵入。

**阶段 1（业务执行）**：
```
1. 解析 SQL → 2. 查询 before image（执行前数据）→ 3. 执行 SQL → 4. 查询 after image（执行后数据）→ 5. 组装 undo log（含 before/after image）
2. 执行 SQL
3. 记录 undo log（含 before image）
4. 注册分支事务到 TC
5. 本地事务提交（SQL + undo log 在同一本地事务）
```

**阶段 2（提交）**：
```
TC 通知 RM 提交：
  - 异步删除 undo log
  - （数据已提交，无需其他操作）
```

**阶段 2（回滚）**：
```
TC 通知 RM 回滚：
  - 根据 undo log 的 before image，生成反向 SQL
  - 执行反向 SQL（恢复数据）
  - 删除 undo log
```

**AT 模式的关键**：
- **前后镜像**：记录 SQL 执行前后的数据状态
- **undo log**：用于回滚
- **一阶段提交**：不像 2PC 持锁到二阶段，AT 模式一阶段就提交本地事务

### 8.5 Seata AT 的隔离性

**写隔离**：
- AT 模式用全局锁（global lock）保证写隔离
- 一阶段：执行 SQL 前，先获取全局锁
- 二阶段提交：释放全局锁
- 二阶段回滚：用 undo log 回滚后释放全局锁

**读隔离**：
- 默认读未提交（Read Uncommitted）
- 可配置读已提交（Read Committed）—— 通过 SELECT FOR UPDATE 走全局锁

### 8.6 Seata 的工业应用

**电商下单场景**：
```
TM（订单服务）：开启全局事务
  RM（订单服务）：创建订单（AT 模式）
  RM（库存服务）：扣减库存（AT 模式）
  RM（优惠券服务）：核销优惠券（AT 模式）
TM：提交全局事务
  TC：通知所有 RM 提交
```

如果库存扣减失败：
```
TM：回滚全局事务
  TC：通知所有 RM 回滚
    订单服务：用 undo log 回滚订单
    优惠券服务：用 undo log 回滚优惠券
```

### 8.7 Seata vs 其他方案

| 方案 | 一致性 | 性能 | 业务侵入 | 适用场景 |
|------|--------|------|---------|---------|
| Seata AT | 最终一致 | 高 | 无 | 通用（默认） |
| Seata TCC | 业务级强一致 | 高 | 高 | 高并发 |
| Seata SAGA | 最终一致 | 中 | 中 | 长事务 |
| Seata XA | 强一致 | 低 | 无 | 金融账户 |
| 2PC（XA） | 强一致 | 低 | 无 | 传统数据库 |
| Percolator | 快照隔离 | 中 | 无 | 大规模分布式 |

### 8.8 Seata TC 高可用与存储模式

**TC 集群高可用**：
- 生产环境 TC 必须集群部署（3 节点）
- 通过注册中心（Nacos / Eureka）做服务发现
- TM/RM 从注册中心获取 TC 列表，负载均衡连接

**存储模式**（Seata 1.4+ 支持 4 种）：

| 模式 | 存储 | 特点 | 适用场景 |
|------|------|------|---------|
| **file** | 本地文件 | 单机，无高可用 | 开发测试 |
| **db** | 数据库 | 持久化，需 DB 高可用 | 生产（传统） |
| **redis** | Redis | 高性能，需 Redis 高可用 | 生产（高性能） |
| **raft** | Raft 共识 | **内置高可用**（基于 sofa-jraft），无外部依赖 | 生产（推荐，1.5+） |

**Raft 模式的意义**：这是与第 4 章"共识算法"的关键呼应——Seata TC 用 Raft 保证自身高可用，TC 内部用共识算法保证事务状态一致。

### 8.9 消息事务（补充）

除了上述方案，**消息事务**是互联网最终一致的另一主流方案：

**RocketMQ 事务消息**：
- 半消息（half message）+ 回查机制
- 流程：发送半消息 → 执行本地事务 → 提交/回滚半消息
- 如果本地事务超时，RocketMQ 回查生产者
- 适合"本地事务 + 消息通知"场景

**Kafka 事务**：
- 幂等生产者 + 事务 API
- 跨分区原子写入
- 适合流处理场景（ Exactly-Once 语义）

**本地消息表**（模式，非框架）：
- 本地事务 + 消息表同库
- 异步发送消息到 MQ
- 消息重试 + 幂等消费
- 适合"无 Seata"的轻量级最终一致

**消息事务 vs Saga**：
- 消息事务：单向通知，无补偿
- Saga：有补偿，可回滚

### 8.10 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | Seata AT 是 2PC 的工程优化——一阶段提交 + undo log 回滚 |
| **江峰** | 形式化 | Seata 的正确性依赖"undo log + 全局锁"两个不变量（读者推导，江峰原书未讨论具体事务协议） |
| **唐伟志** | 工程 | Seata 的核心价值是"业务无侵入"——AT 模式自动生成 undo log |
| **陈东明** | 案例 | 阿里 / 蚂蚁大量使用 Seata，是工业级分布式事务框架 |

### 8.11 本节小结

Seata 是工业级分布式事务框架：
- **四种模式**：AT / TCC / SAGA / XA
- **AT 模式**：自动生成 undo log，业务无侵入
- **三角色**：TC / TM / RM
- **工业应用**：电商下单等微服务场景

带着 Seata 的工业实现进入第 9 节——对比与选型。

## 9. 分布式事务对比与选型

### 9.1 方案对比矩阵

| 方案 | 一致性 | 性能 | 业务侵入 | 适用场景 | 典型实现 |
|------|--------|------|---------|---------|---------|
| 2PC | 强一致 | 低 | 无 | 金融账户 | MySQL XA |
| 3PC | 强一致（分区下不保证） | 低 | 无 | 理论研究 | 几乎无 |
| TCC | 业务级强一致 | 高 | 高 | 高并发 | Seata TCC / Hmily |
| Saga | 最终一致 | 高 | 中 | 长事务 | Seata SAGA / Camunda |
| Percolator | 快照隔离 | 中 | 无 | 大规模分布式 | TiDB |
| Seata AT | 读未提交（默认）/ 读已提交（配置） | 高 | 无 | 通用 | Seata |
| Spanner | 外部一致性（external consistency，强于线性一致） | 中 | 无 | 全球数据库 | Spanner |

### 9.2 选型决策树

```
业务需求
   ↓
是否需要强一致？
   ├─ 是 → 是否跨地域？
   │       ├─ 是 → Spanner（TrueTime + Paxos）
   │       └─ 否 → 2PC（XA）/ Seata XA
   └─ 否 → 是否长事务？
           ├─ 是 → Saga（Seata SAGA / Camunda）
           └─ 否 → 是否高并发？
                   ├─ 是 → TCC（Seata TCC / Hmily）
                   └─ 否 → Seata AT（默认）
```

### 9.3 选型案例

**案例 1：银行转账**
- 需求：强一致 + 跨库
- 推荐：2PC（XA）/ Seata XA
- 理由：金融账户必须强一致

**案例 2：电商下单**
- 需求：高并发 + 最终一致
- 推荐：Seata AT
- 理由：业务无侵入 + 性能好

**案例 3：订单流程（下单→支付→发货→收货）**
- 需求：长事务 + 最终一致
- 推荐：Seata SAGA
- 理由：长流程 + 补偿

**案例 4：秒杀扣库存**
- 需求：极高并发 + 最终一致
- 推荐：TCC（Seata TCC）
- 理由：业务层冻结 + 无锁

**案例 5：跨地域分布式数据库**
- 需求：全球强一致 + SQL
- 推荐：Spanner / CockroachDB
- 理由：TrueTime/HLC + 跨地域

### 9.4 选型的工程考量

**考量 1：一致性需求**
- 金融账户：强一致（2PC / XA）
- 互联网应用：最终一致（TCC / Saga / AT）

**考量 2：并发量**
- 高并发：TCC / Saga / AT（无锁或短锁）
- 低并发：2PC（持锁可接受）

**考量 3：事务时长**
- 短事务（秒级）：2PC / TCC / AT
- 长事务（分钟~天）：Saga

**考量 4：业务侵入**
- 无侵入：2PC / AT / Percolator
- 可侵入：TCC / Saga

**考量 5：现有架构**
- 微服务 + Spring Cloud：Seata
- 传统数据库：XA
- 大数据：Percolator

### 9.5 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | 选型本质是"一致性 vs 性能"的权衡——强一致用 2PC，最终一致用 TCC/Saga |
| **江峰** | 形式化 | 选型必须匹配一致性需求——线性一致 / 快照隔离 / 最终一致（读者推导） |
| **唐伟志** | 工程 | 工程选型要考虑业务侵入 + 现有架构 + 团队熟悉度 |
| **陈东明** | 案例 | 金融用 XA，电商用 Seata，全球数据库用 Spanner——不同场景不同选型 |

### 9.6 本节小结

分布式事务选型：
- **强一致 + 金融**：2PC / XA
- **高并发 + 互联网**：TCC / Seata AT
- **长事务**：Saga
- **大规模分布式**：Percolator / Spanner
- **通用默认**：Seata AT

### 10.2 核心论断

**论断 1：跨节点强一致分布式事务以共识算法为基础设施**
- 共识保证副本一致（CAP C）
- 分布式事务保证业务约束（ACID C）
- **跨节点强一致**事务（2PC/Percolator/Spanner）依赖共识保证副本一致
- **业务层补偿**方案（TCC/Saga）不必然依赖共识——它们在业务层保证原子性，副本一致由底层存储保证

**论断 2：强一致 vs 最终一致是核心权衡**
- 强一致（2PC / XA）：金融账户，性能低
- 最终一致（TCC / Saga / AT）：互联网应用，性能高

**论断 3：Seata 是微服务分布式事务的主流框架**
- AT 模式（无侵入）+ TCC 模式（高并发）+ SAGA 模式（长事务）+ XA 模式（强一致）
- 覆盖绝大多数微服务场景

**论断 4：Percolator / Spanner 是大规模分布式事务的方向**
- 时间戳 + 快照隔离
- TiDB / CockroachDB 跟随
