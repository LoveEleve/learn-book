## 1. 为什么需要 ZAB？

### 1.1 ZooKeeper 的需求

**ZooKeeper**（2008，Yahoo！）是一个为分布式系统提供协调服务的中心化服务。它的核心需求是：

**需求 1：线性一致写**
- 所有写请求按全局顺序执行
- 客户端看到的写顺序一致

**需求 2：高可用**
- 集群中少数节点故障仍能服务
- 通常 3 / 5 / 7 节点部署

**需求 3：低延迟读**
- 读请求可以从任意副本响应（不必须走 Leader）
- 牺牲线性一致读换取低延迟（默认顺序一致读）

**需求 4：容错**
- Leader 故障时能快速切换（秒级）
- 不丢已提交事务

### 1.2 为什么不用 Paxos？

ZooKeeper 设计于 2006-2008 年，当时 Paxos 是唯一成熟的共识算法。但 ZooKeeper 团队（Flavio Junqueira 等）没有直接用 Multi-Paxos，原因：

**原因 1：Paxos 的实现复杂度**
- Multi-Paxos 论文不完整，工程细节需自己设计
- ZooKeeper 需要快速落地，不能花太多时间在算法实现上

**原因 2：ZooKeeper 的需求与 Paxos 的通用性不匹配**
- Paxos 是"通用共识"——任何值达成一致
- ZooKeeper 需要"原子广播"——事务按相同顺序复制
- 通用 Paxos 对 ZooKeeper 来说"过度通用"

**原因 3：性能优化方向不同**
- Paxos 的优化方向是"减少 Prepare 阶段"（Multi-Paxos 的稳定 Leader）
- ZooKeeper 的优化方向是"主从流水线复制"（Leader 串行化 + Follower 流水线 ACK）

### 1.3 ZAB 的设计目标

**ZAB**（ZooKeeper Atomic Broadcast）专为 ZooKeeper 的需求设计：

> **目标**：在 n 个节点（n ≥ 2f+1）的集群中，把 Leader 接收的事务按**全序**原子广播到所有 Follower，保证：
> 1. **Agreement**（一致性）：所有非崩溃节点按相同顺序应用事务
> 2. **Validity**（有效性）：广播的事务必须是 Leader 提议过的
> 3. **Termination**（终止性）：在部分同步假设下，所有非崩溃节点最终应用事务
> 4. **Order**（全序）：事务按全局顺序应用（这是 ZAB 区别于 Paxos 的关键）

**ZAB 与共识的关系**：
- 共识问题：多个节点就单个值达成一致
- 原子广播：多个节点按相同顺序接收一系列值
- **数学等价**（Chandra-Toueg 1996，第 2 章 6.3 节）——原子广播 = 多次共识 + 全序

## 2. ZAB 的设计哲学

### 2.1 原子广播 vs 共识

**共识问题**（Paxos / Raft 解决的）：
- 多个节点就**单个值**达成一致
- 多次共识 → 日志复制（SMR）

**原子广播问题**（ZAB 解决的）：
- Leader 把**一系列事务**按全序广播到所有 Follower
- 所有 Follower 按相同顺序接收

**两者关系**：
- 共识 + 全序 = 原子广播
- 原子广播是共识的特例（事务顺序由 Leader 决定）

**ZAB 的视角**：直接从"原子广播"出发，不绕道"共识"——这让 ZAB 的设计更聚焦于"主从流水线复制"。

### 2.2 主从模型

ZAB 是**强 Leader 模型**（类似 Raft，不同于 Multi-Paxos 的可选 Leader）：

```
Client → Leader ←→ Follower
            ↑
         所有读写
```

**Leader 的职责**：
- 接受客户端写请求
- 分配 zxid（全局唯一递增 ID）
- 把事务广播到所有 Follower
- 决定事务何时提交

**Follower 的职责**：
- 接受 Leader 的广播
- 持久化事务后 ACK
- 应用已提交事务

**与 Raft 的差异**：
- Raft：所有数据流经 Leader（包括读，默认）
- ZAB：写必须走 Leader，读可以从任意 Follower（默认顺序一致读，非线性一致）

### 2.3 两阶段设计

ZAB 把算法分为两个阶段：

**阶段 1：发现 + 同步（Discovery + Synchronization）**
- 集群启动或 Leader 故障时
- 选出新 Leader
- 新 Leader 与所有 Follower 同步状态（确保已提交事务不丢）

**阶段 2：广播（Broadcast）**
- Leader 正常服务期间
- 接受写请求，广播事务
- Follower ACK 后 Leader 提交

**与 Raft 的对比**：
- Raft：选举 + 日志复制 + 安全性（三子问题）
- ZAB：发现同步 + 广播（两阶段）
- 数学等价，但 ZAB 把"选举 + 日志同步"合并为一个阶段

### 2.4 ZAB 的状态机

ZAB 节点有 4 种状态（比 Raft 多 1 种）：

```
                超时，发起选举
  LOOKING ─────────────────→ LOOKING
     │                          │
     │ 选举完成                  │ 选举完成
     ↓                          ↓
  FOLLOWING                  LEADING
     ↑                          │
     │                          │ Leader 故障
     │                          ↓
     └─────────────────────── LOOKING
```

**4 种状态**：
- **LOOKING**：选举中（对应 Raft 的 Candidate）
- **LEADING**：Leader 状态（对应 Raft 的 Leader）
- **FOLLOWING**：Follower 状态（对应 Raft 的 Follower）
- **OBSERVING**：观察者（不参与投票，只接收事务——用于跨地域部署）

**OBSERVING 的特殊性**：
- Observer 不参与 Quorum，不影响可用性
- 主要用于跨地域部署——把 Observer 部署在远端，降低读延迟
- 这是 ZooKeeper 独有的设计（Raft 没有原生 Observer 概念）

### 2.5 本节小结

ZAB 的设计哲学：
- **原子广播视角**（非共识视角）——直接解决"事务全序复制"
- **强 Leader 模型**——所有写经 Leader
- **两阶段设计**——发现同步 + 广播
- **4 种节点状态**——LOOKING / LEADING / FOLLOWING / OBSERVING

带着这个设计哲学进入第 3 节——核心概念。

## 3. 核心概念：zxid 与 epoch

### 3.1 为什么需要 zxid？

ZAB 需要一个**全局唯一、单调递增**的事务 ID，用于：
- 标识事务顺序（全序广播的基础）
- 比较 Leader 的新旧（选举时选 zxid 最大的）
- 检测日志不一致（Follower 与 Leader 同步时）

这个 ID 就是 **zxid**（ZooKeeper Transaction ID）。

### 3.2 zxid 的结构

**zxid 是 64 位整数**，由两部分组成：

```
zxid (64 bit)
├── epoch (32 bit)  ← 高 32 位
└── counter (32 bit) ← 低 32 位
```

**epoch**（任期）：
- 类似 Raft 的 term
- 每次 Leader 切换，epoch 递增
- 保证新 Leader 的事务 ID 大于旧 Leader

**counter**（计数器）：
- 同一 epoch 内单调递增
- 每个事务 counter+1

**比较规则**：
- 先比 epoch：epoch 大者 zxid 大
- epoch 相同时比 counter：counter 大者 zxid 大

**示例**：
```
epoch=1, counter=5  → zxid = 0x0000000100000005 = 4294967301
epoch=2, counter=1  → zxid = 0x0000000200000001 = 8589934593
epoch=2, counter=2  → zxid = 0x0000000200000002 = 8589934594
```

### 3.3 zxid vs Raft term+index

| 维度 | zxid | Raft (term, index) |
|------|------|-------------------|
| 结构 | 64 位整数（epoch + counter） | 两个独立字段（term + index） |
| 单调性 | 全局单调递增 | term 单调，index 在 term 内单调 |
| 比较 | 整数比较 | 先比 term，再比 index |
| 持久化 | 单个字段 | 两个字段 |

**ZAB 的优势**：zxid 是单一整数，比较简单——直接用 `>` 比较。

**Raft 的优势**：term 和 index 分离，语义更清晰——term 表示"任期"，index 表示"日志位置"。

### 3.4 epoch 的作用

**epoch 类似 Raft 的 term**，但有几个关键差异：

**差异 1：epoch 包含在 zxid 中**
- ZAB：epoch 是 zxid 的高 32 位，事务 ID 自带 epoch 信息
- Raft：term 是独立字段，日志条目单独存 term

**差异 2：epoch 切换的时机**
- ZAB：Leader 切换时 epoch+1（每次新 Leader 上任）
- Raft：term 切换的时机更多（选举、发现更高 term）

**差异 3：epoch 的持久化**
- ZAB：epoch 隐含在 zxid 中，持久化 zxid 即持久化 epoch
- Raft：currentTerm 单独持久化

### 3.5 消息类型

ZAB 的核心消息类型（按阶段分组，含辅助消息）：

**阶段 1（发现 + 同步）**：
1. **FOLLOWINFO**：Follower 报告自己的 zxid 和 acceptedEpoch（用于选举）
2. **ACKEPOCH**：Follower 给准 Leader 的 epoch 确认（含 acceptedEpoch，准 Leader 据此确定新 epoch = max(acceptedEpoch)+1）
3. **NEWLEADER**：准 Leader 宣布上任（触发 Follower 同步与 ACK）
4. **DIFF / TRUNC / SNAP**：Leader 发给 Follower 的同步指令（补齐 / 截断 / 快照）

**阶段 2（广播）**：
5. **PROPOSAL**：Leader 提议事务（含 zxid + 事务内容）
6. **ACK**：Follower 确认收到 PROPOSAL（阶段 1 和阶段 2 都用 ACK）
7. **COMMIT**：Leader 通知 Follower 提交事务

> **注**：不同资料对 ZAB 消息类型的枚举略有差异，本篇按 ZooKeeper 工程实现（`Leader.java` / `Follower.java`）归纳为核心 7 类。**待 MCP 核实 ZooKeeper 源码完整消息枚举**。

**与 Raft 的对比**：
- PROPOSAL ≈ Raft 的 AppendEntries（含日志条目）
- ACK ≈ Raft 的 AppendEntriesResponse
- COMMIT ≈ Raft 的 leaderCommit 字段（但 ZAB 是独立消息，Raft 是 AppendEntries 的字段）

### 3.6 ZAB 的持久化状态

每个节点持久化（崩溃后从磁盘恢复）：

```
acceptedEpoch: 已接受的 epoch（类似 Raft 的 currentTerm）
currentEpoch: 当前 epoch
lastZxid: 最后一条事务的 zxid（类似 Raft 的 log[lastIndex]）
dataTree: 数据树（znode 树，类似 Raft 的状态机快照）
```

**为什么持久化 acceptedEpoch？**
- 防止崩溃后用旧 epoch 参与选举
- 类似 Raft 持久化 currentTerm

**为什么持久化 lastZxid？**
- 崩溃后恢复时，需要知道自己的日志进度
- 用于选举时比较（选 lastZxid 最大的节点）

### 3.7 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | 第 5 章 ZAB 的 zxid 设计是"epoch + counter"的组合——epoch 保证 Leader 新旧可比较，counter 保证同 epoch 内全序 |
| **江峰** | 形式化 | zxid 的全序性质是 ZAB 正确性的形式化基础——任意两个 zxid 可比较，保证事务顺序唯一 |
| **唐伟志** | 工程 | 工程上 zxid 是 64 位 long，用位运算高效比较；持久化时单独存 epoch 和 lastZxid |
| **陈东明** | 案例 | ZooKeeper 用 zxid 实现写顺序一致——客户端看到的写顺序与 zxid 顺序一致 |

**4 视角差异与互补**：
- **释慧利 vs 江峰**：释慧利问"zxid 如何设计"（epoch+counter），江峰问"zxid 为什么保证正确性"（全序性质）
- **唐伟志 vs 陈东明**：唐伟志给"工程实现"（64 位 long + 位运算），陈东明给"工业应用"（ZooKeeper 写顺序一致）
- **互补关系**：释慧利给"设计"、江峰给"数学基础"、唐伟志给"工程细节"、陈东明给"工业应用"

### 3.8 本节小结

ZAB 的核心概念：
- **zxid**：64 位整数（epoch 32 位 + counter 32 位），全局唯一单调递增
- **epoch**：类似 Raft 的 term，每次 Leader 切换递增
- **5 种消息**：FOLLOWINFO / LEADING / PROPOSAL / ACK / COMMIT
- **持久化状态**：acceptedEpoch / currentEpoch / lastZxid / dataTree

## 4. 阶段 1：发现 + 同步

### 4.1 阶段 1 的核心问题

阶段 1 回答两个问题：
1. **发现**：如何选出新 Leader？
2. **同步**：新 Leader 如何确保所有 Follower 与自己日志一致？

### 4.2 发现：Leader 选举

**触发条件**：
- 集群启动（所有节点初始 LOOKING）
- Leader 故障（Follower 超时未收到心跳）

**选举流程**：

1. **节点转为 LOOKING**，收集其他节点的 acceptedEpoch
2. **投票给自己**：`vote(myid, lastZxid)`
3. **广播投票**给所有节点
4. **收到其他节点的投票**：
   - 比较投票：(lastZxid, myid) 大者胜出
   - 先比 lastZxid：大者胜
   - lastZxid 相同时比 myid：大者胜
5. **更新自己的投票**：如果收到的投票比自己的大，更新为自己的投票
6. **广播更新后的投票**
7. **收到多数派相同投票**：胜出者成为准 Leader
8. **准 Leader 收集 Follower 的 ACKEPOCH**（含 acceptedEpoch），确定新 epoch = max(acceptedEpoch) + 1
9. **准 Leader 成为正式 Leader**，进入同步阶段（4.3 节）

> **注**：步骤 1 中节点**不直接递增 epoch**，而是收集其他节点的 acceptedEpoch 后，在步骤 8 由准 Leader 确定新 epoch。这避免了多节点并发递增导致 epoch 冲突。

**选举规则的核心**：
- **lastZxid 大者优先**：保证新 Leader 有最完整的日志（类似 Raft 的选举限制）
- **myid 大者次优**：日志相同时用 myid 打破平局（确定性，避免活锁）

**与 Raft 选举的对比**：

| 维度 | ZAB | Raft |
|------|-----|------|
| 触发 | Follower 超时 | Follower 超时 |
| 比较 | lastZxid + myid | lastLogTerm + lastLogIndex |
| 随机化 | 无（用 myid 打破平局） | 随机选举超时 |
| 投票限制 | lastZxid 大者胜 | 日志至少一样新 |
| 活锁风险 | 无（myid 确定性） | 有（需随机化避免） |

**ZAB 的优势**：用 myid 打破平局，避免活锁——但要求 myid 唯一且固定。

**Raft 的优势**：随机化更灵活，不依赖固定节点 ID——但理论上有活锁风险（概率极低）。

### 4.3 同步：Leader 与 Follower 对齐日志

**问题**：新 Leader 上任时，Follower 的日志可能落后或冲突。

**ZAB 的同步策略**：

1. **新 Leader 发送 NEWLEADER 消息**给所有 Follower
2. **Follower 收到 NEWLEADER**，报告自己的 lastZxid（`peerLastZxid`）
3. **Leader 比较 `peerLastZxid` 与自己的日志**，决定发送哪种同步消息：

**DIFF/TRUNC/SNAP 的判定条件**（Leader 视角）：

| 条件 | 发送消息 | 含义 |
|------|---------|------|
| `peerLastZxid` 在 Leader 日志中存在，且 Leader 有更新的 zxid | **DIFF** | Follower 落后，发送 `(peerLastZxid, leaderLastZxid]` 范围内的事务 |
| `peerLastZxid` 在 Leader 日志中**不存在**（Follower 有 Leader 没有的 zxid，通常是旧 Leader 未提交的事务） | **TRUNC** | 让 Follower 截断到 Leader 已有的最大 zxid |
| `peerLastZxid` 远小于 Leader 的 committed zxid（差距超过 `snapCount` / `snapSize` 阈值） | **SNAP** | 发送完整快照（DIFF 数据量太大，不如全量发） |

4. **Follower 完成同步后 ACK NEWLEADER**
5. **Leader 收到多数派 ACK**：宣布同步完成，进入广播阶段

**TRUNC 的场景**：
- 旧 Leader 提议了事务 T（zxid=10），但未达多数派就崩溃
- 新 Leader 没有 T（因为新 Leader 的 lastZxid < 10，或者新 Leader 选 T 但未提交）
- Follower 中有 T → 必须 TRUNC 截断

**与 Raft 的对比**：
- Raft：Leader 通过 nextIndex 回退，找到一致点后覆盖
- ZAB：Leader 直接告诉 Follower 截断或补齐

**ZAB 的特点**：Leader 主动同步（推送 DIFF/TRUNC/SNAP），而非 Raft 的"被动回退"。

### 4.4 NEWLEADER 消息的作用

**NEWLEADER 消息**是 ZAB 阶段 1 的核心：

1. **宣布新 Leader**：Follower 收到后知道新 Leader 是谁
2. **触发同步**：Follower 根据自己的日志状态响应
3. **确认多数派**：Leader 收到多数派 ACK 才进入广播阶段

**为什么需要多数派 ACK？**
- 保证新 Leader 的日志被多数派确认
- 防止"少数派 Leader"——如果只有少数派 ACK，新 Leader 可能没有最新已提交事务

### 4.5 阶段 1 的安全性

**ZAB 阶段 1 保证**：
1. **Leader 唯一**：一个 epoch 只有一个 Leader（多数派选举 + epoch 递增）
2. **日志完整**：新 Leader 有最完整的日志（lastZxid 最大者当选）
3. **已提交不丢**：同步阶段确保所有已提交事务在多数派中（类似 Raft 的选举限制）

**证明（简略）**：
- 假设事务 T（zxid=z）已被提交，意味着多数派 M 复制了 T
- 新 Leader 必须获得多数派 M' 的选票
- M 与 M' 必有交集（鸽笼原理）
- 交集节点既复制了 T 又投票给新 Leader
- 选举规则选 lastZxid 最大者 → 新 Leader 的 lastZxid ≥ z
- **关键前提**：ZAB 日志按 zxid 连续无 gap（事务严格按 zxid 顺序持久化），所以 lastZxid ≥ z 意味着新 Leader 包含所有 zxid ≤ z 的事务，即必有 T

### 4.6 阶段 1 的工程参数

ZooKeeper 的阶段 1 相关参数：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `tickTime` | 2000ms | 基础时间单位 |
| `initLimit` | 10 (× tickTime = 20s) | Leader 选举阶段超时 |
| `syncLimit` | 5 (× tickTime = 10s) | Leader 与 Follower 同步超时 |
| `leaderConnectTimeoutMs` | 5000ms | Follower 连接 Leader 超时 |

**与 Raft 的对比**：
- ZAB 的 initLimit=20s 远大于 Raft 的选举超时 1000ms
- ZAB 倾向"稳定优先"（避免频繁选举），Raft 倾向"快速恢复"
- 这反映了 ZooKeeper（协调服务）vs etcd（KV 存储）的不同需求

### 4.7 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | 第 5 章 ZAB 的阶段 1 是"发现 + 同步"——发现选 Leader（lastZxid 最大），同步保证日志一致 |
| **江峰** | 形式化 | 阶段 1 的正确性依赖"lastZxid 全序 + 多数派公理"——任意两个 zxid 可比较 + 多数派必有交集 |
| **唐伟志** | 工程 | ZooKeeper 的 initLimit=20s 远大于 Raft 的 1000ms，反映"协调服务稳定优先"的设计取向 |
| **陈东明** | 案例 | ZooKeeper 的 Leader 选举在实践中秒级完成（典型 2-5s），20s 是上限 |

**4 视角差异与互补**：
- **释慧利 vs 江峰**：释慧利问"阶段 1 如何工作"（发现+同步），江峰问"阶段 1 为什么正确"（全序+多数派）
- **唐伟志 vs 陈东明**：唐伟志给"参数设计取向"（20s vs 1000ms），陈东明给"实际表现"（2-5s 完成）
- **互补关系**：释慧利给"算法步骤"、江峰给"数学基础"、唐伟志给"参数取向"、陈东明给"工业表现"

### 4.8 本节小结

阶段 1（发现 + 同步）：
- **发现**：Leader 选举（lastZxid 最大 + myid 打破平局）
- **同步**：Leader 主动推送 DIFF/TRUNC/SNAP，Follower 对齐日志
- **NEWLEADER 消息**：宣布新 Leader + 触发同步
- **安全性**：Leader 唯一 + 日志完整 + 已提交不丢

**阶段 1 结束的触发条件**：Leader 收到多数派 ACK NEWLEADER 后，集群进入广播阶段——这意味着新 Leader 的日志已被多数派确认，可以安全接受客户端写请求。

带着阶段 1 的机制进入第 5 节——广播阶段。

## 5. 阶段 2：广播

### 5.1 阶段 2 的核心问题

阶段 2 回答一个问题：**Leader 如何把事务原子广播到所有 Follower？**

"原子广播"包含两个含义：
1. **原子性**：事务要么被所有 Follower 接收，要么都不接收（不存在部分接收）
2. **全序**：所有 Follower 按相同顺序接收事务

### 5.2 广播流程

**Leader 处理写请求的流程**：

1. **客户端发送写请求**到 Leader
2. **Leader 分配 zxid**：`zxid = (epoch << 32) | (++counter)`
3. **Leader 追加事务到本地日志**（持久化）
4. **Leader 并行发送 PROPOSAL** 给所有 Follower
5. **Follower 收到 PROPOSAL**：
   - 持久化事务到本地日志
   - 发送 ACK 给 Leader
6. **Leader 收到多数派 ACK**：
   - 提交事务（COMMIT）
   - 发送 COMMIT 给所有 Follower
7. **Follower 收到 COMMIT**：
   - 应用事务到状态机
8. **Leader 响应客户端**（在 COMMIT 后）

**与 Raft 日志复制的对比**：

| 步骤 | ZAB | Raft |
|------|-----|------|
| 接收请求 | Leader 分配 zxid | Leader 追加日志条目 |
| 持久化 | Leader 持久化 PROPOSAL | Leader 持久化日志条目 |
| 复制 | PROPOSAL 给所有 Follower | AppendEntries 给所有 Follower |
| ACK | Follower 持久化后 ACK | Follower 持久化后 ACK |
| 提交 | Leader 发独立 COMMIT 消息 | Leader 在 AppendEntries 中带 leaderCommit |
| 应用 | Follower 收到 COMMIT 后应用 | Follower 根据 commitIndex 应用 |

**关键差异**：
- ZAB 的 COMMIT 是**独立消息**，Raft 的提交是**AppendEntries 的字段**
- ZAB 的 PROPOSAL + ACK + COMMIT 是三步，Raft 的 AppendEntries + Response 是两步（提交信息在下次 AppendEntries 中捎带）

### 5.3 流水线优化

ZAB 的关键性能优化：**流水线（Pipeline）**。

**没有流水线时**：
```
Leader: PROPOSAL₁ → ACK₁ → COMMIT₁ → PROPOSAL₂ → ACK₂ → COMMIT₂ → ...
Follower: ACK₁ → COMMIT₁ → ACK₂ → COMMIT₂ → ...
```
每个事务必须等待前一个完成——吞吐低。

**有流水线时**：
```
Leader: PROPOSAL₁ → PROPOSAL₂ → PROPOSAL₃ → ACK₁ → ACK₂ → ACK₃ → COMMIT₁ → COMMIT₂ → COMMIT₃
Follower: ACK₁ → ACK₂ → ACK₃ → COMMIT₁ → COMMIT₂ → COMMIT₃
```
Leader 可以连续发送多个 PROPOSAL，不等 ACK——吞吐高。

**流水线的正确性保证**：
- PROPOSAL 按 zxid 顺序发送（FIFO）
- Follower 按 zxid 顺序持久化和 ACK
- Leader 按zxid 顺序提交（COMMIT 也是 FIFO）

**与 Raft 的对比**：
- Raft 也支持流水线（Leader 连续发送 AppendEntries）
- 但 Raft 的提交信息在下次 AppendEntries 中捎带，延迟略高
- ZAB 的独立 COMMIT 消息让提交通知更及时

### 5.4 顺序保证

ZAB 保证事务的全序：

**PROPOSAL 顺序**：
- Leader 按 zxid 递增顺序发送 PROPOSAL
- Follower 按 zxid 顺序接收（FIFO 通道保证）

**COMMIT 顺序**：
- Leader 按 zxid 顺序发送 COMMIT
- Follower 按 zxid 顺序应用

**关键不变量**：
- 如果 Follower 在 zxid=z 提交了事务 T，则所有 zxid < z 的事务都已提交
- 这保证了状态机的全序应用

### 5.5 客户端顺序一致

ZooKeeper 的客户端顺序一致保证：
- 单个客户端看到的写顺序与 zxid 顺序一致
- 单个客户端看到的读顺序与写顺序一致（读自己的写）

**与线性一致性的差异**：
- 线性一致：所有客户端看到的操作顺序与真实时间一致
- 顺序一致：所有客户端看到的操作顺序相同，但不一定与真实时间一致

**ZooKeeper 默认读不保证线性一致**：
- 读可以从任意 Follower 响应
- Follower 可能落后 Leader（异步复制）
- 跨客户端可能看到"回退"

**如何获得线性一致读**：
- 客户端读前调用 `sync()` 操作（等待 Follower 追上 Leader）
- 或所有读走 Leader（性能差）

### 5.6 完整流程示例

考虑 5 节点集群（L, F1, F2, F3, F4），客户端发送 `create /node "data"`：

```
t=1: Client → Leader L: create /node "data"
t=2: L 分配 zxid = (epoch=2 << 32) | counter=5 = 0x0000000200000005
     L 持久化 PROPOSAL(zxid=5, "create /node 'data'")
t=3: L 并行发送 PROPOSAL 给 F1, F2, F3, F4

t=4: F1, F2 持久化 PROPOSAL 后 ACK
     F3 延迟，F4 网络抖动，未及时响应

t=5: L 收到 F1, F2 的 ACK + 自己 = 3 票（多数派 3/5）
     L 提交事务（COMMIT）
     L 发送 COMMIT 给 F1, F2, F3, F4

t=6: L 应用事务到状态机（创建 /node）
     L 响应 Client: 成功

t=7: F1, F2, F3 收到 COMMIT，应用事务
     F4 网络恢复后，收到 PROPOSAL + COMMIT，同步
```

**关键观察**：
- 写入延迟 = Leader 持久化 + 多数派 ACK 往返
- F4 的延迟不影响写入（多数派即可）
- F4 最终会同步（最终一致）

### 5.7 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | 第 5 章 ZAB 的广播是"PROPOSAL + ACK + COMMIT"三步——多数派 ACK 后 Leader 提交，独立 COMMIT 消息通知 Follower |
| **江峰** | 形式化 | 广播的正确性依赖"zxid 全序 + FIFO 通道 + 多数派公理"——三者共同保证事务全序应用 |
| **唐伟志** | 工程 | ZooKeeper 的流水线优化是关键——连续 PROPOSAL 提升吞吐，COMMIT 独立消息保证及时性 |
| **陈东明** | 案例 | ZooKeeper 的写延迟典型 10-50ms（多数派 ACK），读延迟 < 1ms（Follower 本地读） |

**4 视角差异与互补**：
- **释慧利 vs 江峰**：释慧利问"广播如何工作"（三步），江峰问"广播为什么正确"（全序+FIFO+多数派）
- **唐伟志 vs 陈东明**：唐伟志给"工程优化"（流水线），陈东明给"工业性能"（写 10-50ms，读 <1ms）
- **互补关系**：释慧利给"算法步骤"、江峰给"数学基础"、唐伟志给"工程优化"、陈东明给"工业表现"

### 5.8 本节小结

阶段 2（广播）：
- **三步流程**：PROPOSAL → ACK → COMMIT
- **流水线优化**：连续 PROPOSAL 提升吞吐
- **顺序保证**：zxid 全序 + FIFO 通道
- **客户端顺序一致**：单客户端顺序一致，跨客户端不保证线性一致

理解了 ZAB 的两阶段（发现同步 + 广播）后，自然要问：它和 Paxos/Raft 到底差在哪？这是第 6 节的问题。带着广播机制进入第 6 节——与 Paxos/Raft 对比。

## 6. ZAB 与 Paxos/Raft 的对比

### 6.1 三大协议的起源

| 协议 | 年份 | 作者 | 设计动机 |
|------|------|------|---------|
| Paxos | 1990（Lamport 初稿） | Lamport | 通用共识问题 |
| ZAB | 2008（Yahoo!） | Junqueira et al. | ZooKeeper 原子广播需求 |
| Raft | 2014（Ongaro） | Ongaro-Ousterhout | 可理解性 |

**时间线**：Paxos（1990）→ VR（1988，Oki-Liskov）→ ZAB（2008）→ Raft（2014）。**ZAB 和 Raft 都受 Viewstamped Replication（VR）的 view-number 概念启发**（Raft 论文明确引用 VR，未直接引用 ZAB）；ZAB 的 epoch 与 Raft 的 term 是**同源概念**（都源于 VR 的 view-number），非直接因果。

### 6.2 设计哲学对比

| 维度 | Paxos | ZAB | Raft |
|------|-------|-----|------|
| 出发点 | 通用共识 | 原子广播 | 可理解性 |
| Leader 角色 | 可选优化 | 必需设计 | 必需设计 |
| 算法结构 | Prepare + Accept | 发现同步 + 广播 | 选举 + 复制 + 安全性 |
| 设计目标 | 算法最小性 | ZooKeeper 定制 | 易理解易实现 |

**关键洞察**：
- Paxos 是"通用共识"——适用于任何需要达成一致的场合
- ZAB 是"原子广播"——专为 ZooKeeper 的事务全序复制设计
- Raft 是"可理解共识"——用三子问题分解降低复杂度

### 6.3 核心机制对比

| 机制 | Paxos | ZAB | Raft |
|------|-------|-----|------|
| 提案编号 | Proposal ID (round+node) | zxid (epoch+counter) | term |
| Leader 选举 | 未明确（工程实现） | lastZxid + myid | lastLogTerm + lastLogIndex + 随机超时 |
| 日志一致性 | Prepare 阶段保证 | 同步阶段 DIFF/TRUNC | prevLogIndex + prevLogTerm 检查 |
| 提交规则 | 多数派 ACK | 多数派 ACK + COMMIT 消息 | 多数派 ACK + 只提交当前 term |
| 成员变更 | 未明确 | 未明确（ZooKeeper 重新配置） | 联合共识 + 单节点变更 |
| 日志压缩 | 未明确 | Snapshot | Snapshot |

### 6.4 数学等价性

**三个协议在数学上等价**：
- 都保证 Agreement + Validity + 最终 Termination
- 都基于多数派 + Leader + 日志复制
- 都在部分同步 + Crash-recovery 模型下工作

**差异在工程视角**：
- Paxos：从"单值共识"出发，扩展为日志复制
- ZAB：从"原子广播"出发，直接解决全序复制
- Raft：从"可理解性"出发，重新设计共识

### 6.5 工业采用对比

| 系统 | 协议 | 采用原因 |
|------|------|---------|
| Chubby | Multi-Paxos | 历史原因（Raft 未出生） |
| Spanner | Multi-Paxos | 历史原因 + 全球规模优化 |
| MySQL MGR | Paxos 变种（XCom） | 历史原因 |
| ZooKeeper | ZAB | 量身定制 |
| etcd | Raft | 易实现 + 易维护 |
| TiKV | Raft | 易实现 + 多 Raft 组 |
| CockroachDB | Raft | 易实现 + 跨地域 |
| Nacos | Raft（持久）+ Distro（临时） | 易实现 + 双协议 |
| Consul | Raft | 易实现 + 健康检查 |
| SOFA-JRaft | Raft | 蚂蚁开源 |

**趋势**：
- **历史系统**（Chubby / Spanner / MGR / ZooKeeper）：Paxos 或 ZAB
- **新系统**（etcd / TiKV / CockroachDB / Nacos / Consul）：Raft
- **领域差异**：KV / 注册中心倾向 Raft，全球数据库仍用 Paxos，协调服务仍用 ZAB

### 6.6 ZAB 独有的特性

ZAB 有几个 Paxos/Raft 没有的特性：

**特性 1：Observer 节点**
- 不参与 Quorum，只接收事务
- 用于跨地域部署（远端 Observer 降低读延迟）
- Raft 没有原生 Observer（需工程扩展）

**特性 2：顺序一致读（默认）**
- ZooKeeper 默认从 Follower 读，保证单客户端顺序一致
- 不保证线性一致（跨客户端可能回退）
- Raft 默认线性一致读（Leader Read 或 ReadIndex）

**特性 3：临时节点 + Watch**
- ZooKeeper 的临时节点（客户端会话失效自动删除）
- Watch 机制（数据变更通知客户端）
- 这些是 ZooKeeper 的应用层特性，但与 ZAB 的设计紧密相关

### 6.7 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | ZAB 与 Raft 数学等价，差异在工程视角——ZAB 是原子广播，Raft 是共识 |
| **江峰** | 形式化 | 三协议都满足共识三性质，数学上等价 |
| **唐伟志** | 工程 | 工程上 ZAB 的流水线 + Observer 是 ZooKeeper 独特优化 |
| **陈东明** | 案例 | ZooKeeper 是 ZAB 唯一工业实现；Raft 有多个工业实现（etcd/TiKV 等） |

**4 视角差异与互补**：
- **释慧利 vs 江峰**：释慧利问"算法差异在哪"（工程视角），江峰问"数学是否等价"（共识三性质）
- **唐伟志 vs 陈东明**：唐伟志给"工程独特性"（流水线+Observer），陈东明给"工业实现差异"（ZAB 唯一 vs Raft 多个）
- **互补关系**：释慧利给"算法视角"、江峰给"数学等价性"、唐伟志给"工程独特性"、陈东明给"工业实现"

### 6.8 ZAB 与 VR（Viewstamped Replication）的传承

讨论共识协议的演进，必须提到 **VR（Viewstamped Replication，Oki-Liskov 1988）**——它早于 Paxos 论文 2 年，是 ZAB 和 Raft 共同的概念源头。

**VR 的核心思想**：
- view-number（视图编号）：类似 ZAB 的 epoch / Raft 的 term
- 主从复制：Primary 处理请求，Backup 复制
- 视图变更（View Change）：Primary 故障时，Backup 发起视图变更，选新 Primary + 状态同步

**VR → ZAB/Raft 的传承**：

| 概念 | VR（1988） | ZAB（2008） | Raft（2014） |
|------|-----------|-------------|--------------|
| 任期编号 | view-number | epoch | term |
| 主节点 | Primary | Leader | Leader |
| 故障切换 | View Change | 发现 + 同步 | 领导人选举 |
| 状态同步 | 新 Primary 与 Backup 对齐 | DIFF/TRUNC/SNAP | nextIndex 回退 |

**关键认知**：
- ZAB 的 epoch 不是原创——源于 VR 的 view-number
- Raft 的 term 也不是受 ZAB 直接影响——Raft 论文明确引用 VR
- 三者（VR / ZAB / Raft）的同步阶段都是"新主与从节点对齐日志"
- Paxos（1990）与 VR（1988）几乎同期独立提出，但 Paxos 更通用（单值共识），VR 更工程化（主从复制）

**为什么本篇要提 VR？**
- 避免"ZAB 影响 Raft"的历史误植
- 理解共识协议的演进脉络：VR（1988）→ Paxos（1990）→ ZAB（2008）→ Raft（2014）
- VR 的 view-change 机制是理解 ZAB 阶段 1 和 Raft 选举的共同钥匙

### 6.9 本节小结

ZAB 与 Paxos/Raft 的对比：
- **共识三性质等价**：在崩溃故障 + 多数派 + 部分同步假设下，三者都满足 Agreement / Validity / Termination
- **工程视角不同**：ZAB 是原子广播视角，Raft 是共识视角，Paxos 是通用共识视角
- **共同源头**：ZAB 和 Raft 都受 VR（1988）的 view-number 启发
- **ZAB 独有特性**：Observer + 顺序一致读 + 临时节点
- **工业趋势**：新系统倾向 Raft，ZooKeeper 仍用 ZAB（但 Kafka 4.0 已用 KRaft 替代 ZK）

带着这个对比进入第 7 节——ZooKeeper 工业实现。

## 7. ZAB 的工业实现：ZooKeeper

### 7.1 ZooKeeper 的架构

```
              Client
                ↓
         ┌──────────────┐
         │   Leader     │ ← ZAB 协议
         └──────┬───────┘
        ┌───────┼───────┐
        ↓       ↓       ↓
    ┌──────┐ ┌──────┐ ┌──────┐
    │Follower│Follower│Observer│
    └──────┘ └──────┘ └──────┘
```

**节点角色**：
- **Leader**：处理写请求，广播事务
- **Follower**：参与 Quorum，处理读请求
- **Observer**：不参与 Quorum，只处理读请求（跨地域部署）

### 7.2 ZooKeeper 的数据模型

**znode 树**：
- 类似文件系统的层次结构
- 每个 znode 可存储数据（默认 1MB 上限）
- znode 类型：持久 / 临时 / 顺序

**事务日志 + 快照**：
- **事务日志**（`dataLogDir`）：每个事务的 WAL，顺序追加
- **快照**（`dataDir`）：定期全量数据快照
- 恢复时：先加载快照，再 replay 事务日志

### 7.3 ZooKeeper 的关键参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `tickTime` | 2000ms | 基础时间单位 |
| `initLimit` | 10 (20s) | Leader 选举阶段超时 |
| `syncLimit` | 5 (10s) | Leader-Follower 同步超时 |
| `maxClientCnxns` | 60 | 单客户端最大连接数 |
| `autopurge.snapRetainCount` | 3 | 保留的快照数 |
| `autopurge.purgeInterval` | 1 | 自动清理间隔（小时） |
| `jute.maxbuffer` | 1048576 (1MB) | 单个 znode 数据上限 |

### 7.4 ZooKeeper 的应用场景

**场景 1：配置管理**
- 应用从 ZooKeeper 读取配置
- 配置变更时通过 Watch 通知所有应用

**场景 2：服务注册与发现**
- 服务启动时注册到 ZooKeeper（创建临时节点）
- 服务故障时临时节点自动删除
- 客户端通过 Watch 感知服务上下线

**场景 3：分布式锁**
- 多个客户端争抢创建同一节点（创建成功者获得锁）
- 或使用临时顺序节点（避免羊群效应）

**场景 4：Leader 选举**
- 多个服务实例争抢创建 `/leader` 节点
- 创建成功者成为 Leader，其他实例 Watch `/leader`
- Leader 故障时节点删除，触发重新选举

**场景 5：Kafka / HBase / Hadoop 元数据**
- Kafka 早期用 ZooKeeper 存 broker 注册信息和消费者组元数据；**Kafka 2.8+ 引入 KRaft 模式**（基于 Raft 变种）逐步去 ZooKeeper 化，**4.0（2024）已完全移除 ZK 依赖**——这是 ZAB 在大数据生态被 Raft 替代的标志性事件
- HBase 用 ZooKeeper 做 Region Server 注册和 Master 选举
- Hadoop HA 用 ZooKeeper 做 NameNode 主备切换

### 7.5 ZooKeeper 的工程优化

**优化 1：批量提交**
- Leader 可以批量发送 PROPOSAL（多个事务打包）
- 减少网络往返，提升吞吐

**优化 2：Observer 跨地域部署**
- 远端数据中心部署 Observer
- 本地读延迟低，不增加 Quorum 开销

**优化 3：快照 + 日志分离**
- 快照存 `dataDir`，事务日志存 `dataLogDir`
- 建议事务日志用独立磁盘（顺序写性能）

**优化 4：Watch 一次性触发**
- Watch 触发后自动删除（避免重复通知）
- 客户端需要重新注册 Watch
- ZooKeeper 3.6+ 支持持久 Watch（`addWatch` 命令）

### 7.6 ZooKeeper 的局限性

**局限 1：写入性能瓶颈**
- 所有写经 Leader，Leader 是单点
- 典型写吞吐：数千到数万 TPS（取决于事务大小和网络）

**局限 2：存储容量限制**
- 所有数据在内存（znode 树）
- 单集群建议不超过 GB 级数据

**局限 3：不适合大数据存储**
- znode 数据上限 1MB
- 不适合存储大文件或海量数据

**局限 4：跨地域延迟**
- 写必须经 Leader，跨地域写入延迟高
- Observer 只能优化读，不能优化写

### 7.7 ZooKeeper vs etcd

| 维度 | ZooKeeper | etcd |
|------|-----------|------|
| 协议 | ZAB | Raft |
| 数据模型 | znode 树 | KV（扁平） |
| API | 自定义协议 | gRPC + REST |
| 一致性 | 顺序一致读（默认） | 线性一致读（默认） |
| Watch | 一次性触发（3.6+ 持久） | 持久 Watch |
| 事务 | 多操作事务（multi） | 微事务（mini-transaction） |
| 生态 | Hadoop / Kafka / HBase | Kubernetes |
| 语言 | Java | Go |

**选型建议**：
- 大数据生态（Hadoop / Kafka / HBase）：ZooKeeper
- Kubernetes / 云原生：etcd
- 通用协调服务：两者皆可，etcd 更现代

### 7.8 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | ZooKeeper 是 ZAB 唯一工业实现，工程优化围绕"流水线 + Observer"展开 |
| **江峰** | 形式化 | ZooKeeper 的正确性依赖 ZAB 的"zxid 全序 + 多数派"两个不变量 |
| **唐伟志** | 工程 | ZooKeeper 的事务日志 + 快照恢复机制是工程关键 |
| **陈东明** | 案例 | 第 7 章 ZooKeeper 案例——大数据生态（Kafka/HBase/Hadoop）的元数据底座 |

**4 视角差异与互补**：
- **释慧利 vs 江峰**：释慧利问"ZAB 如何优化"（流水线+Observer），江峰问"正确性依赖什么"（zxid+多数派）
- **唐伟志 vs 陈东明**：唐伟志给"工程机制"（日志+快照恢复），陈东明给"工业应用"（大数据生态底座）
- **互补关系**：释慧利给"算法优化"、江峰给"数学基础"、唐伟志给"工程机制"、陈东明给"工业应用"

### 7.9 本节小结

ZooKeeper 是 ZAB 的唯一工业实现：
- **架构**：Leader + Follower + Observer
- **数据模型**：znode 树 + 事务日志 + 快照
- **应用场景**：配置管理 / 服务注册 / 分布式锁 / Leader 选举 / 大数据元数据
- **工程优化**：批量提交 / Observer 跨地域 / 日志分离 / Watch
- **局限性**：写入瓶颈 / 容量限制 / 跨地域延迟

## 8. 本篇总结与第 4 章对比篇演进动机

### 8.1 ZAB 的核心成就

ZAB 完成了 ZooKeeper 的关键设计：
1. **原子广播协议**：专为 ZooKeeper 的事务全序复制设计
2. **zxid 设计**：epoch + counter 的组合，全局唯一单调递增
3. **两阶段设计**：发现同步 + 广播
4. **工业可用**：ZooKeeper 在大数据生态广泛使用

### 8.2 ZAB 与 Paxos/Raft 的关系

**关系**：
- **数学等价**：都保证共识三性质
- **设计视角不同**：ZAB 从原子广播出发，Paxos 从共识出发，Raft 从可理解性出发
- **同源关系**：ZAB 的 epoch 与 Raft 的 term 都受 VR（Viewstamped Replication, 1988）的 view-number 概念启发，是同源概念，非直接因果
- **工业选择**：ZooKeeper 用 ZAB，新系统倾向 Raft

### 8.3 三大协议的历史脉络

```
1990: Paxos（Lamport）
  ↓ 启发
2008: ZAB（Yahoo!）→ ZooKeeper
  ↓ 启发
2014: Raft（Ongaro）→ etcd / TiKV / CockroachDB
```

**演进规律**：
- Paxos 解决"能不能"（理论可行性）
- ZAB 解决"专不专"（ZooKeeper 定制）
- Raft 解决"懂不懂"（可理解性）

### 8.5 关键认知

带着以下 3 个认知进入第 4 章-04 对比篇：

**认知 1：ZAB 是为 ZooKeeper 定制的原子广播协议**
- 不是通用共识算法，是原子广播协议
- 数学上与共识等价，但工程视角不同

**认知 2：zxid 是 ZAB 的核心设计**
- epoch + counter 的组合
- 全局唯一单调递增
- 影响了 Raft 的 term 设计

**认知 3：三大协议在共识三性质上等价，工程视角不同**
- 在崩溃故障 + 多数派 + 部分同步假设下，三者都满足共识的安全性（Agreement / Validity），活性（Termination）依赖部分同步
- **注**：问题模型不同——Paxos 是单值共识，ZAB 是原子广播，Raft 是日志复制 SMR，但数学上可通过 Chandra-Toueg 等价性互转
- Paxos：通用共识（算法最小性）
- ZAB：原子广播（ZooKeeper 定制）
- Raft：可理解性（易实现易维护）
- 选型取决于业务需求和团队熟悉度
