## 1. 为什么需要案例研究？

### 1.1 理论与实践的鸿沟

前 6 章建立了完整的理论框架，但理论与实践之间有鸿沟：

**理论说**："CAP 定理规定分区时 C 和 A 不可兼得"
**实践问**："那 GFS 选了什么？Spanner 选了什么？Dynamo 选了什么？"

**理论说**："共识算法保证 Agreement + Validity + Termination"
**实践问**："那 ZooKeeper 用 ZAB 而非 Paxos，工程上有什么差异？"

**理论说**："快照隔离需要全局时间戳"
**实践问**："那 Spanner 用 TrueTime，TiDB 用 TSO，CockroachDB 用 HLC——各自如何落地？"

本章用 6 个经典案例回答这些"实践问"。

### 1.2 6 个案例的选择理由

| 案例 | 年份 | 代表性 | 理论映射 |
|------|------|--------|---------|
| **GFS** | 2003 | 分布式文件系统奠基 | 分区 + 复制 + 一致性 |
| **BigTable** | 2006 | 列式存储奠基 | 分片 + SSTable + WAL |
| **Dynamo** | 2007 | 最终一致 KV 存储 | 向量时钟 + Quorum + AP |

| **Kafka** | 2011 | 高吞吐消息队列 | ISR + Offset + 分区 |

## 2. GFS（Google File System，2003）

### 2.1 设计动机

**背景**：Google 早期需要处理 PB 级网页数据，传统文件系统（NFS / SAN）无法满足：
- 容量不足（单机 TB 级上限）
- 性能不足（单机 IOPS 上限）
- 成本过高（专用存储设备）

**GFS 的目标**（Ghemawat et al., SOSP 2003）：
- 商品服务器集群
- PB 级容量
- 高吞吐（顺序读写为主）
- 容错（节点故障是常态）

### 2.2 架构

```
┌──────────────────────────────────────┐
│         Master（单点）                │
│  - 元数据（文件名/块位置/副本数）      │
│  - 操作日志（WAL）                    │
└──────────────┬───────────────────────┘
               │
   ┌───────────┼───────────┐
   ↓           ↓           ↓
┌──────┐  ┌──────┐    ┌──────┐
│Chunk  │  │Chunk  │    │Chunk  │
│Server1│  │Server2│    │Server3│
│(64MB) │  │(64MB) │    │(64MB) │
└──────┘  └──────┘    └──────┘
```

**核心组件**：
- **Master**：单点，管理元数据（文件名 → chunk 列表 → chunk 位置）
- **ChunkServer**：存储实际数据，每个 chunk 64MB（远大于传统文件系统的 4KB block）
- **Client**：直接与 ChunkServer 交互（数据不经过 Master）

### 2.3 理论映射

**第 1 章——分布式系统是物理学决定的必然**：
- GFS 用商品服务器集群替代专用存储——这是"单机扩展性已死"的工程验证

**第 3 章——分区与复制**：
- 文件按 64MB chunk 分区（大粒度分区，减少元数据量）
- 每个 chunk 3 副本（复制提供容错）

**第 3 章——一致性模型**：
- GFS 是**松散一致**（relaxed consistency）
- 写操作可能返回"一致"或"已定义"状态
- 适合"写一次读多次"的工作负载（网页爬虫）

**第 4 章——共识算法的缺失**：
- GFS Master 是单点（用 shadow master 做热备）
- 不用共识算法——这是 GFS 的局限（Master 故障需人工切换）

### 2.4 工程取舍

**取舍 1：大 chunk（64MB）vs 小 chunk（4KB）**
- 大 chunk 优势：减少元数据量、顺序读吞吐高
- 大 chunk 劣势：小文件浪费空间（一个 1KB 文件占 64MB）
- GFS 选择大 chunk——因为 Google 工作负载以大文件为主

**取舍 2：单 Master vs 多 Master**
- 单 Master 优势：元数据一致性强、实现简单
- 单 Master 劣势：单点故障、性能瓶颈
- GFS 选择单 Master + shadow——工程简化优先

**取舍 3：松散一致 vs 强一致**
- 松散一致优势：高性能、高可用
- 松散一致劣势：客户端可能读到旧数据
- GFS 选择松散一致——网页爬虫场景可接受

### 2.5 现代启示

**HDFS**（Hadoop Distributed File System）：
- 直接基于 GFS 论文实现
- NameNode = Master，DataNode = ChunkServer
- block 大小 128MB（GFS 是 64MB）

**对象存储**（Ceph / S3 / OSS）：
- 继承 GFS 的"分区+复制"思想
- 但用多副本或 EC（Erasure Coding）替代简单 3 副本

**Colossus**（GFS 的后继）：
- Google 内部用 Colossus 替代 GFS
- 解决单 Master 瓶颈

### 2.6 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | GFS 不用共识算法——Master 单点是工程妥协，反映了 2003 年共识算法工程成熟度不足 |
| **江峰** | 形式化 | GFS 的松散一致性在形式化上弱于线性一致，但适合"写一次读多次"工作负载（读者推导） |
| **唐伟志** | 工程 | GFS 的"大 chunk + 单 Master + 松散一致"三选择是"工作负载驱动设计"的典范 |
| **陈东明** | 案例 | GFS 影响了 HDFS / Ceph / Colossus——"分区+复制"是分布式文件系统的基本模式 |

### 2.7 本节小结

GFS 是分布式文件系统的奠基：
- **大 chunk（64MB）**：减少元数据，适合大文件
- **单 Master**：工程简化，但有单点风险
- **松散一致**：高吞吐，适合网页爬虫
- **3 副本**：容错
- **现代启示**：HDFS / Colossus / 对象存储

带着 GFS 的"分区+复制"思路进入第 3 节——BigTable。

## 3. BigTable（2006）

### 3.1 设计动机

**背景**：Google 需要存储结构化数据（网页索引 / Earth / 历史数据），传统数据库无法满足：
- 容量不足（PB 级）
- 写吞吐不足
- 成本过高

**BigTable 的目标**（Chang et al., OSDI 2006）：
- PB 级结构化数据
- 高写吞吐
- 列式存储（稀疏数据）
- 建立在 GFS 之上

### 3.2 架构

```
┌──────────────────────────────────────┐
│         Master                        │
│  - Tablet 分配/迁移                   │
│  - 元数据 Tablet                      │
└──────────────┬───────────────────────┘
               │
   ┌───────────┼───────────┐
   ↓           ↓           ↓
┌──────┐  ┌──────┐    ┌──────┐
│Tablet │  │Tablet │    │Tablet │
│Server1│  │Server2│    │Server3│
└──────┘  └──────┘    └──────┘
   ↓           ↓           ↓
┌──────────────────────────────────────┐
│              GFS（底层存储）           │
└──────────────────────────────────────┘
```

**核心组件**：
- **Master**：管理 Tablet 分配/迁移
- **Tablet Server**：服务 Tablet（数据分片）
- **GFS**：底层存储（SSTable 文件存在 GFS）

**数据模型**：
- `(row, column, timestamp) → value`
- 行键有序存储
- 列族（column family）预定义，列限定符动态
- 时间戳支持多版本

### 3.3 理论映射

**第 3 章——分区策略**：
- BigTable 按行键范围分区（Tablet）
- Tablet 是数据分片单元（类似 TiKV 的 Region）
- Tablet 自动分裂/合并

**第 3 章——一致性模型**：
- 单行事务（行级 ACID）
- 跨行无事务（最终一致）

**第 6 章——时间戳**：
- 每个值带时间戳（多版本）
- 时间戳用于版本控制和读快照

**第 5 章——WAL**：
- 写操作先写 WAL（commit log，存在 GFS）
- Tablet Server 故障后，重新分配 Tablet + replay WAL

### 3.4 工程取舍

**取舍 1：列式存储 vs 行式存储**
- 列式优势：稀疏数据高效、压缩率高
- 列式劣势：跨列查询慢
- BigTable 选择列式——Google 数据稀疏

**取舍 2：SSTable + MemTable vs B+Tree**
- SSTable 优势：写吞吐高（顺序写）、压缩率高
- SSTable 劣势：读需合并多个 SSTable
- BigTable 选择 SSTable + MemTable——写密集场景

**取舍 3：单行事务 vs 跨行事务**
- 单行优势：实现简单、性能高
- 单行劣势：跨行操作无 ACID
- BigTable 选择单行事务——2006 年跨行事务工程不成熟

### 3.5 现代启示

**HBase**：直接基于 BigTable 论文实现
- HRegion = Tablet
- HFile = SSTable
- HLog = WAL

**Cassandra**：融合 BigTable + Dynamo
- 列式数据模型（来自 BigTable）
- 分布式架构（来自 Dynamo）

**TiDB / Spanner**：融合 BigTable + 共识
- Range 分区（来自 BigTable）
- 多 Raft/Paxos（补充 BigTable 的单点缺陷）

### 3.6 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | BigTable 的单行事务是"在弱一致基础上提供局部强一致"的工程妥协 |
| **江峰** | 形式化 | BigTable 的时间戳多版本是"快照隔离"的简化版（读者推导） |
| **唐伟志** | 工程 | BigTable 的 SSTable + MemTable 是 LSM-Tree 的工业实现，影响了 LevelDB / RocksDB |
| **陈东明** | 案例 | BigTable 影响了 HBase / Cassandra / TiDB——列式存储 + Range 分区是分布式存储的基本模式 |

### 3.7 本节小结

BigTable 是列式存储的奠基：
- **列式数据模型**：稀疏数据高效
- **SSTable + MemTable**：LSM-Tree，高写吞吐
- **Range 分区**：Tablet 自动分裂/合并
- **单行事务**：行级 ACID
- **现代启示**：HBase / Cassandra / TiDB

带着 BigTable 的"列式+分区"思路进入第 4 节——Dynamo。

## 4. Dynamo（2007）

### 4.1 设计动机

**背景**：Amazon 电商需要高可用 KV 存储（购物车 / 会话），传统数据库无法满足：
- 黑色星期五流量峰值
- 节点故障频繁
- 跨地域部署

**Dynamo 的目标**（DeCandia et al., SOSP 2007）：
- **高可用优先**（AP 系统）
- 最终一致
- 去中心化（无 Master）
- 商品服务器

### 4.2 架构

```
        Client
          ↓
   ┌──────────────┐
   │  Coordinator │（任意节点担任）
   └──────┬───────┘
    ┌─────┼─────┐
    ↓     ↓     ↓
  ┌───┐ ┌───┐ ┌───┐
  │N1 │ │N2 │ │N3 │  ← 一致性哈希 + 虚拟节点
  └───┘ └───┘ └───┘
```

**核心组件**：
- **一致性哈希**：节点动态加入/退出，最小迁移
- **虚拟节点**：每个物理节点对应多个虚拟节点（负载均衡）
- **Quorum**：NWR 可调一致性
- **向量时钟**：检测写冲突
- **Merkle Tree**：反熵（anti-entropy）快速同步

### 4.3 理论映射

**第 1 章——CAP 定理**：
- Dynamo 是 **AP 系统**（分区时优先可用性）
- 与 Spanner（CP）形成对比

**第 3 章——分区策略**：
- 一致性哈希 + 虚拟节点（第 3 章 5.4 节）
- 节点加入/退出最小迁移

**第 3 章——复制模式**：
- 无主复制（Leaderless，第 3 章 3.4 节）
- Quorum（NWR，第 3 章 4 节）

**第 3 章——一致性模型**：
- 最终一致（第 3 章 2.8 节）
- 冲突解决：LWW 或向量时钟 + siblings

**第 6 章——向量时钟**：
- Dynamo 用向量时钟检测写冲突（第 6 章 4 节）
- 客户端解决冲突（siblings）

### 4.4 工程取舍

**取舍 1：AP vs CP**
- AP 优势：高可用、跨地域
- AP 劣势：冲突解决复杂
- Dynamo 选择 AP——电商场景可用性优先

**取舍 2：无主 vs 主从**
- 无主优势：无单点、写入并行
- 无主劣势：冲突解决复杂
- Dynamo 选择无主——去中心化

**取舍 3：向量时钟 vs LWW**
- 向量时钟优势：精确检测冲突
- 向量时钟劣势：空间开销 + 客户端复杂
- Dynamo 选择向量时钟——电商冲突可接受

### 4.5 现代启示

**DynamoDB**（AWS 服务）：
- Dynamo 论文的商业化
- 简化为 LWW（牺牲向量时钟的冲突检测）
- 提供 strongly consistent read 选项

**Cassandra**：
- 融合 Dynamo（分布式架构）+ BigTable（列式数据模型）
- 可调一致性（ONE / QUORUM / ALL）
- 从未用向量时钟（选择 LWW）

**Riak**：
- 忠实继承 Dynamo 的向量时钟 + siblings
- 客户端解决冲突

**Voldemort**（LinkedIn）：
- 简化版 Dynamo
- 读修复 + 反熵

### 4.6 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | Dynamo 的 Quorum + 向量时钟是"在 AP 约束下做最大化一致性"的工程方案 |
| **江峰** | 形式化 | Dynamo 的最终一致性在形式化上弱于线性一致，但可用性更强（读者推导） |
| **唐伟志** | 工程 | Dynamo 的"一致性哈希 + 虚拟节点 + Quorum + 向量时钟"是 AP 系统的工程模板 |
| **陈东明** | 案例 | Dynamo 影响了 DynamoDB / Cassandra / Riak / Voldemort——AP 系统的基本模式 |

### 4.7 本节小结

Dynamo 是 AP 系统的奠基：
- **AP 优先**：高可用，最终一致
- **一致性哈希 + 虚拟节点**：去中心化分区
- **Quorum（NWR）**：可调一致性
- **向量时钟**：冲突检测
- **Merkle Tree**：反熵同步
- **现代启示**：DynamoDB / Cassandra / Riak / Voldemort

带着 Dynamo 的"AP"思路进入第 5 节——Spanner（CP 的极致）。

## 5. Spanner（2012）

### 5.1 设计动机

**背景**：Google 需要全球分布式数据库（广告系统 / AdWords），传统数据库无法满足：
- 跨大陆强一致事务
- PB 级数据
- 高可用

**Spanner 的目标**（Corbett et al., OSDI 2012）：
- 全球强一致（外部一致性）
- 跨地域多副本
- SQL 接口
- 建立在 BigTable / Colossus 之上

### 5.2 架构

```
┌──────────────────────────────────────────────┐
│              Spanner 全球部署                  │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐      │
│  │ Zone 1  │  │ Zone 2  │  │ Zone 3  │      │
│  │ (US)    │  │ (EU)    │  │ (Asia)  │      │
│  │ Tablet1 │  │ Tablet1 │  │ Tablet1 │      │
│  │ (Paxos) │  │ (Paxos) │  │ (Paxos) │      │
│  │ Tablet2 │  │ Tablet2 │  │ Tablet2 │      │
│  │ (Paxos) │  │ (Paxos) │  │ (Paxos) │      │
│  └─────────┘  └─────────┘  └─────────┘      │
│       ↕            ↕            ↕            │
│  ┌──────────────────────────────────────┐   │
│  │  TrueTime API（GPS + 原子钟）         │   │
│  └──────────────────────────────────────┘   │
└──────────────────────────────────────────────┘
```

**核心组件**：
- **Tablet**：数据分片单元，每个 Tablet 是一个 Paxos 组
- **Paxos 组**：跨地域多副本（3-5 副本分布在不同 Zone）
- **TrueTime**：GPS + 原子钟提供有界不确定性时间
- **Directory**：数据迁移单位（跨 Tablet）

### 5.3 理论映射

**第 1 章——CAP 定理**：
- Spanner 是 **CP 系统**（强一致，分区时拒绝服务）
- 与 Dynamo（AP）形成对比

**第 4 章——共识算法**：
- 每个 Tablet 是一个 Paxos 组（第 4 章-01）
- Paxos 保证副本一致
- Tablet Leader 处理读写

**第 5 章——分布式事务**：
- 跨 Tablet 事务用 2PC（第 5 章 3 节）
- 2PC 协调者基于 Paxos（高可用）
- TrueTime 时间戳实现快照隔离（第 5 章 7 节 Percolator 思路）

**第 6 章——TrueTime**：
- TrueTime 提供有界不确定性时间（ε < 7ms，第 6 章 6 节）
- Commit Wait 等 ε 过去，保证外部一致性
- 这是 Spanner 的核心创新

### 5.4 工程取舍

**取舍 1：CP vs AP**
- CP 优势：强一致、外部一致性
- CP 劣势：分区时不可用、跨地域延迟高
- Spanner 选择 CP——金融/广告场景一致性优先

**取舍 2：Paxos vs Raft**
- Paxos 优势：2008 年 Raft 未出生
- Paxos 劣势：实现复杂
- Spanner 选择 Paxos——历史原因

**取舍 3：TrueTime vs TSO vs HLC**
- TrueTime 优势：外部一致性
- TrueTime 劣势：成本高（GPS + 原子钟）
- Spanner 选择 TrueTime——Google 有能力负担

**取舍 4：2PC vs 其他**
- 2PC 优势：成熟、强一致
- 2PC 劣势：协调者单点（Spanner 用 Paxos 解决）
- Spanner 选择 2PC + Paxos 协调者

### 5.5 现代启示

**CockroachDB**：
- 受 Spanner 启发
- 用 HLC 替代 TrueTime（避免专用硬件）
- 用 Raft 替代 Paxos（易实现）

**TiDB / TiKV**：
- 受 Spanner 启发
- 用 TSO 替代 TrueTime（中心化但简单）
- 用 Raft 替代 Paxos
- 用 Percolator 事务模型

**YugabyteDB**：
- 受 Spanner 启发
- 用 HLC + Raft

### 5.6 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | Spanner 是 Paxos + 2PC + TrueTime 的组合——三个理论的工业集成 |
| **江峰** | 形式化 | Spanner 的外部一致性在形式化上强于线性一致——加入了真实时间序（读者推导） |
| **唐伟志** | 工程 | Spanner 的"TrueTime + Paxos + 2PC"是"用昂贵硬件换取全球强一致"的工程极致 |
| **陈东明** | 案例 | Spanner 影响了 CockroachDB / TiDB / YugabyteDB——全球分布式数据库的标杆 |

### 5.7 本节小结

Spanner 是全球强一致数据库的标杆：
- **CP 系统**：强一致，分区时拒绝服务
- **Paxos 组**：每个 Tablet 一个 Paxos 组，跨地域多副本
- **TrueTime**：GPS + 原子钟，有界不确定性 ε < 7ms
- **2PC + Paxos 协调者**：跨 Tablet 事务
- **外部一致性**：线性一致 + 真实时间序
- **现代启示**：CockroachDB / TiDB / YugabyteDB

带着 Spanner 的"CP 极致"思路进入第 6 节——Kafka。

## 6. Kafka（2011）

### 6.1 设计动机

**背景**：LinkedIn 需要高吞吐消息队列（活动流 / 日志聚合），传统消息队列（ActiveMQ / RabbitMQ）无法满足：
- 吞吐不足（万级 TPS）
- 持久化开销大
- 扩展性差

**Kafka 的目标**（Kreps et al., 2011）：
- 百万级 TPS
- 持久化（O(1) 顺序写磁盘）
- 分布式扩展
- 分区内有序

### 6.2 架构

```
┌──────────────────────────────────────┐
│         Producer                      │
└──────────────┬───────────────────────┘
               ↓
┌──────────────────────────────────────┐
│         Broker Cluster                │
│  ┌─────────┐  ┌─────────┐  ┌────────┐│
│  │Partition│  │Partition│  │Partition││
│  │  0      │  │  1      │  │  2     ││
│  │(Leader) │  │(Leader) │  │(Leader)││
│  │  ↓      │  │  ↓      │  │  ↓     ││
│  │ Follower│  │ Follower│  │Follower││
│  └─────────┘  └─────────┘  └────────┘│
└──────────────┬───────────────────────┘
               ↓
┌──────────────────────────────────────┐
│         Consumer Group                │
└──────────────────────────────────────┘
```

**核心组件**：
- **Producer**：发送消息到指定 Partition
- **Broker**：存储 Partition，每个 Partition 有 Leader + Follower
- **Consumer Group**：消费消息，分区内有序
- **ZooKeeper**（早期）/ **KRaft**（2.8+）：管理集群元数据

### 6.3 理论映射

**第 1 章——分布式系统是必然**：
- Kafka 用商品服务器集群 + 顺序写磁盘——单机扩展性已死

**第 3 章——分区策略**：
- 按 Partition 分区（第 3 章 5 节）
- Partition 是并行单元

**第 3 章——复制模式**：
- 主从复制（Leader + Follower）
- ISR（In-Sync Replicas）机制——Follower 落后则移出 ISR

**第 6 章——Offset**：
- Partition 内 Offset 单调递增（第 6 章 3 节）
- 提供分区内全序（但非严格 Lamport 时钟——第 6 章已修正）

**第 4 章——共识算法的近似**：
- Kafka ISR 不是严格共识算法
- Leader 选举基于 ZooKeeper/KRaft
- 牺牲强一致换取高吞吐

### 6.4 工程取舍

**取舍 1：顺序写磁盘 vs 随机写**
- 顺序写优势：磁盘吞吐高（HDD 100MB/s vs 随机 1MB/s）
- 顺序写劣势：不适合随机读
- Kafka 选择顺序写——消息流是顺序的

**取舍 2：ISR vs Paxos/Raft**
- ISR 优势：高吞吐（不要求多数派 ACK）
- ISR 劣势：可能丢数据（Leader 切换时 ISR 内数据未同步）
- Kafka 选择 ISR——吞吐优先

**取舍 3：分区有序 vs 全局有序**
- 分区有序优势：并行度高
- 分区有序劣势：跨分区无序
- Kafka 选择分区有序——大多数场景足够

### 6.5 现代启示

**Pulsar**：
- 分离存储层（BookKeeper）+ 计算层（Broker）
- 支持多租户
- 与 Kafka 竞争

**RocketMQ**：
- 阿里开源
- 融合 Kafka + 消息事务（half message + 回查）
- 适合电商场景

**KRaft**（Kafka 2.8+）：
- Kafka 去 ZooKeeper 化
- 用 Raft 变种管理元数据
- Kafka 4.0 完全移除 ZK 依赖

### 6.6 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | Kafka 的 ISR 不是严格共识——牺牲强一致换取高吞吐的工程选择 |
| **江峰** | 形式化 | Kafka 的 Offset 提供分区内全序，但不满足 Lamport 因果保持性（读者推导） |
| **唐伟志** | 工程 | Kafka 的"顺序写 + 零拷贝 + ISR"是高吞吐消息队列的工程模板 |
| **陈东明** | 案例 | Kafka 影响了 Pulsar / RocketMQ / Pulsar——消息队列的基本模式 |

### 6.7 本节小结

Kafka 是高吞吐消息队列的标杆：
- **顺序写磁盘**：O(1) 持久化，高吞吐
- **Partition 分区**：并行单元
- **ISR 复制**：吞吐优先，可能丢数据
- **Offset**：分区内全序
- **KRaft**（2.8+）：去 ZK 化
- **现代启示**：Pulsar / RocketMQ / KRaft

带着 Kafka 的"高吞吐"思路进入第 7 节——ZooKeeper。

## 7. ZooKeeper（2008）

### 7.1 设计动机

**背景**：Yahoo! / Google 需要分布式协调服务（配置管理 / Leader 选举 / 分布式锁），每个系统自己实现协调逻辑太复杂。

**ZooKeeper 的目标**（Hunt et al., ATC 2010）：
- 协调服务（非通用存储）
- 强一致写（顺序一致）
- 高吞吐低延迟
- Watch 机制（数据变更通知）

### 7.2 架构

```
┌──────────────────────────────────────┐
│         Client                        │
└──────────────┬───────────────────────┘
               ↓
┌──────────────────────────────────────┐
│         Leader                        │
│  - 处理写请求                         │
│  - ZAB 广播                           │
└──────────────┬───────────────────────┘
    ┌──────────┼──────────┐
    ↓          ↓          ↓
┌──────┐  ┌──────┐    ┌──────┐
│Follower│  │Follower│   │Observer│
│(投票) │  │(投票) │   │(不投票)│
└──────┘  └──────┘    └──────┘
```

**核心组件**：
- **Leader**：处理写请求，ZAB 广播
- **Follower**：参与 Quorum，处理读请求
- **Observer**：不参与 Quorum，跨地域读优化
- **znode 树**：层次数据模型（持久 / 临时 / 顺序节点）

### 7.3 理论映射

**第 1 章——CAP 定理**：
- ZooKeeper 是 **CP 系统**（强一致写，分区时少数派不可写）

**第 4 章——共识算法**：
- ZAB 协议（第 4 章-03）
- zxid = epoch + counter（第 4 章-03 第 3 节）
- 两阶段：发现同步 + 广播

**第 3 章——一致性模型**：
- 写：顺序一致（所有客户端看到的写顺序一致）
- 读：默认顺序一致（Follower 本地读，可能落后）
- 不保证线性一致（跨客户端可能回退）

**第 6 章——时间戳**：
- zxid 是 epoch + counter 的组合（第 6 章 3 节 Lamport 时钟的变种）

### 7.4 工程取舍

**取舍 1：协调服务 vs 通用存储**
- 协调服务优势：专精、高吞吐
- 协调服务劣势：不适合大数据存储
- ZooKeeper 选择协调服务——聚焦核心能力

**取舍 2：ZAB vs Paxos**
- ZAB 优势：为 ZooKeeper 定制（原子广播）
- ZAB 劣势：非通用共识
- ZooKeeper 选择 ZAB——2008 年 Raft 未出生

**取舍 3：顺序一致读 vs 线性一致读**
- 顺序一致读优势：低延迟（Follower 本地读）
- 顺序一致读劣势：可能读到旧数据
- ZooKeeper 选择顺序一致读——协调服务场景可接受

### 7.5 现代启示

**etcd**：
- 受 ZooKeeper 启发，但用 Raft
- 线性一致读（ReadIndex）
- K8s 元数据底座

**Consul**：
- 受 ZooKeeper 启发，用 Raft
- 强调服务发现 + 健康检查

**Nacos**：
- 阿里开源，混合模型（Raft + Distro）
- Spring Cloud 集成

**Kafka 去 ZK 化**：
- Kafka 2.8+ 用 KRaft 替代 ZooKeeper
- Kafka 4.0 完全移除 ZK——标志 ZK 在消息队列领域被替代

### 7.6 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | ZooKeeper 的 ZAB 是为原子广播定制的共识——与通用 Paxos/Raft 形成对比 |
| **江峰** | 形式化 | ZooKeeper 的顺序一致读在形式化上弱于线性一致，但协调服务场景可接受（读者推导） |
| **唐伟志** | 工程 | ZooKeeper 的"ZAB + Watch + 临时节点"是协调服务的工程模板 |
| **陈东明** | 案例 | ZooKeeper 影响了 etcd / Consul / Nacos——协调服务的基本模式 |

### 7.7 本节小结

ZooKeeper 是协调服务的标杆：
- **CP 系统**：强一致写
- **ZAB 协议**：原子广播（zxid = epoch + counter）
- **顺序一致读**：低延迟，可能落后
- **Watch + 临时节点**：协调服务核心机制
- **Observer**：跨地域读优化
- **现代启示**：etcd / Consul / Nacos

带着 6 个案例的全景进入第 8 节——理论映射总结。

## 8. 理论映射总结

### 8.1 6 个案例的理论映射矩阵

| 案例 | CAP 选择 | 共识算法 | 一致性模型 | 时间方案 | 分区策略 |
|------|---------|---------|-----------|---------|---------|
| **GFS** | CP（松散） | 无（单 Master） | 松散一致 | 物理时钟 | 大 chunk（64MB） |
| **BigTable** | CP（单行） | 无（单 Master） | 单行 ACID | 物理时钟 | Range 分区（Tablet） |
| **Dynamo** | AP | 无（无主） | 最终一致 | 向量时钟 | 一致性哈希 |
| **Spanner** | CP | Paxos | 外部一致性 | TrueTime | Range 分区（Tablet） |
| **Kafka** | CP（分区级） | ISR（非严格共识） | 分区内全序 | Offset | Partition 分区 |
| **ZooKeeper** | CP | ZAB | 顺序一致 | zxid（epoch+counter） | 单分片 |

### 8.2 理论在案例中的演进

**第 1 章——CAP 定理的工业落地**：
- GFS / BigTable / Spanner / ZooKeeper 选 CP（强一致优先）
- Dynamo 选 AP（可用性优先）
- Kafka 选分区级 CP（分区级强一致，跨分区无序）

**第 3 章——分区与复制的工业落地**：
- GFS：大 chunk 分区 + 3 副本
- BigTable：Range 分区 + SSTable
- Dynamo：一致性哈希 + 无主 Quorum
- Spanner：Range 分区 + Paxos 多副本
- Kafka：Partition 分区 + ISR
- ZooKeeper：单分片 + ZAB

**第 4 章——共识算法的工业落地**：
- Spanner：Paxos（每个 Tablet 一个 Paxos 组）
- ZooKeeper：ZAB（原子广播）
- Kafka：ISR（非严格共识）
- GFS / BigTable / Dynamo：不用共识（单 Master 或无主）

**第 5 章——分布式事务的工业落地**：
- Spanner：2PC + TrueTime（外部一致性）
- BigTable：单行事务
- Dynamo：无事务（最终一致）
- 其他：无跨节点事务

**第 6 章——时间方案的工业落地**：
- Spanner：TrueTime（GPS + 原子钟）
- Dynamo：向量时钟
- ZooKeeper：zxid（epoch + counter）
- Kafka：Offset
- GFS / BigTable：物理时钟

### 8.3 理论与工程的对应关系

```
理论层                    工程层
─────────────────────────────────────
CAP 定理          →      GFS/BigTable/Spanner/ZK 选 CP，Dynamo 选 AP
分区策略          →      GFS 大 chunk / BigTable Range / Dynamo 一致性哈希 / Kafka Partition
复制模式          →      GFS 3 副本 / Dynamo 无主 Quorum / Spanner Paxos / Kafka ISR
共识算法          →      Spanner Paxos / ZK ZAB / etcd Raft / Kafka ISR
分布式事务        →      Spanner 2PC+TrueTime / BigTable 单行 / Seata AT
时间方案          →      Spanner TrueTime / Dynamo 向量时钟 / ZK zxid / Kafka Offset
```

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | 6 个案例展示了共识算法的工业选择——Paxos/ZAB/ISR/无共识各有适用场景 |
| **江峰** | 形式化 | 6 个案例验证了 CAP/FLP/一致性模型等理论在工程中的边界（读者推导） |

### 8.5 本节小结

6 个案例的理论映射：
- **CAP**：CP（GFS/BigTable/Spanner/ZK）vs AP（Dynamo）vs 分区级 CP（Kafka）
- **共识**：Paxos（Spanner）/ ZAB（ZK）/ ISR（Kafka）/ 无（GFS/BigTable/Dynamo）
- **时间**：TrueTime（Spanner）/ 向量时钟（Dynamo）/ zxid（ZK）/ Offset（Kafka）
- **分区**：大 chunk / Range / 一致性哈希 / Partition / 单分片

## 9. 对现代系统的启示

### 9.1 从经典到现代的演进

| 经典系统 | 现代继承者 | 演进方向 |
|---------|----------|---------|
| GFS | HDFS / Colossus / Ceph | 多 Master + EC |
| BigTable | HBase / Cassandra / TiDB | 多副本 + 跨行事务 |
| Dynamo | DynamoDB / Cassandra / Riak | 简化冲突解决（LWW） |
| Spanner | CockroachDB / TiDB / YugabyteDB | 去专用硬件（HLC/TSO） |
| Kafka | Pulsar / RocketMQ / KRaft | 去 ZK + 事务消息 |
| ZooKeeper | etcd / Consul / Nacos | Raft + 线性一致读 |

### 9.2 演进的规律

**规律 1：共识算法的普及**
- 经典系统（GFS/BigTable/Dynamo）不用共识
- 现代系统（TiDB/CockroachDB/etcd）普遍用 Raft
- 演进方向：单点 → 共识

**规律 2：时间方案的分化**
- Spanner：TrueTime（最强但最贵）
- CockroachDB：HLC（去专用硬件）
- TiDB：TSO（中心化但简单）
- 演进方向：专用硬件 → 通用方案

**规律 3：事务能力的增强**
- GFS/BigTable：无事务或单行事务
- Dynamo：最终一致
- Spanner/TiDB：跨行事务 + 快照隔离
- 演进方向：弱事务 → 强事务

**规律 4：协调服务的去中心化**
- ZooKeeper：独立协调服务
- etcd/Consul：嵌入 K8s/微服务
- KRaft：Kafka 去协调服务
- 演进方向：独立协调 → 嵌入式 / 去协调

### 9.3 现代系统的选型启示

**启示 1：CAP 选择由业务决定**
- 金融/广告：CP（Spanner/TiDB）
- 电商/社交：AP（Dynamo/Cassandra）
- 消息/日志：分区级 CP（Kafka）

**启示 2：共识算法选 Raft（新系统）**
- 易实现易维护
- 多开源参考

**启示 3：时间方案按预算选**
- Google 级预算：TrueTime
- 普通预算：HLC（CockroachDB）/ TSO（TiDB）

**启示 4：事务按需选**
- 跨行强一致：Spanner/TiDB
- 单行：BigTable/HBase
- 最终一致：Dynamo/Cassandra
- 框架封装：Seata

### 9.4 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | 演进规律是"单点→共识 / 弱事务→强事务 / 专用→通用" |
| **江峰** | 形式化 | 演进的本质是"模型约束逐步放宽 / 工程能力逐步增强"（读者推导） |
| **唐伟志** | 工程 | 现代系统选型应参考经典案例——"在约束下做最大化"的智慧不变 |
| **陈东明** | 案例 | 经典案例是现代系统的源头——HDFS/HBase/Cassandra/TiDB/etcd 都源自 6 个经典 |

### 9.5 本节小结

现代系统的启示：
- **共识普及**：GFS/BigTable/Dynamo 无共识 → 现代系统用 Raft
- **时间分化**：TrueTime / HLC / TSO 按预算选
- **事务增强**：无事务 → 跨行强事务
- **协调去中心化**：独立 ZK → 嵌入式 / 去协调
- **选型启示**：CAP 由业务定 / Raft 新系统 / 时间按预算 / 事务按需

### 10.1 第 7 章总结

第 7 章用 6 个经典案例验证前 6 章理论：
- **GFS**（2003）：大 chunk + 单 Master + 松散一致 → HDFS
- **BigTable**（2006）：列式 + SSTable + 单行事务 → HBase
- **Dynamo**（2007）：AP + 一致性哈希 + 向量时钟 → Cassandra
- **Spanner**（2012）：CP + Paxos + TrueTime + 2PC → CockroachDB/TiDB
- **Kafka**（2011）：顺序写 + ISR + Offset → Pulsar/RocketMQ
- **ZooKeeper**（2008）：ZAB + Watch + 临时节点 → etcd/Consul

**核心论断**：6 个案例展示了"在约束下做最大化"的工程智慧——每个选择都是 CAP/共识/时间/事务理论的权衡落地。

### 10.3 全书的核心认知

**认知 1：分布式系统是当前硬件边界下的主流扩展路径**
- 单机扩展受 4 大物理极限
- 分布式是新系统主流

**认知 2：4 大难题是问题本身的属性**
- 部分失败 + 网络不可靠 + 并发不确定 + 异构故障
- 不是工程缺陷，是物理事实

**认知 3：共识算法是强一致系统的底座**
- Paxos / Raft / ZAB 数学等价
- 跨节点强一致事务依赖共识

**认知 4：时间是分布式系统的暗线**
- 物理时钟不可靠 → 逻辑时钟 → HLC → TrueTime
- 时间戳贯穿共识与事务

**认知 5：工程是"在约束下做最大化"**
- 6 个案例展示了 CAP/共识/时间/事务的权衡落地
- 没有银弹，只有权衡
