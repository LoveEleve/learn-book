## 2. 三大协议对比矩阵

**它们到底差在哪？** 三大协议都满足共识三性质（Agreement / Validity / Termination），但工程实现差异巨大——为什么？因为**等价 ≠ 相同**：算法结构、设计哲学、工程优化方向都不同。本节从 5 个维度系统对比。

### 2.1 设计哲学对比

| 维度 | Paxos（1990） | ZAB（2008） | Raft（2014） |
|------|---------------|-------------|--------------|
| 出发点 | 通用共识 | 原子广播 | 可理解性 |
| 设计目标 | 算法最小性 | ZooKeeper 定制 | 易理解易实现 |
| Leader 角色 | 可选优化 | 必需设计 | 必需设计 |
| 论文完整度 | 不完整（Multi-Paxos 工程自己设计） | 完整（含 ZooKeeper 实现） | 完整（含选举/复制/压缩/成员变更） |
| 影响来源 | Lamport 独立提出 | 与 Paxos 同族但独立设计，epoch 概念受 VR 启发 | 受 Paxos + VR 启发（论文明确引用 VR） |

**关键洞察**：
- **Paxos** 追求"算法最小性"——用最少的机制保证正确性
- **ZAB** 追求"定制化"——为 ZooKeeper 的原子广播需求量身设计
- **Raft** 追求"可理解性"——用三子问题分解降低复杂度

### 2.2 核心机制对比

| 机制 | Paxos | ZAB | Raft |
|------|-------|-----|------|
| 提案编号 | Proposal ID (round+node) | zxid (epoch+counter) | term |
| 日志位置 | 隐式（Multi-Paxos 自己定义） | zxid 的 counter 部分 | 独立的 index 字段 |
| Leader 选举 | 未明确（工程实现） | lastZxid + myid | lastLogTerm + lastLogIndex + 随机超时 |
| 日志一致性 | Prepare 阶段保证 | 同步阶段 DIFF/TRUNC/SNAP | prevLogIndex + prevLogTerm 检查 |
| 提交规则 | 多数派 ACK（论文未明确约束旧 term） | 多数派 ACK + 独立 COMMIT 消息 | 多数派 ACK + 只提交当前 term |
| 同步方式 | 被动（Proposer 重试） | 主动（Leader 推送 DIFF/TRUNC） | 被动（Leader 回退 nextIndex） |
| 成员变更 | 未明确 | 未明确（ZooKeeper 重新配置） | 联合共识 + 单节点变更 |
| 日志压缩 | 未明确 | Snapshot | Snapshot |

### 2.3 工程差异对比

| 差异点 | Paxos | ZAB | Raft |
|--------|-------|-----|------|
| 实现难度 | 高（Multi-Paxos 细节自己设计） | 中（论文完整但定制 ZooKeeper） | 低（论文完整 + 大量开源参考） |
| 流水线优化 | 工程自己实现 | 内置（连续 PROPOSAL） | 内置（连续 AppendEntries） |
| 读一致性 | 工程自己实现 | 默认顺序一致读 | 默认线性一致读（ReadIndex） |
| Observer 节点 | 无 | 原生支持 | 无原生（需工程扩展） |
| 跨地域优化 | 工程自己实现 | Observer 跨地域读 | 工程自己实现 |

### 2.4 数学等价性

**三大协议在共识三性质上等价**：
- 在崩溃故障 + 多数派 + 部分同步假设下，三者都满足 Agreement / Validity / Termination
- **注**：问题模型不同——Paxos 是单值共识，ZAB 是原子广播，Raft 是日志复制 SMR
- 数学上可通过 Chandra-Toueg 1996 等价性互转（共识 ≡ 原子广播）

**等价 ≠ 相同**：
- 算法结构不同（两阶段 vs 三子问题 vs 两阶段）
- 工程优化方向不同（最小性 vs 定制化 vs 可理解性）
- 工业采用不同（历史系统 vs ZooKeeper vs 新系统）

### 2.5 VR 的传承关系

补充第 4 章-03 第 6.8 节提到的 VR（Viewstamped Replication，Oki-Liskov 1988）：

```
VR（1988）→ ZAB（2008）/ Raft（2014）
   ↑
   └─ view-number 概念 → ZAB epoch / Raft term

Paxos（1990，独立提出）
   ↑
   └─ 与 VR 同期，但更通用（单值共识）
```

**VR 的贡献**：
- 首次提出 view-number + 主从复制 + 视图变更
- ZAB 和 Raft 都受 VR 启发（Raft 论文明确引用）
- Paxos 与 VR 独立提出，但 Paxos 更通用

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | 三大协议在第 4-6 章分别推导——Paxos 是基础，ZAB 是原子广播，Raft 是可理解性重设计 |
| **江峰** | 形式化 | 三者都满足共识三性质（江峰第 8 章通用共识形式化，读者套用至三大协议） |
| **唐伟志** | 工程 | 工程实现差异在 Leader 选举、日志同步、读一致性——第 7 章案例研究对比 |
| **陈东明** | 案例 | Chubby 用 Paxos，ZooKeeper 用 ZAB，etcd 用 Raft——每个选择背后都有具体业务场景 |

**4 视角差异与互补**：
- **释慧利 vs 江峰**：释慧利问"算法如何不同"（机制差异），江峰问"数学是否等价"（共识三性质）
- **唐伟志 vs 陈东明**：唐伟志给"工程差异"（选举/同步/读），陈东明给"工业选择"（业务场景驱动）
- **互补关系**：释慧利给"算法视角"、江峰给"数学等价性"、唐伟志给"工程差异"、陈东明给"工业案例"

### 2.7 本节小结

三大协议对比：
- **设计哲学**：Paxos 最小性 / ZAB 定制化 / Raft 可理解性
- **核心机制**：Proposal ID vs zxid vs term / 被动同步 vs 主动同步 vs 被动回退
- **工程差异**：实现难度 / 流水线 / 读一致性 / Observer
- **数学等价**：共识三性质等价，问题模型不同
- **共同源头**：ZAB 和 Raft 都受 VR 启发

## 3. 工业实现对比

### 3.1 9 个主流系统全览

| 系统 | 协议 | 语言 | 一致性 | 分区 | 典型场景 |
|------|------|------|--------|------|---------|
| **etcd** | Raft | Go | 线性一致 | 单分片 | K8s 元数据 |
| **ZooKeeper** | ZAB | Java | 顺序一致 | 单分片 | 大数据生态元数据 |
| **TiKV** | Raft（多组） | Rust | 线性一致 | Range（96MB/256MB） | TiDB 存储 |
| **CockroachDB** | Raft（多组） | Go | 线性一致 | Range（64MB） | 全球 SQL |
| **Spanner** | Paxos（多组） | C++ | 线性一致 | Tablet | Google 全球数据库 |
| **MySQL MGR** | Paxos 变种（XCom） | C | 线性一致 | 手动分库 | MySQL 高可用 |
| **Nacos** | Raft + Distro | Java | CP/AP 切换 | 单分片 | 微服务注册+配置 |
| **Consul** | Raft | Go | 线性一致 | 单分片 | 服务发现+健康检查 |
| **SOFA-JRaft** | Raft | Java | 线性一致 | 用户定义 | 通用 Raft 框架 |

### 3.2 详细对比

#### etcd
- **协议**：Raft（单组）
- **核心特性**：MVCC + Watch + Lease
- **默认读**：ReadIndex Read（3.4+，Follower 可服务线性一致读）
- **典型部署**：3 / 5 / 7 节点
- **数据目录**：`/var/lib/etcd/`（含 `member/wal/` + `member/snap/`）
- **关键参数**：心跳 100ms / 选举超时 1000ms

#### ZooKeeper
- **协议**：ZAB
- **核心特性**：znode 树 + 临时节点 + Watch
- **默认读**：顺序一致读（Follower 本地读，不保证线性一致）
- **典型部署**：3 / 5 / 7 节点 + Observer（跨地域）
- **数据目录**：`dataDir`（快照）+ `dataLogDir`（事务日志）
- **关键参数**：tickTime=2000ms / initLimit=10 / syncLimit=5

#### TiKV
- **协议**：Raft（多组，每个 Region 一个）
- **核心特性**：多 Raft 组 + PD 调度 + Region 自动分裂
- **默认读**：Leader Read + Lease Read
- **典型部署**：多节点集群，单节点数百 Region
- **关键参数**：Region 96MB（5.x+ 推荐 256MB）/ `raftstore.raft-election-timeout-ticks`

#### CockroachDB
- **协议**：Raft（多组，每个 Range 一个）
- **核心特性**：HLC + Leaseholder + 跨地域
- **默认读**：Leaseholder Read
- **典型部署**：跨地域多副本
- **关键参数**：Range 64MB（21.x+ 支持自动调整，**待 MCP 核实当前默认**）

#### Spanner
- **协议**：Paxos（多组，每个 Tablet 一个）
- **核心特性**：TrueTime + Commit Wait + 跨地域强一致
- **默认读**：Leader Read + 只读副本就近读
- **典型部署**：跨大陆多副本
- **关键参数**：TrueTime ε<7ms（2012 论文值，2024 实际未公开）/ Commit Wait ≈ ε

#### MySQL MGR
- **协议**：Paxos 变种（XCom）
- **核心特性**：多副本强一致 + 自动故障切换
- **默认模式**：single-primary（可配置 multi-primary）
- **典型部署**：3-9 节点
- **关键参数**：`group_replication_consistency`（EVENTUAL / BEFORE_ON_PRIMARY_FAILOVER / BEFORE / AFTER / BEFORE_AND_AFTER）

#### Nacos
- **协议**：Raft（持久实例，1.4+ 基于 SOFA-JRaft，1.3 前自研）+ Distro（临时实例）
- **核心特性**：双协议路由（持久 CP / 临时 AP）
- **默认读**：Leader Read
- **典型部署**：3 节点集群
- **关键参数**：`raft.electionTimeout=5000ms`（**待 MCP 核实 2.x**）

#### Consul
- **协议**：Raft（单组）
- **核心特性**：服务发现 + 健康检查 + multi-datacenter
- **默认读**：Leader Read（可配置 stale read）
- **典型部署**：3-5 节点 / 数据中心
- **关键参数**：类似 etcd

#### SOFA-JRaft
- **协议**：Raft（用户定义组数）
- **核心特性**：Java 可嵌入 + ReadIndex + Lease Read
- **核心组件**：Raft Log（RocksDB）+ Snapshot + StateMachine
- **典型应用**：蚂蚁分布式锁 / 配置中心 / RheaDB

### 3.3 工业实现分类

按"协议 + 分区"分类：

```
                     单分片                    多分片
              ┌──────────────────┬──────────────────────────┐
   Raft       │ etcd / Consul    │ TiKV / CockroachDB       │
              │ SOFA-JRaft       │ (多 Raft 组)             │
              ├──────────────────┼──────────────────────────┤
   ZAB        │ ZooKeeper        │ —                        │
              ├──────────────────┼──────────────────────────┤
   Paxos      │ —                │ Spanner / MySQL MGR      │
              │                  │ (多 Paxos 组)            │
              ├──────────────────┼──────────────────────────┤
   混合       │ Nacos            │ —                        │
              │ (Raft + Distro)  │                          │
              └──────────────────┴──────────────────────────┘
```

### 3.4 性能对比

| 系统 | 写吞吐 | 写延迟 | 读延迟 | 数据容量 |
|------|--------|--------|--------|---------|
| etcd | 数万 TPS | 10-50ms | <5ms（ReadIndex） | GB 级 |
| ZooKeeper | 数千-数万 TPS | 10-50ms | <1ms（Follower 本地顺序读）/ >10ms（跨 Leader 同步读） | GB 级 |
| TiKV | 数十万 TPS（集群级别，多 Region 并行） | 10-50ms | <10ms | PB 级 |
| CockroachDB | 数万 TPS | 50-200ms（跨地域） | 10-100ms | PB 级 |
| Spanner | 数万 TPS | 50-200ms（跨地域） | 10-100ms | PB 级 |
| MySQL MGR | 数万 TPS | 30-300ms | <10ms（主） | TB 级 |

**性能差异的根本原因**：
- 单分片 vs 多分片：多分片可横向扩展（TiKV/CockroachDB）
- 同机房 vs 跨地域：跨地域延迟高（Spanner/CockroachDB）
- 强一致读 vs 顺序一致读：顺序一致读快但不保证最新（ZooKeeper）

## 4. 选型决策框架

### 4.1 选型的核心问题

选型回答一个问题：**给定业务需求，该选哪个协议？哪个系统？**

这个问题分两层：
1. **协议层**：Paxos / ZAB / Raft？
2. **系统层**：etcd / ZooKeeper / TiKV / ...？

### 4.2 协议层选型

**协议层选型决策树**：

```
业务需求
   ↓
是否需要拜占庭容错（不可信节点）？
   ├─ 是 → PBFT / PoW / PoS（非本篇范围，参见第 2 章 3.6 节）
   └─ 否 → Crash-recovery 模型
              ↓
       是否需要跨地域强一致？
       ├─ 是 → Paxos + TrueTime（Spanner 路线）
       └─ 否 → 同机房 / 同城
                  ↓
              是否需要定制化原子广播？
              ├─ 是 → ZAB（ZooKeeper 路线）
              └─ 否 → 通用共识
                         ↓
                     Raft（易实现易维护，新系统首选）
```

**协议层选型建议**：
- **新系统**：Raft（易实现易维护，开源参考多）
- **历史系统**：保持原协议（Paxos / ZAB）
- **全球数据库**：Paxos + TrueTime（Spanner 路线）
- **协调服务**：ZAB（ZooKeeper）或 Raft（etcd）

### 4.3 系统层选型

**系统层选型矩阵**：

| 业务需求 | 推荐系统 | 理由 |
|---------|---------|------|
| K8s 元数据 | etcd | K8s 原生支持 |
| 大数据生态元数据 | ZooKeeper | Hadoop/Kafka/HBase 原生集成 |

## 5. 共识算法的演进趋势

### 5.1 演进时间线

```
1978: Lamport 时钟
   ↓
1988: VR（Oki-Liskov）← view-number + 主从复制
   ↓
1990: Paxos（Lamport）← 通用共识
   ↓
1996: Chandra-Toueg ← 共识 ≡ 原子广播等价性
   ↓

   ↓
```

### 5.2 演进的规律

- 早期 Paxos/ZAB（单机房）→ Spanner/CockroachDB（跨地域 + TrueTime/HLC）

- Paxos/ZAB/Raft（Crash-recovery）→ PBFT/PoW/PoS（Byzantine）

**规律 5：从协议到生态**
- 早期只有协议论文 → 现在有完整开源实现（etcd/TiKV/SOFA-JRaft）

### 5.3 未来趋势

**趋势 1：Raft 继续主导新系统**
- 易实现易维护的优势持续

## 6. 共识算法的边界

**共识算法能做什么不能做什么？** 前面 5 节展示了共识算法的能力，本节划清能力边界——哪些问题共识算法解决不了。理解边界才能正确选型，避免"用共识算法解决一切"的误区。

### 6.1 FLP 不可能性

**回顾第 2 章 5.2 节**：异步网络 + 一个节点崩溃 → 不存在确定性共识算法。

**对工业实践的约束**：
- 所有实用共识算法（Paxos/ZAB/Raft）都通过"部分同步假设 + 随机化"绕过 FLP
- **理论上**可能"卡住"（持续分裂），但概率极低
- **工程上**用随机超时 + 心跳 + 故障检测器应对

### 6.2 CAP 定理

**回顾第 1 章 6 节 + 第 2 章 5.3 节**：分区时 C 和 A 不能兼得。

**对工业实践的约束**：
- 共识算法都是 CP 系统（分区时拒绝服务保证 C）
- AP 系统不使用强一致共识算法：Cassandra 用 Quorum + 最终一致；Eureka 用 P2P 复制 + 心跳（非 Quorum）
- **选型时必须明确业务对 C 和 A 的偏好**

### 6.3 拜占庭容错的代价

**回顾第 2 章 3.6 节**：拜占庭故障需要 3f+1 节点 + O(n²) 通信。

**对工业实践的约束**：
- 企业内网（信任域）用 Crash-recovery 共识（Paxos/ZAB/Raft，2f+1 节点）
- 跨信任域（公链 / 联盟链）用 Byzantine 共识（PBFT/PoW/PoS，3f+1 节点）
- **性能代价**：PBFT 的 O(n²) 通信在大集群下性能急剧下降

### 6.4 性能上限

**共识算法的性能上限由以下因素决定**：

| 因素 | 影响 | 优化方向 |
|------|------|---------|
| 网络延迟 | 多数派 ACK 必须等往返 | 跨地域优化（TrueTime/HLC） |
| 磁盘持久化 | 每个事务必须 WAL | 批量写 + 异步刷盘（有数据丢失风险） |
| 多数派大小 | N=2f+1，多数派 = f+1 | 减少副本数（牺牲容错） |
| Leader 单点 | 所有写经 Leader | 多 Raft 组（TiKV/CockroachDB） |

**理论极限**：
- 单 Raft 组写入延迟 ≥ 1 个 RTT（多数派 ACK）
- 同机房：~1ms
- 跨机房：~10ms
- 跨地域：~100ms

**如何突破**？
- 批量提交（多个事务一次共识）
- 流水线（连续提议不等 ACK）
- 多组并行（不同分片并行共识）

### 6.5 共识算法不能解决什么？

**共识算法解决**：
- 多节点就某个值达成一致
- 日志的全序复制
- Leader 选举

**共识算法不解决**：
- 分布式事务的 ACID（需 2PC/TCC/Saga，第 5 章）
- 跨分片事务的协调（需 2PC + 共识，如 Spanner）
- 数据分片策略（需哈希/范围分区，第 3 章 5 节）
- 客户端会话管理（需 lease + heartbeat）
- 网络分区检测（需故障检测器，第 2 章 2.3 节）

### 6.6 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | 共识算法的边界由 FLP/CAP/拜占庭三个不可能性定理决定 |
| **江峰** | 形式化 | 三个不可能性定理是分布式算法的形式化边界——任何算法都不能违反（读者推导，江峰原书未讨论具体协议） |
| **唐伟志** | 工程 | 工程上用"部分同步 + 随机化 + 多组并行"在边界内做最大化 |
| **陈东明** | 案例 | 工业系统（Spanner/CockroachDB）展示了如何在边界内做工程妥协 |

### 6.7 本节小结

共识算法的边界：
- **FLP**：异步下不可能 → 部分同步绕过
- **CAP**：分区时 C/A 不可兼得 → 选择 CP 或 AP
- **拜占庭**：3f+1 节点 + O(n²) 通信 → 企业内网用 Crash-recovery
- **性能上限**：≥ 1 个 RTT → 批量 + 流水线 + 多组并行

**共识算法不解决分布式事务 / 分片 / 会话管理 / 分区检测**——这些需要其他机制配合。

**既然共识算法有上述边界，生产实践中如何在边界内做最大化？** 下一节给出部署、监控、故障处理的实践建议。

## 7. 生产实践建议

### 7.1 部署建议

**建议 1：节点数选择**
- 3 节点：容忍 1 故障，最小可用配置
- 5 节点：容忍 2 故障，推荐生产配置
- 7 节点：容忍 3 故障，高可用配置
- **不要用偶数节点**（4/6 节点不增加容错能力，只增加开销）

**建议 2：跨可用区部署**
- 同城 3 可用区：延迟 < 5ms，可用性高
- 跨地域 3 区域：延迟 50-200ms，可用性极高但写入慢
- **避免单可用区部署**（可用区故障 = 全宕机）

**建议 3：硬件配置**
- CPU：4-8 核（共识本身 CPU 消耗低，但应用层可能高）
- 内存：8-32GB（缓存 + 状态机）
- 磁盘：SSD 必备（WAL 对磁盘 IOPS 敏感）
- 网络：千兆以上（共识消息量小，但批量复制时高）

### 7.2 监控建议

**核心指标**：

| 指标 | 含义 | 告警阈值 |
|------|------|---------|
| Leader 存在性 | 集群是否有 Leader | 无 Leader > 10s |
| Leader 切换次数 | Leader 频繁切换告警 | > 3 次/小时 |
| 提交延迟 | 事务从提议到提交的延迟 | etcd 同机房 > 50ms / 跨地域 > 200ms |
| Follower 落后 | Follower 与 Leader 的日志差距 | > 1000 条 |
| 磁盘使用 | WAL + 快照磁盘占用 | > 80% |
| 网络流量 | 节点间流量 | 异常飙升 |

**监控工具**：
- etcd：etcdctl + Prometheus + Grafana
- ZooKeeper：四字命令（stat/mntr）+ Prometheus exporter
- TiKV：PD 监控 + Grafana

### 7.3 故障处理

**故障 1：Leader 故障**
- 现象：集群无 Leader，写请求失败
- 处理：等待选举完成（通常秒级）
- 如果选举不成功：检查网络 + 节点健康

**故障 2：网络分区**
- 现象：少数派分区无法通信
- 处理：少数派分区自动降级（CP 系统）
- 恢复：分区恢复后自动同步

**故障 3：磁盘满**
- 现象：WAL 无法写入，节点崩溃
- 处理：紧急扩容 + 清理旧快照
- 预防：定期 compact + 自动清理

**故障 4：脑裂**
- 现象：两个 Leader 同时存在
- 原因：共识算法设计正确则不会脑裂（多数派约束）
- 如果发生：检查实现是否有 bug

**故障 5：性能抖动**
- 现象：延迟飙升
- 原因：GC 暂停 / 磁盘抖动 / 网络抖动
- 处理：优化 GC + 独立磁盘 + 网络隔离

### 7.4 容量规划

**etcd / ZooKeeper 容量规划**：
- 数据量：< 10GB（单分片限制）
- QPS：写 < 10K TPS，读 < 100K QPS
- 节点数：3-7

**TiKV / CockroachDB 容量规划**：
- 数据量：PB 级（多分片）
- QPS：写 < 100K TPS（多组分片并行）
- 节点数：根据数据量动态扩展

### 7.5 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | 生产实践的核心是"理解算法的假设边界"——在边界内做最大化 |
| **江峰** | 形式化 | 故障处理的本质是"维护算法的不变量"——Leader 唯一 + 日志一致（读者推导，江峰原书未讨论具体协议） |
| **唐伟志** | 工程 | 部署 / 监控 / 故障处理是工程落地的三大支柱 |
| **陈东明** | 案例 | 工业案例（Chubby/Spanner/ZK）的生产实践是最佳参考 |

### 7.6 本节小结

生产实践建议：
- **部署**：3-7 节点 + 跨可用区 + SSD
- **监控**：Leader 存在性 + 切换次数 + 提交延迟 + Follower 落后
- **故障处理**：Leader 故障等选举 / 分区等恢复 / 磁盘满扩容
- **容量规划**：单分片 < 10GB，多分片 PB 级

## 8. 第 4 章总结与第 5 章预告

### 8.1 第 4 章完整回顾

第 4 章用 4 篇深入分布式共识：

**第 4 章-01 Paxos**：
- 从 FLP + Quorum 推导 Paxos 作为代表性工程解

- Multi-Paxos 稳定 Leader 优化

**第 4 章-02 Raft**：
- 为可理解性而生（三子问题分解）
- 领导人选举（term + 随机超时 + RequestVote）
- 日志复制（AppendEntries + prevLogIndex 检查）
- 安全性（选举限制 + 提交限制）
- 工业实现：etcd / TiKV / CockroachDB / Nacos / Consul / SOFA-JRaft

**第 4 章-03 ZAB**：
- 为 ZooKeeper 定制的原子广播
- zxid（epoch + counter）核心设计
- 两阶段（发现同步 + 广播）
- 工业实现：ZooKeeper

**第 4 章-04 对比与工业实现**（本篇）：
- 三大协议对比矩阵
- 9 个工业实现全览
- 选型决策框架
- 演进趋势 + 算法边界 + 生产实践

### 8.2 第 4 章的核心论断

**论断 1：共识算法是跨节点强一致分布式系统的底座**
- 所有强一致系统（etcd / TiKV / Spanner）都建立在共识之上
- 理解共识是理解这些系统的前提
- **注**：本地事务 / Saga / TCC 不需要共识；AP 系统（Cassandra / Eureka）也不用共识

**论断 2：三大协议数学等价，工程差异**
- Paxos / ZAB / Raft 在共识三性质上等价
- 工程差异在设计哲学（最小性 / 定制化 / 可理解性）

**论断 3：Raft 是新系统主流之一，Paxos 仍在大型全球数据库和数据库复制领域**
- 新系统（etcd / TiKV / CockroachDB）在 KV / 注册中心 / 协调服务领域倾向 Raft
- Paxos 仍是现役：Spanner（Google 全球数据库）/ MySQL MGR（MySQL 8.0+ 主推高可用）
- ZAB 主要在 ZooKeeper（及衍生项目）
- **注**：Kafka ISR / Pulsar BookKeeper 等消息队列用类共识机制，不在三大协议范畴

**论断 4：共识算法有边界**
- FLP / CAP / 拜占庭三个不可能性定理划定边界
- 共识算法不解决分布式事务 / 分片 / 会话管理
