# 第3章 分布式数据基础

> **本章定位**：从第 2 章的"系统模型"到第 4 章的"共识算法"之间的**数据层桥梁**。
>
> 第 2 章建立了"通信 + 故障 + 时间"三维度模型，证明 FLP/CAP 不可能性。但模型本身不解决工程问题——真实的分布式数据库、KV 存储、文件系统如何在模型约束下设计？
>
> 答案是**数据层抽象**：把共识算法（保证日志一致）包装为"复制 + 分区 + 一致性模型"的组合，让业务系统按需选择。本章建立三个核心抽象：
> 1. **一致性模型谱系**（从线性一致到最终一致的 7 个级别）
> 2. **复制模式**（主从 / 多主 / 无主 + Quorum）
> 3. **分区策略**（哈希 / 范围 / 一致性哈希 / 虚拟槽）
>
> 这三者的组合决定了分布式存储系统的能力边界。第 4 章的 Paxos / Raft / ZAB 是"复制模式 + 一致性模型"的工业实现。
>
> **跨书视角声明**：
> - **江峰《分布式高可用算法》**视角为主：第 3 章 3.9 节"多数派"形式化定义 + 第 7 章"共享内存"一致性——给出 Quorum 与一致性的形式化基础
> - **释慧利《深入理解分布式共识算法》**视角次之：Quorum 在 Paxos/Raft 中的应用前提
> - **唐伟志《深入理解分布式系统》**视角给工程映射：复制模式 + 分区策略的工程实现
> - **陈东明《分布式系统与一致性》**视角给案例：第 3-5 章 HDFS / BigTable / MongoDB + 第 6-9 章 RabbitMQ / ZK / Spanner / CockroachDB 的数据层选择
>
> **写作顺序的依赖关系**：
> 1. 一致性模型谱系（基础理论）→ 2. 复制模式（实现机制）→ 3. Quorum 机制（数学基础）→ 4. 分区策略（数据分布）→ 5. C 的辨析 → 6. 真实系统选型（综合应用）

## 1. 为什么需要"分布式数据基础"？

### 1.1 从模型到工程的鸿沟

第 2 章建立了系统模型，证明 FLP/CAP 不可能性。但工程上要做的事比"理解定理"复杂得多：

| 工程问题 | 模型回答 | 工程需要更细的回答 |
|---------|---------|------------------|
| "强一致"到底是什么程度？ | 线性一致性 | 还有更弱的"顺序一致"、"因果一致"等，业务该选哪个？ |
| 副本怎么同步？ | 多副本 + Crash-recovery | 主从复制？多主复制？无主复制？ |
| 多少副本确认算"写入成功"？ | 多数派 | Quorum 的 NWR 怎么配？强一致 vs 最终一致的边界？ |
| 数据怎么分散到多节点？ | 没回答 | 哈希分区？范围分区？一致性哈希？ |
| 重分片时如何最小化迁移？ | 没回答 | 一致性哈希 / 虚拟槽 / 在线迁移？ |

模型给的是"能力边界"，工程需要的是"具体方案"。本章填补这个鸿沟。

### 1.2 数据层的三个核心抽象

分布式数据系统的设计空间由三个抽象决定：

```
                    一致性模型谱系
                    （数据的"新鲜度"承诺）
                          ↓
                    ┌──────────┐
                    │ 数据系统 │ ← 复制模式（副本如何同步）
                    └──────────┘
                          ↑
                    分区策略
                    （数据如何分布）
```

- **一致性模型**：业务读到的数据有多"新"？（线性一致 / 最终一致 / ...）
- **复制模式**：副本之间如何同步？（主从 / 多主 / 无主）
- **分区策略**：数据如何分散到多节点？（哈希 / 范围 / 一致性哈希）

三者的组合决定了一个分布式存储系统的能力与限制。例如：
- **etcd**：线性一致 + 主从复制（Raft）+ 单分片（不分片）→ 强一致但容量有限
- **Cassandra**：可调一致性 + 无主复制（Quorum）+ 一致性哈希分区 → 高可用 + 可扩展
- **TiKV**：线性一致 + 主从复制（Raft）+ Range 分区（多 Raft 组）→ 强一致 + 可扩展

```
   ↓

```

## 2. 一致性模型谱系

### 2.1 一致性模型的核心问题

一致性模型回答一个问题：**业务读到的数据有多"新"？**

直觉上，"强一致"就是读到最新写入。但分布式系统下，"最新"的界定很复杂——多个客户端并发写入时，谁是"最新"？副本同步延迟时，"最新"指哪个副本？网络分区时，"最新"还能保证吗？

不同的一致性模型对这些问题给出不同回答，形成**谱系**（hierarchy）。

**为什么是 7 级？** 这 7 级并非任意分档，而是对"约束渐弱"的离散化划分——每一级都比下一级多保留一项约束：

| 级别 | 保留的约束 | 放弃的约束 |
|------|----------|----------|
| 1. 线性一致 | 全局顺序 + 真实时间顺序 | 无 |
| 2. 顺序一致 | 全局顺序 | 真实时间顺序 |
| 3. 因果一致 | 因果关系顺序 | 全局顺序（并发操作可见顺序不同） |
| 4. 读己写 | 客户端读到自己写 | 因果关系（其他客户端顺序不保证） |
| 5. 单调读 | 客户端不回退 | 读己写（不一定读到自己写） |
| 6. 单调写 | 客户端写顺序 | 单调读 |
| 7. 最终一致 | 最终收敛 | 所有顺序约束 |

**包含关系**：线性一致 ⊃ 顺序一致 ⊃ 因果一致 ⊃ {读己写、单调读、单调写} ⊃ 最终一致。

**教学简化说明**：实际上"读己写 / 单调读 / 单调写"在 Viotti & Vukolić 2016 综述中属于"会话保证（session guarantees）"，与因果一致性的包含关系是因果一致 ⊃ PRAM（含读己写+单调读+单调写），并非简单线性递减。本章按工程常用 7 级划分，便于业务选型理解。

### 2.2 线性一致性（Linearizability）

**定义**（Herlihy-Wing 1990）：所有操作看起来在某个真实时间瞬间**原子地**完成，且操作的全局顺序与真实时间一致。

**关键约束**：
- 如果操作 A 在真实时间上**先于**操作 B 完成，则在全局顺序中 A 必须先于 B
- 每个读操作返回的是"最近一次写操作的值"（在全局顺序中最近的）

**形式化**：
- 操作历史 H 可被线性化为某个**单机串行执行**的历史 H'
- H' 中操作的顺序与 H 中操作的"真实时间区间"相容（若 A 的完成时间 < B 的开始时间，则 A 在 H' 中先于 B）

**真实系统**：
- etcd（Raft + ReadIndex/Lease Read）
- Spanner（Paxos + TrueTime + Commit Wait）
- TiKV（Raft + Leader Read）

**代价**：
- 每次读必须经过 leader（或用 lease）
- 跨地域延迟高（必须等多数派 ACK）

### 2.3 顺序一致性（Sequential Consistency）

**定义**（Lamport 1979）：所有节点看到的操作顺序**相同**，且该顺序与**单个节点**上的程序顺序一致。**不要求**与真实时间一致。

**与线性一致性的差异**：
- 线性一致：全局顺序 = 真实时间顺序
- 顺序一致：全局顺序 ≠ 真实时间顺序，但所有节点看到的顺序相同

**举例**：
```
真实时间线：
  客户端 A：写 x=1 (t=10)
  客户端 B：写 x=2 (t=20)
  客户端 C：读 x   (t=30) → 读到 1
  客户端 D：读 x   (t=40) → 读到 2

顺序一致但非线性一致：全局顺序可以是 [A写1, C读=1, B写2, D读=2]
  - C 在 t=30 读到 1（B 在 t=20 已写但 C 读到旧值 1）
  - 这违反了线性一致（B 写完成于 t=25 早于 C 读开始 t=30，线性一致要求 C 必须读到 B 的写入 2 或之后值）
  - 但满足顺序一致（无跨客户端程序顺序约束，所有客户端看到的全局顺序相同）

线性一致要求：全局顺序必须 [A写1, B写2, C读, D读]，C 必须读到 1 或 2（取决于全局顺序），D 必须读到 2
```

**真实系统**：
- ZooKeeper（客户端顺序一致，但跨客户端不一定线性一致）
- MongoDB（默认读从节点，顺序一致）

**代价**：
- 读可以从任意副本读（不必须走 leader）
- 性能比线性一致好

### 2.4 因果一致性（Causal Consistency）

**定义**：有**因果关系**的操作看到相同顺序，**并发**操作可见顺序不同。

**因果关系的判定**（happened-before，第 2 章 4.3 节定义）：
- 同一节点内：A 先于 B → A → B
- 跨节点：A 发送消息触发 B → A → B
- 传递性：A → B 且 B → C → A → C

**举例**：
```
客户端 A：发帖 "今天天气好"
客户端 B：看到 A 的帖，回复 "确实"
客户端 C：可能先看到 B 的回复，后看到 A 的帖（违反因果）

因果一致要求：C 必须先看到 A 的帖，再看到 B 的回复
```

**真实系统**：
- COPS（Lloyd et al. 2011）
- Dynamo（Amazon 2007 论文中的内部系统，用向量时钟支持因果跟踪；**注**：AWS 对外的 DynamoDB 服务标准 API 只提供 strongly/eventually consistent read，不直接暴露因果一致性选项，因果一致需客户端用条件写入实现）
- MongoDB 3.6+（causal sessions）

**代价**：
- 需要跟踪因果关系（向量时钟 / 版本向量）
- 性能比顺序一致好，比最终一致差

### 2.5 读己写一致性（Read-Your-Writes）

**定义**：客户端能读到**自己**的写。

**为什么需要**：用户发帖后立即刷新页面，应该看到自己的帖子。如果读到了旧副本，用户体验差。

**实现方式**：
- **粘性会话**（Sticky Session）：客户端始终读同一节点（如果该节点是自己写的节点）
- **读主**（Read Primary）：写后立即读走主库
- **时间戳对比**：客户端记录最后写入的时间戳，读时检查副本是否同步到该时间戳

**真实系统**：
- MySQL 主从 + 写后读主
- Redis 主从 + 客户端路由
- Cassandra（读时指定 LOCAL_ONE 但写后用 LOCAL_QUORUM）

### 2.6 单调读一致性（Monotonic Reads）

**定义**：客户端**不会看到**"回退"的数据——一旦读到某个时间的值，后续读到的值至少和它一样新。

**为什么需要**：用户先看到"订单已发货"，刷新后看到"订单待发货"——体验灾难。

**实现方式**：
- 客户端记录最后读到的时间戳，下次读时要求副本至少同步到该时间戳
- 一致性哈希让客户端始终读同一副本

### 2.7 单调写一致性（Monotonic Writes）

**定义**：同一客户端的写操作按其发出顺序生效。

**为什么需要**：用户先改昵称为"A"，再改为"B"——最终值必须是"B"而不是"A"。

**实现方式**：客户端序列化写操作，或服务端用客户端 ID 做串行化。

### 2.8 最终一致性（Eventual Consistency）

**定义**（Vogels 2009）：如果不再有新的写入，所有副本**最终**会收敛到相同值。

**关键点**：
- **不保证何时**收敛（可能 ms 级，也可能小时级）
- **不保证中间状态**（过程中可能读到任意值）
- 是**最弱**的一致性级别

**真实系统**：
- Cassandra（默认 ONE 一致性）
- DynamoDB（默认 eventual consistency read）
- DNS（典型最终一致——TTL 内可能读到旧值）

**冲突解决**：
- **Last-Write-Wins（LWW）**：用时间戳取最新值（Cassandra 默认）
- **Vector Clock + Siblings**：保留冲突版本，客户端解决（Riak）
- **CRDT**（Conflict-free Replicated Data Type）：数据结构本身可合并（Riak 原生支持；**Redis 社区版不原生支持 CRDT**，仅 Redis Enterprise 商业版的 Active-Active 功能基于 CRDT 实现）

### 2.9 一致性谱系图

```
强 ←─────────────────────────────────────────────→ 弱

线性一致 → 顺序一致 → 因果一致 → 读己写 → 单调读 → 单调写 → 最终一致
   │           │           │          │         │         │         │
   │           │           │          │         │         │         └─ 不保证何时收敛
   │           │           │          │         │         └─ 客户端写顺序
   │           │           │          │         └─ 不回退
   │           │           │          └─ 读到自己写
   │           │           └─ 因果序保证
   │           └─ 全局顺序一致（不要求时间）
   └─ 全局顺序 = 真实时间
```

**包含关系**：线性一致 ⊃ 顺序一致 ⊃ 因果一致 ⊃ ... ⊃ 最终一致

**跨书视角对照**（4 本书对"一致性模型"的不同侧重）：

| 作者 | 视角 | 核心论点 | 在本章对应 |
|------|------|---------|-----------|
| **江峰** | 形式化视角 | 一致性模型是分布式算法的"contract"——每种模型对应一组可证明性质，选模型 = 选性质。**江峰 3.9 节多数派定理**给出了 Quorum 的形式化基础 | 第 4 节 Quorum 推导 |
| **释慧利** | 算法视角 | Paxos/Raft 保证线性一致；ZAB 保证顺序一致（客户端视角）；**共识算法的能力边界由一致性模型刻画**——同样的 Raft，配合 ReadIndex 是线性一致，配合 Follower Read 可降级为顺序一致 | 第 4 章共识算法的工程化 |
| **唐伟志** | 工程视角 | 工程上常妥协——**同一系统不同操作用不同级别**（如 Redis 主从：写主线性一致，读从最终一致；etcd 默认线性一致，但可降级为 stale read） | 第 7 节真实系统的选型 |
| **陈东明** | 案例视角 | Spanner 选线性一致（TrueTime）；Dynamo 选最终一致（LWW）；ZooKeeper 选顺序一致（ZAB）——**每个选择背后都有具体的业务场景与代价权衡** | 第 7 节真实系统的选型 |

**4 视角互补与差异**：
- **江峰 vs 释慧利**：江峰问"一致性模型为什么对"（公理化基础），释慧利问"共识算法中如何用"（算法能力边界）——前者是后者的形式化前提
- **唐伟志 vs 陈东明**：唐伟志给"工程如何妥协"（同系统多级别），陈东明给"真实系统如何选"（不同系统不同级别）——前者偏设计模式，后者偏案例库
- **互补关系**：江峰给"为什么"、释慧利给"算法如何用"、唐伟志给"工程如何妥协"、陈东明给"真实系统如何选"——4 个角度共同覆盖一致性模型的全貌

### 2.10 本节小结

一致性模型是 7 级谱系，从线性一致（最强）到最终一致（最弱）。**强一致性代价高（必须 leader 串行化），弱一致性性能好但语义模糊**。业务按需选择——金融账户必须线性一致，用户画像可最终一致。

## 3. 复制模式

### 3.1 复制模式的核心问题

复制（Replication）回答一个问题：**副本之间如何同步？**

不同的同步方式决定了：
- 写入延迟（同步确认 vs 异步确认）
- 数据一致性（强一致 vs 最终一致）
- 可用性（单点故障 vs 多点容错）
- 吞吐量（写放大 vs 并行写入）

### 3.2 主从复制（Leader-Based / Primary-Backup）

**架构**：
```
              Client
                ↓
             ┌──────┐
             │ Leader│
             └──┬───┘
        ┌───────┼───────┐
        ↓       ↓       ↓
    ┌──────┐ ┌──────┐ ┌──────┐
    │Follower1│Follower2│Follower3│
    └──────┘ └──────┘ └──────┘
```

**写入流程**：
1. Client 发送写请求到 Leader
2. Leader 把写操作追加到本地日志
3. Leader 把日志复制到所有 Follower
4. Follower ACK
5. Leader 收到多数派 ACK 后 commit
6. Leader 响应 Client

**读取流程**：
- 默认从 Leader 读（强一致）
- 也可从 Follower 读（最终一致 / 顺序一致）

**故障切换**：
- Leader 宕机 → 选举新 Leader（通过共识算法）
- Follower 宕机 → 重新加入时从 Leader 同步缺失日志

**真实系统**：
- **MySQL 主从**（异步复制）+ MGR（Paxos 半同步）
- **Redis 主从**（异步复制 + 哨兵选主）
- **Kafka**（ISR + Leader Epoch）
- **etcd / ZooKeeper / TiKV**（Raft / ZAB 强一致）

**优势**：
- 写入串行化，避免冲突
- 强一致（线性一致）容易实现
- 客户端路由简单（写必走 Leader）

**劣势**：
- Leader 是单点（写 QPS 受 Leader 能力限制）
- Leader 故障时服务短暂不可用（选举期间）
- 跨地域写入延迟高（必须等多数派 ACK）

### 3.3 多主复制（Multi-Leader）

**架构**：
```
   数据中心 A                  数据中心 B
   ┌──────────┐               ┌──────────┐
   │ Leader A │ ←—异步复制—→ │ Leader B │
   └────┬─────┘               └────┬─────┘
        ↓                          ↓
   Follower A                  Follower B
```

**写入流程**：
1. Client 写入本地数据中心的 Leader
2. Leader 本地确认（不等跨数据中心）
3. Leader 异步复制到其他数据中心
4. 其他数据中心的 Leader 应用变更

**冲突处理**（多主的核心难题）：
- **Last-Write-Wins**：用时间戳取最新（依赖时钟同步）
- **客户端解决**：保留冲突版本，由客户端决定（Riak siblings）
- **CRDT**：数据结构自动合并（如 G-Counter / LWW-Set）
- **业务层合并**：应用逻辑决定（如购物车合并）

**真实系统**：
- **Cassandra**（跨数据中心多主）
- **CouchDB**（多主 + 版本向量）
- **MySQL 双主**（不推荐——容易脑裂）

**优势**：
- 跨地域低延迟（本地数据中心即可写）
- 容忍数据中心级故障
- 写吞吐高（多 Leader 并行）

**劣势**：
- 写冲突复杂（必须设计冲突解决策略）
- 一致性弱（无法线性一致）
- 实现复杂度高

### 3.4 无主复制（Leaderless / Quorum-Based）

**架构**（Dynamo 风格）：
```
              Client
                ↓
         ┌──────────────┐
         │  Coordinator │（任意节点都可担任）
         └──────┬───────┘
        ┌───────┼───────┐
        ↓       ↓       ↓
    ┌──────┐ ┌──────┐ ┌──────┐
    │  N1  │ │  N2  │ │  N3  │
    └──────┘ └──────┘ └──────┘
```

**写入流程**（W 个副本确认）：
1. Client 发送写请求到 Coordinator
2. Coordinator 把写发送到所有 N 个副本
3. W 个副本 ACK 后，Coordinator 响应 Client 成功

**读取流程**（R 个副本返回）：
1. Client 发送读请求到 Coordinator
2. Coordinator 把读发送到所有 N 个副本
3. R 个副本返回值，Coordinator 取最新版本返回 Client

**Quorum 机制**（详见第 4 节）：无主复制依赖 Quorum 协调读写副本数——`W + R > N` 时读写副本有交集，保证强一致；`W + R ≤ N` 时为弱一致。具体推导与配置见第 4 节。

**真实系统**：
- **Cassandra**（可调 W/R）
- **DynamoDB**（Quorum + LWW）
- **Riak**（Quorum + Vector Clock）
- **Voldemort**（LinkedIn 开源）

**优势**：
- 无 Leader 单点
- 写吞吐高（多副本并行）
- 容忍多节点故障

**劣势**：
- 冲突解决复杂
- 强一致需要 Quorum（W+R>N），延迟较高
- 实现复杂度高

### 3.5 复制模式对比

| 模式 | Leader | 写入 | 一致性 | 冲突 | 典型系统 |
|------|--------|------|--------|------|---------|
| 主从 | 单 Leader | 串行 | 强一致（默认） | 无 | etcd / ZK / MySQL / Redis |
| 多主 | 多 Leader | 并行 | 最终一致 | 复杂 | Cassandra / CouchDB |
| 无主 | 无 Leader | 并行 | 可调（Quorum） | 复杂 | DynamoDB / Riak |

### 3.6 复制延迟与一致性 tradeoff

| 同步策略 | 延迟 | 一致性 | 故障丢失风险 |
|---------|------|--------|------------|
| 全同步（所有副本 ACK） | 高 | 强一致 | 无 |
| 多数派同步（Quorum ACK） | 中 | 强一致 | 低（多数派存活） |
| 半同步（1 个副本 ACK） | 低 | 半一致 | 中（ACK 副本也宕机时丢失） |
| 全异步 | 最低 | 最终一致 | 高（主库宕机时未同步数据丢失） |

**真实案例（MySQL 复制策略对比）**：
- **异步复制**：主库写完 binlog 立即 ACK，从库异步同步。延迟 < 1ms，但主库宕机可能丢数据
- **半同步复制**（rpl_semi_sync_master_wait_for_slave_count=1）：主库等至少 1 个从库 ACK。延迟 10-100ms，丢数据风险降低但不消除
- **MGR（Paxos）**：主库等多数派 ACK。延迟 30-300ms，强一致不丢数据

### 3.7 本节小结

复制模式决定了"副本如何同步"：
- **主从**：强一致，Leader 单点
- **多主**：跨地域低延迟，冲突复杂
- **无主**：高可用，Quorum 调一致性

**主从 + 共识算法（Raft/Paxos）是企业内网的主流选择**——强一致 + 自动故障切换。下一节深入 Quorum 的数学基础。

## 4. Quorum 机制（NWR 模型）

### 4.1 Quorum 的核心问题

Quorum 回答一个问题：**多少副本确认算"写入成功"？多少副本返回算"读取成功"？**

直觉上"所有副本"最安全但最慢，"任意一个副本"最快但可能不一致。Quorum 是这两者的数学权衡。

### 4.2 NWR 定义

设系统有 N 个副本，写入需要 W 个副本确认，读取需要 R 个副本返回：

- **N**（Number of replicas）：副本总数
- **W**（Write quorum）：写入确认所需副本数
- **R**（Read quorum）：读取返回所需副本数

**Quorum 定理**：`W + R > N` 保证强一致（读写副本必有交集）

### 4.3 强一致 Quorum 推导

**为什么 `W + R > N` 保证强一致？**

鸽笼原理（Pigeonhole Principle）：
- N 个副本被分成两组：W 个写入副本 + R 个读取副本
- 如果 `W + R > N`，则两组**必有交集**（至少 `W + R - N` 个副本同时被写和读）
- 交集副本持有最新写入，读取时必能从交集中拿到最新值

**举例**：
- N=3, W=2, R=2 → W+R=4 > 3，交集至少 1 个 → 强一致
- N=3, W=1, R=1 → W+R=2 ≤ 3，可能无交集 → 弱一致
- N=5, W=3, R=3 → W+R=6 > 5，交集至少 1 个 → 强一致

### 4.4 Quorum 配置与一致性级别

| 配置 | W | R | 一致性 | 性能 | 典型场景 |
|------|---|---|--------|------|---------|
| 强一致 | N | N | 线性一致 | 最慢 | 金融账户 |
| 多数派 | ⌈(N+1)/2⌉ | ⌈(N+1)/2⌉ | 线性一致 | 中 | etcd / Raft |
| 写优先 | N | 1 | 强一致（写全确认） | 读快写慢 | 读多写少 |
| 读优先 | 1 | N | 强一致（读全副本） | 写快读慢 | 写多读少 |
| 弱一致 | 1 | 1 | 最终一致 | 最快 | 日志 / 缓存 |

### 4.5 Quorum 与故障容忍的关系

设故障容忍 f（允许 f 个副本宕机），则：
- 写入可用：`N - W ≤ f` → `W ≥ N - f`
- 读取可用：`N - R ≤ f` → `R ≥ N - f`
- 强一致：`W + R > N`

**多数派选择**（W = R = ⌈(N+1)/2⌉）：
- N = 2f+1 → W = R = f+1
- 写入可用：N - W = f ≤ f ✓（容忍 f 宕机）
- 读取可用：N - R = f ≤ f ✓
- 强一致：W + R = 2f+2 > 2f+1 ✓

**结论**：多数派 Quorum（W = R = ⌈(N+1)/2⌉）是**同时满足强一致 + 容忍 f 宕机**的最优配置。这就是 Raft / Paxos 选择 2f+1 节点 + 多数派 ACK 的数学基础。

### 4.6 Sloppy Quorum 与 Hinted Handoff

**问题**：严格 Quorum 要求 W 个副本可达，但如果某些副本暂时不可达怎么办？

**Sloppy Quorum**（宽松 Quorum）：
- 写入时，如果目标副本不可达，Coordinator 把写入"借"给其他可达节点
- 这些"借"的节点标记为 hinted（暗示的）
- 原副本恢复后，hinted 节点把数据回传

**真实系统**：
- Cassandra：`hinted_handoff_enabled=true`，hint 保留 3 小时（默认）
- DynamoDB：类似机制

**优势**：
- 提高可用性（即使部分副本宕机也能写）
- 故障恢复后自动同步

**劣势**：
- 期间不保证强一致（读写副本可能无交集）
- hinted 数据可能丢失（hinted 节点也宕机时）

### 4.7 Quorum 在共识算法中的角色

Quorum 是 Paxos / Raft / ZAB 的数学基础：

| 算法 | 写入 Quorum | 读取 Quorum | 说明 |
|------|-----------|-----------|------|
| Paxos | 多数派 | 多数派 | Prepare + Accept 两阶段，每阶段多数派 |
| Raft | 多数派 | 多数派（默认） | 日志复制多数派 ACK；读可走 Leader 或 ReadIndex |
| ZAB | 多数派 | 多数派 | Leader 原子广播，多数派 ACK |

**为什么都是多数派？**
- 多数派是同时满足 `W + R > N`（强一致）和 `W ≤ N - f`（容忍 f 宕机）的最小 Quorum
- 更大的 Quorum（如 W=N）一致性更强但可用性下降
- 更小的 Quorum（如 W=1）可用性更高但一致性弱

### 4.8 Quorum 的工程权衡

| 业务需求 | 推荐配置 | 一致性 | 可用性 |
|---------|---------|--------|--------|
| 金融账户 | W=Quorum, R=Quorum | 强一致 | 容忍少数故障 |
| 商品库存 | W=Quorum, R=Quorum | 强一致 | 容忍少数故障 |
| 用户会话 | W=1, R=1 | 最终一致 | 高可用 |
| 实时推荐 | W=1, R=1 | 最终一致 | 高可用 |
| 配置中心 | W=Quorum, R=Quorum | 强一致 | 容忍少数故障 |

**跨书视角对照**：
- **江峰视角**：Quorum 是分布式算法的形式化基础——多数派公理（majority quorum axiom）保证了共识算法的 safety
- **释慧利视角**：Paxos 的两阶段（Prepare/Accept）每阶段都依赖多数派，这是 Paxos 可证明正确性的关键
- **唐伟志视角**：工程上 Quorum 大小直接影响延迟和可用性，必须按业务权衡
- **陈东明视角**：DynamoDB 用 NWR 可调一致性，Cassandra 用 ONE/QUORUM/ALL 三档

### 4.9 本节小结

Quorum 机制是"副本数 / 写确认数 / 读返回数"的数学权衡：
- `W + R > N` 保证强一致（鸽笼原理）
- **崩溃故障模型下**，多数派 Quorum（W=R=⌈(N+1)/2⌉）是同时满足强一致 + 容忍 f 宕机的最优配置（**注**：拜占庭故障模型下需要 3f+1 节点 + 2f+1 Quorum，参见第 2 章 3.6 节）
- Paxos / Raft / ZAB 都基于多数派 Quorum

## 5. 分区策略

**Quorum 解决了"副本数"问题，但当数据量超过单机容量时，还需分区（Sharding）把数据分散到多节点**。第 4 节的 Quorum 是"纵向冗余"（同一数据多副本），第 5 节的分区是"横向切分"（不同数据分到不同节点）。两者正交组合，构成分布式存储的标准架构。

### 5.1 分区的核心问题

分区（Partitioning / Sharding）回答一个问题：**数据如何分散到多节点？**

单机容量有限（如第 1 章所述），必须分区。但分区方式决定了：
- 数据分布是否均匀
- 范围查询是否高效
- 重分片代价大小
- 热点是否容易处理

### 5.2 哈希分区（Hash Partitioning）

**思路**：对键做哈希，取模分配到分片。

```
shard = hash(key) % N
```

**优势**：
- 数据分布均匀（哈希函数保证）
- 实现简单

**劣势**：
- 范围查询失效（相邻键可能在不同分片）
- **重分片代价大**：N 变化时，几乎所有键都要迁移
  - 从 16 分片扩到 32 分片：迁移量 50% × 2 = 100%（理论上几乎所有键都要重新映射）

**真实系统**：
- 早期 MySQL 分库分表（按 user_id % N）
- Memcached 客户端（twemproxy / libmemcached）用一致性哈希做分片——**Memcached 服务端本身无分布式能力**，一致性哈希在客户端代理层实现

### 5.3 范围分区（Range Partitioning）

**思路**：按键的顺序范围分配到分片。

```
shard 1: [0, 1000)
shard 2: [1000, 2000)
shard 3: [2000, 3000)
...
```

**优势**：
- 范围查询高效（同一范围内数据在同一分片）
- 重分片只需分裂/合并相邻分片

**劣势**：
- 热点问题（如按时间分区，最新数据集中在一个分片）
- 数据分布不均（某些范围数据量大）

**真实系统**：
- HBase（按 RowKey 范围分区，Region 自动分裂）
- TiKV（按 Key Range 分区，每个 Range 一个 Raft 组）
- MongoDB（按 shard key 范围分区，chunk 自动分裂）

### 5.4 一致性哈希（Consistent Hashing）

**动机**：解决简单哈希分区"重分片代价大"的问题。

**思路**（Karger et al. 1997）：
- 把哈希空间组织成环（0 ~ 2³²-1）
- 节点和键都映射到环上
- 键顺时针找到的第一个节点即为其归属

```
       0
     /   \
  N1      N3
   |       |
  N2------N4
       2^32-1

key 的归属：从 key 位置顺时针找到的第一个节点
```

**重分片分析**：
- 加入新节点 N5：只影响 N5 之前一段范围内的键（原本属于下一个节点的部分）
- 移除节点 N3：N3 上的键迁移到下一个节点 N4
- **最小迁移**：每个节点加入/退出只影响一段范围的键

**虚拟节点（Virtual Nodes）**：
- 真实节点映射多个虚拟节点（如 1 个真实节点 = 200 个虚拟节点）
- 解决数据分布不均的问题
- 节点退出时，其虚拟节点分散到多个真实节点

**真实系统**：
- Cassandra（3.x 默认 256 虚拟节点 / 节点；**4.x 起默认改为 16**——CASSANDRA-7032 发现 256 在大集群下导致 token ring 维护开销过高）
- DynamoDB（一致性哈希 + 虚拟节点）
- Memcached（一致性哈希）

### 5.5 虚拟槽（Virtual Slots / Hash Slot）

**思路**（Redis Cluster 设计）：
- 固定数量的"槽"（如 Redis Cluster 16384 个）
- 键通过 CRC16 计算，映射到槽
- 节点负责一段连续的槽范围

```
slot = CRC16(key) % 16384

节点 A：slots 0-5460
节点 B：slots 5461-10922
节点 C：slots 10923-16383
```

**为什么是 16384（2¹⁴）个槽？**（antirez 在 GitHub issue #2576 的官方回复给出 3 个理由）：
1. **心跳 bitmap 大小**：Redis Cluster 心跳包用 bitmap 表示节点负责的槽，16384 / 8 = 2KB；如果是 65536 槽则 bitmap = 8KB，心跳开销过大
2. **集群规模上限**：Redis Cluster 建议集群规模 ≤ 1000 节点（实际硬上限 16384 节点），16384 槽足够每节点平均分到 16 个槽——槽数与节点数匹配
3. **压缩传输**：心跳 bitmap 经过 gzip 压缩后，16384 槽的 bitmap 压缩率高于 65536 槽（节点负责的槽通常连续，压缩效果更好），网络开销更小

工程权衡：16384 是"足够分散 + 心跳开销可接受 + 集群规模匹配"的平衡点。

**重分片分析**：
- 槽是逻辑概念，迁移单位是"槽"
- 槽迁移过程中可同时服务（标记为 MIGRATING / IMPORTING）
- 客户端收到 MOVED / ASK 重定向

**真实系统**：
- Redis Cluster（16384 槽）
- Couchbase（1024 vBucket）

### 5.6 分区策略对比

| 策略 | 数据分布 | 范围查询 | 重分片代价 | 热点处理 | 典型系统 |
|------|---------|---------|----------|---------|---------|
| 哈希分区 | 均匀 | 失效 | 高（几乎全迁移） | 难 | 早期 MySQL |
| 范围分区 | 不均 | 高效 | 低（分裂/合并） | 难（最新数据热点） | HBase / TiKV |
| 一致性哈希 | 均匀（虚拟节点） | 失效 | 低（最小迁移） | 中 | Cassandra / DynamoDB |
| 虚拟槽 | 均匀 | 失效 | 低（按槽迁移） | 中 | Redis Cluster |

### 5.7 重分片的工程实践

**在线重分片**（不停止服务）：
- **TiKV**：`SPLIT TABLE` 命令，分裂 Region（默认 96MB），自动负载均衡
- **Redis Cluster**：`CLUSTER SETSLOT` 命令，槽迁移过程中标记 MIGRATING / IMPORTING
- **Vitess**（YouTube）：在线 vertical / horizontal splitting，业务无感知
- **MongoDB**：balancer 自动迁移 chunk，业务无感知

**离线重分片**（停止服务）：
- 早期 MySQL：停服 + dump + reload + 切换
- 不推荐——影响业务连续性

### 5.8 分区与复制的组合

实际系统通常**分区 + 复制**结合：
- 每个分片有多个副本（提供高可用）
- 不同分片的副本独立（提供可扩展性）

```
分片 1: Leader 1 + Follower 1.1 + Follower 1.2
分片 2: Leader 2 + Follower 2.1 + Follower 2.2
分片 3: Leader 3 + Follower 3.1 + Follower 3.2
```

**多 Raft 组**（TiKV / CockroachDB）：
- 每个分片是一个独立的 Raft 组
- 不同分片的 Leader 可以分布在不同节点（负载均衡）
- 单节点可同时是分片 A 的 Leader + 分片 B 的 Follower

**真实系统**：
- **TiKV**：Range 分区 + 多 Raft 组（每个 Region 一个 Raft 实例）
- **CockroachDB**：Range 分区 + 多 Raft 组 + HLC
- **Cassandra**：一致性哈希 + 无主复制（每分片多副本）

### 5.9 本节小结

分区策略决定了"数据如何分布"：
- 哈希分区：均匀但范围查询失效，重分片代价大
- 范围分区：范围查询高效但易热点
- 一致性哈希：最小迁移 + 虚拟节点均衡
- 虚拟槽：固定槽数 + 心跳优化

**分区 + 复制组合**是分布式存储的标准架构——分片提供可扩展性，副本提供高可用。下一节辨析"一致性"的不同含义。

## 6. 数据一致性 vs 一致性模型

### 6.1 两个"C"的混淆

分布式系统中有两个常被混淆的"C"：
- **ACID 中的 C**（Consistency）：约束一致性（数据满足业务约束）
- **CAP 中的 C**（Consistency）：副本一致性（所有副本数据相同）

二者是**不同维度**的概念，但常被混淆。

### 6.2 ACID 的 C：约束一致性

**定义**：事务执行前后，数据库满足**业务约束**（如外键、唯一性、检查约束）。

**举例**：
- 转账事务：A 扣 100，B 加 100，总额不变（约束：A + B 总和不变）
- 订单事务：扣库存 + 创建订单，库存不能为负（约束：stock ≥ 0）

**实现**：
- 单机：数据库约束 + 触发器 + 应用层校验
- 分布式：分布式事务（2PC / TCC / Saga）+ 业务层补偿

**关键**：ACID C 是**业务层**的概念——数据库不知道"总额不变"，是应用定义的约束。

### 6.3 CAP 的 C：副本一致性

**定义**：所有副本在同一时间看到相同数据（线性一致性）。

**举例**：
- 写入 x=1 后，任意副本读到 x=1（不是旧值 x=0）
- 多客户端并发读，看到的顺序一致

**实现**：
- 共识算法（Paxos / Raft / ZAB）保证副本一致
- Quorum 机制（W+R>N）

**关键**：CAP C 是**系统层**的概念——副本之间数据同步的状态。

### 6.4 二者的关系

```
ACID C（业务约束）          CAP C（副本一致）
       ↓                         ↓
   业务层约束                系统层同步
       ↓                         ↓
   单机事务保证              共识算法保证
       ↓                         ↓
   分布式事务保证            分布式存储保证
```

**关系**：
- ACID C 关注"业务约束是否满足"——可以单机实现（不需要副本）
- CAP C 关注"副本是否一致"——是分布式存储的属性
- 二者**正交**——可以有 ACID C 但无 CAP C（单机数据库），也可以有 CAP C 但无 ACID C（最终一致的 KV 存储）

**真实案例**：
- **MySQL 单机**：ACID C ✓，CAP C ✗（无副本）
- **etcd**：CAP C ✓（Raft），ACID C 部分（事务支持简单）
- **Cassandra**：CAP C ✗（最终一致），ACID C ✗（无强事务）
- **Spanner**：ACID C ✓（分布式事务），CAP C ✓（TrueTime + Paxos）

### 6.5 分布式事务的 C 与共识的 C

**分布式事务**（如 2PC / TCC / Saga）保证 ACID C：
- 2PC：协调者保证所有参与者 commit 或全部 abort
- TCC：业务层补偿保证最终一致
- Saga：长事务通过补偿回滚

**共识算法**（如 Paxos / Raft）保证 CAP C：
- 副本之间日志一致
- 客户端读到的值一致

**关系**：**跨节点强一致**分布式事务的协调者本身可以是共识算法实现（如 Spanner 的协调者基于 Paxos）。**对于跨节点强一致分布式事务，共识是其基础设施**——只有副本一致（CAP C），才能保证事务 ACID。
- 注：本地事务、Saga 长事务、TCC 补偿事务不必然依赖共识算法
- 注：2PC 本身不保证副本一致（参与者崩溃时阻塞），需共识算法作为协调者才能既保证 ACID C 又保证 CAP C——这是 Spanner 的设计动机

### 6.6 BASE 理论：AP 系统的工程实践

BASE（Basically Available + Soft state + Eventually consistent）是 AP 系统的设计指南：
- **Basically Available**：基本可用（允许响应时间增加、功能降级）
- **Soft state**：软状态（允许中间状态存在，不要求即时一致）
- **Eventually consistent**：最终一致（不读时最终所有副本一致）

**与 ACID 的对比**：
| 维度 | ACID | BASE |
|------|------|------|
| 一致性 | 强一致（约束 + 副本） | 最终一致 |
| 可用性 | 牺牲可用性保证一致 | 牺牲一致性保证可用 |
| 性能 | 低（必须协调） | 高（无需协调） |
| 适用场景 | 金融 / 关键业务 | 互联网高并发 |

### 6.7 本节小结

两个"C"是不同维度：
- ACID C：业务约束（应用层）
- CAP C：副本一致（系统层）

二者正交，可以单独保证。**共识算法（Paxos/Raft）保证 CAP C，跨节点强一致分布式事务（2PC/TCC）保证 ACID C**——对于跨节点强一致事务，共识是其基础设施；对于本地事务 / Saga / TCC 补偿事务，不必然依赖共识。

## 7. 真实系统的数据基础选择

### 7.1 本节的核心问题

第 2-6 节建立了三个抽象（一致性模型 / 复制模式 / 分区策略）。本节把它们映射到真实系统——每个生产级分布式存储都做了一组选择，理解这些选择才能正确使用。

### 7.2 MySQL 的数据基础

**一致性模型**：单机 ACID（强一致）+ 主从异步（最终一致）

**复制模式**：主从复制
- 异步复制：主库写完 binlog 即 ACK，从库异步同步
- 半同步复制：主库等 1 个从库 ACK
- MGR：Paxos 多数派 ACK（强一致）

**分区策略**：手动分库分表（ShardingSphere / MyCat）
- 哈希分片（按 user_id % N）
- 范围分片（按时间）
- 广播表（小表全量复制）

**关键工程参数**：
- `binlog_format=ROW`（推荐）
- `sync_binlog=1`（每次写都 sync）
- `innodb_flush_log_at_trx_commit=1`（每次事务都 flush）
- 半同步：`rpl_semi_sync_master_wait_for_slave_count=1`

### 7.3 Redis Cluster 的数据基础

**一致性模型**：最终一致（主从异步复制）

**复制模式**：主从复制
- 主节点写 → 异步同步到从节点
- 主节点宕机时，哨兵/集群选主

**分区策略**：虚拟槽（16384 槽）
- `slot = CRC16(key) % 16384`
- 每个节点负责一段槽范围
- 槽迁移过程中标记 MIGRATING / IMPORTING

**关键工程参数**：
- `cluster-enabled yes`
- `cluster-node-timeout 15000`（节点超时，触发故障转移）
- `cluster-require-full-coverage yes`（默认：所有槽覆盖才提供服务）

### 7.4 Cassandra 的数据基础

**一致性模型**：可调一致性（ONE / QUORUM / ALL / LOCAL_QUORUM）

**复制模式**：无主复制（Dynamo 风格）
- Coordinator 把写发送到所有副本
- W 个副本 ACK 即成功

**分区策略**：一致性哈希 + 虚拟节点
- 默认 256 虚拟节点 / 节点
- 节点加入/退出时最小迁移

**关键工程参数**：
- Replication Factor (RF)：通常 3
- Write Consistency：ONE / QUORUM / ALL
- Read Consistency：ONE / QUORUM / ALL
- `hinted_handoff_enabled=true`

### 7.5 TiKV 的数据基础

**一致性模型**：线性一致（Raft + Leader Read）

**复制模式**：主从复制（Raft）
- 每个 Region 是一个 Raft 组
- Leader 处理读写，Follower 备份

**分区策略**：Range 分区（按 Key Range）
- 默认 Region 大小 96MB（历史默认；5.x+ 推荐 256MB 以减少 Region 数量、降低 Raft 开销）
- Region 自动分裂 / 合并
- PD（Placement Driver）调度 Region 分布

**关键工程参数**：
- Region 大小：96MB（历史默认；5.x+ 推荐 256MB）
- Raft 选举超时：`raft-election-timeout-ticks`
- 多 Raft 组：单节点可同时是多个 Region 的 Leader / Follower

### 7.6 Spanner 的数据基础

**一致性模型**：线性一致（TrueTime + Paxos）

**复制模式**：主从复制（Paxos）
- 每个 Paxos 组跨地域多副本
- Leader 处理读写

**分区策略**：Range 分区 + 自动分裂
- Tablet 是基本单元
- 分裂/合并自动进行

**关键工程参数**：
- TrueTime 不确定性 ε < 7ms（2012 论文典型值，2024 年实际值未公开——但工程上仍为同一量级）
- Commit Wait ≈ ε
- Paxos 组通常 3-5 副本跨地域

### 7.7 MongoDB 的数据基础

**一致性模型**：可调（读偏好 primary / primaryPreferred / secondary / secondaryPreferred / nearest）

**复制模式**：主从复制（副本集）
- Primary 处理写
- Secondary 异步同步 OpLog
- 选举基于 Raft 变种

**分区策略**：Range 分区（shard key 范围）
- Chunk 自动分裂（5.x 之前默认 64MB；**6.0+ 改为自动调整**——unsplittable chunk + auto-merger，不再固定阈值）
- Balancer 自动迁移 chunk

**关键工程参数**：
- 副本集大小：通常 3 / 5 / 7
- Write Concern：w=1 / w=majority / w=all
- Read Concern：local / majority / linearizable

### 7.8 真实系统对比

| 系统 | 一致性 | 复制 | 分区 | 算法 |
|------|--------|------|------|------|
| MySQL | ACID + 异步 | 主从 | 手动 | binlog |
| Redis Cluster | 最终一致 | 主从异步 | 虚拟槽 | 哨兵选主 |
| Cassandra | 可调 | 无主 | 一致性哈希 | Quorum |
| TiKV | 线性一致 | 主从（多 Raft 组） | Range | Raft |
| Spanner | 线性一致 | 主从（Paxos） | Range | Paxos + TrueTime |
| MongoDB | 可调 | 主从（副本集） | Range | Raft 变种 |
| etcd | 线性一致 | 主从（单 Raft 组） | 单分片 | Raft |
| ZooKeeper | 顺序一致 | 主从（ZAB） | 单分片 | ZAB |
| DynamoDB | 可调 | 无主 | 一致性哈希 | Quorum + LWW |

### 7.9 选型决策矩阵

| 业务需求 | 推荐 | 理由 |
|---------|------|------|
| 强一致 + 关系数据 | TiDB / Spanner | ACID + 线性一致 |
| 强一致 + KV | etcd / ZooKeeper | Raft / ZAB + 线性一致 |
| 高可用 + 互联网应用 | Cassandra | 可调一致性 + 无主 |
| 缓存 + 低延迟 | Redis Cluster | 主从 + 虚拟槽 |
| 文档 + 灵活查询 | MongoDB | 副本集 + Range 分区 |
| 跨地域强一致 | Spanner | TrueTime + Paxos |
| 日志 + 高吞吐 | Kafka | ISR + 主从复制 |

**跨书视角对照**：
- **陈东明视角**：第 3-5 章详解 HDFS / BigTable / MongoDB 的数据层选择，第 6-9 章详解 RabbitMQ / ZK / Spanner / CockroachDB
- **唐伟志视角**：第 7 章案例研究对比不同系统的工程取舍
- **江峰视角**：从形式化角度，所有系统的选择都可以归约为"一致性模型 + 复制模式 + 分区策略"三维度组合
- **释慧利视角**：共识算法（Paxos/Raft）是强一致系统的共同底座，复制模式是其上的工程封装

### 7.10 本节小结

真实系统的数据基础选择：
- **强一致 + 关系数据**：TiKV / Spanner（Raft/Paxos + Range 分区）
- **强一致 + KV**：etcd / ZooKeeper（Raft/ZAB + 单分片）
- **高可用 + 互联网**：Cassandra（Quorum + 一致性哈希）
- **缓存 + 低延迟**：Redis Cluster（主从 + 虚拟槽）

**模型选择没有银弹**——每个选择都是业务需求与系统能力的权衡。

## 8. 本章总结与后续路径

### 8.1 三个抽象回顾

| 抽象 | 选项 | 工程主流 |
|------|------|---------|
| 一致性模型 | 线性 / 顺序 / 因果 / 读己写 / 单调读 / 单调写 / 最终 | 线性一致（CP）/ 最终一致（AP） |
| 复制模式 | 主从 / 多主 / 无主 | 主从 + 共识算法 |
| 分区策略 | 哈希 / 范围 / 一致性哈希 / 虚拟槽 | 范围（强一致 KV）/ 一致性哈希（AP）/ 虚拟槽（Redis） |

### 8.2 Quorum 的核心地位

Quorum 是连接"复制模式"和"共识算法"的数学基础：
- `W + R > N` 保证强一致（鸽笼原理）
- 多数派 Quorum（W = R = ⌈(N+1)/2⌉）是强一致 + 故障容忍的最优解
- Paxos / Raft / ZAB 都基于多数派 Quorum

### 8.3 两个 C 的辨析

| C | 含义 | 层次 | 保证手段 |
|---|------|------|---------|
| ACID C | 业务约束 | 应用层 | 事务（2PC / TCC / Saga） |
| CAP C | 副本一致 | 系统层 | 共识算法（Paxos / Raft） |

二者正交，**跨节点强一致**分布式事务以共识为基础设施。

### 8.4 共识算法的工程映射

```
共识算法（Paxos / Raft / ZAB）
       ↓
保证 CAP C（副本一致）
       ↓
支撑主从复制 + 线性一致
       ↓
应用于强一致存储（etcd / TiKV / Spanner）
       ↓
为分布式事务提供基础设施
```

第 4 章将深入 Paxos / Raft / ZAB 的算法细节，理解它们如何在第 2 章的模型约束下实现 Quorum + Leader + Log 的工程化。

### 8.6 关键认知

带着以下 3 个认知进入第 4 章：

**认知 1：一致性是谱系，不是二选一**
- 7 级一致性模型，从线性一致到最终一致
- 业务按需选择，强一致代价高

**认知 2：Quorum 是共识算法的数学基础**
- `W + R > N` 保证强一致
- 多数派 Quorum 是崩溃故障模型下的最优配置（强一致 + 容忍 f 宕机）
- Paxos / Raft / ZAB 都基于多数派

**认知 3：两个 C 是不同维度**
- ACID C（业务约束）vs CAP C（副本一致）
- 共识保证 CAP C，分布式事务保证 ACID C
- 共识是**跨节点强一致**分布式事务的基础设施
