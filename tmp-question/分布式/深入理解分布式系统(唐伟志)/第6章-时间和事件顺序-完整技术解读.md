## 1. 为什么"时间"是分布式系统的核心难题？

### 1.1 单机系统的时间直觉

单机系统下，时间似乎很简单：
- 所有操作在同一台机器上执行
- 操作系统提供单调时钟（`clock_gettime(CLOCK_MONOTONIC)`）
- 事件顺序明确：先 A 后 B，A.timestamp < B.timestamp

单机时间的本质：**单一时间源 + 单调递增**。所有事件都能用同一个时钟排序。

### 1.2 分布式系统打破时间直觉

分布式系统有 N 个节点，每个节点有自己的时钟：
- **时钟漂移**：不同节点的时钟走快走慢不一
- **时钟回拨**：NTP 校时可能导致时钟往回跳
- **无全局时钟**：没有单一时间源能为所有节点提供精确时间

**核心难题**：如何在无全局时钟的情况下，判定事件的先后顺序？

### 1.3 事件顺序的三种类型

分布式系统的事件顺序有三种：

**类型 1：物理顺序（真实时间）**
- 事件 A 在真实时间 10:00:00 发生
- 事件 B 在真实时间 10:00:01 发生
- A 先于 B

**问题**：分布式系统无法精确获取真实时间（第 2 节详解）。

**类型 2：因果顺序（happened-before）**
- 事件 A 发送消息，事件 B 接收该消息
- A 因果先于 B（A → B）

**特点**：不依赖物理时间，只依赖因果关系。

**类型 3：全局一致顺序**
- 所有节点对事件顺序达成一致
- 但不一定与真实时间一致

**共识算法**（第 4 章）就是实现全局一致顺序的机制——所有节点对日志顺序达成一致。

### 1.4 本章的核心问题

本章回答四个核心问题：

1. **物理时钟为什么不可靠？**（第 2 节）
2. **如何用逻辑时钟判定因果顺序？**（第 3-4 节）
3. **如何结合物理时钟与逻辑时钟？**（第 5-6 节）
4. **如何捕获分布式系统的全局状态？**（第 7 节）

## 2. 物理时钟的局限

### 2.1 物理时钟的工作原理

**石英晶振**：
- 现代计算机时钟基于石英晶振
- 晶振频率受温度、电压、老化影响
- 标称频率的偏差称为**时钟漂移（clock drift）**

**时钟漂移率**：
- 典型服务器：1-100 ppm（百万分之一）
- 高质量服务器：0.1-1 ppm
- 每天漂移：1 ppm × 86400 秒 = 86ms

### 2.2 NTP（网络时间协议）

**NTP**（Network Time Protocol）用于同步节点时钟：
- 层次结构：Stratum 0（原子钟/GPS）→ Stratum 1 → Stratum 2 → ...
- 客户端向 NTP 服务器请求时间
- 根据往返延迟（RTT）计算时间偏差（offset）

**NTP 的两种校时模式**：
- **slew**：逐步调整时钟频率（慢校），偏差 < 128ms 时使用
- **step**：直接跳变时钟（快校），偏差 ≥ 128ms 时使用

**step 模式的问题**：时钟可能往回跳（时钟回拨）。

### 2.3 NTP 的精度限制

| 部署 | NTP 同步精度 |
|------|-------------|
| 同机房（局域网） | 1-10ms |
| 跨机房（城域网） | 10-100ms |
| 跨地域（互联网） | 100ms-1s |

**精度限制的原因**：
- 网络延迟不对称（上行 ≠ 下行）
- NTP 服务器本身有误差
- 网络抖动

### 2.4 时钟回拨的工程影响

**时钟回拨**会导致：
- **时间戳乱序**：先写入的数据时间戳可能更晚
- **缓存 TTL 失效**：基于时间戳的过期判断错误
- **分布式锁租约失效**：租约基于时钟，回拨导致租约延长
- **Snowflake ID 冲突**：时间戳回拨导致重复 ID

**生产案例**：
- **Snowflake ID 冲突**：Twitter Snowflake 依赖机器时钟单调递增，NTP 校时回拨导致 ID 重复
- **Redis 锁租约**：基于 EXPIRE 的锁，时钟回拨导致锁提前释放或延长
- **Kafka 消息时间戳**：Producer 时钟回拨导致消息时间戳乱序

### 2.5 物理时钟不可靠的根本原因

**根本原因**：分布式系统没有"全局真实时间"。

**物理时钟的两个不可能**：
1. **不可能完全同步**：NTP 有精度限制（ms 级）
2. **不可能单调递增**：NTP step 模式可能回拨

**结论**：**普通 NTP 同步的物理时钟不能用于判定分布式系统的全局事件顺序**（**注**：Google TrueTime 通过 GPS+原子钟把不确定性降到有界，可判定全局顺序——第 6 节详解）。这是逻辑时钟诞生的根本动机。

### 2.6 chrony：更现代的 NTP 替代

**chrony**（Red Hat 开发）相比传统 ntpd：
- 更快的同步速度
- 更平滑的校时（减少 step 跳变）
- 更好的网络抖动容忍

**生产建议**：现代 Linux 发行版默认使用 chrony 替代 ntpd。

### 2.7 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | 物理时钟不可靠是逻辑时钟诞生的动机——Lamport 1978 提出 happened-before 关系 |
| **江峰** | 形式化 | 物理时钟的不可靠性是分布式算法不依赖物理时间的根本原因（读者推导） |
| **唐伟志** | 工程 | 工程上用 chrony + 监控时钟漂移应对物理时钟问题 |
| **陈东明** | 案例 | Spanner 用 GPS + 原子钟把物理时钟不确定性降到 7ms（第 8 章） |

### 2.8 本节小结

物理时钟的局限：
- **时钟漂移**：1-100 ppm，每天 86ms-8.6s
- **NTP 精度**：1ms-1s（取决于网络）
- **时钟回拨**：导致时间戳乱序、缓存失效、锁租约错误
- **根本原因**：无全局真实时间

带着物理时钟的局限进入第 3 节——Lamport 时钟。

## 3. Lamport 时钟（逻辑时钟的奠基）

### 3.1 Lamport 时钟的核心问题

Lamport 时钟回答一个问题：**如何在不依赖物理时间的前提下，判定事件的先后顺序？**

### 3.2 Lamport 的洞察（1978）

**Lamport 1978 论文《Time, Clocks, and the Ordering of Events in a Distributed System》**的核心洞察：

> "The distribution of the system across a network of computers should be transparent to the users... We define a happened-before relation."

**洞察**：分布式系统中，我们关心的不是"真实时间顺序"，而是"因果关系顺序"。如果事件 A 因果先于事件 B（A → B），那么 A 应该"先于" B。

### 3.3 happened-before 关系（→）

**定义**：
1. **同一节点内**：如果事件 A 先于事件 B 发生，则 A → B
2. **跨节点**：如果事件 A 是"发送消息 m"，事件 B 是"接收消息 m"，则 A → B
3. **传递性**：如果 A → B 且 B → C，则 A → C
4. **并发**：如果既非 A → B 也非 B → A，则 A 与 B 并发（A || B）

**关键**：happened-before 不依赖物理时间，只依赖因果关系。

### 3.4 Lamport 时钟的算法

每个节点维护一个逻辑时钟 `C_i`（整数计数器）：

```
规则 1：本地事件
  C_i = C_i + 1
  事件的时间戳 = C_i

规则 2：发送消息
  C_i = C_i + 1
  发送 (message, C_i)

规则 3：接收消息 (m, C_m)
  C_i = max(C_i, C_m) + 1
  事件的时间戳 = C_i
```

**关键性质**：
- 如果 A → B，则 C(A) < C(B)（因果序保持）
- **但反之不成立**：C(A) < C(B) 不一定意味着 A → B（可能是并发）

### 3.5 Lamport 时钟的局限

**局限**：Lamport 时钟只能检测"因果序"，不能检测"并发"。

**示例**：
- 事件 A（节点 1）：C(A) = 3
- 事件 B（节点 2）：C(B) = 5
- C(A) < C(B)，但无法判断是 A → B 还是 A || B

**为什么这是局限？** 检测并发对于冲突解决很重要（如 DynamoDB 的向量时钟检测写冲突）。

### 3.6 Lamport 时钟的工业应用

**应用 1：Paxos 的 Proposal ID**
- Paxos 的 Proposal ID 是 Lamport 时钟的实例
- `(round_number, node_id)` 单调递增
- 保证：如果 Proposal A 因果先于 Proposal B，则 A.ID < B.ID

**应用 2：Kafka 的 Offset**
- 分区内的 offset 单调递增，提供分区内全序
- **注**：Offset 按 broker 收到顺序分配，不严格满足 Lamport 因果保持性（若 m1 发送后 m2 发送，但 m1 网络延迟晚到，offset 会 m2<m1）——Offset 是分区内全序编号，非严格 Lamport 时钟

**应用 3：Redis 主从复制的 replication offset**
- offset 是 Lamport 时钟的实例
- 用于判断主从同步进度

**应用 4：数据库的自增 ID**
- MySQL 的 auto_increment 是单机 Lamport 时钟
- 分布式场景需用 Snowflake（带 worker ID 的 Lamport 时钟）

### 3.7 Lamport 时钟与全序

**Lamport 时钟可以实现全序**：通过 `(timestamp, node_id)` 组合。

```
如果 C(A) == C(B)：
  比较 node_id：node_id 小者优先
否则：
  C 小者优先
```

**全序的意义**：所有事件都有唯一确定的顺序——这是共识算法的基础（第 4 章 Paxos/Raft 都用类似机制）。

### 3.8 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | 第 1-2 章 Lamport 时钟是共识算法的基础——Paxos 的 Proposal ID 是 Lamport 时钟的实例 |
| **江峰** | 形式化 | Lamport 时钟的 happened-before 关系是分布式算法形式化的基石（读者推导） |
| **唐伟志** | 工程 | 工程上 Lamport 时钟用于 Kafka offset / Redis replication offset / Snowflake ID |
| **陈东明** | 案例 | 数据库自增 ID / Kafka offset / Redis offset 都是 Lamport 时钟的工业实例 |

### 3.9 本节小结

Lamport 时钟：
- **核心思想**：用逻辑计数器替代物理时钟
- **happened-before**：因果顺序不依赖物理时间
- **关键性质**：A → B 则 C(A) < C(B)（但反之不成立）
- **局限**：不能检测并发
- **工业应用**：Paxos Proposal ID / Kafka offset / Redis offset / Snowflake

带着 Lamport 时钟的局限进入第 4 节——向量时钟。

## 4. 向量时钟（精确刻画因果关系）

### 4.1 向量时钟的核心问题

向量时钟回答一个问题：**如何精确判定两个事件是因果有序还是并发？**

### 4.2 向量时钟的算法

每个节点维护一个向量 `V_i = [c_1, c_2, ..., c_n]`，其中 `c_j` 表示节点 i 看到的节点 j 的事件数。

```
规则 1：本地事件
  V_i[i] = V_i[i] + 1

规则 2：发送消息
  发送 (message, V_i)

规则 3：接收消息 (m, V_m)
  V_i[j] = max(V_i[j], V_m[j]) for all j
  V_i[i] = V_i[i] + 1
```

### 4.3 向量时钟的因果关系判定

**比较规则**：
- `V(A) < V(B)` 当且仅当 V(A) 的每个分量都 ≤ V(B) 的对应分量，且至少一个分量严格小于
- `V(A) || V(B)` 当且仅当既非 `V(A) < V(B)` 也非 `V(B) < V(A)`（并发）

**示例**：
```
V(A) = [2, 1, 0]
V(B) = [3, 2, 0]
V(C) = [1, 0, 2]

V(A) < V(B)：A 因果先于 B ✓
V(A) || V(C)：A 与 C 并发 ✓
```

### 4.4 向量时钟的工业应用

**应用 1：Dynamo（Amazon 2007 论文系统，DynamoDB 的前身）**
- Amazon Dynamo 用向量时钟检测写冲突
- 多个客户端并发写同一 key，向量时钟判定是否冲突
- 冲突时保留多个版本（siblings），客户端解决

**应用 2：Riak**
- Riak 继承 Dynamo 的向量时钟
- siblings 机制保留冲突版本

**应用 3：Git 的 commit 关系**
- Git commit 的 parent 关系本质是向量时钟
- merge commit 有多个 parent，体现并发历史

**应用 4：Cassandra（早期）**
- Cassandra 选择 LWW 路线（基于 timestamp），**从未采用向量时钟**——与 Dynamo 形成对比，牺牲冲突检测换取性能
- 牺牲冲突检测换取性能

### 4.5 向量时钟的局限

**局限 1：空间开销 O(n)**
- 向量大小等于节点数 n
- n 增大时，向量空间线性增长

**局限 2：节点动态加入/退出**
- 节点列表变化时，向量管理复杂
- 需要额外的节点 ID 分配机制

**局限 3：客户端需理解向量时钟**
- DynamoDB / Riak 要求客户端处理 siblings
- 增加业务复杂度

### 4.6 向量时钟的优化

**优化 1：Dotted Version Vector**
- 减少向量大小
- 用于 Riak 2.0

**优化 2：ID-based version vector**
- 用客户端 ID 替代节点 ID
- 适合客户端多的场景

### 4.7 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | 向量时钟解决了 Lamport 时钟的局限——精确判定并发 |
| **江峰** | 形式化 | 向量时钟的因果关系判定是分布式冲突检测的形式化基础（读者推导） |
| **唐伟志** | 工程 | 工程上向量时钟的 O(n) 空间开销是主要问题——Dynamo/Riak 用 siblings 解决冲突 |
| **陈东明** | 案例 | Dynamo / Riak 用向量时钟检测写冲突；Cassandra 改用 LWW 牺牲冲突检测 |

### 4.8 本节小结

向量时钟：
- **核心思想**：用向量替代标量，精确刻画因果关系
- **比较规则**：分量比较，判定因果序或并发
- **工业应用**：DynamoDB / Riak / Git / Cassandra（早期）
- **局限**：O(n) 空间 + 节点动态管理 + 客户端复杂度

带着向量时钟的局限进入第 5 节——HLC。

## 5. HLC（混合逻辑时钟）

### 5.1 HLC 的核心问题

HLC 回答一个问题：**如何结合物理时钟的"时间直觉"和逻辑时钟的"因果保证"？**

### 5.2 HLC 的动机

**物理时钟的优势**：有时间直觉（接近真实时间）
**物理时钟的劣势**：不可靠（漂移、回拨）

**逻辑时钟的优势**：因果保证可靠
**逻辑时钟的劣势**：无时间直觉（纯计数器，不知道事件何时发生）

**HLC 的目标**：两者结合——既有因果保证，又有时间直觉。

### 5.3 HLC 的算法（Kulkarni 2014）

HLC 维护 `(physical_ms, logical_count)`：

```
规则 1：本地事件
  physical_ms = max(physical_ms, system_clock())
  if physical_ms 未变：
    logical_count = logical_count + 1
  else：
    logical_count = 0
  时间戳 = (physical_ms, logical_count)

规则 2：发送消息
  发送 (message, HLC)

规则 3：接收消息 (m, m_physical, m_logical)
  physical_ms = max(physical_ms, system_clock(), m_physical)
  if physical_ms == m_physical == 之前的 physical_ms：
    logical_count = max(logical_count, m_logical) + 1
  elif physical_ms == 之前的 physical_ms：
    logical_count = logical_count + 1
  elif physical_ms == m_physical：
    logical_count = m_logical + 1
  else：
    logical_count = 0
```

### 5.4 HLC 的特性

**特性 1：保持因果序**
- 如果 A → B，则 HLC(A) < HLC(B)
- 类似 Lamport 时钟

**特性 2：有时间直觉**
- HLC 的 physical_ms 接近真实时间
- 可以用 HLC 估算事件发生的大致时间

**特性 3：不受时钟回拨影响**
- 如果系统时钟回拨，HLC 的 physical_ms 保持不变（取 max）
- logical_count 兜底，保证单调递增

### 5.5 HLC 的工业应用

**应用 1：CockroachDB**
- CockroachDB 用 HLC 实现跨地域事务排序
- 每个 Transaction 有 HLC 时间戳
- 读操作按 HLC 时间戳获取快照

**应用 2：MongoDB**
- MongoDB 的 clusterTime 是 HLC 的实例
- 用于跨 shard 事务排序

**应用 3：YugabyteDB**
- YugabyteDB 用 HLC（Hybrid Time）
- 类似 CockroachDB

### 5.6 HLC vs Lamport vs 向量时钟

| 时钟 | 因果序 | 并发检测 | 时间直觉 | 空间 | 典型应用 |
|------|--------|---------|---------|------|---------|
| Lamport | 单向 | ✗ | ✗ | O(1) | Paxos / Kafka / Redis |
| 向量时钟 | 精确 | ✓ | ✗ | O(n) | Dynamo / Riak |
| HLC | 单向 | ✗ | ✓ | O(1) | CockroachDB / MongoDB |
| TrueTime | 外部一致性（线性一致+真实时间序） | ✗ | ✓ | O(1) | Spanner |

### 5.7 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | HLC 是 Lamport 时钟的工程优化——加物理时间直觉，保持因果保证 |
| **江峰** | 形式化 | HLC 的正确性依赖"物理时钟有界漂移"假设（读者推导） |
| **唐伟志** | 工程 | CockroachDB 用 HLC 避免对 TrueTime（原子钟）的依赖 |
| **陈东明** | 案例 | CockroachDB / MongoDB / YugabyteDB 都用 HLC |

### 5.8 本节小结

HLC（混合逻辑时钟）：
- **核心思想**：物理时钟 + 逻辑计数器结合
- **特性**：因果保证 + 时间直觉 + 不受回拨影响
- **工业应用**：CockroachDB / MongoDB / YugabyteDB
- **与 Lamport/向量时钟的关系**：Lamport 的增强版（加时间直觉），向量时钟的简化版（O(1) 空间）

**HLC 的局限**：HLC 仍是点时间戳，跨地域事务无法证明 T1 < T2 的真实时间序（HLC 保证因果序但不保证真实时间序）。**全球强一致事务需要"有界不确定性的物理时间"——这是 TrueTime 诞生的直接动机**。带着 HLC 进入第 6 节——TrueTime。

## 6. TrueTime（Google 的全球时间方案）

### 6.1 TrueTime 的核心问题

TrueTime 回答一个问题：**如何在全球范围内提供"有界不确定性"的物理时间？**

### 6.2 TrueTime 的背景

**Spanner**（Corbett et al., OSDI 2012）需要全球强一致事务。挑战：
- 跨大陆延迟 50-200ms
- 物理时钟不可靠（NTP 精度 100ms-1s）
- 强一致事务需要全局时间戳

**Spanner 的解决方案**：TrueTime API。

### 6.3 TrueTime 的工作原理

**TrueTime 的硬件**：
- 每个 Google 数据中心配 GPS 接收器 + 原子钟
- GPS 提供卫星时间
- 原子钟提供高精度本地时间（GPS 故障时备份）

**TrueTime API**：
```python
TT.now()  → [earliest, latest]  # 返回时间区间
TT.after(t)  → bool  # t 是否已完全过去
TT.before(t)  → bool  # t 是否还未到来
```

**关键**：TrueTime 不返回单一时间，而是返回**区间**。区间的宽度 ε 是时间不确定性。

### 6.4 TrueTime 的不确定性 ε

**ε 的来源**：
- GPS 接收器延迟
- 原子钟漂移
- 网络延迟

**ε 的典型值**：
- 2012 Spanner 论文：ε 平均 ~4ms，最大 ~7ms；Commit Wait 平均 ~7ms
- 2024 年实际值未公开（但工程上仍为同一量级）

**Commit Wait**：
- 事务提交时等待 `TT.after(commit_timestamp)` 为真
- 即等待 ε 过去，保证全局一致
- 典型等待 7ms

### 6.5 TrueTime 实现外部一致性

**外部一致性（external consistency）** = 线性一致性 + 真实时间顺序

**TrueTime 如何实现外部一致性**：
1. 事务 T1 在时间戳 t1 提交（等待 Commit Wait）
2. 事务 T2 在 T1 提交后开始
3. T2 的时间戳 t2 > t1（因为 T2 开始时 T1 已提交，TT.now() 的 earliest > t1）
4. 全局顺序与真实时间一致

**这是 Spanner 的核心创新**——通过 TrueTime + Commit Wait，实现全球范围的线性一致性。

### 6.6 TrueTime vs HLC

| 维度 | TrueTime | HLC |
|------|----------|-----|
| 硬件 | GPS + 原子钟 | 普通 NTP |
| 不确定性 | 7ms（有界） | 无界（NTP 精度） |
| 时间保证 | 区间 [earliest, latest] | 点时间戳 |
| Commit Wait | 需要（等 ε） | 不需要 |
| 成本 | 高（专用硬件） | 低（普通服务器） |
| 一致性 | 外部一致性 | 因果一致 + 时间直觉 |
| 典型系统 | Spanner | CockroachDB |

**关键差异**：TrueTime 用昂贵硬件换取有界不确定性，HLC 用逻辑计数器兜底但无有界保证。

### 6.7 TrueTime 的成本与局限

**成本**：
- 每个 Google 数据中心配 GPS + 原子钟
- 专用硬件成本高
- 只有 Google 等巨头能负担

**局限**：
- 外部用户无法直接使用 TrueTime
- CockroachDB / YugabyteDB 用 HLC 替代 TrueTime
- 牺牲外部一致性，换取成本降低

### 6.8 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | TrueTime 通过有界不确定性 + Commit Wait 实现外部一致性 |
| **江峰** | 形式化 | TrueTime 的正确性依赖"ε 有界"假设（读者推导） |
| **唐伟志** | 工程 | TrueTime 成本高，CockroachDB 用 HLC 替代 |
| **陈东明** | 案例 | Spanner 用 TrueTime 实现全球强一致事务（第 8 章） |

### 6.9 本节小结

TrueTime：
- **核心思想**：GPS + 原子钟提供有界不确定性的物理时间
- **API**：`TT.now()` 返回区间 [earliest, latest]
- **Commit Wait**：等 ε 过去，保证全局一致
- **外部一致性**：线性一致 + 真实时间顺序
- **成本**：高（专用硬件），只有 Google 等巨头能负担

**TrueTime 与分布式快照的联系**：TrueTime 用时间戳切分时间，分布式快照用 marker 消息切分时间——两者本质相同，都是"逻辑时间点"的切分。带着 TrueTime 进入第 7 节——分布式快照。

## 7. 分布式快照（Chandy-Lamport 算法）

### 7.1 分布式快照的核心问题

分布式快照回答一个问题：**如何捕获分布式系统的全局状态，而不停止系统？**

### 7.2 为什么需要分布式快照？

**应用场景**：
1. **检查点（Checkpoint）**：周期性保存系统状态，故障恢复
2. **死锁检测**：判断是否死锁
3. **垃圾回收**：判断哪些对象可回收
4. **调试**：分析分布式系统在某个时刻的状态

**挑战**：分布式系统没有"全局时钟"，无法在同一瞬间获取所有节点状态。

### 7.3 Chandy-Lamport 算法（1985）

**Chandy-Lamport 算法**（1985）是分布式快照的经典算法。

**假设**：
- 通道是 FIFO（先进先出）
- 通道可靠（消息不丢失）

**算法流程**：

1. **发起者 P 记录自己的状态**
2. **P 向所有出边通道发送 marker 消息**
3. **节点 Q 收到 marker 消息（首次）**：
   - Q 记录自己的状态
   - Q 在入边通道（marker 来源）上记录"marker 之后的消息"为空
   - Q 在其他入边通道上记录"收到 marker 之前"的消息
   - Q 向所有出边通道发送 marker
4. **节点 Q 收到 marker 消息（非首次）**：
   - Q 在对应入边通道上停止记录消息

**关键**：marker 消息像"快门"，把系统状态切分为"快照前"和"快照后"。

### 7.4 Chandy-Lamport 的正确性

**不变量**：快照捕获的是"一致性状态"——所有节点状态 + 通道消息构成一个可达的系统状态。

**正确性保证**：
- FIFO 通道保证 marker 之后的消息不会进入快照
- marker 之前的消息要么已处理（在节点状态中），要么在通道中（在通道状态中）

### 7.5 分布式快照的工业应用

**应用 1：Flink 的 Checkpoint**
- Flink 用 Chandy-Lamport 变种做 Checkpoint
- Source 节点注入 barrier（类似 marker）
- barrier 随数据流传播
- 节点收到 barrier 后记录状态

**应用 2：Spark Streaming 的微批**
- Spark Streaming 用微批（micro-batch）而非快照
- 每个微批是一个独立的事务

**应用 3：数据库的分布式快照**
- Percolator 用时间戳实现快照隔离（第 5 章）
- Spanner 用 TrueTime 时间戳实现快照

### 7.6 分布式快照与时间戳的关系

**联系**：
- Chandy-Lamport 用 marker 消息切分时间
- Percolator / Spanner 用时间戳切分时间
- 两者本质相同——都是"逻辑时间点"的切分

**差异**：
- Chandy-Lamport：基于消息（marker）
- Percolator/Spanner：基于时间戳（TSO/TrueTime）

### 7.7 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | 分布式快照是时间戳切分的消息版——marker 消息扮演时间戳角色 |
| **江峰** | 形式化 | Chandy-Lamport 的正确性依赖"FIFO 通道 + marker 不变量"（读者推导） |
| **唐伟志** | 工程 | Flink 用 Chandy-Lamport 变种（barrier）做 Checkpoint |
| **陈东明** | 案例 | Flink / Spark Streaming / Percolator / Spanner 都用到分布式快照思想 |

### 7.8 本节小结

分布式快照（Chandy-Lamport 算法）：
- **核心思想**：marker 消息切分时间，捕获一致性状态
- **算法**：发送 marker → 记录状态 → 传播 marker
- **正确性**：FIFO 通道 + marker 不变量
- **工业应用**：Flink Checkpoint / Percolator / Spanner

带着分布式快照进入第 8 节——工业应用。

## 8. 时间在工业系统中的应用

### 8.1 工业系统的时间方案总览

| 系统 | 时间方案 | 一致性 | 典型场景 |
|------|---------|--------|---------|
| Spanner | TrueTime | 外部一致性 | 全球数据库 |
| CockroachDB | HLC | 因果一致 + 时间直觉 | 全球 SQL |
| TiDB | TSO（中心化时间戳） | 线性一致 | 分布式数据库 |
| MongoDB | HLC（clusterTime） | 因果一致 | 文档数据库 |
| Kafka | Offset（Lamport 时钟） | 分区内全序 | 消息队列 |
| Redis | Replication Offset | 主从顺序 | 缓存 |
| DynamoDB | 向量时钟（Dynamo 论文） | 最终一致 | KV 存储 |
| Flink | Barrier（Chandy-Lamport） | 快照一致 | 流处理 |

### 8.2 Spanner：TrueTime + Paxos

**时间方案**：TrueTime（GPS + 原子钟）
**事务模型**：2PC + TrueTime 时间戳
**一致性**：外部一致性（线性一致 + 真实时间顺序）
**Commit Wait**：等 ε（~7ms）过去

**Spanner 的事务流程**：
1. 获取 TrueTime 时间戳 t1
2. 获取锁
3. 写入数据（带 t1）
4. 等待 Commit Wait（`TT.after(t1)` 为真）
5. 释放锁
6. 通知参与者

### 8.3 CockroachDB：HLC + Raft

**时间方案**：HLC（混合逻辑时钟）
**事务模型**：Percolator 变种 + HLC 时间戳
**一致性**：因果一致 + 时间直觉（非外部一致性）
**优势**：无需 TrueTime 硬件，普通服务器即可

### 8.4 TiDB：TSO + Raft

**时间方案**：TSO（Timestamp Oracle，中心化时间戳分配）
**TSO 实现**：PD（Placement Driver）单点分配时间戳
**事务模型**：Percolator + TSO
**一致性**：线性一致
**局限**：TSO 是单点（PD 集群高可用，但仍是中心化）

### 8.5 Kafka：Offset + ISR

**时间方案**：Offset（Lamport 时钟的实例）
**分区顺序**：分区内 offset 单调递增
**跨分区**：无全局顺序（除非用事务 API）
**事务 API**：跨分区原子写入（idempotent producer + transactional API）

### 8.6 Flink：Barrier + Checkpoint

**时间方案**：Barrier（Chandy-Lamport 算法的 marker）
**Checkpoint 流程**：
1. Source 注入 barrier
2. barrier 随数据流传播
3. 算子收到 barrier 后记录状态
4. 所有算子收到 barrier 后 Checkpoint 完成
**Exactly-Once 语义**：Checkpoint + 重放 = Exactly-Once

### 8.7 工业方案的对比

| 维度 | Spanner | CockroachDB | TiDB | Kafka | Flink |
|------|---------|-------------|------|-------|-------|
| 时间方案 | TrueTime | HLC | TSO | Offset | Barrier |
| 硬件需求 | GPS+原子钟 | 普通 | 普通 | 普通 | 普通 |
| 一致性 | 外部一致 | 因果+时间 | 线性一致 | 分区内全序 | 快照一致 |
| 性能 | 中（Commit Wait） | 中 | 高 | 高 | 高 |
| 适用场景 | 全球数据库 | 全球 SQL | 分布式数据库 | 消息队列 | 流处理 |

### 8.8 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | 工业方案的时间选择是"一致性 vs 成本"的权衡——TrueTime 最强但最贵 |
| **江峰** | 形式化 | 工业方案的正确性依赖各自时间方案的不变量（TrueTime ε 有界 / HLC 因果序 / TSO 中心化）（读者推导） |
| **唐伟志** | 工程 | 工程上 TSO 是中心化但简单，HLC 是去中心化但弱，TrueTime 是最强但贵 |
| **陈东明** | 案例 | Spanner / CockroachDB / TiDB / Kafka / Flink 展示了不同时间方案的工业选择 |

### 8.9 本节小结

工业系统的时间方案：
- **Spanner**：TrueTime（外部一致性，成本高）
- **CockroachDB**：HLC（因果一致，成本低）
- **TiDB**：TSO（线性一致，中心化）
- **Kafka**：Offset（分区内全序）
- **Flink**：Barrier（快照一致）

带着这些工业应用进入第 9 节——时间与共识的关系。

## 9. 时间与共识的关系

### 9.1 时间与共识的联系

**共识算法**（第 4 章）实现全局一致顺序——所有节点对事件顺序达成一致。

**时间戳**是共识算法的关键组件：
- Paxos 的 Proposal ID 是 Lamport 时钟的实例
- Raft 的 term 是简化版 Lamport 时钟
- ZAB 的 zxid 是 epoch + counter（类似 HLC）

### 9.2 共识用逻辑时钟而非物理时钟

**为什么共识算法用逻辑时钟？**
1. **物理时钟不可靠**：第 2 节已说明
2. **因果保证**：逻辑时钟保证因果序
3. **全序保证**：`(timestamp, node_id)` 实现全序

### 9.3 时间戳在共识中的作用

| 共识算法 | 时间戳 | 作用 |
|---------|--------|------|
| Paxos | Proposal ID (round + node_id) | 区分提案新旧 |
| Raft | term | 区分 Leader 任期 |
| ZAB | zxid (epoch + counter) | 区分事务新旧 + Leader 任期 |

**关键**：时间戳保证"新提案覆盖旧提案"的安全性。

### 9.4 共识与分布式事务的关系

**分布式事务**（第 5 章）依赖时间戳：
- Percolator 用 TSO 时间戳实现快照隔离
- Spanner 用 TrueTime 时间戳实现外部一致性
- Seata AT 用 XID（事务 ID）协调全局事务

**共识是分布式事务的基础设施**：
- 跨节点强一致事务（Percolator/Spanner）依赖共识保证副本一致
- TSO/PD 本身用共识算法保证高可用

### 9.5 时间的层级

```
物理时钟（不可靠）
   ↓
逻辑时钟（Lamport / 向量时钟）
   ↓
混合时钟（HLC / TrueTime）
   ↓
共识算法（Paxos / Raft / ZAB）
   ↓
分布式事务（2PC / Percolator / Spanner）
   ↓
业务应用（金融 / 电商 / 大数据）
```

**每一层建立在前一层之上**：
- 逻辑时钟解决物理时钟的不可靠
- 混合时钟结合物理与逻辑
- 共识算法用时间戳实现全局一致顺序
- 分布式事务用共识 + 时间戳实现 ACID

### 9.6 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **释慧利** | 算法 | 共识算法的时间戳是 Lamport 时钟的实例——Paxos Proposal ID / Raft term / ZAB zxid |
| **江峰** | 形式化 | 时间的层级是分布式系统理论的核心——从物理到逻辑到共识到事务（读者推导） |
| **唐伟志** | 工程 | 工程上时间戳贯穿共识与事务——从 Proposal ID 到 TSO 到 TrueTime |
| **陈东明** | 案例 | Spanner / TiDB / CockroachDB 展示了时间戳在工业系统中的层级应用 |

### 9.7 本节小结

时间与共识的关系：
- **共识算法用逻辑时钟**：Proposal ID / term / zxid 都是 Lamport 时钟的实例
- **时间戳保证共识安全性**：新提案覆盖旧提案
- **共识是分布式事务的基础设施**：跨节点强一致事务依赖共识
- **时间的层级**：物理 → 逻辑 → 混合 → 共识 → 事务 → 业务

### 10.2 核心论断

**论断 1：物理时钟不可靠是分布式系统的根本难题**
- 时钟漂移 + NTP 精度限制 + 时钟回拨

- Lamport 时钟：单向因果序
- 向量时钟：精确并发检测

**论断 3：TrueTime 是最接近"全局真实时间"的方案**

- 外部一致性 = 线性一致 + 真实时间顺序
- 成本高，只有 Google 等巨头能负担

**论断 4：时间戳贯穿共识与事务**
- 共识算法用逻辑时钟（Proposal ID / term / zxid）
- 分布式事务用时间戳实现快照隔离（Percolator / Spanner）
- 时间是分布式系统的"暗线"
