## 1. 为什么需要"系统模型"？

### 1.1 直觉认知的局限

第 1 章建立了这样的直觉：分布式系统有"部分失败 + 网络不可靠 + 并发不确定 + 异构故障"4 大难题。这个直觉对建立认知框架有效，但**无法支撑严格证明**。

考虑以下问题：

| 问题 | 直觉回答 | 严格回答需要什么？ |
|------|---------|------------------|
| FLP 说"异步网络不可能达成共识" | 什么是"异步"？ | 通信模型的形式化定义 |
| Raft 容忍"少数节点失败" | 什么是"失败"？ | 故障模型的形式化定义 |
| Paxos 需要"多数派" | "多数派"在什么前提下成立？ | 故障模型 + 通信模型的组合 |
| Lamport 时钟解决"顺序问题" | 为什么不用物理时钟？ | 时间模型的不可能性 |

直觉说"网络可能丢包"，但 FLP 定理的证明需要"消息延迟无上界"——这两者**不等价**。TCP 解决了丢包，但**没有解决延迟无上界**。如果对模型的细节理解不到位，就会误以为"用了 TCP 就不会触发 FLP"——这是工程上常见的概念错误。

### 1.2 系统模型的定义

**系统模型（System Model）**是对分布式系统关键属性的形式化抽象，包括：

1. **通信模型**：消息如何传输？延迟有无上界？
2. **故障模型**：节点如何失败？是否能恢复？
3. **时间模型**：如何度量时间？物理时钟还是逻辑时钟？

这三个维度的组合，决定了分布式系统能做什么、不能做什么。**所有分布式算法都有其依赖的系统模型**——离开模型谈算法是无意义的。

### 1.3 模型与工程的关系

系统模型不是纯学术抽象，它**直接对应工程决策**。这里先给出一个直观映射（具体系统在第 7 节展开）：

| 模型选择 | 工程含义 | 典型场景（具体系统第 7 节展开） |
|---------|---------|-------------------------------|
| 异步 + Crash-recovery | 必须用 Raft / Paxos | 强一致 KV 存储 |
| 部分同步 + Crash-recovery | 可以做超时检测 | 协调服务 |
| 异步 + Byzantine 故障 | 必须用 PBFT / PoW | 公链 / 联盟链 |
| 同步 + Crash-stop | 可以做简单多数派 | 受控环境集群（理论假设） |

**真实系统很少完全符合某个理想模型**——工程上往往是"假设部分同步 + 实际异步 + 心跳超时检测"的混合体。但理解理想模型是理解工程妥协的前提。第 7 节会具体分析 etcd / ZooKeeper / Nacos / Spanner / Bitcoin 各自的模型选择。

### 1.4 本章的学习路径

```

```

## 2. 通信模型

### 2.1 通信模型的核心问题

通信模型回答一个问题：**消息从发送到接收，需要多久？**

这个看似简单的问题，在分布式系统中有三种不同的答案，对应三种通信模型：

| 模型 | 消息延迟 | 是否能区分"慢"与"崩溃" |
|------|---------|----------------------|
| 同步（Synchronous） | 有已知上界 | 能 |
| 异步（Asynchronous） | 无上界 | 不能 |
| 部分同步（Partial Synchronous） | 存在上界但未知 / 仅在某段未知 | 部分能 |

### 2.2 同步通信模型

**定义**：消息延迟有**已知上界** `Δ`。

```
发送方 → 消息 → 接收方
        |←— ≤ Δ —→|
```

**关键推论**：
- 如果发送后 `Δ` 时间内未收到响应，可以**确定**对方崩溃（或消息丢失）
- 故障检测是**确定性的**：超时 = 崩溃
- 共识算法可以基于"超时即失败"做简单设计

**真实世界是否存在？** 严格意义上，Internet 不是同步网络——光速、路由器队列、TCP 重传都引入无界延迟。但**某些受控环境可以近似为同步**：
- 同一机柜内的 InfiniBand 网络（μs 级延迟，抖动可控）
- 专用链路连接的数据中心（延迟 < 1ms，抖动 < 100μs）
- 单主机内的进程间通信（共享内存 / Unix Socket）

**同步模型下的共识**：在同步网络中，共识问题是**可解的**——可以通过简单的轮询 + 超时检测实现。例如 2 个节点共识：节点 1 发送值，等待 Δ，若收到 ACK 则决定，否则决定默认值。这个简单算法在异步网络下不成立（无法判断 ACK 是否会到）。

**跨书视角对照**：
- **江峰视角**：同步模型是分布式理论的"理想实验场"——很多算法在同步模型下可证明，但工程上无法直接落地
- **唐伟志视角**：实际工程中"同机房集群"常被近似为同步模型（超时设为 100ms），但这是工程妥协，理论上仍属部分同步

### 2.3 异步通信模型

**定义**：消息延迟**无上界**——消息可以任意延迟，但最终会到达（如果接收方活着）。

```
发送方 → 消息 → 接收方
        |←— ∞ —→|（无上界）
```

**关键推论**：
- **无法区分"慢"与"崩溃"**——一个延迟 10 秒的响应，可能是节点处理慢，也可能是节点崩溃了消息永远不会到
- 故障检测是**不可能确定性的**——任何超时阈值都可能误判
- 共识算法必须**不依赖超时判断崩溃**

**真实世界符合吗？** Internet 严格符合异步模型。虽然实际网络通常有延迟上界（同机房 < 10ms，跨地域 < 1s），但**没有已知的、确定的、所有情况下都成立的上界**——BGP 路由抖动、TCP 重传退避、网络分区都可能让延迟飙升到任意大。

**FLP 不可能性的前提**：异步网络 + 即使一个节点可能崩溃 → 不存在确定性共识算法。这是第 5 节的核心定理。

**异步模型下的"故障检测器"**：由于无法做确定性检测，工程上引入**非完美故障检测器**（Chandra-Toueg 1996）：
- **Completeness**：崩溃的节点最终会被怀疑
- **Accuracy**：活着的节点不被永远怀疑（弱保证）

故障检测器分两类：
- **Perfect Failure Detector**（P）：崩溃立即被检测，活着永不被怀疑（同步模型下可实现）
- **Eventually Perfect Failure Detector**（◇P）：最终崩溃被检测，最终活着不被怀疑（部分同步下可实现）

**真实系统实践**：etcd / ZooKeeper / Nacos 的心跳超时机制，本质上是 ◇P 检测器——理论上可能误判，但"最终"会正确。

### 2.4 部分同步通信模型

**定义**（Dwork-Lynch-Stockmeyer 1988）：存在消息延迟上界 `Δ` 和节点处理速度下界，但**这两个参数未知**；或者**系统在某个未知时刻 `T` 之后变为同步**（在 `T` 之前是异步的）。

```
时间轴：
  异步期（无上界）   |    同步期（有上界 Δ）
─────────────────────├─────────────────────→
                     T（未知）
```

**关键推论**：
- 实际工程中**最贴近真实**的模型
- 算法可以依赖"最终同步"假设，但不能依赖具体上界值
- 共识算法**可解**，但有 liveness 限制（在异步期可能卡住，最终同步后恢复）

**为什么部分同步是工程主流？**
- 真实网络**大多数时候**延迟可控（同机房 < 10ms）
- **偶尔**出现网络抖动 / 分区，延迟飙升
- 工程上用"心跳 + 超时 + 重试"应对偶发抖动
- 算法设计为"在网络正常时保证 liveness，在网络异常时保证 safety"

**Raft / Paxos 的模型假设**：两者都假设**部分同步网络**——这正是它们能绕过 FLP 不可能性的关键。FLP 说"纯异步下不可能"，部分同步提供了一个"逃逸通道"——只要某个时刻网络恢复正常，算法就能继续推进。

### 2.5 通信模型对比

| 维度 | 同步 | 异步 | 部分同步 |
|------|------|------|---------|
| 延迟上界 | 已知 Δ | 无 | 存在但未知 |
| 故障检测 | 确定性 | 不可能 | 概率性 / 最终 |
| 共识可解性 | 可解 | 不可解（FLP） | 可解（最终 liveness） |
| 真实匹配度 | 受控环境 | Internet 严格符合 | Internet 工程近似 |
| 典型算法 | 简单轮询 | 理论分析 | Raft / Paxos / ZAB |

### 2.6 通信模型与 TCP 的关系（常见误解）

**误解**：TCP 提供可靠传输，所以"用了 TCP 就不是异步网络"。

**正解**：
- TCP 解决**同一连接内**的丢失 / 乱序 / 重复（通过序列号 + 重传）
- TCP **没有解决**延迟上界——重传退避会让延迟指数增长，最终 RTO（Linux 初始约 200ms-1s，重传后退避可达分钟级）后断连
- TCP 断连后，应用层无法判断对方是崩溃还是网络分区
- **结论**：用了 TCP 的系统仍然是异步网络（在 FLP 意义下）

**真实案例**：MySQL 主从复制基于 TCP，但主库宕机时从库无法立即感知——必须靠应用层心跳（MGR 的 `group_replication_health_check_interval`）。这印证了"TCP 不解决故障检测"。

### 2.7 本节小结

通信模型决定了"故障检测的能力上限"：
- 同步模型：可以确定性检测故障 → 共识可解
- 异步模型：无法检测故障 → 共识不可解（FLP）
- 部分同步模型：最终可检测故障 → 共识可解（最终 liveness）

**真实分布式系统按部分同步模型设计**——这是 Paxos / Raft / ZAB 共同的隐含假设。下一节讨论"故障"本身的形式化。

## 3. 故障模型

### 3.1 故障模型的核心问题

故障模型回答一个问题：**节点会以什么方式失败？**

不同的失败方式，决定了不同的算法设计：
- 节点崩溃后永不恢复 → 简单但严格
- 节点崩溃后恢复，但状态丢失 → 需要持久化日志
- 节点活着但发送错误消息 → 需要签名验证
- 节点恶意篡改消息 → 需要拜占庭容错

### 3.2 故障模型分类

按"故障严重程度"递增排序：

| 模型 | 描述 | 算法要求 | 典型场景 |
|------|------|---------|---------|
| Crash-stop | 节点崩溃后**永不恢复** | 最简单 | 理论分析 |
| Crash-recovery | 节点崩溃后**可恢复**，状态可能丢失 | 需持久化日志 | Paxos / Raft |
| Omission | 节点**活着但消息丢失**（接收 / 发送 omission） | 需重传机制 | 网络分区 |
| Byzantine | 节点**恶意行为**（篡改 / 伪造消息） | 需签名 + 3f+1 节点 | 区块链 |

### 3.3 Crash-stop 故障模型

**定义**：节点要么正常运行，要么崩溃后**永久消失**（不再恢复）。

```
节点状态：
  正常 ──崩溃──→ 永久退出
```

**特点**：
- 最简单的故障模型
- 崩溃后状态不可恢复——等价于"节点从系统中删除"
- 算法设计无需考虑"恢复后如何同步状态"

**真实匹配度**：严格意义上很少有真实系统符合 Crash-stop——节点通常可重启。但**某些场景近似符合**：
- 容器化部署：Pod 故障被 K8s 重调度，新 Pod 是"新节点"（IP / 卷可能不同）
- 无状态服务：节点故障后新启动，状态从其他节点拉取

**算法意义**：Crash-stop 是**理论分析的标准假设**——FLP 定理、CAP 定理的证明都基于 Crash-stop。但工程上的共识算法（Paxos / Raft）实际是 Crash-recovery 模型。

### 3.4 Crash-recovery 故障模型

**定义**：节点崩溃后**可以恢复**，但内存状态可能丢失，只有**持久化存储**的数据保留。

```
节点状态：
  正常 ──崩溃──→ 故障 ──恢复──→ 重新加入（需同步状态）
```

**关键问题**：
- 内存中的状态（如 Raft 的 `currentTerm`、`votedFor`）会丢失
- 必须把关键状态持久化到磁盘（WAL）
- 恢复后需要从其他节点同步缺失的状态

**Paxos / Raft 的处理**：
- **Paxos**：每个 Acceptor 必须持久化 `highest_promised` 和 `highest_accepted`，恢复后从磁盘读取
- **Raft**：每个节点必须持久化 `currentTerm`、`votedFor`、`log[]`，恢复后从磁盘读取并补齐未复制的日志
- **ZAB**：每个 ZooKeeper 节点持久化事务日志 + 数据快照

**真实案例（etcd）**：
- etcd 数据目录：`/var/lib/etcd/`，包含 `member/wal/`（WAL 日志）和 `member/snap/`（快照）
- 节点重启时先 replay WAL，再从 leader 拉取缺失的日志
- 如果 WAL 损坏，节点无法重新加入集群（必须手动清除数据后重新同步）

**Crash-recovery 的子分类**：
- **稳定存储**（Stable Storage）：崩溃后数据不丢失的存储——理论上的理想模型
- **不稳定存储**（Volatile Storage）：崩溃后数据丢失的存储——内存
- **真实磁盘**：介于两者之间——磁盘故障率约 0.5-2% / 年，RAID / 副本可降低丢失概率

### 3.5 Omission 故障模型

**定义**：节点**活着但消息丢失**，分为：
- **发送 omission**：节点发送消息但消息从未到达（如发送方缓冲区溢出）
- **接收 omission**：消息到达但节点未处理（如接收方缓冲区溢出）

```
节点 A（活着）──消息──✗──节点 B（活着）
                     消息丢失
```

**特点**：
- 节点本身正常，故障在通信链路
- 等价于"网络分区"——某些节点间能通信，某些不能
- TCP 部分缓解 omission（重传），但极端情况下仍会丢消息（缓冲区满 + 连接断开）

**算法应对**：
- **重传 + 幂等**：消息丢失则重发，接收方幂等处理
- **超时检测**：长时间未收到响应则换副本
- **Quorum**：写入多数派即可确认，少数派丢失不影响

**真实案例（Kafka ISR）**：Kafka 的 ISR（In-Sync Replicas）机制处理 follower 的 omission——follower 同步落后超过 `replica.lag.time.max.ms`（默认 10s）则被移出 ISR，leader 只等 ISR 内的 ACK。

### 3.6 Byzantine 故障模型

**定义**：节点可以**任意行为**——包括崩溃、消息丢失、消息篡改、消息伪造、恶意攻击。

```
节点 A → "值=1" → 节点 B
节点 A → "值=2" → 节点 C   （A 对不同节点发不同值，恶意行为）
```

**特点**：
- 最严格的故障模型
- 节点不可信——消息必须签名验证
- 容错需要 **3f+1** 个节点（f 为拜占庭节点数）——下面给出推导
- 性能代价大——PBFT（Practical Byzantine Fault Tolerance）需要 O(n²) 通信

**3f+1 的数学推导**（对比 Crash 模型的 2f+1）：

Crash 模型下：节点要么活着要么崩溃，崩溃节点不发消息。要保证 quorum 交集：`N - f ≥ ⌈(N+1)/2⌉`，解得 `N ≥ 2f+1`。

Byzantine 模型下：f 个拜占庭节点可以**发任意消息**（包括虚假消息），但**不能阻止诚实节点之间的通信**。要保证：
1. **Quorum 交集**：两个 quorum 必须在**诚实节点**上有交集（拜占庭节点可以撒谎，不能作为"已知决定值"的证人）
2. **数学推导**：
   - 总节点数 N = 诚实节点 + 拜占庭节点 = (N-f) + f
   - 要保证 quorum 在诚实节点上有交集，需要诚实节点的 quorum 大小 > (N-f)/2
   - quorum 大小通常为 ⌈(N+1)/2⌉（多数派）
   - 交集大小 ≥ 2·⌈(N+1)/2⌉ - N = 1（至少 1 个节点）
   - 但这 1 个交集节点可能是拜占庭——必须保证交集内**至少 1 个诚实节点**
   - 即：交集大小 > f，即 `2·⌈(N+1)/2⌉ - N > f`
   - 当 N = 3f+1 时：`2·⌈(3f+2)/2⌉ - (3f+1) = 2(f+1) - (3f+1) = f+1 > f` ✓
   - 当 N = 3f 时：`2·⌈(3f+1)/2⌉ - 3f = 2(f+1) - 3f = 2-f`，当 f≥2 时不保证 > f ✗
- **结论**：Byzantine 容错需要 `N ≥ 3f+1`，比 Crash 多 1f 个节点（用于"抵消"拜占庭节点的撒谎能力）

**算法应对**：
- **PBFT**（Castro-Liskov 1999）：1999 年提出，第一个实用的拜占庭容错算法，7 个节点容忍 2 个拜占庭节点
- **PoW**（Proof of Work）：比特币共识，通过算力证明防女巫攻击
- **PoS**（Proof of Stake）：以太坊 2.0 共识，通过权益证明

**为什么 Paxos / Raft 不能容错 Byzantine？**
- Paxos / Raft 假设节点是"诚实的"——消息内容真实
- 拜占庭节点可以伪造 leader 消息，导致多主同时接受写入
- 拜占庭节点可以发送不同值给不同节点，破坏 majority 假设
- **结论**：在企业内网（信任域）可用 Paxos / Raft；跨信任域（如公链）必须用 PBFT / PoW

**真实案例**：
- **Bitcoin**：假设网络中 51% 算力是诚实的（PoW 共识）
- **Ethereum 2.0**：假设 2/3 验证者诚实（PoS + Casper FFG）
- **Hyperledger Fabric**：企业联盟链，用 PBFT 变种

### 3.7 故障模型对比

| 模型 | 节点数要求（容忍 f） | 典型算法通信复杂度 | 信任假设 | 典型算法 |
|------|---------------------|----------|---------|---------|
| Crash-stop | 2f+1 | O(n) | 节点诚实 | 理论分析 |
| Crash-recovery | 2f+1 | O(n) | 节点诚实 | Paxos / Raft / ZAB |
| Omission | 2f+1 | O(n) | 节点诚实 | 同上 + 重传 |
| Byzantine | 3f+1 | O(n²) | 节点不可信 | PBFT / PoW / PoS |

> **注**：通信复杂度是**算法**的属性，不是**模型**的属性——同一模型下可以有不同复杂度的算法（例如 Crash-recovery 模型下，Paxos 是 O(n)，但某些变种可达 O(n²)）。表中列出的是该模型下**典型工业算法**的复杂度。

### 3.8 故障模型与共识算法的对应

```
企业内网（信任域）           跨信任域（公链 / 联盟链）
       ↓                            ↓
Crash-recovery 模型          Byzantine 模型
       ↓                            ↓
   2f+1 节点                  3f+1 节点
       ↓                            ↓
Paxos / Raft / ZAB         PBFT / PoW / PoS
```

**跨书视角对照**：
- **江峰视角**：故障模型是分布式算法形式化的起点——4 种模型对应 4 类算法，每种算法的可证明性质都依赖其模型假设
- **释慧利视角**：Paxos / Raft / ZAB 都基于 Crash-recovery，FLP 定理的"一个节点崩溃"指 Crash-stop（更弱假设，结论更强）
- **唐伟志视角**：工程上的"节点宕机重启"是 Crash-recovery，需要 WAL + 快照机制
- **陈东明视角**：GFS / BigTable / Spanner 都假设 Crash-recovery；Bitcoin / Ethereum 假设 Byzantine

### 3.9 本节小结

故障模型决定了"算法需要应对什么样的失败"：
- Crash-stop：最简单，理论分析用
- Crash-recovery：工程主流，需 WAL 持久化
- Omission：网络层故障，需重传 + Quorum
- Byzantine：恶意行为，需 3f+1 + 签名

**真实企业系统按 Crash-recovery 模型设计**——Paxos / Raft / ZAB 共同假设。下一节讨论"时间"维度。

## 4. 时间模型

### 4.1 时间模型的核心问题

时间模型回答一个问题：**如何度量事件的顺序？**

直觉上，物理时钟就够了——每个事件打一个时间戳，按时间戳排序。但第 1 章已经指出**物理时钟不可靠**。本节深入分析"为什么物理时钟不可靠"以及"逻辑时钟如何解决"。

### 4.2 物理时钟（Physical Clock）

**定义**：每个节点维护一个物理时钟 `T_i`（基于石英晶振），通过 NTP 与标准时间同步。

**根本问题 1：时钟漂移（Clock Drift）**

石英晶振的频率受温度、电压、老化影响，存在偏差：
- 漂移率（drift rate）：典型 1ppm（百万分之一），即每秒漂移 1μs
- 高质量晶振：0.1ppm
- 服务器级晶振：0.01-1ppm

按 1ppm 计算：
- 每天 86,400 × 1e-6 = 86ms
- 每月约 2.6 秒
- 每年约 31 秒

**根本问题 2：NTP 同步精度有限**

| 部署 | NTP 同步精度 |
|------|-------------|
| 同机房（局域网 NTP） | 1-10ms |
| 跨机房（城域网 NTP） | 10-100ms |
| 跨地域（互联网 NTP） | 100ms-1s |

NTP 是**概率性同步**——不能保证所有节点时钟完全一致，只能保证差异在某个范围内。NTP 协议本身有 `offset` 和 `delay` 两个字段，前者表示与参考时钟的偏差，后者表示往返延迟。

**根本问题 3：时钟回拨（Clock Skew）**

NTP 校时如果发现本地时钟快了，会**逐步放慢**（slew）；如果发现慢了，会**逐步加快**。但如果偏差过大（如 > 128ms），NTP 会**直接跳变**（step）——这就是时钟回拨。

时钟回拨的工程影响：
- Snowflake ID 冲突（如第 1 章所述）
- 数据库时间戳乱序（先写入的数据时间戳可能更晚）
- 缓存 TTL 失效（基于时间戳的过期判断错误）
- 分布式锁租约失效（租约基于时钟，回拨导致租约延长）

**生产应对**：
- **chrony 替代 ntpd**：更平滑的同步，更小的跳变概率
- **逻辑时钟替代物理时钟**：因果关系不依赖物理时间
- **混合时钟**：物理时钟 + 逻辑计数器（如 Google TrueTime、HLC）

### 4.3 逻辑时钟（Logical Clock）

**动机**：既然物理时钟不可靠，能否设计一种**不依赖物理时间**的时钟，只关心事件的"因果关系"？

**Lamport 时钟**（Lamport 1978）：

每个节点维护一个单调递增的计数器 `C_i`，规则如下：
1. **本地事件**：`C_i = C_i + 1`
2. **发送消息**：`C_i = C_i + 1`，发送 `(message, C_i)`
3. **接收消息**：`C_i = max(C_i, C_msg) + 1`

**关键性质**：
- 如果事件 A 因果先于事件 B（A → B），则 `C(A) < C(B)`
- **但反之不成立**：`C(A) < C(B)` 不一定意味着 A → B（可能是并发事件）

**因果关系（Happened-Before）**：
- 同一节点内：A 先于 B 发生，则 A → B
- 跨节点：A 发送消息，B 接收该消息，则 A → B
- 传递性：A → B 且 B → C，则 A → C
- 否则：A 与 B **并发**（concurrent），记作 A || B

**Lamport 时钟的局限**：
- 只能检测"因果顺序"，不能检测"并发"
- `C(A) < C(B)` 时无法判断是 A → B 还是 A || B

**真实应用**：
- **数据库 MVCC**：事务 ID 类似 Lamport 时钟，单调递增
- **Kafka Offset**：分区内的 offset 是 Lamport 时钟的实例
- **Redis 主从复制**：replication offset 是 Lamport 时钟

### 4.4 向量时钟（Vector Clock）

**动机**：Lamport 时钟无法判断"并发"，能否设计一种时钟**精确刻画因果关系**？

**向量时钟**（Mattern 1989, Fidge 1988）：

每个节点维护一个向量 `V_i = [c_1, c_2, ..., c_n]`，其中 `c_j` 表示节点 i 看到的节点 j 的事件数。规则：
1. **本地事件**：`V_i[i] = V_i[i] + 1`
2. **发送消息**：发送 `(message, V_i)`
3. **接收消息**：`V_i[j] = max(V_i[j], V_msg[j])` for all j，然后 `V_i[i] = V_i[i] + 1`

**因果关系判断**：
- `V(A) < V(B)` 当且仅当 V(A) 的每个分量都 ≤ V(B) 的对应分量，且至少一个分量严格小于
- `V(A) || V(B)` 当且仅当既非 `V(A) < V(B)` 也非 `V(B) < V(A)`

**真实应用**：
- **Dynamo**（Amazon 2007 论文中的内部系统，非 AWS 对外的 DynamoDB 服务）：使用向量时钟检测写冲突
- **Riak**：向量时钟 + siblings（保留冲突版本，让客户端解决）
- **Cassandra**：早期曾探讨向量时钟，实际采用 **LWW（Last-Write-Wins）基于时间戳** + Merkle Tree 反熵（节省向量时钟的 O(n) 空间开销，代价是丢失因果信息，冲突时直接取最新时间戳）
- **Git**：commit 的 parent 关系本质是向量时钟

**向量时钟的局限**：
- 节点数 N 增大时，向量大小线性增长（O(N) 空间）
- 节点动态加入 / 退出时，向量管理复杂
- 客户端需要理解向量时钟才能解决冲突

### 4.5 混合时钟（Hybrid Clock）

**动机**：物理时钟给"时间直觉"但不可靠，逻辑时钟可靠但无"时间意义"。能否两者结合？

**HLC（Hybrid Logical Clock，Kulkarni 2014）**：

每个节点维护 `(physical_ms, logical_count)`：
1. 本地事件：`physical_ms = max(physical_ms, system_clock())`；如果没变，`logical_count++`；否则 `logical_count = 0`
2. 发送消息：附带 HLC
3. 接收消息：`physical_ms = max(physical_ms, system_clock(), msg.physical_ms)`；`logical_count` 同步

**特性**：
- 保留 Lamport 时钟的因果性（A → B 则 HLC(A) < HLC(B)）
- 保留物理时钟的时间直觉（HLC 接近真实时间）
- 不受时钟回拨影响（logical_count 兜底）

**Google TrueTime**（Spanner 用）：
- 每个数据中心配 GPS + 原子钟
- `TT.now()` 返回 `[earliest, latest]` 区间，保证真实时间在此区间内
- 提交时等待 `TT.after(TT.now())` 为真（即区间完全过去），保证全局顺序
- 典型等待时间 7ms（Commit Wait）

**真实应用**：
- **CockroachDB**：HLC 实现，跨地域强一致
- **MongoDB**：混合时钟（cluster time + counter）
- **Spanner**：TrueTime，全球一致

### 4.6 时间模型对比

| 时钟 | 因果关系 | 时间直觉 | 空间复杂度 | 真实应用 |
|------|---------|---------|----------|---------|
| 物理时钟 | ✗（不可靠） | ✓ | O(1) | 日志时间戳 |
| Lamport 时钟 | ✓（单向） | ✗ | O(1) | Kafka offset / Redis replid |
| 向量时钟 | ✓（精确） | ✗ | O(n) | Dynamo / Riak |
| HLC | ✓（单向） | ✓ | O(1) | CockroachDB |
| TrueTime | ✓（区间） | ✓ | O(1) | Spanner |

**跨书视角对照**（4 本书对"时间模型"的不同侧重）：

### 4.7 本节小结

时间模型决定了"事件顺序的度量能力"：
- 物理时钟：不可靠，但工程易用
- Lamport 时钟：因果单向，无时间直觉
- 向量时钟：因果精确，空间开销大
- HLC / TrueTime：混合，兼顾因果与时间

**真实分布式系统按需求选择**——CP 系统用 HLC / TrueTime，AP 系统用向量时钟检测冲突。下一节将通信模型 + 故障模型组合，推导出不可能性定理。

## 5. 同步性假设与不可能性定理

### 5.1 本节的核心问题

第 2-4 节定义了三个维度的系统模型。本节把它们组合起来，推导出分布式系统的两个**不可能性定理**：
- **FLP 不可能性**（异步网络 + Crash-stop → 共识不可解）
- **CAP 不可能性**（分区容忍 + 一致性 + 可用性不可兼得）

理解这两个定理的形式化前提，才能理解 Paxos / Raft 是如何"绕过"它们的。

### 5.2 FLP 不可能性定理

**陈述**（Fischer-Lynch-Paterson 1985）：

> 在**异步网络**中，如果存在**一个节点可能崩溃**（Crash-stop 模型），则不存在**确定性**共识算法能同时保证 **safety** 和 **liveness**。

**形式化前提**：
- 网络：异步（消息延迟无上界）
- 故障：Crash-stop（节点崩溃后不恢复），最多 1 个节点崩溃
- 算法：确定性（无随机性）
- 共识目标（此处为证明 FLP 给出临时定义，第 6 节再严格展开）：
  - **Safety**：所有非崩溃节点决定的值必须一致，且被某个节点提议过
  - **Liveness**：所有非崩溃节点最终能决定一个值

> **形式化预备概念**：
> - **状态（state）**：算法在某个时刻的全体节点状态 + 通道内消息集合
> - **决策（decision）**：节点确定一个值（0 或 1）后进入 decided 状态
> - **0-valent 状态**：从此状态出发，**所有**可能的执行都最终决策为 0
> - **1-valent 状态**：从此状态出发，**所有**可能的执行都最终决策为 1
> - **单价状态（univalent）**：要么 0-valent 要么 1-valent（即结果已注定，只是还没宣布）
> - **二价状态（bivalent）**：既非 0-valent 也非 1-valent（存在两条执行路径分别决策为 0 和 1，结果未定）
> - **关键事件（critical event）**：使状态从二价变为单价的消息传递事件

**证明核心思路**（反证法 + 三步引理）：

假设存在确定性共识算法 A 满足 safety + liveness。构造一条无穷执行，使其永不决策，与 liveness 矛盾。

1. **存在初始二价状态**（引理 1：存在 bivalent initial state）
2. **从任何二价状态，存在无穷执行序列保持二价**（引理 2：保持 bivalence）
3. **如果算法保证 liveness，则不存在无穷二价执行**——矛盾

**关键引理 1**：存在初始二价状态
- 假设所有初始状态都是单价的（0-valent 或 1-valent）
- 构造两个初始状态 S⁰（所有节点输入 0）和 S¹（所有节点输入 1）
- 在 S⁰ 和 S¹ 之间，按节点编号顺序逐个把输入从 0 改为 1，必存在相邻两个状态 S^i（0-valent）和 S^(i+1)（1-valent），它们只差**第 i+1 个节点的输入**（0 vs 1）
- 考虑第 i+1 个节点在执行初期就崩溃的执行：算法无法区分 S^i 和 S^(i+1)（崩溃节点从未发出消息），但两者决策结果不同——矛盾
- **结论**：必存在初始二价状态

**关键引理 2**：从二价状态可以保持二价（核心引理）
- 设 C 为二价状态，e 为下一个待传递的消息（如 `p→q`）
- 设 `e(C)` 为 e 传递后到达的状态
- 假设 `e(C)` 是单价的（设为 0-valent），即 e 是关键事件
- 构造矛盾：考虑另一条执行 σ，其中 e 的传递被延迟（异步网络允许任意延迟）。σ 中先传递其他消息，到达状态 D
  - 若 D 是 0-valent：则 `e(D)` 也是 0-valent（与假设一致）
  - 若 D 是 1-valent：则 `e(D)` 也应是 1-valent（D 已决定 1）——但 `e(C)` 是 0-valent，从 C 出发有执行到达 D（1-valent）并应用 e，这与"e(C) 是 0-valent"矛盾
  - 若 D 是 bivalent：则 `e(D)` 也可能是 bivalent，递归可构造无穷二价执行
- **结论**：存在某个执行，使状态保持二价

**关键引理 3**：保持二价的无穷执行
- 反复应用引理 2：从初始二价状态 C₀，选择能保持二价的事件 e₁，到达 C₁；再从 C₁ 选择 e₂，到达 C₂；...
- 构造出无穷执行 C₀ → C₁ → C₂ → ...，状态始终保持二价，永不决策
- 与 liveness（所有非崩溃节点最终决策）矛盾
- **结论**：原假设错误，不存在确定性共识算法同时保证 safety + liveness

**FLP 的工程意义**：
- **理论上**：纯异步网络下，确定性共识算法不可能完美
- **工程上**：通过**两条独立路径**绕过 FLP 的"确定性"前提：
  - **路径 1（部分同步假设）**：依赖"网络最终恢复正常"（DLS 1988），多数工业级共识算法走这条路径
  - **路径 2（随机化）**：引入随机性避免活锁（Ben-Or 1983），Raft 的随机选举超时是此路径的轻量化应用

**Raft / Paxos 如何绕过 FLP？**
- **Paxos**：通过 leader 提议 + 多数派接受，**理论上**在极端异步情况下可能活锁（多个 Proposer 互相打断）；**主要通过部分同步假设绕过 FLP**（leader 选举不必然依赖随机化，工程实现可选随机退避降低活锁概率）
- **Raft**：通过 leader + **随机选举超时**（路径 2 的轻量化应用），**理论上**可能卡住（多个 candidate 同时超时），但随机超时使其概率极低；同时依赖部分同步假设保证 liveness
- **本质**：两者都依赖**部分同步假设**（路径 1）作为主要绕过手段；Raft 额外用了**随机化**（路径 2）优化选举；纯异步随机化算法（Ben-Or）不依赖部分同步但性能较差，工程上很少采用

**跨书视角对照**：
- **江峰视角**：FLP 是分布式算法的"基础不可能性定理"——它定义了共识算法的能力边界，所有共识算法都必须以某种方式"绕过"它（部分同步 / 随机化 / 故障检测器）
- **释慧利视角**：第 10 章详解 FLP 证明，释慧利认为 FLP 是"理解 Paxos/Raft 设计动因的钥匙"
- **唐伟志视角**：FLP 是"理论警告"——工程上用部分同步 + 心跳绕过，但不能假装它不存在

### 5.3 CAP 不可能性定理

**陈述**（Gilbert-Lynch 2002）：

> 在**可能发生网络分区**的异步网络中，不存在一个分布式系统能同时保证 **C**（线性一致性）、**A**（可用性）、**P**（分区容忍）。

**形式化前提**：
- 网络：异步（消息延迟无上界）
- 故障：网络分区（Omission 故障）
- 一致性：**线性一致性（Linearizability）**——所有操作看起来在某个真实时间瞬间原子地完成，且操作的全局顺序与真实时间一致（即如果操作 A 在真实时间上先于操作 B 完成，则在全局顺序中 A 必须先于 B）。**注意**：线性一致性比顺序一致性更强——顺序一致性只要求"所有节点看到的操作顺序相同"，但不要求与真实时间一致；线性一致性额外要求与真实时间一致
- 可用性：每个非失败节点的请求在有限时间内返回非错误响应
- 分区容忍：网络分区时系统仍能运作

**证明核心**（反证法）：

1. 假设系统同时保证 C、A、P
2. 节点 N1、N2 发生分区（互不可达）
3. 客户端 C1 写 N1（值 v1）：N1 必须 ACK（A）→ N1 接受 v1
4. 客户端 C2 读 N2：N2 必须 ACK（A）→ N2 返回旧值
5. 但 C 要求 N2 返回 v1 → 矛盾

**CAP 与 FLP 的关系**：
- **FLP** 关注**共识问题**：异步 + 崩溃 → 共识不可解
- **CAP** 关注**一致性 vs 可用性**：异步 + 分区 → 不能同时保证
- 两者都基于"异步网络无法区分慢与崩溃"这一根本困难

**CAP 的常见误解**（已在第 1 章纠正）：
- "三选二"是错的——P 是物理事实，必选
- "CP 系统永远强一致，AP 系统永远弱一致"是错的——只在分区时才需要权衡
- 真正的工程选择是 PACELC（6.5 节）

### 5.4 不可能性定理的工程应对

| 定理 | 形式化前提 | 工程绕过 | 典型算法 |
|------|----------|---------|---------|
| FLP | 异步 + Crash + 确定性 | 部分同步（主）+ 随机化（辅） | Paxos / Raft |
| CAP | 异步 + 分区 | 选择 CP 或 AP | etcd (CP) / Cassandra (AP) |

**关键洞察**：所有"实用"的共识算法都基于**部分同步假设**。FLP 说"纯异步不可能"，部分同步提供了一个"逃逸通道"——只要某个时刻网络恢复正常，算法就能继续推进。这是为什么 Paxos / Raft / ZAB 都有"最终 liveness"保证（最终达成共识，但不保证何时）。

### 5.5 本节小结

两个不可能性定理划定了分布式系统的能力边界：
- FLP：异步 + 崩溃 → 共识不可解（必须引入部分同步 / 随机化）
- CAP：异步 + 分区 → C 和 A 不能兼得（必须选择 CP 或 AP）

**绝大多数工业级共识算法都在这两个边界内做工程妥协**——少数纯异步随机化算法（如 Ben-Or 1983）不依赖部分同步假设，但性能较差，工程上很少采用。下一节给出共识问题的严格形式化定义。

## 6. 共识问题的形式化定义

### 6.1 本节的核心问题

第 4 章会讲 Paxos / Raft / ZAB——它们都"解决共识问题"。但**什么是共识问题**？必须形式化定义，才能讨论算法是否正确。

### 6.2 共识问题的定义

**共识问题（Consensus Problem）**：n 个节点，每个节点提议一个值 `v_i`，算法必须让所有非崩溃节点决定**同一个值** `v`，且 `v` 是某个节点提议过的。

**为什么需要这三个性质？** 通过"缺了会怎样"反推：

| 缺少的性质 | 算法行为 | 工程后果 |
|----------|---------|---------|
| 缺 Termination | 算法可能永不决定，节点一直等 | 系统卡死，请求超时 |
| 缺 Agreement | 不同节点决定不同值 | 数据不一致，账目错乱 |
| 缺 Validity | 决定一个无人提议的值（如默认 0） | "幽灵数据"，业务无意义 |

三个性质缺一不可，共同定义了"正确的共识"。

**形式化目标**（释慧利视角）：

1. **Termination（终止性 / liveness）**：每个非崩溃节点最终决定一个值
2. **Agreement（一致性 / safety）**：所有决定的节点决定同一个值
3. **Validity（有效性 / safety）**：决定的值必须是某个节点提议过的

**注意**：
- Termination 是 liveness 性质（"好事最终发生"）
- Agreement 和 Validity 是 safety 性质（"坏事永不发生"）
- FLP 说"Termination 不能保证"——所以实用算法放宽为"**最终 Termination**"（在部分同步下）

### 6.3 共识问题的变体

共识问题有多种变体，难度递增：

| 变体 | 描述 | 难度 |
|------|------|------|
| **共识（Consensus）** | 节点提议一个值，决定一个值 | 基础 |
| **原子广播（Atomic Broadcast）** | 节点广播消息，所有节点按相同顺序接收 | 等价于共识 |
| **原子提交（Atomic Commit）** | 分布式事务中所有节点 commit 或全部 abort | 比共识弱 |
| **领导者选举（Leader Election）** | 选出一个 leader | 比共识弱 |
| **互斥（Mutual Exclusion）** | 同一时刻只有一个节点进入临界区 | 比共识弱 |

**关键定理**（Chandra-Toueg 1996）：在异步 + Crash-recovery 模型下，**共识问题与原子广播问题等价**。

**为什么原子广播 = 共识？**
- 原子广播：所有节点按相同顺序接收消息
- 共识：所有节点决定同一个值
- 给每个消息分配一个序号 → 序号需要共识
- 反之，共识可以视为"广播一个值"

**真实系统对应**：
- **ZooKeeper ZAB**：原子广播协议（顺序一致 + 全序）
- **etcd Raft**：共识协议（每个日志条目是一次共识）
- **Kafka ISR**：主从复制 + leader epoch（不是严格共识，但有共识特征）

### 6.4 共识算法的分类

按"如何保证 agreement"分：

| 类型 | 思路 | 代表算法 |
|------|------|---------|
| **Quorum-based** | 多数派接受（2f+1 节点容忍 f 崩溃） | Paxos / Raft / **ZAB**（ZAB 本质也是 quorum-based，只是优化为 leader 原子广播） |
| **Randomized** | 随机化避免活锁（绕过 FLP 的"确定性"前提） | Ben-Or（1983，纯异步 + 随机化） |
| **Coordinator-based**（弱） | 协调者决定，但协调者崩溃会阻塞 | 2PC / 3PC（**注**：严格意义上 2PC 不满足共识的 termination 性质——协调者崩溃时参与者可能锁死，故 2PC 不是真正的共识算法，是原子提交协议） |

**为什么需要 majority？**
- 在 Crash-recovery 模型下，n 个节点容忍 f 个崩溃
- 要保证 agreement，必须确保**两个 quorum 有交集**——交集节点知道"已决定的值"
- 数学上：`W + R > N`，通常 `W = R = f + 1 = (N+1)/2`（多数派）

**为什么 majority 是 2f+1？**
- f 个节点崩溃，剩 N - f 个活着
- 需要多数派 ACK，即 ≥ ⌈(N+1)/2⌉
- 容忍 f 崩溃：N - f ≥ ⌈(N+1)/2⌉，即 N ≥ 2f+1

**跨书视角对照**：
- **释慧利视角**：共识的形式化定义是第 1-2 章核心，safety/liveness 区分贯穿全书
- **江峰视角**：第 8 章详解共识算法的分类与等价性，证明原子广播 = 共识
- **唐伟志视角**：第 4 章用 Go 实现简化版 Paxos / Raft，体现形式化定义
- **陈东明视角**：ZAB（ZooKeeper）/ Paxos（Spanner）/ Raft（CockroachDB）的工业实现

### 6.5 共识与状态机复制（SMR）

**状态机复制（State Machine Replication, SMR）**：把共识扩展为"按相同顺序执行一系列命令"。

```
客户端 → 命令序列 → [共识层] → 所有副本按相同顺序执行
```

**关系**：
- 共识：决定单个值
- SMR：决定一系列值的顺序（每次共识决定一个序号的值）
- SMR = 多次共识 + 全序

**真实系统对应**：
- **etcd**：每个 key 的修改是一次共识（Raft 日志条目）
- **ZooKeeper**：每个 znode 的修改是 ZAB 广播
- **TiKV**：每个 Region 的修改是 Raft 日志条目
- **Kafka**：每个分区的消息顺序（不是严格 SMR，但类似）

**SMR 的一致性保证**：
- 线性一致性（Linearizability）：每个操作看起来在某个瞬间原子地完成
- 顺序一致性（Sequential Consistency）：所有客户端看到的操作顺序相同，但不一定与真实时间一致

**Raft 与线性一致性**：
- Raft 保证日志的全序复制（顺序一致）
- 但客户端读取需要"ReadIndex"或"Lease Read"才能保证线性一致
- 否则可能从 stale leader 读到旧值

### 6.6 共识算法的设计约束

基于第 5 节的不可能性定理，共识算法必须满足：

1. **部分同步假设**：依赖网络最终恢复正常（绕过 FLP）
2. **多数派 Quorum**：`W = R = ⌈(N+1)/2⌉`（保证 agreement）
3. **Leader 串行化**：避免多 Proposer 互相打断（避免活锁）
4. **持久化日志**：崩溃后恢复状态（Crash-recovery 假设）
5. **故障检测**：心跳 + 超时（部分同步下的检测器）

**这 5 条约束就是 Paxos / Raft / ZAB 的全部设计空间**。第 4 章将具体展开每个算法如何在约束内做工程取舍。

### 6.7 本节小结

共识问题的形式化定义包含三个性质：
- **Termination**（liveness）：最终决定
- **Agreement**（safety）：决定同一个值
- **Validity**（safety）：决定某个提议的值

**共识问题与原子广播等价**，是状态机复制（SMR）的基础。第 4 章的 Paxos / Raft / ZAB 都在此框架下设计。

**下一节将上述形式化模型映射到真实系统**——看 etcd / ZooKeeper / Nacos / Spanner / Bitcoin 各自选择了什么模型假设，以及这些假设如何决定了它们的工程能力与限制。

## 7. 真实系统的模型选择

### 7.1 本节的核心问题

第 2-6 节定义了模型与定理。本节把它们映射到真实系统——每个生产级分布式系统都隐含做了了一组模型假设，理解这些假设才能正确使用它们。

### 7.2 etcd 的模型选择

**系统模型**：
- 通信：部分同步（同机房 < 10ms，跨机房 < 100ms）
- 故障：Crash-recovery（WAL 持久化）
- 时间：物理时钟 + leader lease（不依赖精确时钟）

**共识算法**：Raft

**关键设计**：
- 心跳间隔：100ms（`--heartbeat-interval`）
- 选举超时：1000ms（`--election-timeout`）
- 多数派：3 节点容忍 1 故障，5 节点容忍 2 故障
- 持久化：`/var/lib/etcd/member/wal/` + `member/snap/`
- 线性一致读：ReadIndex 或 Lease Read

**模型假设的工程体现**：
- 部分同步：选举超时 1000ms 假设"心跳间隔 + 网络抖动 < 1000ms"
- Crash-recovery：WAL 保证崩溃后状态可恢复
- 多数派：3 节点容忍 1 故障，与 Raft 的 2f+1 对应

### 7.3 ZooKeeper 的模型选择

**系统模型**：
- 通信：部分同步（`tickTime=2000ms`，`syncLimit=5` 即 10s）
- 故障：Crash-recovery（事务日志 + 数据快照）
- 时间：物理时钟 + zxid（事务 ID，包含 epoch + counter）

**共识算法**：ZAB（ZooKeeper Atomic Broadcast）

**关键设计**：
- tickTime：2000ms（基础时间单位）
- initLimit：10 × tickTime = 20s（leader 选举阶段超时）
- syncLimit：5 × tickTime = 10s（follower 与 leader 同步超时）
- 多数派：与 Raft 相同，2f+1
- 持久化：`dataDir`（快照）+ `dataLogDir`（事务日志）
- 顺序一致读：客户端顺序一致，但跨客户端不一定线性一致

**ZAB vs Raft 的差异**：
- ZAB 是**原子广播**（leader 串行化所有写）
- Raft 是**共识**（每个日志条目独立共识）
- 数学上等价，但 ZAB 更优化于"leader 广播"场景

### 7.4 Nacos 的模型选择

**系统模型**（混合）：
- 通信：部分同步
- 故障：Crash-recovery
- 时间：物理时钟

**共识算法**：**双协议**
- **Distro 协议**（AP）：临时实例，每个节点独立处理写入，异步同步
- **Raft 协议**（CP）：持久实例，多数派共识

**为什么 Nacos 选择双协议？**
- **临时实例**（如微服务注册）：服务实例频繁上下线，对延迟敏感，容忍短期不一致——AP
- **持久实例**（如配置）：变更频率低，但必须强一致——CP
- **混合模型**：把不同业务需求路由到不同协议

**模型假设的工程体现**：
- AP（Distro）：不依赖多数派，单节点即可接受写入
- CP（Raft）：依赖多数派，与 etcd 相同
- 协议路由：基于 `ephemeral` 字段

> **待 MCP 核实声明**：Nacos 1.x 中 Distro / Raft 的协议路由代码在 `naming/consistency/persistent` 与 `naming/consistency/ephemeral` 包；Nacos 2.x 引入 gRPC 长连接后实现有调整，但持久/临时实例的协议路由边界保留。后续涉及 Nacos 源码时再用 MCP 核实。

### 7.5 Spanner 的模型选择

**系统模型**：
- 通信：部分同步（跨地域 < 250ms）
- 故障：Crash-recovery
- 时间：**TrueTime**（GPS + 原子钟）

**共识算法**：Paxos（每个 Paxos 组）

**关键设计**：
- TrueTime：`TT.now()` 返回 `[earliest, latest]` 区间
- Commit Wait：提交时等待 `TT.after(TT.now())` 为真（典型 7ms）
- Paxos 组：每个分片是一个 Paxos 组，跨地域多副本
- 线性一致：跨地域读写都线性一致

**为什么 Spanner 用 TrueTime？**
- 跨地域强一致需要"全局顺序"——传统物理时钟不可靠
- TrueTime 提供有界不确定性（ε < 7ms）
- Commit Wait 等待 ε，保证两个并发事务不会分配到冲突的时间戳

**模型假设的工程体现**：
- 部分同步 + TrueTime：绕过 FLP（部分同步）+ 解决物理时钟不可靠（TrueTime）
- Paxos：每个 Paxos 组容忍 f 崩溃（2f+1 副本）
- 跨地域：副本分布在不同大陆，延迟 50-250ms

### 7.6 Bitcoin 的模型选择

**系统模型**：
- 通信：异步（P2P 网络，无延迟保证）
- 故障：**Byzantine**（节点可作恶）
- 时间：物理时钟（区块时间戳，精度 ±2 小时）

**共识算法**：**PoW**（Proof of Work）

**关键设计**：
- 算力证明：每个区块需要消耗算力（哈希碰撞）
- 最长链原则：节点接受最长合法链
- 51% 攻击：攻击者控制 51% 算力才能篡改
- 最终性概率：6 个区块确认（约 1 小时）后视为不可逆

**为什么 Bitcoin 用 PoW？**
- 公链场景，节点不可信——Byzantine 故障
- PBFT 需要 3f+1 节点，但公链节点动态加入退出，无法固定 n
- PoW 通过算力证明防女巫攻击（一个 IP 一票 → 一个算力一票）
- 牺牲性能（7 TPS）换信任假设（无需信任任何节点）

**模型假设的工程体现**：
- 异步 + Byzantine：必须用 PoW / PoS
- 概率最终性：6 区块确认后撤销概率极低——具体概率依赖攻击者算力占比 q（Satoshi 论文：q=10% 时撤销概率约 10⁻⁵，q=30% 时约 0.04%，q<10% 才能保证 6 区块后 < 10⁻⁹；当前 Bitcoin 全网算力分布 q 远低于 10%）
- 算力门槛：51% 攻击成本极高（电力 + 硬件）

### 7.7 模型选择对比

| 系统 | 通信 | 故障 | 时间 | 算法 | 一致性 |
|------|------|------|------|------|--------|
| etcd | 部分同步 | Crash-recovery | 物理 + Lease | Raft | 线性一致 |
| ZooKeeper | 部分同步 | Crash-recovery | 物理 + zxid | ZAB | 顺序一致 |
| Consul | 部分同步 | Crash-recovery | 物理 | Raft（与 etcd 同源） | 线性一致 |
| Nacos | 部分同步 | Crash-recovery | 物理 | Raft + Distro | CP/AP 切换 |
| TiKV | 部分同步 | Crash-recovery | 物理 + HLC | 多 Raft 组（每个 Region 一个） | 线性一致 |
| Spanner | 部分同步 | Crash-recovery | TrueTime | Paxos | 线性一致 |
| MongoDB 副本集 | 部分同步 | Crash-recovery | 物理 + cluster time | Raft 变种（选举）+ OpLog 主从复制 | 顺序一致（默认）/ 线性一致（读主） |
| Redis Cluster | 部分同步 | Crash-recovery | 物理 | Gossip + 主从异步复制 | 最终一致（AP） |
| Bitcoin | 异步 | Byzantine | 物理 | PoW | 概率最终 |
| Cassandra | 部分同步 | Crash-recovery | 物理 | Quorum | 最终一致 |

**关键差异点**：
- **etcd vs Consul**：同为 Raft，差异在生态——etcd 是 K8s 底座，Consul 强调服务发现 + 健康检查
- **etcd vs TiKV**：etcd 单 Raft 组，TiKV 多 Raft 组（每个 Region 独立 Raft），可扩展到 PB 级
- **MongoDB vs Redis Cluster**：MongoDB 副本集是 CP（主节点写），Redis Cluster 是 AP（分区时各分片独立写入）

### 7.8 模型选择的工程权衡

| 业务需求 | 推荐模型 | 推荐算法 | 典型系统 |
|---------|---------|---------|---------|
| 强一致 + 低延迟（同机房） | 部分同步 + Crash-recovery | Raft | etcd |
| 强一致 + 跨地域 | 部分同步 + TrueTime | Paxos | Spanner |
| 高可用 + 容忍不一致 | 部分同步 + Crash-recovery | Quorum | Cassandra |
| 公链 + 不可信节点 | 异步 + Byzantine | PoW / PoS | Bitcoin |
| 微服务注册中心 | 混合（AP 注册 + CP 配置） | Distro + Raft | Nacos |

**跨书视角对照**：
- **陈东明视角**：第 6-9 章详细分析 RabbitMQ / ZK / Spanner / CockroachDB 的模型选择与设计权衡
- **唐伟志视角**：第 7 章案例研究对比不同系统的工程取舍
- **江峰视角**：从形式化角度，所有真实系统的选择都可以归约为"通信 + 故障 + 时间"三维度组合
- **释慧利视角**：每个算法的形式化证明都基于特定模型假设，工程选择必须匹配算法的模型前提

### 7.9 本节小结

真实系统的模型选择决定了它的能力与限制：
- **etcd / ZooKeeper**：部分同步 + Crash-recovery + Raft/ZAB → 强一致 + 高可用
- **Spanner**：TrueTime + Paxos → 跨地域强一致
- **Bitcoin**：异步 + Byzantine + PoW → 不可信环境共识
- **Nacos**：混合模型 → 不同业务路由到不同协议

**模型选择没有银弹**——每个选择都是业务需求与系统能力的权衡。理解模型，才能正确选择系统。

## 8. 本章总结与后续路径

### 8.1 三维度模型回顾

| 维度 | 选项 | 工程主流 |
|------|------|---------|
| 通信 | 同步 / 异步 / 部分同步 | 部分同步 |
| 故障 | Crash-stop / Crash-recovery / Omission / Byzantine | Crash-recovery |
| 时间 | 物理 / Lamport / 向量 / HLC / TrueTime | 物理 + Lease（CP）/ HLC（跨地域）|

### 8.2 两个不可能性定理

| 定理 | 前提 | 结论 | 工程应对 |
|------|------|------|---------|
| FLP | 异步 + Crash + 确定性 | 共识不可解 | 部分同步（主）+ 随机化（辅） |
| CAP | 异步 + 分区 | C/A 不可兼得 | 选择 CP 或 AP |

### 8.3 共识问题的形式化

共识问题的三个性质：
1. **Termination**（liveness）：最终决定
2. **Agreement**（safety）：决定同一个值
3. **Validity**（safety）：决定某个提议的值

**共识问题与原子广播等价**，是状态机复制（SMR）的基础。

### 8.4 共识算法的设计约束

基于第 5 节的不可能性定理，共识算法必须：
1. 部分同步假设（绕过 FLP）
2. 多数派 Quorum（保证 agreement）
3. Leader 串行化（避免活锁）
4. 持久化日志（Crash-recovery）
5. 故障检测（心跳 + 超时）

**这 5 条约束就是 Paxos / Raft / ZAB 的设计空间**。

### 8.6 关键认知

带着以下 3 个认知进入第 3 章：

**认知 1：模型决定算法能力**
- 同步模型下共识可解，异步模型下不可解（FLP），部分同步下可解（最终 liveness）
- 离开模型谈算法是无意义的

**认知 2：不可能性定理是设计的起点，不是终点**
- FLP 和 CAP 划定了能力边界
- 所有实用算法都在边界内做工程妥协
- Paxos / Raft 不是"违反"FLP，而是"绕过"——通过部分同步假设

**认知 3：共识问题的形式化定义是后续章节的基础**
- Termination / Agreement / Validity 三性质贯穿 Paxos / Raft / ZAB 的证明
- 共识 = 原子广播 = SMR 的基础
- 第 4 章的算法推导将严格基于本章的定义
