## 1. 为什么需要分区？

### 1.1 分区 vs 复制：两个正交维度

第一篇讲了数据复制——同一份数据的多个副本。分区是另一个维度——不同数据分散到不同节点。

```
复制（纵向冗余）            分区（横向切分）
  同一数据多副本              不同数据分到不同节点
  ┌───┬───┬───┐              ┌───┬───┬───┐
  │ A │ A │ A │              │ A │ B │ C │
  │ v1│ v1│ v1│              └───┴───┴───┘
  └───┴───┴───┘
    3 副本                    3 分片
```

**两者正交组合** = 每个分片有多个副本：

```
分片 1: Leader₁ + Follower₁.₁ + Follower₁.₂
分片 2: Leader₂ + Follower₂.₁ + Follower₂.₂  
分片 3: Leader₃ + Follower₃.₁ + Follower₃.₂
```

这就是 TiKV 的多 Raft 组、Spanner 的多 Paxos 组、Cassandra 的一致性哈希 + 复制因子的基本架构。

### 1.2 分区解决的核心问题

**问题 1：单机容量上限**
- 单机存储有物理上限（唐伟志书第 1 章 1.5 节）
- 分区让数据量线性扩展到 N 台机器

**问题 2：单机吞吐瓶颈**
- 单机读写 QPS 有上限
- 分区让不同数据并行处理

**问题 3：热点**
- 某些数据访问频率远高于其他
- 合理的分区策略可以分散热点

### 1.3 分区的三要素

| 要素 | 核心问题 | 对应模式 |
|------|---------|---------|
| **分片键** | 按什么字段分区？ | Hash(user_id) / Range(timestamp) |
| **路由** | 如何找到数据在哪个分片？ | 固定映射 / 元数据服务 / 一致性哈希环 |
| **重分片** | 加节点时如何迁移数据？ | 哈希取模（全迁移）/ 一致性哈希（最小迁移）/ 虚拟槽（按槽迁移） |

## 2. 固定分区（Hash Partitioning）

### 2.1 模式概述

**核心思想**：对分片键做哈希，取模分配到固定数量的分片。

```
shard = hash(key) % N
```

**示例**：N=4 分片
```
hash("user_001") % 4 = 2 → 分片 2
hash("user_002") % 4 = 0 → 分片 0
hash("user_003") % 4 = 3 → 分片 3
hash("user_004") % 4 = 1 → 分片 1
```

### 2.2 模式的适用场景

| 适用 | 不适用 |
|------|--------|
| 写多读少 | 范围查询（相邻 key 可能在不同分片） |
| 单 key 查询为主 | 需要跨分片 JOIN |
| 分片数稳定（不常扩缩容） | 频繁扩缩容（取模重映射代价大） |

### 2.3 重分片代价：全部重映射

**问题**：N 从 4 扩到 8，几乎所有 key 都要重新映射。

```
原来 4 分片：hash(key) % 4
扩到 8 分片：hash(key) % 8

key="user_001"：hash%4=2（分片2）→ hash%8=？需要重新计算
```

**迁移量**：理论上 50% 的 key 需要迁移（N=4→8：hash%4∈{0,1,2,3} 不变占 50%，hash%4∈{4,5,6,7} 需迁移占 50%）。
- 数据量大时，迁移代价极高
- 需要离线迁移或在线双写

**何时可用**：预分片。如果提前创建足够多的分片（如 128 分片），初期每台机器负责多个分片，扩容时把部分分片迁移到新机器——分片号不变，key 映射不变。

### 2.4 生产案例

**案例 1：MySQL 早期手动分库分表**
- ShardingSphere-JDBC 配置 `sharding-algorithm: hash_mod`
- `user_id % 16` 分到 16 个库
- 扩容困难——通常做 16→32 的双写迁移

**案例 2：Vitess 的哈希分片**
- YouTube 开源的 MySQL 集群中间件
- 支持哈希分片 + 在线重分片（VReplication）
- 用 VReplication 实现增量数据迁移

**案例 3：DynamoDB 的分片（早期）**
- 早期 DynamoDB 用固定分区数
- 分区用完需联系 AWS 扩容（现已改为自适应）

### 2.5 本节小结

固定分区是最简单的分区模式：
- **核心**：`shard = hash(key) % N`，均匀分布
- **优势**：实现简单，分布均匀
- **劣势**：范围查询失效，重分片代价大
- **适用**：分片数稳定 + 单 key 查询 + 写多读少

## 3. 动态分区（Range Partitioning）

### 3.1 模式概述

**核心思想**：按键的**顺序范围**分配分片。分片数量不是预先固定的，而是根据数据量动态分裂与合并。

```
分片 1: [0, 1000)
分片 2: [1000, 2000)
分片 3: [2000, 3000)
...
```

**自动分裂**：当某个分片的数据量超过阈值（如 96MB），自动分裂为两个。

```
分片 2: [1000, 2000) → 数据量 = 100MB > 96MB
  分裂为：
  分片 2a: [1000, 1500)
  分片 2b: [1500, 2000)
```

### 3.2 模式的适用场景

| 适用 | 不适用 |
|------|--------|
| 范围查询（WHERE key BETWEEN a AND b） | 随机 key 访问（不如哈希均匀） |
| 数据量不均匀 | 热点写入（最新数据集中在一个分片） |
| 自动扩容（分片自动分裂） | 分片数需严格控制 |

### 3.3 重分片代价：局部调整

**优势**：只分裂/合并涉及的分片，不需要全量迁移。

```
从 3 分片扩到 4 分片：
  不是重映射所有 key
  只需要把分片 2 分裂为 2a + 2b
  分片 1 和 3 不受影响
```

**代价**：分裂过程中，分片可能不可用或性能下降。需要在线分裂（如 TiKV 的 Region Split）。

### 3.4 生产案例

**案例 1：TiKV 的 Region 分裂**
- TiKV 的 Region 是基于 Key Range 的动态分区
- Region 默认 96MB（历史值，5.x+ 推荐 256MB）
- Region 自动分裂，PD（Placement Driver）调度分布
- 每个 Region 是独立的 Raft 组

**案例 2：HBase 的 Region 分裂**
- HBase 的 Region 基于 RowKey Range
- Region 自动分裂（默认策略：`IncreasingToUpperBoundRegionSplitPolicy`）
- 分裂后 RegionServer 重新分配

**案例 3：CockroachDB 的 Range 分裂**
- CockroachDB 的 Range 基于 Key Range
- Range 默认 64MB（21.x+ 支持自动调整）
- Range 自动分裂 + Leaseholder 读写分离

**案例 4：MongoDB 的 Chunk 自动分裂**
- MongoDB 的 Chunk 基于 shard key 范围
- Chunk 自动分裂（5.x 之前 64MB，6.0+ 自动调整）
- Balancer 自动迁移 chunk

### 3.5 本节小结

动态分区适合范围查询 + 自动扩容：
- **核心**：按键范围分片，自动分裂/合并
- **优势**：范围查询高效，局部重分片
- **劣势**：热点写入（最新数据集中）
- **生产案例**：TiKV / HBase / CockroachDB / MongoDB

## 4. 一致性哈希（Consistent Hashing）

### 4.1 模式概述

**核心思想**：把哈希空间组织成环（0 ~ 2³²-1），节点和 key 都映射到环上，key 顺时针找到的第一个节点即为其归属。

```
       0
     /   \
  N1      N3
   |       |
  N2------N4
       2^32-1

key "user_001" → hash = 10000
  → 顺时针找到 N2（N2 的 hash = 15000）
  → key 归属 N2
```

### 4.2 模式解决的问题

**问题**：固定分区（哈希取模）在 N 变化时需要全量重映射。

**一致性哈希的解答**：N 变化时，只有相邻节点范围内的 key 需要迁移。

```
添加 N5（hash = 8000）：
  N1 的 key 范围：之前负责 (8000, 0]，现在只负责 (4000, 0]
  只有 N1 的部分 key（hash 在 4000~8000）迁移到 N5
  其他节点不受影响

移除 N2：
  N2 上的全部 key 迁移到 N4（N2 顺时针的下一个节点）
  N1、N3 不受影响
```

**迁移量**：添加/移除 1 个节点，只影响 1/N 的 key（而非全部）。

### 4.3 虚拟节点（Virtual Nodes）

**问题**：物理节点少时，哈希环分布不均。

```
3 个节点在环上：
  N1: [0, 1000)
  N2: [1000, 1500)  ← 负责范围太小
  N3: [1500, 2^32-1) ← 负责范围太大
```

**虚拟节点解决**：每个物理节点对应多个虚拟节点（如 200 个），虚拟节点均匀分布在环上。

```
物理节点 N1 → 虚拟节点 N1₁, N1₂, ..., N1₂₀₀
物理节点 N2 → 虚拟节点 N2₁, N2₂, ..., N2₂₀₀
物理节点 N3 → 虚拟节点 N3₁, N3₂, ..., N3₂₀₀

600 个虚拟节点均匀分布 → 每个物理节点负责的 key 量接近均等
```

**虚拟节点数选择**：
- 太少：分布不均
- 太多：token ring 管理开销大
- Cassandra 3.x 默认 256，4.x 改为 16（CASSANDRA-7032：大集群下 token ring 维护开销过高）

### 4.4 生产案例

**案例 1：Cassandra**
- 一致性哈希 + 虚拟节点（3.x: 256, 4.x+: 16）
- 节点加入/退出最小迁移
- 每个 key 复制到 N 个顺时针后继节点（replication factor）

**案例 2：Dynamo（Amazon 2007 论文系统）**
- 一致性哈希 + 虚拟节点
- 每个 key 复制到 N 个顺时针后继节点
- 冲突解决：向量时钟 + siblings

**案例 3：Memcached 客户端**
- twemproxy / libmemcached 在客户端用一致性哈希
- Memcached 服务端本身无分布式能力
- 虚拟节点数：通常 160（libmemcached 默认）

**案例 4：Riak**
- 一致性哈希 + 虚拟节点
- 每个 key 复制到 N 个顺时针后继节点
- 冲突解决：向量时钟 + siblings

### 4.5 本节小结

一致性哈希解决固定分区的重分片代价问题：
- **核心**：哈希环 + 虚拟节点，最小迁移
- **优势**：加/减节点只影响 1/N 的 key
- **局限**：范围查询失效，环管理复杂度
- **生产案例**：Cassandra / Dynamo / Riak / Memcached 客户端

## 5. 虚拟槽（Virtual Slots）

### 5.1 模式概述

**核心思想**：固定数量的"槽"（如 16384 个），key 通过哈希映射到槽，节点负责一段连续的槽范围。

```
slot = CRC16(key) % 16384

节点 A：slots 0-5460
节点 B：slots 5461-10922
节点 C：slots 10923-16383
```

**关键差异**：
- 一致性哈希：节点和 key 都映射到环上
- 虚拟槽：key 映射到槽（槽数固定），节点负责槽范围

### 5.2 模式解决的问题

**问题 1：重分片时 key 映射不变**
- 一致性哈希：节点变化影响 key 映射
- 虚拟槽：key→slot 映射不变，slot→节点 映射可变
- 迁移单位是"槽"而非"key"

**问题 2：槽数固定的工程优化**
- 一致性哈希的 token ring 状态随节点数增长
- 虚拟槽的槽数固定，元数据量恒定

### 5.3 为什么是 16384 个槽？

**Redis Cluster 选择 16384（2¹⁴）的三个理由**（antirez GitHub #2576）：

**理由 1：心跳 bitmap 大小**
- 心跳包用 bitmap 表示节点负责的槽
- 16384 / 8 = 2KB（bitmap 大小）
- 如果是 65536 槽，bitmap = 8KB，心跳开销过大

**理由 2：集群规模上限**
- Redis Cluster 建议集群规模 ≤ 1000 节点
- 16384 槽足够每节点平均 16 个槽——与节点数匹配

**理由 3：压缩传输**
- 心跳 bitmap 经过 gzip 压缩后，16384 槽的压缩率高于 65536 槽
- 节点负责的槽通常连续，压缩效果好

### 5.4 槽迁移流程

```
1. 目标节点：CLUSTER SETSLOT <slot> IMPORTING <source_node_id>
2. 源节点：CLUSTER SETSLOT <slot> MIGRATING <target_node_id>
3. 源节点：CLUSTER GETKEYSINSLOT <slot> <count>（获取要迁移的 key）
4. 源节点：MIGRATE <target_ip> <target_port> <key> 0 <timeout>
5. 重复 3-4 直到槽内所有 key 迁移完毕
6. 任意节点：CLUSTER SETSLOT <slot> NODE <target_node_id>（宣布迁移完成）
```

**关键**：迁移过程中槽仍可服务——如果 key 还在源节点，客户端被重定向到源节点（ASK 重定向）；如果已迁移，客户端被重定向到目标节点（MOVED 重定向）。

### 5.5 生产案例

**Redis Cluster**：最知名的虚拟槽生产系统（Codis 也用 1024 虚拟槽）
- 16384 个槽，CRC16(key) % 16384 映射
- 支持在线槽迁移
- MOVED / ASK 重定向
- 心跳维护槽分配表

**与一致性哈希的对比**：
| 维度 | 一致性哈希 | 虚拟槽 |
|------|----------|--------|
| key 映射 | key → node | key → slot → node |
| 节点变化 | key 重新映射 | slot 重新分配，key 映射不变 |
| 元数据 | 环 + token range | 槽分配表 |
| 迁移单位 | key（无中间抽象） | 槽（逻辑抽象层） |
| 典型系统 | Cassandra / Riak | Redis Cluster |

### 5.6 本节小结

虚拟槽通过"固定槽数"提供中间抽象层：
- **核心**：固定槽数 + slot→node 动态映射
- **优势**：key 映射不变，迁移单位是槽，元数据量恒定
- **局限**：范围查询失效，只适合 KV 场景
- **生产案例**：Redis Cluster（唯一实现，16384 槽）

## 6. 分布式时间模式

### 6.1 为什么需要分布式时间模式？

第一篇讲了数据复制（WAL / Segmented Log），第二篇讲了数据分区。但还有一个贯穿两者的"暗线"——**时间**。跨节点的日志顺序怎么确定？并发写入怎么检测冲突？全局快照怎么分配时间点？

唐伟志书第 6 章已经详讲了 Lamport 时钟 / 向量时钟 / HLC / TrueTime 的理论基础。本篇从**工程模式**角度，讲这三个最常用的分布时间模式的实现选择。

### 6.2 Hybrid Clock（混合时钟）

**模式思想**：结合物理时钟（有时间直觉）和逻辑时钟（因果保证），提供带"时间含义"的单调递增时间戳。

**适用场景**：
| 场景 | 不需要 Hybrid Clock | 需要 Hybrid Clock |
|------|---------------------|-------------------|
| 单机事务 ID | 数据库自增 ID 足够 | — |
| 日志顺序 | Kafka Offset 足够 | 跨分区排序时 |
| 分布式事务时间戳 | — | CockroachDB / MongoDB 的跨节点事务排序 |

**CockroachDB 的 HLC 实现**：
- 每个事务分配 `(physical_ms, logical_count)`
- physical_ms 取 `max(本地物理时钟, 对端物理时钟)`
- 如果 physical_ms 不变，logical_count++
- 结果：单调递增 + 接近真实时间

**MongoDB 的 clusterTime**：
- 每个操作分配 `{clusterTime: Timestamp(epoch, counter)}`
- Timestamp 类似于 HLC 的 (physical, logical)
- 用于因果一致性会话（causal sessions）

### 6.3 Versioned Value（版本化值）

**模式思想**：每个值附带版本号，通过比较版本号判断新旧。

**版本号的形式**：
| 形式 | 示例 | 优势 | 劣势 |
|------|------|------|------|
| 时间戳 | `{"value": "hello", "ts": 1620000000}` | 直观 | 时钟回拨导致错误 |
| 单调计数器 | `{"value": "hello", "version": 42}` | 简单 | 单点分配 |
| 向量时钟 | `{"value": "hello", "vclock": [2,1,0]}` | 检测并发 | O(n) 空间 |
| HLC | `{"value": "hello", "hlc": [1620, 3]}` | 因果+时间 | 非精确定序 |

**Dynamo 的向量时钟应用**：
- 每个值带向量时钟 `[c₁, c₂, ..., cₙ]`
- 并发写入时，向量时钟不可比较 → 冲突
- 冲突版本都保留为 siblings，客户端选择

**Kafka 的 offset 版本化**：
- 每条消息带 offset（单调递增）
- 消费者 commit 的 offset 是 "我读到了 offset N"
- 判断消息新旧：offset 大者新

### 6.4 Gossip 协议与时间

**模式思想**：节点通过周期性随机交换元数据（状态、版本号、时间戳），最终所有节点达到一致——不依赖全局时钟或中心节点。

**Gossip 三阶段**：

```
节点 A                    节点 B
  │                         │
  │ — Gossip (digest) —→   │  阶段 1: A 发送自己的元数据摘要
  │ ← Gossip (digest) —   │  阶段 2: B 回复自己的元数据摘要
  │                         │
  │ — Gossip (delta) —→    │  阶段 3: A 比较两份摘要，发送差异
  │ ← Gossip (delta) —    │          B 也发送差异
  │                         │
  ↓                         ↓
双方更新本地状态
```

**Gossip 与时间的关系**：
- Gossip 传递的信息中包含版本号或时间戳
- 节点根据版本号判断信息新旧（versioned value）
- Gossip 的收敛性依赖"新版本最终覆盖旧版本"

**生产案例**：

| 系统 | Gossip 用途 |
|------|-----------|
| **Cassandra** | 节点发现 / 集群拓扑 / token ring 状态传播 |
| **Redis Cluster** | 节点发现 / 槽分配表传播 / 故障检测 |
| **Consul** | 成员管理 / 健康检查传播 / LAN/WAN Gossip Pool |
| **Dynamo** | 节点发现 / Merkle Tree 比较 |

**Gossip 的核心参数**：
- **fanout**：每次向多少个节点发送 gossip（通常 3 个）
- **interval**：gossip 间隔（通常 1 秒）
- **对 gossip 收敛速度的影响**：fanout 越大、interval 越小，收敛越快，但开销越大

### 6.5 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 第 6 章详讲 Lamport/向量/HLC/TrueTime——本篇从工程模式角度聚焦选型与实现 |
| **释慧利** | 算法 | 共识算法中的时间戳（Proposal ID / term / zxid）是 Lamport 时钟的实例 |
| **江峰** | 形式化 | Gossip 协议的收敛性可形式化——"最终所有节点达到一致"依赖新版本覆盖旧版本（读者推导） |

### 6.6 本节小结

分布式时间模式的三种实践：
- **Hybrid Clock**：物理+逻辑结合，CockroachDB / MongoDB 的分事务时间戳
- **Versioned Value**：带版本号的值，Dynamo 向量时钟 / Kafka offset / Redis CAS
- **Gossip**：分布式元数据收敛，Cassandra / Redis Cluster / Consul 的节点发现与拓扑传播

## 7. 分区与时间的组合应用

### 7.1 模式组合的必然性

**分区 + 时间 = 分布式数据系统的完整能力**。

分区解决"数据分布"，时间解决"事件顺序"。两者组合才能实现：
1. 跨分片时间戳分配（一个事务涉及多个分片，需要一致的时间戳）
2. 分片拆分/合并时的版本一致性（分片 A 和分片 B 合并时，哪个版本是最新的？）
3. 跨分片的因果一致性（分片 A 的写入是否影响分片 B 的读取？）

### 7.2 生产系统的模式组合

| 系统 | 分区策略 | 时间方案 | Gossip | 组合效果 |
|------|---------|---------|--------|---------|
| **TiKV** | Range（Region 自动分裂） | TSO（PD 分配时间戳） | — | Percolator 跨分片事务 |
| **CockroachDB** | Range（自动分裂） | HLC（混合逻辑时钟） | — | 跨 Range 快照隔离 |
| **Cassandra** | 一致性哈希 + 虚拟节点 | LWW（时间戳） | ✓（节点发现+token ring） | 可调一致性 + 去中心化 |
| **Redis Cluster** | 虚拟槽（16384） | configEpoch（非分布式时间方案） | ✓（节点发现+故障检测） | 槽内顺序一致 + 自动故障切换 |
| **Spanner** | Range（Tablet 自动分裂） | TrueTime（GPS+原子钟） | — | 跨 Tablet 外部一致性 |

### 7.3 组合选择的决策维度

| 业务需求 | 分区推荐 | 时间推荐 | Gossip 推荐 | 典型系统 |
|---------|---------|---------|-----------|---------|
| 强一致 KV | Range（动态分裂） | TSO / Raft term | 不需要 | TiKV |
| 全球 SQL | Range | HLC / TrueTime | 不需要 | CockroachDB / Spanner |
| 高可用 KV（AP） | 一致性哈希 | LWW / 向量时钟 | ✓ | Cassandra / Riak |
| 缓存 | 虚拟槽 | 物理时钟 | ✓ | Redis Cluster |

### 7.4 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 分区 + 时间 = 分布式数据系统的完整能力——分区横向扩展，时间纵向排序 |
| **释慧利** | 算法 | 共识算法是分区与时间的交汇点——Raft term 是时间，Raft log 是分区的复制 |
| **江峰** | 形式化 | 分区与时间的组合正确性依赖"分区不变量 + 时间单调性"（读者推导） |

### 7.5 本节小结

分区与时间的组合应用：
- **TiKV**：Range + TSO + Percolator = 跨分片快照隔离
- **CockroachDB**：Range + HLC = 跨 Range 快照隔离
- **Cassandra**：一致性哈希 + LWW + Gossip = AP + 去中心化
- **Redis Cluster**：虚拟槽 + Gossip = 缓存 + 故障自愈
- **Spanner**：Range + TrueTime = 外部一致性

## 8. 对比与选型

### 8.1 分区模式对比矩阵

| 模式 | 范围查询 | 重分片代价 | 分布均衡 | 键映射稳定性 | 规模上限 | 典型系统 |
|------|---------|----------|--------|-----------|--------|---------|
| 固定分区（Hash） | ✗ | 高（全迁移） | 均匀 | 差（N变key变） | N 固定 | MySQL 分库分表 |
| 动态分区（Range） | ✓ | 低（局部分裂） | 不均（热点） | 好（key映射不变） | PB 级 | TiKV / HBase |
| 一致性哈希 | ✗ | 低（1/N 迁移） | 均匀（虚拟节点） | 中（N变部分key变） | 数千节点 | Cassandra / Riak |
| 虚拟槽 | ✗ | 低（按槽迁移） | 均匀 | 好（key→slot不变） | 1000 节点 | Redis Cluster |

### 8.2 时间模式选型

| 业务需求 | 推荐方案 | 典型系统 |
|---------|---------|---------|
| 跨分片强事务 | TSO（TiKV）/ HLC（CockroachDB）/ TrueTime（Spanner） | TiKV / CockroachDB / Spanner |
| 冲突检测 | 向量时钟（Dynamo）/ 版本号（Riak） | Dynamo / Riak |
| 单分区全序 | Offset（Kafka）/ zxid（ZooKeeper）/ term（Raft） | Kafka / ZooKeeper / etcd |
| 去中心化元数据 | Gossip + 版本号 | Cassandra / Consul / Redis Cluster |

### 8.3 选型决策树

```
需要范围查询？
   ├─ 是 → 动态分区（Range）→ TiKV / HBase / CockroachDB
   └─ 否 → 需要低重分片代价？
            ├─ 是 → 需要 key 映射稳定？
            │       ├─ 是 → 虚拟槽 → Redis Cluster
            │       └─ 否 → 一致性哈希 → Cassandra / Riak
            └─ 否 → 固定分区 → MySQL 分库分表（不常扩容时）
```

### 8.4 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 分区 + 时间的选型本质是"一致性 vs 可用性"的工程落地——强一致用 Range+TSO，AP 用一致性哈希+LWW |
| **释慧利** | 算法 | 共识算法在分区+时间选型中扮演枢纽角色——Raft/Paxos 提供强一致的分区复制与 term/zxid 时间戳 |
| **江峰** | 形式化 | 选型必须验证正确性——分区不变量 + 时间单调性 + Gossip 收敛性（读者推导） |

### 8.5 本节小结

分区与时间的对比选型：
- **强一致 + 范围查询**：Range + TSO/HLC/TrueTime
- **AP + 去中心化**：一致性哈希 + LWW + Gossip
- **缓存 + 自动故障切换**：虚拟槽 + Gossip
- **不常扩容**：固定分区（Hash）

## 结语：从模式到架构

回顾本篇，我们完成了数据分区和分布式时间的模式梳理：

- **第 1 节**：分区 vs 复制——正交维度，组合构成分布式数据系统
- **第 2-5 节**：四种分区模式（固定/动态/一致性哈希/虚拟槽）——从最简单到最灵活
- **第 6 节**：三种分布式时间模式（HLC / Versioned Value / Gossip）——时间在分布式系统中的工程实践
- **第 7-8 节**：组合应用与选型——真实系统都是多种模式的组合

**第二篇的核心论断**：分区解决"数据如何分布"，时间解决"事件如何排序"。两者正交组合，构成分布式数据系统的完整能力。四种分区模式（固定/动态/一致性哈希/虚拟槽）按"重分片代价"递增，按"范围查询能力"递减。三种时间模式（HLC / Versioned Value / Gossip）按"一致性"递减，按"去中心化"递增。选型没有银弹——每种模式都是特定业务需求下的最优工程解。
