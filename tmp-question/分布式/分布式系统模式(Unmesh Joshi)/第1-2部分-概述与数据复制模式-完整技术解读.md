## 1. 为什么需要"分布式系统模式"？

### 1.1 理论≠模式：从"为什么"到"怎么做"

唐伟志书第 1-6 章和释慧利书建立了分布式系统的理论框架：CAP / FLP / 共识算法 / 一致性模型 / 时间方案。但理论与工程实践之间有鸿沟：

**理论说**："多数派 Quorum 保证强一致（W+R>N）"
**实践问**："那代码怎么写？Acceptor 状态机怎么实现？日志怎么持久化？"

**理论说**："主从复制实现数据同步"
**实践问**："主从切换时，怎么保证不丢数据？从节点落后太多怎么追上？"

**理论说**："快照隔离需要全局时间戳"
**实践问**："时间戳怎么分配？TSO 挂了怎么办？HLC 的实现细节？"

**设计模式**（Design Pattern）填补了这段鸿沟——它是**经过验证的、可复用的工程解决方案**。

### 1.2 模式的三层价值

| 层 | 价值 | 示例 |
|---|------|------|
| **认知层** | 给常见问题命名，让团队有共同语言 | "这里用 WAL 模式保证持久性" |
| **设计层** | 提供可复用的架构模板 | 主从复制的组件交互图 |
| **实现层** | 给出关键代码结构与工程参数 | etcd 的 WAL 实现、Kafka 的 Segmented Log |

### 1.3 本篇的模式分类

本篇从三个维度组织数据复制模式：

**维度 1：复制拓扑**
- 主从（Leader-Follower）：单点写入，串行复制
- 多主（Multi-Leader）：多点写入，异步同步
- 无主（Leaderless）：Quorum 写入，去中心化

**维度 2：写入流程**
- 同步复制（全同步 / 多数派同步 / 半同步）
- 异步复制
- 流水线复制（连续发送不等 ACK）

**维度 3：日志管理**
- Write-Ahead Log（WAL）：先写日志后执行
- Segmented Log：分段管理，压缩与清理

## 2. 分布式系统的核心挑战与模式分类

### 2.1 核心挑战回顾

唐伟志书第 1 章建立了 4 大难题（部分失败 + 网络不可靠 + 并发不确定 + 异构故障）。数据复制模式要解决的正是这 4 大难题在**数据层**的体现：

| 难题 | 数据层体现 | 对应模式 |
|------|----------|---------|
| 部分失败 | 节点宕机不丢数据 | 主从复制 + WAL |
| 网络不可靠 | 消息延迟导致副本不一致 | Quorum + 多数派确认 |
| 并发不确定 | 多客户端并发写冲突 | 多主复制 + 冲突解决 |
| 异构故障 | 不同节点的磁盘/网络差异 | 异构部署 + 分级复制 |

### 2.2 模式的分类框架

分布式数据模式按功能分为 5 大类：

```
                    ┌─────────────────────┐
                    │    数据复制模式      │
                    │  （主从/多主/无主）   │
                    └──────────┬──────────┘
                               │
        ┌──────────────────────┼──────────────────────┐
        ↓                      ↓                      ↓
┌───────────────┐    ┌───────────────┐    ┌───────────────┐
│  日志管理模式  │    │  一致性模式    │    │  拓扑管理      │
│  (WAL/SegLog)  │    │  (Quorum/因果) │    │  (Leader选举)  │
└───────────────┘    └───────────────┘    └───────────────┘
```

**本篇聚焦**：数据复制模式 + 日志管理模式。

### 2.3 模式决策树

选择数据复制模式的标准决策流程：

```
是否需要强一致写？
   ├─ 是 → 主从复制
   └─ 否 → 是否需要跨地域低延迟？
            ├─ 是 → 多主复制
            └─ 否 → 无主复制（Quorum）
```

**但实际工程中，这三个选择还会交叉**：
- etcd：主从（Raft）+ WAL + Segmented Log（Snapshot）
- Kafka：主从（ISR）+ Segmented Log + 对账
- CockroachDB：主从（Raft）+ 多 Raft 组 + HLC

## 3. 主从复制（Leader-Follower）

### 3.1 模式概述

**核心思想**：所有写操作经过 Leader，Leader 把变更复制到 Follower。

```
              Client
                ↓
             ┌──────┐
             │ Leader│ ← 处理所有写
             └──┬───┘
        ┌───────┼───────┐
        ↓       ↓       ↓
    ┌──────┐ ┌──────┐ ┌──────┐
    │Follower│Follower│Follower│
    └──────┘ └──────┘ └──────┘
```

### 3.2 模式解决的问题

**问题 1：写冲突**
- 无 Leader 时，多个节点独立接受写，可能冲突
- 主从复制通过**单点串行化**消除冲突

**问题 2：读一致性**
- Leader 有最新数据，读 Leader 保证强一致
- Follower 可能落后，读 Follower 是最终一致

**问题 3：故障恢复**
- Leader 故障 → 选举新 Leader
- Follower 故障 → 重新加入时从 Leader 同步

### 3.3 同步策略

**策略 1：全同步（所有 Follower ACK）**
```
Leader → AppendEntries → F1 ACK → F2 ACK → F3 ACK → 提交
```
- 优势：数据不丢
- 劣势：任何一个 Follower 慢都会拖慢整个系统
- 适用：金融账户（一致性优先）

**策略 2：多数派同步（多数派 ACK）**
```
Leader → AppendEntries → F1 ACK → F2 ACK → 提交（F3 可慢）
```
- 优势：容忍少数 Follower 故障或慢
- 劣势：少数 Follower 落后
- 适用：etcd / Raft / ZAB

**策略 3：半同步（至少 1 个 Follower ACK）**
```
Leader → AppendEntries → F1 ACK → 提交（F2/F3 可慢）
```
- 优势：接近于全异步的低延迟
- 劣势：ACK 的 Follower 也宕机时丢数据
- 适用：MySQL 半同步复制

**策略 4：全异步（不等任何 Follower）**
```
Leader → 本地提交 → 异步 AppendEntries
```
- 优势：最低延迟
- 劣势：Leader 宕机可能丢数据
- 适用：MySQL 异步复制 / Redis 主从

### 3.4 故障切换（Failover）

**场景 1：Follower 故障**
- Leader 从 Follower 列表中移除
- Follower 恢复后重新加入，从 Leader 同步日志
- **关键**：Follower 必须知道从哪里开始同步（需要日志索引）

**场景 2：Leader 故障**
- 触发：Follower 超时未收到心跳
- 选举：选日志至少与自己一样新的 Follower 为新 Leader（Raft：先比 lastTerm，term 相同再比 lastLogIndex）
- 避免脑裂：共识算法保证一个 term 最多一个 Leader（Raft）
- **关键**：已提交数据不能丢（选举限制）

### 3.5 生产实现要点

**要点 1：日志索引**
- 每条日志有唯一索引（Raft 的 log index / ZAB 的 zxid / Kafka 的 offset）
- Follower 根据索引判断与 Leader 的差距
- Leader 根据索引判断哪些日志已提交

**要点 2：心跳机制**
- Leader 定期发送心跳（空 AppendEntries）
- Follower 超时未收到心跳 → 触发选举
- **参数选择**：太短误判（网络抖动），太长恢复慢

**要点 3：日志压缩（Snapshot）**
- 日志无限增长，需定期压缩
- Leader 做 Snapshot → 发给 Follower
- Snapshot 之前的日志可删除

**要点 4：读一致性**
- 默认读 Leader（强一致）
- 可降级读 Follower（最终一致）
- ReadIndex（etcd 3.4+）让 Follower 也能服务线性一致读

### 3.6 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 第 3.2 节 主从抄写是数据复制的基础——Leader 串行化保证写入无冲突 |
| **释慧利** | 算法 | 第 4-6 章 Paxos/Raft/ZAB 都是主从复制的共识实现——Prepare+Accept 或 RequestVote+AppendEntries |
| **江峰** | 形式化 | 主从复制的正确性等价于"Leader 唯一性 + 日志不变量"（读者推导） |

### 3.7 本节小结

主从复制是最基础的数据复制模式：
- **核心**：单 Leader 串行写入，日志复制到 Follower
- **同步策略**：全同步 / 多数派 / 半同步 / 全异步
- **故障切换**：Follower 恢复同步 / Leader 故障选举
- **生产要点**：日志索引 + 心跳 + Snapshot + 读一致性

## 4. 多主复制（Multi-Leader）

### 4.1 模式概述

**核心思想**：多个 Leader 同时接受写入，异步同步到其他 Leader。

```
        Client A           Client B
          ↓                   ↓
    ┌──────────┐        ┌──────────┐
    │ Leader A │←异步→│ Leader B │
    └────┬─────┘        └────┬─────┘
         ↓                    ↓
    Follower A            Follower B
```

### 4.2 模式解决的问题

**问题 1：跨地域延迟**
- 主从模式下，所有写必须到同一个 Leader
- 跨地域写入延迟高（中美 RTT 200ms）
- 多主模式：客户端写本地 Leader，异步同步到远端

**问题 2：单点写入瓶颈**
- 主从模式下，所有写经一个 Leader
- 多主模式：多个 Leader 并行写入

### 4.3 模式引入的代价：写冲突

**核心问题**：多个 Leader 同时修改同一数据，如何解决冲突？

```
时刻 t1：Leader A 修改 x=1
时刻 t2：Leader B 修改 x=2（在 t1 之前收到修改，但不知道 t1）
时刻 t3：异步同步后发现 x=1 和 x=2 冲突
```

### 4.4 写冲突的解决策略

**策略 1：避免冲突（冲突预防）**
- 设计上确保同一数据只被一个 Leader 写入
- 例：按用户 ID 哈希，每个用户只路由到一个 Leader
- 局限：无法处理跨用户操作

**策略 2：最后写入胜出（LWW）**
- 每个写入带时间戳，冲突时取时间戳最大的值
- 优势：简单
- 劣势：依赖时钟同步；可能丢失数据（较早写入者的修改被覆盖）

**策略 3：应用层合并（Merge）**
- 保留冲突版本，由应用逻辑合并
- 例：购物车——两个 Leader 各添加了商品，合并后取并集
- 适用：数据类型支持合并的场景（Set / Counter / Registry）

**策略 4：CRDT（Conflict-free Replicated Data Type）**
- 数据结构本身保证合并后结果唯一
- 例：G-Counter（只增不减的计数器）、LWW-Register
- 优势：自动合并，无冲突
- 局限：只支持特定数据类型

**策略 5：向量时钟 + Siblings（冲突保留）**
- 用向量时钟检测冲突
- 冲突时保留所有版本（siblings），由客户端选择
- 适用：Dynamo / Riak
- 局限：客户端复杂（需理解 siblings）

### 4.5 多主 vs 主从的划分

**何时用多主？**

| 条件 | 理由 |
|------|------|
| 跨地域部署 | 本地 Leader 低延迟 |
| 写冲突少 | 同一数据很少被多地同时修改 |
| 数据类型可合并 | Set/Counter 等 CRDT 类型 |
| 最终一致可接受 | 不要求强一致 |

**何时用主从？**

| 条件 | 理由 |
|------|------|
| 同机房部署 | Leader 延迟低（< 1ms） |
| 写冲突风险高 | 主从串行化消除冲突 |
| 强一致要求 | 金融 / 库存 |
| 数据类型不可合并 | 转账 / 扣减 |

### 4.6 生产案例

**案例 1：Cassandra 的多数据中心**
- 每个数据中心独立接受写入（Cassandra 本质是无主架构，多 DC 时每个 DC 是无主集群，跨 DC 异步复制，协调者由客户端指定）
- 跨数据中心异步复制
- 冲突解决：LWW（Last-Write-Wins）
- 一致性级别：可调（ONE / QUORUM / ALL）

**案例 2：CouchDB 的多主同步**
- 每个节点可独立写入
- 用版本向量检测冲突
- 冲突解决：保留所有版本，客户端选择

**案例 3：MySQL 双主（不推荐）**
- 两个节点都是 Master
- 简单场景：一写一读（避免冲突）
- 复杂场景：容易脑裂——建议用 MGR（Paxos 变种）替代

### 4.7 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 第 3.3 节 多主复制解决跨地域延迟——但冲突解决是核心代价 |
| **释慧利** | 算法 | 多主复制本身不是共识算法——需要额外的冲突解决机制（CRDT / LWW / 向量时钟） |
| **江峰** | 形式化 | 多主复制的一致性在形式化上弱于主从——最终一致而非强一致（读者推导） |

### 4.8 本节小结

多主复制解决跨地域延迟和写瓶颈——但代价是写冲突：
- **跨地域部署**：本地 Leader 低延迟
- **冲突解决**：避免冲突 / LWW / 应用层合并 / CRDT / 向量时钟
- **何时用**：跨地域 + 冲突少 + 最终一致可接受
- **何时不用**：同机房 + 强一致 + 数据类型不可合并

## 5. 无主复制（Leaderless）

### 5.1 模式概述

**核心思想**：没有 Leader，任何节点都可接受写入，通过 Quorum（NWR）协调一致性。

```
        Client
          ↓
   ┌──────────────┐
   │  Coordinator │（任意节点担任）
   └──────┬───────┘
    ┌─────┼─────┐
    ↓     ↓     ↓
  ┌───┐ ┌───┐ ┌───┐
  │N1 │ │N2 │ │N3 │
  └───┘ └───┘ └───┘
```

### 5.2 模式解决的问题

**问题 1：Leader 单点**
- 主从模式下，Leader 故障时服务短暂不可用
- 无主模式：任何节点都可接受写入，无单点

**问题 2：写入吞吐瓶颈**
- 主从模式下，所有写入经 Leader
- 无主模式：多节点并行写入

### 5.3 核心机制：Quorum（NWR）

**NWR 定义**（详见唐伟志书第 3 章 4 节）：
- **N**（副本数）：数据保存到 N 个副本
- **W**（写 Quorum）：写入需 W 个副本确认
- **R**（读 Quorum）：读取需 R 个副本返回

**强一致条件**：`W + R > N`

**典型配置**：

| 配置 | W | R | 一致性 | 容错 |
|------|---|---|--------|------|
| 强一致 | N | N | 最强 | 最低（任何故障不可用） |
| 多数派读写 | ⌈(N+1)/2⌉ | ⌈(N+1)/2⌉ | 强一致 | 容忍少≤⌊N/2⌋ 故障 |
| 写多数读单 | ⌈(N+1)/2⌉ | 1 | 一致（读绕后） | 写入容忍 |
| 最终一致 | 1 | 1 | 最终一致 | 最高容忍故障 |

### 5.4 无主复制的关键机制

**机制 1：Hinted Handoff**
- 写入时如果目标副本不可达，Coordinator 把写入"借"给其他可达节点
- 标注为 hinted（暗示的）
- 原副本恢复后，hinted 节点把数据回传
- 例：Cassandra `hinted_handoff_enabled=true`

**机制 2：读修复（Read Repair）**
- 读取时发现副本不一致
- Coordinator 用最新值修复落后副本
- 例：Cassandra 默认开启读修复

**机制 3：反熵（Anti-Entropy）**
- 后台定期比较副本
- 用 Merkle Tree 快速检测差异块
- 差异块同步到一致
- 例：Cassandra 的反熵同步、Dynamo 的 Merkle Tree

### 5.5 生产案例

**案例 1：DynamoDB（AWS 服务）**
- 默认最终一致读
- 可选强一致读（W=⌈(N+1)/2⌉ + R=⌈(N+1)/2⌉）
- 冲突解决：LWW

**案例 2：Cassandra**
- 可调一致性（ONE / QUORUM / ALL / LOCAL_QUORUM）
- Hinted Handoff
- 读修复 + Merkle Tree 反熵
- 冲突解决：LWW

**案例 3：Riak**
- 向量时钟 + Siblings 冲突解决
- Quorum（NWR）
- 读修复 + Hinted Handoff

### 5.6 无主 vs 主从的划分

**何时用无主？**

| 条件 | 理由 |
|------|------|
| 高可用优先（AP） | 无单点故障影响 |
| 写入可重试 | 冲突可解决 |
| 数据类型可合并 | CRDT / Set |
| 最终一致可接受 | 读写一致性可调但不保证 |

**何时用主从？**

| 条件 | 理由 |
|------|------|
| 强一致优先（CP） | Leader 串行化保证 |
| 写入不可冲突 | 金融交易、库存扣减 |
| 数据类型不可合并 | 转账 |
| 简单性优先 | 主从实现更简单 |

### 5.7 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 第 3.4 节 无主复制是 Dynamo 的核心架构——NWR + Hinted Handoff + Merkle Tree |
| **释慧利** | 算法 | 无主复制不用共识算法，而是用 Quorum + 向量时钟 + 读修复保证最终一致 |
| **江峰** | 形式化 | 无主复制的正确性依赖 W+R>N（鸽笼原理）和反熵完整性（读者推导） |

### 5.8 本节小结

无主复制实现高可用，核心机制：
- **NWR**：可调一致性，W+R>N 保证强一致
- **Hinted Handoff**：容忍节点暂时不可达
- **读修复 + 反熵**：后台修复副本一致
- **生产案例**：DynamoDB / Cassandra / Riak

## 6. 流水线复制（Pipeline）

### 6.1 模式概述

**核心思想**：Leader 连续发送多个复制请求，不等前一个 ACK 就发下一个。

```
无流水线：
  Leader → PROPOSAL₁ → ACK₁ → PROPOSAL₂ → ACK₂ → PROPOSAL₃ → ACK₃
  每个 PROPOSAL 必须等前一个 ACK

有流水线：
  Leader → PROPOSAL₁ → PROPOSAL₂ → PROPOSAL₃ → ACK₁ → ACK₂ → ACK₃
  Leader 连续发送 PROPOSAL，不等 ACK
```

### 6.2 模式解决的问题

**问题**：无流水线时，每个复制请求必须等待上一个完成。

- 网络往返延迟 RTT = 1ms（同机房）
- 无流水线：100 个 PROPOSAL 需要 100 × RTT = 100ms
- 有流水线：100 个 PROPOSAL 只需要 1 × RTT = 1ms（忽略处理时间）
- 理想情况下吞吐可达 N 倍（N=流水线深度），实际受 Follower 处理能力和窗口大小限制，通常 5-20x

### 6.3 流水线的工程要求

**要求 1：消息顺序保证**
- PROPOSAL 必须按顺序发送（FIFO 通道）
- ACK 可以不按顺序到达（乱序 ACK）
- 提交必须按顺序（COMMIT 顺序 = PROPOSAL 顺序）

**要求 2：乱序 ACK 处理**
- Leader 可能先收到 PROPOSAL₃ 的 ACK，后收到 PROPOSAL₂ 的 ACK
- Leader 缓冲已收到 ACK 的 PROPOSAL₃
- 等 PROPOSAL₂ 的 ACK 到达后，按顺序提交

**要求 3：流量控制**
- Leader 不能无限发送 PROPOSAL（Follower 可能跟不上）
- 需要窗口机制限制在途 PROPOSAL 数量
- 类似 TCP 的滑动窗口

### 6.4 生产案例

**案例 1：ZooKeeper 的流水线**
- ZAB 协议：Leader 连续发送 PROPOSAL
- Follower 持久化后 ACK
- Leader 收到 ACK 后发送 COMMIT
- 吞吐比无流水线提升 10-50x

**案例 2：Kafka 的批量发送**
- Producer 批量发送消息
- Broker 连续写入日志
- Follower 连续拉取

**案例 3：etcd Raft 的流水线**
- Leader 连续发送 AppendEntries
- Follower 持久化后 ACK
- Leader 按顺序 commitIndex 推进

### 6.5 流水线的局限

**局限 1：不适用于单次操作**
- 流水线的收益来自批量操作
- 单次写入（如单 key put）不受益于流水线

**局限 2：Follower 瓶颈**
- 如果 Follower 处理速度慢，流水线填满后 Leader 被阻塞
- 需要监控 Follower 的处理延迟

**局限 3：乱序 ACK 的复杂度**
- 需要缓冲未提交的 PROPOSAL
- 需要窗口管理

### 6.6 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 流水线是复制性能优化的关键——降低延迟、提升吞吐 |
| **释慧利** | 算法 | Zo zKeeper ZAB 和 Raft 都内置流水线——这不是可选优化，而是复制协议的标准设计 |
| **江峰** | 形式化 | 流水线不改变复制协议的正确性——只改变性能（读者推导） |

### 6.7 本节小结

流水线复制是吞吐优化的关键模式：
- **核心**：连续发送不等 ACK
- **要求**：FIFO 通道 + 乱序 ACK 处理 + 流量控制
- **收益**：吞吐提升 10-100x（批量操作）
- **局限**：不适用单次操作 + Follower 瓶颈

## 7. Write-Ahead Log（WAL）

### 7.1 模式概述

**核心思想**：所有状态变更先写入持久化日志（WAL），再应用到状态机。

```
客户端写操作
    ↓
┌──────────────────────┐
│  1. 写 WAL（持久化）  │  ← 关键：先写日志
└──────────────────────┘
    ↓
┌──────────────────────┐
│  2. 应用到状态机      │  ← 后应用状态
└──────────────────────┘
    ↓
  响应客户端
```

### 7.2 模式解决的问题

**问题 1：崩溃恢复**
- 节点崩溃，内存状态丢失
- WAL 中有所有已提交操作
- 重启后 replay WAL，恢复状态

**问题 2：原子性**
- 操作要么完全执行（WAL 中有记录），要么完全不执行（WAL 中无记录）
- 崩溃恢复时：有 WAL → redo；无 WAL → ignore

**问题 3：复制**
- WAL 是复制的载体
- Leader 的 WAL 复制到 Follower
- Follower 收到 WAL 后应用到本地状态机

### 7.3 WAL 的工程实现

**实现 1：etcd 的 WAL**
- 位置：`/var/lib/etcd/member/wal/`
- 格式：Protobuf 序列化的 Raft 日志条目
- 写策略：每次写入后 `fsync`（强持久）
- 压缩：定期 Snapshot，删除旧 WAL

**实现 2：ZooKeeper 的事务日志**
- 位置：`dataLogDir`
- 格式：ZAB 事务的序列化
- 写策略：每次事务后 `fsync`
- 压缩：定期 Snapshot（`dataDir`）

**实现 3：Kafka 的日志**
- 位置：每个 Partition 的日志目录
- 格式：消息的批量写入
- 写策略：Page Cache + 定期 `fsync`（非每消息）
- 压缩：Log Compaction（按 key 保留最新值）

**实现 4：MySQL 的 Binlog**
- 位置：MySQL 指定的 log 目录
- 格式：Row / Statement / Mixed
- 写策略：`sync_binlog=1`（每次写入后 `fsync`）
- 压缩：Purge Binary Logs

### 7.4 WAL 的写策略

| 策略 | fsync | 持久性 | 性能 | 生产系统 |
|------|-------|--------|------|---------|
| 每次写入 `fsync` | 每次 | 最强 | 最低 | etcd / ZooKeeper / MySQL `sync_binlog=1` |
| 批量 `fsync` | 每 N 次 | 中（丢 N 次数据） | 中 | Kafka |
| 定时 `fsync` | 每秒 1 次 | 中（丢最多 1s 数据） | 中 | Redis（AOF everysec：后台线程每秒 fsync，进程 crash 不丢，OS crash/断电最多丢 1 秒） |
| 异步写 | 不 `fsync` | 最弱（崩溃丢数据） | 最高 | Redis（AOF no 模式） |

### 7.5 WAL 与复制的关系

**WAL 是复制的载体**：
```
Leader WAL
   ↓ (复制)
Follower WAL
   ↓ (应用)
Follower 状态机
```

- Leader 把 WAL 条目发送给 Follower
- Follower 持久化 WAL 后 ACK
- Follower 应用 WAL 到本地状态机

**为什么 WAL 必须持久化？**
- 如果 WAL 不持久化，崩溃后丢失
- 丢失的可能是已 ACK 给 Leader 的条目
- Leader 以为已复制，实际 Follower 丢失
- 导致数据不一致

### 7.6 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | WAL 是持久性与原子性的基础——先写日志后执行 |
| **释慧利** | 算法 | Paxos/Raft/ZAB 都要求 WAL 持久化——不持久化破坏 Safety |
| **江峰** | 形式化 | WAL 的持久性是共识算法 Safety 的前提——Acceptor 必须持久化状态（读者推导） |

### 7.7 本节小结

WAL 是持久化和复制的基石：
- **核心**：先写日志，后应用状态
- **作用**：崩溃恢复 + 原子性 + 复制载体
- **工程实现**：etcd WAL / ZooKeeper 事务日志 / Kafka 日志 / MySQL Binlog
- **写策略**：每次 `fsync`（强持久）/ 批量 `fsync` / 异步写

## 8. Segmented Log（分段日志）

### 8.1 模式概述

**核心思想**：把无限增长的日志分成固定大小的段，便于管理和压缩。

```
单一日志：
  [entry1][entry2][entry3][......][entry1M]

分段日志：
  段1: [entry1..entry1000]
  段2: [entry1001..entry2000]
  段3: [entry2001..entry3000]
  ...
```

### 8.2 模式解决的问题

**问题 1：日志无限增长**
- 单一日志无限增长，磁盘最终耗尽
- 分段后：已做 Snapshot 的段可删除

**问题 2：日志查找效率**
- 单一日志查找 index=50000 的条目：需顺序扫描
- 分段后：跳到正确的段文件，二分查找

**问题 3：日志清理**
- 分段日志可以按时间或大小清理
- 清理旧段不影响新段

### 8.3 Segmented Log 的工程实现

**实现 1：Kafka 的 Segment**
- 每个 Partition 的日志按段组织
- 段大小：1GB（默认）
- 段命名：`00000000000000000000.log`（起始 offset）
- 清理：按时间（retention.ms）或大小（retention.bytes）

**实现 2：etcd 的 Snapshot**
- WAL 分成多个文件
- 压缩时删除 Snapshot 之前的 WAL 文件
- 新文件从 Snapshot 之后开始

**实现 3：BookKeeper 的 Ledger**
- 每个 Ledger 是一个独立段
- Ledger 关闭后不可修改
- 新数据写入新 Ledger

### 8.4 段管理策略

**策略 1：按大小分段**
- 段文件达到 `segment.bytes` 后创建新段
- Kafka 默认 1GB

**策略 2：按时间分段**
- 每 N 小时创建新段
- 适合日志类数据

**策略 3：按 Snapshot 分段**
- 每次 Snapshot 后，之前的段可删除
- etcd / ZooKeeper 的 Snapshot 机制

**策略 4：按 Ledger 分段**
- 每个 Ledger 是独立可关闭的段
- BookKeeper 的设计

### 8.5 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | Segmented Log 是日志管理的基础——Kafka / etcd / BookKeeper 都用到 |
| **释慧利** | 算法 | Segmented Log 与 Snapshot 配合——共识算法的日志压缩依赖它 |
| **江峰** | 形式化 | Segmented Log 不改变日志的不变量——只改变存储方式（读者推导） |

### 8.6 本节小结

Segmented Log 是日志管理的核心模式：
- **核心**：无限日志分成固定大小的段
- **优势**：可删除旧段 + 查找高效 + 清理方便
- **工程实现**：Kafka Segment / etcd Snapshot / BookKeeper Ledger
- **管理策略**：按大小 / 按时间 / 按 Snapshot / 按 Ledger

## 9. 数据复制模式的对比与选型

### 9.1 模式对比矩阵

| 模式 | 一致性 | 写入延迟 | 吞吐 | 容错 | 复杂度 |
|------|--------|---------|------|------|--------|
| 主从（全同步） | 强一致 | 高（等所有 ACK） | 低 | 低（任一 Follower 故障不可写） | 中 |
| 主从（多数派） | 强一致 | 中 | 中 | 中（容忍少数故障） | 中 |
| 主从（全异步） | 最终一致 | 低 | 高 | 低（Leader 故障丢数据） | 低 |
| 多主 | 最终一致 | 低（本地 Leader） | 高 | 高（多 Leader 并行） | 高（冲突解决） |
| 无主（W=R=⌈(N+1)/2⌉） | 强一致 | 中 | 高 | 中（容忍少数故障） | 中 |
| 无主（W=R=1） | 最终一致 | 低 | 最高 | 最高 | 中 |
| 流水线 | 同复制协议 | 低（批量降低延迟） | 最高（批量提升） | 同复制协议 | 高（乱序 ACK） |
| WAL | 单机持久 | 取决于 fsync | 取决于 fsync | 单机恢复 | 低 |
| Segmented Log | 同复制协议 | 同系统 | 同系统 | 同系统 | 中 |

### 9.2 选型决策树

```
需要跨地域低延迟写？
   ├─ 是 → 多主复制
   └─ 否 → 需要强一致？
            ├─ 是 → 需要写入高吞吐？
            │       ├─ 是 → 无主（W=R=⌈(N+1)/2⌉）
            │       └─ 否 → 主从（多数派）
            └─ 否 → 需要最高可用？
                    ├─ 是 → 无主（W=R=1）
                    └─ 否 → 主从（全异步）
```

### 9.3 生产系统的模式组合

| 系统 | 复制拓扑 | 日志管理 | 流水线 | WAL |
|------|---------|---------|--------|-----|
| etcd | 主从（Raft，多数派） | Segmented Log（Snapshot） | ✓ | ✓（每次 fsync） |
| ZooKeeper | 主从（ZAB，多数派） | Segmented Log（Snapshot） | ✓ | ✓（每次 fsync） |
| Kafka | 主从（ISR，不等多数派） | Segmented Log（按大小） | ✓ | 批量 fsync |
| CockroachDB | 主从（Raft，多数派） | Segmented Log（Snapshot） | ✓ | ✓ |
| Cassandra | 无主（Quorum 可调） | Segmented Log（SSTable） | N/A | ✓ |
| DynamoDB | 无主（Quorum 可调） | Segmented Log（SSTable） | N/A | ✓ |

### 9.4 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 选型是 CAP 的工程落地——CP 选主从，AP 选无主，跨地域选多主 |
| **释慧利** | 算法 | 共识算法（Paxos/Raft/ZAB）把主从 + WAL + Segmented Log 组合为工业实现 |
| **江峰** | 形式化 | 每种模式的正确性可形式化验证——不变量 + 故障模型 + 安全证明（读者推导） |

### 9.5 本节小结

数据复制模式选型：
- **强一致**：主从（多数派）→ etcd / Raft / ZAB
- **跨地域**：多主 → Cassandra 多 DC
- **高可用**：无主（W=R=1）→ DynamoDB / Cassandra
- **高吞吐**：无主 + 流水线 → 批量复制
- **模式组合**：几乎所有工业系统都是多模式的组合

## 结语：模式的价值

回顾本篇，我们完成了分布式数据复制模式的系统梳理：

- **第 1-2 节**：模式的认知价值 + 分类框架
- **第 3 节**：主从复制（最基础，强一致）
- **第 4 节**：多主复制（跨地域，冲突解决）
- **第 5 节**：无主复制（高可用，Quorum）
- **第 6 节**：流水线复制（吞吐优化）
- **第 7 节**：WAL（持久化基础）
- **第 8 节**：Segmented Log（日志管理）
- **第 9 节**：对比与选型

**本篇的核心论断**：设计模式不是抽象的概念罗列，而是**经过验证的、可复用的工程解决方案**。每种模式都对应一个具体的工程问题（部分失败 / 网络延迟 / 并发冲突 / 日志增长），每种模式都有明确的适用场景和工程代价。理解模式才能理解"为什么 etcd 用 Raft + WAL + Snapshot、Kafka 用 ISR + Segmented Log、Cassandra 用无主 + Hinted Handoff"——**工业系统的架构就是模式的组合**。
