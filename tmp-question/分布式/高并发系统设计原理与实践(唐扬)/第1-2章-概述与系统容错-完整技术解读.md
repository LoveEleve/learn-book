## 1. 为什么需要高并发系统设计？

### 1.1 从"能跑"到"扛得住"

单机开发阶段，系统只需要"能跑"——业务逻辑正确、数据库读写正常、单元测试通过。但上线之后，面对的不再是 IDE 里的单一请求，而是**数万 QPS 的并发流量**。

**量变引起质变**：
- 100 QPS：任何系统都能跑
- 1000 QPS：需要关注数据库连接池
- 10000 QPS：需要缓存、读写分离、异步化
- 100000 QPS：需要分库分表、消息队列、CDN
- 1000000 QPS：需要全链路压测、弹性伸缩、异地多活

**每个量级都对应一套不同的架构选择**。高并发系统设计不是"选最牛的方案"，而是"为当前的量级选最合适的方案"。

### 1.2 高并发的本质挑战

唐伟志书第 1 章讲的 4 大难题（部分失败/网络不可靠/并发不确定/异构故障）在高并发场景下被**放大**：

| 难题 | 低并发表现 | 高并发放大 |
|------|----------|----------|
| **部分失败** | 偶尔 1 个节点故障 | 大量请求涌入时故障节点拖垮整个系统 |
| **网络不可靠** | 0.01% 丢包无感 | 10000 QPS × 0.01% = 1 个失败/秒，用户可感知 |
| **并发不确定** | 少量锁等待 | 大量锁竞争导致吞吐雪崩 |
| **异构故障** | 个别慢节点 | 慢节点堆积请求 → 连锁故障 |

### 1.3 高并发系统的三层架构

```
                     ┌─────────────────┐
                     │  接入层          │
                     │  (CDN/DNS/LB)   │
                     ├─────────────────┤
                     │  服务层          │
                     │  (RPC/网关/业务) │  ← 本篇重点
                     ├─────────────────┤
                     │  数据层          │
                     │  (DB/Cache/MQ)  │
                     └─────────────────┘
```

**本篇聚焦服务层**——当流量到达业务系统时，如何用容错手段保证可用性。

### 1.4 高并发设计的核心原则

**原则 1：容量规划先行**
- 不知道系统能扛多少 QPS，就不知道什么时候该扩容
- 用全链路压测确定系统容量上限

**原则 2：快速失败**
- 与其让用户等 30 秒超时，不如 100ms 内返回"系统繁忙"
- 快速失败释放资源，避免雪崩

**原则 3：优雅降级**
- 核心流程优先，非核心流程在压力下可降级
- 下单 > 推荐 > 用户画像

**原则 4：过载保护**
- 系统超载时，拒绝额外请求而不是照单全收
- 限流 + 排队 + 熔断

## 2. 高并发核心指标

### 2.1 QPS（每秒查询数）

**定义**：Queries Per Second——系统每秒处理的请求数量。

**计算**：
```
QPS = 并发数 / 平均响应时间（秒）
```

**示例**：Tomcat 200 线程，平均 RT=50ms，理论 QPS 上限 = 200 / 0.05 = 4000。

**QPS 的工程意义**：
- 容量规划的基准指标
- 压测目标：确定系统在给定 RT 下的最大 QPS
- 扩容决策：当前 QPS 接近容量上限时触发扩容

**典型系统的 QPS 量级**：

| 接口 | 典型 QPS | 优化手段 |
|------|---------|---------|
| 商品详情页 | 10000-100000 | CDN +  Redis 缓存 + 静态化 |
| 下单 | 1000-10000 | 异步化 + 削峰填谷 + 库存预扣 |
| 秒杀 | 100000-1000000 | 前端限流 +  排队 + 库存 Redis |
| 支付回调 | 100-1000 | 幂等 + 异步通知 + 对账 |

### 2.2 TPS（每秒事务数）

**定义**：Transactions Per Second——系统每秒处理的事务数量。

**TPS vs QPS**：一个事务可能包含多个请求。

```
用户注册流程：1 个事务 = 3 个请求（校验手机号 + 创建用户 + 发欢迎短信）
QPS = 3000（3 个请求同时计入）
TPS = 1000（1 个事务只计 1 次）
```

**工程上更关注 TPS**：因为它衡量的是"业务吞吐"，而非"接口吞吐"。

### 2.3 RT（响应时间）

**定义**：Response Time——从请求发出到收到响应的时间。

**RT 的分布比平均值重要**：
- P50（中位数）：50% 的请求在此时间内完成
- P99：99% 的请求在此时间内完成——这才是用户体验的指标
- P999：99.9%——长尾请求，通常代表异常

**案例**：某接口平均 RT=100ms，P99=200ms 看似正常。但 P999=5000ms——0.1% 的用户等了 5 秒。日活 100 万，每天 1000 个用户体验极差。

**RT 的组成**：
```
RT = 网络传输时间 + 服务端处理时间 + 数据库/缓存查询时间
```

**优化优先级**：数据库查询 > 服务端处理 > 网络传输（除非跨地域）

### 2.4 并发数

**定义**：同一时刻正在处理的请求数量。

**并发数 vs QPS**：
- QPS = 1000，RT = 10ms → 并发数 = 10（小 QPS 下的短 RT = 低并发）
- QPS = 100，RT = 500ms → 并发数 = 50（高 QT 即使 QPS 低也需高并发）

**Tomcat 线程池配置**：
```yaml
server:
  tomcat:
    threads:
      max: 200          # 最大工作线程数
      min-spare: 10     # 核心线程数
    accept-count: 100   # 当所有线程都在使用时，等待队列长度
```

**公式**：最大并发数 ≈ 最大线程数 + 等待队列长度。Tomcat 默认 = 200 + 100 = 300。

> **注**：YAML 配置路径 `server.tomcat.threads.max` 是 Spring Boot 3.x 语法（2.x 为 `server.tomcat.max-threads`）。

### 2.5 容量规划

**核心问题**：系统能扛多少 QPS？什么时候该扩容？

**容量规划三步法**：
1. **压测**：确定单机 QPS 上限（如 4000 QPS）
2. **冗余**：按 2 倍冗余规划（单机扛 4000，部署 2 台 = 8000 QPS 上限）
3. **监控**：QPS 接近上限的 70% 时触发告警，准备扩容

**容量规划的坑**：
- 压测环境 ≠ 生产环境（数据量、网络、下游依赖不同）
- 单接口压测 ≠ 混合压测（混合场景下瓶颈可能在下游）
- 平稳流量 ≠ 突发流量（秒杀场景 QPS 可能在 1 秒内从 1000 飙升到 100000）

### 2.6 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 第 1 章 4 大难题在高并发下被放大——部分失败→雪崩，网络不可靠→长尾 |
| **释慧利** | 算法 | 共识算法的性能是共识的"隐藏维度"——Paxos 多数派 ACK 延迟影响 QPS |
| **Unmesh** | 模式 | 限流模式是"过载保护"的工程实现——令牌桶/漏桶/滑动窗口 |

### 2.7 本节小结

高并发核心指标：
- **QPS**：容量规划基准（压测目标）
- **TPS**：业务吞吐（关注这个而非 QPS）
- **RT**：P99 比平均值重要（长尾代表异常）
- **并发数**：取决于 QPS × RT
- **容量规划**：压测 → 冗余 → 监控

## 3. 熔断模式（Circuit Breaker）

### 3.1 模式概述

**核心思想**：当下游服务故障时，快速切断对它的调用，让请求直接失败而不是等待超时。故障恢复后，逐步重新开放调用。

```
正常状态              熔断状态              半开状态
  │                     │                     │
  │ —— 请求 ——→         │ —— 请求 ——→         │ —— 请求（探测）——→
  │ ←—— 成功 ——         │ ←—— 立即失败 ——     │ ←—— 成功 → 恢复正常
  │                     │                     │ ←—— 失败 → 回到熔断
  │ 失败次数 > 阈值      │ 熔断时间到           │
  ↓                     ↓                     ↑
 切换到熔断             切换到半开             │
```

### 3.2 熔断器三状态

**状态 1：CLOSED（正常）**
- 请求正常通过
- 统计失败次数或慢调用比例
- 达到阈值 → 切换到 OPEN

**状态 2：OPEN（熔断）**
- 所有请求直接失败（不调用下游）
- 等待一段冷却时间（如 10 秒）
- 冷却结束 → 切换到 HALF-OPEN

**状态 3：HALF-OPEN（探测）**
- 允许少量探测请求通过（如每秒 1 个）
- 探测成功 → 切换到 CLOSED
- 探测失败 → 回到 OPEN，重新计时

### 3.3 熔断的触发条件

**触发条件 1：失败比例**
- 在统计窗口内（如 10 秒），失败请求比例超过阈值（如 50%）
- 同时满足最小请求数（如 20 个，避免冷启动误触发）

**触发条件 2：慢调用比例**
- 在统计窗口内，慢调用（RT > 阈值）比例超过阈值（如 50%）
- 下游服务没崩溃，但变得很慢 → 也需要熔断

**触发条件 3：异常数**
- 在统计窗口内，异常数量超过阈值
- 比失败比例更敏感——适合异常频率低的场景

### 3.4 熔断的工程参数

**Sentinel 的熔断规则**（Sentinel 1.8.x）：
```java
DegradeRule rule = new DegradeRule()
    .setGrade(DEGRADE_GRADE_RT)     // 慢调用比例模式
    .setCount(100)                   // RT 阈值=100ms
    .setTimeWindow(10)               // 熔断恢复时长 10 秒（注：即 recoveryTimeoutMs）
    .setMinRequestAmount(5)          // 最小请求数 5 个
    .setSlowRatioThreshold(0.5)      // 慢调用比例 50%
    .setStatIntervalMs(1000);        // 统计间隔 1 秒（控制时间窗口）
```
> **注**：`timeWindow` 和 `recoveryTimeoutMs` 是同一个参数（熔断恢复时长），不应同时设置。统计窗口由 `statIntervalMs` 控制，默认 1 秒。

**关键参数选择**：

| 参数 | Sentinel 默认 | 说明 |
|------|-------------|------|
| statIntervalMs（统计间隔） | 1000ms | 控制统计窗口，太短易抖动，太长反应慢 |
| minRequestAmount（最小请求数） | 5 | 冷启动保护，避免启动时误触发 |
| slowRatioThreshold（慢调用比例） | 1.0（100%） | 按业务调整（如 0.5=50%） |
| timeWindow（熔断恢复时长） | 0（**必须显式设置**，默认 0 会导致立即恢复） | 太短下游未恢复，太长服务不可用 |
| 熔断时长 | 5 秒 | 太短下游未恢复，太长服务不可用 |

### 3.5 熔断 vs 重试

**熔断不能替代重试，重试不能替代熔断**：

```
错误处理决策树：
  请求失败
    ├─ 是否可重试？（幂等 + 非超时错误）
    │   ├─ 是 → 重试（指数退避）
    │   └─ 否 → 跳到下一步
    └─ 熔断器是否打开？
        ├─ 是 → 直接返回失败
        └─ 否 → 调用下游 → 记录结果
```

**为什么需要熔断？**
- 重试会放大下游压力（N 个客户端同时重试 → 下游压力 ×N）
- 熔断是"放弃"——与其让下游崩溃，不如自己快速失败

### 3.6 生产案例

**案例 1：秒杀场景的熔断**
- 秒杀开始时，库存服务可能被打爆
- Sentinel 熔断器检测到库存服务 RT > 100ms + 慢调用比例 > 50%
- 触发熔断：后续请求直接返回"系统繁忙"
- 10 秒后进入半开：探测请求成功 → 恢复正常

**案例 2：支付回调的熔断**
- 支付回调需要通知订单服务更新状态
- 订单服务故障时，熔断器 OPEN
- 支付回调快速失败 → 支付平台重试 → 订单服务恢复后成功
- 如果没有熔断：支付回调等 30 秒超时 → 支付平台超时 → 用户看到"支付成功但订单未更新"

**案例 3：微服务级联故障**
- 服务 A → 服务 B → 服务 C
- 服务 C 变慢 → 服务 B 的线程被 C 的调用占满 → 服务 A 的线程被 B 的调用占满 → 全链路雪崩
- 每个服务都加熔断 → 雪崩被切断

### 3.7 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 熔断是"部分失败"的工程应对——主动放弃故障节点 |
| **释慧利** | 算法 | 共识算法也有类似"熔断"机制——Leader 超时触发选举，避免等待崩溃节点 |
| **Unmesh** | 模式 | Circuit Breaker 是分布式系统模式中的经典容错模式 |

### 3.8 本节小结

熔断是故障隔离的最后防线：
- **三状态**：CLOSED → OPEN → HALF-OPEN → CLOSED
- **触发条件**：失败比例 / 慢调用比例 / 异常数
- **关键参数**：统计窗口 / 最小请求数 / 熔断时长
- **熔断 vs 重试**：互补，非替代

## 4. 降级模式（Degradation）

### 4.1 模式概述

**核心思想**：在系统压力大或部分服务故障时，主动关闭非核心功能，把资源留给核心流程。

```
正常模式：                  降级模式：
  首页                         首页
  ├─ 推荐（核心）             ├─ 推荐 ✅
  ├─ 广告（非核心）           ├─ 广告 ❌ 降级（返回空）
  ├─ 用户画像（非核心）       ├─ 用户画像 ❌ 降级（用缓存）
  └─ 通知（非核心）           └─ 通知 ❌ 降级（延迟发送）
```

### 4.2 降级的级别

**级别 1：页面降级**
- 关闭非核心页面/模块
- 例：电商首页关闭猜你喜欢模块

**级别 2：功能降级**
- 核心功能保留，非核心功能关闭或简化
- 例：下单流程正常，但优惠券计算简化（用缓存结果）

**级别 3：数据降级**
- 用缓存/默认值替代实时数据
- 例：推荐服务故障 → 返回热门商品列表

**级别 4：完全降级**
- 整个服务不可用，返回静态页面或提示
- 例：秒杀期间超出容量 → 返回"活动太火爆，请稍后再试"

### 4.3 降级 vs 熔断

| 维度 | 熔断 | 降级 |
|------|------|------|
| 触发 | 自动（失败比例超阈值） | 手动/自动（流量/故障） |
| 粒度 | 下游调用级别 | 功能/页面级别 |
| 影响 | 被熔断的下游不可用 | 非核心功能不可用 |
| 恢复 | 自动（半开→关闭） | 手动/自动 |
| 目标 | 保护下游 | 保护自己 |

### 4.4 降级的工程实现

**方式 1：静态开关**
```yaml
degradation:
  recommend: false   # 关闭推荐
  ads: false         # 关闭广告
  coupon: cached     # 优惠券用缓存
```

**方式 2：动态开关（配置中心）**
- Nacos/Apollo 配置中心实时推送降级开关
- 运维人员通过控制台一键降级
- 不需要重启服务

**方式 3：自动降级（监控触发）**
- CPU 使用率 > 80% → 自动降级非核心功能
- QPS 超过容量 90% → 自动降级
- 下游服务熔断 → 自动降级依赖该下游的功能

### 4.5 降级的代价

**代价 1：业务损失**
- 关闭广告 → 广告收入减少
- 关闭推荐 → GMV 下降
- 需要**业务方认同降级策略**——在故障演练中验证

**代价 2：用户体验下降**
- 功能不可用或数据不准确
- 需要给用户合适的提示（而非 404）

**代价 3：降级恢复的复杂性**
- 降级后恢复时，缓存数据可能已过期
- 需要预热或重新加载

### 4.6 生产案例

**案例 1：双 11 前的降级准备**
- 双 11 流量是平时的 10 倍
- 提前准备好降级开关清单
- 核心流程：下单 → 支付 → 库存扣减（不可降级）
- 非核心流程：推荐/广告/积分/通知（可降级）
- **降级顺序**：广告 → 推荐 → 积分 → 通知（从最不核心到次核心）

**案例 2：秒杀场景的数据降级**
- 秒杀时库存服务需处理高并发
- 优惠券服务降级：用活动开始前的缓存结果
- 用户画像降级：用昨天的画像数据
- **核心原则**：库存扣减不能降级

**案例 3：支付服务的降级**
- 支付回调需要记录流水
- 正常情况下：同步写数据库 → 同步写日志中心 → 同步发通知
- 降级模式下：异步写数据库 → 异步发通知 → 日志中心降级（稍后补发）
- **降级不影响支付结果**（只影响通知和日志）

### 4.7 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | CAP——降级本质是"牺牲可用性换取部分可用性" |
| **释慧利** | 算法 | 共识算法不涉及业务降级——这是工程层的概念 |
| **Unmesh** | 模式 | — |

### 4.8 本节小结

降级是有损服务的艺术：
- **四级降级**：页面 → 功能 → 数据 → 完全
- **三种实现**：静态开关 / 动态配置中心 / 自动监控触发
- **代价**：业务损失 + 用户体验 + 恢复复杂度
- **核心原则**：核心流程不可降级，非核心流程按优先级降级

## 5. 隔离模式（Bulkhead / Isolation）

### 5.1 模式概述

**核心思想**：不同业务或不同优先级之间进行资源隔离——某个出问题不拖垮所有。

```
无隔离：                  线程池隔离：
  所有业务                  ThreadPool(核心) ← 核心业务
    ↓                        ThreadPool(非核心) ← 非核心业务
  共用一个线程池              ThreadPool(其他) ← 其他业务
    ↓
  非核心业务占满线程
    ↓
  核心业务无法处理 → 全系统不可用
```

> **注**：Netflix Hystrix 已于 2018 年停止维护，Spring Cloud 已移除。以下以 **Sentinel** 为主线（并发线程数限流，信号量语义），线程池隔离以 **Resilience4j Bulkhead** 为补充。

### 5.2 隔离的级别

**级别 1：线程池隔离**
- 不同业务用独立线程池
- Sentinel 资源隔离通过并发线程数限流实现（类似信号量语义，非独立线程池）
- 例：下单线程池（max=100）、推荐线程池（max=20）

**级别 2：信号量隔离**
- 通过并发计数器限制同时执行的数量
- Sentinel 原生支持：通过并发线程数限流实现信号量语义（`FlowRule.setGrade(RuleConstant.FLOW_GRADE_THREAD)`）
- 线程池隔离可使用 Resilience4j 的 Bulkhead 模式（`BulkheadConfig.builder().maxConcurrentCalls(n)`）
- 例：库存扣减并发数 ≤ 200

**级别 3：进程隔离**
- 核心服务独立部署，不与非核心服务共享资源
- 例：支付服务独立集群

**级别 4：物理隔离**
- 核心服务和普通服务部署在不同物理机/机房
- 例：金融核心交易独立数据中心

### 5.3 线程池隔离 vs 信号量隔离

| 维度 | 线程池隔离 | 信号量隔离 |
|------|----------|----------|
| 原理 | 独立线程池 | 计数器限制并发 |
| 是否异步 | 是（另开线程） | 否（调用线程内执行） |
| 上下文传递 | 需要处理 ThreadLocal | 天然支持（同一线程） |
| 额外开销 | 线程切换开销 | 无额外开销 |
| 超时控制 | 支持（thread timeout） | 不支持（调用线程超时） |
| 熔断配合 | 超时 + 熔断 | 熔断（无超时） |
| 典型框架 | Resilience4j Bulkhead（线程池模式） | Sentinel（并发线程数限流，信号量语义） |

### 5.4 生产案例

**案例 1：微博热点下的隔离**
- 某明星离婚，相关话题 QPS 飙升
- 推荐线程池被相关请求打满
- **如果没有隔离**：发微博也受影响 → 全站不可用
- **有线程池隔离**：推荐线程池满，返回降级结果；核心业务（发微博/看首页）不受影响

**案例 2：电商大促的隔离**
- 下单服务：线程池 max=500
- 推荐服务：线程池 max=50
- 积分服务：线程池 max=20
- 推荐服务被打满 → 推荐降级（返回热门商品）
- **下单不受影响**——这才是隔离的价值

### 5.5 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 隔离是"异构故障"的工程应对——不让慢节点拖垮快节点 |
| **释慧利** | 算法 | — |
| **Unmesh** | 模式 | — |

### 5.6 本节小结

隔离是不把鸡蛋放一个篮子里：
- **四级隔离**：线程池 / 信号量 / 进程 / 物理
- **线程池 vs 信号量**：前者异步+超时，后者同步+高效
- **核心原则**：核心与非核心资源隔离，一个出问题不拖垮全部

## 6. 限流模式（Rate Limiting）

### 6.1 模式概述

**核心思想**：限制系统的请求速率，超过阈值的请求直接拒绝（不处理），保护系统不过载。

```
请求流入
    │
    ▼
┌──────────┐
│ 限流器    │ ← 检查：当前 QPS 是否超过阈值？
└──────────┘
    │
    ├── 未超过 → 正常处理
    │
    └── 超过 → 直接拒绝（429 Too Many Requests）
```

### 6.2 三种限流算法

**算法 1：令牌桶（Token Bucket）**

```
令牌以固定速率放入桶中（如 100 个/秒）
桶的容量有上限（如 200 个）
请求到达时：
  有令牌 → 取走一个令牌，处理请求
  无令牌 → 拒绝请求
```

**优势**：允许突发流量（桶中积攒的令牌可一次性消耗）
**典型实现**：Guava RateLimiter / Sentinel

**算法 2：漏桶（Leaky Bucket）**

```
请求到达时进入桶中
桶以固定速率"漏水"（处理请求）
桶满时新请求被拒绝
```

**优势**：输出速率恒定，适合保护下游
**劣势**：不支持突发（桶满后所有超出请求一律拒绝）

**算法 3：滑动窗口（Sliding Window）**

```
维护一个时间窗口（如 1 秒）
统计窗口内的请求数
超过阈值 → 拒绝
窗口向前滑动，过期数据丢弃
```

**优势**：精确控制，比固定窗口更平滑
**典型实现**：Redis + Lua 脚本

### 6.3 限流的粒度

| 粒度 | 示例 | 实现 |
|------|------|------|
| **全网** | 整个系统 10000 QPS | Nginx / 网关 |
| **服务** | 订单服务 5000 QPS | Sentinel |
| **接口** | 下单接口 2000 QPS | Sentinel |
| **用户** | 每个用户 10 QPS | Redis + 滑动窗口 |
| **IP** | 每个 IP 100 QPS | Nginx + Redis |
| **设备** | 每个设备 50 QPS | Redis |

### 6.4 限流 vs 熔断 vs 降级

| 手段 | 触发时机 | 行为 | 目标 |
|------|---------|------|------|
| **限流** | 请求超过阈值 | 拒绝超出请求 | 保护系统不过载 |
| **熔断** | 下游故障 | 快速失败 | 停止调用故障下游 |
| **降级** | 容量不足/故障 | 关闭非核心功能 | 保证核心流程 |

### 6.5 生产案例

**案例 1：秒杀的限流**
- 秒杀开始前，系统容量已知：库存服务 5000 QPS
- 网关层限流：超过 5000 QPS 直接拒绝
- 被拒绝的用户看到"活动太火爆，请稍后再试"
- **保护库存服务不被打崩**——宁可拒绝用户，不可系统崩溃

**案例 2：API 网关的限流**
- 每个租户 1000 QPS
- 使用 Redis + 滑动窗口实现
- 超限的请求返回 429 + Retry-After header
- 客户端根据 Retry-After 自动退避

**案例 3：短信服务的限流**
- 短信服务有调用次数限制（如 1000 QPS）
- 超过限制会被运营商处罚
- 令牌桶限制：1000 令牌/秒
- 突发流量（营销活动）：令牌桶可积攒 200 个令牌，允许 1200 QPS 的短暂突发

### 6.6 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 限流是"过载保护"的工程实现——控制请求速率，防止雪崩 |
| **释慧利** | 算法 | — |
| **Unmesh** | 模式 | Rate Limiting 是系统保护的第一道防线 |

### 6.7 本节小结

限流是过载保护的第一道防线：
- **三种算法**：令牌桶（支持突发）/ 漏桶（平滑）/ 滑动窗口（精确）
- **六级粒度**：全网 → 服务 → 接口 → 用户 → IP → 设备
- **限流 vs 熔断 vs 降级**：分别是拒绝超量/拒绝故障下游/关闭非核心

## 7. 容错模式的组合与权衡

### 7.1 四种容错手段的关系

```
请求到达
   │
   ▼
┌──────┐  限流：是否超过系统容量？
│ 限流  │  ├─ 是 → 拒绝（429）
└──────┘  └─ 否 → 继续
   │
   ▼
┌──────┐  熔断：下游是否故障？
│ 熔断  │  ├─ 是 → 快速失败
└──────┘  └─ 否 → 继续
   │
   ▼
┌──────┐  隔离：资源是否隔离？
│ 隔离  │  ├─ 核心业务 → 独立线程池
└──────┘  └─ 非核心业务 → 共享线程池
   │
   ▼
┌──────┐  降级：是否需要降级？
│ 降级  │  ├─ 是 → 返回降级结果
└──────┘  └─ 否 → 正常返回
```

### 7.2 真实系统的容错组合

| 系统 | 限流 | 熔断 | 降级 | 隔离 |
|------|------|------|------|------|
| **电商下单** | 接口级限流 | 库存/支付熔断 | 推荐降级 | 核心/非核心隔离 |
| **API 网关** | 租户级限流 | 下游服务熔断 | 非核心路由降级 | — |
| **秒杀** | 全网限制 | 库存服务熔断 | 优惠券降级 | 秒杀独立集群 |
| **支付回调** | — | 订单服务熔断 | 日志异步化 | 支付独立部署 |

### 7.3 容错模式的代价

| 手段 | 代价 |
|------|------|
| 限流 | 拒绝用户请求 → 体验差 → 业务损失 |
| 熔断 | 下游不可用 → 功能缺失 → 体验下降 |
| 降级 | 非核心功能缺失 → 功能不完整 |
| 隔离 | 资源浪费（独立线程池/集群） → 成本增加 |

**核心权衡**：容错手段不是免费的——每种手段都有代价，需要根据业务目标选择组合。

### 7.4 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 4 大难题的工程应对——限流/熔断/降级/隔离分别对应过载/故障/不足/异构 |
| **释慧利** | 算法 | — |
| **Unmesh** | 模式 | — |

### 7.5 本节小结

容错模式的组合：
- **顺序**：限流（第一道）→ 熔断（第二道）→ 隔离（第三道）→ 降级（第四道）
- **每种手段都有代价**——根据业务目标选择组合
- **没有银弹**——不同场景需要不同组合

## 结语：容错是系统高可用的基石

回顾本篇，我们完成了高并发系统容错手段的完整推导：

- **第 1 节**：量变引起质变——每个 QPS 量级对应不同的架构选择
- **第 2 节**：QPS/TPS/RT/并发数——容量规划的四要素
- **第 3 节**：熔断——故障隔离的最后防线
- **第 4 节**：降级——有损服务的艺术
- **第 5 节**：隔离——不把鸡蛋放一个篮子
- **第 6 节**：限流——过载保护的第一道防线
- **第 7 节**：四种容错手段的组合与权衡

**第一篇的核心论断**：高并发系统设计不是"选最牛的方案"，而是"为当前的 QPS 量级选最合适的方案"。限流/熔断/降级/隔离是四种容错手段，每种都有代价——没有银弹，只有权衡。
