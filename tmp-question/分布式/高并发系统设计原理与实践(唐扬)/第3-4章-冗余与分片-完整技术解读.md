# 高并发系统设计原理与实践 · 第二篇 — 冗余与分片

> **本篇定位**：第一篇讲了系统容错（熔断/降级/隔离/限流）——这是"被动防御"：系统出问题时如何止损。本篇讲的是"主动建设"：如何通过冗余和分片让系统在正常状态下就能扛住高并发。

## 1. 为什么需要冗余？

### 1.1 冗余是抵抗"部分失败"的唯一手段

第一篇讲了系统容错——当故障发生时，用熔断/降级/限流止损。但有个前提：**系统不能因为一个故障节点就整体不可用**。这就是冗余的价值——当某个节点挂了，其他节点顶上。

唐伟志书第 1 章提出了分布式系统的 4 大难题，其中"部分失败"是本质特征。冗余是抵抗部分失败、保证服务持续可用的基础手段：

- 单节点：故障 = 系统不可用
- 2 节点：故障 = 50% 概率不可用（取决于哪个节点故障 + 故障切换机制）
- 3 节点（多副本）：故障 = 系统仍可用（多数派存活）

**冗余的数学基础**（唐伟志书第 3 章 4 节）：
- N 个副本，容忍 f 个故障：N ≥ 2f+1
- 3 副本容忍 1 故障，5 副本容忍 2 故障

### 1.2 冗余的两个维度

**维度 1：数据冗余**
- 同一份数据存多个副本
- 副本间的同步方式决定了一致性级别（强一致 / 最终一致）

**维度 2：服务冗余**
- 同一个服务部署多个实例
- 负载均衡在实例间分发请求

两者通常组合使用——每个数据副本由一个服务实例管理：

```
                 负载均衡
                    │
    ┌───────────────┼───────────────┐
    ↓               ↓               ↓
┌──────┐       ┌──────┐       ┌──────┐
│服务A_1│      │服务A_2│      │服务A_3│
│DB(主) │      │DB(从) │      │DB(从) │
└──────┘       └──────┘       └──────┘
```

### 1.3 冗余的度量

| 指标 | 定义 | 计算 |
|------|------|------|
| **可用性** | 系统正常运行时间的比例 | 正常运行时间 / 总时间 |
| **故障容忍** | 系统能容忍的最大同时故障数 | f = ⌊(N-1)/2⌋（多数派） |
| **复制因子** | 数据副本数 | N（通常 3 或 5） |

**可用性级别**：

| 级别 | 可用性 | 年宕机时间 | 典型架构 |
|------|--------|----------|---------|
| 单节点 | 99% | 3.65 天 | 单机部署 |
| 双节点 | 99.9% | 8.76 小时 | 主备 |
| 三节点 | 99.99% | 52.6 分钟 | 同城跨 AZ 三副本 |
| 五节点 | 99.999% | 5.26 分钟 | 异地多活 |

## 2. 多副本冗余

### 2.1 同机房多副本

**架构**：同一机房内部署 3 个副本。这是最简单的冗余模式。

```
      负载均衡
         │
  ┌──────┼──────┐
  │      │      │
  ▼      ▼      ▼
副本1   副本2   副本3
(同一机房的 3 台机器)
```

**适用场景**：初创公司、小规模服务（QPS < 1000）

**容灾能力**：
- ✅ 容忍单节点故障（任意 1 台机器宕机，多数派存活）
- ❌ 无法对抗机房级故障（机房断电/断网 → 全宕）

**实现方式**：
- 数据库：MySQL 主从复制 / MGR
- 缓存：Redis 主从 + Sentinel
- 应用：Nginx + 多实例

### 2.2 同城跨可用区多副本

**架构**：同一城市的 3 个可用区（AZ）各部署一个副本。

```
         全球 DNS
            │
    ┌───────┼───────┐
    ▼       ▼       ▼
 AZ-1    AZ-2    AZ-3
(副本1)  (副本2)  (副本3)
  ←───── 同一城市 ──────→
```

**适用场景**：中等规模服务（QPS 1000-10000），需要一定的容灾能力

**容灾能力**：
- ✅ 容忍单节点故障
- ✅ 容忍单 AZ 故障（机房级故障）
- ❌ 无法对抗城市级故障（地震/大规模断电）

**关键约束**：跨 AZ 延迟 < 5ms（多数派 ACK 延迟可控）

**实现方式**：
- 数据库：MySQL MGR / TiKV 多副本
- 缓存：Redis Cluster
- 应用：K8s 跨 AZ 调度

### 2.3 同城 vs 同机房的抉择

| 维度 | 同机房 | 同城跨 AZ |
|------|--------|----------|
| 延迟 | < 0.5ms | 1-5ms |
| 容灾 | 节点级 | 机房级 |
| 成本 | 低（同机柜） | 中（跨 AZ 带宽） |
| 复杂度 | 低 | 中（跨 AZ 部署） |
| 适用 | QPS < 1000 | QPS 1000-10000 |

### 2.4 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 第 3 章 多副本是数据复制的基础——主从/多主/无主 |
| **Unmesh** | 模式 | 第一篇 主从复制是冗余的标准实现 |
| **释慧利** | 算法 | 共识算法保证多副本的一致性和自动故障切换 |

### 2.5 本节小结

多副本冗余是高可用的基石：
- **同机房**：3 副本，容忍 1 故障，延迟 < 0.5ms
- **同城跨 AZ**：3 副本分布 3 个 AZ，容忍机房级故障，延迟 1-5ms
- **核心约束**：多数派 ACK 延迟决定写延迟

## 3. 同城双活

### 3.1 模式概述

**核心思想**：同一城市的两个数据中心同时提供服务，流量按比例分配到两个中心。

```
            全局 DNS
          (按比例分流)
              │
    ┌─────────┴─────────┐
    │       60%         │       40%
    ▼                   ▼
 数据中心 A          数据中心 B
 ┌─────────┐       ┌─────────┐
 │ 副本 A1  │       │ 副本 B1  │
 │ 副本 A1' │       │ 副本 B2  │
 │  ...     │       │  ...     │
 └─────────┘       └─────────┘
    ↕ (同步复制)       ↕
 ┌─────────────────────────────┐
 │         共享存储 / 同步      │
 └─────────────────────────────┘
```

### 3.2 同城双活的两个模式

**模式 1：Active-Active（全双活）**
- 两个中心都接受写入
- 写入通过跨中心同步保持一致
- 流量按比例分配（如 50:50）

**模式 2：Active-Standby（主备）**
- 主中心接受所有写入
- 备中心接受只读流量
- 主中心故障时，备中心接管写入

**生产上更倾向 Active-Active**：充分发挥两个中心的算力，不浪费资源。

### 3.3 同城双活的工程挑战

**挑战 1：数据同步**
- 跨数据中心延迟 < 5ms（同城双活的前提）
- 强一致：Paxos/Raft 多数派 ACK——3 副本中有 2 个在数据中心 A、1 个在数据中心 B，写入需等两地 ACK
- 最终一致：主中心写、备中心异步同步——备中心可能读到旧数据

**挑战 2：流量分配**
- DNS 按比例分流（60:40 或 50:50）
- 数据中心故障时，自动切换到健康中心
- DNS TTL 影响切换速度（TTL=60s → 最长 60 秒完成全量切换）

**挑战 3：缓存一致性**
- 两个中心的 Redis 缓存可能不一致
- 用失效通知（pub/sub）或双写保证同步
- 同城延迟低（< 5ms），同步代价可接受

### 3.4 生产案例

**案例 1：电商系统同城双活**
- 北京两个数据中心（亦庄 + 望京），延迟 3-5ms
- Active-Active：两个中心都提供完整服务
- 数据库：TiDB 5 副本（3+2 分布）
- 缓存：Redis Sentinel + 主从（跨 AZ 多副本；同城双活场景下用双写失效通知保持缓存一致）
- 流量分配：DNS 50:50

**案例 2：支付系统同城主备**
- 主中心处理所有支付请求
- 备中心处理查询请求 + 异步备份
- 主中心故障 → 备中心接管（需人工确认）
- 数据库：MySQL MGR 3 副本（主中心 2+ 备中心 1）

### 3.5 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 同城双活是 CAP 的权衡——C 或 A 的选择取决于复制方式 |
| **Unmesh** | 模式 | 同城双活是多主复制的工程应用 |
| **释慧利** | 算法 | Paxos/Raft 的多数派机制决定了同城双活的副本分布策略 |

### 3.6 本节小结

同城双活是中等规模系统的高可用方案：
- **Active-Active**：两个中心都接受写入，充分发挥算力
- **数据同步**：同城延迟 < 5ms 使同步复制可行
- **流量分配**：DNS 分流，TTL 影响切换速度
- **容灾能力**：容忍单数据中心故障

## 4. 异地多活

### 4.1 模式概述

**核心思想**：多个城市的数据中心同时提供服务，用户就近访问，数据在多地同步。

```
            全局 DNS
          (就近接入)
              │
    ┌─────────┼─────────┐
    │  北京   │   上海   │   深圳
    ▼         ▼         ▼
 数据中心A  数据中心B  数据中心C
 (北方用户) (华东用户) (南方用户)
    ↕         ↕         ↕
    └───────────────────┘
         (跨地域同步)
```

### 4.2 异地多活的核心难题

**难题 1：跨地域延迟**
- 北京→上海：30ms
- 北京→深圳：40ms
- 中国→美国：200ms

**强一致同步不可行**：多数派 ACK 延迟 = 最慢副本的延迟。跨地域 Paxos/Raft 写入延迟 100-200ms，用户体验不可接受。

**难题 2：数据冲突**
- 北京用户和上海用户同时修改同一数据
- 冲突解决：LWW / 向量时钟 / 业务层合并

**难题 3：回源策略**
- 用户在北京注册，去上海登录 → 上海数据中心没有此用户数据
- 需要回源到北京查询

### 4.3 异地多活的实现模式

**模式 1：单元化（最常见）**
- 按用户 ID 哈希，每个用户固定路由到一个数据中心
- 跨数据中心的用户数据不冲突（同一用户只在一个中心修改）
- 适合电商/社交（用户数据天然可分片）

**模式 2：最终一致 + 冲突解决**
- 所有数据中心都可修改数据
- 跨中心异步同步 + 冲突解决
- 适合协作办公/笔记（多用户协作）

**模式 3：全局强一致（成本最高）**
- 用 Paxos + TrueTime 实现跨地域强一致（**注**：TrueTime ε<7ms 仅在单个大陆内有效，实际 Spanner 部署在单大陆范围内，跨洲通常用异步复制）
- 只有 Spanner / CockroachDB 等能做到
- 适合金融/支付（一致性要求极高）

### 4.4 单元化的工程实现

**用户路由规则**：
```
数据中心 = hash(user_id) % 数据中心数
```

**单元化路由示例**：
```
user_001 → hash % 3 = 0 → 北京
user_002 → hash % 3 = 1 → 上海
user_003 → hash % 3 = 2 → 深圳
```

**跨单元操作**：
- 大部分操作：用户访问自己的数据中心（本地读写，延迟低）
- 跨单元操作（如 A 给 B 转账）：调跨单元 RPC，用分布式事务

**单元化 + 同城双活组合**：
```
北京单元：Active-Active（亦庄 + 望京）
上海单元：Active-Active（浦东 + 松江）
深圳单元：Active-Active（南山 + 福田）
```

### 4.5 生产案例

**案例 1：阿里/蚂蚁异地多活**
- 杭州/上海/深圳/北京 等多地
- 单元化：按用户 ID 路由
- 同城双活 + 异地多活组合
- 数据库：OceanBase（Paxos 跨地域多副本）

**案例 2：微信异地多活**
- 广州/深圳/上海/天津 等多地
- 朋友圈数据按用户 ID 单元化
- 消息数据：用户所在单元写入，接收方单元同步

**案例 3：全球部署（Google Spanner）**
- 北美/欧洲/亚洲 多副本
- Paxos 跨地域强一致
- TrueTime 保证外部一致性
- 写入延迟 50-200ms（跨地域多数派 ACK）

### 4.6 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 第 1 章 CAP——异地多活是 CP vs AP 的极端权衡 |
| **Unmesh** | 模式 | 第二篇 一致性哈希 + 多主复制为异地多活提供分区和同步基础 |
| **释慧利** | 算法 | Spanner 的 Paxos + TrueTime 是跨地域强一致的理论巅峰 |

### 4.7 本节小结

异地多活是大型系统的最高可用架构：
- **跨地域延迟是核心瓶颈**：多数派强一致在跨地域不可行
- **单元化最常见**：按用户 ID 路由，避免跨地域冲突
- 同城双活 + 异地多活组合形成完整容灾体系

## 5. 数据分片

### 5.1 模式概述

**核心思想**：把一张大表拆成多张小表，分散到多个数据库实例上。

```
分片前：                          分片后：
┌───────────────┐        ┌───────┬───────┬───────┐
│  orders (1 亿行)│        │orders_1│orders_2│orders_3│
└───────────────┘        │(0-3000w)│(3000w) │(6000w) │
   单库 QPS 上限 5000     └───────┴───────┴───────┘
                         每个分片 QPS 上限 5000 → 总 QPS 15000
```

### 5.2 分片的两种维度

**垂直拆分**（按列/表）：
- 把不同表分到不同库
- 例：user_db（用户相关）、order_db（订单相关）、product_db（商品相关）
- 优势：不同业务独立扩展
- 劣势：跨库 JOIN 不可用

**水平拆分**（按行）：
- 同一张表按某个字段拆分到多个库
- 例：orders 表按 user_id 分成 16 个库
- 优势：单表容量线性扩展
- 劣势：跨分片查询复杂

**通常会两者结合**：先垂直拆（按业务），再水平拆（按数据量）。

### 5.3 分片键的选择

**选择分片键的 4 个考量**：

| 考量 | 说明 | 举例 |
|------|------|------|
| **均匀分布** | 数据均匀分布到各分片 | user_id 哈希 |
| **查询模式** | 大部分查询带分片键 | 电商场景大部分查询都带 user_id |
| **避免跨分片** | 关联表用同一个分片键 | orders 和 order_items 都用 order_id 分片 |
| **避免热点** | 单个分片不过度集中 | 大 V 的订单集中在一个分片 → 用子分片 |

### 5.4 分片的工程实现

**方案 1：JDBC 层代理（ShardingSphere-JDBC）**
```yaml
sharding:
  tables:
    orders:
      actualDataNodes: ds$->{0..15}.orders_$->{0..15}
      databaseStrategy:
        standard:
          shardingColumn: user_id
          algorithmExpression: ds_${user_id % 16}
      tableStrategy:
        standard:
          shardingColumn: order_id
          algorithmExpression: orders_${order_id % 16}
```

**方案 2：代理层（ShardingSphere-Proxy / Vitess）**
- Proxy 层解析 SQL、路由到正确分片
- 业务无感知（连接 Proxy 和连接单库一样）
- 瓶颈：Proxy 本身成为性能瓶颈

**方案 3：分布式数据库（TiDB / CockroachDB）**
- 数据库原生支持分片（Region / Range）
- 自动分裂 + 迁移
- 跨分片事务原生支持

### 5.5 分片的代价

| 代价 | 说明 |
|------|------|
| **跨分片 JOIN** | 需要应用层合并结果 |
| **跨分片事务** | 需要分布式事务（Seata / Percolator） |
| **分片扩容** | 从 16→32 分片需迁移 50% 数据 |
| **全局唯一 ID** | 不能依赖数据库自增 ID |
| **查询复杂度** | 非分片键查询需扫所有分片 |

### 5.6 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 第 3 章 5 节 分区策略——哈希/范围/一致性哈希 |
| **Unmesh** | 模式 | 第二篇 四种分区模式（固定/动态/一致性哈希/虚拟槽） |
| **释慧利** | 算法 | — |

### 5.7 本节小结

数据分片是横向扩展的核心手段：
- **垂直拆分**：按业务分库
- **水平拆分**：按数据量分表
- **分片键选择**：均匀 + 查询模式 + 避免跨分片 + 避免热点
- **工程实现**：ShardingSphere / Vitess / TiDB

## 6. 服务分片

### 6.1 模式概述

**核心思想**：同一个服务部署多个实例，但每个实例只处理一部分请求（按一致性哈希路由），而非随机分配。

```
普通负载均衡：          服务分片（一致性哈希）：
  请求 → LB → 随机选实例     请求 → hash(user_id) → 固定实例
```

### 6.2 服务分片 vs 无状态服务

| 维度 | 无状态服务 | 服务分片 |
|------|----------|---------|
| 路由 | 随机/轮询 | 一致性哈希（同一用户到同一实例） |
| 实例状态 | 无状态 | 有状态（缓存/连接池） |
| 扩展 | 随意扩缩容 | 需考虑数据迁移 |
| 故障 | 自动切换 | 哈希环重新分配 |

### 6.3 服务分片的典型场景

**场景 1：长连接推送**
- WebSocket 连接绑定到特定实例
- 一致性哈希保证同一用户连接到同一实例
- 单实例故障 → 连接断开，客户端重连到其他实例

**场景 2：本地缓存加速**
- 服务实例本地缓存用户数据
- 一致性哈希保证同一用户请求命中同一实例的缓存
- 实例重启 → 缓存丢失，从数据库重新加载

**场景 3：IM 消息路由**
- 用户在线状态绑定到特定实例
- 消息发给指定用户时，路由到该用户绑定的实例
- 一致性哈希 + 心跳检测

### 6.4 生产案例

**案例 1：微博的 Feed 流**
- 用户关注列表存储在 Redis Cluster
- Feed 生成服务按 user_id 一致性哈希路由
- 同一用户总是命中同一实例，本地缓存关注列表

**案例 2：IM 系统的长连接**
- WebSocket 网关按 user_id 一致性哈希
- 同一用户的连接固定在一台网关
- 网关故障 → 客户端重连 → 重新哈希到其他网关

|------|------|---------|
| **唐伟志** | 理论 | 第 3 章 5.4 节 一致性哈希——最小迁移 |
| **Unmesh** | 模式 | 第二篇 一致性哈希——服务分片的工程应用 |
| **释慧利** | 算法 | — |

### 6.6 本节小结

服务分片让"有状态"成为可能：
- **一致性哈希**：同一用户到同一实例
- **典型场景**：长连接 / 本地缓存 / IM 路由
- **工程代价**：扩缩容需考虑状态迁移

## 7. 冗余与分片的组合

### 7.1 组合矩阵

冗余（纵向）和分片（横向）是正交的，几乎所有生产系统都同时使用两者。

| 架构 | 冗余模式 | 分片模式 | QPS 上限 | 典型系统 |
|------|---------|---------|---------|---------|
| 单机 | 无 | 无 | ~1000 | 小型项目 |
| 主从 | 多副本 | 无 | ~5000 | MySQL 主从 + 读写分离 |
| 同城双活 + 分片 | 双活 | 水平分片 | ~50000 | 中型电商 |
| 异地多活 + 分片 | 异地 | 单元化 | ~500000 | 大厂 |
| 全球多活 | Spanner 级 | 自动分片 | ~1000000 | Google |

### 7.2 真实系统的组合架构

**架构演进路径**：

```
单机部署
   ↓ QPS 增长，加多副本
主从复制（同机房 3 副本）
   ↓ QPS 再增长，加同城双活
同城双活（2 数据中心 Active-Active）
   ↓ 数据量增长，加分片
同城双活 + 数据分片（TiDB/MySQL MGR）
   ↓ 业务全球化，加异地多活
异地多活 + 单元化（阿里/蚂蚁模式）
```

### 7.3 组合的工程考量

**考量 1：冗余越多，成本越高**
- 同机房 3 副本：3 倍硬件成本
- 同城双活：2 倍 + 跨 AZ 带宽成本
- 异地多活：3 倍 + 跨地域带宽成本
- QPS < 1000 时，冗余不是瓶颈，业务逻辑才是

**考量 2：分片越多，复杂度越高**
- 单库：所有 SQL 都能用
- 分片后：跨分片 JOIN/事务不可用
- 分布式 ID + 分布式事务 + 分片扩容都有代价

**考量 3：优先冗余，再分片**
- 冗余解决高可用（主从复制比单机多了容错）
- 分片解决高扩展（主从复制后 QPS 还不够，才分片）
- 不要一上来就"同城双活 + 分片"——复杂度会拖垮团队

### 7.4 跨书视角对照

| 作者 | 视角 | 核心论点 |
|------|------|---------|
| **唐伟志** | 理论 | 冗余 + 分片 = 分布式数据系统的基础架构 |
| **Unmesh** | 模式 | 第一篇 数据复制模式 + 第二篇 分区模式——本篇的工程组合 |
| **释慧利** | 算法 | 共识算法是冗余强一致的基础——Paxos/Raft 保证多副本一致 |

### 7.5 本节小结

冗余与分片的组合：
- **冗余优先，分片其次**：先解决高可用，再解决高扩展
- **成本递增**：每增加一层架构都需要额外硬件和运维成本
- **选择匹配 QPS 量级**：1000 QPS 不必上同城双活，100 万 QPS 必须上异地多活

## 结语：从被动防御到主动建设

回顾本篇，我们完成了冗余与分片的完整推导：

- **第 1 节**：冗余是抵抗部分失败、保证服务持续可用的基础手段
- **第 2 节**：多副本冗余——同机房（节点级容灾）→ 同城跨 AZ（机房级容灾）
- **第 3 节**：同城双活——Active-Active，两个中心同时服务
- **第 4 节**：异地多活——单元化，按用户 ID 路由
- **第 5 节**：数据分片——垂直拆分 + 水平拆分
- **第 6 节**：服务分片——一致性哈希，有状态服务
- **第 7 节**：冗余与分片的组合——优先冗余，再分片

**第二篇的核心论断**：冗余解决高可用（节点故障不影响服务），分片解决高扩展（数据量线性增长）。两者正交组合构成分布式系统的标准架构。从单机到主从到同城双活到异地多活——每增加一层架构，可用性和容量都提升一个量级，但成本和复杂度也在增加。**选择匹配 QPS 量级的架构，而不是盲目追求"最牛"的方案**。
