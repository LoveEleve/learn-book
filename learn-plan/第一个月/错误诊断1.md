# 错误诊断实验 #1

> **目标**：给定故障场景，在限定时间内独立定位问题。这是模拟面试"遇到过什么问题"的最佳训练。
>
> **范围**：覆盖第一个月+第二个月核心知识——内存管理、网络、C++对象模型、G1 GC、故障排查。

## 实验1：内存泄漏定位（40min）

**场景**：你接手一个 C 程序（TCP 代理服务），运行 30 分钟后内存占用从 50MB 增长到 500MB，最终 OOM 被 kill。

**你的任务**：

1. **用 valgrind 定位泄漏点**（15min）
   ```bash
   # 使用 valgrind memcheck
   valgrind --leak-check=full --show-leak-kinds=all ./tcp_proxy &
   # 产生流量
   for i in $(seq 1 100); do curl -s http://localhost:9000/ > /dev/null; done
   kill %1
   # 分析 valgrind 输出：
   # - definitely lost vs indirectly lost vs possibly lost 的区别？
   # - 泄漏栈追踪指向哪个函数？
   ```

2. **用 AddressSanitizer 复现**（10min）
   ```bash
   # 编译时加入 ASan
   gcc -fsanitize=address -g tcp_proxy.c -o tcp_proxy_asan
   ./tcp_proxy_asan
   # 观察 ASan 的错误报告格式
   ```

3. **分析根因并修复**（15min）
   - 常见的 C 内存泄漏模式：忘记 free()、错误路径不释放、循环中重复 malloc
   - 写出修复方案（加入 RAII 包装、使用智能指针类比）
   - 写出你将在面试中如何讲述这个案例（STAR 法则）

**关联知识**：第一个月/第一周/day0-C语言基础.md、第一个月/第三周/day1-2-物理内存.md（伙伴系统/SLAB 分配器与 malloc 的关系）

## 实验2：TCP 连接超时诊断（40min）

**场景**：Nginx 反向代理到后端的 Reactor 服务器，偶尔出现 502 Bad Gateway 错误。后端服务器 CPU 和内存正常。

**你的任务**：

1. **用 tcpdump 抓包分析**（15min）
   ```bash
   # 抓到 502 错误时的网络包
   tcpdump -i lo -nn -s 0 -w /tmp/502_issue.pcap 'port 8080 or port 80' &
   # 模拟请求，等待 502 出现
   ab -n 100 -c 10 http://localhost/
   kill %1
   
   # 分析抓包
   tcpdump -r /tmp/502_issue.pcap -nn
   # 关注：
   # - 是否有 SYN 重传（连接建立慢）
   # - 是否有 FIN 异常的提前关闭
   # - TIME_WAIT 状态连接数
   ```

2. **用 strace 定位后端问题**（10min）
   ```bash
   strace -f -e trace=accept,read,write,close -p $(pgrep reactor_server) &
   # 重新测试
   # 观察系统调用是否正常返回
   ```

3. **给出诊断结论和修复方案**（15min）
   - 可能原因：线程池耗尽、keepalive 超时配置不当、TIME_WAIT 堆积
   - 修复方案：调整 Nginx proxy_read_timeout、后端启用 SO_REUSEADDR、增加线程池大小

**关联知识**：第一个月/第四周/day4-5-Reactor.md、第一个月/第三周/day5-网络层.md（TCP 状态机/TIME_WAIT）

## 实验3：C++ 程序段错误定位（40min）

**场景**：一个多线程 C++ 服务，运行中随机出现 Segmentation fault，core dump 文件已保存。

**你的任务**：

1. **用 GDB 分析 core dump**（15min）
   ```bash
   # 加载 core dump
   gdb ./cpp_service core.12345
   
   # GDB 命令
   (gdb) bt full           # 查看完整调用栈
   (gdb) info threads      # 查看所有线程
   (gdb) thread apply all bt  # 所有线程的调用栈
   (gdb) frame 0           # 查看崩溃帧
   (gdb) info locals       # 查看局部变量
   (gdb) x/4gx $rsp        # 查看栈内存
   ```

2. **判断崩溃类型**（10min）
   - 空指针解引用：`this = 0x0`，vptr 为 NULL
   - 野指针/use-after-free：指针指向已释放的内存
   - 栈溢出：调用栈极深，递归过深或栈内存分配过大
   - 缓冲区溢出：`strcpy` 越界

3. **定位根因并修复**（15min）
   - 虚拟函数调用 + 对象已析构 = vptr 被覆盖 → 破窗效应
   - 多线程访问共享对象无锁保护 → 加入 RAII 锁
   - 写出面试叙事：从 core dump → GDB 定位 → 分析根因 → 修复验证

**关联知识**：第一个月/第四周/day1-2-对象模型上.md（vptr/vtable）、第一个月/第四周/day3-对象模型下.md（析构函数中 vptr 变化）
