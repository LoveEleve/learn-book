## 实验1：MySQL 慢查询诊断（40min）

**场景**：线上订单系统某接口 P99 延迟从 50ms 飙升至 2000ms。监控显示数据库 CPU 使用率 85%，磁盘 IO 正常。

**你的任务**：

1. **用慢查询日志定位问题 SQL**（10min）
   ```sql
   -- 开启慢查询日志
   SET GLOBAL slow_query_log = ON;
   SET GLOBAL long_query_time = 0.1;  -- 100ms 阈值
   
   -- 分析慢查询
   -- 使用 pt-query-digest 或直接查看
   SELECT * FROM mysql.slow_log ORDER BY query_time DESC LIMIT 10;
   ```

2. **用 EXPLAIN 分析执行计划**（15min）
   ```sql
   EXPLAIN SELECT o.*, u.name, p.title
   FROM orders o
   JOIN users u ON o.user_id = u.id
   JOIN products p ON o.product_id = p.id
   WHERE o.created_at > '2026-01-01'
   ORDER BY o.created_at DESC
   LIMIT 20;
   ```
   
   **分析要点**：
   - type 列：ALL（全表扫）→ index → range → ref → const 是否合理？
   - key 列：用到的索引是什么？
   - rows 列：扫描行数是否过大？
   - Extra 列：Using filesort（额外排序）/ Using temporary（临时表）→ 性能杀手

3. **给出优化方案**（15min）
   - 建联合索引：`(created_at, user_id, product_id)`
   - 覆盖索引避免回表
   - 水平分表策略（如果单表过大）

**关联知识**：第三个月/第九周/day1-2-索引.md、第三个月/第九周/day5-MVCC锁.md

## 实验2：Redis 内存持续增长导致 OOM（40min）

**场景**：Redis 集群中一个节点内存使用率持续增长，从 60% 到 95%，触发内存淘汰，大量 key 被驱逐，业务缓存命中率骤降。

**你的任务**：

1. **分析内存使用情况**（10min）
   ```bash
   # 查看内存概况
   redis-cli INFO memory
   # 关注：used_memory_rss / used_memory / mem_fragmentation_ratio
   
   # 查看 key 大小分布
   redis-cli --bigkeys
   
   # 查看无过期时间的 key
   redis-cli --scan --pattern '*' | while read k; do
     ttl=$(redis-cli TTL "$k")
     if [ "$ttl" = "-1" ]; then echo "$k: NO EXPIRE"; fi
   done
   ```

2. **定位问题根因**（15min）
   - 内存碎片率高（mem_fragmentation_ratio > 1.5）→ activedefrag 或重启
   - 大量 key 无过期时间 → 批量加 TTL
   - 某个 key 的 value 异常大（如缓存了大量无用数据）→ 优化数据结构
   - 内存淘汰策略不当（noeviction 直接报错）→ 改为 allkeys-lru

3. **制定优化方案**（15min）
   - 设置 maxmemory-policy allkeys-lru（不是 volatile-lru）
   - 对热点 key 使用不同过期时间（打散，避免缓存雪崩）
   - 设计内存监控告警（used_memory > 80% 预警）
   - 如果数据量确实大，考虑 Redis Cluster 分片

**关联知识**：第三个月/第十周/day4-5-Redis.md、第四个月/第十四周/day1-2-高并发.md（缓存三问题）

## 实验3：分布式事务一致性故障（60min）

**场景**：电商下单流程（订单服务→库存服务→优惠券服务），使用 Seata AT 模式。偶尔出现"订单创建成功但库存未扣减"或"库存已扣减但优惠券未使用"。

**你的任务**：

1. **复现故障场景**（15min）
   ```sql
   -- 检查不一致数据
   -- 订单表中有记录但库存表无扣减记录
   SELECT o.id, o.status, o.created_at
   FROM orders o
   LEFT JOIN inventory_log i ON o.id = i.order_id
   WHERE i.id IS NULL AND o.created_at > '2026-01-01';
   
   -- 检查 undo_log 表
   SELECT * FROM undo_log 
   WHERE log_status != 0  -- 非正常状态
   ORDER BY log_created DESC LIMIT 20;
   ```

2. **分析可能原因**（15min）
   - TC（Transaction Coordinator）宕机 → 分支事务状态未知
   - 网络超时 → 分支事务注册成功但二阶段通知丢失
   - undo_log 表损坏 → 回滚时无法读取 before image
   - 业务异常未被 GlobalTransactional 捕获

3. **设计修复方案**（30min）
   - 加入定时任务扫描长时间未完成的全局事务，人工或自动补偿
   - 对账系统：每日凌晨比对订单表、库存表、优惠券表的数据一致性
   - 监控告警：undo_log 异常状态数量 > 0 立即告警
   - 降级方案：如果 Seata 不可用，临时用本地消息表 + 定时重试代替
   - 面试中如何讲述：先描述故障现象 → 排查工具和方法 → 根因分析 → 短期修复+长期优化

**关联知识**：第三个月/第十一周/day5-分布式事务.md（Seata AT 模式/undo_log 原理）、第三个月/第十一周/day1-2-CAP一致性.md
