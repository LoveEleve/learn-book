**四条主线并行推进**：
- **主线1（MySQL 内核）**：行格式 → B+树索引 → Buffer Pool → redo/undo log → MVCC/ReadView → 锁机制 → Performance Schema/备份恢复
- **主线2（MySQL 高级 + Redis + 中间件）**：并发控制 → 主从复制/GTID/并行复制 → Redis 数据结构 → 持久化/哨兵/集群 → RocketMQ 核心(CommitLog/事务消息) → Kafka 高吞吐/ISR → Etcd Raft/分布式锁
- **主线3（分布式理论）**：CAP/BASE → Paxos 两阶段 → Raft 三子问题 → 分布式事务（2PC/3PC/TCC/Saga）→ 一致性模型
- **主线4（系统设计）**：高并发架构 → 负载均衡 → 缓存架构 → 限流熔断 → 秒杀/短链/Feed流/IM/排行榜

**新增内容（2026-06-17 补充）**：
- **LSM Tree 存储引擎**：B+树之外的索引选择，理解 RocksDB/HBase/Cassandra 的底层原理
- **RocketMQ 核心**：金融级消息队列，掌握 CommitLog 全局顺序写、事务消息半消息+回查、Dledger 选主
- **Kafka 核心**：高吞吐消息队列，掌握 ISR 日志复制、零拷贝、幂等与事务
- **Etcd 与 Raft**：分布式共识引擎，掌握 Raft 三子问题、分布式锁、MVCC 存储
- **MySQL 运维实战**：Performance Schema 诊断、备份恢复（mysqldump/XtraBackup/PITR）、Schema 设计

**四条主线的交汇点**：
- MySQL Buffer Pool LRU ← → Redis 内存淘汰策略 ← → 系统设计中的缓存架构
- MySQL 主从复制 binlog ← → RocketMQ 主从同步（Dledger） ← → Kafka ISR 日志复制 ← → Raft 日志复制
- MySQL 间隙锁 ← → Redis 分布式锁 ← → 分布式事务的并发控制
- MySQL MVCC/ReadView ← → RocketMQ 事务消息半消息 ← → Kafka 幂等生产者 ← → 分布式快照隔离（Percolator）
- MySQL B+树索引 ← → LSM Tree 写优化 ← → RocketMQ CommitLog 顺序写 ← → Kafka Partition 分段写

## 必须掌握的面试话题（第三月）

### MySQL 内核（面试高频区）

1. **InnoDB 行格式与数据页结构**：Compact/Dynamic 行格式的区别？溢出页（overflow page）什么时候触发？Page Directory 的二分查找原理？为什么 InnoDB 页默认 16KB？——与磁盘 IO 单元和操作系统页大小的关系
2. **B+树索引（聚簇索引/二级索引/联合索引/最左前缀）**：为什么 B+树比 B 树更适合磁盘 IO？聚簇索引和非聚簇索引的物理区别是什么？联合索引的最左前缀原则是怎么工作的？能画出 (a,b,c) 联合索引在 B+树上的存储结构吗？——回表是什么？覆盖索引如何避免回表？
3. **Buffer Pool 与 LRU 算法**：Buffer Pool 的三条链表（free/LRU/flush）分别管什么？为什么要用改良的 LRU（midpoint insertion strategy，冷热分离 3/8 位置）而不是标准 LRU？预读机制（线性预读/随机预读）什么时候触发？Buffer Pool 实例数如何设置？
4. **redo log（WAL 机制）**：为什么需要 redo log？redo log buffer → redo log file 的写入时机（事务提交/每秒刷新/buffer 满）？组提交（group commit）如何提高性能？redo log 是物理日志还是逻辑日志？——为什么 redo log 是循环写的，binlog 是追加写的？
5. **undo log 与 MVCC**：undo log 的版本链是怎么组织的？ReadView 在 RC 和 RR 下分别什么时候生成？为什么 RC 下每次 SELECT 都会生成新的 ReadView，而 RR 只在第一次 SELECT 时生成？——MVCC 解决了什么问题？解决了读-写冲突，让读不阻塞写
6. **间隙锁（Gap Lock）与临键锁（Next-Key Lock）**：为什么 RR 隔离级别下需要间隙锁？间隙锁解决幻读的原理是什么？什么场景下间隙锁会降级为行锁或升级为临键锁？——能举例说明不同 SQL（=、>、<、like）在 RR 下分别加什么锁

### Redis（面试高频区）

7. **Redis 五大基本数据结构及其底层实现**：String（SDS，简单动态字符串）、List（3.2 前 ziplist/linkedlist，3.2 后 quicklist）、Hash（ziplist/hashtable）、Set（intset/hashtable）、ZSet（ziplist/skiplist+hashtable）——每种结构的使用场景是什么？底层编码切换的阈值是多少？
8. **Redis 持久化（RDB vs AOF vs 混合持久化）**：RDB 的 COW（写时复制）原理？AOF 的三种刷盘策略（always/everysec/no）？AOF 重写（rewrite）是怎么减少文件大小的？混合持久化（RDB+AOF）的格式是什么？——持久化对性能的影响有多大？
9. **Redis 哨兵（Sentinel）与集群（Cluster）**：哨兵的主观下线（SDOWN）和客观下线（ODOWN）判断标准？哨兵领导者选举（Raft 简化版）的流程？集群模式的哈希槽（16384 个 slot）分配和重分配？——为什么集群用 16384 个槽而不是更多？集群模式下的事务和 Lua 脚本有什么限制？

### 分布式理论（面试区分度区）

10. **CAP 定理**：CAP 的含义（Consistency/Availability/Partition Tolerance）？为什么 CAP 只能三选二？——实际上 P 必须选，所以是 CP vs AP 的权衡。MySQL 主从复制是 CP 还是 AP？Redis 哨兵/集群呢？ZooKeeper 呢？Eureka 呢？——能举例说明不同系统的 CAP 选择
11. **Paxos 两阶段协议**：Basic Paxos 的 Prepare/Promise→Propose/Accept 两个阶段分别做什么？为什么要两阶段（解决活锁+保证安全）？Multi-Paxos 的优化点（领导者租约/省略 Prepare）？——Paxos 为什么难理解？难在哪？
12. **Raft 三子问题**：领导人选举（term/RequestVote RPC/随机超时）、日志复制（AppendEntries RPC/committed index/matchIndex/nextIndex）、安全性（选举限制——Candidate 的日志必须至少和大多数一样新、提交限制——Leader 只能提交当前 term 的日志）——Raft 如何保证线性一致性？Raft vs Paxos 的核心区别？
13. **分布式事务（2PC/3PC/TCC/Saga）**：2PC 的协调者单点故障怎么处理？3PC 的 preCommit 阶段解决了什么问题？TCC（Try/Confirm/Cancel）的补偿机制如何设计？Saga 的事务补偿模式（正向补偿/反向补偿）？——各自适合什么业务场景？
14. **一致性模型谱系**：线性一致性（Linearizability）→ 顺序一致性（Sequential Consistency）→ 因果一致性（Causal Consistency）→ 最终一致性（Eventual Consistency）——如何从强到弱排列？哪些系统实现了哪种一致性？能举例说明因果一致性用向量时钟如何实现吗？

### 系统设计（面试综合区）

15. **高并发系统设计的核心套路**：负载均衡（DNS/L4/L7/一致性哈希）→ 缓存（多级缓存/缓存穿透击穿雪崩/缓存一致性）→ 异步（消息队列解耦/削峰填谷）→ 数据库（读写分离/分库分表/NoSQL 互补）→ 限流熔断降级（令牌桶/漏桶/滑动窗口/Sentinel/Hystrix）——能画出完整的系统架构图并解释每一层的设计决策吗？

## 学习方法提醒（来自学习大纲）

1. **先骨架后细节**：先通读建立脉络（第1遍），再精读核心章节并动手验证（第2遍）——例如 MySQL 先理解 SQL 执行全路径（解析→优化→执行→存储引擎）的宏观流程，再深入 InnoDB 的行格式/B+树/MVCC 细节
2. **多书交叉印证**：同一主题参考 2-3 本不同作者的书——例如 MySQL MVCC 同时看小孩子4919（通俗入门）、书46（内核实现）、书52（锁专题）三种角度
3. **EXPLAIN 分析优先**：每学一个索引知识点，立刻用 EXPLAIN 验证——看执行计划比看书直观 10 倍。看到 type=ALL 就要问"为什么没走索引"，看到 Extra=Using filesort 就要问"怎么建索引能避免排序"
4. **动手搭建优先**：MySQL 主从复制和 Redis 哨兵/集群必须亲手搭建——理解原理的最佳方式是看日志、看状态、手动故障转移
5. **画架构图优先**：系统设计题必须画图——面试官看的是你"能不能把复杂系统画清楚"，而不是听你背诵方案。每个系统设计题至少画 3 张图：整体架构图、核心流程图、数据流图
6. **费曼学习法**：学完后用自己话讲一遍，讲不出来 = 没学会——特别是 CAP 定理、Paxos 两阶段、Raft 三子问题，能脱稿画图讲解才算掌握
7. **面试导向**：每个主题学完后问自己"面试官会怎么问这个问题"，准备 3 分钟回答——InnoDB MVCC/间隙锁、Redis 持久化/集群、CAP/Paxos/Raft、系统设计是最高概率面试题
8. **关联前两个月**：学 MySQL Buffer Pool 时回想 Linux 页缓存（第一个月）、学 Redis RDB 的 COW 时回想 fork() 的 COW 机制（第一个月）、学 Raft 日志复制时回想 Kafka ISR 日志复制（第二个月）——底层知识和中间件知识是理解数据层的钥匙
9. **不要贪多**：第三个月核心书：书45/46/55/57/90/94/103/106/104。其余书作为查阅/补充
