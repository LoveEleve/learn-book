## 本月3个核心知识点（随机抽取）

### 1. G1 SATB 并发标记
**讲述要求**：不是背"初始标记→并发标记→最终标记→筛选回收"的阶段名，而是讲清楚"为什么要有并发标记"和"并发标记中怎么保证不漏标"。用"老师边上课边擦黑板"的比喻：SATB（Snapshot-At-The-Beginning）就是在开始并发标记时给黑板拍张照片，之后擦掉的字通过写屏障记录下来，这样不会漏。

**核心概念链**：STW 问题 → 需要并发标记 → 并发期间 mutator 在修改（漏标风险）→ SATB 机制：标记开始时拍快照 → 并发期间新分配的活对象默认标记为存活 → 写屏障记录引用断裂 → 最终标记处理 SATB 队列。

**参考资料**：第五周/day1-2-G1堆结构.md、第五周/day3-4-SATB标记.md、第五周/day5-混合回收.md

### 2. Kafka ISR vs Raft 的一致性对比
**讲述要求**：不从技术细节讲，而是从"为什么它们选择了不同的方案"这个角度切入。用"快递配送"比喻：Kafka 像顺丰的派送员送到楼下就走了（acks=1 只要 Leader 确认），Raft 像必须本人签收（多数派确认）。为什么 Kafka 可以这样？因为它后续有 ISR 兜底。

**核心概念链**：Kafka ISR（In-Sync Replicas）→ Leader 写 + Follower 异步拉取 → HW（High Watermark）取 ISR 中最小的 LEO → 消费者只能消费到 HW。Raft → Leader 负责日志复制 → AppendEntries RPC → 多数派确认后才提交。两者权衡：Kafka 追求吞吐（允许单点确认），Raft 追求强一致（多数派确认）。

**参考资料**：第八周/day3-4-Kafka.md、第八周/day5-Raft.md

### 3. JMM happens-before 规则
**讲述要求**：不是列举8条规则，而是讲清楚"为什么需要 JMM"。用"两个人用同一个记事本"的比喻——如果没有规则，A 写了内容 B 可能看不到；有了规则，才能有序协作。

**核心概念链**：CPU 和编译器会重排序 → Java 内存模型定义跨线程的可见性规则 → happens-before 是偏序关系 → 程序顺序规则/volatile 规则/锁规则/传递性 → 用 DCL（双重检查锁定）说明 volatile 的必要性 → volatile 禁止指令重排 + 保证可见性。

**参考资料**：第六周/day1-2-JMM.md
