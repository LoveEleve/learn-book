# 第五个月：优化+冲刺层（140h）

> **目标**：打通从 Java 应用到 Linux 内核的性能全链路，掌握 eBPF 追踪与可观测性体系，完成算法冲刺与面试准备，建立端到端性能优化方法论和面试竞争力。
> **核心理念**：前四个月建好了从底层到架构的完整技术体系，第五个月是"能力输出层"——性能优化是"向上诊断、向下追溯"的能力，算法冲刺和面试准备是"把知识转化为面试表现"的能力。
> **月末产出**：能独立完成端到端性能优化（从 perf 火焰图到 JIT 编译优化到内核 eBPF 追踪）、能搭建 Prometheus + Grafana + Jaeger 可观测性体系、能用 bpftrace 写出生产级追踪脚本、LeetCode Hot 100 熟练手写、项目经历 STAR 法则流畅讲述、简历通过率显著提升、完成多轮模拟面试。

>
> **本规划依据**：[学习大纲.md](../学习大纲.md) 第五阶段（性能优化 5本 + eBPF/可观测性 5本）+ [4个月面试冲刺路线.md](../4个月面试冲刺路线.md)

**三条主线并行推进**：
- **主线1（性能优化 + eBPF）**：CPU 性能分析 → Java 性能深入 → 系统全栈调优 → eBPF 追踪 → 可观测性体系

- 性能分析 ← → 算法中的复杂度分析（都是"定位瓶颈"的思维）

## 必须掌握的面试话题（第五月）

### 性能优化（P7 区分度话题）

1. **perf 火焰图分析**：如何用 perf record 采集数据？火焰图的 x 轴和 y 轴分别代表什么？on-CPU 和 off-CPU 火焰图的区别？如何从火焰图中定位性能瓶颈？——"看到一个大平顶"是什么意思？

2. **CPU 微架构性能指标**：IPC（Instructions Per Cycle）和 CPI 的关系？Cache Miss 分为哪几类（L1/L2/L3/LLC）？TLB Miss 什么时候发生？分支预测失败率如何影响性能？

2.5. **Top-Down 微架构分析**：Intel Top-Down 四级分类（Retiring/Bad Speculation/Frontend Bound/Backend Bound）分别代表什么？如何用 `perf stat --topdown` 定位 CPU 瓶颈类型？

3. **JIT 编译深入**：C1 和 C2 编译器分别做什么？分层编译的 5 个级别（0-4）？内联决策的依据是什么？虚方法调用如何内联？逃逸分析优化了什么？

4. **GC 深入调优**：G1 的 IHOP 如何影响并发标记触发时机？MaxGCPauseMillis 设置多大合理？ZGC 的染色指针和 G1 的 Remembered Set 在并发标记上的设计差异？

5. **JMH 高级用法**：JMH 的 @State/@Benchmark/@Setup/@TearDown 注解？@CompilerControl 如何控制内联？Blackhole 的作用？如何避免 DCE？

### eBPF（P7/P8 区分度话题）

6. **eBPF 架构**：eBPF 程序的加载流程（用户态 → BPF 系统调用 → verifier → JIT → attach）？verifier 做了哪些安全检查？maps 的作用是什么？

7. **bpftrace 实战**：kprobe 和 uprobe 的区别？tracepoint 和 USDT 的区别？如何用 bpftrace 统计系统调用耗时？

7.5. **CO-RE/BTF**：CO-RE 解决什么问题？BTF 是什么？vmlinux.h 如何生成？libbpf + CO-RE 和 BCC 的优缺点对比？

8. **XDP 高性能数据路径**：XDP 在数据包处理路径中的位置？XDP 的三种模式（native/offloaded/generic）？XDP 比 DPDK 的优势是什么？

### 可观测性（P7 区分度话题）

9. **可观测性三大支柱**：Metrics、Tracing、Logging 的定位和关联？三者如何通过 TraceID/SpanID 关联？

10. **Prometheus 指标类型**：Counter、Gauge、Histogram、Summary 的区别？Histogram 的 bucket 如何设计？P99 延迟用什么指标类型？

11. **Jaeger 分布式追踪**：Span 和 Trace 的关系？SpanContext 包含哪些字段？采样策略有哪些？

### 系统全栈调优

12. **内存性能分析**：虚拟内存 vs 物理内存、缺页（major/minor page fault）的区别？内存泄漏如何定位？

13. **磁盘 IO 性能**：IOPS、吞吐量、延迟的关系？随机 IO vs 顺序 IO 的性能差异？iostat 的 %util 为什么可能误导？

14. **网络性能调优**：TCP 调优参数（tcp_nodelay/tcp_tw_reuse/tcp_max_syn_backlog）？带宽延迟积（BDP）和 TCP 缓冲区大小的关系？

### 性能方法论

15. **USE 方法论 vs RED 方法**：USE（Utilization/Saturation/Errors）适用于什么场景？RED（Rate/Errors/Duration）适用于什么场景？如何从"系统变慢了"定位到根因？

### 面试准备

16. **算法模板默写**：快排/归并/二分查找/反转链表/LRU Cache/二叉树遍历/DP 背包
17. **系统设计架构图**：秒杀/短链/Feed流/IM/排行榜 完整架构图
18. **STAR 项目自述**：最有挑战的项目、主导的技术改进
19. **面试节奏**：先面非目标公司练手 → 再面目标公司 → 最后面 dream company

## 学习方法提醒

1. **先问题后工具**：不要上来就学 perf 的几十个参数——先问"这段代码 CPU 高是什么原因？"，然后学 perf top 看热点，再学 perf record + 火焰图深入分析
2. **火焰图是性能分析的"X光片"**：每次性能分析都从火焰图开始——看一眼火焰图就能判断瓶颈在哪个函数
3. **eBPF 是"内核的显微镜"**：传统的 perf/strace 只能看到"发生了什么"，eBPF 能让你看到"为什么发生"
4. **可观测性不是监控**：监控告诉你"系统有问题"，可观测性告诉你"为什么有问题"
5. **JMH 写基准测试的陷阱**：JVM 预热（warmup）、DCE（死代码消除）、OSR（栈上替换）——不了解这些陷阱，基准测试的结果就是废纸
6. **USE/RED 方法论是面试"系统变慢"问题的标准答案**：先按 USE 检查资源，再按 RED 检查服务
7. **算法先做后看**：每题先自己写 10-15 分钟，写不出来再看题解。直接看答案等于没练
8. **手写代码**：不要依赖 IDE 自动补全，面试需要白板/编辑器手写
9. **STAR 法则反复打磨**：每个项目至少讲 10 遍，录下来听，直到流畅不卡壳
10. **模拟面试要当真**：穿正装、开摄像头、严格计时。不要因为是模拟就随意
11. **先面非目标公司练手**：把最好的状态留给最想去的公司
12. **每一轮面试后立即复盘**：记录问题和回答质量，为下一轮面试做准备
