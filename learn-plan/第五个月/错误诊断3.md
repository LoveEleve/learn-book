## 实验1：JVM 频繁 Full GC 导致 STW 超时（40min）

**场景**：一个 8GB 堆的 Java 应用（G1 GC），正常运行半年后开始出现频繁 Full GC，每次 STW 时间超过 5 秒，导致服务健康检查超时被 K8s 重启。

**你的任务**：

1. **分析 GC 日志**（10min）
   ```bash
   # JVM 参数开启 GC 日志
   -Xlog:gc*:file=/tmp/gc.log:time,level,tags
   
   # 分析关键指标
   # 关注：Full GC 频率、每次 Full GC 的回收量、GC 前后的堆使用情况
   # 判断是正常的内存压力还是内存泄漏
   ```

2. **用 MAT/JProfiler 分析堆 dump**（15min）
   ```bash
   # 获取堆 dump
   jmap -dump:format=b,file=/tmp/heap.hprof <pid>
   
   # MAT 分析步骤：
   # 1. Leak Suspects Report（泄漏嫌疑报告）
   # 2. 查看 Dominator Tree（支配树）——找到占用最大的对象
   # 3. 追踪 GC Roots 路径 —— 为什么这个对象不能被回收
   # 常见泄漏模式：
   # - ThreadLocal 未 remove（线程池+ThreadLocal=内存泄漏）
   # - 静态 Map 不断 put 不清理
   # - 长生命周期的对象持有短生命周期引用
   ```

3. **给出诊断结论和优化方案**（15min）
   - 如果是内存泄漏：修复代码 + 重启释放
   - 如果是正常压力：增大堆 / 减少缓存 / 缩短对象生命周期
   - 预防措施：-XX:+HeapDumpOnOutOfMemoryError / 监控堆使用率 / GC 频率告警

**关联知识**：第二个月/第五周/day5-混合回收.md（G1 Full GC 触发条件）、第五个月/第十八周/day1-2-Java性能深入.md

## 实验2：微服务 CPU 100% 问题全链路排查（50min）

**场景**：一个 Spring Boot 微服务 CPU 使用率突然飙升至 100%，用户请求超时，但监控显示 QPS 并未增长。

**你的任务**：

1. **用 Arthas 定位 CPU 热点线程**（10min）
   ```bash
   # 安装并启动 Arthas
   java -jar arthas-boot.jar
   
   # 查看最忙的线程
   thread -n 5
   
   # 查看线程的调用栈
   thread <tid>
   
   # 如果怀疑是死循环，用反编译查看方法
   jad com.example.service.OrderService --source-only
   ```

2. **用 perf 火焰图分析 CPU 热点**（15min）
   ```bash
   # 记录 perf 数据
   perf record -F 99 -p <pid> -g -- sleep 30
   # 生成火焰图
   perf script | stackcollapse-perf.pl | flamegraph.pl > flame.svg
   
   # 分析火焰图：
   # - 平台宽度 = CPU 占比
   # - 从下往上 = 调用链
   # - 对比正常时期的火焰图找差异
   ```

3. **用 eBPF 追踪系统调用**（15min）
   ```bash
   # bpftrace 追踪 futex 锁争用（Java synchronized 底层）
   sudo bpftrace -e '
   kprobe:futex_wait {
     @futex_wait[tid] = count();
   }
   interval:s:10 {
     print(@futex_wait);
     clear(@futex_wait);
   }'
   # 如果大量线程在 futex_wait → JVM 锁竞争严重
   ```

4. **给出诊断和修复方案**（10min）
   - 死循环：注意 HashMap 并发 resize（1.7）→ 排查并发容器
   - 锁竞争：synchronized 锁粗化 → 优化锁粒度或改用无锁结构
   - GC 频繁（STW + 并发标记）→ 调整 GC 参数
   - 正则表达式回溯：回溯爆炸 → 优化正则或用普通字符串操作

**关联知识**：第五个月/第十七周/day1-2-CPU性能分析.md、第五个月/第十七周/day3-4-eBPF实战.md

## 实验3：Go 并发 Bug 定位（40min）

**场景**：一个 Go 写的 API 网关，偶发 panic（`send on closed channel`）和 goroutine 泄漏。`go tool pprof` 显示 goroutine 数量持续增长。

**你的任务**：

1. **用 pprof 分析 goroutine 泄漏**（15min）
   ```bash
   # 开启 pprof
   import _ "net/http/pprof"
   
   # 查看 goroutine 数量趋势
   curl http://localhost:6060/debug/pprof/goroutine?debug=1 | grep "^goroutine" | wc -l
   
   # 用 pprof 分析 goroutine 泄漏
   go tool pprof http://localhost:6060/debug/pprof/goroutine
   (pprof) top
   (pprof) list <function_name>
   ```

2. **定位 channel 关闭问题**（15min）
   - `send on closed channel` 的常见模式：
     ```go
     // 错误：多个 goroutine 都可能关闭 channel
     go func() { close(ch) }()
     go func() { close(ch) }()  // panic!
     
     // 正确：只有发送方关闭 channel
     // 或者用 sync.Once 保证只关闭一次
     ```
   - goroutine 泄漏的常见模式：
     ```go
     // 错误：channel 永远没有接收者，发送 goroutine 永远阻塞
     ch := make(chan int)
     go func() { ch <- 1 }()  // 泄漏
     
     // 正确：用 buffered channel 或 select + default + timeout
     ```

3. **用 Go race detector 检测数据竞争**（10min）
   ```bash
   go test -race ./...           # 测试时检测
   go build -race -o gateway .   # 编译时开启 race
   ./gateway                     # 运行（性能有损耗，仅排查用）
   ```

**关联知识**：第六个月/第二十一周/day1-2-Go并发模型.md（channel 原理）、第六个月/第二十一周/day2.5-Go并发原语.md（RWMutex/WaitGroup/Cond）
