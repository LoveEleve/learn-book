# 知识迁移地图

> 跨月知识连接的抽象主题地图。当你学一个新知识但感觉"这好像在哪见过"——恭喜，你在建立知识的深层连接。这些连接是面试中"举一反三"的弹药库。

---

## 核心抽象：缓存模式

缓存是计算机科学中反复出现的核心抽象——用空间换时间，用局部性做加速。

```
M1 TLB（虚拟→物理地址缓存）
│  硬件缓存的典范：4 路组相联、LRU 替换
│
├─→ M2 G1 RSet（跨 Region 引用缓存）
│   记住"谁引用了本 Region 中的对象"，避免全堆扫描
│
├─→ M3 Buffer Pool（数据库页缓存）
│   LRU 链表（old→young 分代）+ 自适应哈希索引
│
├─→ M3 Redis（内存缓存）
│   多级缓存第一层：CDN → Nginx 本地缓存 → Redis → MySQL
│   缓存三问题（穿透/击穿/雪崩）→ 本质是"缓存失效对后端的冲击"
│
└─→ M4 多级缓存架构
     CDN（边缘）→ Nginx 本地缓存（L1）→ Redis（L2）→ Caffeine（L3）→ MySQL
     每层解决不同问题：CDN 减小带宽、Redis 减小 DB 压力、本地缓存减小网络开销
```

**面试金句**：从硬件 TLB 到 G1 RSet 到 Redis 缓存，缓存的核心问题是"什么时候失效？失效后怎么兜底？"——TLB 有缺页中断兜底，G1 有全堆扫描兜底，Redis 有 DB 查询兜底。

## 核心抽象：事件驱动

事件驱动是一种"不轮询、等通知"的编程范式，从内核到应用层一脉相承。

```
M1 epoll（内核事件通知）
│  epoll_create → epoll_ctl 注册 fd → epoll_wait 等待事件
│  事件驱动内核实现：callback 机制把就绪 fd 放入 rdllist
│
├─→ M1 Reactor（应用层事件循环）
│    EventLoop + Channel + Poller + Handler 回调
│    one loop per thread：每个线程一个 epoll
│
├─→ M4 Netty（框架层事件驱动）
│    BossGroup（accept）→ WorkGroup（IO 读写）→ Pipeline 责任链
│    ChannelHandler 双向链表：Inbound 解码→业务→Outbound 编码
│
├─→ M3 Kafka（消息事件流）
│    Producer → Broker（ISR 复制）→ Consumer Group（pull 模型）
│    pull vs push：Kafka 选 pull（消费者控制速率）
│
└─→ M6 Go Channel（CSP 事件通信）
│    goroutine 通过 channel 传递事件，select 多路监听
│    channel ←→ epoll 的哲学共通：不共享内存来通信，而是通过通信来共享内存
```

**面试金句**：从 Linux epoll 到 Netty 到 Kafka，事件驱动的核心是"不要问我，我会通知你"——epoll 通知 fd 就绪，Netty 通知 Handler 执行，Kafka 通知消费者有消息。

## 核心抽象：复制与一致性

分布式系统中最核心的抽象——通过复制保证高可用，通过一致性协议保证复制不出错。

```
M3 MySQL 主从复制（binlog）
│  异步复制 → 半同步复制 → Group Replication
│  宕机切换问题 → MHA/Orchestrator
│
├─→ M2 Raft 日志复制（WAL + 多数派）
│    Leader AppendEntries → Follower 持久化 → 多数派确认 → 提交
│    选举限制 → 保证新 Leader 日志不落后
│
├─→ M3 Kafka ISR 复制
│    Leader → Follower 拉取（Pull），HW = min(LEO) in ISR
│    ISR ≠ Raft：ISR 允许 acks=1（单点确认快），Raft 要求多数派
│
├─→ M2 Etcd Raft 复制
│    Key-Value 的 Raft 实现：MVCC + Watch + Lease
│    对比 Kafka：Kafka 用 Raft 管元数据（KRaft），消息数据仍用 ISR
│
└─→ M4 K8s etcd
│    K8s 的控制平面数据库：所有组件只写 API Server → etcd 持久化
│    Watch 机制 → Controller 调谐循环（观察当前状态→对比期望状态→调整）
```

**面试金句**：从 MySQL binlog 到 Raft 日志到 Kafka 消息，所有复制系统的核心问题是"怎么保证复制不出错"——MySQL 用 GTID 防止重复，Raft 用 Term+Index 保证唯一序，Kafka 用 HW 保证消费者不会读到未确认的消息。

## 核心抽象：分层架构

软件架构的终极模式——每层只与相邻层交互，下层为上层提供抽象。

```
M1 Linux 内核分层
│  用户态 → 系统调用层 → VFS → 文件系统 → 块设备驱动 → 磁盘
│  网络：用户态 → socket → TCP/UDP → IP → 链路层 → 网卡驱动
│
├─→ M4 Tomcat 分层
│    Server → Service → Engine → Host → Context → Wrapper
│    每层通过 Pipeline-Valve 链传递请求
│
├─→ M4 Spring 分层
│    Controller → Service → Repository → 数据源
│    每层通过 AOP 横切（事务/日志/安全）
│
├─→ M4 Docker 分层镜像
│    Overlay2：lowerdir（只读镜像层）→ upperdir（可写容器层）→ merged（统一视图）
│    COPY_ON_WRITE：修改时复制到 upperdir
│
└─→ M4 K8s 分层
│    控制平面（API Server/etcd/Scheduler/Controller Manager）
│    工作节点（kubelet/kube-proxy/Container Runtime）
│    Pod → Deployment → Service → Ingress
```

**面试金句**：从 Linux VFS 到 Tomcat Container 到 Docker OverlayFS，分层架构的核心是"每一层只做自己该做的事"——VFS 只提供接口不关心实现，Tomcat Container 只负责路由不关心业务，Docker 镜像层只关心自己的增量。

## 核心抽象：写时复制（Copy-On-Write = COW）

延迟复制的优化思想——不到万不得已不复制，节省内存和 CPU。

```
M1 fork() COW
│   子进程共享父进程页表，页标记为只读
│   任一进程写入 → 缺页中断 → 复制物理页 → 标记可写
│
├─→ M1 C++ 字符串 COW（std::string，旧实现）
│   多个 string 指向同一内存，写入时复制
│   （C++11 后废弃，因为多线程下锁开销 > 复制开销）
│
├─→ M3 Redis RDB COW
│   BGSAVE：父进程 fork → 子进程用 COW 读内存 → 写入 RDB
│   父进程继续写 → COW 自动复制被修改的页 → RDB 是一致快照
│
├─→ M4 Docker 镜像 COW
│   OverlayFS：下层镜像只读 → 容器修改时复制到上层
│   启动 100 个容器 → 基础镜像只有一份在内存
│
└─→ M2 G1 GC SATB
│   （本质也是 COW 思想）并发标记时拍快照 → 之后变更通过写屏障记录
│   标记过程不受 mutator 干扰 → 保证一致性
```

**面试金句**：从 fork() 到 Redis RDB 到 Docker 镜像，COW 的精髓是"先共享，写入时再复制"——fork() 复制页表但不复制物理页，Redis RDB 子进程读父进程内存不需要拷贝，Docker 镜像层共享基础层节省磁盘。

## 核心抽象：无锁/低锁竞态优化

高并发场景下的性能保障——能不加锁就不加锁。

```
M1 CPU 缓存一致性
│   MESI 协议：用状态机替代全局锁
│   缓存行填充（@Contended）：避免伪共享
│
├─→ M2 JVM synchronized 锁升级
│   偏向锁 → 轻量级锁（CAS 自旋）→ 重量级锁（mutex）
│   大多数场景：偏向锁无竞争 → 轻量级锁短竞争 → 重量级锁高竞争
│
├─→ M6 Go sync.Mutex 模式切换
│   正常模式（自旋）→ 饥饿模式（排队）
│   对比 JVM 锁升级：Go 自旋条件更严格（多核+短临界区+低 P 空闲）
│
├─→ M4 Netty 无锁化设计
│   一个 Channel 绑定一个 EventLoop 线程 → 无锁
│   对比：多个线程处理同一 Channel → 需要锁保护
│
└─→ M3 Disruptor RingBuffer
│   无锁队列：CAS + 预分配 + 缓存行填充
│   单生产者或多生产者模式
```

**面试金句**：从 CPU MESI 到 JVM synchronized 到 Go Mutex 到 Netty，并发优化的核心路径是"尽可能不加锁 → 如果必须加锁，让锁的粒度尽可能细 → 如果还是慢，批量操作分摊锁开销"。

## 核心抽象：垃圾回收

自动内存管理的演进——从引用计数到分代到并发。

```
Java GC 路线：
C1 CMS（并发标记清除，碎片问题）
│
└─→ C2 G1（Region + RSet + SATB + Mixed GC，可预测停顿）
│
└─→ C3 ZGC（染色指针 + 并发整理，< 1ms 停顿）
│
└─→ Go GC 路线：
     C0 并发标记清除（STW 较长）
     │
     └─→ C1 并发三色标记+插入写屏障（需 STW 重扫栈）
     │
     └─→ C2 并发三色标记+混合写屏障（Go 1.8，消除栈重扫 STW）

对比：
┌────────────────┬──────────────┬──────────────┐
│ 维度            │ G1           │ Go 1.8+      │
├────────────────┼──────────────┼──────────────┤
│ 标记算法        │ SATB         │ 混合写屏障    │
│ 并发整理        │ 有（Mixed GC）│ 无（非分代）  │
│ Region           │ 有           │ 无（span）   │
│ 预期停顿         │ ~50ms        │ ~0.5ms      │
│ 堆大小            │ 适合大堆     │ 适合中等堆   │
└────────────────┴──────────────┴──────────────┘
```

**面试金句**：并发标记的核心难题是"漏标"——G1 用 SATB（拍快照），Go 用混合写屏障（Dijkstra + Yuasa），ZGC 用染色指针（读屏障）。三种方案各有取舍，但都遵循一个原则：在并发标记期间，要么记录旧值（SATB），要么记录新值（插入屏障），要么两者都记录（混合屏障）。

## 如何使用这张地图

1. **学新知识时**：查找它所在的抽象主题，找到和已学知识的连接点
2. **面试准备时**：用抽象主题作为"串讲"结构——"我对缓存的理解从硬件 TLB 开始..."
3. **系统设计时**：对照抽象主题检查你是否考虑了对应的问题——例：设计缓存层时，要考虑缓存失效对后端的冲击（缓存三问题）
